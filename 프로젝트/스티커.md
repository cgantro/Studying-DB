## 한 줄 정의
모노레포(FashionApp/Server/ai) 형태의 “실시간 코디 매칭 서비스”에서, 서버 배포 안정화(Healthcheck/Traefik/환경 주입)와 CI 파이프라인 품질을 중심으로 개선을 반복하는 프로젝트.

## 내 역할
- (내 담당) Server 배포 안정화: Healthcheck 경로/권한 정리, 단일 EC2 듀얼 스택(staging/prod) 배포 운영
- (내 담당) GitLab CI 파이프라인 정리: MR 빌드/테스트, develop→staging, main→prod 라우팅
- (내 담당) “문서/정책” 기반 운영: 시크릿 문서화 금지, 런타임 주입 규칙, 모노레포 경계 명시

## 기술 선택 이유 + 대안/트레이드오프(요약)
- GitLab CI(부모 파이프라인 → 하위 파이프라인 트리거):
  - 선택: 모노레포에서 변경 범위 기준으로 파이프라인을 분기하기 쉬움
  - 대안: 단일 거대 파이프라인(속도↓, 책임 경계↓)
- Traefik + Docker Compose(단일 EC2 듀얼 스택):
  - 선택: staging/prod를 같은 서버에서 호스트 기반 라우팅으로 분리
  - 대안: 환경별 EC2 분리(비용↑), k8s(운영 복잡도↑)

## 문제 -> 원인 -> 해결 (요약)
### 1) 배포 Healthcheck가 인증 때문에 실패
- 문제: 배포 스크립트가 `/actuator/health`로 상태 확인을 하는데, 인증이 걸리면 배포가 실패한다.
- 원인: 보안 경계(인증 필요 API)와 운영 헬스체크 경계를 분리하지 않았음.
- 해결: `SecurityConfig`에서 `/actuator/health/**`를 `permitAll`로 열고, 나머지 사용자 자원은 `authenticated`로 유지.
- 대안: 별도 내부용 health endpoint(내부 네트워크 전용) 운영.

### 2) 배포 시 “변수 누락/플레이스홀더 남음”으로 사고 위험
- 문제: `${...}` 플레이스홀더가 남은 상태로 원격에 전달되면, 런타임에서 조용히 깨지거나 잘못된 환경으로 배포될 수 있다.
- 원인: CI 변수/문서/스크립트의 경계가 느슨하면 “정의되지 않은 값”이 섞여 들어간다.
- 해결: deploy 스크립트에서 필수 변수 검증(`require_var`) + placeholder 탐지(`ensure_not_placeholder`)로 실패를 조기에 만든다.

## CS 연결
- 네트워크/운영: reverse proxy 라우팅, 헬스체크, 무중단에 가까운 재기동 전략
- 보안: 공개 엔드포인트/인증 엔드포인트 경계 설정
- SW 공학: 모노레포에서 변경 범위 기반 CI 라우팅, 시크릿 관리 규칙
