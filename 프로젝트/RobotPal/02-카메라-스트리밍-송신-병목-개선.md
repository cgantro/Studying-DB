# RobotPal - 카메라 스트리밍(송신) 병목과 개선

## 문제 정의
- 스트리밍을 켜면 앱 FPS가 흔들리거나 내려간다.
- “렌더링/시뮬레이션”과 “픽셀 캡처/인코딩/전송”이 같은 스레드/타이밍에서 경합하면, 체감 지연이 커진다.

## 원인 1: GPU→CPU readback stall
- 증상: `glReadPixels`가 호출되면 CPU가 GPU 완료를 기다리면서 멈출 수 있다.
- 해결: PBO 2개를 번갈아 쓰는 ping-pong 구조로 “이번 프레임은 GPU가 쓰고, CPU는 이전 프레임을 읽는” 형태로 분리.
- 코드 포인트:
  - Desktop: `Texture::GetAsyncData`에서 `GL_PIXEL_PACK_BUFFER` + `glMapBufferRange` 사용
  - Web: 드라이버 호환성 때문에 `GL_RGBA`로 읽고 필요 시 CPU에서 `RGB`로 변환

## 원인 2: JPEG 인코딩(CPU bound) + 메모리 재할당
- 증상: 해상도 상승/인코딩 품질 고정 환경에서 인코딩이 메인 루프와 경쟁.
- 해결:
  - Producer-Consumer로 분리: `StreamingSystemModule(PostFrame)`은 raw 픽셀만 queue에 넣고 리턴
  - 인코딩 워커 N개: 환경변수 `ROBOTPAL_ENCODE_WORKERS`로 고정 가능하게 만들어 A/B 측정
  - 버퍼 재사용: 워커 루프 밖에서 `reserve`하고 루프마다 `clear`로 재활용
  - RGBA 입력은 RGB로 변환 후 인코딩(불필요한 알파 채널 제거)
- 코드 포인트:
  - `StreamingPipeline::EncodeWorkerLoop` (벡터 재사용, 패킷 구성 `len(4B)+jpeg`)

## 대안과 트레이드오프(송신/인코딩 관점)
- JPEG 유지:
  - 장점: 구현 단순, 디버깅 쉬움, 플랫폼 의존성 낮음
  - 단점: 비트레이트 효율이 낮고 CPU 비용이 큼
- H.264/H.265(하드웨어 인코더 포함)로 전환:
  - 장점: 대역폭/품질 효율이 좋고 전송 FPS를 올리기 쉬움
  - 단점: 파이프라인 복잡도 급증(인코더/컨테이너/디코더), Web/Native 동시 대응 난이도 상승
