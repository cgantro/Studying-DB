# RobotPal - 상태/요약

## 한 줄 정의
- 로봇 시뮬레이터에서 “제어 명령(양방향) + 카메라 스트리밍(단방향)”을 Desktop(TCP) / Web(Emscripten WebSocket)로 붙이고, 병목을 측정해서 파이프라인 단위로 해결한 경험.

## 내가 구현/개선한 축(핵심만)
- 카메라 스트리밍 병목 분석/벤치마크 환경 정리 (`ROBOTPAL_ENCODE_WORKERS`, sink 스크립트, 로그 지표)
- 스트리밍 송신 파이프라인: readback stall 줄이기(PBO), JPEG 인코딩 멀티 워커, 메모리 재사용
- 수신 서버(Python): 최신성 우선(Queue size 1), 디코딩을 thread pool로 분리, 화면용 JPEG pass-through

## 면접에서 바로 쓰는 키워드
- “렌더링 루프를 건드리지 않고, 병목을 다른 스레드/단계로 밀어내서 FPS 변동을 줄였다.”
- “TCP는 메시지 경계가 없어서 길이 헤더 기반으로 수신 측에서 버퍼링 파서를 만들었다.”

## 확인 필요
- 팀 규모/내 정확한 역할 범위(기능 개발 vs 최적화) 구분 문장
- 제어 명령 수신(C++ 쪽)에서 TCP fragmentation 대응이 완전한지 점검 필요(리스크로 남겨둠)
