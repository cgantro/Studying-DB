# Autowing Car - 구조/흐름

## 핵심 데이터 흐름(요약)
- Edge(토잉카/AI) → MQTT(Broker) → Backend(Spring Integration) → WebSocket(STOMP) → 관제/기장 웹

```mermaid
sequenceDiagram
  participant Edge as "Towing Car/AI"
  participant MQTT as "MQTT Broker"
  participant BE as "Spring Boot Backend"
  participant DB as "DB(JPA)"
  participant WS as "STOMP(WebSocket)"
  participant Web as "Web Clients"

  Edge->>MQTT: publish(telemetry/ack/command)
  MQTT->>BE: inbound subscribe
  BE->>BE: validate/parse(topic+payload)
  BE->>DB: persist(필요 시)
  BE->>WS: broadcast(/topic/...)
  WS->>Web: real-time UI update
  Web->>WS: /app/... request
  WS->>BE: command request
  BE->>MQTT: publish(cmd/drive/control)
```

## 설계 포인트
- “즉시성(WS)”과 “기록/무결성(DB)”을 분리해서 생각했다.
- 외부 발행(MQTT)은 트랜잭션 커밋 이후로 밀어야 DB 상태와 메시지 발행이 엇갈리지 않는다(After-commit 패턴).
