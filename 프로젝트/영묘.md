# 영묘 (Unreal 멀티플레이/보이스)

## 현재 상태
- **미완성**
- 근거: 영묘 저장소 마지막 커밋 `2026-03-27 (+09:00)` 기준으로 코드/서버가 존재
- 주의: “게임 완성” 관점에서는 미완성이고, 남은 자산은 보이스/룸 스코프/서버 경계 설계 경험 쪽이 크다.

## 한 줄 정의
Unreal 멀티플레이에서 보이스를 “WebSocket 기반(지연/HoL)”에서 “UDP 릴레이 + Opus(프레임 단위)”로 재설계하고, 룸 코드(RoomCode)로 논리적 격리를 강제한 경험.

## 내 역할
- (내 담당) 보이스 통신 구조 개편: UDP 릴레이 서버 + 클라이언트 패킷 포맷 정리
- (내 담당) 코덱 체계 재구축: UE 기본 스트림 코덱 대신 raw libopus 기반 프레임 코덱 구현
- (내 담당) RoomCode 기반 “메시지/게임플레이” 스코프 규칙 추가(다른 방 이벤트/상태 혼입 차단)

## 핵심 구조(요약)
- C++ GameServer:
  - Lobby: WebSocket(uwebsockets)
  - Voice: UDP + 고정 헤더(룸 코드/화자/페이로드 크기) + 룸 단위 릴레이
- Unreal:
  - RoomCode로 월드 오프셋을 분리(같은 맵을 여러 방이 공유하되 서로 안 겹치게)
  - 수신 메시지는 RoomCode가 다르면 무시(클라 UI/이벤트/게임플레이까지 스코프 적용)
  - Voice: PCM 캡처 → Opus 인코딩 → UDP 송신, 수신은 화자별 코덱으로 디코드 후 재생

## 문제 -> 원인 -> 해결 (요약)
### 1) 보이스 지연/끊김
- 문제: TCP(WebSocket) 계열로 음성을 보내면 “최신 프레임”보다 “순서/재전송”이 우선되어 밀림이 누적됨
- 원인: 스트림 기반 전송의 HoL blocking 성격 + 백프레셔
- 해결: UDP 릴레이로 전환하고 “프레임 단위”로 보내는 구조로 변경

### 2) Raw PCM 전송량 과다
- 문제: PCM을 그대로 보내면 대역폭/CPU 부담이 커짐
- 원인: 무압축 PCM은 샘플레이트/비트수에 비례해서 커짐
- 해결: Opus(VoIP 설정)로 인코딩해서 전송
- 정량 수치: **확인 필요** (측정 조건/기간/방법 문서화되지 않음)

### 3) 멀티스레드 디코딩에서 음질/노이즈 문제
- 문제: 동일 화자 코덱을 여러 스레드에서 동시에 디코드하면 깨질 수 있음
- 원인: 코덱 내부 상태/순서 경합
- 해결: 수신 순서대로 GameThread에서 디코드/재생(깨진 패킷은 드롭)

## CS 연결
- 네트워크: UDP vs TCP, HoL, 패킷 크기/검증, 최신성 우선 전략
- OS/성능: 워커 스레드 분리(UDP 수신→큐→worker), shared payload로 복사 비용 절감
- SW 공학: RoomCode로 스코프 규칙을 코드로 강제(“상태 혼입”을 구조적으로 차단)
