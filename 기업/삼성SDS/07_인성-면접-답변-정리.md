# 인성 면접 답변 정리 (삼성SDS)

작성 원칙:
- “삼성SDS/Brity/FabriX” 같은 **회사 고유 요소를 반드시 포함**한다.
- 프로젝트 근거는 [프로젝트 DB](/프로젝트/프로젝트.md)와 연결한다.
- 수치가 필요한 문장은 근거 문서가 없으면 **면접용 버전에서는 수치를 빼고** 말한다.

## 1) 1분 자기소개
### 질문 의도
- 지원자가 어떤 사람인지, 한 문장으로 “쓸모”가 있는지 확인

### 핵심 답변(면접용)
안녕하세요. 삼성SDS 지원자 홍윤표입니다. 저는 기능을 “추가”하는 것보다, 여러 사용자가 동시에 쓰는 환경에서 데이터 흐름이 꼬이지 않게 만들고 운영 리스크를 줄이는 일을 잘합니다. Autowing에서는 실시간 메시징 흐름과 인증/배포를 묶어 운영 가능한 형태로 만들었고, RobotPal에서는 스트리밍 병목을 측정해 파이프라인 단계로 분리해 개선했습니다. 이런 방식으로 Brity 같은 협업 솔루션이 실제 업무 환경에서 안정적으로 동작하도록 만드는 개발자가 되고 싶습니다.

### 근거 경험(짧게)
- Autowing: MQTT↔WebSocket, 인증/배포까지 E2E
- RobotPal: 스트리밍 병목 측정/개선(측정 문서 있음)

### 꼬리질문 후보
- “운영 리스크”를 어떤 지표로 보나?
- 병목을 어떻게 찾았나?

## 2) 왜 삼성SDS인가
### 질문 의도
- 회사/사업 이해 + 본인 방향성과의 1:1 연결

### 핵심 답변
삼성SDS는 B2B 환경에서 협업/자동화/AI 같은 “업무 흐름을 연결하는 제품”을 만들고, 그 환경에서는 안정성, 권한, 운영성이 곧 제품 경쟁력이라고 봅니다. 저는 프로젝트에서 실시간 데이터 흐름에서 병목과 상태 불일치를 줄이는 쪽을 주로 맡아왔고, 이 역량이 Brity 같은 협업 솔루션의 운영 품질을 올리는 데 직접적으로 쓰일 수 있다고 생각해 지원했습니다.

### 꼬리질문 후보
- 왜 다른 SI/클라우드 회사가 아니라 SDS인가?
- SDS 사업 중 관심 영역은?

## 3) 강점
### 질문 의도
- 같이 일할 때 강점이 “업무로 환산”되는지 확인

### 핵심 답변
제 강점은 문제가 생겼을 때 원인을 끝까지 좁히고, 재현/검증 가능한 형태로 고쳐서 팀이 다시 같은 문제를 겪지 않게 만드는 태도입니다. “고쳤다”로 끝내지 않고, 흐름을 분리하거나 규칙을 문서화해서 운영 리스크를 낮추는 방식으로 일해왔습니다.

## 4) 약점
### 질문 의도
- 약점을 아는지, 관리 가능한지 확인

### 핵심 답변
문제 상황에서 해결책을 빨리 제시하려다, 상대가 받아들이는 속도를 놓친 적이 있습니다. 밴드 활동에서 피드백을 직설적으로 줬다가 팀 분위기가 흔들린 경험 이후로, 먼저 상대가 어떤 의도로 했는지와 제약을 확인한 뒤 “기준/대안/트레이드오프”로 이야기하는 방식으로 바꿨습니다.

## 5) 협업 방식
### 질문 의도
- 소통을 “행동”으로 설명할 수 있는지 확인

### 핵심 답변
말을 많이 하는 것보다, 모두가 같은 기준으로 일하게 만드는 장치를 먼저 만듭니다. 예를 들어 브랜치/커밋 규칙, 공용 문서, 운영 체크리스트를 먼저 잡고, 기술 선택 갈등은 성능/보안/운영 제약을 표로 비교해서 합의합니다. 합의 결과는 문서로 남겨 같은 논쟁이 반복되지 않게 합니다.

## 6) 입사 후 가장 먼저 배우고 싶은 것
### 질문 의도
- “성장 방향”이 회사/직무와 맞는지 확인

### 핵심 답변
제가 가장 먼저 배우고 싶은 건 “B2B 협업 솔루션이 운영에서 깨지는 방식”과, 그걸 막기 위한 표준입니다. 예를 들어 권한/감사 로그, 외부 시스템 연동, 장애 대응 기준 같은 부분은 개인 프로젝트에서 흉내 내기 어렵습니다. Brity 같은 제품이 실제 고객 환경에서 안정적으로 굴러가게 만드는 운영 기준을 빠르게 흡수하고 싶습니다.

### 꼬리질문 후보
- 그 기준을 어떻게 빨리 배우나(문서/로그/온콜/장애 리포트)?
- 본인은 어떤 관측 지표를 먼저 붙이나?

## 7) 3년/5년 뒤 성장 목표
### 질문 의도
- “계획”이 업무 단위로 말이 되는지 확인

### 핵심 답변(3년)
3년 안에는 Brity 계열 서비스에서 한 모듈을 맡아, 기능 구현뿐 아니라 배포/관측/장애 대응까지 스스로 책임질 수 있는 수준이 되고 싶습니다. 특히 실시간 이벤트와 상태 정합성 같은 영역에서, 재현 가능한 장애 분석 루틴을 갖춘 개발자가 되는 게 목표입니다.

### 핵심 답변(5년)
5년 뒤에는 여러 서비스/팀 경계를 넘는 문제를 풀고 싶습니다. 예를 들어 “협업 도구 간 데이터 일관성”이나 “권한/감사 로그 표준화” 같은 주제를 구조로 정리하고, 운영 리스크를 줄이는 방식으로 기여하고 싶습니다.

## 8) 삼성SDS에서 내가 기여할 수 있는 부분
### 질문 의도
- “내 역량이 회사 문제에 어떻게 꽂히는지” 확인

### 핵심 답변
Brity 같은 협업 솔루션은 기능 하나를 붙여도 전체 흐름에 영향을 줍니다. 저는 프로젝트에서 실시간 데이터 흐름에서 병목이나 상태 불일치가 생기면, 원인을 단계로 쪼개서 재현하고 구조로 고치는 일을 해왔습니다. 이 경험을 바탕으로 SDS에서는 1) 실시간 이벤트/상태 정합성이 깨지는 지점 식별, 2) 관측/재현 가능한 로그와 지표 정리, 3) 전송/처리 파이프라인 분리 같은 방식으로 운영 품질을 올리는 데 기여할 수 있습니다.

### 근거 경험(짧게)
- RobotPal: 스트리밍 병목을 측정하고 파이프라인으로 분리(측정 문서 존재)
- Autowing: 실시간 메시징 흐름 + 인증/배포까지 E2E로 구성
- mm-bot: 배치 적재/실시간 질의 분리로 실패 격리

## 9) 실패 경험
### 질문 의도
- 문제를 “통제 가능하게” 만든 경험이 있는지 확인

### 핵심 답변(면접용)
초기에 “기능이 된다”에 집중해서, 운영에서 실패가 어디서 났는지 한 번에 파악하기 어려운 구조로 만든 적이 있습니다. 그 이후에는 1) 경계를 먼저 나누고(수신/처리, 캡처/인코딩/전송), 2) 실패가 났을 때 재현 가능한 로그 포인트를 두고, 3) 대안을 비교한 뒤 결정을 문서로 남기는 방식으로 바꿨습니다.

### 꼬리질문 후보
- 구체적으로 어떤 로그/지표를 남겼나?
- 그 실패를 다음 프로젝트에서 어떻게 예방했나?
