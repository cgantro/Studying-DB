## Spring 기본

### 개념
- Spring Boot는 “Spring 기반 애플리케이션을 빠르게 구성/실행/운영”하기 위한 프레임워크다.
- 핵심은 다음 3가지로 요약할 수 있다.
  1) 관례 기반 자동설정(Auto-configuration)
  2) 스타터 의존성(Starter)로 설정 묶음 제공
  3) 운영 친화 기능(Actuator/Micrometer 등) 결합이 쉬움
- Spring(Core)은 DI/IoC 컨테이너(=ApplicationContext)이며, Boot는 그 위에 “부트스트랩/자동설정/운영 경험”을 얹는다.
- (본 repo 메모) `Spring Boot.md`에 사용 버전이 기록되어 있다: [[CS공부/프레임워크/Spring Boot/Spring Boot]]

### 왜 필요한가
- 복잡한 설정(웹 서버, JSON 직렬화, DB 연결, 보안 등)을 매 프로젝트마다 수작업으로 구성하면 실수가 늘고 생산성이 떨어진다.
- Boot는 “기본값을 제공”해 초기 생산성을 높이고, 필요할 때 override/커스터마이즈할 수 있게 만든다.
- 운영 관점에서 관측/헬스체크/메트릭을 쉽게 붙일 수 있어 “만들고 끝”이 아니라 “운영 가능한 서버”를 만들기 좋아진다.

### 동작 원리
- Boot 동작의 큰 흐름:
  - 애플리케이션 시작 시 ApplicationContext 생성
  - 클래스패스/프로퍼티 기반으로 자동설정 후보를 평가
  - 조건(예: 특정 클래스 존재, 특정 프로퍼티 값)에 맞으면 Bean을 등록
  - 내장 서버(Tomcat/Jetty/Netty 등)를 띄우고, DispatcherServlet/MVC를 붙임(선택한 웹 스택에 따라 다름)
- “조건부 Bean” 사고 모델:
  - 자동설정은 대개 “Bean이 없으면 만들기” 패턴이다.
  - 그래서 커스터마이즈는 “내 Bean을 먼저 제공(override)”하는 방식이 흔하다.

### 장단점 / 한계
- 장점:
  - 빠른 시작, 표준화된 구성, 운영 기능 결합 용이
- 한계:
  - 자동설정이 무엇을 했는지 모르면 장애 분석이 느려진다(“기본값”이 숨어있기 때문).
  - 의존성/조건 충돌로 예상과 다른 Bean이 뜰 수 있다(중복 설정/예상치 못한 auto-config).

### 대안 / 선택 기준
- Spring Boot vs 순수 Spring:
  - Boot가 “편의/운영”을 제공하는 만큼, 단순/특수한 환경에서는 순수 Spring 또는 다른 프레임워크가 선택될 수 있다.
- MVC vs WebFlux:
  - I/O 대기 중심에서 높은 동시성을 원하면 WebFlux(리액티브)가 후보지만, 팀 숙련도/디버깅 난이도를 함께 고려한다.
  - “기술이 좋아서”가 아니라 “워크로드/운영 요구”가 선택 기준이어야 한다.

### 실무/디버깅 포인트
- 예상과 다른 Bean이 뜨면:
  - 어떤 자동설정이 적용됐는지(조건이 왜 true인지)를 추적해야 한다.
  - 프로퍼티 오타/프로파일 적용 여부가 자주 원인이다.
- “로컬에선 되는데 서버에선 안 됨”은 환경변수/프로파일/의존성 차이를 먼저 의심한다.

### 프로젝트 연결
- Spring Boot를 “웹 API + 실시간 통신(WebSocket) + 외부 브로커(MQTT) + Redis/JPA” 같은 조합으로 쓰면, 자동설정/빈 구성 이해가 장애 대응 속도에 직결된다.
  - 관련: [[프로젝트/Autowing-Car]]

## Bean / DI / 싱글톤 동시성

### 개념
- Bean은 Spring 컨테이너가 생성/관리하는 객체다.
- DI(Dependency Injection)는 객체가 의존 대상을 “직접 생성”하는 대신 “외부에서 주입”받는 설계 방식이다.
- Spring의 기본 스코프는 singleton이지만, 이는 “JVM 싱글톤 패턴”이 아니라 “컨테이너 1개당 인스턴스 1개”에 가깝다.
- 싱글톤 빈은 여러 스레드가 동시에 호출할 수 있으므로 “상태를 들고 있으면” 경쟁 조건이 발생할 수 있다.

### 왜 필요한가
- DI는 결합도를 낮춰 테스트/확장성을 높인다(인터페이스 기반, 대역 주입).
- 빈 생명주기/스코프를 이해해야 메모리 누수/초기화 순서/프록시 동작을 설명할 수 있다.
- 싱글톤 동시성 이슈는 실무에서 자주 나오는 장애 유형이다(공유 mutable state, ThreadLocal 오남용 등).

### 동작 원리
- 빈 등록:
  - 컴포넌트 스캔(`@Component`, `@Service` 등) 또는 `@Bean` 메서드로 등록한다.
  - 동일 타입/이름 Bean이 여러 개면 주입 시점에 모호성 문제가 생긴다(`@Qualifier`, `@Primary` 등으로 해결).
- 프록시:
  - AOP/트랜잭션/지연 로딩 등은 프록시로 동작하는 경우가 많다.
  - 프록시는 “호출을 가로채서 부가 기능을 붙인다”는 구조라, self-invocation(자기 자신 메서드 호출)에서 기대대로 동작하지 않을 수 있다.
- 스코프:
  - request/session scope는 웹 요청/세션 단위로 인스턴스가 달라진다.
  - scope proxy를 통해 싱글톤 빈에서도 request scope 의존을 주입할 수 있다(내부 구현은 프록시).
- 싱글톤 동시성:
  - “필드에 상태를 저장”하는 빈은 기본적으로 위험하다.
  - 상태는 요청 스코프/메서드 지역 변수/불변 객체/DB로 보내는 방식으로 처리하는 게 안전하다.

### 장단점 / 한계
- DI는 구조를 깔끔하게 만들지만, 지나치게 복잡한 그래프는 디버깅을 어렵게 한다.
- 프록시는 강력하지만 “실제 클래스가 아닌 프록시 객체가 주입될 수 있음”을 이해해야 한다.
- singleton 기본값은 운영 효율(메모리/초기화)에는 유리하지만, 상태 공유는 항상 주의해야 한다.

### 대안 / 선택 기준
- “상태가 필요한가?”를 먼저 분리한다.
  - 상태가 필요하면: 상태를 외부로(예: DB/Redis) 또는 요청 범위로 제한한다.
  - 상태가 필요 없으면: stateless 서비스를 유지한다.
- 복잡한 빈 그래프는 모듈/패키지 경계(도메인/인프라 분리)로 단순화한다.

### 실무/디버깅 포인트
- 주입 실패/순환 참조:
  - 순환 참조는 설계 냄새인 경우가 많아, 책임을 쪼개거나 방향을 바꾸는 게 보통 더 안전하다.
- 동시성 버그:
  - “가끔 틀린 값이 나온다”는 공유 필드/캐시/정적 상태를 우선 의심한다.
  - ThreadLocal은 트랜잭션/보안 컨텍스트 등에서 쓰이지만, 누수(정리 미흡) 시 심각한 버그를 만든다.

### 프로젝트 연결
- JWT 인증/실시간 이벤트 처리 같은 흐름은 여러 요청/이벤트가 동시에 들어오므로, singleton 빈이 stateless인지가 중요하다.
  - 관련: [[프로젝트/Autowing-Car/03-인증-세션-토큰]]

## Web / REST

### 개념
- REST는 “리소스 중심”으로 API를 설계하는 아키텍처 스타일이다.
- Spring Boot에서 REST API는 보통 Controller(HTTP) → Service(도메인/유스케이스) → Repository(영속성)로 나누어 구현한다.
- Web 설계에서는 “HTTP 메서드/상태 코드/응답 포맷/에러 포맷/멱등성”이 핵심이다.

### 왜 필요한가
- API는 “계약”이라 한번 나가면 바꾸기 어렵다.
- 상태 코드/에러 포맷/멱등성 설계는 클라이언트 개발, 장애 대응, 재시도 정책에 직접 영향을 준다.
- CORS/인증과 결합되면 “서버는 정상이지만 브라우저에서만 실패” 같은 문제가 생길 수 있다.

### 동작 원리
- Controller 책임 경계:
  - HTTP 요청 파싱/검증(입력 DTO), 응답 직렬화, 상태 코드 결정, 인증/인가 결과 반영
  - 비즈니스 규칙 자체는 Service로 내리는 편이 유지보수에 유리하다.
- 상태 코드/응답 제어:
  - `ResponseEntity`, `@ResponseStatus`, 예외 매핑(`@ControllerAdvice`) 등을 사용해 상태 코드와 바디를 제어한다.
- 예외 매핑:
  - 도메인 예외 → HTTP 에러로 변환하는 경계를 한 곳(`@RestControllerAdvice`)에 모으면 API 일관성이 좋아진다.
- CORS:
  - CORS는 “브라우저가 강제하는 정책”이며, 서버는 헤더로 허용 여부를 표현한다.
  - Security가 있으면 “MVC CORS 설정만으로는” 실패할 수 있어 필터 체인에서 CORS가 처리되는지 확인해야 한다.

### 장단점 / 한계
- REST 스타일은 범용적이지만, 모든 API가 리소스 CRUD로 표현되지는 않는다(명령형 API, 비동기 작업 등).
- 응답 공통 포맷은 일관성을 주지만, 지나치면 HTTP 표준(상태코드/헤더)을 희석시킬 수 있다.

### 대안 / 선택 기준
- 설계 기준(권장):
  - 리소스/행위 구분, 멱등성 고려, 에러 모델(코드/메시지/필드 에러) 확정
  - 페이징은 offset vs cursor를 데이터 크기/정렬 안정성 기준으로 선택
- 브라우저 클라이언트가 있으면 CORS/쿠키/토큰 저장 위치까지 같이 설계한다.

### 실무/디버깅 포인트
- “CORS 에러”는 네트워크 실패가 아니라 브라우저 차단일 수 있다.
  - 서버 로그는 정상인데 브라우저 콘솔에서 막히는 패턴을 구분해야 한다.
- 멱등성:
  - 재시도 가능한 API인지(특히 결제/주문) 여부가 운영 안정성에 중요하다.

### 프로젝트 연결
- 실제로는 WebSocket/외부 연동 등과 결합되어 REST 설계가 “운영”으로 연결된다.
  - 관련: [[프로젝트/Autowing-Car/00-상태-요약]]

## Spring MVC / 요청 처리 / 스레드

### 개념
- Spring MVC는 Servlet 기반 웹 프레임워크이며, 기본 모델은 “요청당 스레드(thread-per-request)”다.
- 요청 처리 파이프라인은 크게 `Filter → DispatcherServlet → HandlerMapping/Adapter → Controller → MessageConverter`로 이해하면 좋다.
- 인터셉터(HandlerInterceptor)는 컨트롤러 전후로 동작하고, 필터(Filter)는 서블릿 앞단에서 동작한다.

### 왜 필요한가
- 장애 분석에서 “어느 계층에서 막혔는지”를 설명하려면 요청 흐름을 알아야 한다.
- 스레드 모델을 모르면 타임아웃/블로킹/커넥션 풀 고갈 같은 운영 장애를 설명하기 어렵다.
- 실시간 통신(WebSocket/SSE/Long Polling)은 요청-응답 모델과 다른 제약이 있다.

### 동작 원리
- DispatcherServlet:
  - 프론트 컨트롤러로서 요청을 받아 적절한 핸들러(컨트롤러)로 라우팅한다.
- HandlerMapping/HandlerAdapter:
  - 매핑 규칙(어노테이션 기반 등)에 따라 어떤 컨트롤러/메서드를 호출할지 결정하고 실행한다.
- 파라미터 바인딩:
  - `@RequestParam`, `@PathVariable`, `@RequestBody` 등은 ArgumentResolver/MessageConverter 조합으로 처리된다.
  - JSON 직렬화/역직렬화는 보통 Jackson이 담당하며, 실패 시 400 계열 오류로 이어질 수 있다.
- 스레드와 블로킹:
  - MVC에서는 컨트롤러가 DB/외부 API에서 오래 블로킹되면 해당 요청 스레드가 묶인다.
  - 스레드 풀이 고갈되면 “새 요청이 처리되지 못하는 현상”이 나타날 수 있다.
- 비동기 처리(개념):
  - `@Async`, `Callable/DeferredResult`, SSE 등은 스레드 사용 모델이 달라질 수 있어, 스레드 풀/타임아웃을 분리해 설계해야 한다(세부는 구현 선택에 따라 다름).

### 장단점 / 한계
- MVC(thread-per-request)는 단순하고 디버깅이 쉽지만, 블로킹 I/O가 길면 동시성 확장이 제한된다.
- WebFlux(리액티브)는 동시성 확장에 유리할 수 있지만, 코드/운영 난이도가 올라간다.

### 대안 / 선택 기준
- 선택 기준(권장):
  - “블로킹 I/O 비중”과 “동시 연결 수/지연 목표”
  - 팀의 리액티브 경험/디버깅 역량
  - 관측/장애 대응 도구 체계(APM, tracing)

### 실무/디버깅 포인트
- 느린 요청 분석:
  - 스레드 덤프/trace로 DB/외부 호출/락 대기를 분리한다.
  - upstream(Nginx) 타임아웃과 애플리케이션 타임아웃을 같이 맞춘다(10번/05번 개념정리와 연결).
- 필터/인터셉터/컨트롤러/예외 처리 순서를 헷갈리지 않게 정리해 둔다.

### 프로젝트 연결
- 실시간 통신(WebSocket/STOMP)은 MVC 요청-응답과 다른 파이프라인(핸드셰이크/세션)을 갖는다.

## JPA / DB / 메모리

### 개념
- JPA는 객체-관계 매핑(ORM) 표준이며, Spring Data JPA는 이를 편하게 쓰게 해준다.
- 핵심 키워드:
  - 영속성 컨텍스트(1차 캐시), 변경 감지(Dirty Checking), 지연 로딩(Lazy), N+1, 트랜잭션 경계
- DB/메모리/스레드는 서로 영향을 주며, 특히 커넥션 풀 고갈은 애플리케이션 장애로 바로 이어진다(05번 개념정리와 연결).

### 왜 필요한가
- JPA는 생산성을 높이지만, 동작을 모르면 성능/정합성 이슈를 만들기 쉽다.
- 트랜잭션 경계가 애매하면 LazyInitialization 문제나 예상치 못한 쿼리 폭발이 발생한다.
- DB 커넥션/트랜잭션은 스레드와 결합되어 운영 병목을 만든다.

### 동작 원리
- 영속성 컨텍스트:
  - 엔티티를 “관리 상태”로 두고, 동일 트랜잭션 내에서 동일성(identity)을 보장한다.
  - flush 시점에 변경 내용을 SQL로 반영한다(커밋 직전 또는 flush 호출 시).
- 변경 감지:
  - 엔티티 필드를 바꾸면, 트랜잭션 종료 시점에 변경을 감지해 update를 날릴 수 있다.
  - “명시적 save가 없어도 update가 발생”할 수 있어, 의도와 다르게 쿼리가 나가는지 확인해야 한다.
- Lazy/Eager:
  - Lazy는 접근 시점에 쿼리가 나갈 수 있어, 서비스/컨트롤러 경계에서 트랜잭션이 끝나면 예외가 날 수 있다.
  - Eager는 초기 로딩이 무거워지고 N+1을 유발할 수 있어, 보통 기본은 Lazy + fetch 전략을 필요할 때 최적화한다.
- N+1:
  - 연관 엔티티를 루프에서 접근하면서 쿼리가 반복되는 전형적인 성능 이슈다.
  - fetch join, entity graph, batch size 등으로 제어할 수 있다(선택 기준은 쿼리 형태/페이징/카디널리티에 따라 다름).

### 장단점 / 한계
- ORM은 편하지만, “SQL을 숨겨서” 성능을 놓치기 쉽다.
- 복잡한 조회/리포팅은 ORM보다 SQL 중심(쿼리 최적화)이 더 적합한 경우가 있다.

### 대안 / 선택 기준
- 단순 CRUD는 JPA가 생산성이 좋다.
- 복잡한 쿼리/대량 배치/튜닝이 중요하면:
  - QueryDSL/네이티브 SQL/조회 전용 모델 등을 함께 고려한다.
- 트랜잭션/커넥션 풀 관점은 05번 문서(운영/풀)와 함께 본다.

### 실무/디버깅 포인트
- SQL 로그/슬로우 쿼리/인덱스 점검은 기본이다.
- “응답이 느리다”가 DB 때문인지, 앱 스레드/커넥션 풀 때문인지 분리해 본다.
- OOM/GC와 JPA는 연결될 수 있다(대량 엔티티 로딩/1차 캐시 누적 등).

### 프로젝트 연결
- Spring Boot + JPA + Redis + 실시간 브로드캐스트 흐름에서는 “DB에 남길 것 vs 메모리에 둘 것”의 기준이 중요하다.
  - 관련: [[프로젝트/Autowing-Car]]

## Security / Spring Boot 버전 변화

### 개념
- Spring Security는 필터 체인 기반으로 인증(Authentication)과 인가(Authorization)를 처리한다.
- 웹 보안에서 자주 같이 나오는 축:
  - 세션 vs JWT(무상태) 설계
  - CSRF/XSS 위협 모델
  - CORS와 인증의 결합(쿠키/헤더)
- Spring Boot 버전 변화는 “프레임워크 내부 동작/기본값/호환성”에 영향을 준다.

### 왜 필요한가
- 보안은 “구성 실수”로 바로 취약점/장애가 된다.
- Security는 MVC보다 앞단에서 요청을 막을 수 있어, CORS/인증 실패를 분석할 때 필터 체인 이해가 필수다.
- 버전 업은 운영 리스크이므로, 어떤 변화가 있는지(특히 breaking changes)를 설명할 수 있어야 한다.

### 동작 원리
- 필터 체인:
  - 요청이 들어오면 여러 필터를 거쳐 `SecurityContext`가 채워지고, 인가 규칙이 평가된다.
  - JWT 기반이면 보통 “토큰 파싱/검증 필터”가 앞단에 위치한다.
- CSRF:
  - 브라우저가 쿠키를 자동 전송하는 특성 때문에 발생하는 위협 모델과 연결된다.
  - 무상태 API(JWT 헤더)라도 “쿠키 기반 인증”을 쓰면 CSRF 대응이 필요할 수 있다.
- CORS:
  - MVC에만 설정하면 Security에서 막힐 수 있어, “Security에서 CORS를 허용/적용”하는 위치가 중요하다.
- 버전 변화(개념 수준):
  - Boot 2 → 3 계열에서 네임스페이스 변화(예: `javax` → `jakarta`) 같은 큰 변화가 있었고,
  - 기본 지원 Java 버전 상향 같은 제약이 걸릴 수 있다(정확한 범위는 현재 프로젝트 버전에 맞춰 확인).

### 장단점 / 한계
- Security는 유연하지만 설정 복잡도가 높다.
- “대충 동작하게” 만들면 나중에 정책 변경/취약점 대응이 어렵다.

### 대안 / 선택 기준
- 인증 방식 선택:
  - 브라우저 중심(쿠키/세션) vs API/모바일 중심(토큰/헤더)
  - 로그아웃/토큰 회수 요구(Refresh Token 저장/블랙리스트 등) 여부
- 버전 업:
  - 프레임워크/라이브러리 호환성, 테스트 커버리지, 롤백 전략을 갖춘 뒤 진행한다.

### 실무/디버깅 포인트
- “401/403/리다이렉트”가 나오면:
  - 인증 실패인지(토큰/세션), 인가 실패인지(권한), CORS인지(브라우저 차단)를 분리한다.
  - 필터 체인 로그/디버그 모드로 어느 필터에서 끊겼는지 확인한다.

### 프로젝트 연결
- JWT + Refresh Token(서버 저장) 구조는 실무에서 자주 쓰는 패턴이고, 운영 정책(만료/회전)이 중요하다.
