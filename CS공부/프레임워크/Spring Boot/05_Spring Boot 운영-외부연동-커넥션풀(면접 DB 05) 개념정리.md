## 외부 API 호출 / 복원력

### 개념
- 외부 API 호출은 “내 서비스의 제어 범위 밖”을 호출하는 것이고, 그래서 실패/지연은 기본값이다.
- 복원력(resilience)은 “외부 의존이 흔들려도 내 서비스가 같이 무너지는 것을 막는 설계”를 의미한다.
- 다루는 축:
  - 타임아웃(연결/읽기/전체)
  - 재시도(retry)와 백오프(backoff)
  - 서킷 브레이커(circuit breaker)
  - 벌크헤드(bulkhead) / 격리(격리된 스레드풀/커넥션풀)
  - 폴백(fallback) / 캐시

### 왜 필요한가
- 타임아웃이 없으면 호출 스레드가 무한 대기에 빠져 스레드 풀 고갈 → 전체 장애로 번질 수 있다.
- 재시도를 무조건 걸면:
  - 실패를 더 오래 끌고 가거나,
  - 다운스트림을 더 세게 때려 장애를 확산시키거나,
  - “중복 요청(비멱등)”으로 데이터 문제를 만들 수 있다.
- 결국 외부 연동은 “성공/실패”가 아니라 “실패를 어떤 모양으로 통제할지”가 핵심이다.

### 동작 원리
- HTTP Client 선택(개념):
  - `RestTemplate`: 동기/블로킹 스타일(레거시 성격, 새 개발에서는 대체가 많음)
  - `RestClient`: 동기 스타일이지만 더 현대적인 API(프로젝트/버전에 따라 사용 가능 여부가 달라 `확인 필요`)
  - `WebClient`: 리액티브 기반(논블로킹 사용 가능), 하지만 내부에서 블로킹하면 의미가 줄어든다
- 타임아웃 설정 위치:
  - “클라이언트 라이브러리(HTTP client)” 수준에서 설정하는 게 기본이다.
  - 프레임워크 레벨에서 통일된 데드라인(요청 전체 타임아웃)을 두고, 하위 호출에도 일관되게 적용하는 게 운영에 유리하다.
- Retry:
  - “언제” 재시도할지(네트워크 오류/5xx/429 등), “몇 번” 할지, “간격(지수 백오프/지터)”을 설계해야 한다.
  - 멱등한 요청(GET/PUT 등)이라도, 서버 구현/부작용에 따라 진짜 멱등인지 확인해야 한다.
- Circuit Breaker:
  - 실패율/지연이 임계치를 넘으면 회로를 열어 빠르게 실패시키고, 다운스트림을 보호한다.
  - Half-open 상태에서 일부 트래픽만 흘려 “회복 여부”를 탐색한다.
- Bulkhead:
  - 외부 의존별로 스레드풀/커넥션풀을 분리해, 한 의존의 장애가 전체 요청을 잡아먹지 않게 한다.

### 장단점 / 한계
- 복원력 장치들은 시스템을 안정화하지만, 구성/관측이 없으면 “왜 느린지/왜 실패하는지”를 더 숨길 수 있다.
- 폴백은 편하지만, 잘못 설계하면 조용히 잘못된 데이터를 계속 제공할 수 있다.
- 서킷브레이커/리트라이는 “정상 시 성능”보다 “장애 시 폭발 반경”을 줄이는 데 목적이 있다.

### 대안 / 선택 기준
- 외부 의존 설계 체크리스트(권장):
  - 멱등성: 재시도 가능한 요청인가?
  - 타임아웃: connect/read/전체 데드라인이 있는가?
  - 격리: 의존별 격리(스레드/커넥션)가 있는가?
  - 관측: 실패율/지연/에러 코드를 메트릭으로 보는가?
  - 폴백: 서비스 품질을 유지할 최소 행동은 무엇인가?

### 실무/디버깅 포인트
- “외부 API가 느리다”는 상황에서 분리해야 할 것:
  - DNS 느림 vs TCP 연결 지연 vs TLS 지연 vs 서버 처리 지연(TTFB)
  - 내 서버 스레드 고갈 vs 외부 의존 자체 지연
- 로그만으로 부족하면:
  - 호출 메트릭(성공/실패/지연), 분산 추적(trace), 타임아웃/리트라이 횟수 지표가 필요하다.

### 프로젝트 연결

## Connection Pool / 운영

### 개념
- 풀(Pool)은 “비싼 자원”을 매번 만들지 않고 재사용하기 위한 구조다.
  - DB Connection pool(HikariCP 등)
  - HTTP Connection pool(keep-alive 기반)
  - Thread pool(Executor)
- 운영 장애에서 가장 자주 보는 패턴 중 하나가 “풀 고갈(pool exhaustion)”이다.

### 왜 필요한가
- DB 커넥션은 생성 비용이 크고(DB 핸드셰이크/인증 등), DB가 허용하는 동시 연결 수에도 제한이 있다.
- HTTP도 매 요청마다 연결을 새로 만들면 TCP/TLS 비용이 커서 지연/부하가 커진다(keep-alive/pooling이 중요).
- 풀을 너무 크게 잡으면:
  - DB/외부 시스템이 버티지 못하거나,
  - JVM 스레드/메모리/컨텍스트 스위칭 비용이 커져 전체가 느려질 수 있다.

### 동작 원리
- DB Connection pool(HikariCP):
  - 커넥션을 미리 만들어 두거나 필요 시 생성해 재사용한다.
  - “최대 풀 크기”는 DB의 한계와 애플리케이션의 동시 처리 모델을 함께 고려해 잡아야 한다.
  - 풀에 대기 큐가 생기면 요청 스레드가 커넥션을 기다리며 묶인다.
- HTTP Connection pool:
  - HTTP/1.1 keep-alive로 TCP 연결을 재사용한다.
  - HTTP/2는 연결 하나에서 다중 스트림을 처리할 수 있어(멀티플렉싱) 모델이 달라진다.
  - 서버/프록시(Nginx) 타임아웃과 클라이언트 풀 정책이 맞지 않으면 예기치 않은 끊김/재시도가 발생할 수 있다.
- 풀 고갈의 전형적인 연쇄:
  - 외부(DB/API) 지연 증가 → 커넥션 점유 시간 증가 → 풀 고갈 → 요청 대기 증가 → 스레드 고갈 → 장애 전파

### 장단점 / 한계
- 풀은 성능을 올리지만, 잘못 설정하면 “동시성 폭탄”이 된다.
- 풀을 늘리는 것은 근본 해결이 아닐 수 있다.
  - 느린 쿼리/락 경합/외부 지연이 원인이면, 풀을 늘리면 오히려 더 많은 동시 요청이 DB를 때려 더 느려질 수 있다.

### 대안 / 선택 기준
- 풀 사이즈를 정할 때 같이 보는 것(권장):
  - 평균/피크 QPS, 요청당 DB 쿼리 수, 평균 쿼리 시간
  - DB의 최대 커넥션과 기타 서비스가 쓰는 커넥션
  - 스레드 풀 크기(톰캣 등)와의 관계(스레드가 많아도 DB 커넥션이 적으면 대기가 발생)
- 근본 개선:
  - 쿼리 최적화, 캐시, 배치/비동기 처리, 격리(의존별 풀 분리)

### 실무/디버깅 포인트
- 풀 고갈 증상:
  - DB: 커넥션 획득 대기/타임아웃, 요청 지연 증가, 스레드가 WAITING/BLOCKED로 증가
  - HTTP: keep-alive 재사용 실패, 연결 재수립 증가, 타임아웃 증가
- Actuator:
  - 운영에서 헬스/메트릭을 붙여 “풀 상태/스레드 상태”를 관측하는 게 중요하다(정확한 노출 항목은 구성에 따라 다름).
