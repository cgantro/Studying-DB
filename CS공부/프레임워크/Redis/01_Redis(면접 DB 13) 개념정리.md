## Redis 기본

### 개념
- Redis는 인메모리 기반 Key-Value 저장소이며, 단순 KV뿐 아니라 여러 자료구조와 TTL/원자 연산/메시징 보조 기능을 제공한다.
- 실무에서는 “DB를 대체한다”기보다, **DB 앞단(캐시)** 또는 **임시 상태 저장(세션/토큰/레이트리밋/락)** 으로 쓰는 경우가 많다.
- 핵심 질문은 항상 같다: “이 데이터는 유실되어도 되는가? (유실 시 복구 경로가 있는가?)”

### 왜 필요한가
- DB는 정합성에 강하지만, 모든 트래픽을 DB가 직접 처리하면 지연/부하가 커진다.
- Redis는 낮은 지연으로 “자주 조회되는 데이터”, “짧은 수명 데이터”, “원자 카운트/플래그”를 처리해 시스템 병목을 줄이는 데 도움이 된다.
- TTL/eviction을 이용해 “자동 정리되는 데이터”를 운영하기 좋아, 누수/정리 비용을 낮출 수 있다.

### 동작 원리
- 인메모리 접근 + 이벤트 루프 기반 명령 처리로 낮은 지연을 얻는다.
- 다만 실제 지연은 네트워크 RTT, 직렬화 비용, 값 크기, 느린 명령, 백그라운드 작업(RDB/AOF rewrite 등) 영향까지 포함된다.
- “싱글 스레드”라는 말은 주로 명령 실행이 순차적이라는 의미로 쓰이며, 긴 명령은 전체를 막는 병목이 된다.

### 장단점 / 한계
- 장점: 저지연, TTL, 자료구조/원자 연산, 단순 운영(단일 인스턴스 기준)
- 한계: 메모리 비용, 핫키/스탬피드 같은 운영 문제, 클러스터/복제 도입 시 복잡도 증가

### 대안 / 선택 기준
- 단순 캐시는 애플리케이션 로컬 캐시(Caffeine 등)로도 해결될 수 있다(프로세스 단위).
- 분산 캐시/상태 공유가 필요하면 Redis 같은 중앙 저장소가 유리하다.
- 메시징이 목적이면 Kafka/RabbitMQ 같은 브로커가 더 적합할 수 있다(내구성/재처리/운영 모델 차이).

### 실무/디버깅 포인트
- Redis를 도입하기 전에 “캐시 적중률, DB 부하, 지연 목표”를 명확히 정의해야 한다.
- 커넥션은 매 요청 생성이 아니라 재사용이 기본이고, 타임아웃/풀 설정을 운영 요구에 맞게 잡는다.
- key/value 설계는 운영 가능성(스캔/만료/삭제)까지 포함해 규칙화한다.

### 프로젝트 연결
- Refresh Token을 Redis에 저장하고 TTL로 만료를 제어한 사례가 있다.
  - [[프로젝트/Autowing-Car/03-인증-세션-토큰]]

## 자료구조 / 명령

### 개념
- Redis는 자료구조를 제공해 “애플리케이션에서 직접 구현해야 하는 동시성/집계/정렬”을 서버 측에서 지원한다.
- 자주 쓰는 구조:
  - String: 단일 값/카운터/토큰
  - Hash: 객체형(필드 단위) 데이터
  - List: 순서/큐
  - Set: 중복 없는 집합
  - ZSet: 점수 기반 정렬(랭킹)
- Bitmap/HyperLogLog/BloomFilter/Geo/Streams 등은 “특정 문제에 특화된 도구”로 이해하는 게 안전하다.

### 왜 필요한가
- 자료구조가 맞으면: 코드가 단순해지고, 네트워크 왕복이 줄고, 동시성 제어도 쉬워진다.
- 자료구조가 틀리면: 값이 커지고 연산이 무거워져, Redis 단일 스레드에서 병목이 쉽게 발생한다.

### 동작 원리
- 원자 연산:
  - 단일 명령은 순차 실행되므로 `INCR` 같은 연산을 원자적으로 처리할 수 있다.
  - 복합 연산(읽고 비교하고 쓰기)은 WATCH/Lua로 원자성을 만들어야 한다.
- `KEYS` vs `SCAN`:
  - `KEYS`는 전체 키 공간을 훑어 운영 중 블로킹을 만들 수 있어 위험하다.
  - `SCAN`은 점진적 조회로 스파이크를 줄이지만, 전체를 보면 비용은 여전히 존재한다.

### 장단점 / 한계
- Redis 자료구조는 강력하지만, “데이터 모델링”을 잘못하면 메모리/성능 비용이 급증한다.
- 큰 컬렉션/큰 value는 네트워크와 CPU(직렬화) 비용까지 포함해 병목이 된다.

### 대안 / 선택 기준
- Hash vs JSON 문자열:
  - 필드 단위 갱신/조회가 잦으면 Hash가 유리할 수 있다.
  - 이식/디버깅/스키마 유연성이 더 중요하면 JSON이 유리할 수 있다.
- ZSet은 랭킹/우선순위에 강하지만, 단순 집계라면 카운터/DB 집계가 더 적합할 수 있다.

### 실무/디버깅 포인트
- 키 패턴과 자료구조를 문서로 고정하고, “큰 키(large key)”와 “핫키(hot key)”를 상시 모니터링한다.
- 운영에서 `KEYS` 사용은 원칙적으로 금지하는 팀 규칙을 두는 경우가 많다.

### 프로젝트 연결
- JWT/토큰/레이트리밋 같은 용도는 보통 String/Hash와 TTL이 기본 조합이다.
  - [[프로젝트/Autowing-Car/03-인증-세션-토큰]]

## TTL / 만료 / Eviction / 캐시 패턴

### 개념
- TTL은 키의 수명을 제한해 자동 만료시키는 기능이다.
- eviction은 `maxmemory` 도달 시 어떤 키를 제거할지 결정하는 정책이다.
- 캐시 패턴은 “정합성/성능/운영 난이도”를 바꾸는 선택이다:
  - cache-aside, write-through, write-behind 등

### 왜 필요한가
- TTL이 없으면 캐시가 누적되어 메모리 고갈/eviction 폭주로 이어질 수 있다.
- 캐시는 “정상일 때 빠른 것”보다 “장애/만료 시 시스템이 무너지지 않는 것”이 더 중요하다.

### 동작 원리
- cache-aside:
  - 읽기: 캐시 조회 → 미스면 DB 조회 후 캐시에 저장
  - 쓰기: DB 갱신 후 캐시 무효화(또는 갱신)
- stampede/avalanche/penetration:
  - 스탬피드: 인기 키 만료로 동시 DB 폭주
  - 어벌랜치: 다수 키가 비슷한 시점에 만료
  - 관통: 없는 키 요청이 반복되어 DB를 계속 침
- 만료 처리:
  - 만료 키는 접근 시 제거(lazy) + 백그라운드 정리(active)가 섞여 동작할 수 있다(구체는 설정/버전에 따라 다름).

### 장단점 / 한계
- TTL/eviction은 운영을 쉽게 만들지만, 잘못 설정하면 핵심 데이터가 의도치 않게 사라질 수 있다.
- 캐시 패턴은 단순할수록 좋지만, 요구사항(일관성/지연)에 따라 복잡도를 감수해야 할 때가 있다.

### 대안 / 선택 기준
- TTL/eviction 선택은 “유실 허용”과 직접 연결된다.
  - 유실 가능 캐시: allkeys-lru 같은 정책이 자연스러울 수 있다.
  - 유실 금지 상태: volatile-* 또는 별도 인스턴스 분리 등 전략이 필요하다.
- 어벌랜치는 TTL 지터, 스탬피드는 단일 플라이트(1명만 갱신) 같은 전략으로 완화한다.

### 실무/디버깅 포인트
- 캐시 장애 시 DB 보호 장치(레이트리밋/서킷브레이커/락/폴백)를 준비한다.
- 핫키는 키 샤딩/로컬 캐시/서비스 분산으로 완화한다.

### 프로젝트 연결
- 토큰/세션/캐시는 TTL 정책이 곧 보안/운영 정책이 된다.
  - [[프로젝트/Autowing-Car/03-인증-세션-토큰]]

## Persistence (RDB / AOF)

### 개념
- RDB: 주기적으로 스냅샷을 파일로 저장
- AOF: 수행된 명령을 로그로 기록하고 재생해 복구
- 캐시라도 persistence를 켜면 재시작 후 “워밍업 비용”을 줄이거나 유실 범위를 줄일 수 있다.

### 왜 필요한가
- 장애/재시작 시 완전 초기화가 허용되지 않는 데이터(세션/토큰/작업 상태)가 있을 수 있다.
- persistence는 내구성을 올리지만, 쓰기/디스크/운영 비용을 늘린다.

### 동작 원리
- RDB는 특정 시점 상태를 저장하므로 복구가 단순하고 빠를 수 있다.
- AOF는 fsync 정책에 따라 “유실 범위”와 “쓰기 비용”이 크게 달라진다.
- AOF rewrite는 로그가 커지는 문제를 해결하기 위한 압축 단계다.

### 장단점 / 한계
- RDB: 성능 부담이 비교적 낮을 수 있지만 스냅샷 사이 유실 가능
- AOF: 유실을 줄일 수 있지만 쓰기 비용/운영 복잡도 증가

### 대안 / 선택 기준
- 용도별로 분리하는 전략이 흔하다:
  - 유실 가능 캐시 인스턴스(가벼운 설정)
  - 유실 민감 상태 인스턴스(persistence/복제 강화)
- 최종 선택은 RPO/RTO 요구와 비용(성능/운영)을 함께 비교한다.

### 실무/디버깅 포인트
- persistence를 켰다면 “복구 절차를 실제로 리허설”해야 한다.
- rewrite/스냅샷이 서비스 지연에 영향을 주는지 모니터링하고 스케줄을 조정한다.

### 프로젝트 연결
- Refresh Token 같은 상태는 유실 시 영향이 크므로 persistence/복제/복구 정책을 함께 고려해야 한다.
  - [[프로젝트/Autowing-Car/03-인증-세션-토큰]]

## Replication / Sentinel / Cluster

### 개념
- Replication: 주 노드의 변경을 부 노드로 복제해 읽기 확장/가용성을 확보
- Sentinel: 장애 감지 및 마스터 전환(failover) 관리
- Cluster: 샤딩으로 수평 확장(메모리/처리량) + 일부 고가용성 구성

### 왜 필요한가
- 단일 Redis는 단일 장애점(SPOF)이 될 수 있다.
- 트래픽/데이터가 커지면 단일 인스턴스 용량 한계가 오므로 샤딩이 필요해진다.

### 동작 원리
- 복제는 지연(lag)이 있을 수 있어, 최신성/일관성 요구가 강한 기능은 주 노드 읽기가 필요할 수 있다.
- Cluster는 키를 슬롯 단위로 분배하고, 노드 간에 키가 분산되므로 multi-key 연산이 제한된다.
- 해시 태그(`{...}`)는 특정 키들을 같은 슬롯으로 보내기 위한 키 설계 기법이다.

### 장단점 / 한계
- 복제/클러스터는 가용성과 확장을 제공하지만 운영 복잡도를 크게 올린다.
- failover 중에는 클라이언트 오류/재시도가 발생할 수 있어 애플리케이션 설계(재연결/타임아웃)가 필요하다.

### 대안 / 선택 기준
- 처음부터 클러스터가 정답은 아니다.
  - 데이터/부하가 작으면 단일 + 백업/복구로 시작하고,
  - 성장 시 복제/센티널/클러스터로 확장하는 전략이 안전하다.
- multi-key 원자성이 중요한 서비스는 키 설계/샤딩 전제부터 다시 점검한다.

### 실무/디버깅 포인트
- 복제 지연/읽기 라우팅 정책을 문서로 고정한다.
- 리샤딩/노드 추가는 트래픽/지연에 영향을 줄 수 있어 점진적으로 수행한다.

### 프로젝트 연결
- Redis를 “상태 저장소”로 쓰면 장애 시 영향이 커져, 복제/복구 정책이 더 중요해진다.
  - [[프로젝트/Autowing-Car/03-인증-세션-토큰]]

## Transaction / Lua / Pipeline

### 개념
- MULTI/EXEC: 명령 묶음 실행(단일 노드 기준)
- WATCH: 낙관적 락(키 변경 감지)
- Lua: 여러 명령을 서버 측에서 원자적으로 실행
- Pipeline: 네트워크 왕복을 줄여 throughput을 높이는 최적화(원자성 아님)

### 왜 필요한가
- Redis에서는 “한 키에 대한 단일 명령”은 원자적이지만, 여러 키/여러 단계 로직은 그렇지 않다.
- 원자성과 성능(왕복 감소)을 동시에 달성하려면 Lua/트랜잭션/파이프라인을 상황에 맞게 선택해야 한다.

### 동작 원리
- WATCH는 키가 바뀌면 EXEC를 실패시켜 충돌을 감지하고 재시도를 유도한다.
- Lua는 스크립트가 실행되는 동안 다른 명령이 끼어들지 않아 원자적 규칙을 구현하기 좋다.
- Pipeline은 여러 명령을 묶어 보내 지연을 줄이지만, 중간에 다른 클라이언트 명령이 끼어들 수 있다.

### 장단점 / 한계
- Lua는 강력하지만, 오래 걸리면 Redis 전체를 블로킹할 수 있다.
- 분산 락은 “단일 서버가 내려갔을 때, 네트워크가 분할됐을 때” 같은 실패 모드를 전제로 설계해야 한다.

### 대안 / 선택 기준
- 단순 원자성: 단일 명령으로 해결 가능한지 먼저 본다(`SET NX PX` 등).
- 복합 원자성: Lua가 가장 단순한 경우가 많다.
- 분산 락이 정말 필요한지부터 검토하고, DB 락/작업 큐 직렬화 같은 대안과 비교한다.

### 실무/디버깅 포인트
- “락이 풀리지 않는다/중복 처리된다”는 TTL/해제 로직/재시도 정책을 같이 점검해야 한다.
- 파이프라인/스크립트 도입 전후로 latency/CPU/slowlog를 통해 병목이 바뀌지 않았는지 확인한다.

### 프로젝트 연결
- 토큰 저장/회수 같은 로직은 “원자적 갱신 + TTL” 조합이 핵심이 될 수 있다.
  - [[프로젝트/Autowing-Car/03-인증-세션-토큰]]

## 운영 / 장애 / 모니터링

### 개념
- Redis 운영의 주요 리스크:
  - 메모리 고갈/eviction 폭탄
  - 핫키/큰 키로 인한 지연
  - 백그라운드 작업(RDB/AOF rewrite)로 인한 지연 스파이크
  - failover/리샤딩으로 인한 순간 오류

### 왜 필요한가
- Redis는 빠르지만, 병목이 생기면 전체 요청이 영향을 받기 쉽다(명령 처리 순차성).
- 캐시 장애는 DB 폭주로 이어져 “연쇄 장애”가 되기 쉬워 방어 설계가 필요하다.

### 동작 원리
- `maxmemory`와 eviction 정책이 메모리 압박 시 동작을 결정한다.
- 메모리 단편화는 “사용 중인 데이터 크기”와 “OS가 본 RSS”의 차이를 키워 운영 안정성을 떨어뜨릴 수 있다.
- 모니터링은 `INFO`, `SLOWLOG`, 지연/커넥션/메모리 지표를 기본으로 둔다.

### 장단점 / 한계
- Redis는 인메모리라 비용이 눈에 띄기 쉬워, 데이터 모델/TTL 정책이 조금만 어긋나도 비용이 빠르게 증가한다.

### 대안 / 선택 기준
- 캐시 인스턴스와 상태 인스턴스를 분리해 위험을 격리하는 전략이 흔하다.
- 로컬 캐시와 Redis를 조합하는 계층 캐시도 대안이 될 수 있다.

### 실무/디버깅 포인트
- 느려졌을 때는 “느린 명령/큰 키/핫키/네트워크/메모리 압박”을 순서대로 분리한다.
- 장애 시 DB 보호를 위한 레이트리밋/서킷브레이커/폴백을 애플리케이션에 준비한다.

### 프로젝트 연결
- 운영 이슈가 생겼을 때 Redis/DB/외부 의존을 어떻게 분리해 봤는지 근거를 남겨두면 좋다.
  - `확인 필요`

## 보안

### 개념
- Redis는 내부망에서 신뢰된 서비스끼리 쓰는 전제가 많아, 외부 노출은 위험이 크다.
- ACL/인증/네트워크 격리로 접근을 최소화하는 것이 기본이다.

### 왜 필요한가
- Redis가 외부에 노출되면 데이터 유출/변조뿐 아니라 운영 환경 악용으로 이어질 수 있다.
- 특히 캐시/세션/토큰을 저장하면 보안 사고의 영향이 커진다.

### 동작 원리
- 네트워크 격리(방화벽/서브넷) + 인증 + ACL(최소 권한) 조합으로 보호한다.
- 애플리케이션별 계정을 분리하고 필요한 명령만 허용하는 방식이 안전하다.

### 장단점 / 한계
- 보안을 강화할수록 운영/디버깅 편의가 떨어질 수 있어, 표준 운영 절차(접근 권한/감사)를 같이 둔다.

### 대안 / 선택 기준
- 외부 노출이 필요하면 프록시/게이트웨이/사설망을 통해 경계를 만들고 직접 노출은 피한다.

### 실무/디버깅 포인트
- “보안은 기능이 아니라 기본값”이라, 초기 인프라 구성 단계에서 네트워크/권한을 고정하는 게 비용이 가장 낮다.

### 프로젝트 연결
- 토큰/세션을 Redis에 두면 네트워크 격리와 시크릿 관리가 필수로 따라온다.
