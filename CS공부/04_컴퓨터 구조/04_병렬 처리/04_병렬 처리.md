# 병렬 처리

## 한 줄 정의
병렬 처리(parallel processing)는 **여러 작업을 동시에 진행해 전체 처리 시간을 줄이거나 [[용어 모음집/처리량|처리량]]을 높이는 방법**이다.

쉽게 말하면, 한 사람이 전부 하는 대신 **여러 사람이 나눠서 동시에 일하는 것**이다.

---

## 왜 배우는가
- [[용어 모음집/CPU|CPU]]가 빨라져도 할 일이 많아지면 한계가 온다.
- 영상 처리, 그래픽, 과학 계산, 서버 요청 처리 등은 병렬화가 중요하다.
- 병렬 처리를 이해하면 성능 최적화의 핵심을 볼 수 있다.
- 멀티코어 CPU 시대에는 필수 개념이다.
- 면접에서 [[용어 모음집/병렬성|병렬성]]과 [[용어 모음집/동시성|동시성]]을 구분할 수 있어야 한다.

---

## 초보자용 한 줄 기억법
- **병렬 처리 = 진짜로 여러 일을 동시에 나눠 하는 것**

---

## 병렬 처리를 한 문장으로 다시 말하면
병렬 처리는 **작업을 여러 실행 단위로 나누고, 여러 CPU 코어 또는 실행 자원을 활용해 동시에 처리하는 방식**이다.

---

## 전체 흐름

```text
큰 작업
   ↓
작업 분할
   ↓
여러 코어/스레드에 배분
   ↓
동시에 실행
   ↓
결과 합치기
```

---

## 병렬성 vs 동시성
이 둘은 자주 헷갈린다.

### 동시성(concurrency)
- 여러 작업을 번갈아 진행하는 느낌
- 동시에 "보이게" 만들 수 있음
- 한 CPU에서도 가능

### 병렬성(parallelism)
- 실제로 여러 작업을 같은 시간에 처리
- 여러 코어/하드웨어 자원을 활용

### 아주 쉽게
- 동시성 = 여러 일을 번갈아 잘 처리
- 병렬성 = 여러 일을 진짜 동시에 처리

---

## 병렬성 vs 동시성 비교표

| 항목 | 동시성 | 병렬성 |
|---|---|---|
| 핵심 | 번갈아 처리 | 실제 동시 처리 |
| 필요 자원 | 한 코어도 가능 | 여러 코어/실행 자원 |
| 목적 | 응답성, 구조 분리 | 속도, 처리량 향상 |
| 예시 | 한 CPU에서 여러 요청 처리 | 멀티코어로 계산 분담 |

---

## 병렬 처리가 필요한 이유
- 작업 하나가 너무 크면 오래 걸린다.
- 여러 작업을 나누면 더 빨리 끝낼 수 있다.
- 대규모 데이터 처리에서는 병렬성이 거의 필수다.
- GPU처럼 병렬에 특화된 하드웨어도 있다.

---

## 병렬 처리가 잘 맞는 작업
- 이미지/영상 처리
- 과학 시뮬레이션
- 데이터 분석
- 대량 로그 처리
- 서버 요청 분산 처리
- 벡터/행렬 계산

---

## 병렬 처리가 덜 맞는 작업
- 순서 의존성이 강한 작업
- 작은 작업인데 분할 비용이 큰 경우
- 자원 공유가 너무 많은 작업
- [[용어 모음집/동기화|동기화]] 비용이 큰 작업

---

## 핵심 개념
- **작업 분할**: 큰 일을 작은 조각으로 나눔
- **[[용어 모음집/스레드|스레드]]/[[용어 모음집/프로세스|프로세스]]**: 실행 단위
- **코어(core)**: 실제 실행 자원
- **동기화**: 공유 자원 충돌 방지
- **병목(bottleneck)**: 전체 속도를 잡아먹는 구간
- **오버헤드(overhead)**: 나누고 합치는 데 드는 추가 비용

---

## 작업 분할
병렬 처리의 시작은 **작업을 잘 나누는 것**이다.

### 예시
- 이미지 한 장을 여러 영역으로 나눠 처리
- 데이터 목록을 여러 구간으로 분리
- 요청을 여러 워커가 나눠 처리

### 쉽게 이해하기
- 큰 빵을 여러 조각으로 나눠 각자 먹는 것

---

## 작업 합치기
분할만 하고 끝나면 안 된다. **결과를 다시 합쳐야 한다.**

### 예시
- 부분 합 계산 후 총합 만들기
- 각 타일의 이미지 처리 결과를 합치기
- 여러 워커의 결과를 모아 최종 응답 만들기

### 주의
- 합치는 과정이 병목이 될 수 있다
- 순서가 중요할 수 있다

---

## 멀티코어와 병렬 처리
멀티코어 CPU는 병렬 처리의 중요한 기반이다.

### 쉽게 이해하기
- 코어가 여러 개면 여러 일을 동시에 실제로 돌릴 수 있다
- 단, 코어 수보다 더 많이 쪼갠다고 무조건 빨라지진 않는다

---

## 스레드와 병렬 처리
스레드는 병렬 처리를 구현할 때 자주 쓰이는 도구다.

### 이유
- 가볍다
- 같은 메모리를 공유하기 쉽다
- 작업을 나눠 실행하기 좋다

### 주의
- 공유 때문에 동기화가 필요하다
- 스레드 수가 많다고 무조건 빨라지지 않는다

---

## 프로세스와 병렬 처리
프로세스도 병렬 처리에 쓸 수 있다.

### 장점
- 격리와 안정성이 좋다
- 프로세스가 서로 영향을 덜 준다

### 단점
- 생성/전환 비용이 더 클 수 있다
- 메모리 공유가 어렵다

---

## 병렬 처리 구조 그림

```text
큰 작업
  ├─ 작업 A ─→ 코어 1
  ├─ 작업 B ─→ 코어 2
  ├─ 작업 C ─→ 코어 3
  └─ 작업 D ─→ 코어 4

결과 합치기
```

---

## Amdahl의 법칙
[[용어 모음집/Amdahl의 법칙|Amdahl의 법칙]]은 **병렬화할 수 없는 부분이 전체 성능 향상의 한계를 정한다**는 내용이다.

### 핵심 이해
- 아무리 병렬화를 잘해도 일부분은 직렬로 남는다
- 그 직렬 부분이 전체 속도의 상한을 만든다

### 예시
- 전체 작업의 90%를 병렬화할 수 있어도
- 나머지 10%가 남으면 속도 향상에 한계가 있다

### 아주 쉬운 이해
- 줄 서서 하나씩 해야 하는 구간이 남아 있으면 전체가 그 구간에 묶인다

---

## Amdahl의 법칙 그림

```text
전체 작업 = 직렬 부분 + 병렬 부분

직렬 부분은 1명만 가능
병렬 부분은 여러 명 가능

=> 전체 속도는 직렬 부분에 막힌다
```

---

## 오버헤드(overhead)
오버헤드는 **병렬화를 위해 추가로 지불하는 비용**이다.

### 예시
- 스레드 생성 비용
- 작업 분배 비용
- 동기화 비용
- 결과 합치기 비용
- [[용어 모음집/캐시|캐시]] 불일치 비용

### 중요한 점
- 병렬화가 항상 이득인 건 아니다
- 오버헤드가 이득보다 크면 느려질 수 있다

---

## 병목(bottleneck)
병목은 **전체 성능을 가장 많이 제한하는 구간**이다.

### 예시
- 디스크 I/O가 느림
- 락 경합이 심함
- 합치는 단계가 한 줄로 서 있음

### 쉽게 이해하기
- 물이 병목 구간에서만 좁게 흐르는 현상

---

## 동기화 비용
병렬 처리에서 공유 자원이 많으면 동기화 비용이 커진다.

### 왜 문제인가
- 락 대기
- 경쟁 증가
- 순서 강제
- 데드락 가능성

### 해결 기준
- 공유를 줄인다
- 데이터 분할을 잘한다
- 가능한 순수 계산 부분을 늘린다

---

## 데이터 병렬화와 작업 병렬화

### 데이터 병렬화
- 같은 연산을 여러 데이터에 나눠 적용
- 예: 리스트를 구간별 처리

### 작업 병렬화
- 서로 다른 작업을 동시에 처리
- 예: 렌더링, 입력, 네트워크를 분리

### 비교표

| 항목 | 데이터 병렬화 | 작업 병렬화 |
|---|---|---|
| 방식 | 같은 작업을 데이터에 분산 | 서로 다른 작업을 분산 |
| 예시 | 이미지 타일 처리 | UI/네트워크/렌더링 분리 |
| 장점 | 구조가 단순할 수 있음 | 역할 분리가 명확함 |

---

## SIMD
[[용어 모음집/SIMD|SIMD]]는 **하나의 명령으로 여러 데이터를 동시에 처리하는 방식**이다.

### 예시
- 벡터 연산
- 이미지 픽셀 처리
- 행렬 계산

### 쉽게 이해하기
- 같은 작업을 한 번에 여러 칸에 적용

---

## MIMD
[[용어 모음집/MIMD|MIMD]]는 **여러 명령을 여러 데이터에 대해 동시에 실행하는 방식**이다.

### 쉽게 이해하기
- 각 코어가 다른 일을 할 수 있다
- 일반적인 멀티코어 병렬 처리와 연결된다

---

## SIMD vs MIMD 비교표

| 항목 | SIMD | MIMD |
|---|---|---|
| 의미 | 하나의 명령, 여러 데이터 | 여러 명령, 여러 데이터 |
| 잘 맞는 작업 | 벡터/픽셀/행렬 | 일반 멀티코어 작업 |
| 쉽게 이해하기 | 같은 연산을 한꺼번에 | 각자 다른 일을 병렬 수행 |

---

## GPU와 병렬 처리
GPU는 병렬 처리에 특화된 하드웨어로 자주 쓰인다.

### 특징
- 아주 많은 작은 연산을 동시에 처리하기 좋다
- 그래픽, 딥러닝, 행렬 계산에 강하다

### 쉽게 이해하기
- CPU는 범용 작업에 강하고
- GPU는 대량 병렬 연산에 강하다

---

## 병렬 처리의 장점
- 속도가 빨라질 수 있다
- 처리량이 높아질 수 있다
- 대규모 데이터를 다루기 좋다
- 응답성을 높일 수 있다

---

## 병렬 처리의 단점
- 설계가 어렵다
- 동기화가 필요하다
- 디버깅이 어렵다
- 결과 순서가 비결정적일 수 있다
- 오버헤드가 크면 손해다

---

## 자주 하는 실수
- 병렬성과 동시성을 같은 말로 생각함
  - 바로잡기: 동시성은 여러 작업을 다루는 구조이고, 병렬성은 여러 작업이 실제로 동시에 실행되는 성질이다.
- 스레드만 늘리면 빨라진다고 생각함
  - 바로잡기: 스레드 수를 늘려도 직렬 구간, 락 경합, 문맥 교환, I/O 병목이 있으면 빨라지지 않을 수 있다.
- 오버헤드를 고려하지 않음
  - 바로잡기: 작업 분할, 스레드 생성, 동기화, 결과 병합 비용이 병렬화 이득보다 작아야 효과가 난다.
- 동기화 비용을 무시함
  - 바로잡기: 락 대기와 캐시 일관성 비용은 병렬 처리의 성능을 제한하는 중요한 요소다.
- Amdahl의 법칙을 단순 공식으로만 외움
  - 바로잡기: Amdahl의 법칙은 병렬화할 수 없는 직렬 부분이 전체 속도 향상의 상한을 정한다는 의미다.
- 데이터 분할보다 결과 합치기 문제를 놓침
  - 바로잡기: 데이터를 나누는 것만큼 부분 결과를 올바르고 빠르게 합치는 단계도 병렬 성능에 중요하다.

---

## 실전 예시

### 예시 1: 이미지 필터 적용
- 사진을 여러 조각으로 나눠 각 스레드가 처리
- 마지막에 합쳐 출력

### 예시 2: 서버 요청 처리
- 요청을 여러 워커가 나눠 처리
- 처리량을 높인다

### 예시 3: 수치 계산
- 큰 배열의 합, 평균, 변환을 여러 코어에 나눠 계산

---

## 디버깅할 때 보는 순서
1. 작업이 병렬화 가능한지 본다
2. 직렬 부분이 얼마나 되는지 본다
3. 분할/합치기 비용을 본다
4. 락과 공유 자원이 많은지 본다
5. 스레드 수가 적절한지 본다
6. 캐시 친화성이 있는지 본다
7. 실제 병목이 CPU인지 I/O인지 본다

---

## 비교해서 이해하기

### 병렬 처리 vs 동시성
- 병렬 처리: 실제 동시에 처리
- 동시성: 번갈아 처리해도 동시에 보일 수 있음

### 병렬 처리 vs 순차 처리
- 병렬 처리: 나눠서 동시에
- 순차 처리: 하나씩 차례대로

### SIMD vs 일반 스레드 병렬
- SIMD: 하나의 명령으로 여러 데이터
- 스레드 병렬: 여러 실행 흐름으로 분산

---

## ASCII 그림

### 분할과 합치기
```text
[큰 작업] → [A][B][C][D] → 병렬 실행 → [결과 합치기]
```

### Amdahl 법칙 이해
```text
직렬 부분 ─┐
병렬 부분 ──┼→ 전체 속도 제한
```

### 데이터 병렬
```text
데이터1  데이터2  데이터3  데이터4
   ↓        ↓        ↓        ↓
   같은 연산을 동시에
```

## 체크리스트
- [ ] 병렬 처리를 한 문장으로 설명할 수 있다
- [ ] 병렬성과 동시성 차이를 안다
- [ ] Amdahl의 법칙이 왜 중요한지 안다
- [ ] 오버헤드와 병목을 설명할 수 있다
- [ ] SIMD와 MIMD 차이를 안다
- [ ] 병렬화가 항상 이득은 아님을 안다
- [ ] 작업 분할과 결과 합치기의 중요성을 안다

## 면접 포인트
면접에서는 병렬 처리를 "여러 개를 동시에 하는 것"으로만 말하면 약하다. **병렬성과 동시성 차이, Amdahl의 법칙, 오버헤드, 동기화**까지 연결해야 한다.

### 꼭 말할 수 있어야 하는 것
- 병렬 처리는 여러 작업을 실제로 동시에 처리하는 방식이다
- 병렬성과 동시성은 다르다
- Amdahl의 법칙 때문에 직렬 부분이 전체 성능의 한계를 만든다
- 오버헤드와 동기화 비용이 크면 병렬화가 오히려 손해일 수 있다
- SIMD는 하나의 명령으로 여러 데이터를 처리하고, MIMD는 여러 명령을 병렬로 처리하는 방식이다

### 한 번에 말하는 답변 예시
> 병렬 처리는 작업을 여러 부분으로 나누어 여러 코어에서 동시에 처리하는 방식입니다. 다만 병렬성은 동시성과 다르고, 병렬화할 수 없는 직렬 부분이 있으면 Amdahl의 법칙 때문에 전체 속도 향상에 한계가 있습니다. 또한 작업 분할, 동기화, 결과 합치기 같은 오버헤드가 커지면 오히려 성능이 떨어질 수 있습니다.

- 병목과 오버헤드는 어떻게 구분하나요?

## 한 번 더 요약
- 병렬 처리는 여러 작업을 실제로 동시에 처리하는 방법이다
- 병렬성과 동시성은 다르다
- Amdahl의 법칙이 병렬화의 한계를 정한다
- 오버헤드와 동기화 비용을 꼭 봐야 한다
- 작업 분할과 결과 합치기가 핵심이다

## 초보자용 한 줄 기억법
병렬 처리는 **"큰 일을 여러 명이 동시에 나눠 하는 것"**이다.

## 보강 설명
병렬 처리 주제는 “작업을 나눴을 때 실제로 빨라지는가”를 따져야 한다. 나눌 수 없는 직렬 구간, 동기화 비용, 결과 합치기 비용이 전체 성능의 한계를 만든다.

## 구체적인 예시
예를 들어 이미지 필터는 영역별로 나누면 여러 코어가 동시에 처리할 수 있다. 하지만 마지막에 결과를 합치는 단계가 길거나 공유 락을 자주 잡으면 기대만큼 빨라지지 않는다.
