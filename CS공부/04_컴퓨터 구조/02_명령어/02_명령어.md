
# 명령어

## 한 줄 정의
[[용어 모음집/명령어|명령어]](instruction)는 **[[용어 모음집/CPU|CPU]]가 직접 이해하고 실행하는 최소 단위의 작업 지시**다.

쉽게 말하면, CPU에게 주는 아주 짧은 "할 일 목록"이다.

---

## 왜 배우는가
- 프로그램은 결국 명령어들의 연속으로 실행된다.
- 명령어를 알아야 CPU가 어떻게 계산하는지 이해할 수 있다.
- [[용어 모음집/어셈블리|어셈블리]], 기계어, [[용어 모음집/ISA|ISA]], 주소 지정 방식, 성능 최적화가 모두 연결된다.
- 디버깅과 성능 분석에서 병목을 찾을 때 도움이 된다.
- 면접에서 컴퓨터 구조 기본 질문으로 자주 나온다.

---

## 초보자용 한 줄 기억법
- **명령어 = CPU가 수행하는 한 줄짜리 작업 지시**

---

## 명령어를 한 문장으로 다시 말하면
==명령어는 **CPU가 [[용어 모음집/레지스터|레지스터]]와 메모리를 이용해 특정 연산, 저장, 분기, 입력/출력 등의 동작을 수행하도록 정의된 실행 단위**다.==

---

## 전체 흐름

```text
프로그램
  ↓
명령어들의 모음
  ↓
CPU가 한 줄씩 해석/실행
  ↓
레지스터/메모리 상태 변화
```

---

## 명령어가 왜 중요한가
==프로그램이 아무리 복잡해 보여도, CPU 입장에서는 결국 명령어를 하나씩 처리하는 일이다.==

### 핵심 이해
- 고급 언어 코드 → 컴파일/번역 → 명령어
- CPU는 명령어를 읽고 실행
- 실행 결과가 메모리와 레지스터에 반영됨

즉, 명령어를 이해하면 "프로그램이 실제로 어떻게 움직이는지" 보이기 시작한다.

---

## ISA란
==ISA(Instruction Set Architecture)는 **CPU가 이해할 수 있는 명령어의 규칙과 형식**이다.==

### ISA가 정하는 것
- 어떤 명령어가 있는가
- 각 명령어가 어떤 형식을 가지는가
- 레지스터를 어떻게 다루는가
- 주소를 어떻게 지정하는가
- 메모리 접근 방식은 어떤가

### 쉽게 말하면
- ISA는 CPU와 프로그램 사이의 "약속"이다

---

## ISA의 예시
- x86
- ARM
- [[용어 모음집/RISC|RISC]]-V

### 중요한 점
같은 일을 해도 ISA에 따라 명령어 모양과 실행 방식이 다를 수 있다.

---

## 기계어와 어셈블리

### 기계어(machine code)
- CPU가 직접 실행하는 0과 1의 형태
- 사람에게는 읽기 어렵다

### 어셈블리(assembly)
- 기계어를 사람이 읽기 쉽게 적은 표현
- `mov`, `add`, `jmp` 같은 형태

### 관계
```text
어셈블리 → 어셈블러 → 기계어 → CPU 실행
```

### 쉽게 이해하기
- 어셈블리는 기계어의 사람 버전

---

## 명령어의 구성 요소
명령어는 보통 아래 요소들로 설명할 수 있다.

- **opcode**: 어떤 작업을 할지
- **operand**: 작업 대상이 무엇인지
- **addressing mode**: operand를 어떻게 해석할지

---

## opcode
opcode는 **명령어의 종류**다.

### 예시
- add: 더하기
- load: 메모리에서 읽기
- store: 메모리에 쓰기
- branch: 분기하기
- jump: 점프하기

### 쉽게 이해하기
- opcode는 "동사"에 가깝다

---

## operand
operand는 **명령어가 다루는 대상**이다.

### 예시
- 레지스터 번호
- 메모리 주소
- 즉시값(immediate value)

### 쉽게 이해하기
- operand는 동작의 "목적어"다

---

## 즉시값(immediate)
즉시값은 **명령어 안에 바로 들어 있는 상수 값**이다.

### 예시
- `add r1, r2, #5`
- 여기서 `#5`는 즉시값

### 장점
- 바로 쓸 수 있어 편하다

### 주의
- 크기가 제한될 수 있다

---

## 명령어 형식
명령어 형식은 **opcode와 operand를 어떤 비트 구조로 담는지**를 뜻한다.

### 흔한 구성
- opcode 비트
- 레지스터 번호 비트
- 주소/즉시값 비트
- 모드 비트

### 왜 중요한가
- CPU가 명령어를 빠르게 해석하려면 형식이 명확해야 한다
- 명령어 길이, 디코딩 난이도, 성능과 연결된다

---

## 명령어 형식 예시 그림

```text
[ opcode | operand1 | operand2 | mode ]
```

실제 구조는 ISA마다 다르지만, 이런 식으로 나뉜다고 생각하면 이해하기 쉽다.

---

## 주소 지정 방식(addressing mode)
==주소 지정 방식은 **operand를 실제 값으로 볼지, 레지스터로 볼지, 메모리 주소로 볼지 정하는 방법**이다.==

### 왜 필요하나
같은 명령어라도 데이터가 어디 있는지 표현 방식이 여러 가지일 수 있기 때문이다.

---

## 대표 주소 지정 방식
- 즉시 주소 지정(immediate)
- 레지스터 주소 지정(register)
- 직접 주소 지정(direct)
- 간접 주소 지정(indirect)
- 베이스+오프셋(base + offset)
- 인덱스(indexed)

---

## 주소 지정 방식 비교표

| 방식 | 의미 | 예시로 이해하기 |
|---|---|---|
| immediate | 명령어 안에 값이 바로 있음 | 숫자를 바로 적음 |
| register | 레지스터 값을 사용 | CPU 내부 메모장 사용 |
| direct | 메모리 주소를 직접 지정 | 정확한 집 주소를 적음 |
| indirect | 주소를 가리키는 또 다른 주소를 사용 | 주소록 안의 주소 다시 찾기 |
| base+offset | 기준 주소 + 이동 거리 | 건물 주소 + 호수 |
| indexed | 기준 + 인덱스 | 배열의 n번째 칸 |

---

## 명령어 종류
명령어는 기능에 따라 보통 여러 그룹으로 나눈다.

### 1) 데이터 전달
- load
- store
- move

### 2) 산술/논리 연산
- add
- sub
- and
- or
- xor
- shift

### 3) 제어 흐름
- branch
- jump
- call
- return

### 4) 비교/조건
- compare
- test
- conditional branch

### 5) 시스템/특권 명령
- interrupt
- syscall
- privileged instruction

---

## load/store 구조
load/store 구조는 **메모리 접근을 load와 store로 분리하고, 연산은 주로 레지스터에서 수행하는 방식**이다.

### 쉽게 이해하기
- 메모리에서 값을 가져오는 건 load
- 메모리에 값을 쓰는 건 store
- 계산은 레지스터 안에서 한다

### 장점
- 구조가 단순해진다
- 하드웨어 최적화가 쉬울 수 있다

### 예시
```text
load r1, [mem]
add  r2, r1, #3
store [mem2], r2
```

---

## load/store가 왜 중요한가
- 메모리 접근은 느리다
- 레지스터는 빠르다
- 따라서 계산을 레지스터 중심으로 모아두면 효율적일 수 있다

---

## 명령어와 레지스터
명령어는 대부분 레지스터를 적극적으로 활용한다.

### 이유
- 레지스터가 메모리보다 빠르다
- 명령어 간 데이터 전달이 쉬워진다
- CPU 내부에서 처리하기 좋다

### 쉽게 이해하기
- 레지스터는 CPU 안의 빠른 임시 보관함이다

---

## 분기(branch)
분기는 **다음에 실행할 명령어 주소를 바꾸는 것**이다.

### 예시
- 조건이 참이면 다른 코드로 이동
- 루프 반복
- 함수 호출

### 왜 중요한가
- if, while, for 같은 제어 구조가 분기로 바뀐다
- 프로그램 흐름의 핵심이다

---

## 명령어와 제어 흐름 그림

```text
명령어 1 → 명령어 2 → 명령어 3
                 ↘
                 분기 → 다른 경로
```

---

## 어셈블리 예시
```asm
load r1, [x]
add  r1, r1, #1
store [x], r1
```

### 해석
- x 값을 읽어서
- 1을 더하고
- 다시 x에 저장한다

---

## 명령어 실행 흐름

```text
Fetch → Decode → Execute → Writeback
```

### 각 단계
- Fetch: 명령어를 가져옴
- Decode: 어떤 명령인지 해석
- Execute: 실제 수행
- Writeback: 결과를 저장

---

## 명령어 사이클과 연결
명령어는 [[용어 모음집/명령어 사이클|명령어 사이클]]을 통해 반복해서 처리된다.

### 흐름
- 명령어를 가져오고
- 해독하고
- 실행하고
- 결과를 저장한 뒤
- 다음 명령어로 넘어간다

---

## ==RISC와 CISC 이해==

### RISC
- 단순한 명령어 집합
- 길이와 형식이 비교적 단순한 편
- load/store 구조와 잘 어울린다

### CISC
- 더 복잡하고 다양한 명령어
- 한 명령어가 많은 일을 할 수 있다
- x86이 대표적으로 자주 언급된다

### 비교표

| 항목 | RISC | [[용어 모음집/CISC|CISC]] |
|---|---|---|
| 명령어 수 | 비교적 단순 | 비교적 복잡 |
| 디코딩 | 쉬운 편 | 복잡할 수 있음 |
| 철학 | 단순함과 효율 | 다양한 기능 통합 |
| 예시 | ARM, RISC-V | x86 |

---

## 명령어와 성능
명령어는 단순히 기능만이 아니라 성능과도 연결된다.

### 영향을 주는 요소
- 명령어 길이
- 디코딩 난이도
- [[용어 모음집/분기 예측|분기 예측]]
- 메모리 접근 횟수
- 레지스터 사용 패턴

### 쉽게 이해하기
- 같은 일을 해도 명령어 구성이 다르면 속도가 달라질 수 있다

---

## 파이프라인과 명령어
CPU는 보통 명령어를 한 번에 하나씩만 느리게 처리하지 않고, 여러 단계를 겹쳐서 처리한다.

### 쉽게 이해하기
- 공장 생산라인처럼 명령어를 겹쳐 처리한다
- 분기가 많거나 구조가 복잡하면 효율이 떨어질 수 있다

---

## 실전 예시

### 예시 1: 변수 1 증가
- 메모리에서 읽기(load)
- 1 더하기(add)
- 다시 저장(store)

### 예시 2: if문
- 조건 비교(compare)
- 참/거짓에 따라 branch

### 예시 3: 함수 호출
- call로 주소 이동
- return으로 복귀

## 자주 하는 실수
- 명령어와 ISA를 같은 말로 생각함
  - 바로잡기: 명령어는 CPU가 수행하는 개별 작업이고, ISA는 명령어 집합과 레지스터/주소 지정 규칙 전체를 정한 약속이다.
- 기계어와 어셈블리를 헷갈림
  - 바로잡기: 기계어는 CPU가 직접 읽는 비트열이고, 어셈블리는 그 비트열과 가까운 사람이 읽는 표기다.
- operand와 opcode를 섞음
  - 바로잡기: Opcode는 수행할 연산 종류이고, operand는 그 연산에 사용할 대상 값이나 위치다.
- load/store 구조를 메모리만 쓰는 구조로 오해함
  - 바로잡기: Load/store 구조는 메모리 접근은 load/store로 제한하고 산술 연산은 주로 레지스터 사이에서 수행한다는 의미다.
- 분기와 점프를 같은 말로만 이해함
  - 바로잡기: Jump는 보통 무조건 이동이고, branch는 조건에 따라 이동 여부가 달라지는 제어 흐름 명령이다.
- 주소 지정 방식을 외우기만 하고 실제 의미를 모름
  - 바로잡기: 주소 지정 방식은 피연산자 값을 어디서 어떻게 찾을지 정하는 규칙이다.

## 디버깅할 때 보는 순서
1. 어떤 명령어가 실행되는지 본다
2. opcode와 operand를 구분한다
3. 메모리 접근인지 레지스터 연산인지 본다
4. 분기가 어디서 바뀌는지 본다
5. 주소 지정 방식이 맞는지 본다
6. ISA와 어셈블리 문법이 일치하는지 본다
7. 성능 문제면 load/store 비율과 분기 수를 본다

## 비교해서 이해하기

### 명령어 vs ISA
- 명령어: CPU가 하는 한 줄 작업
- ISA: 그런 명령어들을 정의한 규칙

### 기계어 vs 어셈블리
- 기계어: 0과 1
- 어셈블리: 사람이 읽는 표현

### load vs store
- load: 메모리 → 레지스터
- store: 레지스터 → 메모리

### branch vs jump
- branch: 조건에 따라 이동
- jump: 무조건 이동

```text
[ opcode | operand | mode ]
```

### 실행 흐름
```text
명령어 읽기 → 해석 → 실행 → 결과 저장
```

### load/store
```text
메모리 → load → 레지스터 → 연산 → 레지스터 → store → 메모리
```

## 면접 포인트
면접에서는 명령어를 단순히 "CPU가 실행하는 코드"라고만 말하면 부족하다. **ISA, 주소 지정 방식, 명령어 형식, load/store 구조, 제어 흐름**까지 연결해야 한다.

### 꼭 말할 수 있어야 하는 것
- 명령어는 CPU가 이해하는 최소 실행 단위다
- ISA가 명령어 집합과 형식을 정의한다
- 기계어는 0과 1이고 어셈블리는 사람이 읽기 쉽게 쓴 표현이다
- 명령어는 opcode와 operand로 구성되고 주소 지정 방식이 중요하다
- load/store 구조에서는 메모리 접근과 연산을 분리한다

### 한 번에 말하는 답변 예시
> 명령어는 CPU가 직접 실행하는 최소 단위의 작업 지시입니다. ISA가 어떤 명령어와 형식을 사용할지 정의하고, 어셈블리는 이를 사람이 읽기 쉬운 형태로 표현한 것입니다. 명령어는 보통 opcode와 operand로 구성되며, operand를 어떻게 해석하는지에 따라 주소 지정 방식이 달라집니다. 또한 load/store 구조에서는 메모리 접근을 load와 store로 분리하고 연산은 주로 레지스터에서 수행합니다.

### 꼬리 질문 예시
- ISA는 왜 필요한가요?

## 한 번 더 요약
- 명령어는 CPU가 실행하는 최소 단위다
- ISA가 명령어의 규칙을 정의한다
- 기계어는 0과 1, 어셈블리는 사람이 읽는 표현이다
- opcode/operand/addressing mode가 핵심이다
- load/store, branch, jump 같은 분류를 함께 봐야 한다

## 초보자용 한 줄 기억법
명령어는 **"CPU에게 하는 짧은 작업 지시문"**이다.

## 보강 설명
명령어 주제는 소프트웨어와 하드웨어의 경계다. 고급 언어 코드는 컴파일을 거쳐 ISA가 정의한 명령어와 레지스터 사용 규칙에 맞춰 실행된다.

## 구체적인 예시
예를 들어 `x = arr[i]`는 배열 시작 주소와 인덱스를 이용해 실제 메모리 주소를 계산하고, load 명령으로 값을 레지스터에 가져오는 흐름으로 바뀔 수 있다.
