# 캐시 일관성

## 한 줄 정의
[[용어 모음집/캐시 일관성|캐시 일관성]](cache coherence)은 **여러 코어가 각자 캐시를 가질 때 같은 메모리 주소의 값을 서로 모순 없이 보게 만드는 문제와 해결 방식**이다.

## 왜 문제가 생기는가
멀티코어 [[용어 모음집/CPU|CPU]]에서는 코어마다 L1/[[용어 모음집/L2 캐시|L2 캐시]]가 있을 수 있다. 두 코어가 같은 변수 `x`를 캐시에 올린 뒤, 한 코어가 `x`를 수정하면 다른 코어의 캐시에는 오래된 값이 남을 수 있다.

```text
Core 1 cache: x = 10
Core 2 cache: x = 10

Core 1이 x = 20으로 수정

Core 2 cache는 아직 x = 10일 수 있음
```

## 해결 방향
- 한 코어가 값을 수정하면 다른 코어의 같은 캐시 라인을 무효화한다.
- 또는 수정된 값을 다른 캐시에 알려준다.
- 캐시 라인 상태를 추적해 읽기/쓰기 가능 여부를 관리한다.

## MESI 프로토콜 이해
[[용어 모음집/MESI|MESI]]는 대표적인 캐시 일관성 프로토콜이다.

| 상태 | 의미 |
|---|---|
| Modified | 이 캐시에만 최신 값이 있고 메모리와 다름 |
| Exclusive | 이 캐시에만 있고 메모리와 같음 |
| Shared | 여러 캐시에 있고 메모리와 같음 |
| Invalid | 유효하지 않은 캐시 라인 |

## 예시
두 [[용어 모음집/스레드|스레드]]가 같은 카운터를 계속 증가시키면 같은 캐시 라인을 두 코어가 번갈아 수정하게 된다. 이때 캐시 라인 무효화가 계속 발생해 성능이 크게 떨어질 수 있다. 이런 현상을 줄이려면 공유 쓰기를 줄이거나, 스레드별 지역 카운터를 두고 나중에 합치는 방식이 좋다.

## False sharing
서로 다른 변수를 쓰더라도 같은 캐시 라인에 들어 있으면 불필요한 일관성 트래픽이 생길 수 있다.

예시:
```text
cache line: [thread1_count][thread2_count]
```

두 값은 논리적으로 다르지만 같은 캐시 라인에 있어 서로의 수정이 충돌처럼 보일 수 있다.

## 자주 헷갈리는 점
- 캐시 일관성은 캐시와 메모리 사이의 단순 [[용어 모음집/동기화|동기화]] 문제가 아니라 여러 코어의 관찰 결과를 맞추는 문제다.
- 일관성이 맞아도 프로그램의 실행 순서까지 항상 직관적으로 보장되는 것은 아니다. 메모리 모델과 동기화 원시도 함께 봐야 한다.
- 락을 쓰면 [[용어 모음집/상호 배제|상호 배제]]뿐 아니라 메모리 가시성 문제도 함께 해결하는 경우가 많다.

## 면접 포인트
캐시 일관성은 멀티코어에서 같은 메모리 값을 여러 캐시가 들고 있을 때 생기는 불일치를 해결하는 개념이다. MESI 같은 프로토콜은 캐시 라인 상태를 관리하며, false sharing은 서로 다른 데이터가 같은 캐시 라인에 있어 성능이 떨어지는 사례다.

## 보강 설명
캐시 주제는 “빠른 CPU와 느린 메모리 사이의 간격을 어떻게 줄이는가”가 핵심이다. 지역성이 좋은 접근은 캐시 적중률을 높이고, 캐시 미스나 일관성 비용은 성능을 떨어뜨린다.

## 구체적인 예시
예를 들어 배열을 앞에서부터 순서대로 읽으면 한 캐시 라인에 들어온 주변 원소를 이어서 쓰기 때문에 빠르다. 반대로 큰 배열을 무작위로 접근하거나 여러 코어가 같은 캐시 라인을 계속 수정하면 미스와 일관성 트래픽이 늘어난다.
