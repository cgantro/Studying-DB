# L1 L2 L3와 쓰기 정책

## 한 줄 정의
L1, L2, [[용어 모음집/L3 캐시|L3 캐시]]는 **[[용어 모음집/CPU|CPU]]와 메모리 사이에 놓인 계층형 고속 저장소**이고, 쓰기 정책은 **캐시에 쓴 데이터를 메모리에 언제 반영할지 정하는 방식**이다.

## 캐시 계층
| 계층 | 특징 | 쉽게 이해하기 |
|---|---|---|
| L1 | 가장 빠르고 가장 작음 | 코어 바로 옆의 책상 |
| L2 | L1보다 크지만 조금 느림 | 책상 옆 서랍 |
| L3 | 여러 코어가 공유하는 경우가 많음 | 팀 공용 자료함 |
| 메모리 | 크지만 캐시보다 느림 | 멀리 있는 창고 |

## 왜 여러 단계로 나누는가
작고 빠른 캐시만 크게 만들기는 어렵다. 그래서 매우 빠른 L1, 중간 크기의 L2, 더 큰 L3를 계층화해 속도와 용량 사이의 균형을 맞춘다.

```text
CPU core → L1 → L2 → L3 → Main Memory
```

## 쓰기 정책
### Write-through
캐시에 쓰는 동시에 메모리에도 바로 반영한다.

장점:
- 메모리와 캐시 값이 비교적 일관됨
- 구현과 복구가 단순할 수 있음

단점:
- 쓰기마다 메모리 반영 비용이 커질 수 있음

### Write-back
캐시에 먼저 쓰고, 나중에 캐시 라인이 교체될 때 메모리에 반영한다.

장점:
- 반복 쓰기에 유리
- 메모리 접근 횟수를 줄일 수 있음

단점:
- 캐시와 메모리 값이 일시적으로 다를 수 있음
- dirty bit 같은 관리 정보가 필요함

## 예시
같은 변수 `count`를 반복해서 증가시키는 경우 write-back은 캐시 안에서 여러 번 갱신한 뒤 마지막에 한 번만 메모리에 반영할 수 있어 효율적이다.

## 자주 헷갈리는 점
- L1이 항상 전체 시스템에서 공유되는 것은 아니다. 보통 코어별 L1이 있고, L3는 공유되는 구조가 흔하다.
- write-back은 "메모리에 안 쓴다"가 아니라 "나중에 쓴다"는 뜻이다.
- 캐시 정책은 성능뿐 아니라 일관성, 전력, 구현 복잡도와도 관련된다.

## 면접 포인트
캐시는 CPU에 가까울수록 빠르고 작으며, 멀수록 느리지만 크다. 쓰기 정책은 write-through와 write-back으로 나눌 수 있고, write-back은 dirty bit를 이용해 나중에 메모리에 반영한다.

## 보강 설명
캐시 주제는 “빠른 CPU와 느린 메모리 사이의 간격을 어떻게 줄이는가”가 핵심이다. 지역성이 좋은 접근은 캐시 적중률을 높이고, 캐시 미스나 일관성 비용은 성능을 떨어뜨린다.

## 구체적인 예시
예를 들어 배열을 앞에서부터 순서대로 읽으면 한 캐시 라인에 들어온 주변 원소를 이어서 쓰기 때문에 빠르다. 반대로 큰 배열을 무작위로 접근하거나 여러 코어가 같은 캐시 라인을 계속 수정하면 미스와 일관성 트래픽이 늘어난다.
