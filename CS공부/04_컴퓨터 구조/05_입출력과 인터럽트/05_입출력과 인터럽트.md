# 입출력과 인터럽트

## 한 줄 정의
입출력과 인터럽트는 **[[용어 모음집/CPU|CPU]], 메모리, 주변 장치가 데이터를 주고받고, 장치가 CPU에게 사건을 알리는 방식**이다.

쉽게 말하면, 키보드 입력, 디스크 읽기, 네트워크 패킷 도착 같은 일이 생겼을 때 **하드웨어와 CPU가 어떻게 서로 신호를 주고받는지**를 보는 주제다.

---

## 왜 배우는가
- 컴퓨터는 CPU만으로 동작하지 않는다.
- 디스크, 네트워크 카드, 키보드, 모니터 같은 장치와 계속 통신한다.
- 인터럽트를 이해하면 [[용어 모음집/운영체제|운영체제]]의 [[용어 모음집/시스템 콜|시스템 콜]], [[용어 모음집/스케줄링|스케줄링]], I/O 대기 흐름이 더 잘 보인다.
- DMA와 memory-mapped I/O는 성능과 하드웨어 제어를 설명할 때 중요하다.

---

## 전체 흐름

```text
장치에서 사건 발생
   ↓
인터럽트 또는 상태 레지스터로 알림
   ↓
CPU가 현재 작업 잠시 중단
   ↓
인터럽트 핸들러 실행
   ↓
필요한 데이터 이동 또는 상태 갱신
   ↓
원래 작업으로 복귀
```

---

## 핵심 개념
- **버스(bus)**: CPU, 메모리, 장치를 연결하는 통로
- **I/O 장치**: 입력/출력을 담당하는 주변 장치
- **컨트롤러**: 장치를 제어하고 CPU와 연결하는 하드웨어
- **폴링(polling)**: CPU가 장치 상태를 반복 확인하는 방식
- **인터럽트(interrupt)**: 장치가 CPU에게 사건을 알리는 방식
- **DMA**: CPU 개입을 줄이고 장치가 메모리와 직접 데이터 이동을 하는 방식
- **memory-mapped I/O**: 장치 [[용어 모음집/레지스터|레지스터]]를 메모리 주소처럼 접근하는 방식

---

## 폴링 vs 인터럽트

| 항목 | 폴링 | 인터럽트 |
|---|---|---|
| 방식 | CPU가 계속 확인 | 장치가 알림 |
| 장점 | 단순함 | CPU 낭비 감소 |
| 단점 | 바쁜 대기 가능 | 처리 흐름이 복잡 |
| 어울리는 경우 | 매우 짧고 잦은 확인 | 드문 이벤트, I/O 완료 |

---

## DMA가 필요한 이유
큰 데이터를 장치에서 메모리로 옮길 때 CPU가 바이트마다 개입하면 비효율적이다.

```text
장치 → DMA 컨트롤러 → 메모리
                 ↓
             완료 인터럽트
```

CPU는 설정과 완료 처리에 집중하고, 실제 대량 복사는 DMA가 맡는다.

---

## 운영체제와 연결
- 인터럽트는 [[용어 모음집/커널|커널]]이 장치 이벤트를 처리하는 출발점이다
- I/O 완료 인터럽트는 잠든 [[용어 모음집/프로세스|프로세스]]를 깨울 수 있다
- DMA는 디스크/네트워크 I/O 성능에 중요하다
- memory-mapped I/O는 드라이버가 장치를 제어하는 방식과 연결된다

---

## 면접 포인트
> 입출력은 CPU와 주변 장치가 데이터를 주고받는 과정이고, 인터럽트는 장치가 CPU에게 이벤트를 알리는 방식입니다. 폴링은 CPU가 상태를 계속 확인하는 방식이라 단순하지만 낭비가 있을 수 있고, 인터럽트는 이벤트가 생겼을 때만 처리해 효율적입니다. 큰 데이터 이동에는 CPU 부담을 줄이기 위해 DMA를 사용합니다.

## 보강 설명
입출력 주제는 CPU가 느린 장치와 어떻게 협력하는지 보는 영역이다. CPU가 계속 기다릴지, 장치가 인터럽트로 알릴지, DMA로 데이터 이동을 맡길지에 따라 효율이 달라진다.

## 구체적인 예시
예를 들어 디스크에서 큰 파일을 읽을 때 CPU가 바이트마다 직접 옮기면 비효율적이다. DMA를 사용하면 장치가 메모리로 직접 데이터를 옮기고, 완료되면 인터럽트로 CPU에게 알린다.
