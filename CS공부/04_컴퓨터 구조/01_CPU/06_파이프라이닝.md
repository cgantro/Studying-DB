# 파이프라이닝

## 한 줄 정의
[[용어 모음집/파이프라이닝|파이프라이닝]]은 **[[용어 모음집/명령어|명령어]] 실행 단계를 여러 구간으로 나누고, 여러 명령어가 각 단계를 동시에 지나가게 해서 [[용어 모음집/처리량|처리량]]을 높이는 기법**이다.

---

## 왜 필요한가
한 명령어가 모든 단계를 끝낼 때까지 다음 명령어를 기다리면 [[용어 모음집/CPU|CPU]] 자원이 놀게 된다.

```text
명령어1: Fetch Decode Execute Memory Write
명령어2:                         Fetch Decode ...
```

파이프라이닝은 이런 단계를 겹쳐 처리한다.

---

## 파이프라인 흐름

```text
Cycle 1: I1 Fetch
Cycle 2: I1 Decode   | I2 Fetch
Cycle 3: I1 Execute  | I2 Decode   | I3 Fetch
Cycle 4: I1 Memory   | I2 Execute  | I3 Decode | I4 Fetch
Cycle 5: I1 Write    | I2 Memory   | I3 Execute| I4 Decode
```

한 명령어의 지연 시간이 크게 줄어드는 것보다는, **전체 처리량**이 좋아지는 개념이다.

---

## 핵심 개념
- **latency**: 한 명령어가 끝날 때까지 걸리는 시간
- **throughput**: 단위 시간당 처리되는 명령어 수
- **hazard**: 파이프라인 진행을 막는 위험 상황
- **stall**: hazard 때문에 파이프라인을 잠시 멈추는 것
- **flush**: 잘못 가져온 명령어를 버리는 것
- **forwarding**: 아직 저장 전인 결과를 뒤 명령어에 바로 전달하는 것
- **branch prediction**: 분기 결과를 미리 예측하는 것

---

## 파이프라인 hazard

| 종류 | 원인 | 예시 |
|---|---|---|
| 구조적 hazard | 같은 하드웨어 자원 충돌 | 명령어 fetch와 데이터 접근이 같은 메모리 사용 |
| 데이터 hazard | 이전 명령어 결과가 아직 준비되지 않음 | `add` 결과를 바로 다음 명령어가 사용 |
| 제어 hazard | 분기 때문에 다음 명령어 주소가 불확실 | if/loop 분기 |

---

## 데이터 hazard 예시

```asm
add r1, r2, r3
sub r4, r1, r5
```

두 번째 명령어는 `r1`이 필요하지만, 첫 번째 명령어 결과가 아직 [[용어 모음집/레지스터|레지스터]]에 쓰이지 않았을 수 있다.

### 대응
- stall
- forwarding
- 컴파일러의 명령어 재배치

---

## 제어 hazard와 분기 예측
분기 명령어는 다음에 어떤 주소의 명령어를 가져와야 할지 불확실하게 만든다.

```text
if 조건이 참이면 A
아니면 B
```

CPU는 흐름을 멈추지 않기 위해 분기 결과를 예측한다. 예측이 틀리면 잘못 가져온 명령어를 버리고 다시 가져와야 한다.

---

## 깊은 파이프라인의 trade-off
파이프라인 단계를 깊게 나누면 [[용어 모음집/클럭|클럭]]을 높이기 쉬울 수 있지만, [[용어 모음집/분기 예측|분기 예측]] 실패나 stall 비용이 커질 수 있다.

| 장점 | 단점 |
|---|---|
| 높은 클럭 가능성 | 분기 실패 비용 증가 |
| 처리량 향상 가능 | 제어가 복잡해짐 |
| 단계별 작업 단순화 | hazard 관리 비용 증가 |

---

## 파이프라인과 병렬성
파이프라이닝은 여러 명령어를 동시에 처리하는 것처럼 보이지만, 각 명령어가 서로 다른 단계를 지나가는 방식이다.

- 명령어 수준 [[용어 모음집/병렬성|병렬성]](ILP)을 활용한다
- 여러 코어에서 동시에 다른 작업을 하는 병렬 처리와는 구분된다

---

## 자주 하는 실수
- 파이프라인이 한 명령어의 실행 시간을 무조건 줄인다고 생각함
  - 바로잡기: 파이프라이닝의 핵심 효과는 한 명령어의 지연 시간 감소보다 단위 시간당 완료 명령어 수 증가다.
- hazard를 단순 오류로만 생각함
  - 바로잡기: Hazard는 오류라기보다 파이프라인 진행을 막는 의존성, 자원 충돌, 분기 불확실성이다.
- stall과 flush를 같은 것으로 봄
  - 바로잡기: Stall은 기다리기 위해 파이프라인을 멈추는 것이고, flush는 잘못 가져온 명령어를 버리는 것이다.
- 분기 예측이 항상 맞는다고 생각함
  - 바로잡기: 분기 예측은 성능 최적화 기법이지만 틀릴 수 있으며, 실패하면 잘못 가져온 명령어를 버리는 비용이 든다.
- 파이프라이닝과 멀티코어 병렬 처리를 같은 개념으로 봄
  - 바로잡기: 파이프라이닝은 한 코어 안에서 여러 명령어의 실행 단계를 겹치는 기법이고, 멀티코어 병렬 처리는 여러 코어가 독립 실행 흐름을 나눠 처리하는 방식이다.

---

## 면접 포인트
> 파이프라이닝은 명령어 실행 단계를 나누고 여러 명령어가 서로 다른 단계를 동시에 지나가게 해서 처리량을 높이는 기법입니다. 한 명령어의 latency를 줄이기보다는 전체 throughput을 높이는 데 초점이 있습니다. 단, 구조적 hazard, 데이터 hazard, 제어 hazard가 발생할 수 있고, 이를 해결하기 위해 stall, forwarding, branch prediction, flush 같은 기법을 사용합니다.

---

## 체크리스트
- [ ] 파이프라이닝이 처리량을 높이는 기법임을 안다
- [ ] latency와 throughput 차이를 설명할 수 있다
- [ ] 구조적/데이터/제어 hazard를 구분할 수 있다
- [ ] forwarding과 stall을 설명할 수 있다
- [ ] 분기 예측 실패와 flush를 연결할 수 있다
---

## 보강 설명
CPU 주제는 “명령어가 실제로 어떻게 실행되는가”를 기준으로 연결해서 보는 것이 좋다. [[용어 모음집/PC|PC]]가 명령어 주소를 가리키고, [[용어 모음집/제어장치|제어장치]]가 명령어를 해석하며, 레지스터와 [[용어 모음집/ALU|ALU]]가 값을 주고받고, 결과가 다시 레지스터나 메모리에 반영된다.

## 구체적인 예시
예를 들어 `a = b + c` 같은 코드는 메모리에서 값을 읽고, 레지스터에 올리고, ALU에서 더한 뒤, 결과를 다시 저장하는 여러 명령어로 바뀐다. 파이프라이닝이 있으면 여러 명령어의 단계가 겹쳐 실행되지만, 데이터 의존성이나 분기 때문에 멈출 수 있다.
