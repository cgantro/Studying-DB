# 클럭 IPC와 성능

## 한 줄 정의
[[용어 모음집/CPU|CPU]] 성능은 **[[용어 모음집/클럭|클럭]] 속도뿐 아니라 한 클럭에 처리하는 [[용어 모음집/명령어|명령어]] 수, 메모리 대기, [[용어 모음집/분기 예측|분기 예측]], 파이프라인 효율이 함께 결정하는 결과**다.

---

## 핵심 개념
- **클럭(clock)**: CPU 동작의 기준 박자
- **클럭 주파수**: 1초에 몇 번 박자가 반복되는지
- **[[용어 모음집/CPI|CPI]]**: 명령어 하나당 평균 몇 클럭이 필요한지
- **[[용어 모음집/IPC|IPC]]**: 한 클럭에 평균 몇 개 명령어를 처리하는지
- **latency**: 한 작업이 끝날 때까지 걸리는 시간
- **throughput**: 단위 시간당 [[용어 모음집/처리량|처리량]]

---

## 클럭만으로 성능을 판단하면 안 되는 이유
같은 3GHz CPU라도 내부 구조와 실행 효율이 다르면 실제 성능이 달라진다.

```text
성능 ≈ 클럭 × IPC
```

여기에 [[용어 모음집/캐시 미스|캐시 미스]], 메모리 대기, 분기 예측 실패, 전력/발열 제한까지 영향을 준다.

---

## CPI와 IPC

| 지표  | 의미                         | 좋을수록    |
| --- | -------------------------- | ------- |
| CPI | 명령어 하나를 처리하는 데 필요한 평균 클럭 수 | 낮을수록 좋음 |
| IPC | 한 클럭에 처리하는 평균 명령어 수        | 높을수록 좋음 |
|     |                            |         |

대략적으로 `IPC = 1 / CPI`처럼 생각할 수 있지만, 실제 CPU에서는 여러 실행 유닛과 파이프라인 때문에 더 복잡하다.

---

## CPU 시간 공식 이해

```text
CPU 시간 = 명령어 수 × CPI × 클럭 사이클 시간
```

또는

```text
CPU 시간 = 명령어 수 × CPI / 클럭 주파수
```

성능 개선은 보통 세 가지 방향이다.

- 명령어 수를 줄인다
- CPI를 낮춘다
- 클럭 주파수를 높인다

---

## 성능에 영향을 주는 요소

| 요소 | 영향 |
|---|---|
| 캐시 미스 | CPU가 메모리를 기다리게 함 |
| 분기 예측 실패 | 파이프라인 flush 발생 |
| 데이터 의존성 | 뒤 명령어가 앞 결과를 기다림 |
| 명령어 종류 | 복잡한 명령어는 더 많은 비용 가능 |
| 병렬 실행 유닛 | 동시에 처리 가능한 명령어 수 증가 |
| 전력/발열 | 높은 클럭 유지에 제한 |

---

## latency와 throughput

### latency
한 작업이 완료될 때까지의 시간이다.

### throughput
일정 시간 동안 처리한 작업량이다.

[[용어 모음집/파이프라이닝|파이프라이닝]]은 보통 latency보다 throughput 개선에 더 직접적이다.

---

## 예시

### 클럭은 높지만 느릴 수 있는 경우
- 캐시 미스가 많다
- 분기 예측 실패가 많다
- 메모리 접근이 많다
- 데이터 의존성 때문에 stall이 많다

### 클럭은 낮아도 빠를 수 있는 경우
- IPC가 높다
- 캐시 효율이 좋다
- 분기 예측이 잘 맞는다
- 같은 일을 더 적은 명령어로 처리한다

---

## 자주 하는 실수
- GHz가 높으면 무조건 빠르다고 생각함
  - 바로잡기: 성능은 클럭, IPC, 캐시 적중률, 분기 예측, 메모리 대기 시간, 코어 구조가 함께 결정한다.
- 코어 수와 단일 [[용어 모음집/스레드|스레드]] 성능을 섞음
  - 바로잡기: 코어 수는 여러 작업 처리량에 유리하고, 단일 스레드 성능은 한 코어의 클럭, IPC, 캐시, 분기 예측 성능에 더 좌우된다.
- IPC와 클럭을 따로만 봄
  - 바로잡기: 실행 시간은 대략 명령어 수, CPI/IPC, 클럭이 함께 결정하므로 IPC와 클럭을 따로 떼어 보면 오해하기 쉽다.
- 캐시 미스를 CPU와 무관한 문제로 생각함
  - 바로잡기: 캐시 미스는 CPU가 필요한 데이터를 제때 받지 못하게 만들어 파이프라인 정지와 메모리 대기 시간을 늘리는 CPU 성능 문제다.
- benchmark 결과를 모든 작업에 일반화함
  - 바로잡기: 벤치마크는 특정 작업 조건의 결과이므로 실제 서비스의 데이터 크기, I/O, 캐시 패턴과 맞는지 확인해야 한다.

---

## 면접 포인트
> CPU 성능은 클럭만으로 결정되지 않습니다. 같은 클럭이라도 한 클럭에 처리할 수 있는 명령어 수인 IPC가 다르면 성능이 달라지고, 캐시 미스나 분기 예측 실패, 데이터 의존성으로 파이프라인이 멈추면 실제 처리량이 떨어집니다. 그래서 성능은 명령어 수, CPI/IPC, 클럭, [[용어 모음집/메모리 계층|메모리 계층]], 파이프라인 효율을 함께 봐야 합니다.

---

## 체크리스트
- [ ] 클럭과 성능을 같은 말로 보지 않는다
- [ ] CPI와 IPC 차이를 설명할 수 있다
- [ ] CPU 시간 공식을 이해적으로 안다
- [ ] 캐시 미스와 분기 예측이 성능에 미치는 영향을 안다
- [ ] latency와 throughput을 구분할 수 있다
---

## 보강 설명
CPU 주제는 “명령어가 실제로 어떻게 실행되는가”를 기준으로 연결해서 보는 것이 좋다. PC가 명령어 주소를 가리키고, [[용어 모음집/제어장치|제어장치]]가 명령어를 해석하며, [[용어 모음집/레지스터|레지스터]]와 [[용어 모음집/ALU|ALU]]가 값을 주고받고, 결과가 다시 레지스터나 메모리에 반영된다.

## 구체적인 예시
예를 들어 `a = b + c` 같은 코드는 메모리에서 값을 읽고, 레지스터에 올리고, ALU에서 더한 뒤, 결과를 다시 저장하는 여러 명령어로 바뀐다. 파이프라이닝이 있으면 여러 명령어의 단계가 겹쳐 실행되지만, 데이터 의존성이나 분기 때문에 멈출 수 있다.
