# 명령어 사이클

## 한 줄 정의
[[용어 모음집/명령어 사이클|명령어 사이클]]은 **[[용어 모음집/CPU|CPU]]가 명령어를 가져오고, 해석하고, 실행하고, 결과를 저장하는 반복 흐름**이다.

---

## 핵심 개념
- Fetch: 명령어를 가져온다
- Decode: 명령어를 해석한다
- Execute: 실제 연산이나 동작을 수행한다
- Memory Access: 필요하면 메모리를 읽거나 쓴다
- Write-back: 결과를 [[용어 모음집/레지스터|레지스터]]에 저장한다
- Interrupt Check: [[용어 모음집/인터럽트|인터럽트]]나 예외를 확인한다

---

## 전체 흐름

```text
Fetch
  ↓
Decode
  ↓
Execute
  ↓
Memory Access
  ↓
Write-back
  ↓
Interrupt Check
  ↓
다음 명령어
```

모든 명령어가 모든 단계를 똑같이 쓰는 것은 아니지만, 큰 흐름은 이렇게 이해하면 좋다.

---

## Fetch
Fetch는 [[용어 모음집/PC|PC]]가 가리키는 주소에서 명령어를 가져오는 단계다.

```text
PC → 명령어 메모리/캐시 → IR
```

이후 PC는 보통 다음 명령어 주소로 갱신된다. 단, 분기나 점프가 있으면 다른 주소로 바뀔 수 있다.

---

## Decode
Decode는 [[용어 모음집/IR|IR]]에 들어 있는 명령어를 해석하는 단계다.

- opcode 확인
- operand 위치 확인
- 필요한 레지스터 읽기
- 제어 신호 준비

---

## Execute
Execute는 실제 동작을 수행하는 단계다.

| 명령어 종류 | Execute에서 하는 일 |
|---|---|
| 산술/논리 | [[용어 모음집/ALU|ALU]] 연산 |
| load/store | 주소 계산 |
| branch | 조건 판단과 분기 대상 계산 |
| compare | 플래그 갱신 |

---

## Memory Access
메모리 접근이 필요한 명령어는 이 단계에서 [[용어 모음집/캐시|캐시]]나 메모리에 접근한다.

- load: 메모리에서 값을 읽음
- store: 메모리에 값을 씀

단순 ALU 명령어는 이 단계가 거의 필요 없을 수 있다.

---

## Write-back
Write-back은 연산이나 load 결과를 레지스터에 기록하는 단계다.

```text
ALU 결과 또는 메모리 읽기 결과 → 목적 레지스터
```

---

## 인터럽트와 예외
CPU는 명령어 사이클 중간 또는 명령어 경계에서 인터럽트/예외를 확인한다.

```text
현재 명령어 처리
   ↓
인터럽트 확인
   ↓
필요하면 현재 상태 저장
   ↓
인터럽트 핸들러 실행
   ↓
원래 흐름 복귀
```

---

## 단일 사이클, 멀티 사이클, 파이프라인

| 방식 | 쉽게 이해하기 |
|---|---|
| 단일 사이클 | 한 명령어를 한 긴 사이클에 처리 |
| 멀티 사이클 | 한 명령어를 여러 짧은 단계로 나눠 처리 |
| 파이프라인 | 여러 명령어의 단계를 겹쳐 처리 |

---

## 자주 하는 실수
- 모든 명령어가 같은 시간을 쓴다고 생각함
  - 바로잡기: 명령어마다 필요한 단계와 메모리 접근 여부가 달라 실행 시간과 사이클 수가 달라질 수 있다.
- Fetch와 Decode를 같은 단계로 봄
  - 바로잡기: Fetch는 명령어를 메모리/캐시에서 가져오는 단계이고, Decode는 가져온 명령어의 의미와 피연산자를 해석하는 단계다.
- Write-back이 항상 메모리에 쓰는 것이라고 생각함
  - 바로잡기: Write-back은 결과를 최종 목적지에 반영하는 단계이며, 많은 명령어에서는 메모리가 아니라 레지스터에 쓴다.
- 인터럽트가 명령어 실행 흐름과 무관하다고 생각함
  - 바로잡기: 인터럽트는 현재 명령어 흐름을 잠시 중단하고 핸들러로 제어를 넘긴 뒤, 저장된 상태를 바탕으로 복귀한다.
- 명령어 사이클과 [[용어 모음집/파이프라이닝|파이프라이닝]]을 같은 개념으로 봄
  - 바로잡기: 명령어 사이클은 한 명령어의 처리 흐름이고, 파이프라이닝은 여러 명령어의 단계들을 겹쳐 처리량을 높이는 기법이다.

---

## 면접 포인트
> 명령어 사이클은 CPU가 명령어를 처리하는 반복 흐름입니다. 보통 fetch에서 PC가 가리키는 명령어를 가져오고, decode에서 opcode와 operand를 해석한 뒤, execute에서 ALU 연산이나 주소 계산을 수행합니다. 필요하면 memory access 단계에서 메모리를 읽고 쓰며, write-back 단계에서 결과를 레지스터에 저장합니다. 인터럽트가 있으면 CPU 상태를 저장하고 핸들러로 이동할 수 있습니다.

---

## 체크리스트
- [ ] Fetch, Decode, Execute, Memory, Write-back을 설명할 수 있다
- [ ] PC와 IR이 명령어 사이클에서 어떻게 쓰이는지 안다
- [ ] load/store와 ALU 명령어의 단계 차이를 안다
- [ ] 인터럽트 처리와 명령어 사이클을 연결할 수 있다
- [ ] 명령어 사이클과 파이프라인 차이를 안다
---

## 보강 설명
CPU 주제는 “명령어가 실제로 어떻게 실행되는가”를 기준으로 연결해서 보는 것이 좋다. PC가 명령어 주소를 가리키고, [[용어 모음집/제어장치|제어장치]]가 명령어를 해석하며, 레지스터와 ALU가 값을 주고받고, 결과가 다시 레지스터나 메모리에 반영된다.

## 구체적인 예시
예를 들어 `a = b + c` 같은 코드는 메모리에서 값을 읽고, 레지스터에 올리고, ALU에서 더한 뒤, 결과를 다시 저장하는 여러 명령어로 바뀐다. 파이프라이닝이 있으면 여러 명령어의 단계가 겹쳐 실행되지만, 데이터 의존성이나 분기 때문에 멈출 수 있다.
