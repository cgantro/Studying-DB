# CPU

## 한 줄 정의
[[용어 모음집/CPU|CPU]](Central Processing Unit)는 **프로그램의 [[용어 모음집/명령어|명령어]]를 순서대로 처리해 계산, 제어, 데이터 이동을 수행하는 컴퓨터의 핵심 장치**다.

쉽게 말하면, 메모리에 있는 명령어를 하나씩 읽어서 **무슨 일을 해야 하는지 판단하고 실제로 실행하는 부품**이다.

---

## 왜 배우는가
- 프로그램은 결국 CPU가 실행하는 명령어들의 흐름이다.
- CPU를 이해하면 코드가 실제로 어떻게 실행되는지 보인다.
- [[용어 모음집/레지스터|레지스터]], [[용어 모음집/ALU|ALU]], [[용어 모음집/제어장치|제어장치]], [[용어 모음집/캐시|캐시]], 파이프라인이 모두 성능과 연결된다.
- [[용어 모음집/운영체제|운영체제]]의 [[용어 모음집/문맥 교환|문맥 교환]], [[용어 모음집/인터럽트|인터럽트]], [[용어 모음집/스케줄링|스케줄링]]도 CPU 상태 저장과 관련이 있다.

---

## CPU를 한 문장으로 다시 말하면
CPU는 **[[용어 모음집/명령어 사이클|명령어 사이클]]을 반복하면서 레지스터와 메모리의 값을 읽고, ALU로 계산하고, 제어장치로 실행 흐름을 조정하는 장치**다.

---

## 핵심 개념
- **명령어(instruction)**: CPU가 수행하는 최소 작업 지시
- **[[용어 모음집/PC|PC]](Program Counter)**: 다음에 가져올 명령어 주소
- **[[용어 모음집/IR|IR]](Instruction Register)**: 현재 해석 중인 명령어
- **레지스터(register)**: CPU 내부의 빠른 임시 저장 공간
- **ALU**: 산술/논리 연산 장치
- **제어장치**: 제어 신호를 만들어 각 부품을 움직이는 장치
- **[[용어 모음집/클럭|클럭]](clock)**: CPU 동작의 박자
- **파이프라인(pipeline)**: 명령어 처리 단계를 겹쳐 [[용어 모음집/처리량|처리량]]을 높이는 기법

---

## 전체 흐름

```text
PC가 다음 명령어 주소를 가리킴
   ↓
메모리/캐시에서 명령어를 가져옴
   ↓
IR에 명령어 저장
   ↓
제어장치가 명령어 해석
   ↓
레지스터에서 operand 준비
   ↓
ALU 또는 실행 유닛이 처리
   ↓
결과를 레지스터/메모리에 저장
   ↓
PC 갱신
```

---

## CPU 내부 부품 관계

```text
        제어장치
     신호 ↓   ↑ 상태
레지스터 → ALU → 결과
   ↑             ↓
   └── 캐시/메모리 ┘
```

---

## CPU와 메모리의 관계
- CPU는 매우 빠르다
- [[용어 모음집/RAM|RAM]]은 CPU보다 느리다
- 그래서 레지스터와 캐시가 중간에서 속도 차이를 줄인다
- 메모리 접근이 많으면 CPU가 기다리는 시간이 늘어난다

---

## CPU 성능을 볼 때
CPU 성능은 클럭만으로 결정되지 않는다.

| 요소                      | 의미                                      |
| ----------------------- | --------------------------------------- |
| 클럭                      | 1초에 몇 번 박자를 만들 수 있는가                    |
| [[용어 모음집/IPC\|IPC]]     | 한 클럭에 평균 몇 개 명령어를 처리하는가                 |
| 캐시 효율                   | 메모리 접근 [[용어 모음집/대기 시간\|대기시간]]을 얼마나 줄이는가 |
| [[용어 모음집/분기 예측\|분기 예측]] | 다음 실행 흐름을 얼마나 잘 맞히는가                    |
| 파이프라인 효율                | 단계가 멈추지 않고 잘 흐르는가                       |

---

## 운영체제와 연결
- 문맥 교환은 CPU 레지스터 상태를 저장하고 복원하는 일과 연결된다
- 인터럽트가 오면 CPU는 현재 흐름을 멈추고 [[용어 모음집/커널|커널]]의 처리 코드로 이동한다
- 스케줄러는 어떤 [[용어 모음집/프로세스|프로세스]]가 CPU를 사용할지 정한다
- [[용어 모음집/시스템 콜|시스템 콜]]은 [[용어 모음집/사용자 모드|사용자 모드]]에서 [[용어 모음집/커널 모드|커널 모드]]로 CPU 실행 권한이 바뀌는 흐름이다

---

## 자주 하는 실수
- CPU를 단순 계산기처럼만 생각함
  - 바로잡기: CPU는 계산뿐 아니라 명령어 fetch/decode, 제어 신호 생성, 메모리 접근, 분기 처리, 인터럽트 처리까지 수행한다.
- 클럭이 높으면 무조건 빠르다고 생각함
  - 바로잡기: 성능은 클럭, IPC, 캐시 적중률, 분기 예측, 메모리 대기 시간, 코어 구조가 함께 결정한다.
- 레지스터와 캐시를 같은 것으로 봄
  - 바로잡기: 레지스터는 ALU가 바로 사용할 값과 CPU 실행 상태를 담는 아주 작은 저장소이고, 캐시는 RAM 접근을 줄이기 위해 자주 쓰는 데이터/명령어를 저장하는 계층이다.
- 명령어 사이클과 파이프라인을 구분하지 못함
  - 바로잡기: 명령어 사이클은 한 명령어의 처리 흐름이고, 파이프라이닝은 여러 명령어의 단계들을 겹쳐 처리량을 높이는 기법이다.
- CPU 성능 병목을 항상 CPU 내부 문제로만 생각함
  - 바로잡기: CPU 내부뿐 아니라 캐시 미스, 메모리 대역폭, 디스크/네트워크 I/O, 락 경합도 전체 성능 병목이 될 수 있다.

## 면접 포인트
면접에서는 CPU를 **명령어 실행 흐름**으로 설명하는 것이 좋다.

### 꼭 말할 수 있어야 하는 것
- CPU는 명령어를 가져와 해석하고 실행한다
- 제어장치, ALU, 레지스터가 핵심 부품이다
- PC는 다음 명령어 주소를, IR은 현재 명령어를 담는다
- 성능은 클럭, IPC, 캐시, 파이프라인, 분기 예측의 영향을 받는다
- 운영체제의 문맥 교환은 CPU 상태 저장/복원과 연결된다

### 한 번에 말하는 답변 예시
> CPU는 메모리에 있는 명령어를 가져와 해석하고 실행하는 중앙 처리 장치입니다. 제어장치가 명령어를 해석해 제어 신호를 만들고, ALU가 산술/논리 연산을 수행하며, 레지스터가 중간값과 상태를 빠르게 저장합니다. CPU는 fetch, decode, execute, write-back 흐름을 반복하고, 성능은 클럭뿐 아니라 IPC, 캐시 효율, 파이프라인, 분기 예측에도 영향을 받습니다.

---

## 체크리스트
- [ ] CPU를 명령어 실행 흐름으로 설명할 수 있다
- [ ] 제어장치, ALU, 레지스터 역할을 구분할 수 있다
- [ ] PC와 IR의 역할을 안다
- [ ] CPU 성능을 클럭만으로 판단하지 않는다
- [ ] CPU와 운영체제 문맥 교환의 연결을 안다

## 보강 설명
CPU 주제는 “명령어가 실제로 어떻게 실행되는가”를 기준으로 연결해서 보는 것이 좋다. PC가 명령어 주소를 가리키고, 제어장치가 명령어를 해석하며, 레지스터와 ALU가 값을 주고받고, 결과가 다시 레지스터나 메모리에 반영된다.

## 구체적인 예시
예를 들어 `a = b + c` 같은 코드는 메모리에서 값을 읽고, 레지스터에 올리고, ALU에서 더한 뒤, 결과를 다시 저장하는 여러 명령어로 바뀐다. 파이프라이닝이 있으면 여러 명령어의 단계가 겹쳐 실행되지만, 데이터 의존성이나 분기 때문에 멈출 수 있다.
