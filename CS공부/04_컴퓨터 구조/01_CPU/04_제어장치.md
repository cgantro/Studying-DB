# 제어장치

## 한 줄 정의
[[용어 모음집/제어장치|제어장치]]는 **[[용어 모음집/명령어|명령어]]를 해석해 [[용어 모음집/CPU|CPU]] 내부 부품들이 언제 무엇을 해야 하는지 제어 신호를 만들어 내는 장치**다.

---

## 핵심 개념
- 명령어 디코딩을 수행한다
- [[용어 모음집/레지스터|레지스터]], [[용어 모음집/ALU|ALU]], 메모리, [[용어 모음집/PC|PC]]에 제어 신호를 보낸다
- CPU의 데이터 이동과 실행 순서를 지휘한다
- 구현 방식으로 하드와이어드 제어와 마이크로프로그램 제어가 있다

---

## 제어장치 흐름

```text
IR에 저장된 명령어
   ↓
제어장치가 opcode 해석
   ↓
필요한 제어 신호 생성
   ↓
레지스터/ALU/메모리/PC 동작
```

---

## 제어 신호 예시

| 신호 | 의미 |
|---|---|
| RegRead | 레지스터 값 읽기 |
| RegWrite | 레지스터에 결과 쓰기 |
| ALUOp | ALU 연산 종류 선택 |
| MemRead | 메모리 읽기 |
| MemWrite | 메모리 쓰기 |
| PCWrite | 다음 명령어 주소 갱신 |
| Branch | 조건 분기 처리 |

---

## 명령어별 제어 흐름 이해

### add 명령어
```text
레지스터 읽기 → ALU 덧셈 → 결과 레지스터 쓰기
```

### load 명령어
```text
주소 계산 → 메모리 읽기 → 레지스터 쓰기
```

### branch 명령어
```text
조건 비교 → 플래그 확인 → PC 변경 여부 결정
```

명령어마다 필요한 제어 신호 조합이 다르다.

---

## 하드와이어드 제어
하드와이어드 제어는 회로로 제어 로직을 직접 구성하는 방식이다.

### 장점
- 빠를 수 있다
- 단순한 명령어 집합에 잘 맞는다

### 단점
- 수정이 어렵다
- 명령어가 복잡하면 회로 설계가 어려워진다

---

## 마이크로프로그램 제어
마이크로프로그램 제어는 명령어 실행을 더 작은 마이크로 명령들의 흐름으로 제어하는 방식이다.

### 장점
- 복잡한 명령어를 다루기 쉽다
- 수정과 확장이 상대적으로 유연할 수 있다

### 단점
- 하드와이어드보다 느릴 수 있다
- 제어 저장소가 필요하다

---

## 비교표

| 항목 | 하드와이어드 | 마이크로프로그램 |
|---|---|---|
| 구현 | 회로 중심 | 마이크로 명령 중심 |
| 속도 | 빠른 편 | 상대적으로 느릴 수 있음 |
| 유연성 | 낮음 | 높음 |
| 어울리는 구조 | 단순 명령어 | 복잡한 명령어 |

---

## 자주 하는 실수
- 제어장치가 실제 산술 계산을 한다고 생각함
  - 바로잡기: 제어장치는 계산을 직접 하기보다 명령어를 해석하고 ALU, 레지스터, 메모리 접근을 제어하는 신호를 만든다.
- 제어 신호를 소프트웨어 명령과 같은 것으로 봄
  - 바로잡기: 제어 신호는 CPU 내부 하드웨어를 움직이는 전기적/논리적 신호이고, 소프트웨어 명령은 ISA가 정의한 실행 지시다.
- 하드와이어드가 항상 더 좋다고 생각함
  - 바로잡기: 하드와이어드 제어는 빠를 수 있지만 수정이 어렵고, 마이크로프로그램 방식은 유연하지만 해석 계층 비용이 생길 수 있다.
- 마이크로프로그램을 일반 프로그램과 완전히 같은 것으로 생각함
  - 바로잡기: 마이크로프로그램은 일반 애플리케이션이 아니라 CPU 내부에서 복잡한 명령을 더 작은 제어 단계로 풀기 위한 제어 저장 방식이다.
- 제어장치와 명령어 디코딩의 관계를 놓침
  - 바로잡기: 명령어 디코딩은 opcode와 operand를 해석해 어떤 제어 신호가 필요한지 결정하는 과정이다.

---

## 면접 포인트
> 제어장치는 명령어를 해석해 CPU 내부 부품에 필요한 제어 신호를 보내는 장치입니다. 예를 들어 add 명령어라면 레지스터를 읽고 ALU에 덧셈 신호를 보내고 결과를 레지스터에 쓰도록 지시합니다. 구현 방식에는 빠르지만 수정이 어려운 하드와이어드 제어와, 유연하지만 상대적으로 느릴 수 있는 마이크로프로그램 제어가 있습니다.

---

## 체크리스트
- [ ] 제어장치가 계산 장치가 아니라 지휘 장치임을 안다
- [ ] 제어 신호 예시를 말할 수 있다
- [ ] 명령어별로 필요한 제어 흐름이 다름을 안다
- [ ] 하드와이어드와 마이크로프로그램 방식을 비교할 수 있다
---

## 보강 설명
CPU 주제는 “명령어가 실제로 어떻게 실행되는가”를 기준으로 연결해서 보는 것이 좋다. PC가 명령어 주소를 가리키고, 제어장치가 명령어를 해석하며, 레지스터와 ALU가 값을 주고받고, 결과가 다시 레지스터나 메모리에 반영된다.

## 구체적인 예시
예를 들어 `a = b + c` 같은 코드는 메모리에서 값을 읽고, 레지스터에 올리고, ALU에서 더한 뒤, 결과를 다시 저장하는 여러 명령어로 바뀐다. [[용어 모음집/파이프라이닝|파이프라이닝]]이 있으면 여러 명령어의 단계가 겹쳐 실행되지만, 데이터 의존성이나 분기 때문에 멈출 수 있다.
