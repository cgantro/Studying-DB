# 문맥 교환과 동기화

## 한 줄 정의
[[용어 모음집/문맥 교환|문맥 교환]](context switching)은 **[[용어 모음집/CPU|CPU]]가 실행하던 작업의 상태를 저장하고 다른 작업의 상태를 복원하는 과정**이고, [[용어 모음집/동기화|동기화]]는 **여러 실행 흐름이 공유 자원을 안전하게 사용하도록 순서와 접근을 조절하는 것**이다.

## 문맥 교환 흐름
```text
현재 작업 중단
→ 레지스터, PC, SP 등을 PCB/TCB에 저장
→ 다음 작업 선택
→ 다음 작업의 PCB/TCB에서 상태 복원
→ 실행 재개
```

## 문맥 교환이 일어나는 경우
- 타이머 [[용어 모음집/인터럽트|인터럽트]]로 CPU 할당 시간이 끝남
- 실행 중인 작업이 I/O를 기다림
- 더 높은 우선순위 작업이 도착함
- [[용어 모음집/시스템 콜|시스템 콜]]이나 인터럽트 처리 후 [[용어 모음집/스케줄링|스케줄링]]이 필요함

## 비용
문맥 교환은 유용하지만 공짜가 아니다.

- [[용어 모음집/레지스터|레지스터]] 저장/복원 비용
- [[용어 모음집/커널 모드|커널 모드]] 전환 비용
- 스케줄러 실행 비용
- [[용어 모음집/캐시|캐시]]와 [[용어 모음집/TLB|TLB]] 적중률 저하 가능성

## 동기화가 필요한 이유
여러 [[용어 모음집/스레드|스레드]]가 같은 변수나 자료구조를 동시에 수정하면 결과가 깨질 수 있다.

예시:
```text
count = 10
Thread A: count를 읽음
Thread B: count를 읽음
Thread A: 11 저장
Thread B: 11 저장
```

두 번 증가해야 하는데 최종 값이 11이 될 수 있다. 이런 문제를 경쟁 상태(race condition)라고 한다.

## 동기화 도구
- [[용어 모음집/뮤텍스|뮤텍스]]: 한 번에 하나의 스레드만 [[용어 모음집/임계 구역|임계 구역]] 진입
- [[용어 모음집/세마포어|세마포어]]: 정해진 개수만큼 동시 접근 허용
- 조건 변수: 특정 조건이 만족될 때까지 대기
- 원자 연산: 중간에 끼어들 수 없는 단일 연산

## 자주 헷갈리는 점
- 문맥 교환은 CPU를 효율적으로 나눠 쓰게 하지만 너무 잦으면 오버헤드가 커진다.
- 동기화는 정확성을 위해 필요하지만 과하면 [[용어 모음집/병렬성|병렬성]]이 줄어든다.
- 락은 공유 자원 보호용이지 모든 성능 문제의 해결책은 아니다.

## 면접 포인트
문맥 교환은 현재 실행 상태를 [[용어 모음집/PCB|PCB]]/[[용어 모음집/TCB|TCB]]에 저장하고 다음 작업의 상태를 복원하는 과정이다. 동기화는 공유 자원 접근 순서를 제어해 race condition을 막는 것이며, 락 경합은 성능 저하를 만들 수 있다.

## 보강 설명
프로세스와 스레드 주제는 실행 단위와 자원 소유 단위를 구분하는 것이 중요하다. 프로세스는 주소 공간과 자원을 갖고, 스레드는 그 안에서 CPU를 받는 실행 흐름으로 동작한다.

## 구체적인 예시
예를 들어 브라우저는 여러 프로세스로 탭을 분리하고, 각 프로세스 안에서 렌더링이나 네트워크 작업을 여러 스레드로 나눌 수 있다. 문맥 교환 때는 CPU 상태가 저장되고 다른 실행 흐름의 상태가 복원된다.

## 헷갈릴 때 구분 기준
- 이름이 비슷한 용어는 “무엇을 관리하는가”, “언제 발생하는가”, “어떤 비용을 줄이는가”로 구분한다.
- 성능 관련 개념은 항상 병목과 오버헤드를 같이 본다.
- [[용어 모음집/운영체제|운영체제]] 개념은 사용자 프로그램 관점과 커널 내부 관점을 나누어 설명한다.

## 면접 답변에 붙이면 좋은 문장
이 개념은 단독으로 외우기보다 실제 실행 흐름에서 어떤 상태를 바꾸고 어떤 비용을 줄이거나 늘리는지까지 연결해서 설명하면 좋다.
