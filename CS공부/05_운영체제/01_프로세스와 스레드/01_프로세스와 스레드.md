# 프로세스와 스레드

## 한 줄 정의
[[용어 모음집/프로세스|프로세스]](process)는 **실행 중인 프로그램의 독립된 작업 단위**이고, [[용어 모음집/스레드|스레드]](thread)는 **그 프로세스 안에서 실제로 실행 흐름을 담당하는 더 작은 단위**다.

쉽게 말하면:
- 프로세스 = 작업실 하나
- 스레드 = 그 작업실 안에서 일하는 일꾼들

---

## 왜 배우는가
- [[용어 모음집/운영체제|운영체제]]의 핵심은 여러 작업을 동시에 돌리는 것이다.
- 프로세스와 스레드를 알면 멀티태스킹, [[용어 모음집/동시성|동시성]], 성능, 안전성을 이해할 수 있다.
- 서버가 왜 여러 프로세스/스레드를 쓰는지 설명할 수 있다.
- 브라우저, 게임, DB, 웹 서버 구조를 이해하는 데 중요하다.
- 면접에서 거의 기본으로 묻는 주제다.

---

## 초보자용 한 줄 기억법
- **프로세스 = 독립된 프로그램 실행 단위**
- **스레드 = 그 안에서 실제로 일하는 흐름**

---

## 프로세스와 스레드를 한 문장으로 다시 말하면
프로세스와 스레드는 **프로그램 실행을 여러 단위로 나누어, 안전한 격리와 효율적인 협업을 가능하게 하는 운영체제의 기본 개념**이다.

---

## 전체 흐름

```text
프로그램 실행
   ↓
프로세스 생성
   ↓
프로세스 안에 스레드 생성
   ↓
스레드들이 CPU를 번갈아 사용
   ↓
작업 완료
```

---

## 핵심 개념
- **프로세스**: 실행 중인 프로그램
- **스레드**: 프로세스 내부의 실행 흐름
- **주소 공간**: 프로세스가 사용하는 메모리 영역
- **[[용어 모음집/PCB|PCB]]**: 프로세스 제어 블록
- **[[용어 모음집/TCB|TCB]]**: 스레드 제어 블록
- **[[용어 모음집/문맥 교환|문맥 교환]](context switch)**: 실행 중인 단위를 바꾸는 과정
- **[[용어 모음집/동기화|동기화]]**: 스레드끼리 안전하게 공유 자원을 쓰게 하는 조정

---

## 프로세스란
프로세스는 **실행 중인 프로그램 하나를 독립적인 단위로 보는 것**이다.

### 특징
- 자기만의 주소 공간을 가진다
- 다른 프로세스와 메모리가 분리된다
- 안전성이 높다
- 생성과 전환 비용이 비교적 크다

### 예시
- 브라우저 탭
- 메모장 실행 창
- 서버의 worker process

---

## 스레드란
스레드는 **프로세스 안에서 실제로 [[용어 모음집/CPU|CPU]]를 받아 실행되는 흐름의 단위**다.

### 특징
- 같은 프로세스의 메모리를 공유한다
- 스택은 각 스레드가 따로 가진다
- 생성과 전환이 프로세스보다 가볍다
- 공유가 쉬운 대신 동기화가 필요하다

### 예시
- UI 처리 스레드
- 네트워크 처리 스레드
- 백그라운드 작업 스레드

---

## 프로세스 vs 스레드 비교표

| 항목 | 프로세스 | 스레드 |
|---|---|---|
| 정의 | 실행 중인 프로그램 | 프로세스 내부 실행 흐름 |
| 주소 공간 | 독립적 | 공유 |
| 메모리 보호 | 강함 | 약함(공유로 인한 위험) |
| 생성 비용 | 상대적으로 큼 | 상대적으로 작음 |
| 전환 비용 | 큼 | 작음 |
| 안정성 | 높음 | 공유 때문에 더 복잡 |
| 동기화 | 상대적으로 덜 복잡 | 매우 중요 |

---

## 메모리 관점 비교

### 프로세스
- 독립된 주소 공간
- 다른 프로세스와 직접 메모리를 공유하지 않음
- 충돌이 적다

### 스레드
- 힙과 전역 영역을 공유하는 경우가 많다
- 스택은 각자 따로 가진다
- 공유 때문에 빠르지만 위험도 있다

---

## 주소 공간 그림

```text
프로세스 A          프로세스 B
┌───────────┐       ┌───────────┐
│ code      │       │ code      │
│ data      │       │ data      │
│ heap      │       │ heap      │
│ stack     │       │ stack     │
└───────────┘       └───────────┘
```

### 쉽게 이해하기
- 프로세스끼리는 벽이 있다
- 스레드끼리는 같은 방을 같이 쓴다

---

## 스레드의 메모리 공유 그림

```text
같은 프로세스 안

[code]
[data]
[heap] ← 공유
[stack1] ← 스레드 A
[stack2] ← 스레드 B
```

### 중요한 점
- 공유 영역은 협업에 좋다
- 스택은 각자 따로라서 함수 호출 흐름은 분리된다

---

## PCB(Process Control Block)
PCB는 **프로세스의 상태 정보를 담는 자료구조**다.

### 들어갈 수 있는 정보
- 프로세스 ID
- 상태(Ready, Running, Waiting 등)
- 프로그램 카운터
- [[용어 모음집/레지스터|레지스터]] 값
- 우선순위
- 메모리 정보
- 열린 파일 정보

### 왜 필요한가
- 문맥 교환 시 복원해야 하기 때문
- 운영체제가 프로세스를 관리하려면 상태가 필요하기 때문

---

## TCB(Thread Control Block)
TCB는 **스레드의 상태 정보를 담는 자료구조**다.

### 들어갈 수 있는 정보
- 스레드 ID
- 상태
- 레지스터 정보
- 프로그램 카운터
- 스택 포인터
- [[용어 모음집/스케줄링|스케줄링]] 정보

### 쉽게 이해하기
- PCB가 프로세스의 신분증이라면
- TCB는 스레드의 신분증이다

---

## PCB vs TCB 비교표

| 항목 | PCB | TCB |
|---|---|---|
| 대상 | 프로세스 | 스레드 |
| 저장 정보 | 프로세스 상태, 자원 정보 | 스레드 상태, 실행 흐름 정보 |
| 역할 | 프로세스 관리 | 스레드 관리 |
| 연결 | 주소 공간/파일/자원 | 스택/레지스터/실행 흐름 |

---

## 문맥 교환(context switch)
문맥 교환은 **CPU가 현재 실행 중인 프로세스나 스레드의 상태를 저장하고, 다른 실행 단위의 상태를 복원하는 과정**이다.

### 왜 필요한가
- CPU는 한 번에 하나만 실행한다
- 여러 작업을 번갈아 실행해야 한다

### 저장할 것
- 레지스터
- 프로그램 카운터
- 상태 정보
- 스택 관련 정보

### 비용이 드는 이유
- 저장/복원 작업이 있다
- [[용어 모음집/캐시|캐시]] 친화성이 떨어질 수 있다
- 너무 자주 일어나면 오버헤드가 커진다

---

## 문맥 교환 흐름

```text
현재 작업 실행
   ↓
상태 저장
   ↓
다른 작업 상태 복원
   ↓
새 작업 실행
```

---

## 프로세스 문맥 교환 vs 스레드 문맥 교환

| 항목     | 프로세스 교환       | 스레드 교환          |
| ------ | ------------- | --------------- |
| 바뀌는 범위 | 더 큼           | 더 작음            |
| 주소 공간  | 바뀔 수 있음       | 같은 프로세스면 공유     |
| 비용     | 일반적으로 큼       | 일반적으로 작음        |
| 느낌     | 작업실을 통째로 갈아끼움 | 같은 작업실 안 일꾼만 교대 |

---

## 스레드를 쓰는 이유
- 같은 메모리를 공유하기 쉬움
- 작업을 동시에 진행하기 좋음
- 응답성을 높일 수 있음
- I/O 대기 동안 다른 일을 할 수 있음

### 예시
- UI는 멈추지 않고
- 백그라운드에서 파일 다운로드
- 네트워크 요청 처리

---

## 프로세스를 쓰는 이유
- 서로 격리되어 안전하다
- 한 프로세스가 죽어도 다른 프로세스에 영향이 덜하다
- 보안과 안정성이 중요할 때 좋다

### 예시
- 브라우저 탭 분리
- 서버 워커 분리
- 격리된 실행 환경

---

## 멀티프로세싱 vs 멀티스레딩

### 멀티프로세싱
- 여러 프로세스를 함께 사용
- 격리와 안정성에 유리
- 자원 사용이 더 클 수 있음

### 멀티스레딩
- 하나의 프로세스 안에 여러 스레드
- 공유와 효율에 유리
- 동기화가 중요

---

## 멀티프로세싱 vs 멀티스레딩 비교표

| 항목 | 멀티프로세싱 | 멀티스레딩 |
|---|---|---|
| 격리 | 강함 | 약함 |
| 공유 | 어렵다 | 쉽다 |
| 성능 | 프로세스 비용 있음 | 스레드가 가벼움 |
| 안정성 | 높음 | 동기화 실수에 취약 |
| 적합한 경우 | 독립성 중요 | 협업과 빠른 처리 중요 |

---

## 동기화와의 관계
스레드는 메모리를 공유하므로 동기화가 필요하다.

### 왜
- 동시에 같은 값을 바꾸면 race condition이 생긴다
- 락, [[용어 모음집/세마포어|세마포어]], 모니터 같은 도구가 필요하다

### 쉽게 이해하기
- 같은 칠판을 여러 사람이 동시에 쓰는 것과 비슷하다

---

## 프로세스 간 통신(IPC)
프로세스는 메모리를 공유하지 않기 때문에 **프로세스 간 통신**이 필요할 수 있다.

### 예시
- 파이프
- 소켓
- 공유 메모리
- 메시지 큐

### 쉽게 이해하기
- 프로세스끼리는 벽이 있으니 창구가 필요하다

---

## 스레드 간 통신
스레드는 같은 주소 공간을 공유하므로 통신이 쉽다.

### 하지만
- 쉬운 대신 위험하다
- 동기화 없으면 데이터가 꼬일 수 있다

---

## 프로세스 생성과 스레드 생성

### 프로세스 생성
- 상대적으로 무겁다
- 주소 공간 설정이 필요하다

### 스레드 생성
- 상대적으로 가볍다
- 같은 프로세스 자원을 재사용한다

### 기억법
- 프로세스 생성 = 새 집 마련
- 스레드 생성 = 같은 집에 사람 한 명 더 들이기

---

## 실전 예시

### 예시 1: 브라우저
- 탭을 프로세스로 분리해 안정성 확보
- 탭 내부에는 여러 스레드가 동작

### 예시 2: 웹 서버
- 요청마다 스레드를 쓰거나
- 프로세스 풀/스레드 풀을 쓸 수 있다

### 예시 3: 게임
- 렌더링, 입력, 네트워크, 물리 계산을 나누어 처리

---

## 자주 하는 실수
- 프로세스와 프로그램을 같은 말로 생각함
  - 바로잡기: 프로그램은 디스크에 있는 실행 파일이고, 프로세스는 그 프로그램이 메모리에 올라와 실행 중인 인스턴스다.
- 스레드가 메모리를 전부 따로 가진다고 생각함
  - 바로잡기: 스레드는 같은 프로세스의 코드/데이터/힙을 공유하지만 각자 스택과 레지스터 실행 상태를 가진다.
- 프로세스가 무조건 느리다고만 생각함
  - 바로잡기: 프로세스는 생성/전환 비용은 크지만 격리와 안정성이 좋아 상황에 따라 더 적절할 수 있다.
- 스레드가 많으면 무조건 빠르다고 생각함
  - 바로잡기: 스레드가 너무 많으면 문맥 교환, 락 경합, 메모리 사용량 때문에 오히려 느려질 수 있다.
- context switch를 단순 교대라고만 이해함
  - 바로잡기: 문맥 교환은 실행 대상을 바꾸는 것뿐 아니라 PC, SP, 범용 레지스터, 상태 레지스터 같은 CPU 상태를 저장하고 복원하는 작업이다.
- 스레드 안전과 프로세스 격리를 섞음
  - 바로잡기: 스레드 안전은 공유 메모리 동시 접근 문제이고, 프로세스 격리는 주소 공간 분리로 서로의 메모리를 보호하는 문제다.

---

## 디버깅할 때 보는 순서
1. 문제 단위가 프로세스인지 스레드인지 본다
2. 메모리가 공유되는지 확인한다
3. 동기화가 필요한 구간인지 본다
4. 문맥 교환이 너무 잦은지 본다
5. 프로세스 간 통신이 필요한지 확인한다
6. 충돌이 격리 부족 때문인지 본다
7. 성능 문제면 스레드 수와 큐 구조를 본다

---

## 비교해서 이해하기

### 프로그램 vs 프로세스
- 프로그램: 디스크에 있는 실행 파일
- 프로세스: 실행 중인 프로그램

### 프로세스 vs 스레드
- 프로세스: 독립된 작업 단위
- 스레드: 프로세스 내부 실행 흐름

### 멀티프로세싱 vs 멀티스레딩
- 멀티프로세싱: 분리와 안전성
- 멀티스레딩: 공유와 효율성

---

## ASCII 그림

### 프로세스와 스레드
```text
프로세스
 ├─ 스레드 A
 ├─ 스레드 B
 └─ 스레드 C
```

### 문맥 교환
```text
A 실행 → 상태 저장 → B 상태 복원 → B 실행
```

### 격리 vs 공유
```text
프로세스: 벽이 있음
스레드: 같은 방 공유
```

- [ ] 문맥 교환이 왜 비싼지 설명할 수 있다
- [ ] 프로세스와 스레드의 메모리 차이를 안다

## 면접 포인트
면접에서는 "프로세스와 스레드 차이"를 단순 암기보다 **메모리, 격리, 비용, 동기화**까지 묶어서 말해야 한다.

### 꼭 말할 수 있어야 하는 것
- 프로세스는 독립된 주소 공간을 가진 실행 단위다
- 스레드는 프로세스 내부의 실행 흐름이다
- 프로세스는 격리가 강하고 스레드는 공유가 쉽다
- 문맥 교환은 상태 저장/복원 비용이 들어간다
- 스레드는 동기화가 중요하고, 프로세스는 IPC가 중요하다

### 한 번에 말하는 답변 예시
> 프로세스는 실행 중인 프로그램의 독립된 단위이고, 스레드는 그 안에서 실제 실행 흐름을 담당하는 단위입니다. 프로세스는 주소 공간이 분리되어 안정성이 높지만 생성과 문맥 교환 비용이 크고, 스레드는 메모리를 공유해 가볍지만 동기화가 필요합니다. 그래서 프로세스는 격리와 안정성이 중요할 때, 스레드는 공유와 효율성이 중요할 때 주로 사용합니다.

### 꼬리 질문 예시
- PCB와 TCB는 뭐가 다른가요?
- 문맥 교환이 왜 비싼가요?
- 스레드가 많으면 무조건 빠른가요?
- 프로세스끼리는 어떻게 통신하나요?
- 스레드 안전은 왜 필요한가요?

## 한 번 더 요약
- 프로세스는 독립된 실행 단위다
- 스레드는 프로세스 내부 실행 흐름이다
- 프로세스는 격리, 스레드는 공유가 강점이다
- 문맥 교환은 비용이 든다
- 동기화와 IPC를 함께 이해해야 한다

## 초보자용 한 줄 기억법
프로세스와 스레드는 **"독립된 작업실과 그 안에서 일하는 일꾼들"**이다.

## 보강 설명
프로세스와 스레드 주제는 실행 단위와 자원 소유 단위를 구분하는 것이 중요하다. 프로세스는 주소 공간과 자원을 갖고, 스레드는 그 안에서 CPU를 받는 실행 흐름으로 동작한다.

## 구체적인 예시
예를 들어 브라우저는 여러 프로세스로 탭을 분리하고, 각 프로세스 안에서 렌더링이나 네트워크 작업을 여러 스레드로 나눌 수 있다. 문맥 교환 때는 CPU 상태가 저장되고 다른 실행 흐름의 상태가 복원된다.
