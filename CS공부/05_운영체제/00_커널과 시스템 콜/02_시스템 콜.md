# 시스템 콜

## 한 줄 정의
[[용어 모음집/시스템 콜|시스템 콜]]은 **사용자 프로그램이 [[용어 모음집/운영체제|운영체제]] [[용어 모음집/커널|커널]]의 기능을 요청하는 공식 인터페이스**다.

---

## 핵심
- 일반 함수 호출과 다르게 [[용어 모음집/커널 모드|커널 모드]]로 전환된다
- 파일, [[용어 모음집/프로세스|프로세스]], 메모리, 네트워크 작업에 사용된다
- 커널은 요청을 검증하고 권한을 확인한 뒤 처리한다
- 시스템 콜은 강력하지만 호출 비용이 있다

---

## 시스템 콜이 필요한 이유
사용자 프로그램은 하드웨어와 핵심 자원을 직접 다룰 수 없다.

```text
파일 열기
메모리 매핑
프로세스 생성
소켓 송수신
```

이런 작업은 커널의 보호와 조정이 필요하다.

---

## 시스템 콜 흐름

```text
사용자 코드
   ↓
라이브러리 함수 호출
   ↓
시스템 콜 번호와 인자 준비
   ↓
트랩 명령으로 커널 진입
   ↓
커널이 처리
   ↓
결과/에러 코드 반환
```

---

## 라이브러리 함수와 시스템 콜

| 구분 | 의미 |
|---|---|
| 라이브러리 함수 | 프로그래머가 호출하는 편한 API |
| 시스템 콜 | 커널에게 실제 요청하는 낮은 수준 인터페이스 |

예를 들어 `printf`는 내부적으로 버퍼링을 하다가 필요할 때 파일 쓰기 시스템 콜로 이어질 수 있다.

---

## 대표 분류

| 분류      | 예시 작업                        |     |
| ------- | ---------------------------- | --- |
| 프로세스 관리 | 생성, 종료, 대기                   |     |
| 파일 관리   | 열기, 읽기, 쓰기, 닫기               |     |
| 메모리 관리  | 매핑, 보호, 해제                   |     |
| 장치 관리   | 장치 읽기/쓰기                     |     |
| 통신      | 파이프, 소켓, [[용어 모음집/IPC\|IPC]] |     |
| 정보 조회   | 시간, 프로세스 정보                  |     |

---

## 자주 하는 실수
- 시스템 콜을 일반 함수 호출과 같은 비용으로 생각함
  - 바로잡기: 시스템 콜은 사용자 모드에서 커널 모드로 전환하고 권한 검사를 거치므로 일반 함수 호출보다 비용이 크다.
- 라이브러리 함수와 시스템 콜을 완전히 같은 것으로 봄
  - 바로잡기: 라이브러리 함수는 사용자 공간 코드일 수 있고, 그 내부에서 필요할 때만 시스템 콜을 호출한다.
- 시스템 콜이 권한 검사를 하지 않는다고 생각함
  - 바로잡기: 시스템 콜은 파일 권한, 메모리 접근, 프로세스 권한 같은 보호 규칙을 커널에서 검사한다.
- 파일 I/O가 모두 즉시 디스크 접근이라고 생각함
  - 바로잡기: 파일 I/O는 페이지 캐시나 버퍼 캐시를 거칠 수 있어 매 호출이 즉시 디스크 접근으로 이어지는 것은 아니다.

---

## 면접 포인트
> 시스템 콜은 사용자 프로그램이 커널 기능을 요청하는 인터페이스입니다. 일반 함수 호출과 달리 [[용어 모음집/사용자 모드|사용자 모드]]에서 커널 모드로 전환되며, 커널은 인자와 권한을 검사한 뒤 파일, 프로세스, 메모리 같은 자원을 처리합니다. 모드 전환 비용이 있으므로 시스템 콜은 일반 함수 호출보다 비싼 작업입니다.

## 보강 설명
커널과 시스템 콜 주제는 “사용자 프로그램이 왜 하드웨어를 직접 제어하지 못하고 운영체제를 거치는가”가 핵심이다. 보호와 안정성을 위해 권한을 나누고, 필요한 작업은 시스템 콜로 요청한다.

## 구체적인 예시
예를 들어 파일 읽기는 단순 함수 호출처럼 보여도 내부적으로는 사용자 모드에서 커널 모드로 전환되어 권한 검사, [[용어 모음집/파일 시스템|파일 시스템]] 처리, 디스크 I/O 요청이 진행된다.
