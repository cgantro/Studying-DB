# 파일 시스템

## 한 줄 정의
[[용어 모음집/파일 시스템|파일 시스템]](file system)은 **디스크 같은 저장 장치 위의 데이터를 파일과 [[용어 모음집/디렉터리|디렉터리]]라는 형태로 정리하고, 찾고, 읽고, 쓰고, 보호하는 규칙과 구조**다.

쉽게 말하면, 저장 장치에 있는 데이터를 그냥 아무 데나 두는 게 아니라 **"이름 붙이고, 폴더로 나누고, 권한을 붙여서" 관리하는 체계**다.

---

## 왜 배우는가
- 파일은 거의 모든 프로그램이 사용한다.
- 파일 시스템을 이해하면 저장/읽기/삭제가 왜 느릴 수 있는지 이해할 수 있다.
- 경로, 권한, 링크, [[용어 모음집/저널링|저널링]], [[용어 모음집/inode|inode]] 같은 개념을 설명할 수 있다.
- 서버 장애, 데이터 손상, 디스크 성능 문제를 이해하는 데 중요하다.
- 면접에서 inode, 디렉터리, 저널링은 자주 묻는 기본 주제다.

---

## 초보자용 한 줄 기억법
- **파일 시스템 = 디스크 위에 파일과 폴더를 정리하는 [[용어 모음집/운영체제|운영체제]]의 방식**

---

## 파일 시스템을 한 문장으로 다시 말하면
파일 시스템은 **저장 장치의 데이터를 계층적 이름 공간, 메타데이터, 접근 권한, 데이터 블록 관리를 통해 안전하고 효율적으로 다루는 구조**다.

---

## 전체 흐름

```text
프로그램이 파일 요청
   ↓
경로 탐색
   ↓
디렉터리 확인
   ↓
inode / 메타데이터 조회
   ↓
데이터 블록 위치 확인
   ↓
읽기 or 쓰기 수행
```

---

## 파일 시스템이 하는 일
- 파일 이름을 관리한다
- 폴더 구조를 관리한다
- 파일의 실제 데이터 위치를 관리한다
- 권한을 검사한다
- 파일 생성/삭제/수정/이동을 처리한다
- 장애가 나도 복구할 수 있게 도와준다

---

## 파일과 디렉터리

### 파일
- 데이터를 담는 단위
- 문서, 이미지, 실행 파일, 로그 등

### 디렉터리
- 파일과 다른 디렉터리를 담는 폴더
- 계층 구조를 만든다

### 쉽게 말하면
- 파일은 내용물
- 디렉터리는 서랍/상자

---

## 디렉터리 구조
파일 시스템은 보통 트리 구조로 보인다.

```text
/
├─ home/
│  └─ user/
│     ├─ docs/
│     └─ photo.png
├─ var/
│  └─ log/
└─ etc/
```

### 왜 좋은가
- 큰 저장 공간을 논리적으로 나눌 수 있다
- 이름 충돌을 줄일 수 있다
- 탐색이 쉬워진다

---

## 경로(path)
경로는 **파일이나 디렉터리의 위치를 나타내는 이름**이다.

### 절대 경로
- 루트(`/`)부터 시작
- 전체 위치를 정확히 표현

### 상대 경로
- 현재 위치를 기준으로 표현

### 예시
- `/home/user/doc.txt`
- `../images/a.png`

---

## 파일 시스템의 핵심 개념
- **inode**
- **메타데이터**
- **데이터 블록**
- **디렉터리 엔트리**
- **권한**
- **링크**
- **저널링**

---

## inode란
inode는 **파일의 이름을 제외한 정보를 담는 메타데이터 구조**다.

### inode에 들어갈 수 있는 정보
- 파일 크기
- 소유자
- 권한
- 생성/수정 시간
- 실제 데이터 블록 위치
- 링크 수

### 중요한 이해 포인트
- 파일 이름은 디렉터리에 있고
- 파일의 실체 정보는 inode에 있다

---

## inode와 파일 이름의 차이
### 파일 이름
- 사람이 보는 이름
- 디렉터리에 저장됨

### inode
- 파일의 실제 속성 정보
- 시스템이 관리하는 정보

### 비유
- 파일 이름 = 사람 이름표
- inode = 신분증/기록카드

---

## inode와 디렉터리 관계
디렉터리는 보통 **파일 이름 ↔ inode 번호**를 연결한다.

```text
파일 이름 ──> inode 번호 ──> 실제 메타데이터 + 데이터 블록 위치
```

### 쉽게 이해하기
- 디렉터리는 주소록
- inode는 상세 카드

---

## 메타데이터란
메타데이터는 **파일 내용이 아닌 파일에 대한 정보**다.

### 예시
- 크기
- 권한
- 소유자
- 생성 시간
- 수정 시간

### 왜 중요한가
- 파일을 읽지 않아도 파일 정보를 알 수 있다
- 정렬, 검색, 권한 검사에 쓰인다

---

## 데이터 블록
데이터 블록은 **실제 파일 내용이 저장되는 공간**이다.

### 예시
- 문서의 본문
- 이미지 데이터
- 실행 파일의 바이너리
- 로그 내용

### 쉽게 이해하기
- inode가 "어디에 있는지"를 알고
- 데이터 블록이 "내용"을 담는다

---

## 파일 읽기 흐름

```text
파일 이름 요청
   ↓
디렉터리에서 inode 찾기
   ↓
inode에서 블록 위치 확인
   ↓
디스크에서 데이터 읽기
   ↓
프로그램에 전달
```

---

## 파일 쓰기 흐름

```text
파일 수정 요청
   ↓
권한 검사
   ↓
빈 블록 또는 기존 블록 확인
   ↓
데이터 기록
   ↓
메타데이터 갱신
   ↓
필요하면 저널 반영
```

---

## 저널링(journaling)
저널링은 **파일 시스템 변경 내용을 먼저 기록해 두고, 장애가 나도 복구할 수 있게 하는 방식**이다.

### 왜 필요한가
- 저장 중 전원이 꺼질 수 있다
- 중간 상태가 꼬일 수 있다
- 파일 시스템 구조가 망가지는 것을 줄이기 위해

### 쉽게 이해하기
- 본 작업 전에 메모장에 "무슨 일을 할지" 적어 두는 것

### 장점
- 복구가 쉬워진다
- 장애 후 일관성을 유지하기 좋다

### 단점
- 쓰기 비용이 늘 수 있다
- 성능 오버헤드가 생길 수 있다

---

## 링크(link)
링크는 **하나의 파일을 다른 이름으로 참조하는 것**이다.

### 하드 링크
- 같은 inode를 가리키는 다른 이름
- 실제 데이터는 하나
- 삭제해도 링크 수가 남아 있으면 데이터가 유지될 수 있다

### 심볼릭 링크(soft link)
- 다른 파일의 경로를 가리키는 별도 파일
- 원본이 없어지면 끊어질 수 있다

---

## 하드 링크 vs 심볼릭 링크 비교표

| 항목 | [[용어 모음집/하드 링크|하드 링크]] | [[용어 모음집/심볼릭 링크|심볼릭 링크]] |
|---|---|---|
| 가리키는 대상 | 같은 inode | 파일 경로 |
| 원본 삭제 영향 | 덜 받음 | 원본 없으면 깨질 수 있음 |
| 다른 파일시스템 | 제한적 | 가능 |
| 쉽게 이해하기 | 같은 사람의 다른 이름 | 주소를 적은 메모 |

---

## 권한(permission)
파일 시스템은 **누가 어떤 파일을 읽고/쓰고/실행할 수 있는지**를 관리한다.

### 대표 권한
- read
- write
- execute

### 대상
- 소유자
- 그룹
- 기타 사용자

### 왜 중요한가
- 보안과 직결된다
- 실수로 중요한 파일을 바꾸는 일을 줄인다

---

## 권한 이해 그림

```text
소유자: rwx
그룹:    r-x
기타:    r--
```

### 쉽게 말하면
- 누구는 읽기만
- 누구는 수정까지
- 누구는 실행까지

---

## 파일 시스템의 성능 포인트
- 순차 읽기/쓰기
- 랜덤 읽기/쓰기
- [[용어 모음집/캐시|캐시]]
- 메타데이터 접근
- 저널링 오버헤드

### 중요한 이해 포인트
- 디스크는 [[용어 모음집/RAM|RAM]]보다 훨씬 느리다
- 자주 작은 파일을 읽으면 오버헤드가 커질 수 있다
- 캐시와 버퍼가 성능에 큰 영향을 준다

---

## 캐시와 버퍼 이해
### 캐시
- 자주 쓰는 데이터를 빨리 꺼내기 위한 임시 저장

### 버퍼
- 쓰기 작업을 바로 디스크에 하지 않고 모아 두는 공간

### 왜 필요한가
- 디스크 접근 횟수를 줄인다
- 성능을 높인다

---

## 파일 시스템과 단편화
파일 시스템도 저장 공간이 조각날 수 있다.

### 내부 단편화
- 할당된 블록 안에 남는 공간

### 외부 단편화
- 빈 공간은 있는데 연속된 큰 공간이 부족한 상태

### 영향
- 큰 파일 저장이 비효율적일 수 있다
- 성능 저하가 생길 수 있다

---

## 파일 시스템 구조 이해

```text
[디렉터리]
   ├─ 파일 이름 → inode
   └─ inode → 데이터 블록 위치
```

### 이렇게 나누는 이유
- 이름과 내용, 위치를 분리하면 관리가 쉬워진다
- 파일 이름이 바뀌어도 내용은 그대로 둘 수 있다

---

## 실전 예시

### 예시 1: 문서 저장
- 파일 이름으로 저장
- 메타데이터 기록
- 실제 본문은 데이터 블록에 저장

### 예시 2: 사진 파일 열기
- 경로 탐색
- inode 조회
- 데이터 블록 읽기

### 예시 3: 로그 파일
- 계속 append
- 커질수록 블록 할당이 늘어남
- 저널링이 장애 복구에 도움

---

## 자주 하는 실수
- inode와 파일 이름을 같은 것으로 생각함
  - 바로잡기: inode는 파일 메타데이터와 데이터 블록 위치를 담고, 파일 이름은 디렉터리 엔트리가 inode 번호에 매핑한다.
- 디렉터리가 파일 내용을 담는다고 생각함
  - 바로잡기: 디렉터리는 파일 내용 자체가 아니라 이름과 inode 번호의 매핑을 담는 특별한 파일이다.
- 심볼릭 링크와 하드 링크를 헷갈림
  - 바로잡기: 하드 링크는 같은 inode를 여러 이름이 가리키고, 심볼릭 링크는 다른 경로를 가리키는 별도 파일이다.
- 저널링이 백업과 같다고 생각함
  - 바로잡기: 저널링은 변경 기록으로 파일 시스템 일관성 복구를 돕는 기능이지, 과거 파일 내용을 보존하는 백업이 아니다.
- 파일 시스템 성능을 디스크 속도만으로 판단함
  - 바로잡기: 파일 시스템 성능은 디스크 속도뿐 아니라 캐시, 저널링, inode 탐색, 작은 파일 개수, 동기화 쓰기 여부에 영향을 받는다.
- 권한을 소유자 기준으로만 봄
  - 바로잡기: 권한은 소유자뿐 아니라 그룹과 기타 사용자에 대해 읽기/쓰기/실행 권한을 나누어 판단한다.

---

## 디버깅할 때 보는 순서
1. 경로가 맞는지 확인한다
2. 권한 문제가 있는지 본다
3. 링크가 끊어졌는지 본다
4. inode와 디렉터리 구조를 의심해 본다
5. 저널링 복구가 필요한 상태인지 본다
6. 디스크 공간 부족이나 단편화를 확인한다
7. 캐시/버퍼로 인해 보이는 차이인지 본다

## 파일 시스템과 다른 개념 비교

### 파일 시스템 vs 저장 장치
- 저장 장치: 물리적 공간
- 파일 시스템: 그 공간을 쓰는 규칙

### inode vs 디렉터리
- inode: 파일 정보 카드
- 디렉터리: 이름과 inode를 연결하는 표

### 하드 링크 vs 심볼릭 링크
- 하드 링크: 같은 실체의 다른 이름
- 심볼릭 링크: 경로를 가리키는 별칭

### 저널링 vs 백업
- 저널링: 장애 복구용 기록
- 백업: 원본 손실 대비 복사본

## ASCII 그림

### 디렉터리와 inode
```text
/home/user/doc.txt
        ↓
   디렉터리 엔트리
        ↓
     inode #123
        ↓
   데이터 블록들
```

### 링크
```text
fileA ── 같은 inode
fileB ── 같은 inode
```

### 심볼릭 링크
```text
link → /real/path/fileA
```

### 저널링
```text
변경 전 기록 → 실제 반영 → 완료
```

## 체크리스트
- [ ] 파일 시스템을 한 문장으로 설명할 수 있다
- [ ] inode와 파일 이름 차이를 안다
- [ ] 디렉터리와 메타데이터 역할을 설명할 수 있다
- [ ] 하드 링크와 심볼릭 링크 차이를 안다
- [ ] 저널링이 왜 필요한지 안다
- [ ] 권한(read/write/execute)을 설명할 수 있다
- [ ] 파일 시스템이 성능과 복구에 영향을 준다는 점을 안다

## 면접 포인트
면접에서는 inode를 단순 정의로 외우기보다 **디렉터리, 메타데이터, 데이터 블록의 관계**까지 말해야 한다.

### 꼭 말할 수 있어야 하는 것
- 파일 시스템은 디스크 위 데이터를 파일/디렉터리로 관리하는 구조다
- 디렉터리는 이름과 inode를 연결하고, inode는 메타데이터와 데이터 블록 위치를 담는다
- 저널링은 장애 후 복구를 돕는다
- 하드 링크는 같은 inode를 공유하고, 심볼릭 링크는 경로를 참조한다
- 권한은 보안과 직결된다

### 한 번에 말하는 답변 예시
> 파일 시스템은 저장 장치의 데이터를 파일과 디렉터리 형태로 관리하는 규칙과 구조입니다. 디렉터리는 파일 이름과 inode를 연결하고, inode는 파일의 메타데이터와 실제 데이터 블록 위치를 담습니다. 저널링은 쓰기 도중 장애가 나도 일관성을 복구하는 데 도움이 되고, 하드 링크와 심볼릭 링크는 파일을 참조하는 방식이 다릅니다.

### 꼬리 질문 예시
- inode에는 뭐가 들어가나요?
- 하드 링크와 심볼릭 링크 차이는 뭔가요?
- 저널링은 왜 필요한가요?
- 디렉터리는 파일 내용을 저장하나요?
- 권한은 어떻게 관리되나요?

## 보강 설명
파일 시스템 주제는 사람이 보는 파일 이름과 저장 장치의 실제 블록 배치를 연결하는 구조를 이해하는 것이 중요하다. 이름, inode, 데이터 블록, 권한, 복구 정보가 나뉘어 관리된다.

## 구체적인 예시
예를 들어 `/home/me/a.txt`를 열면 각 디렉터리에서 이름을 조회해 inode를 찾고, inode가 가리키는 데이터 블록을 읽는다. 저널링은 중간에 전원이 꺼져도 구조를 복구하기 쉽게 만든다.
