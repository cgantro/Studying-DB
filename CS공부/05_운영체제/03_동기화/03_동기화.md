# 동기화

## 한 줄 정의
[[용어 모음집/동기화|동기화]](synchronization)는 **여러 [[용어 모음집/스레드|스레드]]나 [[용어 모음집/프로세스|프로세스]]가 공유 자원에 접근할 때, 데이터가 꼬이지 않도록 접근 순서와 개수를 조절하는 것**이다.

쉽게 말하면, 여러 실행 흐름이 같은 것을 만질 때 **누가 언제 들어가고, 몇 명까지 들어가고, 어떤 조건이 될 때까지 기다릴지 정하는 규칙**이다.

---

## 왜 필요한가
프로그램은 겉으로 한 줄처럼 보여도 실제로는 여러 단계로 실행될 수 있다.

```text
count++
```

이 한 줄은 실제로는 대략 다음처럼 나뉠 수 있다.

```text
1. count 값을 읽음
2. 읽은 값에 1을 더함
3. 다시 count에 저장함
```

두 스레드가 동시에 이 과정을 실행하면 문제가 생긴다.

```text
count = 0

스레드 A: count 읽음 -> 0
스레드 B: count 읽음 -> 0
스레드 A: 1 저장
스레드 B: 1 저장

예상 결과: 2
실제 결과: 1
```

이처럼 실행 순서에 따라 결과가 달라지는 문제가 [[용어 모음집/동기화|동기화]]의 출발점이다.

---

## 핵심 용어 흐름

```text
공유 자원
   ↓
동시에 접근하면 문제 가능
   ↓
race condition
   ↓
임계 구역 식별
   ↓
뮤텍스/세마포어/조건 변수 등으로 보호
```

---

## 공유 자원
공유 자원은 여러 실행 흐름이 함께 접근할 수 있는 대상이다.

### 예시
- 전역 변수
- 공유 큐
- 공유 배열
- 파일
- 로그 버퍼
- DB 커넥션 풀
- 메모리 캐시
- 계좌 잔액
- 재고 수량

공유 자원이 모두 위험한 것은 아니다. **읽기만 한다면** 보통 문제가 적다. 문제가 되는 경우는 주로 **읽고 수정하고 다시 쓰는 작업**이 동시에 일어날 때다.

---

## Race Condition
race condition은 **실행 순서나 타이밍에 따라 결과가 달라지는 상황**이다.

### 예시: 재고 감소
```text
재고 = 1

사용자 A: 재고 확인 -> 1
사용자 B: 재고 확인 -> 1
사용자 A: 주문 성공, 재고 0 저장
사용자 B: 주문 성공, 재고 0 저장
```

결과적으로 재고는 1개였는데 주문은 2개가 성공할 수 있다.

### 핵심 원인
- 읽기와 쓰기 사이에 다른 실행 흐름이 끼어든다.
- 공유 자원 접근 순서가 보장되지 않는다.
- 실행 타이밍에 따라 결과가 달라진다.

---

## 임계 구역
[[용어 모음집/임계 구역|임계 구역]](critical section)은 **동시에 여러 실행 흐름이 들어가면 안 되는 코드 구간**이다.

### 예시
```text
lock 획득
재고 확인
재고 감소
주문 성공 처리
lock 해제
```

여기서 `재고 확인 -> 재고 감소 -> 주문 성공 처리`가 하나의 임계 구역이다. 이 구간은 중간에 다른 스레드가 끼어들면 안 된다.

### 중요한 점
임계 구역은 무조건 큰 코드 블록이 아니다. **공유 자원을 실제로 읽고 쓰는 최소 구간**이어야 한다.

```text
나쁜 방식:
함수 전체를 lock으로 감싼다.

좋은 방식:
공유 자원을 읽고 수정하는 부분만 짧게 lock으로 감싼다.
```

---

## 뮤텍스
[[용어 모음집/뮤텍스|뮤텍스]](mutex)는 **한 번에 하나의 실행 흐름만 임계 구역에 들어가게 하는 도구**다.

```text
mutex.lock()
임계 구역
mutex.unlock()
```

### 언제 쓰는가
- 공유 변수 수정
- 공유 큐 삽입/삭제
- 파일에 동시에 쓰기
- 계좌 잔액 변경
- 재고 감소

### 예시
```text
스레드 A: lock 획득 -> count 수정 -> unlock
스레드 B: lock 대기 -> A가 unlock한 뒤 count 수정
```

### 핵심
뮤텍스는 **상호 배제**가 목적이다. 즉, 한 구역에 한 번에 하나만 들어가게 한다.

---

## 세마포어
[[용어 모음집/세마포어|세마포어]](semaphore)는 **정해진 개수만큼의 실행 흐름만 통과시키는 카운터 기반 동기화 도구**다.

```text
semaphore.wait()
자원 사용
semaphore.signal()
```

카운터가 3이면 동시에 3개까지 들어갈 수 있고, 카운터가 1이면 하나만 들어갈 수 있다.

### 카운팅 세마포어
카운팅 세마포어는 여러 개의 동일한 자원을 관리할 때 쓴다.

```text
DB 커넥션 3개
세마포어 초기값 = 3

스레드 A 입장 -> 남은 허용 수 2
스레드 B 입장 -> 남은 허용 수 1
스레드 C 입장 -> 남은 허용 수 0
스레드 D 대기
```

### 이진 세마포어
이진 세마포어는 값이 0 또는 1인 세마포어다. 뮤텍스처럼 한 번에 하나만 통과시키는 데 쓸 수 있다.

단, 뮤텍스는 보통 **잠근 스레드가 해제한다**는 소유권 의미가 강하고, 세마포어는 **카운터와 신호 전달** 의미가 더 강하다.

---

## 세마포어도 임계 구역에 쓸 수 있는가
쓸 수 있다. 다만 어떤 임계 구역인지에 따라 다르다.

### 한 번에 하나만 들어가야 하는 임계 구역
```text
count = count + 1
```

이런 구간은 동시에 2개가 들어가면 값이 깨질 수 있다. 따라서 뮤텍스나 값이 1인 세마포어를 써야 한다.

```text
semaphore 초기값 = 1
```

### N개까지 들어가도 되는 제한 구역
```text
DB 커넥션 사용
```

DB 커넥션이 3개라면 동시에 3개까지는 안전하다. 이때는 값이 3인 세마포어를 쓴다.

```text
semaphore 초기값 = 3
```

정리하면, 세마포어가 임계 구역에 못 쓰이는 것이 아니다. **한 명만 들어가야 하는 임계 구역에는 값 1로 쓰고, N개까지 허용되는 구역에는 값 N으로 쓴다.**

---

## 뮤텍스와 세마포어 비교

| 구분 | 뮤텍스 | 세마포어 |
|---|---|---|
| 핵심 목적 | 상호 배제 | 허용 개수 제어, 신호 전달 |
| 동시에 통과 가능한 수 | 보통 1개 | N개 가능 |
| 소유권 | 잠근 실행 흐름이 푸는 의미가 강함 | 소유권 의미가 약할 수 있음 |
| 대표 사용 | 공유 변수, 임계 구역 보호 | 연결 풀, 버퍼 칸 수, 생산자-소비자 |
| 비유 | 열쇠 1개 | 빈자리 개수 카운터 |

---

## 조건 변수
조건 변수(condition variable)는 **어떤 조건이 만족될 때까지 기다리거나, 조건이 바뀌었음을 알려 주는 도구**다.

뮤텍스가 "한 명만 들어오게 하기"라면, 조건 변수는 "지금 들어가도 되는 상태인지 기다리기"에 가깝다.

### 예시
```text
버퍼가 비어 있음
소비자: 꺼낼 것이 없으므로 기다림

생산자: 버퍼에 데이터 넣음
생산자: 소비자를 깨움
```

### 주의
조건 변수는 보통 뮤텍스와 함께 사용한다.

```text
mutex 획득
조건 확인
조건이 아니면 wait
조건이 맞으면 작업
mutex 해제
```

## 생산자-소비자 문제와 연결
[[용어 모음집/생산자 소비자 문제|생산자-소비자 문제]]는 동기화의 대표 예시다.

여기서는 두 가지 문제가 동시에 나온다.

1. 버퍼 자료구조에 동시에 접근하면 안 된다.
2. 버퍼가 비었거나 가득 찼을 때는 기다려야 한다.

그래서 보통 다음 도구를 함께 쓴다.

| 문제 | 필요한 도구 |
|---|---|
| 버퍼 내부 자료구조 보호 | [[용어 모음집/뮤텍스|뮤텍스]] |
| 빈 칸 개수 관리 | [[용어 모음집/세마포어|세마포어]] |
| 찬 칸 개수 관리 | [[용어 모음집/세마포어|세마포어]] |
| 특정 조건까지 대기 | 조건 변수 또는 세마포어 |

---

## 자주 하는 실수
- 동기화를 "무조건 lock을 거는 것"으로만 이해함
  - 바로잡기: 동기화는 접근 순서와 조건을 조절하는 넓은 개념이고, 뮤텍스/세마포어/조건 변수/원자 연산은 그 도구다.
- 임계 구역을 너무 크게 잡음
  - 바로잡기: 공유 자원을 실제로 읽고 쓰는 최소 구간만 보호해야 병렬성이 줄어드는 것을 막을 수 있다.
- 뮤텍스와 세마포어를 "하나는 임계 구역용, 하나는 임계 구역에 못 쓰는 것"으로 이해함
  - 바로잡기: 세마포어도 값이 1이면 한 번에 하나만 통과시키는 데 쓸 수 있다. 다만 뮤텍스는 상호 배제, 세마포어는 개수 제어와 신호 전달에 더 초점이 있다.
- 세마포어 wait/signal 순서를 대충 둠
  - 바로잡기: 생산자-소비자 문제에서는 빈 칸을 먼저 확인한 뒤 버퍼 lock을 잡아야 데드락 위험을 줄일 수 있다.
- 조건 확인을 `if`로만 처리함
  - 바로잡기: 조건 변수에서는 깨어난 뒤에도 조건이 여전히 맞는지 다시 확인해야 하므로 보통 `while`로 조건을 검사한다.
- 동기화하면 항상 안전하고 성능도 좋아진다고 생각함
  - 바로잡기: 동기화는 정확성을 위해 필요하지만, 대기 시간과 문맥 교환 비용을 만들 수 있다.

---

## 면접 포인트
- 동기화는 공유 자원에 대한 동시 접근을 안전하게 조절하는 것이다.
- race condition은 실행 순서에 따라 결과가 달라지는 문제다.
- 임계 구역은 동시에 여러 실행 흐름이 들어가면 안 되는 코드 구간이다.
- 뮤텍스는 한 번에 하나만 들어가게 하는 상호 배제 도구다.
- 세마포어는 허용 가능한 동시 접근 개수를 세는 도구다.
- 생산자-소비자 문제에서는 버퍼 보호와 버퍼 상태 대기를 구분해야 한다.

### 답변 예시
> 동기화는 여러 스레드가 공유 자원에 동시에 접근할 때 race condition이 생기지 않도록 접근 순서와 조건을 제어하는 것입니다. 공유 변수 수정처럼 한 번에 하나만 들어가야 하는 구간은 뮤텍스로 보호하고, DB 커넥션 풀이나 bounded buffer처럼 N개까지 허용되는 자원은 세마포어로 개수를 관리할 수 있습니다. 생산자-소비자 문제에서는 버퍼 자료구조 자체는 뮤텍스로 보호하고, 빈 칸과 찬 칸의 개수는 세마포어로 관리합니다.
