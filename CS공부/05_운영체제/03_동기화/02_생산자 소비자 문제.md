# 생산자 소비자 문제

## 한 줄 정의
[[용어 모음집/생산자 소비자 문제|생산자-소비자 문제]]는 **생산자가 데이터를 만들고 소비자가 데이터를 가져가는 구조에서, 제한된 버퍼를 안전하게 공유하기 위한 대표 [[용어 모음집/동기화|동기화]] 문제**다.

핵심은 두 가지다.

1. 버퍼가 가득 차면 생산자는 기다려야 한다.
2. 버퍼가 비어 있으면 소비자는 기다려야 한다.
3. 버퍼 자료구조 자체는 동시에 수정되면 안 된다.

---

## 기본 구조

```text
생산자 Producer
   ↓ 데이터 넣기
[ 제한된 버퍼 bounded buffer ]
   ↓ 데이터 꺼내기
소비자 Consumer
```

### 생산자
생산자는 데이터를 만든 뒤 버퍼에 넣는다.

예시:
- 웹 요청을 큐에 넣는 스레드
- 로그 메시지를 로그 큐에 넣는 스레드
- 카메라 프레임을 버퍼에 넣는 스레드
- 다운로드한 데이터를 처리 큐에 넣는 스레드

### 소비자
소비자는 버퍼에서 데이터를 꺼내 처리한다.

예시:
- 큐에서 요청을 꺼내 처리하는 워커 스레드
- 로그 큐에서 메시지를 꺼내 파일에 쓰는 스레드
- 프레임을 꺼내 인코딩하는 스레드
- 작업 큐에서 job을 꺼내 실행하는 스레드

---

## 왜 문제가 생기는가
버퍼 크기가 무한하지 않기 때문이다.

```text
버퍼 크기 = 3

[ ][ ][ ]  빈 상태
[A][B][C]  가득 찬 상태
```

### 버퍼가 가득 찬 경우
```text
[A][B][C]

생산자: D를 넣고 싶음
문제: 넣을 칸이 없음
해결: 빈 칸이 생길 때까지 대기
```

### 버퍼가 비어 있는 경우
```text
[ ][ ][ ]

소비자: 데이터를 꺼내고 싶음
문제: 꺼낼 데이터가 없음
해결: 데이터가 들어올 때까지 대기
```

### 동시에 버퍼를 수정하는 경우
```text
생산자 A: tail 위치 확인
생산자 B: tail 위치 확인
생산자 A: 데이터 삽입
생산자 B: 같은 위치에 데이터 삽입
```

이 경우 버퍼 내부 상태가 깨질 수 있다. 그래서 버퍼 자체는 [[용어 모음집/임계 구역|임계 구역]]으로 보호해야 한다.

---

## 필요한 동기화 도구

생산자-소비자 문제는 보통 세 가지를 구분해서 생각한다.

| 역할 | 의미 | 보통 쓰는 도구 |
|---|---|---|
| 버퍼 보호 | 버퍼 자료구조를 한 번에 하나만 수정 | [[용어 모음집/뮤텍스|뮤텍스]] |
| 빈 칸 개수 | 생산자가 넣을 수 있는 자리 수 | [[용어 모음집/세마포어|세마포어]] `empty` |
| 찬 칸 개수 | 소비자가 꺼낼 수 있는 데이터 수 | [[용어 모음집/세마포어|세마포어]] `full` |

여기서 중요한 점은 **뮤텍스와 세마포어가 하는 일이 다르다**는 것이다.

```text
뮤텍스:
버퍼 내부 자료구조를 동시에 수정하지 못하게 함

empty 세마포어:
빈 칸이 몇 개 남았는지 셈

full 세마포어:
꺼낼 데이터가 몇 개 있는지 셈
```

---

## 초기값
버퍼 크기가 N이라면 보통 이렇게 시작한다.

```text
mutex = 1
empty = N
full = 0
```

### 의미
| 값 | 의미 |
|---|---|
| `mutex = 1` | 버퍼 수정은 한 번에 하나만 가능 |
| `empty = N` | 처음에는 빈 칸이 N개 |
| `full = 0` | 처음에는 꺼낼 데이터가 0개 |

예를 들어 버퍼 크기가 3이면:

```text
mutex = 1
empty = 3
full = 0
```

---

## 생산자 흐름

```text
생산자:
1. 데이터 생성
2. empty.wait()    빈 칸이 있을 때까지 기다림
3. mutex.lock()    버퍼에 혼자 접근
4. 버퍼에 데이터 삽입
5. mutex.unlock()  버퍼 접근 종료
6. full.signal()   찬 칸이 하나 늘었다고 알림
```

### 왜 empty를 먼저 기다리는가
버퍼가 가득 찼는데 mutex를 먼저 잡으면 문제가 생길 수 있다.

```text
나쁜 흐름:
생산자: mutex 획득
생산자: empty 대기

소비자: 데이터를 꺼내야 empty가 늘어남
소비자: 그런데 mutex를 못 얻어서 버퍼에 접근 못 함
```

이러면 생산자는 empty를 기다리고, 소비자는 mutex를 기다리는 상황이 될 수 있다. 그래서 보통 **자리가 있는지 먼저 확인하고, 그다음 버퍼 lock을 잡는다.**

---

## 소비자 흐름

```text
소비자:
1. full.wait()     꺼낼 데이터가 있을 때까지 기다림
2. mutex.lock()    버퍼에 혼자 접근
3. 버퍼에서 데이터 제거
4. mutex.unlock()  버퍼 접근 종료
5. empty.signal()  빈 칸이 하나 늘었다고 알림
6. 데이터 처리
```

### 왜 full을 먼저 기다리는가
버퍼가 비어 있는데 mutex를 잡고 기다리면 생산자가 데이터를 넣을 수 없다.

```text
나쁜 흐름:
소비자: mutex 획득
소비자: full 대기

생산자: 데이터를 넣어야 full이 늘어남
생산자: 그런데 mutex를 못 얻어서 버퍼에 접근 못 함
```

그래서 소비자는 **꺼낼 데이터가 있는지 먼저 확인하고, 그다음 버퍼 lock을 잡는다.**

---

## 전체 흐름 예시
버퍼 크기가 2라고 하자.

```text
초기 상태
buffer = [ ][ ]
empty = 2
full = 0
```

### 생산자 A가 데이터 X 삽입
```text
empty.wait()  -> empty = 1
mutex.lock()
buffer = [X][ ]
mutex.unlock()
full.signal() -> full = 1
```

### 생산자 B가 데이터 Y 삽입
```text
empty.wait()  -> empty = 0
mutex.lock()
buffer = [X][Y]
mutex.unlock()
full.signal() -> full = 2
```

### 생산자 C가 데이터 Z 삽입 시도
```text
empty.wait()
empty = 0이므로 대기
```

### 소비자가 하나 꺼냄
```text
full.wait()   -> full = 1
mutex.lock()
buffer = [ ][Y]
mutex.unlock()
empty.signal() -> empty = 1
```

이제 생산자 C가 다시 진행할 수 있다.

## 코드처럼 보기

### 생산자
```text
produce item

wait(empty)
wait(mutex)

insert item into buffer

signal(mutex)
signal(full)
```

### 소비자
```text
wait(full)
wait(mutex)

remove item from buffer

signal(mutex)
signal(empty)

consume item
```

여기서 `empty/full`은 개수를 세는 세마포어이고, `mutex`는 버퍼 내부를 보호하는 이진 세마포어 또는 뮤텍스라고 보면 된다.

---

## 조건 변수로 푸는 방식
세마포어 대신 뮤텍스와 조건 변수를 사용할 수도 있다.

### 생산자
```text
mutex.lock()

while buffer is full:
    notFull.wait(mutex)

insert item
notEmpty.signal()

mutex.unlock()
```

### 소비자
```text
mutex.lock()

while buffer is empty:
    notEmpty.wait(mutex)

remove item
notFull.signal()

mutex.unlock()
```

### 왜 while을 쓰는가
깨어났다고 해서 조건이 반드시 만족된 상태라고 확신하면 안 된다. 여러 소비자가 동시에 깨어났거나, 깨어난 뒤 다른 스레드가 먼저 데이터를 가져갔을 수 있다.

그래서 조건 변수에서는 보통 다음처럼 쓴다.

```text
while 조건이 만족되지 않음:
    wait
```

## 실무 예시

### 로그 처리
```text
여러 요청 스레드 -> 로그 큐 -> 로그 writer 스레드
```

- 요청 스레드는 로그 메시지를 만든다.
- 로그 writer는 큐에서 메시지를 꺼내 파일에 쓴다.
- 큐가 가득 차면 요청 스레드는 기다리거나 로그를 버리는 정책을 선택해야 한다.

### 웹 서버 작업 큐
```text
accept 스레드 -> 요청 큐 -> worker 스레드들
```

- accept 스레드는 들어온 요청을 큐에 넣는다.
- worker 스레드는 큐에서 요청을 꺼내 처리한다.
- 요청 큐가 비어 있으면 worker는 기다린다.

### 영상 처리
```text
프레임 읽기 -> 프레임 버퍼 -> 인코딩
```

- 프레임 reader는 영상을 읽어 버퍼에 넣는다.
- encoder는 버퍼에서 프레임을 꺼내 인코딩한다.
- reader가 너무 빠르면 버퍼가 가득 차고, encoder가 너무 빠르면 버퍼가 빈다.

---

## 자주 하는 실수
- 버퍼 보호와 개수 관리를 같은 문제로 봄
  - 바로잡기: 버퍼 자료구조 보호는 뮤텍스의 역할이고, 빈 칸/찬 칸 개수 관리는 세마포어의 역할이다.
- `mutex`를 먼저 잡고 `empty`나 `full`을 기다림
  - 바로잡기: 기다리는 동안 mutex를 잡고 있으면 상대편이 버퍼를 수정하지 못해 데드락이 생길 수 있다.
- `empty`와 `full` 초기값을 반대로 둠
  - 바로잡기: 처음 버퍼가 비어 있다면 `empty = 버퍼 크기`, `full = 0`이다.
- 생산자가 `full.signal()`을 빼먹음
  - 바로잡기: 데이터를 넣은 뒤에는 소비자가 깨어날 수 있도록 full을 증가시켜야 한다.
- 소비자가 `empty.signal()`을 빼먹음
  - 바로잡기: 데이터를 꺼낸 뒤에는 생산자가 들어올 수 있도록 empty를 증가시켜야 한다.
- 조건 변수를 `if`와 함께 사용함
  - 바로잡기: 깨어난 뒤에도 조건이 만족되는지 다시 확인해야 하므로 `while`로 검사하는 것이 안전하다.

---

## 면접 포인트
- 생산자-소비자 문제는 bounded buffer를 공유할 때 생기는 대표 동기화 문제다.
- 버퍼가 가득 차면 생산자는 기다려야 하고, 버퍼가 비면 소비자는 기다려야 한다.
- 버퍼 자료구조 접근은 뮤텍스로 보호한다.
- 빈 칸 수는 `empty`, 찬 칸 수는 `full` 세마포어로 관리한다.
- wait 순서가 중요하다. 보통 `empty/full`을 먼저 기다리고, 그다음 `mutex`를 잡는다.

### 답변 예시
> 생산자-소비자 문제는 생산자가 bounded buffer에 데이터를 넣고 소비자가 꺼낼 때 생기는 동기화 문제입니다. 버퍼가 가득 차면 생산자는 기다려야 하고, 버퍼가 비면 소비자는 기다려야 합니다. 보통 버퍼 자료구조 자체는 뮤텍스로 보호하고, 빈 칸 개수는 empty 세마포어, 찬 칸 개수는 full 세마포어로 관리합니다. 생산자는 empty를 기다린 뒤 mutex를 잡고 데이터를 넣고 full을 증가시키며, 소비자는 full을 기다린 뒤 mutex를 잡고 데이터를 꺼내고 empty를 증가시킵니다.
