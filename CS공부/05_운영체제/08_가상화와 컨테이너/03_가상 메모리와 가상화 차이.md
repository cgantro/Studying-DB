# 가상 메모리와 가상화 차이

## 한 줄 정의
[[용어 모음집/가상 메모리|가상 메모리]]는 **각 [[용어 모음집/프로세스|프로세스]]에게 자기만의 큰 메모리 주소 공간이 있는 것처럼 보이게 하는 메모리 관리 기법**이고, [[용어 모음집/가상화|가상화]]는 **하나의 물리 컴퓨터나 실행 환경을 여러 독립 환경처럼 보이게 하는 더 넓은 기술**이다.

짧게 말하면:

```text
가상 메모리 = 메모리 주소를 속인다
가상화 = 컴퓨터/OS/실행 환경을 속인다
```

---

## 먼저 구분하기

| 구분           | 가상 메모리                                                                                           | 가상화                                                                                   |
| ------------ | ------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------- |

| 대상           | 프로세스 하나하나                                                                                        | VM, 컨테이너, 실행 환경                                                                       |
| 목적           | 주소 공간 분리, 보호, 메모리 효율                                                                             | 격리, 배포, 여러 환경 실행                                                                      |
| 핵심 기술        | [[용어 모음집/페이징\|페이징]], [[용어 모음집/페이지 테이블\|페이지 테이블]], [[용어 모음집/TLB\|TLB]], [[용어 모음집/페이지 폴트\|페이지 폴트]] | [[용어 모음집/하이퍼바이저\|하이퍼바이저]], [[용어 모음집/namespace\|namespace]], [[용어 모음집/cgroup\|cgroup]] |

## 가상 메모리를 왜 쓰는가
프로그램 입장에서는 메모리를 이렇게 보고 싶다.

```text
내 코드
내 전역 변수
내 힙
내 스택
```

즉, 다른 프로그램이 어디에 올라와 있는지 신경 쓰지 않고 **나 혼자 메모리를 쓰는 것처럼** 실행되고 싶다.

하지만 실제 물리 메모리([[용어 모음집/RAM|RAM]])는 하나뿐이다.

```text
물리 RAM
[프로세스 A 일부][커널 일부][프로세스 B 일부][빈 공간][프로세스 A 다른 부분]
```

가상 메모리는 이 문제를 해결한다.

```text
프로세스 A가 보는 주소 공간
0x0000 ---------------- 0xFFFF

프로세스 B가 보는 주소 공간
0x0000 ---------------- 0xFFFF

실제 물리 메모리
여기저기 흩어져 있음
```

프로세스 A와 B가 같은 가상 주소를 쓰더라도, 운영체제는 각 프로세스의 [[용어 모음집/페이지 테이블|페이지 테이블]]을 이용해 서로 다른 물리 주소로 바꿔 준다.

---

## 주소가 바뀐다는 말
프로세스가 사용하는 주소는 보통 물리 주소가 아니라 **가상 주소**다.

```text
프로세스 코드:
arr[0] 읽기
   ↓
CPU가 가상 주소 0x1000 접근
   ↓
MMU가 페이지 테이블 확인
   ↓
물리 주소 0xA300으로 변환
   ↓
RAM 접근
```

흐름은 이렇게 볼 수 있다.

```text
가상 주소
   ↓
페이지 테이블
   ↓
물리 주소
   ↓
RAM
```

여기서 핵심은 **프로세스는 물리 주소를 직접 알 필요가 없다**는 점이다.

---

## 같은 주소인데 다른 메모리인 예시
두 프로세스가 있다고 하자.

```text
프로세스 A: 0x1000에 x 저장
프로세스 B: 0x1000에 y 저장
```

겉으로는 둘 다 `0x1000`을 쓴다. 하지만 실제 매핑은 다를 수 있다.

```text
프로세스 A의 0x1000 -> 물리 메모리 0xA000
프로세스 B의 0x1000 -> 물리 메모리 0xB000
```

그래서 A가 `0x1000`에 값을 써도 B의 `0x1000`은 바뀌지 않는다. 이게 프로세스 주소 공간 분리다.

## 가상 메모리의 장점

### 1. 프로세스 보호
프로세스 A가 프로세스 B의 메모리를 마음대로 건드리지 못한다.

```text
A의 주소 공간
B의 주소 공간
커널 주소 공간
```

운영체제는 페이지마다 읽기/쓰기/실행 권한을 둘 수 있다.

```text
코드 영역: 읽기 + 실행
데이터 영역: 읽기 + 쓰기
스택 영역: 읽기 + 쓰기
```

### 2. 프로그램 작성이 쉬워짐
프로그램은 "내가 실제 RAM의 어디에 올라가는지" 몰라도 된다.

```text
프로그램 입장:
나는 0번지부터 큰 연속 공간을 쓰는 것처럼 보임

운영체제 입장:
실제로는 물리 메모리 여기저기에 나눠 배치
```

### 3. 물리 메모리를 효율적으로 사용
프로그램 전체를 한 번에 RAM에 올리지 않아도 된다. 실제로 필요한 페이지만 올릴 수 있다. 이것이 [[용어 모음집/요구 페이징|요구 페이징]]이다.

```text
프로그램 전체: 1GB
지금 실제로 쓰는 부분: 50MB

처음부터 1GB 전체를 RAM에 올리지 않고,
필요한 페이지부터 올릴 수 있다.
```

### 4. 스왑 사용 가능
RAM이 부족하면 당장 덜 쓰는 페이지를 디스크의 [[용어 모음집/스왑|스왑]] 영역으로 내보낼 수 있다.

```text
RAM 부족
   ↓
덜 쓰는 페이지를 디스크로 이동
   ↓
필요할 때 다시 RAM으로 가져옴
```

단, 디스크는 RAM보다 훨씬 느리므로 스왑이 너무 많아지면 [[용어 모음집/스래싱|스래싱]]이 생길 수 있다.

## 페이지와 페이지 테이블
가상 메모리는 보통 메모리를 일정한 크기의 조각으로 나눠 관리한다.

```text
가상 메모리 조각 = 페이지
물리 메모리 조각 = 프레임
```

예를 들어 페이지 크기가 4KB라면:

```text
가상 페이지 0 -> 물리 프레임 12
가상 페이지 1 -> 물리 프레임 5
가상 페이지 2 -> 아직 RAM에 없음
```

이 매핑 정보를 담는 표가 페이지 테이블이다.

```text
프로세스 A 페이지 테이블
가상 페이지 | 물리 프레임 | 권한 | 현재 RAM에 있음?
0          | 12        | R-X | yes
1          | 5         | RW- | yes
2          | 없음      | RW- | no
```

---

## TLB는 왜 필요한가
매번 메모리에 접근할 때마다 페이지 테이블을 뒤지면 느리다. 그래서 CPU 안에는 최근 주소 변환 결과를 저장하는 [[용어 모음집/TLB|TLB]]가 있다.

```text
가상 주소 접근
   ↓
TLB에 변환 정보 있음?
   ├─ 있음: 바로 물리 주소 확인
   └─ 없음: 페이지 테이블 확인
```

TLB는 주소 변환용 캐시라고 이해하면 된다.

---

## 페이지 폴트는 무조건 에러인가
아니다. [[용어 모음집/페이지 폴트|페이지 폴트]]는 CPU가 접근한 가상 페이지가 현재 RAM에 없거나, 접근 권한에 문제가 있을 때 발생한다.

### 정상적인 페이지 폴트
요구 페이징에서는 아직 RAM에 올리지 않은 페이지에 처음 접근할 때 페이지 폴트가 날 수 있다.

```text
프로세스가 어떤 코드/데이터에 처음 접근
   ↓
해당 페이지가 RAM에 없음
   ↓
페이지 폴트
   ↓
운영체제가 디스크에서 페이지를 가져옴
   ↓
다시 실행
```

이 경우는 정상적인 지연 처리다.

### 비정상적인 페이지 폴트
없는 주소나 권한이 없는 주소에 접근하면 프로그램 오류가 된다.

```text
NULL 포인터 접근
읽기 전용 페이지에 쓰기
다른 프로세스 메모리 접근
```

이런 경우는 segmentation fault 같은 오류로 이어질 수 있다.

---

## 가상화는 무엇이 다른가
가상화는 메모리 주소 하나가 아니라 **실행 환경 전체**를 가짜로 보여주는 기술이다.

### VM
가상머신은 하드웨어 전체가 있는 것처럼 보이게 해서, 그 위에 별도 OS를 실행한다.

```text
물리 서버
   ↓
하이퍼바이저
   ├─ VM 1: Linux
   ├─ VM 2: Windows
   └─ VM 3: Linux
```

VM 안의 OS는 자신이 진짜 하드웨어 위에서 실행되는 것처럼 느낀다.

### 컨테이너
컨테이너는 별도 OS를 통째로 띄우는 것이 아니라, 같은 커널을 공유하면서 프로세스가 보는 환경을 분리한다.

```text
호스트 OS 커널
   ├─ 컨테이너 A: 자기 파일 시스템, 자기 프로세스 목록처럼 보임
   └─ 컨테이너 B: 자기 파일 시스템, 자기 프로세스 목록처럼 보임
```

컨테이너는 [[용어 모음집/namespace|namespace]]로 보이는 자원을 분리하고, [[용어 모음집/cgroup|cgroup]]으로 CPU/메모리 사용량을 제한한다.

---

## 둘이 같이 쓰이는 경우
가상 메모리와 가상화는 별개 개념이지만 실제 시스템에서는 함께 쓰인다.

### 일반 프로세스
```text
프로세스
   ↓
가상 메모리 사용
   ↓
물리 RAM 접근
```

### 컨테이너 안의 프로세스
```text
컨테이너 안 프로세스
   ↓
자기 프로세스 주소 공간 사용
   ↓
호스트 커널의 가상 메모리 관리
   ↓
물리 RAM 접근
```

컨테이너도 결국 호스트 커널 위의 프로세스이므로 가상 메모리를 사용한다.

### VM 안의 프로세스
```text
게스트 프로세스의 가상 주소
   ↓
게스트 OS의 물리 주소처럼 보이는 주소
   ↓
하이퍼바이저가 실제 물리 주소로 매핑
   ↓
물리 RAM
```

VM에서는 주소 변환이 한 단계 더 복잡해진다. 게스트 OS도 자기 나름의 가상 메모리를 관리하고, 하이퍼바이저도 게스트가 보는 물리 메모리를 실제 물리 메모리에 매핑해야 한다.

## 가장 헷갈리는 지점

### "가상 메모리도 가상화 아닌가?"
넓게 보면 가상 메모리도 추상화 또는 가상화의 한 종류처럼 볼 수 있다. 하지만 운영체제 공부에서는 보통 구분해서 부른다.

```text
가상 메모리:
프로세스의 주소 공간을 가짜로 보여주는 메모리 관리 기법

가상화:
VM/컨테이너처럼 실행 환경 전체를 분리해서 보여주는 기술
```

### "컨테이너마다 메모리가 따로 있나?"
물리 RAM이 컨테이너마다 따로 박혀 있는 것은 아니다. 컨테이너 안의 프로세스들은 호스트 커널의 가상 메모리 관리를 그대로 사용한다. 다만 cgroup으로 "이 컨테이너는 메모리를 얼마까지 쓸 수 있다" 같은 제한을 걸 수 있다.

### "프로세스가 보는 메모리 크기와 실제 RAM 크기는 같은가?"
같지 않을 수 있다. 프로세스는 큰 가상 주소 공간을 볼 수 있지만, 실제로 RAM에 올라온 페이지는 그중 일부일 수 있다.

## 비유

### 가상 메모리
각 학생에게 똑같이 1번부터 시작하는 사물함 번호표를 준 것과 비슷하다.

```text
학생 A의 1번 사물함 -> 실제 사물함 20번
학생 B의 1번 사물함 -> 실제 사물함 35번
```

학생들은 둘 다 "내 1번 사물함"이라고 생각하지만 실제 위치는 다르다.

### 가상화
하나의 건물을 여러 독립 사무실처럼 나눠 쓰게 만드는 것과 비슷하다.

```text
회사 A는 자기 사무실만 보임
회사 B는 자기 사무실만 보임
각 회사는 독립된 공간처럼 사용
```

## 자주 하는 실수
- 가상 메모리를 "가짜 RAM을 만들어서 RAM이 실제로 늘어나는 것"으로 이해함
  - 바로잡기: 가상 메모리는 주소 공간을 분리하고 매핑하는 기법이다. 스왑을 이용해 RAM보다 큰 프로그램을 실행할 수는 있지만, 실제 RAM 속도와 용량이 마법처럼 늘어나는 것은 아니다.
- 가상 주소와 물리 주소를 같은 것으로 생각함
  - 바로잡기: 프로세스가 쓰는 주소는 가상 주소이고, 실제 RAM 위치는 페이지 테이블을 통해 찾는 물리 주소다.
- 페이지 폴트를 무조건 오류라고 생각함
  - 바로잡기: 요구 페이징에서는 필요한 페이지를 RAM에 올리기 위해 정상적으로 페이지 폴트가 발생할 수 있다.
- 컨테이너가 VM처럼 별도 커널을 가진다고 생각함
  - 바로잡기: 컨테이너는 호스트 커널을 공유하고, namespace와 cgroup으로 실행 환경을 격리한다.
- 가상 메모리와 가상화를 같은 범위의 개념으로 생각함
  - 바로잡기: 가상 메모리는 프로세스 주소 공간 문제이고, 가상화는 실행 환경 격리 문제다.

---

## 면접 포인트
- 가상 메모리는 프로세스마다 독립적인 가상 주소 공간을 제공한다.
- 같은 가상 주소라도 프로세스마다 다른 물리 주소로 매핑될 수 있다.
- 페이지 테이블은 가상 주소를 물리 주소로 바꾸는 매핑 정보를 가진다.
- TLB는 주소 변환을 빠르게 하기 위한 캐시다.
- 페이지 폴트는 정상적인 요구 페이징에서도 발생할 수 있다.
- 가상화는 VM/컨테이너처럼 실행 환경 전체를 격리하거나 추상화하는 기술이다.

### 답변 예시
> 가상 메모리는 각 프로세스가 자기만의 독립된 주소 공간을 가진 것처럼 보이게 하고, 실제로는 페이지 테이블을 통해 가상 주소를 물리 주소로 변환하는 메모리 관리 기법입니다. 반면 가상화는 하나의 물리 시스템 위에서 여러 VM이나 컨테이너처럼 독립 실행 환경을 만드는 더 넓은 기술입니다. 둘 다 실제 자원을 다르게 보이게 한다는 공통점은 있지만, 가상 메모리는 프로세스의 주소 공간을 다루고 가상화는 실행 환경 전체를 다룬다는 점이 다릅니다.
