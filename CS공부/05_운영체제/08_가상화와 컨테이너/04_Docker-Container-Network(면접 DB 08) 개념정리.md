## Docker 기본

### 개념
- Docker는 “이미지(Image)로 애플리케이션과 실행 환경을 패키징하고, 컨테이너(Container)로 실행”하는 플랫폼이다.
- 이미지: 실행 파일/라이브러리/설정 등을 포함한 불변 패키지(레이어 기반).
- 컨테이너: 이미지의 인스턴스(실행 중인 프로세스 + 격리된 환경).
- 컨테이너는 VM이 아니라, 커널을 공유하면서 namespace/cgroup으로 격리되는 구조로 이해하는 게 핵심이다.

### 왜 필요한가
- “로컬/CI/서버 환경 차이”를 줄여 재현성을 높인다.
- 배포 단위를 표준화해, 롤백/스케일/이식성을 높인다.
- 팀 협업에서 “실행 방법을 코드(Dockerfile/Compose)로 문서화”하는 효과가 있다.

### 동작 원리
- 레이어:
  - Dockerfile의 각 단계가 레이어로 쌓이고, 변경된 레이어만 다시 빌드/전송할 수 있다.
  - 빌드 캐시를 잘 활용하면 CI에서 시간이 크게 줄 수 있다(단, 수치 단정은 금지).
- 실행:
  - 컨테이너는 결국 “호스트 커널 위에서 도는 프로세스”이고, PID/네트워크/마운트가 격리된다.
- 저장:
  - 컨테이너의 쓰기 레이어는 휘발성일 수 있어, 데이터는 볼륨(Volume) 등으로 분리하는 게 일반적이다.

### 장단점 / 한계
- 장점: 재현성, 배포 표준화, 격리, 이식성
- 한계: 커널 공유(커널 취약점 영향), 네트워크/스토리지/권한 모델을 이해해야 운영이 안정적이다.

### 대안 / 선택 기준
- VM은 커널까지 격리되지만 오버헤드가 크다.
- 컨테이너는 가볍지만 격리 경계가 VM보다 얇을 수 있어 보안 정책이 필요하다(최소 권한, 이미지 스캔, 루트리스 등).

### 실무/디버깅 포인트
- “Docker 자체가 느리다/안 된다”보다,
  - 이미지 빌드(캐시/레이어)
  - 네트워크(포트/브리지/DNS)
  - 볼륨/권한(마운트/UID/GID)
  - 리소스 제한(cgroup)
중 어디인지 먼저 분리하는 게 중요하다.

### 프로젝트 연결
- 배포 자동화에서 Docker 이미지 빌드/푸시/실행은 가장 흔한 기본 흐름이다.
  - 관련: [[프로젝트/Autowing-Car]]
  - 관련: [[프로젝트/스티커]]

## Container 실행 / 상태 / 메모리

### 개념
- 컨테이너 상태는 “프로세스 상태”와 연결된다.
- 자원 제한은 cgroup 기반이며, 메모리/CPU 제한을 걸 수 있다.
- JVM 같은 런타임은 컨테이너 메모리 제한을 잘못 해석하면 OOM/GC 문제를 만들 수 있다(자세한 옵션은 환경별로 달라 `확인 필요`).

### 왜 필요한가
- 운영에서 흔한 장애:
  - 컨테이너가 바로 종료됨(entrypoint/command/환경변수/권한)
  - OOMKilled(메모리 제한 초과)
  - CPU 스로틀링(과도한 CPU 제한/경합)
- 컨테이너는 “프로세스 격리”이기 때문에, 프로세스가 죽으면 컨테이너도 끝난다.

### 동작 원리
- 실행:
  - `ENTRYPOINT`/`CMD` 조합으로 실행 명령이 결정된다.
  - foreground 프로세스가 종료되면 컨테이너도 종료되는 모델이므로, 데몬을 백그라운드로 돌리는 식은 설계가 꼬이기 쉽다.
- 로그:
  - 컨테이너 표준 출력/에러를 로그로 모으는 방식이 일반적이다.
  - 파일 로그를 컨테이너 내부에만 남기면, 컨테이너 재시작/교체 시 사라질 수 있다(볼륨/로그 수집 설계 필요).
- 메모리:
  - 제한을 걸면 커널이 OOM 정책에 따라 프로세스를 죽일 수 있다.
  - 애플리케이션은 자신이 가질 수 있는 메모리를 “정확히 인지하고 제한에 맞게 설정”해야 한다.

### 장단점 / 한계
- 컨테이너 격리는 운영을 단순화하지만, “호스트 커널” 문제가 전체에 영향을 줄 수 있다.
- 리소스 제한은 안정성을 주지만, 너무 타이트하면 성능/지연 문제를 만든다.

### 대안 / 선택 기준
- 리소스 제한 설정은 “서비스 특성 + 부하 패턴”을 근거로 잡아야 한다.
- 메모리 사용이 큰 서비스는:
  - 힙/캐시/버퍼 정책을 명확히 하고,
  - 관측 지표를 붙인 뒤,
  - 단계적으로 제한을 조정하는 게 안전하다.

### 실무/디버깅 포인트
- 컨테이너가 죽으면:
  - 종료 코드/이벤트(OOMKilled 여부)/로그를 먼저 본다.
  - healthcheck/재시작 정책이 “원인을 숨기고 무한 재시작”을 만들지 않는지 점검한다.

### 프로젝트 연결
- 배포 형태(Docker/JAR)에 따라 리소스/파일 경로가 달라지는 문제는 실제로 자주 발생한다.
  - 관련: [[프로젝트/Autowing-Car/04-맵-리소스-로딩-Docker-JAR]]

관련 질문: [[CS공부/12_면접/전체 질문 DB/08-Docker-Container-Network-질문#Container 실행 / 상태 / 메모리]]

## Volume / Network

### 개념
- 볼륨은 “데이터 지속성”을 위해 컨테이너의 수명과 데이터를 분리하는 장치다.
- 네트워크는 컨테이너 간 통신/포트 노출을 위한 가상 네트워크 계층이다.
- `ports`는 호스트↔컨테이너 포트 매핑이고, 같은 네트워크 내부 통신과는 다른 문제다.

### 왜 필요한가
- DB/파일/업로드 데이터 등을 컨테이너 내부에만 두면 컨테이너 교체 시 데이터가 사라진다.
- 네트워크를 모르면 “컨테이너끼리 통신이 안 된다”, “localhost가 다르게 동작한다” 같은 문제를 해결하기 어렵다.

### 동작 원리
- Volume vs bind mount:
  - bind mount: 호스트 디렉토리를 그대로 컨테이너에 마운트(개발에 편하지만 권한/경로 의존).
  - volume: Docker가 관리하는 저장소 영역(이식성/관리성에서 유리).
- 네트워크:
  - bridge 네트워크에서 컨테이너는 가상 인터페이스/IP를 갖고, NAT/포워딩으로 외부와 통신한다.
  - 컨테이너 내부의 `localhost`는 “그 컨테이너 자신”이다. 호스트를 가리키지 않는다.

### 장단점 / 한계
- bind mount는 개발 편의가 좋지만, 운영에서 환경 의존/권한 문제를 만들 수 있다.
- 네트워크는 단순해 보이지만, 방화벽/보안그룹/프록시/Nginx까지 얽히면 원인 추적이 어려워진다.

### 대안 / 선택 기준
- 운영에서는 데이터는 volume, 설정은 환경변수/시크릿, 로그는 표준 출력 기반 수집을 기본으로 두는 편이 단순하다.
- 네트워크 설계는 “외부 노출은 최소”, “내부 통신은 네트워크로 격리”를 기본으로 한다.

### 실무/디버깅 포인트
- 통신 문제는 계층별로 쪼갠다.
  - DNS(이름) → TCP 연결(포트/라우팅) → TLS(인증서) → HTTP(상태 코드)
- 컨테이너가 호스트 서비스에 접근해야 하면 별도 방법이 필요하다(환경별로 다름, `확인 필요`).

### 프로젝트 연결
- 컨테이너 네트워크를 공용 네트워크로 묶지 않으면 Redis/Backend 간 통신이 깨질 수 있다는 이슈가 기록되어 있다.
  - 관련: [[프로젝트/Autowing-Car/02-CI-CD-배포-이슈]]

관련 질문: [[CS공부/12_면접/전체 질문 DB/08-Docker-Container-Network-질문#Volume / Network]]

## Docker Compose / 운영

### 개념
- Docker Compose는 여러 컨테이너(서비스)를 “한 번에” 정의/실행하기 위한 도구다.
- 운영에서는 Compose 파일이 사실상 “로컬/스테이징/프로덕션 환경 정의” 역할을 한다.

### 왜 필요한가
- DB/캐시/백엔드/프록시를 함께 띄우는 구성은 수작업이 번거롭고 실수가 많다.
- Compose는 서비스 간 네트워크/볼륨/환경변수 등을 코드로 관리해 재현성을 높인다.

### 동작 원리
- `ports` vs `expose`:
  - `ports`: 호스트에 포트를 공개(외부 접근)
  - `expose`: 컨테이너 네트워크 내부에 포트를 “문서화/노출”(외부 공개는 아님)
- 환경변수/시크릿:
  - 런타임 설정은 이미지에 박지 않고 환경으로 주입하는 편이 안전하다.
  - 민감정보는 로그/이미지에 남지 않게 주의해야 한다.

### 장단점 / 한계
- Compose는 단일/소규모 운영에 유용하지만, 대규모 오케스트레이션(Kubernetes 등)과는 운영 모델이 다르다.
- Compose 파일이 커지면 환경별 분기/재사용이 어려워질 수 있다(구조화 필요).

### 대안 / 선택 기준
- 단일 호스트 운영/개발 환경이면 Compose가 효율적이다.
- 멀티 노드/자동 확장/자가 치유가 필요하면 오케스트레이터 도입을 검토한다.

### 실무/디버깅 포인트
- “DB는 떴는데 앱이 연결 실패”는 보통:
  - 네트워크 분리, 잘못된 호스트명/포트, 초기화 순서(ready 전에 접속), 인증 정보 문제
를 의심한다.
- 의존 서비스 readiness를 고려해 재시도/헬스체크를 설계한다.

### 프로젝트 연결
- 배포에서 런타임 변수를 `deploy.env`로 주입하는 패턴이 정리되어 있다.
  - 관련: [[프로젝트/Autowing-Car/02-CI-CD-배포-이슈]]

관련 질문: [[CS공부/12_면접/전체 질문 DB/08-Docker-Container-Network-질문#Docker Compose / 운영]]

## Docker 배포 / 트러블슈팅

### 개념
- 배포는 “이미지 빌드 → 레지스트리 푸시 → 서버에서 pull/run(또는 compose up)” 흐름으로 이해하면 된다.
- 트러블슈팅은 “빌드/전송/실행/통신/리소스” 중 어디에서 깨졌는지 분리하는 게 핵심이다.

### 왜 필요한가
- 배포 실패는 대부분 코드보다 “환경/네트워크/권한/설정” 때문이다.
- 재현 가능한 배포를 만들려면 빌드 산출물과 런타임 변수를 분리해야 한다.

### 동작 원리
- 이미지 태깅:
  - `latest` 같은 모호한 태그보다, 커밋 SHA/버전 태그로 불변 배포 단위를 만드는 편이 운영에서 안전하다.
- 롤백:
  - 이전 이미지 태그로 되돌릴 수 있어야 한다.
  - 데이터 스키마 변경이 있으면 롤백이 어려워질 수 있어, 마이그레이션 전략을 따로 둔다.

### 장단점 / 한계
- 컨테이너 배포는 빠르지만, 관측(로그/메트릭/헬스)이 없으면 장애 시 원인 추적이 느려진다.
- 호스트 1대에 많은 컨테이너를 올리면 리소스 경합/네트워크 복잡도가 커질 수 있다.

### 대안 / 선택 기준
- 배포 전략(단계적):
  - 단일 인스턴스면 단순 재시작이 가능하지만, 무중단/점진 배포가 필요하면 프록시/로드밸런서/헬스체크 기반으로 설계한다.

### 실무/디버깅 포인트
- 로그/상태:
  - 컨테이너 로그, 프로세스 종료 코드, 포트 바인딩 충돌 여부부터 확인한다.
- 네트워크:
  - “도메인은 맞는데 접속이 안 됨”은 DNS/방화벽/Nginx/업스트림을 순서대로 본다(11/10번 개념정리와 연결).

### 프로젝트 연결
- Docker/JAR 환경에서 리소스 로딩이 깨진 사례가 정리되어 있다.
  - [[프로젝트/Autowing-Car/04-맵-리소스-로딩-Docker-JAR]]

## 보안 / CI 연동

### 개념
- 컨테이너 보안은 “이미지/런타임/네트워크/시크릿” 축으로 본다.
- CI 연동은 “빌드/테스트/스캔/푸시”를 자동화하고, 배포까지 이어질 수 있다.

### 왜 필요한가
- 컨테이너는 배포가 쉬운 만큼, 취약한 이미지/시크릿 노출이 빠르게 확산될 수 있다.
- CI에서 외부 패키지 다운로드/레지스트리 접근이 실패하면 빌드가 흔들린다(네트워크/인증 문제).

### 동작 원리
- 보안 기본:
  - 최소 권한(루트 실행 회피), 불필요한 패키지 제거, 이미지 스캔, 시크릿 분리
- CI에서 흔한 문제:
  - 레지스트리 로그인/권한, 네트워크(DNS/방화벽), Docker-in-Docker 설정

### 장단점 / 한계
- 보안 강화를 하면 편의가 떨어질 수 있다(권한/디버깅 제한).
- CI 안정성을 높이려면 캐시/재시도/미러/아티팩트 보관 같은 운영 설계가 필요하다.

### 대안 / 선택 기준
- CI runner가 어디에 있는지(깃헙 호스티드 vs 셀프 호스트)와 네트워크 제약이 선택 기준이 된다.
- 시크릿은 환경변수로 주입하되, 로그/아티팩트에 노출되지 않게 규칙을 둔다.

### 실무/디버깅 포인트
- “CI에서만 실패”는 대개 네트워크/DNS/권한/리소스 부족이 원인이다.
- 서비스 컨테이너(DB 등)는 “준비 완료” 전에 테스트가 시작되면 실패하므로 readiness를 확보한다.

## Docker Compose DNS / Network

### 개념
- Compose에서 서비스 이름으로 통신이 되는 이유는, 같은 네트워크에 붙은 컨테이너들 사이에 DNS 기반 서비스 디스커버리가 제공되기 때문이다.
- “같은 네트워크가 아니면 이름으로 접근이 안 되는 이유”는 DNS가 네트워크 스코프 내에서 동작하기 때문이다.

### 왜 필요한가
- 마이크로 서비스/멀티 컨테이너 환경에서 IP는 바뀔 수 있으므로, 이름 기반 접근이 안정적이다.
- DNS 문제는 “앱은 정상인데 연결이 안 된다”로 나타나며, 원인 분리가 어렵다.

### 동작 원리
- service name 해석:
  - 동일 Compose 프로젝트/네트워크 내에서 `backend` 같은 서비스 이름은 해당 컨테이너 IP로 해석된다.
  - 컨테이너 재시작으로 IP가 바뀌어도 이름은 그대로여서 운영이 단순해진다.
- `localhost`:
  - 컨테이너 내부에서 `localhost`는 자기 자신이므로, 다른 서비스(예: DB)에 접근하려면 서비스 이름/컨테이너 IP를 써야 한다.
- DNS 장애 패턴:
  - 클라이언트(내 PC)에서는 되는데 컨테이너에서만 실패하면 `/etc/resolv.conf`/네트워크 설정/내부 DNS를 의심한다.

### 장단점 / 한계
- 이름 기반 접근은 편하지만, 네트워크를 잘못 나누면 “절대 연결 안 됨”이 된다.
- DNS 캐시/전파 문제까지 얽히면 트러블슈팅이 더 어려워질 수 있다.

### 대안 / 선택 기준
- 네트워크 설계는 “외부 노출”과 “내부 통신”을 분리하고, 내부는 가능한 한 같은 네트워크로 묶어 단순화한다.
- 프록시(Nginx/Traefik) 도입 시, 서비스 이름 기반 upstream과 네트워크 연결이 맞는지부터 점검한다.

### 실무/디버깅 포인트
- “DB는 떴는데 앱이 연결 실패”에서 우선 보는 것:
  - 같은 네트워크인지, 서비스 이름이 맞는지, 포트가 맞는지, DB readiness가 끝났는지
- DNS 확인은 컨테이너 내부에서 직접 해야 한다(호스트에서 되는지는 별개일 수 있음).
