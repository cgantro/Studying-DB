# 블로킹 논블로킹 동기 비동기 I/O

## 한 줄 정의
블로킹/논블로킹은 **호출한 쪽이 기다리는지 여부**이고, 동기/비동기는 **작업 완료를 누가 어떤 방식으로 확인하는지**에 가깝다.

---

## 핵심
- 블로킹은 결과가 준비될 때까지 기다린다
- 논블로킹은 준비되지 않았으면 즉시 돌아온다
- 동기는 호출 흐름에서 완료를 확인한다
- 비동기는 완료 알림, 콜백, 이벤트 등으로 나중에 처리한다
- 네 개념은 서로 섞일 수 있어 구분해서 말해야 한다

---

## 블로킹 I/O

```text
read 호출
   ↓
데이터 올 때까지 대기
   ↓
결과 반환
```

구현은 단순하지만, 기다리는 동안 해당 [[용어 모음집/스레드|스레드]]는 다른 일을 못 할 수 있다.

---

## 논블로킹 I/O

```text
read 호출
   ↓
데이터 있음? ── 예 → 반환
        │
        아니오
        ↓
즉시 실패/재시도 신호 반환
```

호출자는 나중에 다시 확인하거나 이벤트 감시와 함께 사용한다.

---

## 동기와 비동기 이해

| 구분 | 쉽게 이해하기 |
|---|---|
| 동기 | 내가 완료 여부를 직접 확인하며 흐름을 이어감 |
| 비동기 | 작업을 맡겨 두고 완료 알림을 받음 |

---

## 네 개념 비교

| 구분 | 질문 |
|---|---|
| 블로킹/논블로킹 | 호출한 쪽이 기다리는가? |
| 동기/비동기 | 완료 결과를 어떤 흐름으로 받는가? |

---

## 실전 예시

### 블로킹 동기
- 파일 읽기를 호출하고 끝날 때까지 기다림

### 논블로킹 동기
- 소켓을 논블로킹으로 두고 반복 확인

### 이벤트 기반 I/O
- 여러 소켓을 감시하다가 준비된 것만 처리

### 비동기 I/O
- 작업을 요청하고 완료 이벤트/콜백으로 결과 처리

---

## 자주 하는 실수
- 논블로킹과 비동기를 같은 뜻으로 씀
  - 바로잡기: 논블로킹은 호출이 바로 반환되는지의 문제이고, 비동기는 완료 결과를 나중에 통지받는 실행 모델이다.
- 블로킹이면 무조건 나쁜 방식이라고 생각함
  - 바로잡기: 블로킹은 단순하고 흐름을 이해하기 쉬워 작업 수가 적거나 별도 스레드로 분리된 경우 적절할 수 있다.
- 비동기면 무조건 빠르다고 생각함
  - 바로잡기: 비동기는 대기 시간을 잘 활용할 수 있지만 작업 자체가 빨라지는 것은 아니며 콜백/이벤트 관리 복잡도가 생긴다.
- 이벤트 루프가 [[용어 모음집/CPU|CPU]] 작업까지 자동으로 병렬 처리한다고 착각함
  - 바로잡기: 이벤트 루프는 I/O 대기를 효율적으로 다루지만 CPU 계산을 자동으로 여러 코어에 병렬화하지는 않는다.

---

## 면접 포인트
> 블로킹과 논블로킹은 호출한 스레드가 기다리는지의 차이이고, 동기와 비동기는 완료 결과를 어떤 흐름으로 받는지의 차이입니다. 블로킹 I/O는 단순하지만 대기 비용이 있고, 논블로킹 I/O는 즉시 돌아와 여러 작업을 다루기 좋습니다. 비동기 I/O는 완료를 나중에 알림으로 받아 처리합니다.

## 보강 설명
입출력 주제는 CPU가 느린 장치와 어떻게 협력하는지 보는 영역이다. CPU가 계속 기다릴지, 장치가 [[용어 모음집/인터럽트|인터럽트]]로 알릴지, [[용어 모음집/DMA|DMA]]로 데이터 이동을 맡길지에 따라 효율이 달라진다.

## 구체적인 예시
예를 들어 디스크에서 큰 파일을 읽을 때 CPU가 바이트마다 직접 옮기면 비효율적이다. DMA를 사용하면 장치가 메모리로 직접 데이터를 옮기고, 완료되면 인터럽트로 CPU에게 알린다.
