# 스케줄링

## 한 줄 정의
[[용어 모음집/스케줄링|스케줄링]](scheduling)은 **[[용어 모음집/CPU|CPU]]라는 한정된 자원을 어떤 [[용어 모음집/프로세스|프로세스]]나 [[용어 모음집/스레드|스레드]]에게 어떤 순서로 줄지 결정하는 정책**이다.

쉽게 말하면, 여러 작업이 줄 서 있을 때 **"누가 먼저 CPU를 쓸지"를 정하는 줄 세우기 규칙**이다.

---

## 왜 배우는가
- CPU는 동시에 모든 작업에게 줄 수 없다.
- 어떤 작업을 먼저 실행하느냐에 따라 체감 속도가 달라진다.
- 서버, 데스크톱, 모바일, 실시간 시스템은 요구사항이 다르다.
- 스케줄링은 [[용어 모음집/응답 시간|응답 시간]], [[용어 모음집/대기 시간|대기 시간]], [[용어 모음집/처리량|처리량]], 공정성, 기아 문제와 연결된다.
- 면접에서 자주 나오는 [[용어 모음집/운영체제|운영체제]] 기본 개념이다.

---

## 초보자용 한 줄 기억법
- **스케줄링 = CPU를 누구에게 얼마나 빨리 줄지 정하는 것**

---

## 스케줄링을 한 문장으로 다시 말하면
스케줄링은 **여러 실행 흐름 사이에서 CPU 사용 순서를 정해 시스템 전체의 성능과 공정성을 조절하는 운영체제의 핵심 기능**이다.

---

## 전체 흐름

```text
여러 프로세스/스레드 준비
        ↓
대기열(ready queue)
        ↓
스케줄러가 선택
        ↓
CPU 할당
        ↓
실행 종료 or 대기 or 선점
        ↓
다시 대기열로
```

---

## 스케줄링이 필요한 이유
CPU는 매우 빠르지만 한 번에 한 작업만 할 수 있다.

### 그래서 생기는 문제
- 누가 먼저 실행될까?
- 오래 걸리는 작업이 있으면 짧은 작업은 얼마나 기다릴까?
- 중요한 작업을 먼저 처리해야 할까?
- 모든 작업을 공평하게 처리해야 할까?

스케줄링은 이 질문들에 대한 답이다.

---

## 핵심 개념
- **CPU burst**: CPU를 실제로 사용하는 구간
- **I/O burst**: 입출력 작업을 기다리는 구간
- **Ready queue**: CPU를 기다리는 작업 목록
- **Dispatcher**: 실제로 CPU를 넘겨주는 역할
- **Context switch**: 현재 실행 중인 작업을 바꾸는 것

---

## CPU burst와 I/O burst
프로세스는 보통 CPU만 계속 쓰는 게 아니라 CPU와 I/O를 번갈아 사용한다.

### 예시
- CPU 계산 → 디스크 읽기 대기 → CPU 계산 → 네트워크 대기

### 쉽게 이해하기
- 어떤 작업은 CPU를 오래 쓰고
- 어떤 작업은 잠깐 쓰고 자주 기다린다

이 차이를 이용하면 스케줄링을 더 잘할 수 있다.

---

## 스케줄링의 분류
- **선점형(preemptive)**
- **비선점형(non-preemptive)**

---

## 선점형 스케줄링
선점형은 **실행 중인 작업을 중간에 빼앗아 다른 작업에게 CPU를 줄 수 있는 방식**이다.

### 장점
- 응답성이 좋다
- 중요한 작업을 빨리 처리할 수 있다
- 인터랙티브 환경에 유리하다

### 단점
- context switch 비용이 생긴다
- 구현이 복잡하다
- 공정성 문제를 잘 설계해야 한다

### 예시
- 라운드 로빈
- 선점형 [[용어 모음집/우선순위 스케줄링|우선순위 스케줄링]]

---

## 비선점형 스케줄링
비선점형은 **한 번 CPU를 받으면 작업이 끝나거나 스스로 대기할 때까지 뺏지 않는 방식**이다.

### 장점
- 단순하다
- 구현이 쉬운 편이다
- context switch가 적을 수 있다

### 단점
- 긴 작업이 CPU를 오래 잡을 수 있다
- 응답성이 떨어질 수 있다
- 짧은 작업이 오래 기다릴 수 있다

### 예시
- [[용어 모음집/FCFS|FCFS]]
- 일부 비선점형 [[용어 모음집/SJF|SJF]]

---

## 선점형 vs 비선점형 비교표

| 항목 | 선점형 | 비선점형 |
|---|---|---|
| CPU 중간 회수 | 가능 | 불가능 |
| 응답성 | 좋음 | 상대적으로 떨어짐 |
| 구현 복잡도 | 높음 | 낮음 |
| context switch | 많아질 수 있음 | 적을 수 있음 |
| 대표 예시 | RR, 선점형 Priority | FCFS, 비선점형 SJF |

---

## 스케줄링 목표
스케줄링은 단순히 빨리 돌리는 게 아니다. 여러 목표가 있다.

- **응답 시간(Response time)** 줄이기
- **대기 시간(Waiting time)** 줄이기
- **턴어라운드 타임(Turnaround time)** 줄이기
- **처리량(Throughput)** 높이기
- **공정성(Fairness)** 확보하기
- **기아([[용어 모음집/Starvation|Starvation]])** 방지하기

---

## 주요 지표

### 응답 시간
- 요청하고 처음 반응이 올 때까지의 시간
- 사용자 체감에 중요하다

### 대기 시간
- 실행되기 전에 기다린 총 시간
- 스케줄링 효율을 볼 때 중요하다

### 턴어라운드 타임
- 작업이 들어와서 끝날 때까지 걸린 전체 시간

### 처리량
- 단위 시간당 끝난 작업 수

---

## 지표 비교표

| 지표 | 의미 | 중요할 때 |
|---|---|---|
| 응답 시간 | 처음 반응까지의 시간 | UI, 인터랙티브 |
| 대기 시간 | 실행 전 기다린 시간 | 배치 작업 최적화 |
| 턴어라운드 타임 | 시작부터 종료까지 | 전체 완료 시간 평가 |
| 처리량 | 시간당 완료 작업 수 | 서버, 배치 시스템 |

---

## 대표 알고리즘
- **FCFS(First Come First Served)**
- **SJF(Shortest Job First)**
- **SRTF(Shortest Remaining Time First)**
- **RR([[용어 모음집/Round Robin|Round Robin]])**
- **Priority Scheduling**
- **[[용어 모음집/MLFQ|MLFQ]](Multi-Level Feedback Queue)**
- **[[용어 모음집/CFS|CFS]] / [[용어 모음집/EEVDF|EEVDF]]**
- **실시간 정책: [[용어 모음집/SCHED_FIFO|SCHED_FIFO]], [[용어 모음집/SCHED_RR|SCHED_RR]], [[용어 모음집/SCHED_DEADLINE|SCHED_DEADLINE]]**

---

## FCFS
FCFS는 **먼저 온 작업부터 먼저 처리하는 방식**이다.

### 장점
- 매우 단순하다
- 공정해 보인다
- 이해하기 쉽다

### 단점
- 평균 대기 시간이 길어질 수 있다
- 긴 작업이 앞에 있으면 뒤 작업이 막힌다
- convoy effect가 생길 수 있다

### convoy effect
- 긴 작업 하나가 앞에 있으면 전체가 줄줄이 밀리는 현상

### 비유 
- 은행 창구에서 먼저 온 사람부터 처리

---

## SJF
SJF는 **실행 시간이 가장 짧은 작업을 먼저 처리하는 방식**이다.

### 장점
- 평균 대기 시간을 줄이는 데 유리하다
- 짧은 작업을 빨리 끝낼 수 있다

### 단점
- 미래의 실행 시간을 알아야 한다는 문제가 있다
- 긴 작업이 계속 밀릴 수 있다
- 기아 상태가 생길 수 있다

### 쉽게 이해하기
- 짧은 일부터 먼저 끝내서 전체 평균을 낮추는 전략

---

## RR(Round Robin)
RR은 **각 작업에 일정한 시간 조각(time quantum)을 나눠 주고, 시간이 끝나면 다음 작업으로 넘기는 방식**이다.

### 장점
- 응답성이 좋다
- 공정성이 좋다
- 인터랙티브 환경에 적합하다

### 단점
- quantum이 너무 작으면 context switch가 많아진다
- quantum이 너무 크면 FCFS처럼 된다

### 쉽게 이해하기
- 돌아가며 차례대로 조금씩 CPU를 쓰는 방식

---

## Priority Scheduling
Priority Scheduling은 **우선순위가 높은 작업을 먼저 처리하는 방식**이다.

### 장점
- 중요한 작업을 빨리 처리할 수 있다
- 시스템 정책을 반영하기 좋다

### 단점
- 낮은 우선순위 작업이 기아 상태에 빠질 수 있다
- 우선순위 조정이 어려울 수 있다

---

## SRTF
SRTF(Shortest Remaining Time First)는 **남은 실행 시간이 가장 짧은 작업을 먼저 실행하는 선점형 SJF**다.

### 예시
```text
A가 10ms짜리 작업으로 실행 중이다.
그런데 B가 2ms짜리 작업으로 새로 들어왔다.

SRTF는 A를 잠시 멈추고 B를 먼저 실행할 수 있다.
```

### 장점
- 짧은 작업의 응답과 완료가 빠르다
- 평균 대기 시간을 줄이는 데 유리하다

### 단점
- 남은 실행 시간을 정확히 알기 어렵다
- 짧은 작업이 계속 들어오면 긴 작업이 굶을 수 있다

---

## MLFQ
[[용어 모음집/MLFQ|MLFQ]](Multi-Level Feedback Queue)는 **여러 우선순위 큐를 두고 작업의 행동에 따라 큐를 올리거나 내리는 방식**이다.

### 동작 방식
- 처음 들어온 작업은 높은 우선순위 큐에서 시작한다.
- CPU를 오래 쓰면 CPU-bound 작업으로 보고 낮은 큐로 내린다.
- 짧게 실행하고 I/O 대기를 자주 하면 interactive 작업으로 보고 높은 우선순위를 유지할 수 있다.
- 오래 기다린 작업은 starvation 방지를 위해 다시 높은 큐로 올릴 수 있다.

### 예시
```text
Q0: 높은 우선순위, 짧은 quantum
Q1: 중간 우선순위, 중간 quantum
Q2: 낮은 우선순위, 긴 quantum

마우스 입력 처리처럼 짧게 자주 실행되는 작업은 Q0에 남기 쉽고,
긴 계산 작업은 Q1, Q2로 내려갈 수 있다.
```

### 왜 중요한가
MLFQ는 "짧은 작업을 우대하고, 긴 작업은 뒤로 보내되, 너무 오래 기다리면 다시 올린다"는 아이디어를 담고 있다. 현대 OS의 동적 우선순위 스케줄링을 이해할 때 좋은 중간 모델이다.

---

## 현대 OS의 실제 스케줄링
실제 운영체제는 FCFS, SJF, RR, Priority 중 하나만 단순히 고르지 않는다. 일반 작업, 실시간 작업, 배치 작업, 컨테이너 그룹을 나누고, 각 영역에 맞는 정책을 적용한다.

### Linux 일반 작업: CFS와 EEVDF
[[용어 모음집/CFS|CFS]](Completely Fair Scheduler)는 일반 프로세스가 CPU를 공정하게 나눠 쓰도록 설계된 Linux 스케줄러다. 핵심은 각 작업의 virtual runtime을 추적해서 **CPU를 상대적으로 덜 받은 작업**을 먼저 실행하는 것이다.

최근 Linux에서는 [[용어 모음집/EEVDF|EEVDF]](Earliest Eligible Virtual Deadline First) 방향으로 일반 작업 스케줄링이 바뀌었다. EEVDF는 작업이 CPU를 받아야 할 "자격"과 virtual deadline을 계산해서, 공정성을 유지하면서 지연 시간에 민감한 작업을 더 잘 다루려는 방식이다.

```text
CFS:
CPU를 덜 받은 작업을 고른다.

EEVDF:
CPU를 받을 자격이 있는 작업 중 virtual deadline이 가장 빠른 작업을 고른다.
```

### Linux 정책 이름
| 정책                                                  | 사용 환경              | 설명                                     |
| --------------------------------------------------- | ------------------ | -------------------------------------- |
| [[용어 모음집/SCHED_OTHER]] 또는 SCHED_NORMAL | 일반 작업              | 대부분의 일반 프로세스가 쓰는 기본 정책                 |
| SCHED_BATCH                                         | 배치 작업              | 상호작용성보다 처리량이 중요한 작업                    |
| SCHED_IDLE                                          | 매우 낮은 중요도 작업       | CPU가 남을 때 실행되어도 되는 작업                  |
| [[용어 모음집/SCHED_FIFO]]                   | 실시간 작업             | 같은 우선순위에서는 스스로 양보하거나 막힐 때까지 계속 실행      |
| [[용어 모음집/SCHED_RR]]                       | 실시간 작업             | 같은 실시간 우선순위 안에서 round-robin            |
| [[용어 모음집/SCHED_DEADLINE]]           | deadline 기반 실시간 작업 | runtime, deadline, period로 CPU 요구량을 표현 |

### cgroup과 컨테이너 CPU 제한
서버와 컨테이너 환경에서는 프로세스 하나만 보는 것이 아니라 [[용어 모음집/cgroup|cgroup]] 단위로 CPU 사용량을 제한한다.

```text
컨테이너 A: CPU 50%
컨테이너 B: CPU 25%
컨테이너 C: CPU 25%
```

이렇게 설정하면 컨테이너 A 안에 스레드가 많아도 그룹 전체가 정해진 CPU 몫 이상을 계속 쓰지 못한다. 그래서 Kubernetes나 Docker 같은 환경에서는 "프로세스 스케줄링"뿐 아니라 "그룹 단위 CPU 배분"도 같이 이해해야 한다.

### Windows 스케줄링
Windows는 스레드 우선순위를 기반으로 선점 스케줄링을 한다. 높은 우선순위의 실행 가능한 스레드가 먼저 CPU를 받고, 같은 우선순위 안에서는 time slice를 나누어 실행한다.

```text
프로세스 priority class
        +
스레드 priority level
        ↓
스레드 base priority
```

예를 들어 UI 입력 스레드는 사용자 반응성을 위해 상대적으로 높은 우선순위를 받을 수 있고, 백그라운드 계산 스레드는 낮은 우선순위로 둘 수 있다. 다만 너무 높은 우선순위를 오래 유지하면 시스템 스레드나 입력 처리까지 방해할 수 있다.

---

## 정책 선택 기준
| 환경 | 중요한 목표 | 어울리는 정책 방향 |
|---|---|---|
| 대화형 UI | 응답 시간 | RR, MLFQ, 동적 우선순위 |
| 배치 작업 | 처리량, 평균 완료 시간 | SJF 추정, SCHED_BATCH |
| 일반 서버 | 공정성, 처리량, 지연 시간 균형 | CFS, EEVDF |
| 컨테이너 서버 | 그룹별 CPU 격리 | cgroup CPU quota/share |
| 실시간 제어 | 마감 시간, 예측 가능성 | SCHED_FIFO, SCHED_RR, SCHED_DEADLINE |

## 알고리즘 비교표

| 알고리즘 | 핵심 | 장점 | 단점 |
|---|---|---|---|
| FCFS | 먼저 온 순서 | 단순함 | 긴 작업이 줄을 막음 |
| SJF | 짧은 작업 우선 | 평균 대기 시간 감소 | 긴 작업 기아 가능 |
| SRTF | 남은 시간이 짧은 작업 우선 | 짧은 작업 응답성 좋음 | 실행 시간 예측 필요 |
| RR | 시간 조각 분배 | 응답성/공정성 좋음 | context switch 비용 |
| Priority | 우선순위 우선 | 중요한 작업 우선 | 낮은 우선순위 기아 |
| MLFQ | 여러 큐와 동적 이동 | 대화형 작업에 유리 | 규칙 설계가 복잡 |
| CFS/EEVDF | 공정한 CPU 몫과 지연 시간 균형 | 현대 Linux 일반 작업에 적합 | 내부 개념이 복잡 |

---

## 기아(starvation)
기아는 **특정 작업이 계속 밀려서 CPU를 거의 못 받는 상태**다.

### 예시
- 우선순위가 낮은 작업이 계속 뒤로 밀림
- 짧은 작업만 계속 들어와 긴 작업이 실행되지 못함

### 해결
- aging

---

## aging
aging은 **오래 기다린 작업의 우선순위를 점점 높여 주는 방식**이다.

### 왜 필요한가
- 기아를 완화하기 위해
- 오래 기다린 작업도 언젠가 실행되게 하기 위해

### 쉽게 이해하기
- 오래 줄 서 있으면 점점 먼저 해주는 규칙

---

## 스케줄링의 흐름 그림

```text
작업 도착
   ↓
Ready Queue
   ↓
스케줄러 선택
   ↓
CPU 실행
   ↓
시간 종료? I/O? 완료?
   ├─ 완료 → 종료
   ├─ I/O → 대기
   └─ 선점 → 다시 큐
```

---

## context switch
context switch는 **실행 중인 작업의 상태를 저장하고 다른 작업의 상태를 복원하는 과정**이다.

### 왜 필요한가
- CPU를 다른 작업에게 넘기려면 현재 상태를 보존해야 한다

### 비용
- 저장/복원 시간이 든다
- 너무 자주 일어나면 성능이 떨어진다

### 쉽게 이해하기
- 작업을 바꿀 때 책갈피를 꽂고 다른 책으로 넘어가는 느낌

---

## 스케줄링과 context switch 관계
- 선점형이 많을수록 context switch가 늘 수 있다
- RR에서 quantum이 너무 작으면 switch가 과해진다
- 성능과 응답성 사이의 균형이 중요하다

---

## CPU-bound vs I/O-bound

### CPU-bound
- CPU를 많이 쓰는 작업
- 계산 위주

### I/O-bound
- 입출력 대기가 많은 작업
- 디스크, 네트워크 대기 위주

### 스케줄링에서 왜 중요한가
- 짧은 CPU burst 작업과 I/O 대기가 많은 작업은 스케줄링 전략이 달라질 수 있다

---

## 멀티레벨 큐 이해
실제 시스템은 단순한 하나의 큐만 쓰지 않을 수도 있다.

### 예시
- 인터랙티브 작업 큐
- 배치 작업 큐
- 시스템 작업 큐

### 이유
- 작업 성격이 다르기 때문이다

---

## 실시간 시스템
[[용어 모음집/실시간 시스템|실시간 시스템]]은 **시간 안에 반드시 반응해야 하는 시스템**이다.

### 예시
- 자동차 제어
- 의료 장비
- 산업 제어

### 중요 포인트
- 처리량보다 마감 시간 준수가 더 중요할 수 있다
- 스케줄링 정책이 매우 중요하다

---

## 데스크톱/서버/실시간 시스템 비교

| 환경 | 중요한 것 | 스케줄링 이해 |
|---|---|---|
| 데스크톱 | 응답성 | 사용자 반응 빠르게 |
| 서버 | 처리량/공정성 | 많은 요청을 균형 있게 |
| 실시간 | 마감 시간 | 정해진 시간 안에 반드시 |

---

## 자주 하는 실수
- FCFS가 무조건 공정하다고만 생각함
  - 바로잡기: FCFS는 도착 순서 기준이라 단순하지만, 긴 작업이 앞에 있으면 짧은 작업이 오래 기다리는 convoy effect가 생길 수 있다.
- SJF가 왜 이론적으로 좋은지 모르고 외움
  - 바로잡기: SJF는 짧은 작업을 먼저 실행해 평균 대기 시간을 줄일 수 있지만 실행 시간 예측과 starvation 문제가 있다.
- RR의 quantum 크기 영향 이해 못함
  - 바로잡기: RR의 quantum이 너무 작으면 문맥 교환 비용이 커지고, 너무 크면 FCFS처럼 반응성이 떨어질 수 있다.
- Priority와 aging 관계를 모름
  - 바로잡기: Priority 스케줄링은 낮은 우선순위 작업이 굶지 않도록 오래 기다린 작업의 우선순위를 올리는 aging이 필요할 수 있다.
- 교과서 알고리즘이 실제 OS에서 그대로 하나만 쓰인다고 생각함
  - 바로잡기: 실제 OS는 일반 작업, 실시간 작업, 배치 작업, 컨테이너 그룹을 나누고 CFS/EEVDF, 실시간 정책, cgroup CPU 제한 같은 여러 정책을 함께 사용한다.
- CFS를 단순한 Round Robin으로 이해함
  - 바로잡기: CFS는 같은 시간 조각을 순서대로 주는 방식이 아니라, virtual runtime을 비교해서 CPU를 덜 받은 작업을 우선 실행하는 공정성 기반 정책이다.
- 실시간 스케줄링을 "무조건 빠른 스케줄링"으로 생각함
  - 바로잡기: 실시간 스케줄링은 평균 속도보다 정해진 시간 안에 실행되는 예측 가능성이 중요하다. SCHED_DEADLINE은 runtime, deadline, period 같은 조건으로 CPU 사용량을 표현한다.
- 선점형/비선점형을 헷갈림
  - 바로잡기: 선점형은 커널이 실행 중인 작업을 중간에 빼앗을 수 있고, 비선점형은 작업이 스스로 CPU를 내놓을 때까지 기다린다.
- context switch 비용을 과소평가함
  - 바로잡기: 문맥 교환은 실행 대상을 바꾸는 것뿐 아니라 PC, SP, 범용 레지스터, 상태 레지스터 같은 CPU 상태를 저장하고 복원하는 작업이다.
- 응답 시간과 처리량을 같은 지표로 생각함
  - 바로잡기: 응답 시간은 첫 반응까지의 시간이고, 처리량은 단위 시간에 완료한 작업 수다.

---

## 디버깅할 때 보는 순서
1. 어떤 작업이 CPU를 많이 쓰는지 본다
2. 응답 시간이 중요한지 처리량이 중요한지 본다
3. 선점형인지 비선점형인지 확인한다
4. 큐가 한쪽으로 몰리는지 본다
5. 기아 상태가 있는지 본다
6. context switch가 너무 잦은지 본다
7. 우선순위/aging 정책이 있는지 본다

- SJF: 짧은 것부터

### RR vs Priority

- 선점형: 중간에 빼앗을 수 있음
- 비선점형: 끝날 때까지 기다림

## 면접 포인트
면접에서는 알고리즘 이름만 말하지 말고 **어떤 목표에 맞는지**까지 말해야 한다.

### 꼭 말할 수 있어야 하는 것
- 스케줄링은 CPU를 어떤 순서로 누구에게 줄지 정하는 정책이다
- 선점형은 중간 회수가 가능하고, 비선점형은 그렇지 않다
- FCFS는 단순하지만 convoy effect가 생길 수 있다
- SJF는 평균 대기 시간을 줄일 수 있지만 기아가 생길 수 있다
- RR은 공정성과 응답성이 좋지만 context switch 비용이 있다
- Priority는 중요한 작업을 먼저 처리하지만 낮은 우선순위가 기아에 빠질 수 있다
- aging은 기아를 완화하는 방법이다
- MLFQ는 여러 우선순위 큐를 사용해 대화형 작업과 CPU-bound 작업을 다르게 다룬다
- 현대 Linux 일반 작업은 CFS/EEVDF 계열로 공정성과 지연 시간의 균형을 맞춘다
- 실시간 정책은 평균 속도보다 deadline과 예측 가능성이 중요하다

### 한 번에 말하는 답변 예시
> 스케줄링은 CPU를 어떤 작업에게 어떤 순서로 줄지 정하는 운영체제의 정책입니다. FCFS, SJF, RR, Priority가 대표적이고, 각각 단순성, 평균 대기 시간, 응답성, 우선순위 반영 같은 장단점이 다릅니다. 선점형은 실행 중인 작업을 뺏을 수 있고 비선점형은 끝날 때까지 기다립니다. 실제 시스템에서는 응답 시간, 처리량, 공정성, 기아 방지의 균형을 보고 선택합니다.

### 현대 OS까지 포함한 답변 예시
> 교과서에서는 FCFS, SJF, RR, Priority로 기준을 배우지만 실제 OS는 하나만 쓰지 않습니다. Linux 일반 작업은 CFS에서 EEVDF 방향으로 발전해 공정성과 지연 시간을 함께 다루고, 실시간 작업은 SCHED_FIFO, SCHED_RR, SCHED_DEADLINE 같은 별도 정책을 씁니다. 서버와 컨테이너 환경에서는 cgroup으로 그룹별 CPU 몫도 제한합니다.

### 꼬리 질문 예시
- SJF가 왜 평균 대기 시간을 줄이나요?
- RR의 time quantum은 왜 중요한가요?
- 기아 상태를 어떻게 완화하나요?
- context switch 비용은 왜 생기나요?
- 실시간 시스템은 왜 다른 기준을 쓰나요?
- CFS와 RR은 무엇이 다른가요?
- EEVDF는 왜 CFS 다음 흐름으로 등장했나요?
- 컨테이너에서는 CPU 스케줄링을 어떻게 제한하나요?

## 보강 설명
스케줄링 주제는 CPU라는 제한된 자원을 어떤 기준으로 나눠 줄지 결정하는 문제다. 좋은 정책은 시스템 목적에 따라 달라지며, 응답 시간과 처리량, 공정성 사이에는 트레이드오프가 있다.

## 구체적인 예시
예를 들어 대화형 시스템은 사용자가 빠르게 반응을 느끼도록 RR 같은 시분할 정책이 유리할 수 있다. 반면 배치 작업은 전체 처리량과 평균 대기 시간을 더 중요하게 볼 수 있다.
