# 실시간 시스템

## 한 줄 정의
[[용어 모음집/실시간 시스템|실시간 시스템]](real-time system)은 **작업의 정답이 맞는 것뿐 아니라, 정해진 시간 안에 끝나는 것도 correctness에 포함되는 시스템**이다.

쉽게 말하면, 실시간 시스템에서는 "언젠가 처리됨"이 아니라 **"마감 시간 안에 처리됨"**이 중요하다.

---

## 왜 스케줄링에서 나오는가
실시간 시스템은 [[용어 모음집/스케줄링|스케줄링]]을 이해할 때 중요한 예외 사례다.

일반적인 데스크톱이나 서버는 보통 다음을 중요하게 본다.

- 평균 [[용어 모음집/응답 시간|응답 시간]]
- 전체 [[용어 모음집/처리량|처리량]]
- [[용어 모음집/CPU|CPU]] 사용률
- 사용자 체감 속도

하지만 실시간 시스템은 다음이 더 중요하다.

- deadline을 지키는가
- 최악의 경우에도 시간 안에 끝나는가
- 실행 시간이 예측 가능한가
- 우선순위가 높은 작업이 필요한 순간에 CPU를 받을 수 있는가

그래서 실시간 시스템은 단순히 "빠른 시스템"이 아니라 **시간 제약을 지켜야 하는 시스템**이다.

---

## 실시간 시스템의 핵심

```text
입력 발생
   ↓
정해진 시간 안에 계산
   ↓
정해진 시간 안에 출력 또는 제어
```

예를 들어 자동차 브레이크 제어 시스템은 브레이크 입력을 받은 뒤 너무 늦게 반응하면 안 된다. 계산 결과가 맞아도 5초 뒤에 나오면 의미가 없다.

---

## 일반 시스템과 실시간 시스템 비교

| 구분 | 일반 시스템 | 실시간 시스템 |
|---|---|---|
| 핵심 목표 | 평균 성능, 사용자 체감, 처리량 | deadline 준수 |
| 느린 응답 | 불편함, 성능 저하 | 실패, 사고, 품질 문제 |
| 중요한 지표 | 평균 응답 시간, 처리량 | 최악 실행 시간, deadline miss |
| 스케줄링 기준 | 공정성, 효율, 응답성 | 예측 가능성, 우선순위, 마감 시간 |
| 예시 | 웹 서버, 코드 에디터, 메신저 | 자동차 제어, 의료 장비, 산업 제어 |

---

## Hard Real-Time
Hard real-time system은 deadline을 놓치면 시스템 실패로 보는 실시간 시스템이다.

### 예시
- 자동차 에어백 전개
- 항공기 제어
- 산업용 로봇 안전 정지
- 심장 박동 조율기 같은 의료 장비

### 특징
- deadline miss가 치명적이다.
- 평균적으로 빠른 것만으로는 부족하다.
- 최악의 경우 실행 시간까지 고려해야 한다.

```text
에어백 센서 감지
   ↓
정해진 시간 안에 에어백 전개
   ↓
늦으면 실패
```

---

## Soft Real-Time
Soft real-time system은 deadline을 놓쳐도 시스템이 완전히 실패하지는 않지만 품질이 떨어지는 시스템이다.

### 예시
- 동영상 재생
- 온라인 게임
- 화상 회의
- 오디오 스트리밍

### 특징
- deadline miss가 바로 치명적 실패는 아니다.
- 늦으면 끊김, 지연, 품질 저하가 발생한다.
- 일부 지연은 허용되지만 계속 늦으면 사용자 경험이 나빠진다.

```text
동영상 프레임 준비
   ↓
화면 출력 시점 안에 도착
   ↓
늦으면 프레임 드롭 또는 끊김
```

---

## Firm Real-Time
Firm real-time system은 deadline을 넘긴 결과가 가치가 없어지는 시스템이다. Hard real-time처럼 치명적 사고는 아니지만, 늦은 결과는 버려진다.

### 예시
- 실시간 주식 시세 처리
- 실시간 센서 데이터 분석
- 일부 광고 입찰 시스템

### 특징
- 늦은 결과는 사용할 가치가 없다.
- deadline miss가 반복되면 시스템 품질이 크게 떨어진다.

```text
센서 데이터 도착
   ↓
100ms 안에 판단해야 함
   ↓
200ms 뒤 결과는 이미 늦어서 폐기
```

---

## Periodic Task와 Aperiodic Task

### Periodic Task
Periodic task는 일정 주기마다 반복되는 작업이다.

```text
10ms마다 센서 읽기
20ms마다 제어값 계산
33ms마다 화면 프레임 출력
```

실시간 시스템에서는 periodic task가 많다. 주기가 정해져 있으면 CPU 사용량을 계산하고 deadline을 예측하기 쉽다.

### Aperiodic Task
Aperiodic task는 정해진 주기 없이 외부 사건에 의해 발생하는 작업이다.

```text
버튼 입력
네트워크 패킷 도착
긴급 센서 이벤트
```

언제 발생할지 모르기 때문에 최악의 경우를 고려해야 한다.

## Deadline, Period, Runtime
실시간 스케줄링에서는 다음 세 가지가 자주 나온다.

| 용어 | 의미 | 예시 |
|---|---|---|
| runtime | 한 번 실행될 때 필요한 CPU 시간 | 2ms 동안 계산 필요 |
| period | 작업이 반복되는 주기 | 10ms마다 실행 |
| deadline | 언제까지 끝나야 하는지 | 시작 후 10ms 안에 완료 |

예를 들어 다음과 같은 작업이 있다고 하자.

```text
runtime = 2ms
period = 10ms
deadline = 10ms
```

이 말은 **10ms마다 한 번 작업이 생기고, 각 작업은 2ms의 CPU 시간이 필요하며, 10ms 안에 끝나야 한다**는 뜻이다.

---

## 실시간 시스템에서 중요한 계산

### CPU 이용률
작업 하나의 CPU 이용률은 보통 다음처럼 생각한다.

```text
CPU 이용률 = runtime / period
```

예를 들어:

```text
작업 A: runtime 2ms, period 10ms
CPU 이용률 = 2 / 10 = 20%

작업 B: runtime 1ms, period 5ms
CPU 이용률 = 1 / 5 = 20%
```

둘을 합치면 CPU 이용률은 40%다. 단순히 평균만 보면 충분해 보일 수 있지만, deadline이 겹치면 스케줄링이 어려워질 수 있다.

---

## 대표 실시간 스케줄링 정책

### Rate Monotonic Scheduling
Rate Monotonic Scheduling은 **주기가 짧은 작업일수록 높은 우선순위를 주는 고정 우선순위 방식**이다.

```text
작업 A: period 10ms -> 높은 우선순위
작업 B: period 50ms -> 낮은 우선순위
```

주기가 짧다는 것은 더 자주 실행되어야 한다는 뜻이므로 더 높은 우선순위를 준다.

### Earliest Deadline First
EDF(Earliest Deadline First)는 **deadline이 가장 가까운 작업을 먼저 실행하는 방식**이다.

```text
작업 A deadline: 30ms 뒤
작업 B deadline: 10ms 뒤

EDF는 B를 먼저 실행한다.
```

### Linux SCHED_DEADLINE
[[용어 모음집/SCHED_DEADLINE|SCHED_DEADLINE]]은 Linux의 deadline 기반 실시간 스케줄링 정책이다. 작업이 필요한 runtime, deadline, period를 커널에 알려 주고, 커널은 그 조건을 기준으로 스케줄링한다.

## 스케줄링과 연결해서 이해하기
일반적인 [[용어 모음집/Round Robin|Round Robin]]이나 [[용어 모음집/CFS|CFS]]는 공정성과 평균 응답성을 중요하게 본다. 반면 실시간 시스템은 공정성보다 deadline 준수가 더 중요할 수 있다.

예를 들어 일반 시스템에서는 모든 작업이 공평하게 CPU를 나눠 갖는 것이 좋아 보인다.

```text
A도 조금 실행
B도 조금 실행
C도 조금 실행
```

하지만 실시간 시스템에서는 마감 시간이 가까운 작업을 먼저 실행해야 할 수 있다.

```text
A deadline 5ms 남음
B deadline 50ms 남음

A를 먼저 실행해야 함
```

## 자주 하는 실수
- 실시간 시스템을 "엄청 빠른 시스템"으로 이해함
  - 바로잡기: 실시간 시스템의 핵심은 평균 속도가 아니라 정해진 시간 안에 끝나는 예측 가능성이다.
- Hard real-time과 Soft real-time을 구분하지 않음
  - 바로잡기: Hard real-time은 deadline miss가 실패이고, Soft real-time은 품질 저하로 이어진다.
- 처리량이 높으면 실시간 요구사항도 만족한다고 생각함
  - 바로잡기: 처리량이 높아도 특정 작업이 deadline을 놓치면 실시간 시스템에서는 실패할 수 있다.
- CFS 같은 일반 스케줄러와 실시간 스케줄러를 같은 기준으로 비교함
  - 바로잡기: CFS는 일반 작업의 공정성을 목표로 하고, 실시간 스케줄링은 deadline과 예측 가능성을 목표로 한다.
- deadline을 "빨리 끝내면 좋은 목표 시간" 정도로 생각함
  - 바로잡기: 실시간 시스템에서 deadline은 시스템 요구사항이며, 특히 Hard real-time에서는 지키지 못하면 실패다.

---

## 면접 포인트
- 실시간 시스템은 정답의 정확성뿐 아니라 시간 안에 완료되는지도 중요하다.
- Hard real-time은 deadline을 놓치면 실패이고, Soft real-time은 품질 저하가 발생한다.
- 실시간 시스템은 평균 응답 시간보다 최악 실행 시간과 deadline miss 여부가 중요하다.
- 실시간 스케줄링에서는 runtime, period, deadline을 함께 본다.
- Rate Monotonic은 주기가 짧은 작업 우선, EDF는 deadline이 가까운 작업 우선이다.

### 답변 예시
> 실시간 시스템은 작업 결과가 맞는 것뿐 아니라 정해진 시간 안에 완료되는 것도 중요한 시스템입니다. 일반 시스템은 평균 응답 시간이나 처리량을 주로 보지만, 실시간 시스템은 deadline miss가 핵심입니다. Hard real-time은 deadline을 놓치면 시스템 실패가 되고, Soft real-time은 동영상 끊김처럼 품질 저하로 이어집니다. 그래서 스케줄링에서도 공정성보다 예측 가능성과 deadline 준수가 더 중요할 수 있습니다.
