# 응답 시간 대기 시간 처리량

## 한 줄 정의
[[용어 모음집/응답 시간|응답 시간]], [[용어 모음집/대기 시간|대기 시간]], [[용어 모음집/처리량|처리량]]은 **[[용어 모음집/스케줄링|스케줄링]] 정책이 좋은지 평가하는 대표 지표**다.

## 응답 시간
요청이 들어온 뒤 첫 반응이 나오기까지 걸리는 시간이다.

예시:
- 사용자가 키를 눌렀을 때 화면에 반응이 나타나기까지
- 웹 요청을 보낸 뒤 첫 바이트가 도착하기까지

대화형 시스템에서는 응답 시간이 중요하다.

## 대기 시간
[[용어 모음집/프로세스|프로세스]]가 준비 큐에서 [[용어 모음집/CPU|CPU]]를 받지 못하고 기다린 총 시간이다.

예시:
```text
도착 → 준비 큐 대기 → 실행 → I/O 대기 → 준비 큐 대기 → 완료
```

여기서 준비 큐에서 기다린 시간이 대기 시간에 해당한다.

## 처리량
단위 시간 동안 완료한 작업 수다.

예시:
- 1초에 처리한 요청 수
- 1분에 완료한 배치 작업 수

서버나 배치 처리 시스템에서는 처리량이 중요하다.

## 반환 시간
반환 시간(turnaround time)은 작업이 도착해서 완료될 때까지의 전체 시간이다.

```text
반환 시간 = 완료 시각 - 도착 시각
```

## 지표 간 트레이드오프
- 응답 시간을 줄이려고 자주 [[용어 모음집/문맥 교환|문맥 교환]]하면 처리량이 떨어질 수 있다.
- 처리량을 높이려고 긴 작업을 몰아서 실행하면 짧은 작업의 응답 시간이 나빠질 수 있다.
- 공정성을 높이면 특정 작업 최적화는 어려울 수 있다.

## 예시
RR 스케줄링에서 quantum을 작게 잡으면 사용자는 빠른 반응을 느낄 수 있지만, 문맥 교환이 너무 많아져 전체 처리량이 감소할 수 있다.

## 면접 포인트
스케줄링 평가는 하나의 지표만 보면 안 된다. 대화형 시스템은 응답 시간, 배치 시스템은 처리량, 전체 효율은 대기 시간과 반환 시간을 함께 본다.

## 보강 설명
스케줄링 주제는 CPU라는 제한된 자원을 어떤 기준으로 나눠 줄지 결정하는 문제다. 좋은 정책은 시스템 목적에 따라 달라지며, 응답 시간과 처리량, 공정성 사이에는 트레이드오프가 있다.

## 구체적인 예시
예를 들어 대화형 시스템은 사용자가 빠르게 반응을 느끼도록 RR 같은 시분할 정책이 유리할 수 있다. 반면 배치 작업은 전체 처리량과 평균 대기 시간을 더 중요하게 볼 수 있다.
