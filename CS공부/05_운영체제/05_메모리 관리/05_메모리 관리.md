# 메모리 관리

## 한 줄 정의
메모리 관리는 **[[용어 모음집/운영체제|운영체제]]가 한정된 메모리를 여러 프로그램에 안전하고 효율적으로 나눠 주는 방식**이다.

쉽게 말하면, 컴퓨터가 여러 프로그램을 동시에 돌릴 때 **"누가 어디를 얼마나 쓰고, 부족하면 어떻게 할지"를 정하는 일**이다.

---

## 왜 배우는가
- 프로그램이 커질수록 메모리를 그냥 쓰기만 해서는 안 된다.
- 여러 프로그램이 동시에 실행되면 메모리 충돌이 생길 수 있다.
- [[용어 모음집/가상 메모리|가상 메모리]], [[용어 모음집/페이지 테이블|페이지 테이블]], [[용어 모음집/페이지 폴트|페이지 폴트]], [[용어 모음집/스왑|스왑]] 같은 개념을 이해하려면 메모리 관리가 필요하다.
- 성능 문제를 볼 때도 메모리 관리가 핵심인 경우가 많다.
- 면접에서 [[용어 모음집/페이징|페이징]], [[용어 모음집/세그멘테이션|세그멘테이션]], 가상 메모리는 기본 질문이다.

---

## 초보자용 한 줄 기억법
- **메모리 관리 = 메모리를 공평하게 나누고, 부족하면 넘치지 않게 조절하는 것**

---

## 메모리 관리를 한 문장으로 다시 말하면
메모리 관리는 **프로그램마다 독립된 주소 공간을 제공하고, 실제 물리 메모리를 효율적으로 배분하며, 부족할 때는 보조 저장장치를 활용하는 운영체제의 기능**이다.

---

## 전체 흐름

```text
프로그램 실행
   ↓
가상 주소 사용
   ↓
페이지 테이블 조회
   ↓
물리 주소로 변환
   ↓
RAM 접근
   ↓
부족하면 페이지 폴트
   ↓
디스크/스왑에서 가져옴
```

---

## 메모리 관리가 하는 일
- 프로그램마다 자기 메모리가 있는 것처럼 보이게 한다
- 실제 [[용어 모음집/RAM|RAM]]을 나눠 쓴다
- 너무 많이 쓰면 페이지를 내리고 다시 올린다
- 주소 변환을 통해 안전성을 높인다
- 메모리 단편화를 줄이려 한다

---

## 핵심 개념
- **가상 주소(virtual address)**: 프로그램이 보는 주소
- **물리 주소(physical address)**: 실제 RAM 주소
- **페이지(page)**: 고정 크기로 나눈 메모리 단위
- **프레임(frame)**: 실제 물리 메모리 쪽의 페이지 크기 단위
- **페이지 테이블**: 가상 주소와 물리 주소를 대응시키는 표
- **[[용어 모음집/TLB|TLB]]**: 주소 변환을 빠르게 하는 [[용어 모음집/캐시|캐시]]
- **페이지 폴트**: 필요한 페이지가 메모리에 없을 때 발생하는 이벤트
- **스왑(swap)**: RAM이 부족할 때 디스크를 임시로 쓰는 방식

---

## 가상 메모리란
가상 메모리는 **프로그램이 실제 RAM보다 큰 연속된 메모리를 가진 것처럼 보이게 하는 기술**이다.

### 왜 필요한가
- 각 프로그램을 격리할 수 있다
- 주소 충돌을 막을 수 있다
- 실제 물리 메모리를 효율적으로 나눠 쓸 수 있다
- 프로그램이 RAM보다 큰 공간을 쓰는 듯한 경험을 줄 수 있다

### 쉽게 말하면
- 각 프로그램에게 "자기만의 큰 방"을 주는 것처럼 보이게 한다
- 실제로는 뒤에서 운영체제가 방을 배정하고 조정한다

---

## 페이징(paging)
페이징은 **메모리를 고정 크기의 페이지로 나눠 관리하는 방식**이다.

### 특징
- 고정 크기 단위로 나눈다
- 페이지 테이블로 주소를 변환한다
- 외부 단편화를 줄이기 좋다
- 현대 운영체제에서 매우 기본적이다

### 장점
- 메모리 할당이 비교적 규칙적이다
- 단편화 문제를 줄이는 데 유리하다
- 가상 메모리와 잘 맞는다

### 단점
- 페이지 테이블 관리 비용이 있다
- 주소 변환이 필요하다
- 페이지 폴트가 나면 느리다

---

## 세그멘테이션(segmentation)
세그멘테이션은 **의미 단위로 메모리를 나누는 방식**이다.

### 의미 단위 예시
- 코드
- 데이터
- 스택
- 힙

### 장점
- 프로그램의 논리 구조와 잘 맞는다
- 코드, 데이터, 스택처럼 의미가 드러난다

### 단점
- 외부 단편화가 생길 수 있다
- 관리가 복잡할 수 있다

### 기억법
- 페이징: 크기 기준
- 세그멘테이션: 의미 기준

---

## 페이징 vs 세그멘테이션 비교표

| 항목 | 페이징 | 세그멘테이션 |
|---|---|---|
| 분할 기준 | 고정 크기 | 의미 단위 |
| 장점 | 단편화 감소 | 구조가 직관적 |
| 단점 | 주소 변환 필요 | 단편화 가능 |
| 예시 | 페이지, 프레임 | 코드/데이터/스택 |
| 현대 OS | 매우 흔함 | 개념적으로 중요 |

---

## 페이지와 프레임
### 페이지(page)
- 가상 메모리를 같은 크기 조각으로 나눈 것
- 프로그램이 보는 단위

### 프레임(frame)
- 물리 메모리를 같은 크기 조각으로 나눈 것
- 실제 RAM 쪽 단위

### 쉬운 연결
- 페이지 ↔ 프레임이 1:1처럼 매핑된다

---

## 페이지 테이블
페이지 테이블은 **가상 주소를 물리 주소로 바꿔 주는 매핑표**다.

### 역할
- 어떤 가상 페이지가 어떤 물리 프레임에 있는지 기억한다
- 권한 정보(읽기/쓰기/실행)도 함께 담을 수 있다
- 페이지 폴트와 주소 변환의 핵심이다

### 비유
- 가상 주소는 책의 목차
- 페이지 테이블은 목차와 실제 페이지를 연결하는 색인

---

## 주소 변환 흐름

```text
가상 주소
   ↓
페이지 번호 + 오프셋 분해
   ↓
페이지 테이블 조회
   ↓
물리 프레임 번호 획득
   ↓
물리 주소 생성
   ↓
RAM 접근
```

---

## TLB란
TLB(Translation Lookaside Buffer)는 **최근에 사용한 주소 변환 결과를 빠르게 기억하는 캐시**다.

### 왜 필요한가
- 페이지 테이블을 매번 메모리에서 찾으면 느리다
- 주소 변환은 매우 자주 일어난다
- 그래서 자주 쓰는 매핑을 캐시해 둔다

### 쉽게 말하면
- 주소 변환의 빠른 메모장

### TLB가 좋으면
- 주소 변환이 빨라진다
- 전체 메모리 접근 성능이 좋아질 수 있다

---

## 페이지 폴트(page fault)
페이지 폴트는 **필요한 페이지가 RAM에 없을 때 발생하는 사건**이다.

### 흐름
1. 프로그램이 어떤 페이지를 읽으려 함
2. 그 페이지가 메모리에 없음
3. 운영체제가 디스크나 스왑에서 가져옴
4. 다시 실행

### 왜 느린가
- RAM보다 디스크가 훨씬 느리기 때문이다
- 페이지 폴트가 잦으면 성능이 크게 떨어진다

---

## 페이지 폴트 흐름 그림

```text
프로그램 접근
   ↓
메모리에 있음? ── 예 → 바로 실행
        │
        아니오
        ↓
페이지 폴트
        ↓
디스크/스왑에서 적재
        ↓
다시 실행
```

---

## 스왑(swap)
스왑은 **RAM이 부족할 때 디스크의 일부를 임시 메모리처럼 사용하는 것**이다.

### 왜 쓰나
- 메모리가 부족해도 프로그램을 계속 돌리기 위해
- 일부 데이터를 디스크로 내리고 다른 데이터를 올리기 위해

### 단점
- 디스크는 RAM보다 매우 느리다
- 스왑이 많아지면 시스템 전체가 느려진다

### 쉽게 이해하기
- 책상 위가 부족해서 책을 서랍에 넣어 두는 느낌

---

## 스왑이 늘면 생기는 현상
- 프로그램이 느려진다
- 응답이 버벅인다
- 시스템 전체가 무거워진다
- 심하면 거의 멈춘 것처럼 보인다

---

## 단편화(fragmentation)
메모리 관리에서 중요한 문제다.

### 내부 단편화
- 할당된 공간 내부에 남는 빈 공간
- 고정 크기 블록을 쓰다 보니 일부가 비는 것

### 외부 단편화
- 전체 빈 공간은 충분하지만 연속된 큰 공간이 없는 것
- 세그멘테이션에서 특히 언급된다

---

## 단편화 비교표

| 종류 | 의미 | 쉽게 말하면 |
|---|---|---|
| 내부 단편화 | 블록 내부가 남음 | 방은 받았는데 일부가 남음 |
| 외부 단편화 | 조각난 빈 공간 | 빈 방은 있는데 큰 방이 없음 |

---

## 메모리 할당과 단편화
### 고정 크기 방식
- 내부 단편화가 생길 수 있다

### 가변 크기 방식
- 외부 단편화가 생길 수 있다

### 페이징과 단편화
- 페이징은 외부 단편화를 줄이는 데 유리하다
- 대신 내부 단편화가 조금 생길 수 있다

---

## 메모리 보호
메모리 관리는 단순한 배분이 아니라 **보호**도 포함한다.

### 보호의 예
- 다른 [[용어 모음집/프로세스|프로세스]] 메모리 침범 방지
- 읽기 전용/쓰기 가능 구분
- 실행 권한 제어

### 왜 중요한가
- 프로그램 하나가 다른 프로그램을 망가뜨리지 못하게 한다
- 운영체제 안정성이 높아진다

---

## 공유 메모리와 메모리 관리
가끔은 여러 프로세스가 메모리를 일부 공유해야 한다.

### 예시
- 공유 라이브러리
- 메모리 맵 파일
- [[용어 모음집/IPC|IPC]]

### 주의
- 공유는 성능에 좋지만 [[용어 모음집/동기화|동기화]] 문제가 생길 수 있다
- 메모리 관리와 동기화를 함께 봐야 한다

---

## 메모리 계층 이해
메모리는 한 덩어리가 아니라 속도/용량이 다른 계층으로 생각하는 게 좋다.

```text
레지스터 → 캐시 → RAM → 디스크
빠름  ↑               ↓ 느림
작음  ↓               ↑ 큼
```

### 쉽게 이해하기
- 빠를수록 작고 비싸다
- 클수록 느리고 싸다
- 운영체제는 이 차이를 관리한다

---

## 메모리 관리의 실제 작업
메모리 관리는 단순히 "나눠 준다"가 아니라 여러 세부 작업을 포함한다.

### 1) 할당(allocation)
- 필요한 메모리를 요청에 맞게 준다
- 프로세스 시작, 객체 생성, 버퍼 생성 때 일어난다

### 2) 해제(deallocation)
- 더 이상 필요 없는 메모리를 회수한다
- 누수가 생기지 않게 관리한다

### 3) 매핑(mapping)
- 가상 주소와 물리 주소를 연결한다
- 페이지 테이블이 핵심이다

### 4) 보호(protection)
- 다른 프로그램의 메모리를 침범하지 못하게 한다
- 권한 정보가 여기서 쓰인다

### 5) 교체(replacement)
- RAM이 부족하면 어떤 페이지를 내릴지 정한다
- [[용어 모음집/페이지 교체|페이지 교체]] 알고리즘이 여기에 해당한다

---

## 연속 할당과 비연속 할당

### 연속 할당
- 메모리를 한 덩어리로 붙여서 할당하는 방식
- 단순하지만 외부 단편화가 생기기 쉽다

### 비연속 할당
- 메모리를 여러 조각으로 나눠 여기저기 배치하는 방식
- 현대 운영체제의 페이징과 잘 맞는다

### 비교표

| 항목 | 연속 할당 | 비연속 할당 |
|---|---|---|
| 배치 | 한 덩어리 | 여러 조각 |
| 장점 | 단순함 | 유연함 |
| 단점 | 단편화에 취약 | 주소 변환 필요 |
| 대표 비유 | 큰 방 하나 | 여러 칸의 창고 |

---

## 요구 페이징(demand paging)
[[용어 모음집/요구 페이징|요구 페이징]]은 **필요한 순간에만 페이지를 메모리에 올리는 방식**이다.

### 왜 쓰나
- 처음부터 모든 페이지를 올리면 비효율적이다
- 실제로 안 쓰는 페이지까지 메모리에 둘 필요가 없다

### 쉽게 이해하기
- 책 전체를 책상 위에 두지 않고, 필요한 쪽만 펼쳐 보는 느낌

### 연결
- 페이지 폴트가 발생하면 그때 페이지를 가져온다

---

## 복사 시점 지연(Copy-on-Write, COW)
Copy-on-Write는 **처음에는 메모리를 공유하다가, 수정이 일어날 때만 복사하는 방식**이다.

### 왜 좋나
- 처음부터 복사하지 않아서 빠르다
- 메모리 사용을 줄일 수 있다

### 예시
- 프로세스 fork 후 부모/자식이 같은 페이지를 읽기만 하면 공유
- 누군가 쓰려고 할 때만 별도 복사

### 쉽게 이해하기
- 처음엔 같은 서류를 같이 보고, 수정할 때만 사본을 만드는 느낌

---

## 페이지 교체(page replacement)
페이지 교체는 **RAM이 꽉 찼을 때 어떤 페이지를 내릴지 정하는 일**이다.

### 왜 중요한가
- 잘못 고르면 페이지 폴트가 늘어난다
- 성능 차이가 크게 난다

### 대표 정책
- FIFO
- [[용어 모음집/LRU|LRU]]
- Clock(Second Chance)

---

## FIFO 페이지 교체
FIFO는 **가장 먼저 들어온 페이지를 먼저 내보내는 방식**이다.

### 장점
- 단순하다

### 단점
- 오래 쓰는 중요한 페이지도 먼저 나갈 수 있다
- 직관과 다르게 성능이 나쁠 수 있다

---

## LRU 페이지 교체
LRU는 **가장 오래 사용하지 않은 페이지를 내보내는 방식**이다.

### 장점
- [[용어 모음집/지역성|지역성]](locality)을 잘 활용한다
- 일반적으로 실용성이 좋다

### 단점
- 정확한 LRU는 구현 비용이 있다
- 실제 시스템은 근사치를 많이 쓴다

---

## Clock(Second Chance)
Clock은 **LRU를 근사하는 대표적인 방식**이다.

### 쉽게 이해하기
- 최근에 사용한 페이지에는 한 번 더 기회를 준다
- 원형 시계처럼 돌면서 교체 대상을 찾는다

### 장점
- FIFO보다 낫고 LRU보다 단순할 수 있다

---

## 페이지 교체 알고리즘 비교표

| 알고리즘 | 핵심 | 장점 | 단점 |
|---|---|---|---|
| FIFO | 먼저 들어온 것 제거 | 단순함 | 성능이 나쁠 수 있음 |
| LRU | 오래 안 쓴 것 제거 | 지역성 활용 | 구현 비용 있음 |
| Clock | 한 번 더 기회 | 실용적 근사 | 완전한 LRU는 아님 |

---

## 지역성(locality)
지역성은 **프로그램이 한 번 쓴 메모리 주변이나 최근에 쓴 메모리를 다시 쓸 가능성이 높다**는 성질이다.

### 종류
- 시간 지역성: 최근에 쓴 것을 다시 쓸 가능성
- 공간 지역성: 가까운 주소를 연속으로 쓸 가능성

### 왜 중요한가
- 캐시, 페이지 교체, 메모리 설계 전반에 영향을 준다

---

## Thrashing
Thrashing은 **페이지 폴트가 너무 자주 발생해 실제 작업보다 페이지 교체에 더 많은 시간을 쓰는 상태**다.

### 증상
- [[용어 모음집/CPU|CPU]] 사용률이 오히려 떨어짐
- 디스크 I/O가 과도해짐
- 시스템 전체가 매우 느려짐

### 쉽게 이해하기
- 일은 못 하고 페이지 옮기기만 반복하는 상황

---

## Working set
Working set은 **프로그램이 일정 시간 동안 실제로 자주 쓰는 페이지들의 집합**이다.

### 왜 중요하나
- 이 집합이 RAM에 충분히 있어야 성능이 좋다
- working set이 너무 크면 thrashing 위험이 커진다

---

## 메모리 압박이 심할 때 나타나는 현상
- 페이지 폴트 증가
- 스왑 증가
- 응답 지연 증가
- thrashing
- OOM(메모리 부족으로 프로세스 종료) 가능성

---

## OOM 이해
메모리가 너무 부족하면 운영체제나 런타임이 일부 프로세스를 종료해 전체 시스템을 살리기도 한다.

### 쉽게 이해하기
- 한 방에 너무 많은 사람이 들어와서, 일부를 내보내야 전체가 움직이는 상황

---

## 메모리 단편화 더 자세히

### 내부 단편화 예시
- 4KB 페이지를 할당했는데 실제로는 3KB만 사용
- 1KB가 내부적으로 남는다

### 외부 단편화 예시
- 빈 공간이 1KB, 2KB, 1KB로 흩어져 있음
- 4KB 연속 공간이 필요하면 못 넣는다

### 왜 어려운가
- 메모리가 충분해 보여도 연속 공간이 없을 수 있기 때문이다

---

## 버디 시스템(Buddy System)
[[용어 모음집/버디 시스템|버디 시스템]]은 **메모리를 2의 거듭제곱 크기로 나누고 합치는 방식**이다.

### 장점
- 분할/병합이 비교적 쉽다
- 관리가 단순한 편이다

### 단점
- 내부 단편화가 생길 수 있다

### 쉽게 이해하기
- 큰 블록을 반씩 쪼개고, 짝이 맞으면 다시 합친다

---

## 슬랩 할당자(Slab Allocator)
[[용어 모음집/슬랩 할당자|슬랩 할당자]]는 **비슷한 크기의 객체를 빠르게 할당하기 위한 기법**이다.

### 주로 어디서 쓰나
- [[용어 모음집/커널|커널]] 내부 객체
- 자주 생성/삭제되는 작은 객체

### 쉽게 이해하기
- 같은 크기 컵을 미리 여러 개 준비해 두는 느낌

---

## 메모리 관리와 캐시의 관계
- 페이지 교체는 캐시와 비슷한 생각을 가진다
- 자주 쓰는 것을 남기고 덜 쓰는 것을 내린다
- 지역성이 좋을수록 성능이 좋아진다

## 실전 예시

### 예시 4: 프로그램 fork
- 부모 프로세스 복제 시 바로 전부 복사하면 비싸다
- Copy-on-Write로 효율을 높인다

### 예시 5: 메모리가 갑자기 바빠짐
- 새 작업이 많아져 페이지 폴트가 늘어난다
- working set이 RAM보다 커지면 느려진다

### 예시 6: 오래된 탭/프로그램 정리
- 사용하지 않는 페이지를 교체 정책으로 내린다

---

## 메모리 관리 체크포인트
- 가상 주소와 물리 주소를 구분했는가
- 페이지와 프레임을 헷갈리지 않는가
- TLB miss와 page fault를 구분하는가
- 스왑이 늘어도 괜찮다고 착각하지 않는가
- 단편화, thrashing, working set을 같이 보고 있는가


---

## 실전 예시

### 예시 1: 브라우저 탭을 많이 열었을 때
- 탭마다 메모리가 필요하다
- RAM이 부족하면 스왑이 증가할 수 있다
- 시스템이 느려질 수 있다

### 예시 2: 큰 이미지 편집
- 큰 버퍼가 필요하다
- 메모리 압박이 생길 수 있다
- 페이지 폴트와 스왑이 문제가 될 수 있다

### 예시 3: 서버가 갑자기 느려짐
- 메모리 누수
- 과도한 캐시
- 스왑 증가
- 페이지 폴트 증가

---

## 자주 하는 실수
- 가상 메모리를 실제 RAM보다 큰 물리 메모리로 착각함
  - 바로잡기: 가상 메모리는 실제 RAM을 늘리는 것이 아니라 프로세스마다 독립적인 주소 공간을 제공하고 필요한 부분을 물리 메모리에 매핑하는 기법이다.
- 페이지와 프레임을 섞음
  - 바로잡기: 페이지는 가상 주소 공간의 고정 크기 블록이고, 프레임은 물리 메모리의 같은 크기 블록이다.
- 페이지 폴트가 단순 에러라고 생각함
  - 바로잡기: 페이지 폴트는 잘못된 접근일 수도 있지만, 요구 페이징이나 COW처럼 정상적인 메모리 로딩 과정에서도 발생한다.
- 스왑이 늘어도 별 문제 없다고 생각함
  - 바로잡기: 스왑은 메모리 압박을 완화하지만 디스크 I/O가 필요해 과도하면 성능이 급격히 떨어진다.
- 내부/외부 단편화를 헷갈림
  - 바로잡기: 내부 단편화는 할당된 블록 안의 남는 공간이고, 외부 단편화는 빈 공간이 흩어져 큰 연속 공간을 만들기 어려운 상태다.
- TLB와 페이지 테이블 역할을 섞음
  - 바로잡기: 페이지 테이블은 주소 변환의 원본 자료구조이고, TLB는 최근 변환 결과를 저장하는 빠른 캐시다.
- 메모리 보호를 배분과 분리해서 생각하지 않음
  - 바로잡기: 메모리 관리는 공간 배분뿐 아니라 읽기/쓰기/실행 권한을 통해 프로세스 간 보호도 담당한다.

---

## 디버깅할 때 보는 순서
1. 메모리 사용량이 얼마나 되는지 본다
2. 스왑이 늘고 있는지 본다
3. 페이지 폴트가 잦은지 본다
4. TLB miss가 많은지 본다
5. 특정 프로세스가 메모리를 많이 먹는지 본다
6. 내부/외부 단편화가 의심되는지 본다
7. 가상 메모리와 실제 메모리 요구가 맞는지 본다

---

## 비교해서 이해하기

### 가상 메모리 vs 실제 메모리
- 가상 메모리: 프로그램이 보는 주소 공간
- 실제 메모리: RAM의 실제 주소 공간

### 페이지 vs 프레임
- 페이지: 가상 메모리 조각
- 프레임: 물리 메모리 조각

### 페이지 테이블 vs TLB
- 페이지 테이블: 전체 매핑 정보
- TLB: 자주 쓰는 매핑 캐시

### 페이징 vs 세그멘테이션
- 페이징: 고정 크기
- 세그멘테이션: 의미 단위

```text

   ↓

```

```text
접근

 ↓
페이지 폴트

```

```text
빠름/작음
레지스터 → 캐시 → RAM → 디스크
느림/큼
```

## 면접 포인트
면접에서는 메모리 관리가 단순히 "RAM 관리"가 아니라 **주소 변환, 보호, 성능, 단편화까지 포함하는 개념**이라는 점을 말해야 한다.

### 꼭 말할 수 있어야 하는 것
- 가상 메모리는 프로그램에 큰 연속 주소 공간을 제공한다
- 페이지 테이블로 가상 주소를 물리 주소로 변환한다
- TLB는 주소 변환을 빠르게 하는 캐시다
- 페이지 폴트는 메모리에 없는 페이지를 가져올 때 발생한다
- 스왑이 늘면 성능이 크게 떨어질 수 있다
- 페이징은 외부 단편화를 줄이는 데 유리하고, 세그멘테이션은 의미 단위로 나눈다

### 한 번에 말하는 답변 예시
> 메모리 관리는 운영체제가 프로그램마다 가상 주소 공간을 제공하고, 실제 물리 메모리를 효율적으로 나누어 쓰는 과정입니다. 가상 주소는 페이지 테이블을 통해 물리 주소로 변환되고, TLB는 이 변환을 빠르게 해 줍니다. 메모리가 부족하면 페이지 폴트가 발생하고 디스크나 스왑에서 페이지를 가져오게 되며, 이 과정이 자주 발생하면 성능이 크게 떨어집니다.

### 꼬리 질문 예시
- 페이지 폴트가 왜 느린가요?
- TLB는 왜 필요한가요?
- 내부 단편화와 외부 단편화 차이는 뭔가요?
- 가상 메모리와 실제 메모리는 어떻게 다른가요?
- 페이징이 세그멘테이션보다 널리 쓰이는 이유는 뭔가요?

## 한 번 더 요약
- 메모리 관리는 운영체제가 메모리를 나누고 보호하고 변환하는 기능이다
- 가상 메모리, 페이지 테이블, TLB, 페이지 폴트, 스왑이 핵심이다
- 페이징은 고정 크기, 세그멘테이션은 의미 단위다
- 스왑이 늘면 시스템이 느려진다
- 단편화와 보호도 함께 고려해야 한다

## 보강 설명
메모리 관리 주제는 가상 주소와 물리 주소를 분리하고, 제한된 물리 메모리를 효율적으로 쓰는 방법을 다룬다. 주소 변환, 페이지 폴트, 교체 정책, 커널 할당자가 서로 연결된다.

## 구체적인 예시
예를 들어 프로세스가 가상 주소에 접근하면 TLB나 페이지 테이블을 통해 물리 프레임을 찾는다. 해당 페이지가 메모리에 없으면 페이지 폴트가 발생하고 운영체제가 디스크에서 페이지를 가져올 수 있다.
