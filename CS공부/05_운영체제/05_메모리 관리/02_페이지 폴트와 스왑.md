# 페이지 폴트와 스왑

## 한 줄 정의
[[용어 모음집/페이지 폴트|페이지 폴트]](page fault)는 **[[용어 모음집/프로세스|프로세스]]가 접근한 가상 페이지가 현재 물리 메모리에 없거나 접근할 수 없을 때 발생하는 예외**이고, [[용어 모음집/스왑|스왑]]은 **메모리 부족 시 일부 페이지를 디스크로 내보내거나 다시 가져오는 기법**이다.

## 페이지 폴트 흐름
```text
페이지 접근
→ 페이지 테이블 valid bit 확인
→ 메모리에 없음
→ page fault 발생
→ 운영체제가 디스크에서 페이지 로드
→ 페이지 테이블 갱신
→ 명령 재시작
```

## 페이지 폴트 종류
- 메모리에 없는 페이지 접근
- 접근 권한 위반
- 아직 실제 물리 페이지를 배정하지 않은 demand paging 상황
- copy-on-write에서 쓰기 발생

## 스왑
스왑은 물리 메모리가 부족할 때 덜 필요한 페이지를 디스크 영역으로 내보내고, 나중에 필요하면 다시 읽어오는 방식이다.

장점:
- 실제 메모리보다 큰 주소 공간을 사용할 수 있다.

단점:
- 디스크 접근은 메모리보다 매우 느리다.
- page fault가 많아지면 시스템이 급격히 느려질 수 있다.

## Thrashing
[[용어 모음집/스래싱|스래싱]](thrashing)은 메모리가 부족해 페이지를 계속 내보내고 다시 가져오느라 실제 작업보다 [[용어 모음집/페이지 교체|페이지 교체]]에 시간을 더 많이 쓰는 상태다.

## 예시
프로그램이 큰 배열 여러 개를 동시에 사용하고 물리 메모리가 부족하면 일부 페이지가 스왑으로 밀려난다. 이후 그 페이지를 다시 접근하면 page fault가 발생하고 디스크에서 다시 읽어와야 한다.

## 자주 헷갈리는 점
- 모든 page fault가 오류는 아니다. demand paging에서는 정상적인 실행 과정에서도 발생한다.
- page fault는 비싸다. 특히 디스크 I/O가 필요하면 비용이 매우 커진다.
- 스왑은 메모리 확장처럼 보이지만 성능상 메모리와 같지 않다.

## 면접 포인트
페이지 폴트는 필요한 페이지가 메모리에 없거나 접근 권한이 맞지 않을 때 발생한다. [[용어 모음집/운영체제|운영체제]]는 페이지를 로드하고 [[용어 모음집/페이지 테이블|페이지 테이블]]을 갱신한 뒤 명령을 재시작할 수 있다. 스왑이 과하면 스래싱이 발생한다.

## 보강 설명
메모리 관리 주제는 가상 주소와 물리 주소를 분리하고, 제한된 물리 메모리를 효율적으로 쓰는 방법을 다룬다. 주소 변환, 페이지 폴트, 교체 정책, [[용어 모음집/커널|커널]] 할당자가 서로 연결된다.

## 구체적인 예시
예를 들어 프로세스가 가상 주소에 접근하면 TLB나 페이지 테이블을 통해 물리 프레임을 찾는다. 해당 페이지가 메모리에 없으면 페이지 폴트가 발생하고 운영체제가 디스크에서 페이지를 가져올 수 있다.
