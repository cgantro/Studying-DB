# Web / HTTP / 인증 / CORS / DNS - 답변 정리

원본 질문: [[CS공부/12_면접/전체 질문 DB/11-Web-HTTP-인증-CORS-DNS-질문]]

---

## 1. HTTP란 무엇인가요?
### 답변(면접용)
- 웹에서 클라이언트-서버가 리소스를 주고받기 위한 애플리케이션 계층 프로토콜이다.
- 요청(Request)과 응답(Response) 메시지를 교환한다.
- 기본은 무상태(stateless)라서 상태는 쿠키/세션/토큰 등으로 보완한다.

---

## 2. HTTP Request와 Response의 구조를 설명해보세요.
### 답변(면접용)
- Request: 시작줄(메서드/경로/버전) + 헤더 + 바디(선택).
- Response: 시작줄(버전/상태코드) + 헤더 + 바디(선택).
- 헤더는 메타데이터, 바디는 실제 데이터(JSON/HTML/바이너리 등)다.

---

## 3. HTTP Method GET, POST, PUT, PATCH, DELETE의 차이는 무엇인가요?
### 답변(면접용)
- GET: 조회, POST: 생성/처리(비멱등인 경우가 많음).
- PUT: 전체 치환(멱등), PATCH: 부분 수정(멱등 보장 여부는 설계에 달림).
- DELETE: 삭제(멱등으로 설계하는 경우가 많음).

---

## 4. GET과 POST의 차이는 무엇인가요?
### 답변(면접용)
- GET은 조회에 쓰고, 캐싱/프리페치 등의 영향을 받을 수 있다.
- POST는 서버 상태 변경/처리에 쓰는 경우가 많고, 바디로 데이터 전달이 일반적이다.
- 민감정보는 URL에 노출될 수 있어 GET 쿼리에 넣지 않는 게 안전하다.

---

## 5. PUT과 PATCH의 차이는 무엇인가요?
### 답변(면접용)
- PUT은 리소스를 “전체 치환”하는 의미로 쓰는 게 일반적이다.
- PATCH는 “부분 수정”을 의미하며 변경 필드만 전달하는 방식이 흔하다.
- 둘 다 멱등성/검증/동시성(ETag 등)을 같이 고려해야 한다.

---

## 6. HTTP Status Code 2xx, 3xx, 4xx, 5xx의 차이는 무엇인가요?
### 답변(면접용)
- 2xx: 성공, 3xx: 리다이렉션, 4xx: 클라이언트 요청 문제, 5xx: 서버 문제.
- 4xx는 “요청을 고쳐야” 재시도가 의미 있고, 5xx는 서버/의존성 점검이 우선이다.
- 운영에서는 코드만 보지 말고 로그/지표로 원인을 같이 본다.

---

## 7. 200, 201, 204의 차이는 무엇인가요?
### 답변(면접용)
- 200 OK: 일반 성공(응답 바디가 있는 경우가 많음).
- 201 Created: 생성 성공(보통 Location 헤더로 새 리소스 위치를 안내).
- 204 No Content: 성공했지만 응답 바디가 없음(삭제/부분 처리 등에 사용).

---

## 8. 301과 302의 차이는 무엇인가요?
### 답변(면접용)
- 301은 영구 이동, 302는 임시 이동 의미로 쓰는 게 일반적이다.
- 브라우저/캐시가 301을 더 적극적으로 기억할 수 있어 운영에 영향이 있다.
- 메서드 유지가 필요하면 307/308도 고려한다.

---

## 9. 400, 401, 403, 404의 차이는 무엇인가요?
### 답변(면접용)
- 400: 잘못된 요청(검증 실패, 파라미터 오류 등).
- 401: 인증 필요/실패, 403: 인증은 됐지만 권한 없음.
- 404: 리소스 없음(또는 존재를 숨기기 위해 404로 응답하는 전략도 있음).

---

## 10. 500, 502, 503, 504의 차이는 무엇인가요?
### 답변(면접용)
- 500: 애플리케이션 내부 오류.
- 502: 게이트웨이/프록시가 업스트림에서 잘못된 응답을 받음.
- 503: 서비스 이용 불가(과부하/점검), 504: 업스트림 응답 대기 중 타임아웃.

---

## 11. HTTP/1.1의 Keep-Alive는 무엇인가요?
### 답변(면접용)
- 하나의 TCP 연결을 여러 HTTP 요청/응답에 재사용하는 기능이다.
- 연결/핸드셰이크 비용을 줄여 레이턴시를 낮춘다.
- 유휴 연결 관리(타임아웃/풀)까지 같이 설계해야 한다.

---

## 12. HTTP/1.1의 Head-of-Line Blocking 문제는 무엇인가요?
### 답변(면접용)
- 하나의 연결에서 앞 요청이 지연되면 뒤 요청이 함께 막힐 수 있는 현상이다.
- HTTP/1.1은 파이프라이닝/순서 문제로 실무에서 제한이 있었다.
- 이를 개선하려고 HTTP/2 멀티플렉싱, HTTP/3(QUIC) 등이 등장했다.

---

## 13. HTTP/2는 HTTP/1.1의 어떤 문제를 개선했나요?
### 답변(면접용)
- 멀티플렉싱으로 한 연결에서 여러 스트림을 병렬로 처리해 HOL을 줄였다.
- HPACK 헤더 압축, 서버 푸시(현재는 활용이 제한적) 같은 기능이 있다.
- 연결 수를 줄여도 동시성이 나오는 방향으로 개선됐다.

---

## 14. HTTP/2 Multiplexing이란 무엇인가요?
### 답변(면접용)
- 하나의 TCP 연결에서 여러 요청/응답 스트림을 동시에 흘려보내는 방식이다.
- 순서대로만 처리하던 제약을 줄여 레이턴시를 개선한다.
- 단, TCP 레벨 HOL(패킷 손실 시) 한계는 남아 HTTP/3가 보완한다.

---

## 15. HTTP/3는 어떤 특징을 가지나요?
### 답변(면접용)
- TCP 대신 QUIC(UDP 기반) 위에서 동작한다.
- 연결/암호화가 통합되어 핸드셰이크 비용을 줄이고, 스트림 단위 HOL을 줄인다.
- 네트워크 환경(모바일, 손실)에서 이점이 커질 수 있다.

---

## 16. TCP와 UDP 기반 웹 통신의 차이는 무엇인가요?
### 답변(면접용)
- TCP는 연결/신뢰성/순서 보장을 제공하고, UDP는 그 보장이 없다.
- HTTP/1.1/2는 TCP 위, HTTP/3는 QUIC(UDP 기반) 위에서 동작한다.
- UDP 기반은 신뢰성/혼잡 제어를 애플리케이션(프로토콜)에서 구현한다.

---

## 17. TLS Handshake는 왜 필요한가요?
### 답변(면접용)
- 통신 암호화에 필요한 키를 합의하고, 서버(필요 시 클라이언트) 신원을 검증한다.
- 세션 키를 만들고 이후 데이터는 대칭키로 빠르게 암호화한다.
- 실패하면 인증서/호스트명/체인/시간 오차 등을 의심한다.

---

## 18. HTTPS는 HTTP와 무엇이 다른가요?
### 답변(면접용)
- HTTPS는 HTTP를 TLS로 감싼 형태로, 암호화/무결성/인증을 제공한다.
- 중간자 공격/도청/변조 위험을 줄인다.
- 운영에서는 인증서 관리(만료, 체인, 배포)가 필수다.

---

## 19. 인증서가 만료되면 어떤 문제가 발생하나요?
### 답변(면접용)
- 브라우저/클라이언트가 신뢰하지 않아 경고 또는 접속 차단이 발생한다.
- 자동 갱신이 실패하면 장애로 이어지므로 만료 모니터링이 필요하다.
- 서버 시간 오차도 “만료처럼” 보이게 만들 수 있다.

---

## 20. 브라우저가 HTTPS 인증서를 검증하는 과정은 어떻게 되나요?
### 답변(면접용)
- 인증서 체인이 신뢰할 수 있는 루트 CA로 이어지는지 확인한다.
- 인증서의 유효 기간, 도메인(SAN/호스트명) 일치 여부를 검증한다.
- 필요하면 폐기(Revocation) 관련 정보도 확인할 수 있다(환경에 따라 다름).
