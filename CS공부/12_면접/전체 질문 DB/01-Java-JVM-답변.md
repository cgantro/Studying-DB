# Java / JVM - 답변 정리

원본 질문: [[CS공부/12_면접/전체 질문 DB/01-Java-JVM-질문]]

---

## 1. Java의 핵심 특징을 설명해보세요.
### 답변(면접용)
- Java는 JVM 위에서 동작하는 정적 타입 언어다.
- 소스는 바이트코드로 컴파일되고, JVM이 인터프리터/JIT로 실행한다.
- 런타임이 GC로 메모리를 관리해 안전성을 높인다.

---

## 2. Java가 플랫폼 독립적이라고 하는 이유는 무엇인가요?
### 답변(면접용)
- OS/CPU용 기계어가 아니라 JVM용 바이트코드로 컴파일한다.
- 플랫폼별 JVM이 바이트코드를 실행해 “한 번 빌드, 여러 환경 실행”이 가능하다.
- 단, 네이티브 의존성(JNI 등)이 있으면 독립성이 깨질 수 있다.

---

## 3. JVM, JRE, JDK의 차이를 설명해보세요.
### 답변(면접용)
- JVM: 바이트코드를 실행하는 가상 머신(클래스 로딩, 실행 엔진, GC 포함).
- JRE: 실행 환경(JVM + 표준 라이브러리).
- JDK: 개발 도구 모음(JRE + `javac`, 디버거, 진단 도구 등).

---

## 4. bytecode란 무엇인가요?
### 답변(면접용)
- `javac`가 `.java`를 컴파일해서 만드는 JVM용 중간 코드다(`.class`).
- JVM이 로딩/검증 후 인터프리터 또는 JIT로 실행한다.
- 바이트코드는 플랫폼 독립성을 확보하는 핵심 매개다.

---

## 5. Java 소스 코드가 실행되기까지의 과정을 설명해보세요.
### 답변(면접용)
- 컴파일: `.java` → `javac` → `.class`(바이트코드).
- 실행: JVM이 클래스를 로딩/검증하고 인터프리터로 실행을 시작한다.
- 자주 실행되는 코드는 JIT가 네이티브 코드로 최적화한다.

---

## 6. `javac`는 어떤 역할을 하나요?
### 답변(면접용)
- Java 소스 코드를 바이트코드(`.class`)로 컴파일한다.
- 컴파일 단계에서 타입 체크, 문법 검증, 어노테이션 처리 등이 일어난다.
- 결과물은 JVM이 실행 가능한 형태다.

---

## 7. Class Loader는 어떤 역할을 하나요?
### 답변(면접용)
- 필요한 시점에 클래스를 로딩해서 JVM 메모리(메타스페이스 등)에 적재한다.
- 부모-자식 위임 모델로 중복 로딩과 클래스 충돌을 줄인다.
- 클래스패스/모듈 경계가 로딩 범위와 충돌을 좌우한다.

---

## 8. Class Loading 과정의 Loading, Linking, Initialization을 설명해보세요.
### 답변(면접용)
- Loading: 바이트코드를 읽어 클래스 구조를 JVM 내부 표현으로 만든다.
- Linking: 검증/준비/해결 과정으로, 정적 필드 기본값 할당과 참조 해결이 포함된다.
- Initialization: 클래스 초기화(정적 초기화 블록, 정적 필드 초기화)를 수행한다.

---

## 9. Bytecode Verifier는 왜 필요한가요?
### 답변(면접용)
- 악의적/손상된 바이트코드가 JVM 안전성을 깨는 것을 막기 위해서다.
- 타입 안정성, 스택 규칙, 접근 규칙 등을 검증한다.
- 검증 실패 시 클래스 로딩 자체가 거부될 수 있다.

---

## 10. JIT Compiler는 어떤 역할을 하나요?
### 답변(면접용)
- 자주 실행되는 바이트코드를 네이티브 코드로 컴파일해 성능을 올린다.
- 런타임 프로파일(핫스팟)을 기반으로 최적화를 적용한다.
- 최적화가 누적되므로 워밍업 구간 성능이 다를 수 있다.

---

## 11. Interpreter와 JIT Compiler의 차이는 무엇인가요.
### 답변(면접용)
- 인터프리터는 바이트코드를 한 줄씩 해석해 실행한다(시작이 빠름).
- JIT는 특정 코드를 네이티브로 컴파일해 반복 실행 성능을 높인다.
- JVM은 보통 “인터프리터로 시작 → 핫 코드 JIT”로 섞어서 쓴다.

---

## 12. JVM이 HotSpot이라는 이름을 가지는 이유는 무엇인가요?
### 답변(면접용)
- 실행 중 “자주 호출되는 지점(핫스팟)”을 찾아 최적화하는 전략에서 유래했다.
- 프로파일 기반으로 인라이닝 등 공격적 최적화를 적용한다.
- 결과적으로 장기 실행 서비스에서 성능 이점이 커진다.

---

## 13. Java에서 `public static void main(String[] args)`의 각 키워드를 설명해보세요.
### 답변(면접용)
- `public`: JVM이 외부에서 엔트리포인트를 호출할 수 있게 공개.
- `static`: 객체 생성 없이 클래스 레벨로 호출 가능.
- `void`/`main`/`String[] args`: 반환 없음, 약속된 이름, 커맨드라인 인자.

---

## 14. primitive type과 reference type의 차이는 무엇인가요?
### 답변(면접용)
- primitive는 값 자체를 저장하고, reference는 객체를 가리키는 참조값을 저장한다.
- reference 타입의 객체 본체는 보통 힙에 있고, 참조값은 스택 프레임에 있을 수 있다.
- `null` 가능 여부, 동등성 비교 방식이 달라진다.

---

## 15. `int`와 `Integer`의 차이는 무엇인가요?
### 답변(면접용)
- `int`는 primitive, `Integer`는 wrapper 객체(reference)다.
- wrapper는 `null` 가능, 제네릭/컬렉션에 필요, 오토박싱 비용이 생길 수 있다.
- `Integer`는 비교 시 `==` 사용에 특히 주의가 필요하다.

---

## 16. autoboxing과 unboxing이란 무엇인가요?
### 답변(면접용)
- 오토박싱: primitive → wrapper 자동 변환, 언박싱: wrapper → primitive 자동 변환.
- 편의성이 높지만, 반복 연산에서 객체 생성/GC 비용이 생길 수 있다.
- `null` wrapper를 언박싱하면 `NullPointerException`이 발생한다.

---

## 17. autoboxing이 성능 문제를 만들 수 있는 경우는 언제인가요?
### 답변(면접용)
- 루프/대량 연산에서 boxing/unboxing이 반복되면 객체 생성과 GC 부담이 커진다.
- `List<Integer>`처럼 wrapper 기반 컬렉션을 수치 연산에 과하게 쓰면 손해가 커진다.
- 가능하면 primitive 특화 API(`IntStream` 등)나 자료구조 선택으로 줄인다.

---

## 18. Java는 pass-by-value라고 하는데, reference type에서는 어떻게 동작하나요?
### 답변(면접용)
- Java는 항상 “값 복사”로 전달한다.
- reference 타입은 “참조값(주소 비슷한 값)”이 복사되어 전달된다.
- 따라서 객체 필드 변경은 반영되지만, 파라미터 재할당은 호출자에 영향이 없다.

---

## 19. `==`와 `equals()`의 차이는 무엇인가요?
### 답변(면접용)
- `==`: primitive는 값 비교, reference는 동일 참조(동일성) 비교.
- `equals()`: 논리적 동등성 비교를 위해 사용하며 클래스가 재정의할 수 있다.
- 문자열/값 객체 비교는 보통 `equals()`가 기본이다.

---

## 20. `equals()`와 `hashCode()`를 함께 재정의해야 하는 이유는 무엇인가요?
### 답변(면접용)
- 해시 기반 컬렉션의 계약 때문이다(`HashMap`, `HashSet`).
- `equals()`가 `true`면 `hashCode()`도 같아야 버킷 탐색이 일관된다.
- 둘 중 하나만 깨지면 조회 실패, 중복 삽입 같은 문제가 생긴다.

---

## 21. `String`은 왜 immutable인가요?
### 답변(면접용)
- 불변이면 공유(String Pool) 시 안전하고, 해시 캐시 등 최적화가 가능하다.
- 보안(클래스 로딩, 경로, 쿼리 등)에서 문자열 변경에 따른 위험을 줄인다.
- 변경이 잦으면 `StringBuilder` 같은 가변 타입을 쓴다.

---

## 22. String Pool이란 무엇인가요?
### 답변(면접용)
- 리터럴 문자열을 JVM이 공유 저장하는 영역/메커니즘이다.
- 같은 리터럴은 같은 인스턴스를 재사용해 메모리를 아낀다.
- `new String()`은 풀 재사용과 별개로 새 객체를 만들 수 있다.

---

## 23. `new String("abc")`와 `"abc"`의 차이는 무엇인가요?
### 답변(면접용)
- `"abc"`는 리터럴로 String Pool에 올라가고, 재사용될 수 있다.
- `new String("abc")`는 풀의 리터럴과 별도로 힙에 새 객체를 만든다.
- 동일성(`==`)은 달라질 수 있고, 동등성(`equals`)은 같을 수 있다.

---

## 24. `String.intern()`은 어떤 역할을 하나요?
### 답변(면접용)
- 문자열을 String Pool에 등록하고, 풀의 대표 인스턴스를 돌려준다.
- 동일 문자열의 참조를 통일해 메모리/비교 비용을 줄일 수 있다.
- 남용하면 풀에 과도한 문자열이 쌓여 오히려 문제가 될 수 있다.

---

## 25. `String`, `StringBuilder`, `StringBuffer`의 차이는 무엇인가요?
### 답변(면접용)
- `String`은 불변, `StringBuilder`/`StringBuffer`는 가변이다.
- `StringBuffer`는 동기화로 스레드 안전을 제공하지만 보통 느리다.
- 단일 스레드 문자열 조합은 `StringBuilder`가 일반적인 선택이다.

---

## 26. 문자열 비교에서 `==`를 쓰면 안 되는 이유는 무엇인가요?
### 답변(면접용)
- `==`는 문자열 내용이 아니라 참조 동일성을 비교한다.
- 리터럴 풀 재사용 여부에 따라 결과가 달라져 버그가 된다.
- 문자열 내용 비교는 `equals()`(또는 `equalsIgnoreCase`)를 쓴다.

---

## 27. `final` 키워드는 변수, 메서드, 클래스에서 각각 어떤 의미인가요?
### 답변(면접용)
- 변수: 재할당 불가(참조 변수면 “참조 자체”만 고정).
- 메서드: 오버라이드 금지.
- 클래스: 상속 금지.

---

## 28. `final` reference 변수의 내부 객체 상태는 변경 가능한가요?
### 답변(면접용)
- 가능하다. `final`은 참조 재할당만 막고, 객체 내부 불변을 보장하지 않는다.
- 내부 상태까지 불변으로 만들려면 객체 설계를 불변으로 해야 한다.
- 컬렉션은 방어적 복사/언모디파이어블 래퍼 등을 고려한다.

---

## 29. `static` 키워드의 의미는 무엇인가요?
### 답변(면접용)
- 인스턴스가 아니라 클래스에 귀속되는 멤버를 만든다.
- 클래스 로딩 시점/초기화 시점과 결합되어 생명주기가 길다.
- 전역 상태처럼 쓰면 테스트/동시성/메모리 측면에서 문제가 된다.

---

## 30. static 메서드에서 instance 변수에 직접 접근할 수 없는 이유는 무엇인가요?
### 답변(면접용)
- static은 인스턴스 없이 호출 가능해서 “어느 객체의 필드인지”가 정해지지 않는다.
- 접근하려면 특정 인스턴스 참조가 필요하다.
- 반대로 static 필드/메서드는 인스턴스 없이도 접근 가능하다.

---

## 31. Checked Exception과 Unchecked Exception의 차이는 무엇인가요?
### 답변(면접용)
- checked: 컴파일러가 처리(try/catch 또는 throws)를 강제한다.
- unchecked(RuntimeException): 강제하지 않으며 주로 프로그래밍 오류/복구 불가 상황에 쓴다.
- “복구 가능성”과 “API 사용성”을 같이 보고 선택한다.

---

## 32. Java에서 예외를 너무 넓게 catch하면 어떤 문제가 생기나요?
### 답변(면접용)
- 진짜 원인을 숨기고 장애 분석이 어려워진다.
- 복구 불가능한 예외까지 삼켜서 시스템이 더 나쁜 상태로 진행될 수 있다.
- 필요한 범위만 잡고, 로그/재throw/변환 정책을 명확히 한다.

---

## 33. try-with-resources는 어떤 문제를 해결하나요?
### 답변(면접용)
- 자원(close 필요)의 누수를 줄이기 위해 자동 `close()`를 보장한다.
- 예외가 발생해도 `close()`가 호출되도록 언어 차원에서 지원한다.
- `AutoCloseable`/`Closeable`을 구현한 자원에 적용된다.

---

## 34. `AutoCloseable`은 어떤 역할을 하나요?
### 답변(면접용)
- `try-with-resources`에서 자동 정리 대상이 되기 위한 인터페이스다.
- `close()` 메서드를 제공해 자원 해제를 표준화한다.
- 구현 시 close에서 예외를 던질 수 있어 처리 전략을 정해야 한다.

---

## 35. `throw`와 `throws`의 차이는 무엇인가요?
### 답변(면접용)
- `throw`: 실제로 예외를 발생시키는 문장.
- `throws`: 메서드 시그니처에 “이 예외를 던질 수 있다”를 선언.
- checked 예외는 `throws`로 전파하거나 잡아야 한다.

---

## 36. JVM Runtime Data Area를 설명해보세요.
### 답변(면접용)
- 메서드 영역(메타스페이스), 힙, 스택(프레임), PC 레지스터, 네이티브 메서드 스택으로 구성된다.
- 힙은 공유 영역, 스택은 스레드별로 분리된다.
- GC는 주로 힙을 대상으로 동작한다.

---

## 37. Heap 영역에는 무엇이 저장되나요?
### 답변(면접용)
- new로 생성된 객체 인스턴스와 배열이 저장되는 영역이다.
- 스레드 간 공유되므로 동시성 이슈는 주로 힙 객체 공유에서 생긴다.
- GC의 주요 대상이며, 크기/세대 구성에 따라 성능이 달라진다.

---

## 38. Stack 영역에는 무엇이 저장되나요?
### 답변(면접용)
- 메서드 호출 단위의 스택 프레임(지역 변수, 파라미터, 리턴 주소 등)이 쌓인다.
- 스레드별로 분리되어 있어 공유되지 않는다.
- 깊은 재귀/큰 지역 변수는 스택 오버플로를 유발할 수 있다.

---

## 39. Method Area 또는 Metaspace에는 무엇이 저장되나요?
### 답변(면접용)
- 클래스 메타데이터(클래스 구조, 메서드/필드 정보 등)가 저장된다.
- HotSpot에서는 PermGen 대신 Metaspace를 사용한다(개념적으로).
- 클래스 로더 누수 같은 문제가 메타스페이스 메모리로 이어질 수 있다.

---

## 40. PC Register와 Native Method Stack은 어떤 역할을 하나요?
### 답변(면접용)
- PC Register: 각 스레드가 현재 실행 중인 바이트코드 위치를 추적한다.
- Native Method Stack: JNI 등 네이티브 메서드 호출을 위한 스택을 관리한다.
- 둘 다 스레드별로 존재하며, 호출/실행 흐름 유지에 필요하다.

---

## 41. Java 객체는 Heap에 생성된다고 하는데, 모든 객체가 반드시 Heap에 남나요?
### 답변(면접용)
- 일반적으로 객체는 힙에 할당되지만, 최적화로 달라질 수 있다.
- Escape Analysis로 스레드 밖으로 “탈출하지 않는” 객체는 스택 할당/분해될 수 있다.
- 구현/옵션/상황에 따라 달라질 수 있어 단정은 피한다(확인 필요).

---

## 42. Escape Analysis란 무엇인가요?
### 답변(면접용)
- 객체가 메서드/스레드 밖으로 참조가 나가는지(escape) 분석하는 최적화다.
- escape 하지 않으면 스택 할당, 동기화 제거 같은 최적화가 가능하다.
- 결과적으로 객체 생성/GC/락 비용을 줄일 수 있다.

---

## 43. Java에서도 메모리 누수가 발생할 수 있나요?
### 답변(면접용)
- 발생한다. GC가 있어도 “참조가 남아 있으면” 회수되지 않는다.
- 주로 컬렉션에 계속 쌓아두거나 캐시/리스너/ThreadLocal 정리가 안 되는 경우다.
- 누수는 힙 사용량 증가, GC 빈도 증가, 응답 지연으로 나타난다.

---

## 44. Java 메모리 누수가 발생하는 대표적인 경우는 무엇인가요?
### 답변(면접용)
- 전역(static) 컬렉션/캐시에 객체가 계속 쌓이는 경우.
- 리스너/콜백 등록 후 해제 누락, 클래스 로더 참조 누수.
- ThreadLocal에 큰 객체를 넣고 remove를 안 하는 경우.

---

## 45. static Collection이 메모리 누수를 만들 수 있는 이유는 무엇인가요?
### 답변(면접용)
- static은 클래스 생명주기와 함께 오래 살아남아 참조가 쉽게 해제되지 않는다.
- 컬렉션에서 제거하지 않으면 객체가 계속 참조되어 GC가 회수하지 못한다.
- 캐시를 쓴다면 크기 제한/만료 정책이 필요하다.

---

## 46. ThreadLocal이 메모리 누수를 만들 수 있는 경우는 언제인가요?
### 답변(면접용)
- 스레드 풀 환경에서 ThreadLocal 값이 스레드에 남아 재사용될 때가 대표적이다.
- 요청 단위 값(유저/트랜잭션 컨텍스트 등)을 넣고 remove를 안 하면 누수가 된다.
- 라이브러리 사용 시 finally에서 정리하는 습관이 중요하다.

---

## 47. GC가 필요한 이유는 무엇인가요?
### 답변(면접용)
- 수동 메모리 해제 실수(누수, 댕글링)를 줄이고 개발 생산성을 높인다.
- 살아있는 객체 그래프를 기준으로 회수해 메모리 안전성을 확보한다.
- 대신 STW/스루풋/지연 같은 트레이드오프가 있다.

---

## 48. GC Root란 무엇인가요?
### 답변(면접용)
- GC가 “도달 가능성”을 판단할 때 출발점이 되는 참조 집합이다.
- 스레드 스택의 지역 변수, static 참조, JNI 참조 등이 대표적이다.
- 루트에서 도달 가능한 객체는 “살아있다”고 본다.

---

## 49. Minor GC와 Major GC의 차이는 무엇인가요?
### 답변(면접용)
- Minor GC는 주로 Young 영역을 대상으로 하는 회수다.
- Major(또는 Old) GC는 Old 영역까지 포함한 큰 회수로, 영향이 더 클 수 있다.
- 정확한 용어/범위는 GC 구현에 따라 다를 수 있어 확인이 필요하다.

---

## 50. Stop-the-world란 무엇인가요?
### 답변(면접용)
- GC 같은 작업을 위해 애플리케이션 스레드를 일시 정지하는 구간이다.
- STW가 길면 지연 시간이 튀고 타임아웃/장애로 이어질 수 있다.
- 목표는 “STW 시간을 줄이기”와 “예측 가능하게 만들기”다.

---

## 51. Young Generation과 Old Generation을 나누는 이유는 무엇인가요?
### 답변(면접용)
- 대부분 객체는 금방 죽는다는 경험칙(weak generational hypothesis)을 활용한다.
- Young은 빠르게 자주 회수하고, 오래 사는 객체는 Old로 승격해 관리한다.
- 전체 GC 비용을 낮추고 성능을 안정화하려는 목적이다.

---

## 52. 객체가 Old Generation으로 승격되는 기준은 무엇인가요?
### 답변(면접용)
- Young GC를 여러 번 생존한 객체가 임계치를 넘으면 승격된다.
- Survivor 공간 압박 등으로 조기 승격이 발생할 수도 있다.
- 구체 임계치/정책은 GC 종류/옵션에 따라 달라진다(확인 필요).

---

## 53. G1 GC는 어떤 특징을 가지나요?
### 답변(면접용)
- 힙을 리전(Region) 단위로 나누고, 우선순위 기반으로 회수한다.
- 목표는 긴 STW를 피하면서 예측 가능한 지연을 제공하는 것이다.
- Mixed GC 등으로 Young과 일부 Old를 함께 관리한다.

---

## 54. ZGC는 어떤 목적의 GC인가요?
### 답변(면접용)
- 매우 짧은 STW를 목표로 하는 저지연(low-latency) GC다.
- 대규모 힙에서도 지연을 줄이는 방향으로 설계되어 있다.
- 대신 환경/옵션/워크로드에 따라 비용과 효과가 달라진다.

---

## 55. Java에서 `System.gc()`를 직접 호출하는 것이 권장되지 않는 이유는 무엇인가요?
### 답변(면접용)
- GC는 JVM이 힙/부하 상태를 보고 최적 시점을 선택하는 게 일반적이다.
- 강제 호출은 불필요한 STW/성능 저하를 유발할 수 있다.
- 메모리 문제는 호출로 숨기기보다 누수/객체 생명주기 원인을 찾아야 한다.

---

## 56. Java 8에서 도입된 주요 기능은 무엇인가요?
### 답변(면접용)
- 람다, 스트림, 디폴트 메서드, `Optional` 같은 함수형 스타일 요소가 핵심이다.
- 날짜/시간 API(`java.time`) 등 표준 라이브러리도 강화되었다.
- 코드 스타일과 성능(박싱/병렬 스트림 등) 트레이드오프를 같이 본다.

---

## 57. Lambda Expression은 기존 익명 클래스와 무엇이 다른가요?
### 답변(면접용)
- 함수형 인터페이스를 더 간결하게 구현하기 위한 문법이다.
- 캡처/스코프 차이(예: effectively final)와 `this` 바인딩이 익명 클래스와 다르다.
- 목적은 보일러플레이트를 줄이고 동작(behavior)을 전달하는 것이다.

---

## 58. Stream API의 장점과 단점은 무엇인가요?
### 답변(면접용)
- 장점: 파이프라인으로 읽기 쉬운 데이터 처리, 중간 연산/최종 연산 분리.
- 단점: 디버깅이 어려울 수 있고, 박싱/람다 비용으로 느려질 수 있다.
- 성능이 중요하면 루프와 비교해 선택한다.

---

## 59. Optional은 어떤 문제를 해결하기 위해 도입되었나요?
### 답변(면접용)
- “값이 없을 수 있음”을 명시적으로 표현해 NPE를 줄이려는 목적이다.
- 호출자가 값 존재 여부를 처리하도록 유도한다.
- 필드 타입으로 남용하면 직렬화/성능/가독성 문제가 생길 수 있다.

---

## 60. Java 8 default method는 왜 필요했나요?
### 답변(면접용)
- 인터페이스에 메서드를 추가할 때 기존 구현체를 깨지 않기 위해 도입됐다.
- 공통 구현을 인터페이스 레벨에서 제공할 수 있다.
- 다중 상속처럼 충돌 가능성이 있어 설계에 주의가 필요하다.

---

## 61. Java 11이 기업 환경에서 많이 쓰이는 이유는 무엇인가요?
### 답변(면접용)
- 장기 지원(LTS) 버전으로 운영 안정성과 업그레이드 비용 측면에서 선호된다.
- Java 8 대비 언어/라이브러리/성능 개선이 누적되어 있다.
- 정책상 “LTS 중심 업그레이드” 전략이 흔하다.

---

## 62. Java 11의 `var`는 어떤 상황에서 유용하고 언제 피해야 하나요?
### 답변(면접용)
- 지역 변수 타입 추론으로 반복되는 타입 표기를 줄여 가독성을 높인다.
- 타입이 명확한 경우(팩토리 메서드, 빌더 등)에 특히 유용하다.
- 타입이 모호해져 읽기 어려우면 명시 타입을 유지하는 게 낫다.

---

## 63. Java 17의 `record`는 어떤 문제를 해결하나요?
### 답변(면접용)
- 불변 데이터 전달용 타입(DTO/VO)에서 보일러플레이트를 줄인다.
- `equals/hashCode/toString` 등을 자동 생성해 실수를 줄인다.
- 도메인 로직이 많은 타입에는 맞지 않을 수 있다.

---

## 64. Java 17의 sealed class는 어떤 상황에서 유용한가요?
### 답변(면접용)
- 상속 가능한 하위 타입을 제한해 모델을 닫힌 형태로 만들 수 있다.
- 패턴 매칭/스위치와 결합하면 분기 누락을 줄일 수 있다.
- 라이브러리 API 안정성과 도메인 모델링에 도움이 된다.

---

## 65. Java 21의 Virtual Thread는 기존 Platform Thread와 무엇이 다른가요?
### 답변(면접용)
- Virtual Thread는 적은 OS 스레드 위에 많은 경량 스레드를 멀티플렉싱한다.
- 블로킹 I/O가 많을 때 동시성 처리 효율을 높이는 방향이다.
- DB 커넥션 풀 같은 외부 자원 병목은 그대로라서 같이 설계해야 한다.
