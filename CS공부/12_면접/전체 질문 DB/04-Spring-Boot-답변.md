# Spring Boot - 답변 정리

원본 질문: [[CS공부/12_면접/전체 질문 DB/04-Spring-Boot-질문]]

---

## 1. Spring Framework를 사용하는 이유는 무엇인가요?
### 답변(면접용)
- 객체 생성/의존 관리(IoC/DI)로 결합을 낮추고 테스트를 쉽게 한다.
- 공통 인프라(트랜잭션, AOP, MVC 등)를 표준 방식으로 제공한다.
- 팀 단위 개발에서 구조/관례로 일관성을 만들기 좋다.

---

## 2. Spring Boot와 Spring Framework의 차이는 무엇인가요?
### 답변(면접용)
- Spring Framework는 핵심 기능(컨테이너, MVC 등) 자체다.
- Spring Boot는 자동 설정/스타터/운영 기능으로 “쉽게 실행/배포”를 돕는다.
- Boot가 Framework 위에서 개발 경험을 단순화한다.

---

## 3. Spring Boot의 핵심 장점은 무엇인가요?
### 답변(면접용)
- Auto Configuration으로 설정을 줄이고 빠르게 시작할 수 있다.
- Starter로 의존성 조합이 표준화된다.
- Actuator, 외부 설정, 프로파일로 운영/환경 분리가 쉬워진다.

---

## 4. IoC란 무엇인가요?
### 답변(면접용)
- 객체 생성/생명주기 제어 흐름이 애플리케이션 코드에서 컨테이너로 넘어가는 것이다.
- 개발자는 객체 관계를 선언하고, 컨테이너가 조립한다.
- 결합도를 낮추고 확장/테스트를 쉽게 한다.

---

## 5. DI란 무엇인가요?
### 답변(면접용)
- 객체가 직접 의존성을 생성하지 않고, 외부에서 주입받는 방식이다.
- 생성자 주입이 의존성 명확성/불변성 측면에서 유리하다.
- DIP(추상 의존)와 함께 쓰면 교체/테스트가 쉬워진다.

---

## 6. IoC Container는 어떤 역할을 하나요?
### 답변(면접용)
- Bean 생성/조립/수명주기 관리와 의존성 주입을 수행한다.
- AOP 프록시, 이벤트, 스코프 같은 부가 기능도 제공한다.
- 설정/어노테이션 기반으로 컴포넌트를 스캔한다.

---

## 7. Bean이란 무엇인가요?
### 답변(면접용)
- Spring 컨테이너가 관리하는 객체를 Bean이라고 한다.
- 생성/주입/수명주기가 컨테이너에 의해 관리된다.
- 애플리케이션 구성 요소(서비스, 리포지토리 등)가 주 대상이다.

---

## 8. Bean 생명주기를 설명해보세요.
### 답변(면접용)
- 생성 → 의존성 주입 → 초기화 콜백 → 사용 → 소멸 콜백 순서로 볼 수 있다.
- 초기화/소멸 훅(@PostConstruct, @PreDestroy 등)으로 자원 준비/정리를 한다.
- 스코프에 따라 수명주기(싱글톤/리퀘스트 등)가 달라진다.

---

## 9. Component Scan은 어떻게 동작하나요?
### 답변(면접용)
- 지정한 패키지 범위에서 컴포넌트 어노테이션을 찾아 Bean으로 등록한다.
- 필터/조건으로 포함/제외를 제어할 수 있다.
- 스캔 범위를 넓히면 의도치 않은 Bean 충돌이 생길 수 있다.

---

## 10. Auto Configuration은 어떻게 동작하나요?
### 답변(면접용)
- 클래스패스 의존성과 설정 값을 조건으로 필요한 Bean을 자동 등록한다.
- `@Conditional...` 조건으로 “있으면 등록/없으면 등록 안 함” 형태로 동작한다.
- 문제 발생 시 자동 설정 리포트/로그로 어떤 설정이 적용됐는지 확인한다.

---

## 11. `@SpringBootApplication`에는 어떤 어노테이션이 포함되어 있나요?
### 답변(면접용)
- 보통 `@SpringBootConfiguration`, `@EnableAutoConfiguration`, `@ComponentScan` 조합이다.
- 애플리케이션 시작점에서 자동 설정과 스캔을 켠다.
- 위치(패키지 루트)가 스캔 범위를 결정한다.

---

## 12. Starter Dependency는 어떤 장점이 있나요?
### 답변(면접용)
- 기능 단위로 의존성 조합을 표준화해 충돌/누락을 줄인다.
- 버전 호환을 BOM로 관리해 일관성을 만든다.
- 신규 프로젝트 세팅 시간을 크게 줄인다.

---

## 13. `application.yml`과 `application.properties`의 차이는 무엇인가요?
### 답변(면접용)
- 표현 방식 차이(yml은 계층 구조 표현이 편함, properties는 단순 키-값).
- 기능적으로는 동일한 설정을 담을 수 있다.
- 팀 기준(가독성, 충돌, 포맷)을 정해 통일하는 게 좋다.

---

## 14. Profile을 사용하는 이유는 무엇인가요?
### 답변(면접용)
- 개발/테스트/운영 환경별 설정을 분리하기 위해서다.
- DB, 로깅, 외부 API 엔드포인트 같은 값이 환경마다 달라진다.
- 잘못된 프로파일로 운영에 개발 설정이 들어가는 실수를 막는다.

---

## 15. 환경변수로 설정을 주입하는 이유는 무엇인가요?
### 답변(면접용)
- 이미지/아티팩트는 동일하게 두고 런타임에 값만 바꿀 수 있다.
- 비밀 정보(키/비밀번호)를 코드/레포에서 분리하기 좋다.
- 12-factor app 스타일로 운영 자동화에 유리하다.

---

## 16. `@Component`, `@Service`, `@Repository`, `@Controller`의 차이는 무엇인가요?
### 답변(면접용)
- 모두 컴포넌트 스캔 대상으로 Bean 등록을 유도한다.
- 역할을 표현해 가독성을 높이고, 일부는 부가 동작(예: 예외 변환) 힌트를 준다.
- 의미를 살려 구분하면 계층 구조가 명확해진다.

---

## 17. `@RestController`와 `@Controller`의 차이는 무엇인가요?
### 답변(면접용)
- `@RestController`는 기본적으로 응답 바디(JSON 등)를 반환하는 컨트롤러다.
- `@Controller`는 뷰 렌더링을 포함할 수 있고, 바디 반환은 `@ResponseBody`가 필요할 수 있다.
- API 서버는 보통 `@RestController`를 쓴다.

---

## 18. 생성자 주입을 권장하는 이유는 무엇인가요?
### 답변(면접용)
- 의존성이 명확하고, 불변 필드로 만들기 쉽다.
- 테스트에서 주입이 간단하고, 순환 참조를 조기에 발견하기 쉽다.
- 런타임 NPE 같은 문제를 줄인다.

---

## 19. 필드 주입이 권장되지 않는 이유는 무엇인가요?
### 답변(면접용)
- 테스트가 어렵고, 의존성이 숨겨져 객체가 불완전한 상태가 되기 쉽다.
- final 불변으로 만들기 어렵다.
- 순환 참조가 런타임까지 숨어 있을 수 있다.

---

## 20. 순환 참조는 왜 발생하나요?
### 답변(면접용)
- A가 B를, B가 A를 의존하는 구조가 되면 발생한다.
- 책임 분리가 잘못되었거나 계층이 꼬인 신호인 경우가 많다.
- 구조를 바꾸지 않으면 프록시/지연 주입 같은 꼼수로 숨기게 된다.

---

## 21. 순환 참조를 해결하는 방법은 무엇인가요?
### 답변(면접용)
- 설계를 바꿔 의존 방향을 끊는 게 1순위다(책임 분리, 중간 추상 도입).
- 이벤트/콜백/인터페이스로 결합을 낮출 수 있다.
- 임시로는 지연 주입(ObjectProvider 등)으로 끊을 수 있지만 원인 해결이 우선이다.

---

## 22. Spring Bean은 기본적으로 어떤 scope를 가지나요?
### 답변(면접용)
- 기본은 singleton scope다(컨테이너 내 1개 인스턴스).
- 요청/세션 스코프 등은 웹 환경에서 별도로 쓸 수 있다.
- 싱글톤은 공유되므로 상태를 두면 위험하다.

---

## 23. Singleton Bean은 멀티스레드 환경에서 안전한가요?
### 답변(면접용)
- 인스턴스가 하나라는 뜻이지 스레드 안전을 보장하지 않는다.
- 무상태(stateless)로 만들면 안전해지기 쉽다.
- 공유 상태가 필요하면 동기화/불변/스코프 분리 등을 고려한다.

---

## 24. Service Bean을 무상태로 설계해야 하는 이유는 무엇인가요?
### 답변(면접용)
- 싱글톤 서비스는 여러 요청이 동시에 접근하므로 상태가 섞일 수 있다.
- 상태를 메서드 지역 변수/요청 스코프 등으로 내리면 안전하다.
- 상태가 필요하면 저장소(DB/캐시)나 별도 컨텍스트로 분리한다.

---

## 25. Service 클래스에 요청별 상태를 instance 변수로 저장하면 어떤 문제가 생기나요?
### 답변(면접용)
- 다른 요청과 상태가 섞여 데이터 오염/보안 문제가 생길 수 있다.
- 동시성 버그가 비결정적으로 발생한다.
- 요청별 상태는 파라미터로 전달하거나 요청 스코프에 둔다.

---

## 26. Spring MVC에서 여러 요청이 같은 Controller 인스턴스를 공유해도 되는 이유는 무엇인가요?
### 답변(면접용)
- 컨트롤러도 보통 싱글톤이고, 요청 상태를 인스턴스 필드에 두지 않는다는 전제다.
- 요청 데이터는 메서드 파라미터/요청 스코프에서 처리된다.
- 필드에 상태를 두면 즉시 문제가 된다.

---

## 27. Thread-safe한 Spring Bean을 만들려면 어떻게 설계해야 하나요?
### 답변(면접용)
- 기본은 무상태 + 불변 데이터 구조를 선호한다.
- 공유 상태가 필요하면 동기화/락/원자 연산/동시성 컬렉션을 사용한다.
- 스코프(요청/프로토타입)로 상태를 분리하는 것도 방법이다.

---

## 28. Request Scope Bean은 언제 사용하나요?
### 답변(면접용)
- 요청 단위로 유지해야 하는 상태(요청 ID, 사용자 컨텍스트)를 담을 때 사용한다.
- 싱글톤에 주입 시 프록시로 동작해 주의가 필요하다.
- 과도한 사용은 복잡도를 키워 필요한 경우에만 쓴다.

---

## 29. Prototype Scope Bean은 Singleton Bean에 주입될 때 어떤 문제가 생기나요?
### 답변(면접용)
- 주입 시점에 한 번 생성된 인스턴스가 계속 사용될 수 있다.
- “요청마다 새로 만들고 싶다”는 기대와 다르게 동작한다.
- Provider/ObjectFactory 등을 써서 필요 시점에 생성하게 한다.

---

## 30. `ObjectProvider`는 어떤 상황에서 사용하나요?
### 답변(면접용)
- 지연 조회/선택적 의존성 주입이 필요할 때 쓴다.
- 프로토타입을 매번 새로 만들거나, 순환 참조를 완화하는 데도 쓰인다.
- 남용하면 의존 관계가 숨겨져 설계가 불명확해질 수 있다.

---

## 31. Spring Boot에서 REST API를 구현할 때 Controller의 책임은 어디까지라고 보나요?
### 답변(면접용)
- 요청/응답 변환(DTO), 상태코드 설정, 입력 검증 같은 웹 계층 책임을 맡는다.
- 비즈니스 규칙은 서비스 계층으로 내려 응집도를 유지한다.
- 얇은 컨트롤러(Thin Controller)가 유지보수에 유리한 편이다.

---

## 32. Spring Boot에서 HTTP Method별 요청을 매핑하는 어노테이션은 무엇인가요?
### 답변(면접용)
- `@GetMapping`, `@PostMapping`, `@PutMapping`, `@PatchMapping`, `@DeleteMapping` 등이 대표적이다.
- 공통으로는 `@RequestMapping(method=...)`도 있다.
- 경로 변수/쿼리/바디 매핑 어노테이션과 함께 사용한다.

---

## 33. Spring Boot에서 상태 코드와 응답 바디를 함께 제어하려면 어떤 방식을 사용하나요?
### 답변(면접용)
- `ResponseEntity`로 상태 코드/헤더/바디를 함께 설정할 수 있다.
- 예외 기반으로 처리하려면 예외 매핑(@ControllerAdvice)과 조합한다.
- 표준 응답 포맷이 있으면 바디 래핑 전략도 같이 정한다.

---

## 34. Spring Boot에서 예외별 HTTP Status Code를 어떻게 매핑하나요?
### 답변(면접용)
- `@ControllerAdvice` + `@ExceptionHandler`로 예외를 잡아 응답을 만든다.
- 예외 클래스에 `@ResponseStatus`로 기본 상태를 지정할 수도 있다.
- 공통 에러 코드/메시지 정책을 팀 기준으로 유지한다.

---

## 35. Spring Boot에서 CORS 설정을 전역으로 적용하는 방법은 무엇인가요?
### 답변(면접용)
- MVC 레벨에서는 `WebMvcConfigurer#addCorsMappings`로 전역 설정이 가능하다.
- Spring Security 사용 시 Security 필터 체인에서 CORS를 켜고 소스(CorsConfigurationSource)를 맞춘다.
- 환경별 허용 origin은 설정으로 분리하는 게 안전하다.

---

## 36. Spring Security가 적용된 상황에서 CORS 설정 위치를 잘못 잡으면 어떤 문제가 생기나요?
### 답변(면접용)
- preflight(OPTIONS)가 인증/인가에 막혀 401/403이 날 수 있다.
- MVC CORS만 설정해도 Security 필터에서 차단되면 실패한다.
- “CORS는 브라우저 정책, Security는 서버 정책”이라 둘을 함께 맞춰야 한다.

---

## 37. `@RequestBody`, `@RequestParam`, `@PathVariable`의 차이는 무엇인가요?
### 답변(면접용)
- `@PathVariable`: URL 경로 변수, `@RequestParam`: 쿼리/폼 파라미터.
- `@RequestBody`: 요청 바디(JSON 등)를 객체로 역직렬화한다.
- 입력 검증과 에러 응답 정책을 같이 설계한다.

---

## 38. Validation은 어느 계층에서 처리하는 것이 좋나요?
### 답변(면접용)
- 기본 형식 검증은 컨트롤러(입력 DTO)에서 먼저 막는 게 일반적이다.
- 도메인 규칙/비즈니스 검증은 서비스/도메인 계층에서 처리한다.
- 에러 응답은 전역 예외 처리로 일관되게 만든다.

---

## 39. `@Valid`와 `@Validated`의 차이는 무엇인가요?
### 답변(면접용)
- 둘 다 Bean Validation을 트리거하지만, `@Validated`는 그룹 검증에 유리하다.
- `@Valid`는 표준(Jakarta) 어노테이션이다.
- 팀에서는 그룹 검증 필요 여부에 따라 선택한다.

---

## 40. Global Exception Handler는 왜 필요한가요?
### 답변(면접용)
- 예외를 한 곳에서 공통 포맷/상태코드로 매핑해 일관성을 만든다.
- 컨트롤러에 try/catch가 퍼지는 걸 막는다.
- 로깅/트레이싱 키를 함께 남기기 좋다.

---

## 41. `@ControllerAdvice`는 어떤 역할을 하나요?
### 답변(면접용)
- 여러 컨트롤러에 공통으로 적용되는 예외 처리/바인딩/모델 속성을 제공한다.
- `@ExceptionHandler`와 조합해 예외→응답 매핑을 담당한다.
- 범위를 패키지/어노테이션으로 제한할 수도 있다.

---

## 42. API 응답을 공통 포맷으로 감싸는 것의 장단점은 무엇인가요?
### 답변(면접용)
- 장점: 클라이언트 처리 일관성(코드/메시지/데이터)과 에러 표준화.
- 단점: HTTP 상태 의미를 흐릴 수 있고 단순 API도 과해질 수 있다.
- 최소 포맷만 표준화하고, 예외는 문서로 합의하는 게 안전하다.

---

## 43. API 버전 관리는 어떻게 할 수 있나요?
### 답변(면접용)
- URL(`/v1`), 헤더, 서브도메인 등 방식이 있다.
- 하위 호환 유지가 가능하면 버전 증가를 늦추고, 깨지는 변경이면 버전을 분리한다.
- 문서/클라이언트 배포 정책과 함께 결정한다.

---

## 44. REST API에서 세션을 잘 쓰지 않는 이유는 무엇인가요?
### 답변(면접용)
- 무상태로 만들면 서버 확장/로드밸런싱이 단순해진다.
- 세션 공유/동기화 비용과 장애 전파를 줄일 수 있다.
- 대신 인증/폐기/보안 모델을 다른 방식으로 설계해야 한다.

---

## 45. Spring Boot에서 `@ResponseStatus`와 `ResponseEntity`로 상태 코드를 제어하는 방식의 차이는 무엇인가요?
### 답변(면접용)
- `@ResponseStatus`는 고정된 상태 코드를 어노테이션으로 지정할 때 간단하다.
- `ResponseEntity`는 런타임 조건에 따라 상태/헤더/바디를 유연하게 바꿀 수 있다.
- “고정 규칙이면 어노테이션, 분기/헤더 제어면 ResponseEntity”로 정리할 수 있다.

---

## 46. Spring MVC에서 요청 하나가 처리되는 흐름을 설명해보세요.
### 답변(면접용)
- 필터 체인 → DispatcherServlet → 핸들러 매핑/어댑터 → 컨트롤러 실행 흐름이다.
- 파라미터 바인딩/검증 후 컨트롤러가 서비스 호출, 결과를 뷰/바디로 렌더링한다.
- 인터셉터는 컨트롤러 전/후, 필터는 서블릿 앞단에서 동작한다.

---

## 47. DispatcherServlet은 어떤 역할을 하나요?
### 답변(면접용)
- Spring MVC의 프론트 컨트롤러로, 요청을 받아 적절한 컨트롤러로 라우팅한다.
- 핸들러 매핑/어댑터, 예외 처리, 뷰 리졸버 등 MVC 구성 요소를 조정한다.
- 공통 처리(로깅, 예외, 바인딩)가 중앙화된다.

---

## 48. HandlerMapping과 HandlerAdapter는 어떤 역할을 하나요?
### 답변(면접용)
- HandlerMapping은 요청 URL/메서드에 맞는 핸들러(컨트롤러 메서드)를 찾는다.
- HandlerAdapter는 선택된 핸들러를 실제로 호출할 수 있게 실행을 중개한다.
- 다양한 컨트롤러 스타일을 확장 가능하게 만든 구조다.

---

## 49. Filter와 Interceptor의 차이는 무엇인가요?
### 답변(면접용)
- Filter는 서블릿 스펙 레벨로 요청/응답을 감싸고, MVC 바깥에서도 동작한다.
- Interceptor는 Spring MVC 레벨로 컨트롤러 전/후에 동작한다.
- 인증/로깅 등은 둘 다 가능하지만 적용 범위/정보 접근이 다르다.

---

## 50. ArgumentResolver는 어떤 역할을 하나요?
### 답변(면접용)
- 컨트롤러 메서드 파라미터를 특정 방식으로 만들어 주입하는 확장 지점이다.
- 인증 사용자, 요청 컨텍스트 같은 값을 편하게 주입할 수 있다.
- 남용하면 컨트롤러가 숨은 의존을 갖게 되어 문서화가 필요하다.

---

## 51. Tomcat은 요청을 어떤 방식으로 멀티스레드 처리하나요?
### 답변(면접용)
- 커넥터가 요청을 받아 워커 스레드 풀에서 요청 처리를 수행한다.
- 요청 하나당 스레드가 할당되는 모델이 기본(블로킹 I/O 기반)이다.
- 스레드 풀 설정과 대기 큐가 성능/안정성에 영향을 준다.

---

## 52. Tomcat request thread pool이 고갈되면 어떤 일이 발생하나요?
### 답변(면접용)
- 신규 요청이 대기하거나 타임아웃/거절로 이어질 수 있다.
- 지연이 늘면서 연쇄 타임아웃과 장애 전파가 발생한다.
- 병목(DB, 외부 API, 락, GC)을 먼저 찾아야 한다.

---

## 53. Spring MVC는 blocking I/O 기반인가요?
### 답변(면접용)
- 전통적인 Spring MVC + 서블릿 컨테이너 모델은 블로킹 I/O 기반이다.
- 요청 처리 스레드가 I/O 대기 동안 묶일 수 있다.
- 대안으로 WebFlux(리액티브)나 Virtual Thread 같은 선택지가 있다.

---

## 54. `@Async`는 어떤 상황에서 사용하나요?
### 답변(면접용)
- 요청 스레드를 오래 잡지 않도록 백그라운드 작업을 분리할 때 쓴다.
- 메일 발송, 비동기 처리, 느린 외부 호출 같은 작업이 대상이다.
- 트랜잭션/컨텍스트 전파와 스레드풀 튜닝이 중요하다.

---

## 55. `@Async` 사용 시 주의할 점은 무엇인가요?
### 답변(면접용)
- 스레드풀 크기/큐/거부 정책을 설정하지 않으면 폭주할 수 있다.
- 예외는 호출자 스레드로 바로 전파되지 않아 처리 전략이 필요하다.
- 트랜잭션/보안 컨텍스트가 자동 전파되지 않는 경우를 주의한다.

---

## 56. `@Async`가 같은 클래스 내부 호출에서 동작하지 않는 이유는 무엇인가요?
### 답변(면접용)
- Spring AOP 프록시 기반이라 “프록시를 통한 호출”에서만 적용된다.
- 자기 자신 메서드 호출은 프록시를 우회해 어노테이션이 적용되지 않는다.
- 분리된 빈으로 빼거나 프록시 경로로 호출하게 구조를 바꾼다.

---

## 57. ThreadPoolTaskExecutor 설정에서 corePoolSize, maxPoolSize, queueCapacity는 각각 무엇인가요?
### 답변(면접용)
- core: 기본적으로 유지하는 스레드 수.
- max: 큐가 가득 찼을 때 확장 가능한 최대 스레드 수.
- queueCapacity: 대기 큐 크기(초과 시 스레드 확장 또는 거부 정책).

---

## 58. 비동기 작업에서 예외 처리는 어떻게 해야 하나요?
### 답변(면접용)
- `Future/CompletableFuture`로 결과/예외를 수집하거나, 비동기 예외 핸들러를 둔다.
- 실패를 로그로만 남기지 말고 재시도/알림/보상 같은 정책을 정한다.
- 작업이 중요한 경우 영속 큐/잡 시스템을 고려한다.

---

## 59. Virtual Thread를 Spring Boot에서 사용할 때 기대 효과는 무엇인가요?
### 답변(면접용)
- 블로킹 I/O가 많을 때 더 많은 동시 요청을 적은 OS 스레드로 처리할 수 있다.
- 코드 스타일(동기 코드)을 유지하면서 동시성을 올릴 수 있다.
- 외부 자원(DB 커넥션 등) 병목은 그대로라 함께 튜닝해야 한다.

---

## 60. Virtual Thread를 사용해도 DB Connection Pool이 병목이 될 수 있는 이유는 무엇인가요?
### 답변(면접용)
- DB 커넥션 수는 제한되어 있고, 커넥션이 없으면 대기한다.
- 스레드가 많아도 커넥션/쿼리 처리량이 한계면 지연은 그대로다.
- 풀 크기/쿼리 최적화/트랜잭션 길이 관리가 핵심이다.

---

## 61. JPA를 사용하는 이유는 무엇인가요?
### 답변(면접용)
- 객체 중심으로 DB를 다뤄 개발 생산성을 높인다.
- 반복적인 CRUD와 매핑을 줄이고, 영속성 컨텍스트로 변경 감지를 지원한다.
- 대신 쿼리/성능 이슈를 이해하고 제어해야 한다.

---

## 62. ORM이란 무엇인가요?
### 답변(면접용)
- 객체(Object)와 관계형 DB(Relation) 간 매핑을 제공하는 기술이다.
- SQL을 완전히 숨기기보다는 “생산성과 일관성”을 높이는 도구로 본다.
- 복잡한 조회는 JPQL/QueryDSL/네이티브 쿼리로 보완한다.

---

## 63. JPA와 Hibernate의 차이는 무엇인가요?
### 답변(면접용)
- JPA는 표준 인터페이스(스펙)이고, Hibernate는 대표 구현체다.
- 구현체는 추가 기능/튜닝 옵션을 제공할 수 있다.
- 코드 의존은 가능하면 JPA 표준에 두는 게 이식성에 유리하다.

---

## 64. Entity는 왜 기본 생성자가 필요한가요?
### 답변(면접용)
- 프록시/리플렉션으로 객체를 생성할 수 있어야 한다.
- JPA가 엔티티를 로딩/프록시 생성할 때 기본 생성자를 사용한다.
- 접근 제한은 둘 수 있지만(JPA 요구 충족) 완전히 없으면 안 된다.

---

## 65. Entity에 setter를 무분별하게 열면 어떤 문제가 생기나요?
### 답변(면접용)
- 도메인 규칙이 깨지고, 어디서 상태가 바뀌는지 추적이 어려워진다.
- 변경 감지로 인해 의도치 않은 업데이트가 발생할 수 있다.
- 의미 있는 변경 메서드로 상태 변경을 통제하는 게 안전하다.

---

## 66. 영속성 컨텍스트란 무엇인가요?
### 답변(면접용)
- 엔티티를 1차 캐시로 관리하고 동일성 보장, 변경 감지를 제공하는 컨텍스트다.
- 트랜잭션 범위에서 엔티티 상태를 추적한다.
- flush 시점에 변경 사항이 DB에 반영된다.

---

## 67. 1차 캐시란 무엇인가요?
### 답변(면접용)
- 영속성 컨텍스트 내부에 엔티티를 ID로 캐시하는 공간이다.
- 같은 트랜잭션에서 같은 엔티티 재조회 시 DB를 다시 치지 않을 수 있다.
- 트랜잭션 범위 캐시라서 전역 캐시와는 다르다.

---

## 68. Dirty Checking이란 무엇인가요?
### 답변(면접용)
- 트랜잭션 내 엔티티 변경을 스냅샷과 비교해 업데이트를 자동 생성하는 기능이다.
- 개발자가 명시적으로 update 쿼리를 쓰지 않아도 된다.
- 변경 범위/트랜잭션 길이를 관리하지 않으면 성능 이슈가 될 수 있다.

---

## 69. Flush와 Commit의 차이는 무엇인가요?
### 답변(면접용)
- flush는 영속성 컨텍스트 변경을 DB에 SQL로 반영하는 과정이다.
- commit은 트랜잭션을 최종 확정하는 것이다.
- flush가 commit과 항상 같은 시점은 아니며, 쿼리 실행 전 flush가 일어날 수 있다.

---

## 70. 지연 로딩과 즉시 로딩의 차이는 무엇인가요?
### 답변(면접용)
- 지연 로딩은 실제 사용 시점에 연관 객체를 조회한다.
- 즉시 로딩은 엔티티 로딩 시점에 연관 객체까지 함께 조회한다.
- 기본은 지연 로딩을 선호하고, 필요한 경우 fetch join 등으로 제어한다.

---

## 71. N+1 문제가 발생하는 이유는 무엇인가요?
### 답변(면접용)
- 연관 로딩 과정에서 부모 N개 조회 후 자식 조회가 N번 추가로 발생할 때 생긴다.
- 지연 로딩 + 반복 접근 패턴에서 흔하다.
- fetch join/배치 사이즈/엔티티 그래프 등으로 완화한다.

---

## 72. N+1 문제를 해결하는 방법은 무엇인가요?
### 답변(면접용)
- fetch join으로 필요한 연관을 한 번에 가져온다.
- 배치 로딩 설정으로 여러 연관을 묶어 조회한다.
- 조회 전용 DTO 쿼리로 필요한 데이터만 가져오는 것도 방법이다.

---

## 73. Fetch Join이란 무엇인가요?
### 답변(면접용)
- JPQL에서 연관 엔티티를 조인하면서 함께 로딩하는 기법이다.
- N+1을 줄이는데 효과적이다.
- 컬렉션 fetch join은 페이징/중복 행 문제를 주의해야 한다.

---

## 74. EntityGraph는 언제 사용하나요?
### 답변(면접용)
- 특정 조회에서 어떤 연관을 로딩할지 선언적으로 지정하고 싶을 때 쓴다.
- fetch join을 코드로 쓰지 않고 로딩 그래프를 분리할 수 있다.
- 복잡한 조회는 여전히 쿼리 최적화가 필요하다.

---

## 75. 대량 조회에서 JPA 영속성 컨텍스트가 메모리 문제를 만들 수 있는 이유는 무엇인가요?
### 답변(면접용)
- 조회한 엔티티가 1차 캐시에 쌓여 메모리가 증가한다.
- 변경 감지 스냅샷까지 유지하면 부담이 커진다.
- 페이징/스트리밍/flush&clear로 컨텍스트를 관리한다.

---

## 76. 대량 insert/update에서 `flush()`와 `clear()`를 사용하는 이유는 무엇인가요?
### 답변(면접용)
- 일정 단위로 DB에 반영하고 영속성 컨텍스트를 비워 메모리를 관리한다.
- 1차 캐시/스냅샷 누적을 방지한다.
- 배치 처리에서는 청크 전략과 함께 사용한다.

---

## 77. Open Session In View는 무엇이고 어떤 장단점이 있나요?
### 답변(면접용)
- 웹 요청 전체 동안 영속성 컨텍스트를 열어 지연 로딩을 가능하게 하는 전략이다.
- 장점은 개발 편의, 단점은 트랜잭션 경계가 흐려지고 성능/설계 문제가 숨겨질 수 있다.
- 운영에서는 필요 시점에만 로딩하도록 설계를 명확히 하는 게 안전하다.

---

## 78. Transaction이란 무엇이고 어떤 장단점이 있나요?
### 답변(면접용)
- 여러 DB 작업을 하나의 원자적 단위로 묶는 개념이다.
- 장점은 일관성/원자성 보장, 단점은 락/경합/지연이 늘 수 있다.
- 트랜잭션은 짧게, 필요한 범위만 잡는 게 기본이다.

---

## 79. `@Transactional`은 내부적으로 어떻게 동작하나요?
### 답변(면접용)
- AOP 프록시로 메서드 호출을 감싸 트랜잭션 시작/커밋/롤백을 수행한다.
- 예외 유형에 따라 롤백 규칙이 다를 수 있다.
- 자기 호출에서는 프록시가 적용되지 않는 점을 주의한다.

---

## 80. Transaction Context가 ThreadLocal과 관련되는 이유는 무엇인가요?
### 답변(면접용)
- 같은 스레드에서 DB 커넥션/트랜잭션 상태를 묶어 전달해야 한다.
- 이를 위해 스레드 로컬에 트랜잭션 리소스를 바인딩하는 방식이 흔하다.
- 스레드가 바뀌면 컨텍스트가 전파되지 않아 비동기 처리에서 주의가 필요하다.

---

## 81. Spring Security는 어떤 역할을 하나요?
### 답변(면접용)
- 인증/인가를 필터 체인 기반으로 제공하는 보안 프레임워크다.
- 세션/토큰/OAuth2 등 다양한 인증 방식을 지원한다.
- 보안 헤더/CSRF/CORS 같은 웹 보안 구성도 함께 다룬다.

---

## 82. Security Filter Chain이란 무엇인가요?
### 답변(면접용)
- 요청이 들어올 때 순서대로 실행되는 보안 필터들의 체인이다.
- 인증 처리, 인가 결정, 예외 처리 등이 필터로 구성된다.
- 필터 순서가 동작을 좌우하므로 설정 시 순서를 이해해야 한다.

---

## 83. 인증과 인가의 차이는 무엇인가요?
### 답변(면접용)
- 인증(Authentication)은 “누구인가”를 확인하는 것이다.
- 인가(Authorization)는 “무엇을 할 수 있는가” 권한을 확인하는 것이다.
- 인증 후 인가가 수행되는 흐름이 일반적이다.

---

## 84. JWT를 사용하는 이유는 무엇인가요?
### 답변(면접용)
- 무상태로 인증 정보를 전달해 확장에 유리하다.
- 마이크로서비스/분산 환경에서 중앙 세션 저장소 의존을 줄일 수 있다.
- 단, 폐기/탈취 대응이 어려워 운영 설계가 필수다.

---

## 85. Access Token과 Refresh Token의 차이는 무엇인가요?
### 답변(면접용)
- Access token은 API 호출에 사용하고, 만료를 짧게 가져간다.
- Refresh token은 access 재발급에 사용하며 보통 더 긴 수명을 가진다.
- refresh는 서버 저장/회전/폐기 정책이 중요하다.

---

## 86. Cookie 기반 JWT와 Authorization Header 기반 JWT의 차이는 무엇인가요?
### 답변(면접용)
- 쿠키는 자동 전송되어 편하지만 CSRF 대응이 필요하다.
- 헤더는 명시 전송이라 CSRF 표면은 줄지만 XSS/저장 방식 위험을 봐야 한다.
- 도메인 구성과 CORS 정책이 선택에 크게 영향을 준다.

---

## 87. Spring Security에서 CSRF 보호를 적용/비활성화할 때 판단 기준은 무엇인가요?
### 답변(면접용)
- 쿠키 기반 세션 인증처럼 브라우저 자동 전송이 있으면 CSRF 보호가 기본적으로 필요하다.
- 완전한 stateless API + 헤더 기반 토큰이면 CSRF 표면이 줄어 비활성화하는 경우가 있다.
- 위협 모델(XSS/CSRF)과 호출 주체(브라우저/서버)를 먼저 확정하고 결정한다.

---

## 88. HttpOnly, Secure, SameSite 옵션의 의미는 무엇인가요?
### 답변(면접용)
- HttpOnly: JS 접근 제한(XSS 피해 감소), Secure: HTTPS에서만 전송.
- SameSite: 크로스 사이트 요청에서 쿠키 전송 제한(CSRF 피해 감소).
- 인증 쿠키를 쓴다면 세 옵션을 조합해 기본 방어선을 만든다.

---

## 89. Spring Boot 2에서 3으로 넘어갈 때 가장 큰 변화는 무엇인가요?
### 답변(면접용)
- 기반이 Spring Framework 6/Jakarta EE로 바뀌며 `javax` → `jakarta` 전환이 크다.
- JDK 기준 상향, 일부 의존성/설정 변화가 동반된다.
- 마이그레이션은 패키지 변경과 호환성 검증이 핵심이다.

---

## 90. `javax.*`에서 `jakarta.*`로 바뀐 것이 왜 중요한가요?
### 답변(면접용)
- 패키지명이 바뀌어 컴파일/런타임에서 직접적인 호환성 깨짐이 발생한다.
- 라이브러리(서블릿, JPA 등)도 Jakarta 버전으로 맞춰야 한다.
- 마이그레이션 범위가 넓어 의존성 정합성 관리가 중요해진다.
