# Java Collection / 동시성 / 네트워크 - 답변 정리

원본 질문: [[CS공부/12_면접/전체 질문 DB/02-Java-Collection-동시성-네트워크-질문]]

---

## 1. `ArrayList`와 `LinkedList`의 차이는 무엇인가요?
### 답변(면접용)
- `ArrayList`는 연속 배열 기반이라 인덱스 접근이 빠르고, 중간 삽입/삭제는 비싸다.
- `LinkedList`는 노드 연결 기반이라 중간 삽입/삭제는 유리할 수 있지만, 인덱스 접근이 느리다.
- 캐시 지역성/메모리 오버헤드 때문에 실무에서는 `ArrayList`가 기본 선택인 경우가 많다.

---

## 2. `HashMap`의 내부 동작 방식을 설명해보세요.
### 답변(면접용)
- 키의 `hashCode()`로 버킷 인덱스를 정하고, 버킷에 엔트리를 저장한다.
- 조회는 해시 → 버킷 → 키 `equals()`로 최종 매칭한다.
- 리사이즈(rehash) 비용이 있어 초기 용량/부하율을 고려한다.

---

## 3. `HashMap`에서 해시 충돌은 어떻게 처리되나요?
### 답변(면접용)
- 같은 버킷에 여러 엔트리가 들어가는 충돌이 발생할 수 있다.
- 일반적으로 버킷 내에서 체이닝(연결 리스트/트리 구조)으로 여러 엔트리를 관리한다.
- 충돌이 많으면 조회가 느려지므로 좋은 해시/키 설계가 중요하다.

---

## 4. `HashMap`과 `ConcurrentHashMap`의 차이는 무엇인가요?
### 답변(면접용)
- `HashMap`은 동기화를 제공하지 않아 동시 수정에 안전하지 않다.
- `ConcurrentHashMap`은 동시 접근을 고려해 락 분할/원자 연산 등을 제공한다.
- `ConcurrentHashMap`도 “복합 연산(check-then-act)”은 원자 API로 처리해야 한다.

---

## 5. `HashSet`은 중복을 어떻게 판단하나요?
### 답변(면접용)
- 내부적으로 보통 해시 기반 구조를 사용해 중복을 판단한다.
- `hashCode()`로 후보를 좁히고, `equals()`로 최종 동일 여부를 결정한다.
- 따라서 `equals/hashCode` 계약을 깨면 중복/조회가 깨진다.

---

## 6. `TreeMap`과 `HashMap`의 차이는 무엇인가요?
### 답변(면접용)
- `TreeMap`은 정렬(비교) 기반으로 키 순서를 유지한다.
- `HashMap`은 해시 기반이라 정렬을 보장하지 않고 평균적으로 빠른 조회를 제공한다.
- 정렬/범위 조회가 필요하면 `TreeMap`, 단순 키 조회가 대부분이면 `HashMap`을 고려한다.

---

## 7. `PriorityQueue`는 내부적으로 어떤 자료구조를 사용하나요?
### 답변(면접용)
- 일반적으로 배열 기반 이진 힙(Heap)으로 구현된다.
- 최솟값/최댓값을 빠르게 꺼내는 데 유리하고, 임의 위치 삭제/검색은 비싸다.
- 정렬이 아니라 “우선순위 기준 top”을 유지하는 구조다.

---

## 8. `Stack`보다 `Deque`를 권장하는 이유는 무엇인가요?
### 답변(면접용)
- `Stack`은 오래된 동기화 기반 구현이라 현대 코드에서는 선호도가 낮다.
- `Deque`(예: `ArrayDeque`)는 스택/큐 양쪽 용도로 더 유연하고 성능이 좋은 편이다.
- 동시성이 필요하면 별도 동시성 자료구조를 고려한다.

---

## 9. `Iterator` 사용 중 Collection을 수정하면 어떤 문제가 생기나요?
### 답변(면접용)
- 일반 컬렉션은 순회 중 구조 변경 시 `ConcurrentModificationException`이 날 수 있다(fail-fast).
- 예외가 안 나더라도 순회 결과가 깨질 수 있다.
- 수정이 필요하면 `Iterator.remove()` 같은 안전한 방법이나 동시성 컬렉션을 쓴다.

---

## 10. `Collections.synchronizedList()`와 `CopyOnWriteArrayList`의 차이는 무엇인가요?
### 답변(면접용)
- `synchronizedList`는 모든 접근을 동기화해 단순하지만, 경쟁이 크면 느려질 수 있다.
- `CopyOnWriteArrayList`는 쓰기 시 복사 비용이 크지만, 읽기 위주에서 락 경쟁이 적다.
- 읽기/쓰기 비율과 데이터 크기를 보고 선택한다.

---

## 11. Java에서 스레드를 생성하는 방법에는 무엇이 있나요?
### 답변(면접용)
- `Thread` 상속, `Runnable` 구현, `Callable`+`Future`(보통 Executor로 실행) 등이 있다.
- 실무에서는 직접 생성보다 `ExecutorService`로 스레드 풀을 쓰는 경우가 많다.
- 생성 비용/자원 관리/예외 처리 정책을 한 곳에서 통제하기 좋다.

---

## 12. `Thread`, `Runnable`, `Callable`의 차이는 무엇인가요?
### 답변(면접용)
- `Thread`는 실행 단위를 표현하는 클래스, `Runnable`은 반환값 없는 작업 인터페이스다.
- `Callable`은 반환값/예외를 가질 수 있고, 결과는 `Future`로 받는다.
- 작업(What)과 실행(How)을 분리하려면 `Runnable/Callable`+Executor 조합이 낫다.

---

## 13. `Future`와 `CompletableFuture`의 차이는 무엇인가요?
### 답변(면접용)
- `Future`는 비동기 결과를 “조회(get)” 중심으로 다루며 조합이 불편하다.
- `CompletableFuture`는 콜백/조합(`thenApply`, `thenCompose`)이 가능해 비동기 파이프라인에 유리하다.
- 예외 처리/타임아웃/스레드풀 지정 같은 운영 포인트를 같이 설계해야 한다.

---

## 14. `ExecutorService`를 사용하는 이유는 무엇인가요?
### 답변(면접용)
- 스레드 생성/재사용을 풀로 관리해 비용을 줄이고 자원을 통제한다.
- 작업 큐/스레드 수/거부 정책 등으로 시스템 부하를 제어할 수 있다.
- 직접 스레드 생성은 누수/폭주/예외 처리 누락이 생기기 쉽다.

---

## 15. 스레드를 직접 생성하는 것보다 스레드 풀을 사용하는 이유는 무엇인가요?
### 답변(면접용)
- 스레드 생성/컨텍스트 스위칭 비용이 크고, 무제한 생성은 시스템을 망가뜨린다.
- 풀은 동시 실행량을 제한하고, 큐로 버퍼링해 과부하를 완화한다.
- 모니터링/튜닝 포인트가 명확해 운영이 쉽다.

---

## 16. CPU-bound 작업과 I/O-bound 작업에서 스레드 풀 크기 전략은 어떻게 달라지나요?
### 답변(면접용)
- CPU-bound는 코어 수 근처로 제한해 컨텍스트 스위칭을 줄이는 쪽이 일반적이다.
- I/O-bound는 대기 시간이 길어 더 많은 스레드를 둘 수 있지만, 병목(커넥션 풀 등)을 함께 본다.
- “스레드 수”보다 “병목 자원”을 먼저 찾는 게 안전하다.

---

## 17. `synchronized`는 어떤 문제를 해결하나요?
### 답변(면접용)
- 임계 구역을 보호해 동시 수정으로 인한 데이터 레이스를 막는다.
- 진입/탈출 시 가시성(메모리 배리어)도 확보한다.
- 락 경합이 크면 성능 저하/데드락 위험이 있어 범위를 최소화한다.

---

## 18. 객체 락과 클래스 락의 차이는 무엇인가요?
### 답변(면접용)
- 객체 락은 특정 인스턴스 단위로 잠그고, 인스턴스가 다르면 서로 영향이 없다.
- 클래스 락은 클래스(보통 `Class` 객체) 단위로 잠가 인스턴스가 달라도 공유 락이 된다.
- 락 범위가 커질수록 병렬성이 떨어진다.

---

## 19. `volatile`은 어떤 문제를 해결하나요?
### 답변(면접용)
- 변수의 가시성 문제를 줄여 “다른 스레드가 최신 값을 보게” 한다.
- 재정렬을 제한해 특정 순서 보장을 강화한다.
- 단, 복합 연산(증가 등)의 원자성은 보장하지 않는다.

---

## 20. `volatile`이 원자성을 보장하지 못한다는 말은 무슨 뜻인가요?
### 답변(면접용)
- `x++` 같은 연산은 읽기→계산→쓰기의 여러 단계라 중간에 끼어들 수 있다.
- `volatile`은 최신 값을 보게는 하지만, 여러 단계가 한 덩어리로 실행되진 않는다.
- 원자성이 필요하면 락이나 원자 클래스(예: `AtomicInteger`)를 쓴다.

---

## 21. `AtomicInteger`는 어떻게 동시성 문제를 해결하나요?
### 답변(면접용)
- CAS(Compare-And-Set) 같은 원자적 하드웨어 연산을 활용한다.
- 락 없이도 일부 연산(증가, 교체 등)을 안전하게 만들 수 있다.
- 경합이 매우 크면 CAS 재시도가 많아져 성능이 나빠질 수 있다.

---

## 22. CAS란 무엇인가요?
### 답변(면접용)
- “현재 값이 기대 값이면 새 값으로 바꾸는” 원자 연산이다.
- 실패하면 다시 읽고 재시도하는 방식으로 락 없이 동시성을 만든다.
- ABA 문제 같은 함정이 있어 상황에 따라 추가 장치가 필요하다.

---

## 23. `ReentrantLock`은 `synchronized`와 무엇이 다른가요?
### 답변(면접용)
- 공정성, tryLock, 락 인터럽트, 조건 변수 등 더 많은 제어 기능을 제공한다.
- `synchronized`는 문법 수준이라 사용이 단순하고, JVM 최적화 혜택을 받을 수 있다.
- 둘 다 올바른 범위/순서로 써야 데드락을 피한다.

---

## 24. Immutable 객체가 멀티스레딩에서 유리한 이유는 무엇인가요?
### 답변(면접용)
- 상태 변경이 없으니 락 없이 공유해도 데이터 레이스 위험이 낮다.
- 공유/캐시/재사용에 유리하고, reasoning이 쉬워진다.
- 대신 변경이 잦으면 새 객체 생성 비용이 늘 수 있다.

---

## 25. Java에서 `UnknownHostException`은 어떤 경우에 발생하나요?
### 답변(면접용)
- 도메인 이름을 IP로 해석(DNS 조회)하지 못할 때 발생한다.
- 잘못된 호스트명, DNS 장애, 네트워크 설정 문제 등이 원인 후보다.
- 재현 시 `nslookup/dig`와 애플리케이션 DNS 캐시 정책도 같이 본다.

---

## 26. Java HTTP Client에서 connect timeout과 read timeout의 차이는 무엇인가요?
### 답변(면접용)
- connect timeout: TCP 연결(또는 TLS 포함 연결 수립)이 일정 시간 내 성립하지 않으면 실패.
- read timeout: 연결 후 응답을 읽는 과정에서 일정 시간 데이터가 안 오면 실패.
- 둘을 구분해야 “연결 문제 vs 서버 처리 지연”을 나눠 진단할 수 있다.

---

## 27. 외부 API 호출 시 timeout을 설정하지 않으면 어떤 문제가 생기나요?
### 답변(면접용)
- 외부가 멈췄을 때 요청이 무기한 대기하며 스레드/커넥션 풀이 고갈될 수 있다.
- 장애가 전파되어 내 서비스 전체가 느려지거나 타임아웃이 연쇄로 발생한다.
- 타임아웃은 “기본값 + 의존성별 예외” 정책으로 관리하는 게 안전하다.

---

## 28. Java 애플리케이션에서 DNS 변경이 바로 반영되지 않을 수 있는 이유는 무엇인가요?
### 답변(면접용)
- DNS TTL로 인해 브라우저/OS/리졸버/라이브러리 계층에서 캐시가 남을 수 있다.
- JVM도 내부적으로 DNS 캐시를 둘 수 있어 즉시 반영이 안 될 수 있다.
- 장애 시 “어디에 캐시가 걸렸는지” 계층별로 확인한다(확인 필요).

---

## 29. Java에서 TLS 인증서 검증 실패 시 어떤 예외가 발생할 수 있나요?
### 답변(면접용)
- 핸드셰이크 단계에서 실패하면 `SSLHandshakeException` 류가 대표적이다.
- 원인은 만료/호스트명 불일치/SAN 없음/신뢰 체인 문제 등 다양하다.
- 클라이언트 truststore 설정과 서버 인증서 체인을 함께 점검한다.
