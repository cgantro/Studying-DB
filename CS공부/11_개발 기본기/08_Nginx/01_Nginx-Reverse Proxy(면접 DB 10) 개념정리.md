## Nginx 기본

### 개념
- Nginx는 이벤트 기반(event-driven) 웹 서버이자 리버스 프록시로 많이 사용된다.
- 실무에서 Nginx가 맡는 역할:
  - 외부 트래픽 수용(프론트 도어)
  - 리버스 프록시(업스트림으로 전달)
  - TLS 종료(HTTPS termination)
  - 정적 파일 서빙
  - 압축/캐싱/레이트 리밋 같은 공통 기능
- “웹 서버(Nginx)”와 “애플리케이션 서버(WAS, 예: Spring Boot 내장 Tomcat)”의 책임을 구분하는 것이 중요하다.

### 왜 필요한가
- 애플리케이션 서버를 직접 외부에 노출하면:
  - TLS/정적 파일/캐싱/압축 같은 공통 요구를 앱이 떠안게 되고,
  - 장애/성능/보안 설정이 분산되어 운영 난이도가 오른다.
- Nginx 앞단에서 공통 기능을 처리하면 앱은 비즈니스 로직에 집중할 수 있고, 운영 정책(타임아웃/로그/보안 헤더)도 한 곳에서 관리하기 쉬워진다.

### 동작 원리
- 워커 모델:
  - 마스터 프로세스가 설정 로드/워커 관리, 워커 프로세스가 요청 처리를 담당한다.
  - 이벤트 기반으로 많은 연결을 효율적으로 처리할 수 있다(구체 동작은 OS 이벤트 모델(epoll/kqueue 등)에 의존).
- 설정 구조:
  - `http { server { location { ... } } }` 블록 구조로 라우팅/정책을 정의한다.
  - `location` 매칭 규칙(정확/접두/정규식) 이해가 중요하다.

### 장단점 / 한계
- 장점: 높은 처리 효율, 성숙한 생태계, 설정 기반 운영
- 한계:
  - 설정이 복잡해지면 디버깅이 어려워질 수 있다.
  - 동적 서비스 디스커버리/복잡한 라우팅이 필요하면 별도 게이트웨이/서비스 메시가 더 적합할 수 있다.

### 대안 / 선택 기준
- 대안: Apache, Envoy, Traefik, API Gateway 제품 등.
- 선택 기준:
  - 필요한 기능(TLS, L7 라우팅, 관측, 인증 연동)
  - 운영 환경(Docker/K8s), 동적 라우팅 필요성
  - 팀의 운영 경험과 디버깅 역량

### 실무/디버깅 포인트
- “Nginx 문제”로 보이는 현상도 실제로는 업스트림(앱/DB/외부 API) 문제일 수 있다.
- 그래서 Nginx에서는 최소한:
  - access/error 로그를 남기고,
  - 업스트림 응답 시간/상태를 로깅할 수 있게 하고,
  - 타임아웃을 명확히 설정해 장애 전파를 제한한다.

### 프로젝트 연결
- Nginx를 실제로 운영한 프로젝트 근거가 있으면 연결한다: `확인 필요`

## Reverse Proxy / Load Balancing

### 개념
- 리버스 프록시는 “클라이언트 앞에서 서버를 대신 받아” 내부 서버(업스트림)로 요청을 전달하는 프록시다.
- 로드밸런싱은 여러 업스트림 인스턴스로 트래픽을 분산하는 기능이다.

### 왜 필요한가
- 앱 인스턴스를 여러 대로 늘리면(수평 확장) 외부 트래픽을 분산해야 한다.
- 장애/배포/스케일링에서 “클라이언트는 하나의 도메인만 바라보고” 내부 구성은 자유롭게 바뀌게 하는 것이 운영에 유리하다.

### 동작 원리
- `proxy_pass`:
  - location에서 지정한 업스트림으로 요청을 전달한다.
  - 전달 시 헤더(`Host`, `X-Forwarded-*`)가 어떻게 설정되는지에 따라 앱에서 원본 정보 인식이 달라진다.
- 로드밸런싱 알고리즘(개념):
  - 라운드로빈, 가중치, IP 해시, 최소 연결 등(정확한 지원/동작은 설정에 따라 다름).
- 헬스 체크:
  - 업스트림이 죽었을 때 빠르게 빼기 위한 장치가 필요하다(오픈소스 Nginx의 기능 범위는 구성에 따라 다를 수 있어 `확인 필요`).

### 장단점 / 한계
- 로드밸런싱은 확장성을 주지만, 세션/인증 방식에 따라 sticky session 같은 요구가 생길 수 있다.
- 앱이 상태를 서버 메모리에 들고 있으면(scale-out 시) 분산이 어려워지므로, 상태는 외부 저장소로 보내는 설계가 유리한 경우가 많다.

### 대안 / 선택 기준
- L4(LB) vs L7(프록시):
  - 단순 TCP 분산이면 L4도 가능하지만, 헤더/라우팅/인증/캐싱 같은 기능은 L7이 필요하다.
- 로드밸런서 선택은 “운영 환경(클라우드 LB/K8s ingress)”과 함께 결정한다.

### 실무/디버깅 포인트
- 502/504 같은 오류는 “Nginx가 나쁘다”가 아니라 업스트림 연결/응답 문제인 경우가 많다.
- 분산 환경에서는 access log에 upstream 주소/상태/응답 시간을 남겨야 원인 분리가 가능하다.

### 프로젝트 연결
- 다중 인스턴스/로드밸런싱 운영 경험 연결은 `확인 필요`

## Header / HTTPS / 보안

### 개념
- Nginx는 HTTPS(TLS) 종료 지점이 될 수 있고, 그때 인증서/프로토콜/보안 헤더를 중앙에서 관리할 수 있다.
- 헤더는 “클라이언트 정보 전달(Forwarded)”과 “보안 정책(CSP/HSTS 등)”에 핵심이다.

### 왜 필요한가
- TLS를 앱 인스턴스마다 구성하면 운영 복잡도가 급격히 올라간다.
- 보안 헤더/리다이렉트 정책을 프록시에서 통일하면 설정 누락을 줄일 수 있다.
- 프록시를 거치면 앱은 원본 IP/프로토콜을 잃기 쉬워, `X-Forwarded-For`, `X-Forwarded-Proto` 같은 헤더가 중요해진다.

### 동작 원리
- HTTPS termination:
  - 클라이언트↔Nginx 구간은 TLS로 보호되고, Nginx↔업스트림은 HTTP 또는 HTTPS가 될 수 있다.
  - 업스트림까지 HTTPS로 가면 내부 구간도 암호화되지만, 운영/인증서 관리 비용이 늘 수 있다.
- 보안 헤더(개념):
  - HSTS: HTTPS 강제
  - CSP: 스크립트/리소스 로딩 제한(XSS 완화)
  - X-Content-Type-Options 등
- 인증/인가:
  - Nginx 자체 인증(Basic auth 등) 또는 앱 보안(Spring Security 등)과의 책임 분리를 결정해야 한다.

### 장단점 / 한계
- 프록시에서 TLS/보안 헤더를 처리하면 통일성이 좋지만, 앱 요구사항과 불일치하면 장애가 날 수 있다.
- 업스트림이 HTTPS일 때 SNI/인증서 검증 같은 추가 설정이 필요할 수 있다(환경별로 다름, `확인 필요`).

### 대안 / 선택 기준
- 내부망까지 암호화할지:
  - 규정/보안 요구가 강하면 내부 TLS(mTLS 포함)를 고려한다.
  - 단일 호스트/신뢰된 내부망이면 복잡도를 줄이기 위해 HTTP로 두는 선택도 있다.

### 실무/디버깅 포인트
- HTTPS 문제는:
  - 인증서 만료/도메인 불일치/체인 문제/시간 동기화가 흔한 원인이다.
- 프록시 뒤에서 앱이 원본 프로토콜/클라이언트 IP를 알아야 한다면 forwarded 헤더를 신뢰하는 설정이 필요하다(프레임워크 쪽 설정 포함).

### 프로젝트 연결
- 인증서/HTTPS 운영 경험 연결은 `확인 필요`

## 정적 파일 / 압축 / 캐싱

### 개념
- Nginx는 정적 파일 서빙에 강하고, gzip/brotli 압축(구성에 따라), 캐싱/헤더 제어를 앞단에서 처리할 수 있다.
- 정적 파일은 앱 서버를 거치지 않게 분리하면 앱 리소스를 비즈니스 처리에 집중할 수 있다.

### 왜 필요한가
- 정적 리소스(이미지/JS/CSS)는 트래픽 비중이 커질 수 있고, 압축/캐싱이 효과적이다.
- 캐싱을 잘하면 지연과 서버 부하가 줄지만, 잘못하면 오래된 리소스가 남아 버그처럼 보일 수 있다.

### 동작 원리
- 압축:
  - 텍스트 계열(JS/CSS/JSON)은 압축 효과가 크지만, 이미 압축된 바이너리는 효과가 작다.
- 캐싱 헤더:
  - `Cache-Control`, `ETag`, `Last-Modified` 같은 메커니즘으로 클라이언트/중간 캐시 동작을 제어한다.
- 정적 파일 위치:
  - `root`/`alias` 설정으로 파일 경로 매핑이 결정되며, 실수하면 의도치 않은 파일 노출이 생길 수 있다.

### 장단점 / 한계
- 캐싱은 성능을 올리지만, 무효화(invalidation)가 어려울 수 있다.
- 압축은 CPU 비용이 있으므로, 대상 MIME/레벨을 조절해야 한다.

### 대안 / 선택 기준
- CDN:
  - 정적 자산은 CDN으로 보내고, 오리진은 Nginx/앱 서버로 두는 구성이 흔하다.
- 빌드 산출물에 해시를 붙여 캐시 무효화를 단순화하는 전략(예: `app.<hash>.js`)이 자주 쓰인다.

### 실무/디버깅 포인트
- “배포했는데 변경이 안 보임”은 캐시 문제일 수 있다.
  - 브라우저 캐시, CDN 캐시, 프록시 캐시를 분리해서 본다.

### 프로젝트 연결
- 정적 자산 캐싱/CDN 운영 경험 연결은 `확인 필요`

## WebSocket / 로그 / 트러블슈팅

### 개념
- WebSocket은 HTTP 업그레이드(Upgrade)로 시작해 장기 연결을 유지하는 프로토콜이다.
- 프록시(Nginx) 앞단에서 WebSocket을 프록시할 때는 “업그레이드 헤더”와 “타임아웃”이 중요하다.
- 트러블슈팅은 access/error 로그 + 업스트림 로그 + 네트워크 타이밍을 결합해 원인을 분리한다.

### 왜 필요한가
- 실시간 기능은 “연결이 길게 유지”되므로 타임아웃/버퍼/프록시 설정이 맞지 않으면 끊김이 잦아진다.
- 502/504 같은 오류는 사용자에게는 “서버가 안 된다”로 보이므로, 빠르게 원인을 분리할 수 있어야 한다.

### 동작 원리
- WebSocket 프록시(개념):
  - HTTP 요청에 `Upgrade: websocket`이 포함되고, 서버가 101 Switching Protocols로 전환한다.
  - 프록시가 업그레이드 헤더를 전달하지 않으면 연결이 실패할 수 있다.
- 로그:
  - access log에는 요청/상태/응답 시간, 가능하면 upstream 응답 시간/상태를 포함한다.
  - error log에는 업스트림 연결 실패/타임아웃 같은 원인이 남는다.

### 장단점 / 한계
- WebSocket은 강력하지만, 운영에서는 연결 수/메모리/하트비트/재연결 전략까지 고려해야 한다.
- 프록시 설정 하나로 끊김이 생길 수 있어 환경별 테스트가 필요하다.

### 대안 / 선택 기준
- 단방향 푸시가 목적이면 SSE가 더 단순할 수 있다.
- 양방향/실시간 상호작용이면 WebSocket이 적합한 경우가 많다.

### 실무/디버깅 포인트
- 502 vs 504:
  - 502는 업스트림 연결/응답 자체가 비정상(프로토콜/연결 실패)일 가능성이 높고,
  - 504는 업스트림 응답을 타임아웃으로 기다리다 실패한 경우가 많다(일반적인 해석, 환경에 따라 다를 수 있음).
- upstream 응답 시간이 길면:
  - 앱 처리 지연, DB 지연, 외부 API 지연, 또는 커넥션 풀 고갈을 의심한다.

### 프로젝트 연결
- WebSocket(STOMP) 사용 프로젝트 흐름이 기록돼 있다.

## 배포 구조

### 개념
- 흔한 배포 구조:
  - (외부) Nginx → (내부) 앱 서버 여러 대 → DB/캐시
  - 또는 Docker Compose에서 Nginx 컨테이너가 `backend` 같은 서비스로 프록시
- 프록시/앱/DB의 역할 경계를 명확히 해야 장애 분석이 빨라진다.

### 왜 필요한가
- 배포 구조를 모르면 “어디서 끊기는지”를 감으로 찍게 된다.
- 구조를 알면 DNS/방화벽/Nginx/업스트림/DB를 순서대로 확인할 수 있다.

### 동작 원리
- 헬스 체크:
  - 업스트림이 준비되기 전에 트래픽이 들어오면 실패하므로, 헬스 체크/리드니스가 필요하다.
- 무중단 배포:
  - 롤링/블루그린/카나리 등은 프록시가 트래픽을 어떻게 분배하는지와 결합된다.

### 장단점 / 한계
- 단일 서버는 단순하지만 SPOF가 된다.
- 다중 서버는 안정성이 좋아질 수 있지만, 설정/관측/배포가 복잡해진다.

### 대안 / 선택 기준
- 요구사항에 따라:
  - 단일 서버 + 빠른 롤백
  - 다중 서버 + 점진 배포
중 선택하고, 관측/롤백 체계를 세트로 준비한다.

### 실무/디버깅 포인트
- 장애 시 확인 순서 예시:
  - DNS 해석 → 방화벽/포트 → Nginx 로그 → 업스트림 로그/헬스 → DB/외부 의존

### 프로젝트 연결
- Docker 기반 배포 이슈/구조는 프로젝트 문서에 일부 기록돼 있다.

## DNS / Upstream

### 개념
- 업스트림에 도메인을 쓰면 DNS 조회 시점/캐시 정책에 따라 “IP 변경이 즉시 반영되지 않을 수 있다”.
- Docker Compose 환경에서는 업스트림을 서비스 이름으로 두는 패턴이 많고, 이때는 “동일 네트워크”와 “내부 DNS”가 핵심이다.

### 왜 필요한가
- 운영에서 IP가 바뀌는 것은 흔하다(오토스케일/교체/재배포).
- DNS/캐시를 모르면 “계속 옛날 서버로 간다” 같은 현상을 설명하기 어렵다.

### 동작 원리
- DNS 캐시:
  - OS/리졸버/애플리케이션이 캐시를 가질 수 있다.
  - Nginx도 설정/구성에 따라 도메인 해석 결과를 유지할 수 있어, IP 변경을 반영하려면 reload가 필요해질 수 있다(구체 동작은 설정과 버전에 따라 달라 `확인 필요`).
- `resolver`:
  - 런타임 DNS 재해석이 필요한 경우 `resolver` 지시어로 DNS 서버를 지정하는 패턴이 사용된다(정확한 적용 범위는 설정에 따라 다름).
- Docker 환경:
  - Nginx 컨테이너가 `backend`로 접근하려면 같은 Docker 네트워크에 있어야 이름 해석/연결이 된다.

### 장단점 / 한계
- DNS 기반 라우팅은 유연하지만, TTL/캐시/전파 때문에 “일시적 불일치”가 생길 수 있다.

### 대안 / 선택 기준
- 동적 서비스 디스커버리 요구가 크면:
  - 서비스 메시/디스커버리 시스템을 도입하거나,
  - 오케스트레이터(K8s) 서비스/엔드포인트를 사용하는 패턴을 고려한다.

### 실무/디버깅 포인트
- “업스트림 IP가 바뀌었는데 계속 옛 IP로 감”:
  - DNS TTL/캐시, Nginx reload 여부, 컨테이너 네트워크 변경을 순서대로 확인한다.
- 컨테이너 환경에서는 “호스트에서 되는지”가 아니라 “컨테이너 안에서 되는지”로 확인해야 한다.
