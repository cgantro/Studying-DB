# 디버깅

## 한 줄 정의
문제가 발생했을 때 **원인을 찾아 재현하고, 가설을 세우고, 검증해서, 다시 안 생기게 만드는 과정**이다.

## 왜 배우는가
- 버그는 기능을 만든 뒤에 반드시 나온다.
- 같은 에러라도 원인은 다를 수 있다.
- 디버깅을 잘하면 문제를 빨리 고치고, 비슷한 버그를 다시 막을 수 있다.
- 면접이나 실무에서는 "무엇을 만들 수 있냐"보다 "문제를 어떻게 푸냐"를 많이 본다.

## 디버깅을 한 문장으로 생각하기
디버깅은 **추측**이 아니라 **증거를 모아 원인을 좁혀 가는 과정**이다.

---

## 전체 흐름
```text
문제 발생
  ↓
증상 확인
  ↓
재현 가능하게 만들기
  ↓
범위 좁히기
  ↓
가설 세우기
  ↓
검증하기
  ↓
수정하기
  ↓
재발 방지하기
```

## 핵심 개념
- **재현**: 같은 문제가 다시 나오게 만드는 것
- **범위 좁히기**: 전체 시스템 중 어디서 문제가 생겼는지 줄여 가는 것
- **가설**: "이 부분이 문제일 것 같다"는 임시 추측
- **검증**: 로그, 테스트, 브레이크포인트, 실험으로 가설을 확인하는 것
- **재발 방지**: 테스트 추가, 리팩터링, 경계값 처리, 문서화

---

## 디버깅 사고방식
### 1) 증상과 원인을 분리한다
- 증상: 화면이 안 뜬다, 응답이 느리다, 값이 틀리다
- 원인: DB 쿼리 문제일 수도 있고, 네트워크 문제일 수도 있고, 입력 검증 문제일 수도 있다
- 증상만 보고 바로 원인을 단정하면 틀리기 쉽다

### 2) "무조건 맞는 답"을 먼저 찾지 않는다
- 처음부터 정답을 맞히려 하지 말고 후보를 좁힌다
- 한 번에 하나씩 바꾸고 결과를 본다
- 동시에 여러 개를 고치면 뭐가 원인이었는지 알 수 없다

### 3) 가장 최근 변경부터 의심한다
- 버그는 최근에 바꾼 코드에서 나오는 경우가 많다
- 다만 최근 변경이 항상 원인은 아니므로 증거로 확인해야 한다

### 4) 눈으로 보기보다 로그를 믿는다
- 사람은 흐름을 착각하기 쉽다
- 로그, 에러 메시지, 스택 트레이스가 가장 강한 증거다

---

## 디버깅 도구 맵

```text
버그 조사 도구
├─ 로그
├─ 브레이크포인트
├─ 테스트
├─ print / trace
├─ 재현 시나리오
└─ 비교 실험
```

### 로그
- 언제, 어떤 입력, 어떤 상태였는지 남긴다
- 운영 환경 문제를 확인할 때 특히 중요하다
- 로그는 많을수록 좋지 않고, **핵심이 보이도록** 남겨야 한다

### 브레이크포인트
- 특정 줄에서 실행을 멈추고 변수 상태를 본다
- 흐름이 복잡할 때 유용하다
- 조건부 브레이크포인트를 쓰면 더 효율적이다

### 테스트
- 버그를 재현하는 최소 테스트를 만들 수 있다
- 수정 후 다시 같은 문제가 안 생기는지 확인한다
- 회귀 방지에 아주 중요하다

### print / trace
- 가장 단순하지만 빠르다
- 작은 범위를 빨리 확인할 때 좋다
- 다만 너무 많이 찍으면 오히려 흐름이 복잡해진다

---

## 문제를 좁히는 방법
### 범위 좁히기 체크리스트
- 입력값이 문제인가?
- 특정 사용자/특정 환경에서만 발생하는가?
- 특정 시간대에만 발생하는가?
- 특정 브라우저/OS/버전에서만 발생하는가?
- 최근 배포 이후 생겼는가?
- 항상 발생하는가, 가끔 발생하는가?
- 같은 요청을 여러 번 보내면 항상 같은 결과가 나오는가?

### 아주 흔한 접근 순서
1. 재현한다
2. 로그를 본다
3. 입력값을 확인한다
4. 최근 변경을 본다
5. 관련 함수와 의존성을 확인한다
6. 최소한으로 고친다
7. 다시 재현해 본다

---

## 디버깅 흐름도
```text
[버그 발생]
      |
      v
[재현 가능?] -- 아니오 --> [재현 조건 만들기]
      |
     예
      v
[입력/상태 확인]
      |
      v
[가설 A/B/C 세우기]
      |
      v
[하나씩 검증]
      |
      v
[원인 찾음]
      |
      v
[최소 수정]
      |
      v
[다시 테스트]
      |
      v
[회귀 방지용 테스트/문서 추가]
```

## 버그 종류별로 보는 관점

### 1) 논리 버그
- 코드가 에러 없이 실행되지만 결과가 틀린 경우
- 예: 조건문 반대로 작성, 비교 연산 실수, off-by-one
- 가장 찾기 어려운 유형 중 하나다

### 2) 런타임 에러
- 실행 중 예외가 터지는 경우
- 예: null 참조, 배열 범위 초과, 타입 오류, divide by zero
- 스택 트레이스를 먼저 본다

### 3) 성능 버그
- 정답은 맞지만 너무 느린 경우
- 예: 중복 쿼리, 불필요한 반복문, 인덱스 미사용, 무한 대기
- 어디서 시간이 쓰이는지 측정해야 한다

### 4) 환경 문제
- 내 컴퓨터에서는 되는데 서버에서는 안 되는 경우
- 예: 버전 차이, 권한, 네트워크, 환경 변수, 인코딩 차이
- 코드보다 배포 환경부터 확인해야 할 때가 많다

### 5) 동시성 버그
- 여러 작업이 동시에 실행될 때만 나타나는 버그
- 예: race condition, deadlock, lost update
- 재현이 어렵고 로그가 특히 중요하다

## 자주 쓰는 디버깅 전략

### A. 이분법적으로 줄이기
- 문제 구간을 반씩 잘라 본다
- "이 함수 전까지는 정상인가?"
- "이 API 전과 후 중 어디서 깨지는가?"

### B. 비교하기
- 정상 동작하는 케이스와 실패 케이스를 비교한다
- 입력, 환경, 시간, 사용자, 버전을 비교한다

### C. 최소 재현 만들기
- 실제 앱 전체가 아니라 최소 코드로 문제를 재현한다
- 재현 코드가 짧을수록 원인 찾기가 쉬워진다

### D. 하나만 바꾸기
- 조건을 여러 개 바꾸지 않는다
- 한 번에 한 변수만 바꿔야 원인과 결과를 연결할 수 있다

### E. 의심 목록 만들기
- 입력값 문제
- 상태 문제
- 캐시 문제
- DB 문제
- 네트워크 문제
- 권한 문제
- 타이밍 문제

## 초보자도 이해하기 쉬운 예시

### 예시 1: 버튼을 눌러도 화면이 안 바뀜
가능한 원인:
- 클릭 이벤트가 연결되지 않음
- 상태 업데이트가 안 됨
- API 응답은 왔는데 렌더링 조건이 잘못됨
- CSS 때문에 바뀌었는데 안 보임

어떻게 확인할까:
- 클릭 로그 찍기
- 상태 값 출력하기
- 네트워크 응답 보기
- 렌더 조건 확인하기

### 예시 2: 로그인은 성공했는데 메인 페이지가 안 열림
가능한 원인:
- 토큰 저장 실패
- 리다이렉트 경로 오류
- 권한 체크 로직 문제
- 서버 응답은 성공인데 프론트 분기 문제

### 예시 3: 가끔만 주문이 중복 생성됨
가능한 원인:
- 버튼 연타
- API 중복 호출
- 트랜잭션 처리 문제
- 동시성 문제
- 재시도 로직 문제

## 비교표 느낌으로 정리한 디버깅 도구

### 로그 vs 브레이크포인트 vs 테스트
- **로그**
  - 장점: 운영 환경에서도 확인 가능
  - 단점: 미리 심어 두지 않으면 놓칠 수 있음
  - 잘 맞는 상황: 서버 문제, 재현 어려운 문제

- **브레이크포인트**
  - 장점: 현재 상태를 자세히 볼 수 있음
  - 단점: 실시간 환경에서는 쓰기 어렵다
  - 잘 맞는 상황: 로컬 디버깅, 흐름 추적

- **테스트**
  - 장점: 재현과 회귀 방지에 강함
  - 단점: 먼저 재현 코드를 만들어야 한다
  - 잘 맞는 상황: 수정 후 검증, 반복 발생 버그

### print vs 로그
- **print**
  - 빠르고 가볍다
  - 임시 확인에 좋다
- **로그**
  - 레벨과 형식을 관리할 수 있다
  - 나중에 분석하기 좋다

## 실전 디버깅 순서 예시

### 서버 에러가 뜰 때
1. 에러 메시지 전체를 읽는다
2. 스택 트레이스를 확인한다
3. 어떤 입력에서 터지는지 본다
4. 최근 배포 내용을 확인한다
5. 관련 로그를 찾는다
6. DB, 네트워크, 환경 변수 여부를 본다
7. 최소 수정 후 다시 확인한다

### 프론트 화면이 안 바뀔 때
1. 이벤트가 실제로 호출되는지 본다
2. 상태 값이 바뀌는지 본다
3. API 응답이 정상인지 본다
4. 렌더 조건이 맞는지 본다
5. 스타일 때문에 안 보이는지 본다

### 느린 쿼리가 있을 때
1. EXPLAIN을 본다
2. 인덱스를 타는지 확인한다
3. WHERE/JOIN 조건을 본다
4. 필요 없는 컬럼/정렬이 없는지 본다
5. 데이터 양과 실행 횟수를 본다

## 디버깅할 때 자주 하는 실수
- 증상만 보고 너무 빨리 결론 내리기
- 한 번에 여러 개를 고치기
- 로그 없이 감으로만 판단하기
- 정상 케이스와 실패 케이스를 비교하지 않기
- 재현되지 않는 버그를 억지로 추측으로만 고치기
- 수정 후 회귀 테스트를 안 하기
- 환경 차이를 무시하기

- [ ] 수정 후 다시 같은 문제가 안 생기는지 확인했는가?
- [ ] 재발 방지용 테스트를 추가했는가?

## 면접 포인트
- 디버깅은 "문제를 고치는 과정"이 아니라 "원인을 좁혀 증명하는 과정"이라고 말하면 좋다.
- 실제 사례를 하나 준비해 두면 좋다.
- 문제 해결 과정은 보통 아래 순서로 말한다.

```text
문제 확인 → 재현 → 원인 후보 설정 → 검증 → 수정 → 재발 방지
```

- 면접에서는 결과보다 **어떻게 접근했는지**를 더 본다.
- "로그를 어떻게 설계했는지", "왜 그 원인을 의심했는지"를 설명할 수 있어야 한다.

---

## 한 번 더 요약
- 디버깅은 추측이 아니라 검증이다.
- 재현이 가능해야 원인을 찾기 쉽다.
- 한 번에 하나씩 범위를 좁혀야 한다.
- 수정 후에는 재발 방지까지 해야 진짜 해결이다.

## 초보자용 한 줄 기억법
- 디버깅은 **문제가 왜 생겼는지 증거를 모아 찾아가는 과정**이다.
## 더 깊게 보기
- 개발 기본기는 협업, 안정성, 운영을 버티게 해 주는 실전 기술이다.
- 디버깅의 핵심은 "증상을 줄여서 원인을 보이게 만드는 것"이다.
- 문제가 생긴 순간부터 바로 고치기보다, 재현 조건을 고정해야 한다.
- 한 번에 여러 수정을 하면 어떤 수정이 원인이었는지 알 수 없다.
- 운영 환경에서는 로그가 가장 중요한 증거인 경우가 많다.
- 동시성 버그와 환경 버그는 재현이 어려워서 더 체계적으로 접근해야 한다.

## 흐름도
```text
문제 발생
  ↓
재현 가능?
  ├─ 아니오 → 재현 조건 만들기
  └─ 예
      ↓
로그/상태 확인
      ↓
가설 세우기
      ↓
하나씩 검증
      ↓
수정
      ↓
재발 방지
```

## 예시를 조금 더
- 버튼 클릭이 안 되면 이벤트 등록, 상태 변경, 렌더링 조건 순으로 본다.
- 로그인 후 이동이 안 되면 토큰 저장과 라우팅 분기를 확인한다.
- DB 중복 저장은 버튼 중복 클릭, 재시도, 트랜잭션 처리까지 의심한다.

## 자주 헷갈리는 점
- 증상과 원인을 같은 것으로 착각한다.
- 감으로 고치고 확인을 생략한다.
- 수정 후 회귀 테스트를 하지 않는다.

## 면접 예상 질문
- 문제를 처음 발견했을 때 가장 먼저 무엇을 확인하나요?
- 재현이 어려운 버그는 어떻게 접근하나요?
- 디버깅 후 재발 방지를 위해 무엇을 하나요?

## 초보자용 기억법
- 디버깅: "이게 왜 필요한지"까지 설명할 수 있으면 이해한 것이다.
