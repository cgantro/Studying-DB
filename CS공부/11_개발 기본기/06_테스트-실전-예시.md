# 테스트 실전 예시

## 왜 따로 보는가
이 페이지는 테스트를 **실제로 어떻게 생각하고 쓰는지**를 보여 준다.

테스트는 정의를 외우는 것보다, 문제를 보고 "어떤 테스트를 써야 하지?"를 판단하는 게 더 중요하다.

---

## 1. 실전 작성 순서
가장 기본 흐름은 다음과 같다.

```text
요구사항 읽기
  ↓
무엇을 보장해야 하는지 말로 바꾸기
  ↓
성공/실패/경계값 찾기
  ↓
테스트 데이터 준비
  ↓
실행
  ↓
결과 검증
  ↓
실패 시 원인 수정
  ↓
회귀 방지용 테스트 유지
```

---

## 2. 예시 1: 합계 계산
### 요구사항
두 숫자를 더한 값을 반환해야 한다.

### 생각할 것
- 양수 + 양수
- 0 포함
- 음수 포함
- 큰 수

### 테스트 포인트
- `1 + 2 = 3`
- `0 + 5 = 5`
- `-1 + 1 = 0`

### 왜 쉬운가
입력과 결과가 단순해서 테스트 구조를 익히기 좋다.

---

## 3. 예시 2: 할인 정책
### 요구사항
- 10,000원 이상이면 10% 할인
- 10,000원 미만이면 할인 없음

### 테스트 포인트
- 9,999원은 할인 안 됨
- 10,000원은 할인 됨
- 10,001원도 할인 됨

### 핵심
경계값은 반드시 본다.

---

## 4. 예시 3: 로그인
### 요구사항
아이디와 비밀번호가 맞아야 로그인 성공.

### 성공 케이스
- 올바른 아이디와 비밀번호
- 토큰 발급 여부
- 이후 화면 이동 여부

### 실패 케이스
- 비밀번호 틀림
- 아이디 없음
- 입력 비어 있음
- 권한 없는 사용자

### 주의
로그인은 단순 반환값보다 **흐름**이 중요할 수 있다.

---

## 5. 예시 4: 주문 생성
### 요구사항
주문을 생성하면 저장되고 재고가 줄어야 한다.

### 봐야 할 것
- 저장이 됐는가?
- 재고 차감이 됐는가?
- 결제 실패 시 롤백되는가?
- 중복 요청이 막히는가?

### 어떤 테스트가 적절한가
- 로직 자체는 단위 테스트
- DB와 함께 보는 건 통합 테스트
- 사용자가 구매하는 흐름은 E2E

---

## 6. 실전에서 자주 쓰는 패턴
### Arrange / Act / Assert
```text
Arrange: 준비
Act: 실행
Assert: 검증
```

이 3단계가 보이면 테스트 코드는 읽기 쉬워진다.

### Given / When / Then
BDD 스타일로도 많이 쓴다.

- Given: 주어진 상황
- When: 실행
- Then: 결과

둘은 이름만 다르고 본질은 비슷하다.

---

## 7. 테스트 이름 잘 짓는 법
좋은 이름은 읽자마자 의도를 알 수 있어야 한다.

### 나쁜 예
- `test1`
- `check`
- `userTest`

### 좋은 예
- `주문_금액이_0이면_생성되지_않는다`
- `비밀번호가_틀리면_로그인에_실패한다`
- `장바구니가_비어있으면_결제할_수_없다`

---

## 8. 자주 부딪히는 실수
### 1) 구현 내부를 너무 많이 검사함
리팩터링할 때마다 테스트가 깨진다.

### 2) 공유 상태를 씀
- 이전 테스트가 남긴 DB 데이터
- 전역 변수
- 캐시

### 3) 외부 시간에 의존함
현재 시각에 따라 결과가 달라지면 테스트가 불안정해진다.

### 4) 랜덤값을 그대로 씀
결과가 매번 달라진다.

### 5) 실패 메시지가 약함
깨졌을 때 왜 깨졌는지 모르겠다.

---

## 9. 테스트하기 어려운 코드의 신호
- 함수가 너무 길다
- 조건문이 너무 많다
- DB/HTTP/파일 처리가 한 함수에 섞여 있다
- 상태가 여기저기 흩어져 있다
- 한 함수가 여러 일을 한다

이런 코드는 테스트보다 먼저 구조 개선이 필요할 수 있다.

---

## 10. 테스트가 설계를 돕는 방식
테스트는 단순 검사 도구가 아니라 설계 도구이기도 하다.

예:
- DB 접근을 분리하자
- 시간 조회를 주입하자
- 계산 로직과 I/O를 나누자
- 입력 검증을 한 곳에 모으자

즉, 테스트를 쓰다 보면 자연스럽게 더 잘 나뉜 코드로 간다.

---

## 11. 실전 판단 체크
다음 질문에 답할 수 있으면 테스트 이해이 올라간다.

- 이 코드는 무엇을 보장해야 하는가?
- 성공과 실패는 무엇인가?
- 경계값은 어디인가?
- 외부 의존성은 무엇인가?
- 테스트가 느려지는 이유는 무엇인가?
- 실패했을 때 어디를 먼저 봐야 하는가?

---

## 12. 한 줄 요약
- 테스트는 요구사항을 코드로 다시 확인하는 과정이다.
- 성공 케이스만 보지 말고 실패와 경계값을 봐야 한다.
- 테스트 이름이 읽히면 코드도 읽기 쉬워진다.
- 테스트하기 어려운 코드는 설계 신호를 보여 준다.
