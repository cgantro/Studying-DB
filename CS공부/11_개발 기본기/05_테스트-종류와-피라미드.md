# 테스트 종류와 피라미드

## 왜 따로 보는가
테스트는 전부 같은 것이 아니다. **무엇을 확인하느냐**, **얼마나 넓게 보느냐**, **얼마나 자주 돌리느냐**에 따라 역할이 다르다.

이 노트는 테스트의 큰 그림을 정리한 보조 페이지다.

---

## 1. 테스트 종류를 나누는 기준
테스트 종류는 보통 다음 기준으로 구분한다.

- 범위가 얼마나 큰가?
- 실제 시스템에 얼마나 가까운가?
- 실행이 얼마나 빠른가?
- 실패 원인을 얼마나 쉽게 찾을 수 있는가?

정리하면 이렇게 볼 수 있다.

```text
작은 범위 / 빠름 / 원인 찾기 쉬움 → 단위 테스트
중간 범위 / 적당한 속도 / 연결 검증 → 통합 테스트
큰 범위 / 느림 / 실제 흐름 확인 → E2E 테스트
```

---

## 2. 단위 테스트
### 정의
가장 작은 기능 단위를 검증하는 테스트다.

### 무엇을 보나
- 함수가 올바른 값을 반환하는가?
- 조건 분기가 맞는가?
- 예외를 올바르게 던지는가?

### 장점
- 빠르다.
- 실패 원인이 분명하다.
- 많이 쌓아 두기 좋다.

### 단점
- 실제 시스템 연결 문제는 놓칠 수 있다.
- 과하게 쪼개면 구현 세부사항에 묶일 수 있다.

### 예시
- 세금 계산 함수
- 문자열 정리 함수
- 유효성 검사 함수

---

## 3. 통합 테스트
### 정의
두 개 이상 구성요소가 잘 연결되는지 보는 테스트다.

### 무엇을 보나
- 서비스와 DB가 잘 연결되는가?
- API 요청이 저장소까지 잘 전달되는가?
- 모듈 간 데이터 흐름이 맞는가?

### 장점
- 실제와 더 가깝다.
- 단위 테스트가 놓치는 문제를 잡는다.

### 단점
- 준비가 복잡하다.
- 속도가 느려질 수 있다.
- 실패 원인을 찾기 더 어렵다.

### 예시
- 회원가입 시 DB 저장 확인
- 주문 생성 후 재고 차감 확인

---

## 4. E2E 테스트
### 정의
사용자가 실제로 서비스를 쓰는 흐름 전체를 검증하는 테스트다.

### 무엇을 보나
- 로그인부터 결제까지 흐름이 이어지는가?
- 사용자가 보는 화면이 예상대로 바뀌는가?
- 프론트/백엔드/DB가 함께 맞물리는가?

### 장점
- 가장 현실적이다.
- 사용자 입장에서 큰 문제를 잡는다.

### 단점
- 느리다.
- 불안정해지기 쉽다.
- 유지보수 비용이 높다.

### 예시
- 회원가입 → 로그인 → 상품 구매 → 주문 완료

---

## 5. 회귀 테스트
### 정의
기존에 잘 되던 기능이 바뀐 뒤에도 계속 잘 되는지 확인하는 테스트다.

### 포인트
회귀 테스트는 별도 종류라기보다 **목적**이다.

- 단위 테스트도 회귀 테스트가 될 수 있다.
- 통합 테스트도 회귀 테스트가 될 수 있다.
- E2E 테스트도 회귀 테스트가 될 수 있다.

---

## 6. 스모크 테스트
### 정의
핵심 기능이 최소한 살아 있는지 빠르게 확인하는 테스트다.

### 예시
- 서버가 켜지는가?
- 핵심 API가 응답하는가?
- 배포 후 첫 화면이 열리는가?

### 포인트
깊게 검증하기보다 **큰 고장 여부를 빨리 확인**하는 데 쓴다.

---

## 7. 테스트 더블
실제 대신 쓰는 가짜 객체나 가짜 구현이다.

### 종류
- Mock: 호출 여부를 검증
- Stub: 정해진 값을 반환
- Fake: 단순화된 실제 비슷한 구현
- Spy: 호출 기록을 남기며 일부 실제 동작

### 언제 쓰나
- DB, 네트워크, 파일처럼 느리거나 불안정한 의존성이 있을 때
- 실패 상황을 인위적으로 만들고 싶을 때
- 빠른 단위 테스트를 만들고 싶을 때

### 주의
너무 많이 쓰면 실제 동작과 멀어질 수 있다.

---

## 8. 테스트 피라미드
### 기본 구조
```text
         E2E
      ─────────
     통합 테스트
   ─────────────
    단위 테스트
 ────────────────
```

### 왜 이런 모양인가
- 아래로 갈수록 빠르고 싸다.
- 아래로 갈수록 실패 원인이 쉽다.
- 위로 갈수록 현실적이지만 비싸다.

### 실전 해석
- 단위 테스트를 많이 쌓는다.
- 통합 테스트는 핵심 연결만 잘 쌓는다.
- E2E는 사용자 흐름의 중요한 부분만 잡는다.

---

## 9. 피라미드가 깨지는 흔한 경우
### E2E가 너무 많은 경우
- 테스트가 느려진다.
- 조금만 바뀌어도 자주 깨진다.
- 유지보수가 힘들다.

### 단위 테스트만 많은 경우
- 실제 연결 문제를 놓칠 수 있다.
- 코드가 잘게 쪼개져도 실제 시스템 품질은 보장되지 않을 수 있다.

### 통합 테스트가 없는 경우
- 컴포넌트가 따로는 맞는데 붙이면 깨지는 문제가 생긴다.

---

## 10. 어떤 테스트를 언제 쓰는가
- **단위 테스트**: 계산, 분기, 입력 검증, 순수 로직
- **통합 테스트**: DB 저장, 메시지 큐, 외부 서비스 연동
- **E2E 테스트**: 로그인, 결제, 회원가입처럼 실제 사용자 흐름

---

## 11. 초보자용 판단법
### 질문 1
"이건 함수 하나의 규칙을 보는가?"
- 그렇다면 단위 테스트일 가능성이 높다.

### 질문 2
"여러 컴포넌트가 붙어서도 맞는가?"
- 그렇다면 통합 테스트일 가능성이 높다.

### 질문 3
"사용자 입장에서 끝까지 되는가?"
- 그렇다면 E2E 테스트다.

---

## 12. 한 줄 요약
- 단위 테스트는 작고 빠르다.
- 통합 테스트는 연결을 본다.
- E2E 테스트는 사용자 흐름을 본다.
- 피라미드는 아래가 넓고 위가 좁아야 현실적이다.
