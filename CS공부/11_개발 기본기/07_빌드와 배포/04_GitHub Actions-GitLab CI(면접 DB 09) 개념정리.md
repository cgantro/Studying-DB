## CI/CD 기본

### 개념
- CI(Continuous Integration): 변경을 자주 통합하고 자동 검증(빌드/테스트/정적 분석)을 돌리는 문화/체계.
- CD(Continuous Delivery/Deployment): 검증된 산출물을 자동으로 전달/배포하는 체계.
- 파이프라인은 “단계(stage) + 작업(job)”으로 구성되고, 실패를 빠르게 발견하는 게 목적이다.

### 왜 필요한가
- 수동 빌드/테스트/배포는 실수와 누락이 쌓인다.
- CI는 “품질 게이트(quality gate)”로 회귀 버그를 줄이고, CD는 “배포를 일상화”해 리스크를 낮춘다.
- 운영 관점에서 CI/CD는 “재현 가능한 산출물”을 만드는 과정이기도 하다.

### 동작 원리
- 기본 흐름:
  - 소스 변경 → 빌드/테스트 → 아티팩트 생성 →(선택) 이미지 빌드/푸시 → 배포
- 분리 원칙:
  - 빌드 산출물은 불변(immutable)로 만들고, 환경 설정은 런타임에 주입한다.
  - 파이프라인에서 비밀값은 시크릿으로 관리하고 로그/아티팩트에 남기지 않는다.

### 장단점 / 한계
- 자동화는 안정성을 높이지만, 파이프라인이 복잡해지면 유지보수 비용이 생긴다.
- CI가 느리면 팀 생산성이 떨어지므로 캐시/병렬/스코프 최적화가 필요하다.

### 대안 / 선택 기준
- 모든 테스트를 매번 돌릴지, 변경 범위 기반으로 나눌지(단위/통합/e2e) 전략이 필요하다.
- “완벽한 파이프라인”보다 “실제 실패를 줄이는 최소 게이트부터” 구축하는 게 보통 효과적이다.

### 실무/디버깅 포인트
- CI 실패는 크게:
  - 코드(테스트/빌드 깨짐)
  - 환경(네트워크/DNS/권한/리소스)
  - 의존성(패키지 레지스트리/외부 서비스)
로 나뉜다.
- 간헐 실패(flaky)는 원인을 기록하고 재현 가능한 형태로 좁혀야 한다(타임아웃/네트워크/병렬성).

### 프로젝트 연결
- GitLab CI/CD와 Docker 배포를 함께 다룬 프로젝트 기록이 있다.

## GitHub Actions

### 개념
- GitHub Actions는 GitHub 이벤트(push, PR 등)에 반응해 워크플로우를 실행하는 CI/CD 플랫폼이다.
- 주요 구성 요소(개념):
  - workflow: YAML로 정의된 자동화 흐름
  - job/step: 실행 단위/명령 단계
  - runner: 워크플로우를 실제로 실행하는 머신(호스티드/셀프 호스트)
  - artifact/cache: 산출물/캐시 저장

### 왜 필요한가
- GitHub에 코드가 있으면 PR 기반 검증/코드 리뷰 흐름과 자연스럽게 결합된다.
- “저장소 이벤트 → 자동 검증”을 표준화하기 쉬워 팀 룰을 만들기 좋다.

### 동작 원리
- runner 네트워크:
  - GitHub-hosted runner는 GitHub가 제공하는 환경이고, 네트워크/권한 제약이 있을 수 있다.
  - self-hosted runner는 사내/개인 환경에서 운영하므로 네트워크 접근 범위가 달라질 수 있다(보안/격리 정책에 따라).
- 캐시:
  - 의존성 다운로드를 줄이기 위해 캐시를 쓰지만, 캐시 오염/충돌이 실패 원인이 될 수 있다.
- 시크릿:
  - 레지스트리 로그인/배포 키 같은 민감정보는 시크릿으로 관리한다.

### 장단점 / 한계
- GitHub Actions는 편하지만, 복잡한 배포/네트워크 요구가 있으면 self-hosted 운영이 필요할 수 있다.
- self-hosted는 자유도가 크지만 보안/유지보수 부담이 커진다.

### 대안 / 선택 기준
- 조직/보안 요구에 따라:
  - 외부 네트워크 접근 제한이 있으면 self-hosted가 필요할 수 있다.
  - 간단한 CI면 hosted runner로 충분할 수 있다.

### 실무/디버깅 포인트
- “패키지 다운로드 실패”는:
  - DNS/프록시/방화벽, 레이트리밋, 레지스트리 장애를 의심한다.
- “권한 문제”는:
  - 토큰 스코프/레포 권한, 환경 보호 규칙(배포 보호)을 점검한다.

### 프로젝트 연결
- GitHub Actions를 실제로 어느 프로젝트에 적용했는지 연결은 `확인 필요`.

## GitLab CI

### 개념
- GitLab CI는 `.gitlab-ci.yml` 기반으로 파이프라인을 정의하고, runner가 실행한다.
- GitLab의 강점은 레지스트리/이슈/권한/배포 흐름이 한 플랫폼에 묶일 수 있다는 점이다.

### 왜 필요한가
- 사내/프로젝트 환경에 맞춘 runner 운영이 쉬운 경우가 많다(네트워크 접근/도구 설치).
- Docker-in-Docker 같은 구성으로 이미지 빌드/푸시를 파이프라인에 쉽게 넣을 수 있다(단, 보안/네트워크 고려 필요).

### 동작 원리
- runner/컨테이너:
  - runner executor 방식(shell/docker 등)에 따라 job이 격리되는 방식이 달라진다.
- service container:
  - 테스트용 DB/캐시를 service로 띄우고 job 컨테이너에서 접근할 수 있다(통신 모델은 executor 구성에 의존).
- 아티팩트/캐시:
  - 빌드 산출물/테스트 리포트는 아티팩트로 남기고, 의존성은 캐시로 최적화한다.

### 장단점 / 한계
- self-hosted runner는 네트워크/환경 제어가 가능하지만, 장애/보안 패치 책임이 생긴다.
- Docker-in-Docker는 편하지만, 네트워크/권한/TLS 설정이 꼬이면 자주 실패한다.

### 대안 / 선택 기준
- 이미지 빌드는:
  - DinD가 필요할지, 호스트 도커 소켓 바인딩이 가능할지(보안 고려), 별도 빌드 서버를 둘지 선택한다.

### 실무/디버깅 포인트
- “DinD 연결 실패”는:
  - 포트/TLS/환경변수 설정 문제일 가능성이 높다.
  - 네트워크가 막혀 registry 접근이 안 되는 경우도 흔하다.

### 프로젝트 연결
- GitLab CI DinD 이슈가 정리되어 있다.
  - [[프로젝트/Autowing-Car/02-CI-CD-배포-이슈]]

## Build / Test / Quality Gate

### 개념
- Quality gate는 “이 기준을 통과해야 merge/deploy 가능” 같은 정책이다.
- 일반적으로 아래를 묶는다:
  - 빌드 성공
  - 테스트(단위/통합)
  - 정적 분석/포맷팅(선택)
  - 커버리지(선택)
  - 보안 스캔(선택)

### 왜 필요한가
- 실패를 “PR 단계”에서 막지 않으면, 장애가 배포 후에 발견돼 비용이 커진다.
- 품질 기준을 자동화하면 리뷰가 “논리/설계”에 집중할 수 있다.

### 동작 원리
- 테스트 계층:
  - 단위 테스트는 빠르고 안정적이지만, 통합 문제(DB/네트워크)은 못 잡을 수 있다.
  - 통합/e2e는 느리고 flaky해질 수 있어, 최소 세트만 게이트로 두는 전략이 필요하다.
- 외부 의존:
  - 외부 API에 의존하는 테스트는 CI에서 불안정해지기 쉬워, mock/stub 서버로 대체하는 경우가 많다.

### 장단점 / 한계
- 게이트를 너무 무겁게 하면 개발 속도가 느려진다.
- 게이트를 너무 가볍게 하면 품질이 떨어진다.
- “팀 목표(속도 vs 안정)”에 맞춘 균형이 필요하다.

### 대안 / 선택 기준
- 권장 전략:
  - PR: 빠른 단위/핵심 통합만
  - main/nightly: 더 무거운 통합/e2e/보안 스캔
  - 배포 전: 환경 의존 테스트/스모크 테스트

### 실무/디버깅 포인트
- flaky 테스트는:
  - 타임아웃/병렬 실행/테스트 데이터 공유/외부 의존을 의심한다.
- 품질 게이트 실패는 “원인/대응 룰”을 문서화해야 운영 비용이 줄어든다.

### 프로젝트 연결
- 파이프라인 품질 게이트를 어떻게 구성했는지 프로젝트 근거는 `확인 필요`.

## Docker / 배포 연동

### 개념
- CI/CD에서 Docker 연동은 “컨테이너 이미지가 곧 배포 산출물”이 되는 패턴이다.
- 배포는 “어디에/어떻게/어떤 권한으로”가 핵심이고, SSH/레지스트리/시크릿이 결합된다.

### 왜 필요한가
- 이미지로 배포하면 환경 일관성이 좋아지고, 롤백도 이미지 태그로 단순해질 수 있다.
- 다만 네트워크/권한/시크릿 관리가 제대로 되지 않으면 오히려 위험해진다.

### 동작 원리
- 이미지 태그 전략:
  - 커밋 SHA/버전 태그로 불변성을 확보한다.
- 배포 단계:
  - SSH 접속 실패는 DNS/방화벽/키 권한/서버 상태로 나눠 점검한다.
  - 배포 전후 스모크 테스트/헬스 체크로 “살아있음”을 확인한다.

### 장단점 / 한계
- 자동 배포는 속도/일관성이 좋지만, 실패 시 영향이 커질 수 있어 롤백과 보호 장치가 필요하다.

### 대안 / 선택 기준
- 배포 방식 선택:
  - 단일 서버: 단순 SSH + compose
  - 여러 서버: 롤링/블루그린/카나리 같은 전략 고려

### 실무/디버깅 포인트
- 배포 실패 시 체크 순서:
  - 레지스트리 접근(로그인/권한)
  - 서버 접근(네트워크/키)
  - 컨테이너 실행(환경변수/포트/헬스)

### 프로젝트 연결
- 배포 자동화와 런타임 변수 주입 패턴이 기록돼 있다.
  - [[프로젝트/Autowing-Car/02-CI-CD-배포-이슈]]

## 운영 안정성 / 실무 질문

### 개념
- 운영 안정성은 “실패를 빨리 감지하고, 빨리 복구하고, 재발을 줄이는 체계”다.
- CI/CD에서 운영 안정성에 기여하는 요소:
  - 재현 가능한 산출물
  - 관측 가능한 배포(릴리즈 노트/메트릭 비교/로그)
  - 롤백 가능성
  - 시크릿/권한 통제

### 왜 필요한가
- 배포는 장애의 가장 큰 원인 중 하나다(변경 자체가 리스크).
- 그래서 배포를 두려워하지 않으려면, 자동화+관측+롤백이 세트로 필요하다.

### 동작 원리
- 네트워크 관점:
  - runner가 외부 레지스트리/패키지 저장소에 접근하지 못하면, DNS/프록시/방화벽/인증을 의심한다.
  - “간헐 실패”는 레이트리밋/네트워크 품질/캐시 오염을 의심한다.
- 서비스 컨테이너:
  - DB 준비 전에 테스트가 실행되면 실패하므로, readiness 확보(healthcheck/wait-for-it) 같은 전략이 필요하다.

### 장단점 / 한계
- 안정성을 높이는 장치는 파이프라인을 복잡하게 만들 수 있다.
- 하지만 장애 한 번의 비용이 크기 때문에, 반복되는 실패 유형부터 자동화/검증을 넣는 게 효과적이다.

### 대안 / 선택 기준
- 우선순위:
  - “자주 깨지는 것”과 “깨지면 영향이 큰 것”부터 자동화한다.
  - 외부 의존이 큰 테스트는 mock/stub로 대체하거나 분리한다.

### 실무/디버깅 포인트
- 실패 원인을 문서화하고 재발 방지 룰로 만들면(캐시 규칙, 타임아웃, 재시도 정책) 운영 비용이 크게 줄어든다.
- 네트워크/보안 실패는 “코드가 아니라 환경”임을 빠르게 판단하는 것이 중요하다.
