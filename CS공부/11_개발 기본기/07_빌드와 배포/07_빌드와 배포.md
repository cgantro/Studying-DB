# 빌드와 배포

## 한 줄 정의
소스 코드를 **실행 가능한 산출물로 만들고**, 그 결과를 **운영 환경에 안전하게 반영하는 과정**이다.

## 왜 배우는가
- 코드가 완성돼도 사용자에게 전달되지 않으면 의미가 없다.
- 빌드와 배포를 이해하면 "내 컴퓨터에서는 되는데 서버에서는 안 됨" 같은 문제를 더 잘 설명할 수 있다.
- 배포는 기능 추가보다 **안정성, 롤백, 모니터링**이 더 중요할 때가 많다.
- 실무에서는 빌드/배포가 자동화되어야 실수가 줄고 속도가 빨라진다.

## 빌드와 배포를 먼저 분리해서 이해하기

```text
[소스 코드]
   ↓
[빌드]
   ↓
[산출물(artifact)]
   ↓
[배포]
   ↓
[운영 환경에서 실행]
```

- **빌드**: 코드, 의존성, 설정을 묶어 실행 가능한 형태로 만드는 단계
- **배포**: 그렇게 만들어진 산출물을 서버/클러스터/앱스토어/컨테이너 환경에 올리는 단계

## 자주 헷갈리는 개념
- **빌드(build)**: 만들기
- **배포(deploy)**: 올리기
- **릴리스(release)**: 사용자에게 공개하는 행위
- **런(run)**: 실제 실행
- **배포 성공**과 **서비스 정상 동작**은 같은 뜻이 아니다

---

## 전체 흐름

```text
개발자 코드 작성
   ↓
의존성 설치
   ↓
테스트 실행
   ↓
빌드 수행
   ↓
산출물 생성
   ↓
배포 대상 환경 선택
   ↓
배포 실행
   ↓
헬스체크/모니터링
   ↓
문제 없으면 운영 반영 완료
```

## 핵심 개념
- **산출물(artifact)**: 빌드 결과물
- **환경 분리**: 개발/스테이징/운영은 다를 수 있다
- **자동화**: 사람이 직접 반복하지 않도록 만든다
- **롤백**: 문제가 생겼을 때 이전 버전으로 되돌린다
- **무중단 배포**: 사용자 영향 없이 새 버전으로 바꾸는 전략

---

## 빌드가 하는 일
### 1) 코드 변환
- 소스 코드를 실행 가능한 형태로 바꾼다.
- 언어에 따라 컴파일, 번들링, 패키징이 포함된다.

### 2) 의존성 묶기
- 외부 라이브러리, 런타임, 설정 파일을 함께 관리한다.
- 필요한 버전을 고정하지 않으면 환경마다 결과가 달라질 수 있다.

### 3) 검사
- 빌드 전에 테스트나 린트가 함께 돌기도 한다.
- 잘못된 코드가 배포되지 않도록 막는 역할도 한다.

### 4) 산출물 생성
- JAR, WAR, EXE, 이미지 파일, Docker 이미지, 정적 번들 등이 될 수 있다.

---

## 배포가 하는 일
### 1) 실행 환경 반영
- 서버에 새 파일을 올리거나 컨테이너를 새로 띄운다.
- 환경 변수, 시크릿, 네트워크 설정도 맞춰야 한다.

### 2) 트래픽 전환
- 새 버전이 준비되면 사용자 요청이 새 버전으로 가도록 바꾼다.
- 이때 무중단 배포 전략을 쓸 수 있다.

### 3) 건강 상태 확인
- 배포 후 응답이 정상인지 확인한다.
- 실패하면 바로 롤백할 수 있어야 한다.

---

## 빌드와 배포의 차이

| 구분 | 빌드 | 배포 |
|---|---|---|
| 목적 | 실행 가능한 결과물 생성 | 그 결과물을 운영 환경에 반영 |
| 입력 | 소스 코드, 의존성 | 빌드 산출물, 환경 설정 |
| 결과 | artifact | 서비스 버전 교체 |
| 실패 원인 | 컴파일 에러, 테스트 실패, 패키징 문제 | 설정 오류, 서버 오류, 트래픽 전환 실패 |
| 대표 도구 | Gradle, Maven, npm, webpack, Docker build | SSH, CI/CD, Kubernetes, 배포 스크립트 |

---

## 배포가 어려운 이유
- 코드가 맞아도 **환경 차이** 때문에 실패할 수 있다.
- 개발 환경에서는 보이던 파일이 운영 환경에는 없을 수 있다.
- 서버 권한, 포트, 환경 변수, 시크릿, 데이터베이스 설정이 다를 수 있다.
- 배포는 기술뿐 아니라 운영 안정성 문제이기도 하다.

---

## 환경 분리
### 개발 환경
- 개발자가 로컬에서 확인하는 환경
- 빠르게 바꾸고 실험하기 좋다

### 스테이징 환경
- 운영과 비슷하게 만든 사전 검증 환경
- 배포 전 최종 점검용

### 운영 환경
- 실제 사용자가 접속하는 환경
- 장애 비용이 크므로 가장 조심해야 한다

```text
개발 → 스테이징 → 운영
  낮은 위험        높은 위험
```

---

## 배포 전략
- **롤링 배포**
  - 조금씩 순서대로 새 버전으로 바꾼다
  - 자원 사용이 효율적이지만 전환 중 혼합 상태가 생길 수 있다

- **블루-그린 배포**
  - 같은 환경을 두 벌 두고 한쪽으로 트래픽을 바꾼다
  - 롤백이 빠르다

- **카나리 배포**
  - 일부 사용자에게만 먼저 새 버전을 보낸다
  - 문제를 작은 범위에서 먼저 발견할 수 있다

- **A/B 배포**
  - 서로 다른 버전을 비교 실험한다
  - 기능 검증이나 실험에 쓰인다

---

## 배포 전략 비교

| 전략 | 장점 | 단점 | 적합한 상황 |
|---|---|---|---|
| 롤링 | 자원 효율 좋음 | 혼합 버전이 잠깐 생김 | 일반적인 서비스 |
| 블루-그린 | 롤백 빠름, 전환 단순 | 인프라 비용 큼 | 안정성이 중요한 서비스 |
| 카나리 | 위험을 작게 시작 | 관찰과 운영이 복잡 | 사용자 영향이 큰 서비스 |
| A/B | 실험에 유리 | 분석 설계가 필요 | 제품 실험, UX 검증 |

---

## 롤백이 중요한 이유
- 새 버전이 항상 더 좋은 것은 아니다.
- 장애가 나면 빠르게 이전 버전으로 돌아가야 한다.
- 롤백이 느리면 장애 시간이 길어지고 피해가 커진다.

### 롤백 시 확인할 것
- 어떤 버전으로 돌아갈지
- 데이터베이스 변경이 되돌릴 수 있는지
- 캐시나 세션이 새 버전과 충돌하지 않는지
- 트래픽이 다시 이전 버전으로 정상 전환되는지

---

## 빌드와 배포 워크플로우

```text
1. 코드를 푸시한다
2. CI가 테스트를 돌린다
3. 빌드 산출물을 만든다
4. 이미지를 만들거나 패키지를 만든다
5. 스테이징에 먼저 배포한다
6. 헬스체크를 한다
7. 운영에 배포한다
8. 모니터링한다
9. 이상하면 롤백한다
```

---

## 실전 예시
### 예시 1: 웹 서비스 배포
- React 프론트엔드를 빌드하면 정적 파일이 나온다.
- 백엔드 Java 애플리케이션은 JAR 파일로 패키징된다.
- CI가 테스트 후 빌드하고, 서버나 컨테이너 플랫폼에 배포한다.

### 예시 2: Docker 기반 배포
- `Dockerfile`로 이미지를 만든다.
- 이미지는 어떤 환경에서도 동일한 실행 단위를 만든다.
- 운영에서는 컨테이너를 재시작하거나 새 이미지로 교체한다.

### 예시 3: 앱스토어 배포
- 모바일 앱은 빌드 후 스토어 심사를 거쳐 공개된다.
- 서버 배포와 달리 사용자 기기 업데이트 속도도 고려해야 한다.

---

## 자주 하는 실수
- 내 컴퓨터에서만 되는 설정을 운영에 그대로 기대함
  - 바로잡기: 운영 환경은 OS, 네트워크, 권한, 환경 변수가 다를 수 있으므로 배포 설정을 코드/문서/CI로 재현 가능하게 관리해야 한다.
- 빌드 성공을 서비스 정상과 같은 의미로 착각함
  - 바로잡기: 빌드 성공은 산출물이 만들어졌다는 뜻이고, 서비스 정상은 실행, 헬스체크, 주요 기능 검증까지 통과해야 한다.
- 배포 후 검증 없이 끝내버림
  - 바로잡기: 배포 후에는 헬스체크, 로그, 메트릭, 핵심 시나리오를 확인해야 한다.
- 롤백 계획 없이 배포함
  - 바로잡기: 배포 전에는 이전 버전 복구 방법, DB 변경 되돌림 가능성, 설정 복구 절차를 준비해야 한다.
- 환경 변수를 코드에 직접 박아 넣음
  - 바로잡기: 테스트에서는 이 요소를 그대로 의존하면 결과가 흔들리므로 fixture, mock, 고정 seed, 테스트 전용 설정으로 통제해야 한다.
- 버전이 다른 서버를 섞어서 운영함
  - 바로잡기: 서버별 버전 차이는 재현 어려운 장애를 만들 수 있어 배포 상태와 버전을 추적해야 한다.
- 로그/모니터링 없이 배포함
  - 바로잡기: 바르게는 로그의 목적, 레벨, 필드 이름, 추적 ID, 보안 마스킹 기준을 정하고 필요한 지점에만 남긴다.

## 초보자용 비교
### 수동 배포 vs 자동 배포
- **수동 배포**
  - 사람이 직접 파일 복사, 재시작, 확인
  - 단순하지만 실수가 많다
- **자동 배포**
  - 정해진 절차를 스크립트/CI가 수행
  - 반복 작업이 안정적이고 빠르다

### 빌드 성공 vs 서비스 정상
- **빌드 성공**
  - 코드가 패키징됐다는 뜻
- **서비스 정상**
  - 사용자가 실제로 문제 없이 이용할 수 있다는 뜻

이 둘은 전혀 같지 않다.

## 면접 포인트
- 빌드와 배포의 차이를 분리해서 설명한다.
- 배포 전략의 종류와 장단점을 말할 수 있으면 좋다.
- 롤백과 헬스체크를 배포의 일부로 설명해야 한다.
- "빌드 성공"과 "서비스 정상"이 다르다는 점을 말하면 이해도가 높아 보인다.

### 면접에서 자주 나오는 질문
- 빌드와 배포는 뭐가 다른가요?
- 롤링/블루-그린/카나리 배포 차이는 뭔가요?
- 배포 후 문제가 생기면 어떻게 하나요?
- CI/CD는 왜 필요한가요?
- 운영 환경과 개발 환경은 왜 분리하나요?

## 한 번 더 요약
- 빌드는 실행 가능한 결과물을 만드는 일이다.
- 배포는 그 결과물을 운영 환경에 안전하게 반영하는 일이다.
- 배포는 단순 복사가 아니라 환경, 검증, 롤백, 모니터링까지 포함한다.
- 전략과 자동화가 없으면 배포는 금방 위험해진다.

## 초보자용 한 줄 기억법
- 빌드와 배포는 **코드를 서비스로 바꾸는 마지막 과정**이다.
