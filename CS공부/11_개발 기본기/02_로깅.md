# 로깅

## 한 줄 정의
로깅(logging)은 **프로그램이 언제, 어떤 입력으로, 어떤 상태에서, 무엇을 했는지**를 기록해 두는 일이다.

쉽게 말해, 시스템이 남기는 **사건 기록장**이다.
문제가 생겼을 때 "무슨 일이 있었는지"를 나중에 복원할 수 있게 해 준다.

---

## 로깅을 먼저 한 문장으로 이해하기
로깅은 **사건의 흔적을 남겨서, 나중에 원인을 추적할 수 있게 만드는 기술**이다.

디버깅이 "문제를 찾는 활동"이라면, 로깅은 그 활동을 가능하게 해 주는 **증거 수집 도구**다.

```text
문제 발생
   ↓
로그 확인
   ↓
가설 세우기
   ↓
원인 좁히기
   ↓
수정
   ↓
재발 방지
```

---

## 왜 중요한가
로그는 단순히 "에러를 찍는 것"이 아니다. 실무에서는 아래 역할을 동시에 한다.

- **문제 원인 추적**: 재현이 어려운 버그도 로그가 있으면 흐름을 따라갈 수 있다.
- **운영 상태 확인**: 서버가 정상인지, 느린지, 실패가 많은지 볼 수 있다.
- **장애 대응 속도 향상**: 운영 환경에서는 디버거를 붙이기 어렵기 때문에 로그가 핵심 증거가 된다.
- **분석과 통계**: 어떤 기능이 많이 쓰이는지, 어디서 실패가 많은지 볼 수 있다.
- **보안/감사**: 누가 언제 무엇을 했는지 남겨야 하는 경우가 있다.
- **협업**: 다른 사람이 만든 코드도 로그만 보면 흐름을 이해하기 쉬워진다.

### 로그가 없으면 생기는 일
- "제 컴퓨터에서는 되는데요?" 상태가 길어진다.
- 장애가 나도 어디서 깨졌는지 모르고 감으로 고치게 된다.
- 같은 버그가 반복된다.
- 개발자와 운영자가 서로 다른 이야기를 하게 된다.

### 로그가 있으면 좋은 점
- 증거를 기반으로 이야기할 수 있다.
- 재현이 어려운 문제도 좁혀 갈 수 있다.
- 배포 후 문제를 빠르게 찾을 수 있다.
- 나중에 보더라도 당시 상황을 복원할 수 있다.

---

## 로그는 어디에 쓰이나
로그는 "나중에 찾아보는 기록"이지만, 실제로는 아주 넓게 쓰인다.

### 1) 디버깅
- 에러가 어디서 났는지 찾는다.
- 어떤 입력이 문제였는지 확인한다.
- 특정 분기 조건이 왜 걸렸는지 본다.

### 2) 운영
- 요청이 얼마나 실패하는지 확인한다.
- 특정 API가 느린지 본다.
- 갑자기 오류가 늘었는지 확인한다.

### 3) 보안
- 로그인 실패/성공 기록
- 권한 변경 기록
- 관리자 행동 기록
- 비정상 접근 시도 기록

### 4) 감사(audit)
- 누가, 언제, 무엇을 변경했는지 증명해야 할 때 사용한다.

### 5) 분석
- 어떤 기능이 자주 사용되는지
- 사용자 흐름이 어디서 끊기는지
- 특정 이벤트가 얼마나 발생하는지

---

## 전체 그림
로그는 보통 이렇게 흐른다.

```text
사용자 행동 / 시스템 사건
          ↓
애플리케이션 코드에서 로그 생성
          ↓
stdout / 파일 / 로그 프레임워크
          ↓
수집기(Agent) / 전송기
          ↓
저장소(ELK, Loki, CloudWatch 등)
          ↓
검색 / 필터 / 대시보드 / 알람
          ↓
장애 분석 / 운영 판단 / 보안 감사
```

즉, 로그는 "찍고 끝"이 아니라, **나중에 찾고 해석하는 것까지 포함**한다.

---

## 로깅의 핵심 개념

### 1) 로그 레벨(log level)
로그의 중요도와 심각도를 구분하는 기준이다.

보통 아래처럼 사용한다.

- **TRACE**: 아주 자세한 흐름 추적
- **DEBUG**: 개발 중 확인용 상세 정보
- **INFO**: 정상 동작의 주요 사건
- **WARN**: 주의가 필요한 상황, 아직은 동작함
- **ERROR**: 요청 실패, 예외 발생, 조치가 필요한 문제
- **FATAL / CRITICAL**: 서비스가 심각하게 망가진 상황

> 프로젝트마다 레벨 이름이나 개수는 조금 다를 수 있다.

### 2) 구조화 로그(structured logging)
문자열 한 줄만 남기는 대신, **필드(key-value)** 형태로 남기는 방식이다.

예:
```text
level=INFO event=login_success userId=42 latencyMs=31 ip=1.2.3.4
```

장점:
- 검색하기 쉽다
- 필터링과 집계가 쉽다
- 사람이 읽어도 되고 기계가 파싱하기도 쉽다

### 3) 상관관계 ID(correlation id)
한 번의 요청 또는 하나의 작업 흐름을 식별하기 위한 ID다.

예:
- requestId
- traceId
- correlationId
- transactionId

이 값이 있으면 여러 서비스의 로그를 묶어서 볼 수 있다.

```text
클라이언트 요청 하나
  ↓
API 서버
  ↓
인증 서버
  ↓
결제 서버
  ↓
DB
```

이런 흐름에서 같은 requestId가 들어 있으면, 나중에 로그를 따라가며 전체 과정을 복원할 수 있다.

### 4) 감사 로그(audit log)
누가, 언제, 무엇을 했는지 남기는 기록이다.

예:
- 관리자 권한 변경
- 계정 삭제
- 결제 상태 변경
- 개인정보 열람

감사 로그는 일반 디버깅 로그보다 더 민감하고, 위변조 방지가 중요하다.

### 5) 접근 로그(access log)
HTTP 요청/응답처럼 **들어온 요청 자체**를 기록한 로그다.

보통 이런 정보가 들어간다.
- 요청 시각
- 요청 경로
- 응답 코드
- 응답 시간
- 클라이언트 IP
- User-Agent

### 6) 애플리케이션 로그(application log)
비즈니스 로직이 실행되며 남기는 로그다.

예:
- 회원가입 성공
- 재고 부족
- 장바구니 업데이트
- 결제 승인 실패

---

## 무엇을 로그로 남겨야 하나
로그는 많이 찍는다고 좋은 게 아니다. **나중에 원인을 복원하는 데 필요한 정보**를 남겨야 한다.

### 보통 유용한 정보
- **시간**: 언제 발생했는가
- **레벨**: 얼마나 심각한가
- **이벤트 이름(event)**: 무슨 사건인가
- **요청 식별자(requestId / traceId)**: 어떤 요청에 속하는가
- **사용자/도메인 ID**: 누구와 관련 있는가
- **입력 요약**: 어떤 값이 들어왔는가
- **출력 요약**: 무엇이 결과였는가
- **상태/결과**: 성공인지 실패인지
- **실패 이유**: 왜 실패했는가
- **소요 시간(latencyMs)**: 얼마나 걸렸는가
- **외부 의존성 정보**: DB, API, 큐 호출 결과

### 특히 잘 남겨야 하는 지점
- 요청 시작 / 종료
- 외부 API 호출 전후
- DB 조회 / 저장 전후
- 상태가 바뀌는 순간
- 예외가 발생한 순간
- 재시도(retry) 발생 시점
- 권한 변경, 결제, 삭제 같은 중요한 작업

### 너무 자세히 다 찍어야 할까?
아니다. 다음 기준으로 생각하면 된다.

- **재현이 어려운가?** → 더 자세히 남겨야 한다.
- **중요한 상태 변화인가?** → 꼭 남겨야 한다.
- **반복적으로 너무 자주 발생하는가?** → 줄여야 한다.
- **민감정보가 포함되는가?** → 남기면 안 된다.

---

## 절대 남기면 안 되는 것
로그는 편하지만, 잘못 남기면 보안 사고가 된다.

### 금지 또는 강한 주의 대상
- 비밀번호
- 인증 토큰 / 세션 토큰
- 카드번호 전체
- 주민등록번호
- 개인정보 원문
- 비밀 키 / API 키
- 내부 관리자용 비밀 값

### 필요하면 이렇게 한다
- **마스킹**: 일부만 보이게 가린다.
  - 예: `card=1234-****-****-5678`
- **해시**: 원문을 저장하지 않고 식별만 한다.
- **최소 수집**: 정말 필요한 값만 남긴다.

> 원칙: **나중에 봤을 때 도움이 되지만, 유출되면 위험한 정보는 최대한 덜 남긴다.**

---

## 로그 레벨은 어떻게 생각하면 좋은가
레벨은 "중요도 스위치"다. 레벨이 잘 설계되면 운영이 쉬워진다.

### 레벨별 판단 기준
- **TRACE**: 아주 세밀한 흐름
- **DEBUG**: 개발 중 흐름 확인
- **INFO**: 정상 동작의 핵심 사건
- **WARN**: 이상 징후지만 당장은 동작
- **ERROR**: 실패, 예외, 처리 불가
- **FATAL/CRITICAL**: 서비스 전체에 치명적

### 아주 단순한 기준
- **정상 흐름의 주요 사건** → INFO
- **문제가 될 수 있는 이상 징후** → WARN
- **실패나 예외** → ERROR
- **개발 중 상세 추적** → DEBUG

### 레벨을 잘 나누는 이유
- 너무 많은 로그가 찍히면 진짜 문제를 못 본다.
- 운영 중에는 보통 INFO 이상만 보거나, 상황에 따라 DEBUG를 잠깐 켠다.
- 레벨이 잘 설계되면 장애 대응이 훨씬 쉬워진다.

## 구조화 로그 vs 평문 로그
이 비교는 실무에서 정말 중요하다.

| 항목 | 평문 로그 | 구조화 로그 |
|---|---|---|
| 형태 | 사람이 읽기 쉬운 문장 | key-value / JSON |
| 검색 | 어렵다 | 쉽다 |
| 집계 | 어렵다 | 쉽다 |
| 대시보드 | 약하다 | 강하다 |
| 초기 작성 | 편하다 | 약간 번거롭다 |
| 운영 활용 | 제한적 | 매우 좋다 |

### 평문 로그 예시
```text
주문 생성 실패
```

문제:
- 어떤 주문인지 모른다.
- 어떤 사용자와 관련 있는지 모른다.
- 왜 실패했는지 모른다.
- 검색할 키가 부족하다.

### 구조화 로그 예시
```text
level=ERROR event=order_create_failed requestId=abc123 userId=42 orderId=1001 reason=payment_timeout latencyMs=802
```

장점:
- 검색하기 쉽다.
- 대시보드와 알람에 붙이기 좋다.
- 필드 기준으로 필터링할 수 있다.

### 초보자용 이해
- 평문 로그는 **메모** 같다.
- 구조화 로그는 **표 형식의 기록** 같다.

## 로그와 디버깅의 관계
로그와 디버깅은 가깝지만 완전히 같지는 않다.

- **디버깅**: 문제의 원인을 찾는 활동 전체
- **로깅**: 그 원인을 찾도록 증거를 남기는 도구

즉, 로깅은 디버깅의 핵심 재료다.

```text
문제 발생
  ↓
로그 확인
  ↓
가설 세움
  ↓
검증
  ↓
수정
```

로그가 잘 남아 있으면 디버깅 시간이 크게 줄어든다.

---

## correlation ID / request ID / trace ID
비슷하게 들리지만, 초보자일수록 구분해 두면 좋다.

### requestId
- 보통 **하나의 HTTP 요청**을 식별한다.
- API 게이트웨이나 서버 앞단에서 생성하는 경우가 많다.

### traceId
- 분산 시스템에서 **한 요청의 전체 흐름**을 식별한다.
- 여러 서비스에 걸친 호출을 한 줄로 엮는 데 유리하다.

### correlationId
- 더 넓게는 **관련된 여러 로그를 묶는 식별자**다.
- 업무 시스템마다 정의가 조금 다를 수 있다.

### 왜 필요한가
분산 시스템에서는 로그가 서비스별로 흩어진다.

```text
사용자 요청 1개
  ↓
API Gateway
  ↓
User Service
  ↓
Order Service
  ↓
Payment Service
  ↓
DB / 외부 API
```

이때 공통 ID가 없으면 로그가 조각조각 흩어져 보인다.
공통 ID가 있으면 **한 요청의 여정**을 따라갈 수 있다.

### 실전 이해
- 작은 프로젝트: requestId 하나만 있어도 큰 도움이 된다.
- 마이크로서비스: traceId + spanId 같은 개념이 더 중요해진다.
- 외부 연동: 상대 시스템의 ID와 우리 시스템의 ID를 같이 남기면 좋다.

---

## 감사 로그(audit log)
감사 로그는 일반 로그보다 더 엄격하게 다뤄야 한다.

### 감사 로그가 필요한 이유
- 누가 권한을 바꿨는지
- 누가 데이터를 삭제했는지
- 누가 민감정보를 열람했는지
- 결제 상태나 정산 정보가 바뀌었는지

### 일반 로그와의 차이
- 일반 로그: 디버깅/운영 중심
- 감사 로그: 책임 추적/법적 증빙/보안 중심

### 감사 로그에 들어가면 좋은 것
- 실행자 ID
- 대상 ID
- 작업 종류
- 이전 값 / 이후 값
- 시각
- 요청 ID
- 결과
- 수행 경로(IP, 기기 정보 등 필요 시)

### 주의점
- 감사 로그는 삭제하면 안 되는 경우가 많다.
- 위변조 방지가 중요하다.
- 접근 권한을 엄격히 관리해야 한다.

## 로그, 메트릭, 트레이싱 비교
이 셋은 서로 대체재가 아니라 **서로 보완재**다.

| 구분 | 로그 | 메트릭 | 트레이싱 |
|---|---|---|---|
| 질문 | 무슨 일이 있었나? | 전체 상태는 어떤가? | 요청이 어디를 지나갔나? |
| 형태 | 텍스트 / 구조화 사건 기록 | 숫자 시계열 | 호출 경로 |
| 강점 | 맥락이 풍부함 | 추세/이상 탐지 | 분산 시스템 추적 |
| 약점 | 양이 많아지기 쉬움 | 세부 맥락이 약함 | 개별 상세 메시지는 약함 |
| 잘 맞는 상황 | 오류 원인 분석 | CPU, 에러율, 응답시간 | 서비스 간 병목 추적 |

### 한 줄 비교
- **로그** = 무슨 일이 있었는지 자세히 기록
- **메트릭** = 전체 상태를 숫자로 요약
- **트레이싱** = 요청의 경로를 연결해서 추적

### 실제로는 함께 쓴다
예를 들어 응답이 느리다고 하자.

1. **메트릭**으로 느린 구간이 늘었는지 본다.
2. **트레이싱**으로 어느 서비스에서 시간이 걸렸는지 본다.
3. **로그**로 그 시점의 입력값과 예외를 확인한다.

---

## access log / application log / audit log 비교
이 셋은 역할이 다르다.

- **접근 로그(access log)**: 요청 자체를 기록
  - 예: `GET /orders 200 53ms`
- **애플리케이션 로그(application log)**: 비즈니스 처리 과정을 기록
  - 예: `order_create_failed reason=payment_timeout`
- **감사 로그(audit log)**: 누가 무엇을 했는지 기록
  - 예: `admin_id=7 action=permission_changed target_user=42`

### 왜 나눠야 하나
하나로 다 해결하려고 하면 정보가 섞인다.
- 운영자는 요청량과 응답시간을 보고 싶다.
- 개발자는 비즈니스 단계와 예외를 보고 싶다.
- 보안/감사 담당자는 변경 이력을 보고 싶다.

역할을 나누면 각자 필요한 정보를 더 쉽게 찾는다.

## 로그를 설계할 때 생각할 질문

### 기본 질문
- 이 로그를 누가 볼까? 개발자? 운영자? 보안 담당자?
- 나중에 어떤 문제를 찾기 위해 이 로그가 필요한가?
- 어떤 키로 검색할 것인가?
- 너무 자주 찍혀서 소음이 되지는 않는가?
- 민감정보는 없는가?

### 실무 질문
- 한 요청을 끝까지 따라갈 수 있는가?
- 실패 이유가 충분히 드러나는가?
- 재현하기 어려운 문제를 복원할 수 있는가?
- 장애 알람과 연결할 수 있는가?
- 배포 후에도 형식이 유지되는가?

## 로깅 워크플로우: 실제로 어떻게 설계하나
로깅은 생각보다 순서가 중요하다.

### 1단계: 목적을 먼저 정한다
"왜 로그가 필요한가?"를 먼저 정한다.

예:
- 장애 원인 찾기
- 주문 처리 추적
- 결제 실패 분석
- 관리자 행동 감사

### 2단계: 로그를 남길 사건을 고른다
모든 줄에 로그를 넣는 것이 아니다.

주로 이런 지점이 좋다.
- 요청 시작/끝
- 외부 API 호출 전/후
- DB 저장 전/후
- 상태 전이 시점
- 예외 발생 시점

### 3단계: 공통 필드를 정한다
팀에서 통일할 기본 필드를 정해 둔다.

예:
- timestamp
- level
- event
- requestId
- userId
- result
- latencyMs

### 4단계: 레벨 기준을 정한다
팀마다 기준이 다르면 운영이 지저분해진다.

예:
- INFO: 정상 흐름
- WARN: 이상 징후
- ERROR: 실패 / 예외

### 5단계: 민감정보를 검토한다
로그는 한 번 퍼지면 회수가 어렵다.

체크:
- 토큰이 들어갔는가?
- 비밀번호가 들어갔는가?
- 개인정보가 그대로 들어갔는가?
- 마스킹이 필요한가?

### 6단계: 출력 형식을 결정한다
- 콘솔(stdout)
- 파일
- JSON
- 로그 수집기로 바로 전달

### 7단계: 실제 검색 시나리오로 검증한다
로그는 "남겼다"가 끝이 아니다.

운영자가 이렇게 찾을 수 있어야 한다.
- requestId로 한 요청을 따라가기
- userId로 특정 사용자 행동 확인
- errorCode로 실패 패턴 확인
- 시간 범위로 장애 구간 확인

### 8단계: 배포 후 점검한다
- 로그가 너무 많지 않은가?
- 형식이 깨지지 않았는가?
- 필드명이 일관적인가?
- 민감정보가 노출되지 않았는가?

## 로그가 실제로 어떻게 동작하는가

### 1단계: 사건이 일어난다
사용자가 버튼을 누르거나, 서버가 요청을 받거나, 예외가 발생한다.

### 2단계: 코드가 로그를 작성한다
개발자는 중요한 지점에 로그 문장을 심는다.

예:
```text
주문 생성 시작
주문 검증 완료
결제 요청 전송
결제 응답 수신
주문 생성 완료
```

### 3단계: 로그가 출력된다
대개 다음 중 하나로 간다.

- **표준 출력(stdout)**: 컨테이너 환경에서 많이 사용
- **파일**: 전통적인 서버 환경에서 사용
- **중앙 로그 시스템**: 여러 서버의 로그를 한곳에 모음

### 4단계: 수집된다
로그 수집기(agent)가 파일이나 stdout을 읽어서 중앙 저장소로 보낸다.

### 5단계: 저장되고 검색된다
운영자는 검색창에서 requestId, 사용자 ID, 에러 메시지, 시간 범위로 찾는다.

### 6단계: 분석에 사용된다
- 장애 원인 분석
- 트래픽 확인
- 비정상 패턴 탐지
- 보안 조사

## 실전 예시

### 예시 1: 로그인 실패
상황:
- 사용자가 로그인 버튼을 눌렀다.
- 화면에는 "로그인 실패"만 보인다.

좋은 로그 예시:
```text
level=WARN event=login_failed requestId=abc123 userId=unknown ip=1.2.3.4 reason=invalid_password
```

이 로그로 알 수 있는 것:
- 어떤 요청인지
- 어디서 왔는지
- 왜 실패했는지
- 같은 요청을 어떻게 추적할지

### 예시 2: 주문 중복 생성
상황:
- 사용자가 주문 버튼을 두 번 눌렀는지, 서버가 두 번 처리했는지 모른다.

로그에 있어야 할 정보:
- requestId
- userId
- orderId
- 처리 단계
- 재시도 여부

예:
```text
level=INFO event=order_create_started requestId=abc123 userId=42 cartId=8
level=INFO event=payment_requested requestId=abc123 orderId=1001
level=INFO event=order_create_finished requestId=abc123 orderId=1001 result=success
```

### 예시 3: 서버 500 에러
상황:
- API 응답은 500이다.
- 프론트에서는 원인을 알 수 없다.

필요한 로그:
- 어떤 엔드포인트였는지
- 어떤 입력이 들어왔는지
- 어느 단계에서 예외가 났는지
- 예외 메시지와 스택 트레이스

예:
```text
level=ERROR event=payment_callback_failed requestId=abc123 orderId=1001 error=NullPointerException step=save_payment_result
```

### 예시 4: 느린 응답
상황:
- 화면은 정상인데 너무 느리다.

로그에 들어가면 좋은 정보:
- 처리 시간(latency)
- 외부 API 호출 시간
- DB 쿼리 시간
- 재시도 횟수

예:
```text
level=INFO event=order_list_finished requestId=abc123 latencyMs=842 dbMs=731 apiMs=84 result=success
```

### 예시 5: 결제 승인 실패
상황:
- 결제 게이트웨이는 실패를 돌려줬다.
- 실패가 사용자 문제인지, 외부 API 문제인지 구분해야 한다.

예:
```text
level=ERROR event=payment_approve_failed requestId=abc123 orderId=1001 provider=payco status=503 reason=provider_unavailable retryable=true
```

이런 로그가 있으면 다음 판단이 가능하다.
- 우리 코드 문제인지
- 외부 서비스 문제인지
- 재시도해도 되는지
- 알람을 올려야 하는지

## 로그에서 자주 보는 필드

### 기본 필드
- **timestamp**: 언제 발생했는가
- **level**: 심각도는 어느 정도인가
- **event**: 어떤 사건인가
- **message**: 사람이 읽는 설명
- **requestId / traceId**: 요청 추적용 ID
- **userId**: 어떤 사용자와 관련 있는가
- **orderId / paymentId / sessionId**: 도메인 식별자
- **status / result**: 성공인지 실패인지
- **reason / error**: 실패 이유
- **latencyMs**: 걸린 시간

### 필드 작성 팁
- 숫자는 숫자답게 남기면 집계하기 쉽다.
- ID는 검색하기 좋게 항상 같은 이름으로 유지한다.
- 너무 긴 문장은 핵심이 안 보인다.
- 이벤트 이름은 동사+상태로 맞추면 좋다.
  - 예: `order_create_started`, `order_create_finished`, `order_create_failed`

## 좋은 로그의 조건
좋은 로그는 많기만 한 로그가 아니다. 아래 조건을 만족해야 한다.

### 1) 정확해야 한다
- 거짓말을 하면 안 된다.
- 실패를 성공으로 찍으면 안 된다.
- 예외를 무시하면 안 된다.

### 2) 충분해야 한다
- 나중에 봤을 때 "무슨 요청인지" 알 수 있어야 한다.
- 핵심 입력값과 결과가 보여야 한다.
- 너무 적으면 쓸모가 없다.

### 3) 과하지 않아야 한다
- 모든 변수를 다 찍을 필요는 없다.
- 불필요한 디버그 로그가 너무 많으면 중요한 신호가 묻힌다.

### 4) 일관돼야 한다
- 같은 형식, 같은 필드, 같은 용어를 써야 검색이 쉽다.
- 팀마다 표현이 제각각이면 분석이 어려워진다.

### 5) 검색 가능해야 한다
- requestId, userId, orderId 같은 키로 찾을 수 있어야 한다.
- 문자열 하나만 던지는 것보다 필드화가 좋다.

### 6) 민감정보를 넣지 않아야 한다
- 비밀번호
- 인증 토큰
- 주민번호
- 카드번호 전체
- 개인정보 원문

이런 정보는 원칙적으로 로그에 남기면 안 된다.

### 7) 시간 정보가 있어야 한다
- 언제 발생했는지 알아야 원인을 추적할 수 있다.
- 시간대가 섞이면 순서를 놓치기 쉽다.

## 로그를 볼 때의 사고방식
로그는 그냥 읽는 게 아니라, **질문을 가지고 읽는 것**이다.

### 읽을 때 던질 질문
- 이 요청은 누구의 요청인가?
- 어떤 입력이 들어왔는가?
- 어디까지 정상적으로 갔는가?
- 처음으로 이상해진 지점은 어디인가?
- 실패 원인이 메시지에 직접 보이는가?
- 같은 requestId의 앞뒤 로그는 무엇인가?

### 로그를 읽는 순서
1. 에러 로그부터 찾는다.
2. 같은 requestId나 orderId를 찾는다.
3. 바로 앞 로그를 본다.
4. 처음 시작 로그까지 거슬러 올라간다.
5. 이상한 값이나 누락된 값을 찾는다.

## 자주 하는 실수

### 1) 너무 많이 찍는다
- 모든 함수 시작과 끝에 로그를 넣으면 소음이 된다.
  - 바로잡기: 바르게는 로그의 목적, 레벨, 필드 이름, 추적 ID, 보안 마스킹 기준을 정하고 필요한 지점에만 남긴다.
- 중요한 로그가 묻힌다.
  - 바로잡기: 바르게는 로그의 목적, 레벨, 필드 이름, 추적 ID, 보안 마스킹 기준을 정하고 필요한 지점에만 남긴다.

### 2) 너무 적게 찍는다
- 실패했는데 이유가 없다.
  - 바로잡기: 실패 로그에는 실패한 작업, 입력 식별자, 예외 타입, 원인 메시지, 외부 응답 코드처럼 원인 추적에 필요한 정보를 남겨야 한다.
- 나중에 보면 아무것도 못 찾는다.
  - 바로잡기: 나중에 분석하려면 요청 ID, 사용자/리소스 식별자, 처리 단계, 결과, 소요 시간처럼 검색 가능한 맥락을 남겨야 한다.

### 3) 민감정보를 그대로 찍는다
- 보안 사고가 될 수 있다.
  - 바로잡기: 로그는 운영자가 보는 데이터이자 저장되는 데이터이므로 토큰, 비밀번호, 주민번호, 개인 정보는 마스킹하거나 남기지 않아야 한다.
- 토큰, 비밀번호, 개인 정보는 절대 그대로 남기지 않는다.
  - 바로잡기: 민감 정보는 로그에 원문으로 남기지 않고 마스킹, 해시, 제거 기준을 적용해야 한다.

### 4) 형식이 제각각이다
- 어떤 곳은 userId, 어떤 곳은 uid, 어떤 곳은 memberId.
  - 바로잡기: 같은 의미의 필드는 서비스 전반에서 하나의 이름으로 통일해야 검색과 집계가 가능하다.
- 검색할 때 다 따로 놀게 된다.
  - 바로잡기: 필드 이름과 값 형식을 통일해야 로그 검색, 대시보드 집계, 알림 조건이 한 번에 동작한다.

### 5) 레벨을 잘못 쓴다
- 모든 걸 ERROR로 찍으면 진짜 에러가 안 보인다.
  - 바로잡기: 로그 레벨은 장애 대응 기준에 맞게 ERROR/WARN/INFO/DEBUG를 구분해야 진짜 문제가 묻히지 않는다.
- INFO인데 사실은 문제인 경우도 있다.
  - 바로잡기: 문제 징후는 WARN이나 ERROR로 올리고, 정상 흐름 설명은 INFO로 남겨 운영자가 우선순위를 구분할 수 있게 해야 한다.

### 6) 예외를 숨긴다
- catch 후 아무 로그도 없으면 원인 파악이 불가능해진다.
  - 바로잡기: 바르게는 로그의 목적, 레벨, 필드 이름, 추적 ID, 보안 마스킹 기준을 정하고 필요한 지점에만 남긴다.

### 7) print로 임시 디버깅만 하고 끝낸다
- 개발할 때는 편하지만, 운영에서는 못 찾는다.
  - 바로잡기: 운영 로그는 로컬 콘솔 출력보다 중앙 수집, 구조화 필드, 검색 가능성, 보존 기간을 기준으로 설계해야 한다.
- 임시 print는 나중에 정리해야 한다.
  - 바로잡기: 임시 출력은 운영 로그 정책과 다르므로 배포 전 제거하거나 정식 로거로 바꿔야 한다.

### 8) 시간 기준이 맞지 않는다
- 서버 시간, 타임존, 밀리초 단위가 뒤섞이면 분석이 어렵다.
  - 바로잡기: 로그 시간은 UTC 또는 명확한 타임존과 밀리초 단위 기준으로 통일해야 추적이 쉽다.

### 9) requestId를 안 넣는다
- 여러 서비스가 엮이면 어디서 나온 로그인지 연결이 안 된다.
  - 바로잡기: 바르게는 로그의 목적, 레벨, 필드 이름, 추적 ID, 보안 마스킹 기준을 정하고 필요한 지점에만 남긴다.

### 10) 로그만 믿고 메트릭/알람을 안 만든다
- 로그는 상세하지만, 전체 상태를 빠르게 보기엔 부족하다.
  - 바로잡기: 바르게는 로그의 목적, 레벨, 필드 이름, 추적 ID, 보안 마스킹 기준을 정하고 필요한 지점에만 남긴다.

### 11) 이벤트 이름이 매번 다르다
- 같은 사건인데 코드마다 이름이 다르면 집계가 안 된다.
  - 바로잡기: 같은 사건은 이벤트명과 필드명을 표준화해야 대시보드, 알림, 검색 쿼리에서 같은 기준으로 묶을 수 있다.

### 12) 로그를 너무 늦게 추가한다
- 장애가 난 뒤에야 "아, 이 정보가 필요했네"를 깨닫기 쉽다.
  - 바로잡기: 장애 전에 주요 요청 흐름, 외부 의존성 호출, 상태 전이, 실패 사유를 로그 설계에 포함해야 복구 중 추가 배포 없이 분석할 수 있다.
- 중요 흐름은 미리 설계해 두는 편이 좋다.
  - 바로잡기: 주문, 결제, 인증, 배포 같은 핵심 흐름은 성공/실패/외부 호출/소요 시간을 미리 정해 로그로 남겨야 한다.

## 초보자용 비교 정리

### 로그가 필요한 순간과 아닌 순간
- **필요한 순간**
  - 버그 원인을 찾고 싶다.
  - 운영 장애를 추적하고 싶다.
  - 흐름을 나중에 다시 복원하고 싶다.

- **과한 순간**
  - 너무 단순한 함수인데 로그만 잔뜩 있는 경우
  - 매 반복마다 같은 메시지를 찍는 경우
  - 이미 충분히 명확한 코드에 불필요한 로그를 넣는 경우

### 좋은 로그와 나쁜 로그
- **좋은 로그**
  - 짧지만 핵심이 있다.
  - 검색 가능한 키가 있다.
  - 누가 봐도 상황이 보인다.

- **나쁜 로그**
  - 감탄사처럼만 적혀 있다.
  - "에러 발생"만 있고 이유가 없다.
  - 같은 문구가 수천 번 반복된다.

## 로그 메시지 작성 패턴

### 패턴 1: 시작/종료 로그
```text
level=INFO event=payment_start requestId=abc123 orderId=1001
level=INFO event=payment_finish requestId=abc123 orderId=1001 result=success latencyMs=120
```

장점:
- 처리 시간 계산이 쉽다.
- 중간에 어디서 끊겼는지 찾기 쉽다.

### 패턴 2: 단계별 로그
```text
level=INFO event=checkout_validate_done requestId=abc123
level=INFO event=checkout_price_calculated requestId=abc123 total=32000
level=INFO event=checkout_payment_requested requestId=abc123
```

장점:
- 복잡한 흐름을 단계별로 볼 수 있다.

### 패턴 3: 실패 이유 포함 로그
```text
level=ERROR event=upload_failed requestId=abc123 fileName=a.png reason=file_too_large maxSizeMb=10
```

장점:
- 왜 실패했는지 바로 보인다.

### 패턴 4: 외부 시스템 연동 로그
```text
level=INFO event=inventory_api_called requestId=abc123 provider=warehouse timeoutMs=2000
level=ERROR event=inventory_api_failed requestId=abc123 provider=warehouse status=504 retryable=true
```

장점:
- 내부 문제인지 외부 문제인지 구분하기 좋다.

## 실무에서 로그를 더 잘 쓰는 팁

### 1) 로그 메시지와 필드를 분리한다
메시지는 사람이 읽는 설명, 필드는 검색용 데이터로 생각하면 좋다.

예:
- 메시지: "주문 생성 실패"
- 필드: `orderId=1001`, `reason=payment_timeout`

### 2) 같은 사건은 같은 event 이름을 쓴다
- order_create_started
- order_create_finished
- order_create_failed

이렇게 맞추면 집계가 쉽다.

### 3) 로그와 에러를 혼동하지 않는다
- 예외를 잡았다고 자동으로 해결된 건 아니다.
- 해결되지 않은 문제는 반드시 드러나야 한다.

### 4) 단순한 기준을 팀에서 합의한다
- 어떤 상황을 WARN으로 볼 것인가?
- 어떤 필드를 공통으로 넣을 것인가?
- requestId 이름은 무엇으로 통일할 것인가?

### 5) 운영자가 찾는 방식으로 남긴다
개발자가 편한 로그와 운영자가 찾기 쉬운 로그는 다를 수 있다.

운영자는 보통 이렇게 찾는다.
- 시간 범위
- requestId
- 사용자 ID
- 주문 ID
- 에러 코드

이 키를 기준으로 설계하면 실제로 유용해진다.

## 운영 환경에서 로그가 특히 중요한 이유

### 운영 환경은 디버거를 붙이기 어렵다
- 로컬에서는 브레이크포인트를 걸 수 있다.
- 하지만 운영 서버에서는 그렇게 하기 어렵다.
- 그래서 운영 문제는 로그가 거의 유일한 단서가 된다.

### 운영 환경은 재현이 어렵다
- 사용자 데이터가 다르다.
- 트래픽이 다르다.
- 타이밍이 다르다.
- 환경 변수와 설정이 다를 수 있다.

### 운영에서는 빠른 판단이 중요하다
- 지금 장애가 큰지 작은지
- 특정 사용자만 영향받는지
- 바로 롤백해야 하는지
- 더 지켜봐도 되는지

이런 판단은 로그 없이는 어렵다.

## 로그의 한계
로깅은 강력하지만 만능은 아니다.

### 로그만으로 부족한 경우
- 전체 서비스의 추세를 빠르게 보려면 메트릭이 필요하다.
- 요청 경로의 병목을 한 번에 보려면 트레이싱이 필요하다.
- 실시간 경보를 주려면 알람 시스템이 필요하다.

### 로그의 단점
- 양이 너무 많아질 수 있다.
- 검색하지 않으면 쓸모를 못 느낀다.
- 구조가 나쁘면 오히려 혼란스럽다.
- 민감정보 유출 위험이 있다.

> 결론: **로그는 필요하지만, 로그만 있으면 충분하지는 않다.**

## 운영자 체크리스트

### 배포 전 체크
- [ ] 중요한 경계 지점에 로그가 있는가?
- [ ] requestId 또는 traceId가 전달되는가?
- [ ] 실패 원인이 로그에 보이는가?
- [ ] 민감정보가 마스킹되어 있는가?
- [ ] 로그 레벨이 적절한가?
- [ ] 같은 이벤트가 너무 많이 찍히지 않는가?
- [ ] 형식이 일관적인가?
- [ ] 검색 키(userId, orderId, requestId)가 있는가?

### 장애 발생 시 체크
- [ ] 에러 로그를 먼저 찾았는가?
- [ ] 같은 requestId로 앞뒤 흐름을 봤는가?
- [ ] 정상 케이스와 비교했는가?
- [ ] 최근 배포와 관련 있는가?
- [ ] 특정 사용자/특정 환경에서만 발생하는가?
- [ ] 메트릭과 함께 확인했는가?
- [ ] 재현 가능한 최소 조건을 찾았는가?

### 로그 품질 점검 체크
- [ ] 검색 가능한 필드가 있는가?
- [ ] 시간, 레벨, 이벤트명이 있는가?
- [ ] 너무 장황하지 않은가?
- [ ] 너무 빈약하지 않은가?
- [ ] 보안 사고를 만들 정보가 없는가?
- [ ] 팀에서 합의한 형식과 맞는가?

## 초보자를 위한 예시 코드로 이해하기
언어마다 문법은 다르지만, 생각은 비슷하다.

### 나쁜 예
```text
뭔가 이상함
```

문제:
- 무엇이 이상한지 모른다.
- 어디서 이상한지 모른다.
- 검색이 안 된다.

### 조금 나은 예
```text
회원가입 실패
```

문제:
- 여전히 정보가 부족하다.

### 좋은 예
```text
level=ERROR event=signup_failed requestId=abc123 userId=42 reason=email_already_exists
```

이제야 다음이 보인다.
- 어떤 사건인지
- 누구와 관련 있는지
- 왜 실패했는지

### 더 좋은 예
```text
level=ERROR event=signup_failed requestId=abc123 userId=42 emailHash=9f3a... step=duplicate_check reason=email_already_exists retryable=false
```

이 정도면 운영에서 꽤 유용하다.

## 로그를 읽는 실제 순서
문제가 생기면 아래 순서로 보는 습관이 좋다.

1. **에러 로그**를 찾는다.
2. 같은 **requestId / traceId / orderId**를 찾는다.
3. 바로 앞의 **정상 로그**를 본다.
4. 시작 로그부터 종료 로그까지 흐름을 본다.
5. 실패가 난 **첫 지점**을 찾는다.
6. 정상 케이스와 비교한다.
7. 외부 API/DB/재시도 여부를 확인한다.

```text
start → validate → call external → save db → finish
                   ↑
               여기서 실패?
```

## 면접 포인트
면접에서는 "로그를 남긴다"보다 **왜, 어떻게, 어떤 기준으로 남기는지**를 설명할 수 있어야 한다.

### 자주 나오는 질문
- 로그와 모니터링의 차이는 무엇인가요?
- 구조화 로그를 쓰는 이유는 무엇인가요?
- 어떤 정보를 반드시 남기고, 어떤 정보는 남기면 안 되나요?
- requestId / correlationId는 왜 필요한가요?
- 운영 장애를 로그로 어떻게 추적하나요?
- DEBUG 로그와 INFO 로그는 어떻게 구분하나요?
- 로그와 메트릭, 트레이싱은 어떤 관계인가요?

### 답변할 때 핵심
- 운영 환경에서는 로그가 가장 중요한 증거가 될 수 있다.
- 검색 가능성과 일관성이 중요하다.
- 민감정보는 절대 남기면 안 된다.
- requestId로 흐름을 연결하면 문제 추적이 쉬워진다.
- 로그는 재현이 어려운 문제를 푸는 데 큰 도움이 된다.
- 로그만으로 부족할 때는 메트릭과 트레이싱을 함께 써야 한다.

### 좋은 설명 흐름
```text
문제 발생 → 필요한 정보 정의 → 로그 설계 → 검색/추적 → 원인 규명 → 재발 방지
```

### 면접에서 한 번은 말해두면 좋은 문장
- "로그는 나중에 문제를 재구성할 수 있게 남기는 증거입니다."
- "좋은 로그는 사람이 읽을 수 있고, 기계가 검색할 수 있어야 합니다."
- "운영에서는 재현보다 추적이 먼저인 경우가 많습니다."
- "구조화 로그와 requestId가 있으면 분산 환경에서 원인 추적이 훨씬 쉬워집니다."

### 면접에서 추가로 말하면 좋은 포인트
- 로그 레벨을 무작정 높이는 것보다, 의미 있는 지점에 정확히 남기는 것이 더 중요하다.
- 감사 로그는 디버깅 로그와 목적이 다르므로 분리해야 한다.
- 민감정보 보호는 로깅 설계의 일부다.

---

## 초보자용 기억법
- 로깅은 **사건을 나중에 다시 볼 수 있게 기록하는 것**이다.
- 좋은 로그는 **정확하고, 충분하고, 찾기 쉽고, 안전해야 한다**.
- 로그가 있으면 디버깅, 운영, 분석이 모두 쉬워진다.
- 로그만으로는 부족할 수 있으니 메트릭과 트레이싱도 함께 봐야 한다.

---

## 한 번 더 요약
- 로깅은 시스템의 기억을 남기는 일이다.
- 운영 환경에서는 로그가 핵심 증거다.
- 구조화 로그와 requestId가 특히 중요하다.
- 로그는 많기만 하면 안 되고, 읽을 수 있어야 한다.
- 민감정보는 절대 그대로 남기지 않는다.
- 감사 로그, 접근 로그, 애플리케이션 로그는 역할이 다르다.
- 로그 / 메트릭 / 트레이싱은 서로 보완한다.
