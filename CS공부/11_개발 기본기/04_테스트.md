# 테스트

## 한 줄 정의
테스트는 **코드가 기대한 대로 동작하는지 자동으로 확인하는 절차**다.

즉, 사람이 매번 눈으로 확인하던 검사를 코드가 대신하게 만드는 것이다.

---

## 왜 중요한가
테스트는 단순히 "버그를 찾는 도구"가 아니다. 더 정확히는 **버그가 다시 들어오지 못하게 막는 안전장치**다.

- 기능을 바꿀 때 기존 기능이 망가지는 것을 빨리 찾는다.
- 사람이 매번 수동으로 확인하지 않아도 된다.
- 변경이 쉬워진다. 코드 수정이 무서워지지 않는다.
- 협업할 때 서로의 작업을 검증하기 쉽다.
- 운영 중 장애를 줄이고, 배포에 대한 자신감을 높인다.

테스트가 없으면 개발자는 늘 이런 생각을 하게 된다.

```text
"이거 바꾸면 다른 데 깨지는 거 아니야?"
```

테스트가 있으면 이 불안이 줄어든다.

```text
"바꾸고 테스트 돌려보면 되겠네."
```

이 차이가 엄청 크다.

---

## 테스트를 한 문장으로 다시 말하면
테스트는 **내 코드에 대해 내가 만든 작은 질문서**다.

예를 들면:

- 이 함수에 1을 넣으면 2가 나오는가?
- 비어 있는 리스트를 넣으면 에러 없이 처리하는가?
- 로그인 실패 시 올바른 메시지를 주는가?

이 질문에 자동으로 답하게 만드는 것이 테스트다.

---

## 테스트가 해결하는 문제
### 1) 회귀 방지
회귀(regression)는 **예전에 되던 기능이 나중에 다시 망가지는 현상**이다.

예:
- 가격 계산 함수 수정 → 할인 로직이 깨짐
- 로그인 로직 수정 → 기존 토큰 검증이 깨짐
- UI 리팩터링 → 버튼 클릭이 안 됨

테스트는 이 회귀를 빨리 잡는다.

### 2) 변경의 안전망
코드는 거의 항상 바뀐다.

- 요구사항이 바뀜
- 성능 개선이 필요함
- 버그 수정이 필요함
- 구조를 더 깔끔하게 바꾸고 싶음

테스트가 있으면 바꾼 뒤 바로 검증할 수 있다.

### 3) 설계 피드백
테스트하기 어려운 코드는 종종 구조가 나쁘다.

예:
- 너무 많은 책임을 가진 함수
- 외부 의존성이 뒤엉킨 코드
- 상태가 숨겨진 코드

테스트를 쓰다 보면 자연스럽게 더 분리된 구조로 개선하게 된다.

---

## 테스트의 큰 종류
테스트는 목적과 범위에 따라 나뉜다.

### 1) 단위 테스트(Unit Test)
- 가장 작은 단위의 코드가 맞는지 본다.
- 보통 함수, 메서드, 클래스 하나 수준을 확인한다.
- 빠르고, 실패 지점이 명확하다.
- 외부 시스템(DB, 네트워크, 파일)에 의존하지 않게 작성하는 경우가 많다.

예:
- `add(1, 2)`가 `3`을 반환하는가?
- 세금 계산 함수가 올바른 값을 내는가?

### 2) 통합 테스트(Integration Test)
- 여러 구성 요소가 함께 잘 동작하는지 본다.
- 예: 서비스 + DB, API + 저장소, 모듈 간 연결
- 단위 테스트보다 현실적이지만, 더 느리고 복잡하다.

예:
- 회원가입 요청이 오면 DB에 실제로 저장되는가?
- 주문 생성 시 재고 차감과 결제 기록이 함께 처리되는가?

### 3) E2E 테스트(End-to-End Test)
- 사용자가 실제로 앱을 쓰는 흐름 전체를 확인한다.
- 화면, API, DB까지 한 번에 이어서 본다.
- 가장 현실적이지만 가장 느리고, 유지보수가 어렵다.

예:
- 로그인 → 상품 선택 → 결제 → 주문 완료까지 되는가?

### 4) 회귀 테스트(Regression Test)
- 새로 수정한 뒤 기존 기능이 깨지지 않았는지 확인한다.
- 엄밀히는 별도 테스트 종류라기보다 **테스트 목적**에 가깝다.
- 단위/통합/E2E 모두 회귀 테스트가 될 수 있다.

### 5) 스모크 테스트(Smoke Test)
- 핵심 기능이 최소한 살아 있는지 빠르게 확인한다.
- 배포 직후나 큰 변경 후에 유용하다.
- 깊게 검증하기보다는 "일단 크게 안 깨졌는가"를 본다.

### 6) 리그레션이 아닌 샌티 테스트(Sanity Test)
- 특정 변경이 정말 맞는지 아주 좁게 확인하는 검사.
- 팀마다 쓰는 의미가 조금 다를 수 있다.

## 테스트 피라미드
테스트는 많이 쌓는다고 무조건 좋은 게 아니다. **어떤 테스트를 얼마나 쌓을지**가 중요하다.

보통 아래처럼 생각한다.

```text
         E2E
      ─────────
     통합 테스트
   ─────────────
    단위 테스트
 ────────────────
```

### 아래층이 넓어야 하는 이유
#### 단위 테스트가 가장 많아야 하는 이유
- 빠르다.
- 싸다.
- 실패 원인을 찾기 쉽다.
- 자주 실행하기 좋다.

#### 통합 테스트는 중간 정도
- 실제 연결을 검증하므로 가치가 크다.
- 하지만 환경 준비가 필요하고 느리다.

#### E2E는 가장 적게
- 실제 사용자 흐름을 검증하는 데 중요하다.
- 하지만 느리고 불안정해지기 쉽다.
- 너무 많으면 유지보수 비용이 커진다.

### 피라미드가 의미하는 것
테스트 피라미드는 이렇게 말한다.

- **기본 검증은 단위 테스트로 많이 한다.**
- **연결 검증은 통합 테스트로 적당히 한다.**
- **사용자 흐름 검증은 E2E로 핵심만 한다.**

즉, 모든 것을 E2E로 확인하려고 하면 너무 느리고 비싸다.

---

## 테스트는 어떻게 동작하는가
테스트는 보통 아래 3단계로 생각하면 쉽다.

```text
1. 준비(Arrange)
2. 실행(Act)
3. 검증(Assert)
```

### 1) 준비(Arrange)
- 테스트할 데이터를 만든다.
- 필요한 객체, 입력값, 환경을 준비한다.

예:
- 숫자 1과 2를 준비한다.
- 로그인된 사용자 정보를 만든다.
- 가짜 DB 응답을 준비한다.

### 2) 실행(Act)
- 실제 코드를 한 번 실행한다.

예:
- `add(1, 2)` 호출
- 주문 생성 API 요청
- 결제 처리 함수 실행

### 3) 검증(Assert)
- 결과가 기대와 맞는지 확인한다.

예:
- 결과가 3인지 확인
- DB에 데이터가 들어갔는지 확인
- 에러 메시지가 정확한지 확인

이 흐름을 짧게 적으면 이렇게 된다.

```text
준비 → 실행 → 검증
```

이 패턴을 머릿속에 넣어 두면 테스트 코드가 훨씬 읽기 쉬워진다.

## 테스트 더블: 진짜 대신 쓰는 가짜들
테스트를 작성하다 보면 실제 DB나 API를 그대로 쓰기 어려울 때가 많다. 이때 **테스트 더블(test double)** 을 쓴다.

### 대표적인 종류
#### Mock
- "이 함수가 이렇게 호출되었는가?"를 확인하는 가짜
- 호출 여부, 호출 횟수, 호출 인자 등을 검증할 때 쓴다.

#### Stub
- 미리 정해진 값을 돌려주는 가짜
- 입력에 따른 고정 응답이 필요할 때 쓴다.

#### Fake
- 실제 구현과 비슷하지만 훨씬 단순한 가짜
- 예: 인메모리 DB, 간단한 저장소 구현

#### Spy
- 실제 동작을 일부 하면서 호출 기록도 남기는 형태

### 왜 쓰는가
- 외부 시스템이 느리거나 불안정할 수 있다.
- 테스트가 너무 느려지는 것을 막는다.
- 원하는 상황을 쉽게 만들 수 있다.
- 실패 조건을 인위적으로 만들 수 있다.

### 주의할 점
- 너무 많이 가짜로 바꾸면 실제와 멀어진다.
- 가짜는 편하지만, 실제 환경과 다르면 테스트가 믿기 어려워진다.

---

## 테스트와 디버깅의 관계
둘은 비슷해 보이지만 목적이 다르다.

### 테스트
- 미리 정해 둔 기대값과 비교한다.
- 반복 실행에 강하다.
- 회귀 방지에 좋다.

### 디버깅
- 이미 발생한 문제의 원인을 찾는다.
- 현재 상태를 추적하는 데 강하다.
- 증거를 보고 범위를 좁힌다.

간단히 말하면:

- **디버깅**은 "왜 깨졌는지"를 찾는 과정
- **테스트**는 "앞으로도 안 깨지는지"를 확인하는 과정

## 비교로 이해하기
### 단위 테스트 vs 통합 테스트 vs E2E
- 단위 테스트
  - 범위: 가장 작음
  - 속도: 가장 빠름
  - 장점: 원인 파악이 쉽다
  - 단점: 실제 연결 문제를 놓칠 수 있다

- 통합 테스트
  - 범위: 여러 컴포넌트 연결
  - 속도: 중간
  - 장점: 실제 동작에 더 가깝다
  - 단점: 환경 세팅이 필요하다

- E2E 테스트
  - 범위: 사용자 흐름 전체
  - 속도: 가장 느림
  - 장점: 실제 사용자 관점에 가깝다
  - 단점: 불안정하고 유지보수 비용이 높다

### 테스트와 수동 확인
- 수동 확인
  - 장점: 즉시 눈으로 확인 가능
  - 단점: 매번 사람이 해야 함, 반복에 약함

- 자동 테스트
  - 장점: 빠르게 반복 가능, 누락이 줄어듦
  - 단점: 처음 작성 비용이 듦

### 테스트와 로깅
- 테스트
  - 기대값을 검증한다.
  - 주로 개발/CI에서 사용한다.

- 로깅
  - 실행 중 실제 상태를 기록한다.
  - 운영에서 원인 분석에 중요하다.

둘은 경쟁 관계가 아니라 서로 보완한다.

## 테스트를 쓰는 실제 워크플로우
테스트는 아무 코드나 바로 쓰는 것이 아니라 보통 이런 순서로 한다.

```text
문제 또는 요구사항 이해
  ↓
무엇을 검증할지 정하기
  ↓
가장 작은 테스트부터 작성
  ↓
실패를 확인
  ↓
코드 구현 또는 수정
  ↓
성공하도록 맞추기
  ↓
리팩터링
  ↓
회귀 테스트 추가
```

### 초보자에게 특히 중요한 순서
#### 1) 요구사항을 문장으로 바꾸기
예:
- "숫자 두 개를 더하면 합을 반환해야 한다"
- "비밀번호가 비어 있으면 가입이 실패해야 한다"

#### 2) 경계값부터 보기
경계값은 실수를 많이 하는 구간이다.

예:
- 0
- 1
- 빈 문자열
- null / undefined
- 최대값 근처
- 배열 길이 0 / 1

#### 3) 실패 케이스도 생각하기
성공 케이스만 테스트하면 반쪽짜리다.

예:
- 정상 로그인
- 비밀번호 틀림
- 존재하지 않는 사용자
- 권한 없는 사용자

#### 4) 테스트 이름을 읽기 쉽게 쓰기
좋은 테스트 이름은 문장처럼 읽힌다.

예:
- `should_return_sum_when_two_numbers_are_added`
- `주문_금액이_0이면_결제할_수_없다`

## 실전 예시
### 예시 1: 숫자 더하기 함수
가장 단순한 단위 테스트 예시다.

```text
함수: add(a, b)
기대: a + b를 반환
```

검증할 것:
- `add(1, 2) = 3`
- `add(0, 5) = 5`
- `add(-1, 1) = 0`

왜 좋은가:
- 입력과 출력이 명확하다.
- 실패하면 원인이 바로 보인다.

### 예시 2: 할인 계산
예:
- 10,000원 이상이면 10% 할인
- 10,000원 미만이면 할인 없음

검증할 것:
- 9,999원 → 할인 없음
- 10,000원 → 할인 적용
- 경계값이 정확한지 확인

여기서 중요한 것은 **경계값 테스트**다.

### 예시 3: 로그인 검증
예:
- 아이디와 비밀번호가 맞으면 성공
- 틀리면 실패 메시지

검증할 것:
- 성공 시 토큰이 발급되는가
- 실패 시 정확한 에러가 나오는가
- 비어 있는 입력은 어떻게 처리하는가

### 예시 4: 주문 생성
예:
- 주문 요청이 들어오면 DB에 저장하고 재고를 줄인다.

검증할 것:
- 주문이 저장되는가
- 재고가 차감되는가
- 결제 실패 시 롤백되는가

이건 단위 테스트만으로는 부족할 수 있고, 통합 테스트가 필요할 수 있다.

## 테스트를 잘 쓰는 기준
### 좋은 테스트의 특징
- **빠르다**: 자주 돌릴 수 있어야 한다.
- **명확하다**: 무엇이 실패했는지 바로 알 수 있어야 한다.
- **독립적이다**: 다른 테스트에 영향을 주지 않아야 한다.
- **재현 가능하다**: 같은 조건이면 같은 결과가 나와야 한다.
- **유지보수 가능하다**: 요구사항이 바뀌면 같이 고칠 수 있어야 한다.

### 나쁜 테스트의 특징
- 너무 느리다.
- 실패 이유가 불분명하다.
- 여러 테스트가 같은 상태를 공유한다.
- 구현 내부를 지나치게 많이 알고 있다.
- 실제 요구사항보다 "구현 방식"에 묶여 있다.

---

## 자주 하는 실수
### 1) 성공 케이스만 테스트함
실패 케이스를 빼먹으면 실제 장애를 놓치기 쉽다.

### 2) 한 테스트에 너무 많은 것을 검증함
하나의 테스트는 가능한 한 하나의 의도를 가져야 한다.

### 3) 테스트끼리 서로 의존함
A 테스트가 지나야 B 테스트가 성공하는 구조는 매우 위험하다.

### 4) 외부 상태를 방치함
- DB 데이터
  - 바로잡기: 테스트에서는 이 요소를 그대로 의존하면 결과가 흔들리므로 fixture, mock, 고정 seed, 테스트 전용 설정으로 통제해야 한다.
- 시간
  - 바로잡기: 테스트에서는 이 요소를 그대로 의존하면 결과가 흔들리므로 fixture, mock, 고정 seed, 테스트 전용 설정으로 통제해야 한다.
- 랜덤값
  - 바로잡기: 테스트에서는 이 요소를 그대로 의존하면 결과가 흔들리므로 fixture, mock, 고정 seed, 테스트 전용 설정으로 통제해야 한다.
- 환경 변수
  - 바로잡기: 테스트에서는 이 요소를 그대로 의존하면 결과가 흔들리므로 fixture, mock, 고정 seed, 테스트 전용 설정으로 통제해야 한다.
- 캐시
  - 바로잡기: 테스트에서는 이 요소를 그대로 의존하면 결과가 흔들리므로 fixture, mock, 고정 seed, 테스트 전용 설정으로 통제해야 한다.

이런 것들이 테스트를 흔든다.

### 5) 구현 디테일에 과하게 묶임
리팩터링할 때마다 테스트가 깨지면 테스트 자체가 부담이 된다.

### 6) 테스트를 너무 많이 가짜로 처리함
너무 많이 mock하면 실제 동작과 멀어질 수 있다.

### 7) 실패 메시지가 불친절함
나중에 테스트가 깨졌을 때 원인을 바로 알 수 없어진다.

---

## 테스트를 쉽게 만드는 습관
### 1) 순수 함수에 가깝게 설계하기
입력 → 출력이 분명한 함수는 테스트하기 쉽다.

### 2) 의존성을 분리하기
- DB 접근
- HTTP 요청
- 파일 읽기
- 시간 조회

이런 것들은 핵심 로직과 분리할수록 테스트가 쉬워진다.

### 3) 작은 함수로 나누기
함수가 너무 크면 테스트 포인트가 흐려진다.

### 4) 테스트 데이터 만들기를 단순하게 하기
복잡한 준비 코드가 많아지면 테스트가 읽기 어려워진다.

### 5) 실패 메시지를 분명하게 하기
무엇이 기대와 달랐는지 한눈에 보이게 한다.

## 면접 포인트
면접에서는 정의만 외운 사람과, 실제로 써 본 사람을 구분하려고 한다.

### 자주 나오는 질문
- 테스트가 왜 필요한가요?
- 단위 테스트와 통합 테스트의 차이는 무엇인가요?
- 테스트 피라미드를 설명해 보세요.
- 테스트 더블(mock/stub/fake)은 왜 쓰나요?
- 테스트하기 좋은 코드의 특징은 무엇인가요?
- 재현이 어려운 버그를 테스트로 어떻게 막을 수 있나요?

### 답변할 때의 핵심
#### 1) 품질 관점
테스트는 품질을 높이고 회귀를 줄인다고 말할 수 있어야 한다.

#### 2) 비용 관점
초기에 조금 더 들지만, 장기적으로 수동 확인 비용과 장애 비용을 줄인다.

#### 3) 설계 관점
테스트는 좋은 구조를 만들도록 압력을 준다.

#### 4) 실무 관점
배포 전 검증, 협업, 리팩터링, 장애 대응에 모두 도움이 된다.

### 말하기 좋은 한 줄
- "테스트는 버그를 100% 없애는 도구가 아니라, 변경의 위험을 줄이는 안전망입니다."

## 개발 흐름 속 테스트 위치
테스트는 보통 아래 위치에서 중요하다.

```text
요구사항 정리
  ↓
코드 작성
  ↓
테스트 작성 / 수정
  ↓
CI에서 자동 실행
  ↓
배포 전 검증
  ↓
운영 중 회귀 방지
```

즉, 테스트는 개발 마지막에 덧붙이는 장식이 아니라 **개발 흐름 전체에 붙어 있는 안전장치**다.

---

## 개발 기본기 관점에서의 테스트
테스트는 개발 기본기 중에서도 특히 중요하다.

- Git은 변경을 관리한다.
- 디버깅은 원인을 찾는다.
- 로깅은 실제 상황을 남긴다.
- 배포는 변경을 세상에 내보낸다.
- 테스트는 그 변경이 안전한지 확인한다.

이 다섯 개는 따로 놀지 않고 서로 연결된다.

```text
코드 수정 → 테스트 → 배포 → 로그 확인 → 디버깅 → 재수정
```

---

## 한 번 더 요약
- 테스트는 코드가 기대대로 동작하는지 자동으로 확인하는 절차다.
- 핵심 목적은 버그를 빨리 찾는 것뿐 아니라 **회귀를 막는 것**이다.
- 단위 테스트, 통합 테스트, E2E 테스트가 있으며, 아래로 갈수록 빠르고 위로 갈수록 현실적이다.
- 테스트는 보통 준비 → 실행 → 검증 순서로 작성한다.
- 좋은 테스트는 빠르고, 독립적이고, 읽기 쉬워야 한다.
- 테스트하기 좋은 코드는 대체로 구조도 좋다.

테스트는 **"이 코드가 맞는지 자동으로 다시 물어보는 장치"**다.
