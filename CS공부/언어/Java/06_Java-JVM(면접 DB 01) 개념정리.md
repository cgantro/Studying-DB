## Java 기본

### 개념
- Java는 “소스 → 바이트코드”로 컴파일하고, JVM이 이를 실행하는 언어다.
- 플랫폼 독립성은 “바이트코드가 OS/CPU에 중립적이고, 각 플랫폼에 JVM 구현이 존재한다”는 의미에 가깝다.
- 실행 관점에서 JVM은 크게 `클래스 로딩 → 검증/링킹 → 실행(인터프리터/JIT) → GC/런타임 관리`의 역할을 맡는다.
- `public static void main(String[] args)`는 JVM이 엔트리포인트로 호출하는 약속된 시그니처다.

### 왜 필요한가
- “왜 JVM이 필요한가?”를 설명할 수 있어야 런타임 이슈(성능/메모리/클래스 로딩 문제)를 추적할 수 있다.
- 서버 운영에서는 “소스 코드”보다 “JVM 옵션/GC/스레드/클래스 로딩”이 장애/성능에 더 직접적으로 영향을 주는 경우가 많다.
- 인터프리터/JIT의 조합을 이해하면 워밍업, 최적화, 프로파일 기반 컴파일 같은 현상을 자연스럽게 설명할 수 있다.

### 동작 원리
- 컴파일:
  - `javac`가 `.java`를 `.class`(바이트코드)로 컴파일한다.
  - 바이트코드는 JVM 명령 집합 기반이며, 특정 CPU의 기계어가 아니다.
- 클래스 로딩:
  - Class Loader가 클래스를 로딩하고, Linking(Verification/Preparation/Resolution)과 Initialization이 이어진다.
  - “처음 한 번만 로딩되는가?”는 로더/격리(ClassLoader hierarchy) 설계에 따라 달라질 수 있다(웹 컨테이너/플러그인 구조 등).
- 실행:
  - 초기에는 인터프리터가 실행을 시작하고, 자주 실행되는 경로(Hot path)는 JIT가 컴파일/최적화한다.
  - JIT는 프로파일 데이터를 바탕으로 인라이닝, 탈가상화, 루프 최적화 등을 적용한다(최적화는 상황에 따라 되돌릴 수도 있음).
- 검증:
  - Bytecode Verifier는 타입 안전성/스택 프레임 검증 등으로 “명백히 위험한 바이트코드”를 막아 JVM 안정성에 기여한다.

### 장단점 / 한계
- 장점:
  - 이식성(동일 바이트코드), 풍부한 생태계, 런타임 최적화(JIT), GC로 인한 메모리 안전성(상대적).
- 한계:
  - 워밍업(초기 성능), 런타임 오버헤드, GC pause 가능성, 네이티브 코드 대비 낮은 “최대 자원 제어” 가능성.
- “Java는 느리다/빠르다” 같은 단정은 의미가 약하다.
  - 어떤 워크로드인지(짧은 CLI vs 장기 실행 서버), 어떤 GC/옵션/프로파일 상태인지에 따라 결과가 크게 달라진다.

### 대안 / 선택 기준
- JVM 언어(예: Kotlin)는 동일 JVM 위에서 동작하므로 런타임 특성(클래스로딩/GC/JIT)은 공유한다.
- 짧게 실행되는 작업/초저지연이 절대적이면 네이티브 언어(C/C++/Rust 등)도 대안이 될 수 있다.
- 실무 선택은 “개발 생산성/팀 숙련도/운영 난이도/성능 목표/장애 대응 체계”의 합으로 결정하는 게 안전하다.

### 실무/디버깅 포인트
- “실행이 느리다”를 만나면 보통 아래 순서로 좁힌다.
  - CPU 바운드인지 / I/O 바운드인지(스레드 덤프, 프로파일, 지표로 구분)
  - GC pause가 문제인지(메모리/GC 로그/지표)
  - 락 경합/스레드 고갈인지(스레드 덤프, executor 큐, 커넥션 풀)
  - 클래스 로딩/리플렉션/프록시 비용인지(초기 지연, startup profile)
- `==` vs `equals()`, `equals/hashCode` 계약은 컬렉션 동작과 직결된다(자세한 건 02번 개념정리로 연결).

### 프로젝트 연결
- Java/JVM 지식은 Spring Boot 운영 이슈(스레드/GC/커넥션풀)로 이어진다.
- 프로젝트 연결은 Spring Boot/운영 문서에서 구체적으로 다룬다: `확인 필요`

## String / 객체 / 예외

### 개념
- `String`은 불변(immutable) 객체이며, 문자열 리터럴은 String Pool로 재사용될 수 있다.
- 불변성은 “동시성 안전성”과 “캐싱 가능성(hashCode 캐시)”을 제공하지만, 문자열 조작을 무심코 하면 임시 객체가 늘어날 수 있다.
- 예외(Exception)는 “정상 흐름과 예외 흐름의 분리”를 위한 메커니즘이며, Checked/Unchecked로 분류된다.

### 왜 필요한가
- 문자열은 API/로그/키/식별자 등 다양한 곳에서 핵심 자료형이라, 불변성/풀/비교 규칙을 잘못 이해하면 버그가 오래 남는다.
- 예외는 운영 장애 분석에서 “어떤 계층에서 어떤 유형이 터졌는지”를 설명하는 언어이기도 하다.
- try-with-resources는 자원 누수(파일/소켓/스트림)와 직결되어 운영 안정성에 영향을 준다.

### 동작 원리
- String Pool:
  - 리터럴 `"abc"`는 풀에 올라가 재사용될 수 있다.
  - `new String("abc")`는 풀과 별개 객체를 생성할 수 있다(구현/최적화는 JVM 버전에 따라 다를 수 있음).
  - `intern()`은 문자열을 풀 기준으로 정규화하려는 시도지만, 남용하면 메모리/GC 부담이 생길 수 있다.
- 문자열 조작:
  - `String` 연결을 반복하면 중간 객체가 많이 생길 수 있다.
  - `StringBuilder`는 가변 버퍼로 효율적인 누적을 제공하고, `StringBuffer`는 동기화를 제공한다(대부분의 단일 스레드 누적에는 `StringBuilder`가 일반적).
- 예외:
  - Checked Exception은 호출자에게 처리를 강제하는 성격, Unchecked(Runtime) Exception은 프로그래밍 오류/상태 오류로 분류하는 관습이 있다.
  - 너무 넓은 catch는 원인을 흐리고, 장애 원인 추적을 어렵게 만든다(로깅/리커버리 전략이 중요).
- try-with-resources:
  - `AutoCloseable`을 대상으로 scope 종료 시 `close()`를 호출해 자원을 정리한다.
  - 예외가 중첩될 때 suppressed exception이 붙을 수 있어, 로그에서 같이 확인해야 한다.

### 장단점 / 한계
- `String` 불변은 안전하지만, 대량 조작/파싱에서는 성능 비용이 될 수 있다.
- Checked Exception 남발은 호출자 코드를 오염시킬 수 있어, 도메인/계층 경계에서 전략적으로 설계해야 한다.
- 예외를 “흐름 제어”로 쓰면(정상 케이스를 예외로 처리) 성능/가독성이 나빠질 수 있다.

### 대안 / 선택 기준
- 문자열 누적/포맷이 많으면:
  - `StringBuilder`, 스트림/버퍼 기반 처리, 메시지 포맷팅 전략 등을 고려한다.
- 예외 처리:
  - “복구 가능한가?”를 기준으로 처리 계층을 나눈다(컨트롤러/서비스/인프라).
  - 외부 연동 오류는 재시도/서킷브레이커와 결합해 설계한다(05번 개념정리로 연결).

### 실무/디버깅 포인트
- 문자열 비교는 `==` 대신 `equals()`가 기본이다(동일성/동등성 구분).
- 예외 로그는 “최상단 메시지”보다 stack trace의 원인(caused by) 체인을 끝까지 보는 습관이 중요하다.
- 자원 누수(파일/소켓)는 운영 중 핸들 고갈로 번질 수 있어, try-with-resources 사용 여부를 먼저 본다.

### 프로젝트 연결
- 외부 API 호출/소켓/파일 등 자원을 쓰는 코드에서 try-with-resources/정확한 예외 분류가 운영 안정성에 직결된다.
- 구체 프로젝트 근거 링크는 `확인 필요`

## JVM 메모리 관리

### 개념
- JVM 메모리는 크게 Heap(객체), Stack(호출 프레임/지역 변수), Metaspace(클래스 메타데이터) 관점으로 본다.
- GC는 “더 이상 참조되지 않는 객체”를 수집해 메모리 누수 위험을 줄이지만, pause/throughput trade-off가 존재한다.
- “메모리 누수”는 JVM에서는 보통 “수집되지 않는 참조가 남아 객체가 계속 살아있는 상태”를 의미한다.

### 왜 필요한가
- 서버 장애에서 메모리 관련 이슈는 흔하다(OutOfMemoryError, 잦은 Full GC, 지연 스파이크 등).
- GC는 평균보다 p95/p99 지연에 영향을 크게 줄 수 있어, 성능 문제를 “평균”만 보고 놓치기 쉽다.
- 힙만 보지 말고 스레드/메타스페이스/직접 메모리(Direct Buffer)도 같이 보아야 한다.

### 동작 원리
- 객체 생명주기 관점:
  - 많은 객체는 짧게 살고(young), 일부는 오래 산다(old).
  - GC는 이 특성을 이용해 세대(young/old)를 나누고 효율을 높인다(구체 알고리즘은 GC 종류에 따라 다름).
- Root와 도달성:
  - 스택, 정적 필드, JNI 참조 등 GC Root에서 도달 가능한 객체는 살아남는다.
  - “캐시/전역 맵/ThreadLocal” 같은 구조는 의도치 않게 참조를 오래 잡아둘 수 있다.
- 힙/메타스페이스:
  - 클래스 메타데이터는 Metaspace에 쌓이며, 클래스 로더 누수는 Metaspace 고갈로 이어질 수 있다.

### 장단점 / 한계
- GC는 개발 생산성을 높이지만, pause/튜닝/관측 부담이 있다.
- 힙을 키우면 일시적으로 Full GC 빈도가 줄 수 있지만, “더 큰 pause”로 돌아올 수 있다.
- GC 이슈는 애플리케이션 로직(할당 패턴/캐시 전략)과 결합되어 있어 JVM 옵션만으로 해결이 안 되는 경우가 많다.

### 대안 / 선택 기준
- GC 튜닝은 “원인(할당률/객체 수명/누수)”을 먼저 파악한 뒤 옵션을 건드리는 게 안전하다.
- 메모리 민감 서비스는 캐시 정책/객체 재사용/데이터 구조 개선 같은 코드 레벨 개선이 더 큰 효과를 내기도 한다.
- 관련 상세는 기존 문서로 연결한다: [[CS공부/언어/Java/03_JVM 메모리와 GC 실무]]

### 실무/디버깅 포인트
- 운영에서 기본적으로 보고 싶은 것:
  - 힙 사용량(young/old), GC 횟수/시간, allocation rate(가능하면), Full GC 유무
  - 스레드 수, 커넥션 풀 상태(스레드/DB/HTTP)
- “OOM이 났다”면:
  - 힙 덤프/GC 로그 확보 여부부터 확인하고, 누수 의심 객체 그래프를 따라간다.
  - ThreadLocal, 캐시, 대형 컬렉션, 대량 문자열, 버퍼(Direct) 등을 우선 의심한다.

### 프로젝트 연결
- Spring 기반 서버에서 “트랜잭션/ThreadLocal/커넥션풀”과 메모리/스레드 문제는 함께 나타나는 경우가 많다.
- 프로젝트 근거 링크는 Spring Boot 운영 문서 쪽에서 구체화: `확인 필요`

## Java 8 / 11 / 17 / 21 변화

### 개념
- Java의 주요 변화는 “언어 문법”뿐 아니라 “표준 라이브러리/런타임/GC/도구” 개선을 포함한다.
- 실무에서는 보통 LTS(예: 8, 11, 17, 21) 중심으로 업그레이드 계획을 잡는다.

### 왜 필요한가
- 프레임워크(Spring Boot 등) 지원 범위가 Java 버전에 의해 결정되는 경우가 많다.
- 성능/운영 관점에서 진단 도구/GC/런타임 개선이 누적되어, 버전 업이 곧 운영 안정성으로 이어질 수 있다.

### 동작 원리
- 버전별 기능은 “컴파일 타깃/바이트코드 레벨”과 “런타임 동작”을 함께 본다.
- 마이그레이션은 보통 아래 축으로 점검한다.
  - 컴파일/빌드(언어/Gradle/Maven 플러그인)
  - 런타임(옵션/GC 기본값 변화)
  - 서드파티 라이브러리 호환성

### 장단점 / 한계
- 최신 버전은 기능/성능이 좋아질 수 있지만, 라이브러리 호환성/운영 리스크가 있다.
- “무조건 최신”이 정답이 아니라, 서비스 성격과 업그레이드 역량(테스트/롤백/관측)에 따라 결정해야 한다.

### 대안 / 선택 기준
- 업그레이드 기준(권장):
  - 프레임워크 지원 여부(예: Spring Boot 3.x는 Java 17+ 요구) 확인
  - 핵심 라이브러리(JPA/Netty/observability 등) 호환성 확인
  - 성능/운영 이득(진단, GC 개선) vs 마이그레이션 비용 비교
- 버전별 핵심 정리는 기존 문서로 연결:
  - [[CS공부/언어/Java/01_Java 17 핵심]]
  - [[CS공부/언어/Java/02_Java 21 핵심]]

### 실무/디버깅 포인트
- 버전 업 시 가장 흔한 문제는 “의존성 충돌/리플렉션 제한/빌드 플러그인 호환성” 계열이다.
- 런타임 옵션은 기본값이 바뀔 수 있어, 기존 튜닝 옵션이 역효과를 내지 않는지 점검해야 한다.
