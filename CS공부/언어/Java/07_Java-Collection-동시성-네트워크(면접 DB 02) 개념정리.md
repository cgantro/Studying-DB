## Collection

### 개념
- Java Collection Framework는 “자료구조(리스트/셋/맵/큐)”를 공통 인터페이스로 제공해, 구현 교체와 알고리즘 재사용을 쉽게 하는 표준 라이브러리 집합이다.
- 핵심 축:
  - `List`: 순서/중복 허용 (`ArrayList`, `LinkedList`)
  - `Set`: 중복 비허용 (`HashSet`, `TreeSet`)
  - `Map`: 키-값 (`HashMap`, `TreeMap`, `ConcurrentHashMap`)
  - `Queue/Deque`: 큐/덱 (`ArrayDeque`, `PriorityQueue`)
- “무엇을 쓰느냐”는 시간복잡도보다도 **메모리/캐시 지역성/동시성/요구 기능(정렬/범위조회)**에 의해 결정되는 경우가 많다.

### 왜 필요한가
- 컬렉션 선택이 성능/안정성에 직접 영향을 준다.
  - 예: `HashMap` 키 계약(`equals/hashCode`)을 깨면 조회/중복 제거가 망가진다.
  - 예: 대량 데이터 처리에서 `LinkedList`는 포인터 추적 비용(캐시 미스)으로 체감 성능이 크게 떨어질 수 있다.
- 운영에서는 컬렉션이 곧 메모리/GC 압력으로 이어진다.
  - 같은 데이터라도 “객체 수가 많다”는 것 자체가 GC 비용을 올린다.

### 동작 원리
- `ArrayList` vs `LinkedList`:
  - `ArrayList`는 연속 배열 기반이라 인덱스 접근이 빠르고, 끝에 추가는 amortized O(1)이다.
  - `LinkedList`는 노드가 흩어져 있어 순차 접근 자체가 느려질 수 있고, 중간 삽입/삭제도 “노드 위치를 이미 알고 있을 때”만 싸다.
  - 실무에서는 “순회/조회가 대부분”이면 `ArrayList`가 기본 선택인 경우가 많다.
- `HashMap` 핵심:
  - 키의 `hashCode()`로 버킷을 고르고, 버킷 내부에서 `equals()`로 키를 비교해 엔트리를 찾는다.
  - 해시 충돌은 체이닝(연결 리스트/트리 형태)으로 처리한다(구체 구현은 JDK 버전에 따라 세부가 달라질 수 있음).
  - 로드 팩터와 리사이징은 성능과 메모리의 trade-off다(리사이징은 순간 비용이 크다).
- `HashSet`의 중복 판단:
  - 내부적으로 `HashMap`을 이용하는 구조로 이해하면 편하다(키만 쓰고 값은 더미).
  - 결국 중복 판단은 `equals/hashCode` 계약에 의해 결정된다.
- `TreeMap`/`TreeSet`:
  - 정렬(키/원소의 자연 순서 또는 Comparator)에 기반한 탐색 트리 구조라 범위 조회/정렬이 필요할 때 유리하다.
  - 단순 조회/삽입은 해시 기반보다 보통 더 비싸다.
- `PriorityQueue`:
  - 우선순위 기반으로 “최소/최대”를 빠르게 뽑기 위한 힙(Heap) 구조를 사용한다.
  - “정렬된 전체 순회”가 아니라 “가장 중요한 것 1개”를 반복적으로 뽑는 문제에 적합하다.
- `Stack`보다 `Deque`:
  - `Stack`은 레거시/동기화 오버헤드/설계 측면에서 권장되지 않는 경우가 많아, `ArrayDeque`를 스택처럼 쓰는 패턴이 흔하다.

### 장단점 / 한계
- 컬렉션은 만능이 아니라 요구사항을 구체화해야 한다.
  - “순서 유지가 필요한가?”, “정렬이 필요한가?”, “중복 제거가 필요한가?”, “동시 접근이 있는가?”를 먼저 결정한다.
- 해시 기반 컬렉션은 키 설계가 망가지면 최악이 된다.
  - 가변 키(필드 변경) + `equals/hashCode`에 그 필드를 포함하면 조회가 깨질 수 있다.
- 반복 중 수정(ConcurrentModification):
  - 일반적으로 Iterator 순회 중 구조적 변경은 `ConcurrentModificationException` 같은 형태로 감지될 수 있다(항상 강제되는 동기화가 아니라 fail-fast 성격).

### 대안 / 선택 기준
- 기본 선택(경험칙):
  - 대부분의 “리스트 + 순회/조회”는 `ArrayList`.
  - “중복 제거 + 빠른 포함 여부”는 `HashSet`.
  - “키-값”은 `HashMap`, “정렬/범위”는 `TreeMap`.
- 동시 접근이 있으면 “무조건 동기화”가 아니라 사용 패턴을 먼저 본다.
  - 읽기 위주: `CopyOnWriteArrayList` 같은 전략도 가능(쓰기 비용 큼).
  - 복합 연산(check-then-act): `ConcurrentHashMap`의 원자 API(`compute`, `putIfAbsent`)를 사용해야 한다.

### 실무/디버깅 포인트
- 컬렉션 성능 문제는 “자료구조”보다 “데이터 크기/접근 패턴/객체 수”가 원인인 경우가 많다.
- 메모리 문제에서 큰 컬렉션은 우선 의심 대상이다(누수/캐시/전역 맵/ThreadLocal 등).
- `Collections.synchronizedXxx`는 단순 래핑이라, 반복/복합 연산의 원자성을 보장하지 못할 수 있다(사용 시 주의).

### 프로젝트 연결
- 인증/세션/토큰 같은 상태를 서버에 들고 가는 설계(예: Refresh Token 저장)는 “Map/캐시 구조”의 수명/삭제 정책이 중요하다.
  - 관련: [[프로젝트/Autowing-Car/03-인증-세션-토큰]]

## Java 멀티스레딩 / 동시성

### 개념
- 동시성(concurrency)은 “여러 작업이 겹쳐 진행되는 구조”를 다루고, 병렬성(parallelism)은 “물리적으로 동시에 실행”되는 경우를 말한다.
- Java에서 동시성의 핵심은 “공유 상태(shared mutable state)”를 어떻게 안전하게 다루느냐이다.
- 용어를 분리해서 설명해야 한다.
  - Race condition: 실행 타이밍에 따라 결과가 바뀌는 버그
  - Data race: 동기화 없는 공유 변수 접근(메모리 모델 관점에서 정의)
  - Deadlock: 서로가 가진 락을 기다리며 영원히 진행하지 못함
  - Starvation: 특정 스레드가 계속 기회를 못 얻음

### 왜 필요한가
- 서버 애플리케이션은 요청을 동시에 처리해야 하므로 동시성은 필수다.
- 동시성 버그는 재현이 어렵고, 한 번 터지면 데이터 정합성/장애로 이어져 비용이 크다.
- Spring Boot 같은 프레임워크에서도 “싱글톤 빈”이 기본이기 때문에, 상태를 공유하면 경쟁 조건이 발생할 수 있다(04번 개념정리와 연결).

### 동작 원리
- 메모리 가시성(visibility)과 happens-before:
  - 한 스레드가 쓴 값이 다른 스레드에서 “언제 보이는지”는 CPU/컴파일러/JIT 최적화에 의해 달라질 수 있다.
  - Java Memory Model은 `synchronized`, `volatile`, `Lock`, `Atomic*` 등이 어떤 가시성/순서를 보장하는지 정의한다.
- `synchronized`:
  - 임계 구역을 직렬화하고, 락 획득/해제 시점에 메모리 가시성(플러시/리로드)에 관련된 보장을 제공한다.
  - 장점은 단순함, 단점은 락 경합/우선순위 역전/병목 가능성이다.
- `volatile`:
  - 단일 변수의 가시성을 높여 주지만, “복합 연산(읽고-수정-쓰기)”을 원자적으로 만들어주지는 않는다.
  - 그래서 카운터 같은 건 `AtomicLong`/`LongAdder` 같은 원자/분산 카운터를 고려한다.
- `Atomic*`:
  - CAS(Compare-And-Swap) 기반으로 락 없이(또는 최소 락으로) 원자 연산을 제공한다.
  - 하지만 무한 재시도/경합이 심하면 성능이 떨어질 수 있어, 쓰기 경합 패턴에서는 다른 구조가 유리할 수 있다.
- Executor / Thread pool:
  - 매 요청/작업마다 스레드를 만들면 생성/스케줄링 비용이 크므로, 풀로 재사용한다.
  - 큐가 무한이면 지연이 쌓이고, 제한 큐면 거절(rejection) 정책을 설계해야 한다.
- Virtual Thread(개념 수준):
  - “OS 스레드와 1:1로 묶이지 않는 경량 스레드”로, blocking I/O 중심의 동시성 모델을 단순화하는 방향이다.
  - 다만 모든 문제가 자동으로 해결되는 건 아니고, 락 경합/공유 상태 문제는 여전히 남는다(세부는 `확인 필요`로 두고 필요 시 별도 문서화).

### 장단점 / 한계
- 락 기반은 이해하기 쉽지만 경합이 병목이 될 수 있다.
- 락 프리/원자 연산은 빠를 수 있지만, 사고 모델이 어렵고 디버깅이 힘들다.
- 동시성 문제는 “코드 레벨”뿐 아니라 “설계 레벨(상태 공유 최소화)”에서 해결하는 게 효과적이다.

### 대안 / 선택 기준
- 기본 원칙:
  - 공유 상태를 줄이고, 불변 객체/지역 변수/메시지 전달로 설계를 바꾼다.
  - 필요하면 동기화하지만, “락을 잡는 범위/시간”을 최소화한다.
- 데이터 구조 선택:
  - 동시 읽기/쓰기가 많은 맵이면 `ConcurrentHashMap`.
  - 읽기 위주 리스트면 `CopyOnWriteArrayList`도 후보(쓰기 많으면 부적합).
- 프레임워크/런타임 선택:
  - I/O 중심이면 이벤트 루프(예: Netty)나 Virtual Thread 같은 접근이 고려 대상일 수 있다(서비스 성격에 따라 선택).

### 실무/디버깅 포인트
- 증상 기반 체크리스트:
  - 처리량 저하: 락 경합/스레드 풀 포화/GC/외부 의존성 지연
  - 간헐적 데이터 이상: race condition, 복합 연산의 비원자성, 잘못된 캐시
  - 정지: 데드락(스레드 덤프에서 락 대기 체인 확인)
- 도구:
  - 스레드 덤프/락 상태 확인, executor 큐/활성 스레드 수 지표, APM trace 등(환경에 따라).

### 프로젝트 연결
- 실시간 처리(WebSocket/MQTT)처럼 “동시 이벤트”가 많은 시스템은 공유 상태를 어떻게 나누는지가 핵심이다.
  - 관련(실시간 흐름): [[프로젝트/Autowing-Car/01-구조-흐름]]

## Java Network / DNS

### 개념
- 네트워크 문제는 Java 코드 문제라기보다 “DNS/라우팅/방화벽/TLS/타임아웃” 같은 환경 문제인 경우가 많다.
- Java에서 외부 API 호출/DB 접속 실패는 보통 다음 축으로 나뉜다.
  - 이름 해석(DNS) 문제
  - 연결 수립(TCP) 문제
  - 보안 계층(TLS) 문제
  - 애플리케이션 계층(HTTP) 문제
- `UnknownHostException`은 대표적인 DNS/이름 해석 실패 신호다.

### 왜 필요한가
- 운영에서 “외부 API가 느리다/간헐 실패한다”는 문제는 자주 나오고, 원인을 네트워크 계층별로 분리해 설명해야 한다.
- 타임아웃을 설정하지 않으면 스레드가 대기 상태로 묶여 스레드 풀 고갈 → 장애 전파로 이어질 수 있다.
- Docker/컨테이너 환경에서는 “내 PC에서는 되는데 컨테이너에서는 안 됨” 같은 DNS 차이가 자주 발생한다(08번 개념정리와 연결).

### 동작 원리
- DNS 캐시:
  - 브라우저/OS/리졸버처럼 JVM 내부 또는 OS 레벨에서도 캐시가 관여할 수 있다.
  - Java에서도 DNS 결과가 즉시 반영되지 않는 경우가 생길 수 있는데, 런타임/보안 설정/캐시 정책에 의해 좌우될 수 있다(세부 설정값은 환경별로 달라 `확인 필요`).
- 타임아웃 종류(개념):
  - connect timeout: TCP 연결을 만드는 데 걸리는 최대 시간
  - read timeout: 연결이 만들어진 뒤 응답 바이트를 기다리는 최대 시간
  - 전체 요청 타임아웃: 요청 전체에 대한 데드라인(라이브러리/클라이언트별로 설정 방식이 다름)
- TLS 예외:
  - 인증서 만료/도메인 불일치/신뢰 체인 실패 등은 “서버가 살아 있어도” 연결이 실패하게 만든다.
  - Java는 신뢰 저장소(truststore) 설정/배포 환경 CA 구성에 따라 동작이 달라질 수 있다(구체 예외 타입은 라이브러리별로 달라 `확인 필요`).

### 장단점 / 한계
- 네트워크는 “느린 게 정상”일 수 있어, 재시도/캐시/타임아웃/서킷브레이커 같은 전략이 필요하다.
- 무작정 재시도는 장애를 키울 수 있다(05번 개념정리: 복원력 패턴 참고).

### 대안 / 선택 기준
- 외부 호출 기본 정책(권장):
  - 타임아웃은 반드시 설정
  - 재시도는 “멱등성/부하/장애 전파”를 고려해 제한적으로
  - 관측(로그/메트릭/추적)으로 DNS/연결/TLS/TTFB를 분리
- DNS 문제가 의심되면 OS/컨테이너 레벨에서 먼저 확인:
  - `nslookup`, `dig` 등으로 레코드/리졸버 차이를 확인(네트워크 문서 참고).

### 실무/디버깅 포인트
- “이름은 되는데 연결이 안 됨” vs “이름부터 안 됨”을 먼저 가른다.
  - 이름 실패: `UnknownHostException`, NXDOMAIN, 리졸버 문제
  - 연결 실패: 방화벽/보안그룹/포트/라우팅, 서버 다운
  - TLS 실패: 인증서/시간 동기화/중간자 프록시/신뢰 저장소
- 애플리케이션에서는 타임아웃 미설정이 가장 치명적이다(스레드/커넥션 고갈로 번짐).
