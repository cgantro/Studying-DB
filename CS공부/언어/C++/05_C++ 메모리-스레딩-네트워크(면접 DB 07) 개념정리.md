## 메모리 / RAII / 스마트 포인터

### 개념
- C++에서 가장 중요한 설계 질문은 “소유권(ownership)과 수명(lifetime)”이다.
- RAII(Resource Acquisition Is Initialization)는 “자원 획득을 객체 수명에 묶어, 소멸자에서 해제”하는 패턴이다.
- 자원은 메모리만이 아니라 락, 파일 핸들, 소켓, 스레드, GPU 리소스 등을 포함한다.

### 왜 필요한가
- 누수/이중 해제/댕글링 포인터는 재현이 어렵고, 운영에서 치명적이다.
- 예외가 발생해도 자원이 해제되도록 구조를 만들려면 RAII가 사실상 필수다.
- 스마트 포인터는 소유권을 타입으로 표현해 코드 리뷰/유지보수를 쉽게 만든다.

### 동작 원리
- 스택/힙 수명:
  - 스택 객체는 스코프 종료 시 소멸자가 자동 호출된다.
  - 힙 객체는 `new/delete`로 수명을 수동 관리하지만, modern C++에서는 `std::unique_ptr`/`std::shared_ptr`로 관리하는 게 일반적이다.
- `unique_ptr`:
  - 단독 소유(복사 불가, 이동 가능). “누가 지우는가”가 명확하다.
- `shared_ptr` / `weak_ptr`:
  - 참조 카운팅 기반 공유 소유.
  - 소유 그래프가 복잡해지고, 순환 참조가 생기면 누수(해제 불가)가 될 수 있어 `weak_ptr`로 끊는다.
- 예외 안전성:
  - RAII는 예외가 터져도 스택을 풀며 소멸자를 호출하기 때문에, “중간 실패”에 강해진다.
  - 단, 소멸자에서 예외를 던지면 더 큰 문제가 되므로 일반적으로 소멸자는 예외를 던지지 않도록 설계한다.

### 장단점 / 한계
- 스마트 포인터는 안전하지만, 무분별한 `shared_ptr`는 소유 관계를 흐리고 성능 비용(원자적 refcount)도 생길 수 있다.
- “포인터가 있다 = 소유권”이 아니라, 소유/비소유를 구분해야 한다.

### 대안 / 선택 기준
- 권장 선택(경험칙):
  - 기본은 값 타입/스택 수명
  - 동적 할당이 필요하면 `unique_ptr`
  - 공유 소유가 진짜 필요할 때만 `shared_ptr`(그리고 순환 참조 점검)
- 인터페이스는 “소유권 이동”이 있는지 타입/함수 시그니처로 드러내는 게 좋다.

### 실무/디버깅 포인트
- 크래시가 나면 먼저 의심할 것:
  - 수명 종료 후 접근(use-after-free), double free, 범위 밖 접근(out-of-bounds)
- 도구:
  - sanitizers, 정적 분석, 코어 덤프/스택 트레이스(환경에 따라 준비가 다르므로 `확인 필요`).

### 프로젝트 연결
- 실시간/네트워크 코드에서는 “버퍼 수명”이 가장 흔한 버그 원인이라 RAII/소유권 모델이 핵심이 된다.
  - 관련: [[프로젝트/RobotPal]]

## 멀티스레딩

### 개념
- 멀티스레딩에서 핵심은 “데이터 레이스를 없애고, 진행(progress) 문제(데드락/기아)를 피하는 것”이다.
- C++은 메모리 모델이 명시되어 있어, `std::atomic`과 happens-before 관계를 이해해야 안전한 코드를 만들 수 있다.

### 왜 필요한가
- 성능/응답성을 올리기 위해 병렬 처리/비동기 처리가 필요하지만, 동시성 버그는 비용이 매우 크다.
- 특히 네트워크/실시간 시스템에서는 스레드 설계가 지연/처리량을 결정한다.

### 동작 원리
- 스레드 생성/조인:
  - `std::thread`는 생성 시 실행을 시작하고, `join()`으로 종료를 기다린다.
  - 조인 누락은 종료 시점 문제/리소스 누수로 이어질 수 있어 RAII 래핑이 유용하다.
- 동기화:
  - `std::mutex` + `std::lock_guard`/`std::unique_lock`로 임계구역을 보호한다.
  - 조건 변수(`std::condition_variable`)는 “상태가 만족될 때까지 기다리기”를 구현한다(스푸리어스 웨이크업 고려).
- atomic:
  - 원자 변수는 단일 변수의 원자성/가시성/순서를 제어한다.
  - 메모리 오더(memory order)는 성능과 안전성의 trade-off이며, 기본은 `seq_cst`로 시작하고 필요할 때만 조정하는 게 안전하다(세부 튜닝은 `확인 필요`).

### 장단점 / 한계
- 락은 단순하지만 경합이 병목이 될 수 있다.
- 락 프리/atomic 기반은 성능이 좋을 수 있지만 설계/검증 난이도가 높다.
- 스레드 수를 늘리는 게 항상 성능을 올리진 않는다(컨텍스트 스위칭/캐시/경합 비용).

### 대안 / 선택 기준
- 기본 원칙:
  - 공유 상태를 줄이고(메시지 큐/작업 분리), 필요한 곳만 최소 락을 건다.
  - 데이터 구조를 스레드별로 분리(sharding)하거나, producer-consumer 구조로 단순화한다.
- 설계 선택은 워크로드(CPU 바운드 vs I/O 바운드)와 지연 목표에 따라 달라진다.

### 실무/디버깅 포인트
- 데드락/경합 분석:
  - 락 순서 규칙을 정하고, 락을 잡는 시간/범위를 줄인다.
- 실시간 시스템에서는 “최신 데이터 우선”인지 “순서/정확성 우선”인지 정책을 먼저 결정한다(네트워크 설계와 연결).

### 프로젝트 연결
- 음성/실시간 스트리밍은 “순서/재전송”보다 “최신성”이 중요한 경우가 있어 프로토콜 선택과 스레드 설계가 맞물린다.
  - 관련: [[프로젝트/영묘]]

## Network

### 개념
- C++ 표준 라이브러리는 네트워크 API가 제한적이라, 보통 OS 소켓 API 또는 외부 라이브러리(Boost.Asio 등)를 사용한다.
- TCP는 바이트 스트림(stream)이고, UDP는 메시지(datagram) 기반이다.
- “TCP에는 message boundary가 없다”는 말은, 송신 측의 `send()` 단위가 수신 측 `recv()` 단위로 보장되지 않는다는 의미다.

### 왜 필요한가
- 네트워크 문제는 “항상 일어나는 정상 상황”이라, 부분 송수신/지연/끊김을 전제로 설계해야 한다.
- 구조체를 그대로 전송하는 같은 단순 구현은 엔디안/패딩/정렬/ABI 차이로 깨질 수 있다.

### 동작 원리
- TCP 스트림:
  - 수신은 “얼마나 읽을지”를 애플리케이션이 정해야 한다.
  - 그래서 길이 프리픽스(length-prefix), 구분자(delimiter), 고정 길이 프레임 같은 프레이밍 전략이 필요하다.
- 부분 송수신:
  - `send`/`recv`는 요청한 바이트 수를 항상 처리하지 않을 수 있다(버퍼/윈도우/스케줄링 영향).
  - 따라서 루프를 돌며 “남은 바이트를 모두 처리”하는 코드를 작성해야 한다.
- 엔디안:
  - 네트워크 바이트 오더는 보통 big-endian으로 정의되고, 호스트는 little-endian이 흔하다.
  - `htonl/ntohl/htons/ntohs`는 호스트↔네트워크 바이트 오더 변환을 돕는다.
- TLS:
  - C++에서 TLS를 직접 구현하기보다 OpenSSL 같은 라이브러리를 사용한다(구현 복잡도/보안성 때문).
  - 서버 이름(SNI), 인증서 검증 같은 개념은 라이브러리 API를 통해 설정한다.

### 장단점 / 한계
- OS 소켓 API는 강력하지만, 비동기/이벤트 루프/에러 처리까지 포함하면 구현량이 커진다.
- 외부 라이브러리를 쓰면 생산성이 오르지만, 빌드/배포/ABI 호환성 문제가 생길 수 있다.

### 대안 / 선택 기준
- 동기 vs 비동기:
  - 단순/낮은 동시성이면 동기 모델이 단순하다.
  - 높은 동시성이 필요하면 이벤트 루프(비동기) 모델이 유리할 수 있다(디버깅 난이도는 상승).
- 전송 포맷:
  - 구조체 raw 전송 대신, 명시적인 직렬화 포맷(바이너리 프로토콜/JSON/Protobuf 등)을 선택한다.

### 실무/디버깅 포인트
- 네트워크 read/write는 “언제든 끊길 수 있음”을 가정하고, 타임아웃/재시도/재연결 전략을 세운다.
- 소켓 자원은 RAII로 묶어 누수/예외 상황을 줄인다(소켓 닫힘 누락은 운영 장애로 이어짐).

### 프로젝트 연결
- WebSocket/TCP/UDP 선택은 “순서/재전송 vs 최신성/지연”의 trade-off로 설명할 수 있다.
  - 관련: [[프로젝트/영묘]]

## Build / Linking

### 개념
- 빌드는 “컴파일(오브젝트 생성) + 링크(실행 파일/라이브러리 결합)”로 나뉜다.
- 링크 방식:
  - 정적 링크(static): 바이너리에 라이브러리 코드 포함
  - 동적 링크(dynamic): 런타임에 공유 라이브러리(DLL/so) 로드
- ABI 호환성은 “컴파일러/표준 라이브러리/컴파일 옵션”이 달라질 때 바이너리 호환이 깨질 수 있다는 의미다.

### 왜 필요한가
- 실무에서는 기능 구현보다 “빌드/링킹/배포” 문제로 시간이 많이 든다.
- Debug/Release 차이는 최적화/정의되지 않은 동작(UB) 노출/타이밍 차이를 만들 수 있어, “릴리즈에서만 터지는 버그”가 생긴다.
- Header-only 라이브러리는 편하지만, 컴파일 시간/코드 부풀림/ODR 문제를 만들 수 있다.

### 동작 원리
- Header-only:
  - 정의가 헤더에 있어야 템플릿 인스턴스화를 할 수 있어, 헤더만으로 배포되는 경우가 있다.
  - 하지만 모든 포함 지점에서 컴파일되므로 빌드 시간이 늘어날 수 있다.
- Debug/Release:
  - 최적화가 켜지면 타이밍/인라이닝/메모리 배치가 달라져, UB가 더 잘 드러나거나 다른 형태로 나타날 수 있다.
- 정적 vs 동적 링크:
  - 정적: 배포가 단순(의존성 포함)하지만 바이너리가 커질 수 있고, 보안 업데이트 반영이 늦을 수 있다.
  - 동적: 여러 프로그램이 공유 라이브러리를 재사용할 수 있지만, 런타임 의존성(버전/경로) 문제가 생길 수 있다.
- CMake target 관리:
  - target 단위로 include/link를 관리하면 “전역 설정 오염”을 줄이고 의존성 추적이 쉬워진다.

### 장단점 / 한계
- 빌드/링킹은 “문법”이 아니라 “환경”이라 재현이 중요하다.
- 컴파일러/표준 라이브러리 버전 차이가 큰 문제를 만들 수 있다(특히 배포/플러그인 구조).

### 대안 / 선택 기준
- 라이브러리 배포 전략:
  - 제품/플랫폼 요구에 따라 정적/동적/헤더온리 중 선택한다.
  - ABI 안정성이 중요한 경우, C API 경계로 감싸거나(플러그인), 같은 toolchain을 강제하는 전략을 쓴다.

### 실무/디버깅 포인트
- “빌드만 깨짐”은 다음을 우선 확인한다.
  - 컴파일 옵션 차이, include path 우선순위, 링크 순서, 런타임 로딩 경로
- 재현 가능한 빌드(고정된 toolchain/의존성) 체계를 갖추는 것이 장기적으로 비용을 줄인다.

### 프로젝트 연결
- 네이티브 프로젝트는 CI에서 빌드/링킹 문제가 자주 나오므로, 파이프라인에 build cache/의존성 고정이 중요하다.
  - 구체 프로젝트 근거는 `확인 필요`
