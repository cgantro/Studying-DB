## C++ 기본

### 개념
- C++은 “성능 + 자원 제어”에 강점이 있는 시스템 프로그래밍 언어다.
- 핵심 사고 모델:
  - “어떤 자원을 누가 소유(ownership)하고, 언제 해제(release)하는가”
  - “복사(copy) vs 이동(move) vs 참조(reference)로 데이터가 어떻게 전달되는가”
  - “정적 타입/컴파일 타임 검증”을 어디까지 활용할 것인가
- modern C++은 “포인터 산술”보다 “타입/RAII/표준 라이브러리” 중심으로 안전성을 끌어올리는 방향이다.

### 왜 필요한가
- 성능/지연이 중요한 영역(게임/로보틱스/미디어/네트워크/저수준 라이브러리 등)에서 C++이 자주 쓰인다.
- “메모리/스레드/네트워크” 문제를 OS 관점까지 내려가 설명할 수 있어야 실무 디버깅이 가능하다.
- 면접에서 C++은 언어 문법보다 “자원 관리/예외 안전성/ABI/빌드” 같은 실무가 자주 나온다.

### 동작 원리
- 컴파일/링킹:
  - 헤더는 선언(인터페이스), 소스는 정의(구현)로 분리되는 패턴이 많다.
  - 템플릿은 컴파일 타임에 인스턴스화되므로, 정의가 헤더에 있어야 하는 경우가 많다.
- 객체 수명:
  - 자동 저장 기간(스택) / 동적 저장 기간(힙) / 정적 저장 기간(전역/정적) 모델을 구분한다.
  - 생성자/소멸자 호출 타이밍이 자원 해제 타이밍과 직결된다(RAII로 연결).

### 장단점 / 한계
- 장점:
  - 제로 오버헤드 추상화(개념적으로), 세밀한 성능 튜닝, 시스템 호출/네이티브 라이브러리 결합 용이
- 한계:
  - 메모리 안전성은 개발자가 책임져야 하며(UB 위험), 빌드/ABI/의존성 관리 난이도가 높을 수 있다.

### 대안 / 선택 기준
- 비슷한 영역 대안:
  - Rust: 메모리 안전성을 강하게 제공(대신 학습/생태계/FFI 고려)
  - Java/C#: 생산성/런타임 관리(대신 GC/런타임 오버헤드)
- 선택 기준은 “성능 목표”뿐 아니라 “팀 숙련도/빌드 파이프라인/디버깅 체계”까지 포함해야 한다.

### 실무/디버깅 포인트
- UB(정의되지 않은 동작)를 피하는 습관:
  - 초기화, 수명, 범위, 경쟁 조건을 항상 의식한다.
- 크래시/메모리 오류는 재현이 어려울 수 있어, sanitizers/로그/코어 덤프 같은 도구 체계를 갖추는 게 중요하다(환경에 따라 다름).

### 프로젝트 연결
- C++로 실시간 통신/멀티스레딩을 다룬 경험이 있으면, 자원 관리와 동시성 설계를 함께 설명할 수 있다.
  - 관련: [[프로젝트/RobotPal]]

## Pointer / Reference / const

### 개념
- 포인터(pointer)는 “주소를 값으로 가진 변수”, 레퍼런스(reference)는 “대상을 다른 이름으로 참조하는 별칭”으로 이해하면 안전하다.
- const는 “수정 불가”라는 의미지만, 무엇이 const인지(포인터 자체 vs 가리키는 대상)를 구분해야 한다.

### 왜 필요한가
- C++에서 성능/자원 제어는 결국 “데이터를 어떻게 전달/소유하느냐”로 귀결된다.
- 포인터/레퍼런스/const 규칙을 틀리면:
  - 불필요한 복사(성능)
  - 댕글링 포인터/참조(안정성)
  - API 오용(캡슐화 붕괴)
로 이어진다.

### 동작 원리
- 포인터:
  - null 가능, 재할당 가능, 수명/소유권을 명시하지 않으면 위험해질 수 있다.
  - 현대 C++에서는 “소유 포인터” 대신 smart pointer로 소유권을 표현하는 쪽이 일반적이다.
- 레퍼런스:
  - 보통 null이 아니고(참조는 반드시 대상이 있어야 함), 재바인딩이 불가하다.
  - 함수 인자로는 “복사 비용 줄이기 + 원본 접근”을 위해 많이 쓰인다.
- const:
  - `const T&`는 “읽기 전용 참조”로 복사 비용을 줄이는 대표 패턴이다.
  - `T* const p`(포인터 상수) vs `const T* p`(가리키는 대상 상수)의 차이를 명확히 설명해야 한다.

### 장단점 / 한계
- 포인터/레퍼런스는 강력하지만, 잘못 쓰면 수명 문제를 만든다.
- const는 안전하지만, 과도한 const-correctness는 코드가 복잡해질 수 있다(팀 스타일/코드베이스에 맞춰 일관되게 적용).

### 대안 / 선택 기준
- 소유권 표현:
  - 단독 소유: `std::unique_ptr`
  - 공유 소유: `std::shared_ptr`(주의: 소유 그래프 복잡, 사이클 문제)
  - 비소유 관찰: raw pointer / reference / `std::weak_ptr`(상황에 따라)
- API 설계:
  - “소유권이 넘어가나?”를 타입으로 드러내는 것이 유지보수에 유리하다.

### 실무/디버깅 포인트
- 댕글링 위험을 줄이려면:
  - 수명 정책을 문서/타입으로 고정하고, 임시 객체 참조를 피한다.
  - 참조 멤버/포인터 멤버는 객체 수명 관계를 명확히 해야 한다.

### 프로젝트 연결
- 전송 계층/버퍼 같은 코드에서는 포인터/수명/const가 곧 안정성이다.
  - 관련: [[프로젝트/RobotPal/01-구조-흐름]]

## 객체지향 C++

### 개념
- C++의 객체지향은 “가상 함수/다형성/상속”뿐 아니라, “값 타입(value semantics) + RAII”까지 포함해 이해하는 게 실무적이다.
- C++에서는 “상속 다형성”이 강력하지만 비용(가상 호출/힙 할당/수명 관리)이 있으므로 조합/템플릿 기반 다형성도 자주 사용한다.

### 왜 필요한가
- 인터페이스 기반 설계는 테스트/확장에 유리하지만, C++에서는 수명/소유권/예외 안전성까지 같이 설계해야 한다.
- 가상 소멸자/슬라이싱(object slicing) 같은 C++ 특유의 함정을 피해 가야 한다.

### 동작 원리
- 다형성:
  - 런타임 다형성: virtual 함수 + vtable
  - 컴파일타임 다형성: 템플릿(정적 다형성), CRTP 같은 패턴
- 자원 관리:
  - 클래스는 소멸자에서 자원을 해제하도록 설계하면 예외가 있어도 누수 가능성이 줄어든다(RAII).
- 슬라이싱:
  - 기반 클래스 값으로 파생 클래스를 복사하면 파생 부분이 잘리는 문제가 생길 수 있다.
  - “값으로 저장”할지 “포인터/참조로 다룰지”를 의도적으로 선택해야 한다.

### 장단점 / 한계
- 상속은 확장 포인트가 되지만 결합도를 높일 수 있다.
- 템플릿은 성능과 타입 안전성을 주지만, 컴파일 시간/오류 메시지/바이너리 크기에 영향을 줄 수 있다.

### 대안 / 선택 기준
- 런타임 다형성 vs 정적 다형성:
  - 플러그인/동적 교체가 중요하면 런타임 다형성
  - 성능/인라이닝이 중요하고 타입이 컴파일 타임에 고정되면 정적 다형성
- 클래스 설계는 “소유권/수명”을 먼저 확정한 뒤에 상속/조합을 결정하는 게 안전하다.

### 실무/디버깅 포인트
- polymorphic base에는 가상 소멸자를 두는 습관이 필요하다.
- 예외 안전성(강/기본 보장)을 고려해 자원 해제가 누락되지 않는 구조를 만든다.

### 프로젝트 연결
- 네트워크/전송 계층을 추상화할 때 런타임 다형성(인터페이스) 또는 템플릿(정적 다형성) 중 선택 기준을 설명할 수 있다.
  - 관련: [[프로젝트/RobotPal]]

## Move / Template / STL

### 개념
- Move semantics는 “자원(버퍼/핸들) 소유권을 복사 대신 이동”해 성능을 올리는 C++ 핵심 기능이다.
- 템플릿은 타입에 대한 일반화를 제공하고, STL(표준 라이브러리)은 컨테이너/알고리즘/이터레이터로 조합 가능한 재사용 구조를 제공한다.

### 왜 필요한가
- 큰 객체를 매번 복사하면 성능/메모리 비용이 커진다.
- 템플릿/알고리즘을 쓰면 “자료구조 교체/정책 교체”가 쉬워지고, 성능도 컴파일 타임 최적화를 기대할 수 있다.

### 동작 원리
- Move:
  - rvalue reference(`T&&`)는 “이동 가능한 임시/소유권을 넘길 수 있는 값”을 표현한다.
  - 이동 생성자/이동 대입은 내부 포인터를 넘기고 원본을 비워 “복사 비용”을 줄인다.
  - `std::move`는 실제 이동이 아니라 “이동 가능하다고 캐스팅”하는 표현이다(이동은 타입의 move ctor/assignment가 수행).
- 복사 생략(RVO/NRVO):
  - 반환에서 불필요한 복사가 생략될 수 있어, “무조건 move가 이득”이라고 단정할 수 없다(컴파일러 최적화에 따라 다름).
- 템플릿/STL:
  - 컨테이너는 “저장”, 알고리즘은 “처리”, 이터레이터는 “연결”이라는 분리가 핵심이다.
  - 커스텀 타입을 컨테이너 키로 쓸 때는 비교/해시/수명/복사 비용을 고려해야 한다.

### 장단점 / 한계
- Move는 성능에 큰 도움을 주지만, moved-from 객체의 상태를 잘못 가정하면 버그가 될 수 있다.
- 템플릿은 강력하지만 컴파일 시간 증가/오류 메시지 복잡/코드 부풀림 문제가 있다.

### 대안 / 선택 기준
- 큰 객체 전달은:
  - `const T&`(읽기) 또는 `T&&`(소유권 이전) 중 의도를 선택한다.
- 컨테이너 선택은:
  - 데이터 크기/접근 패턴/메모리 배치(캐시)를 같이 본다(단순 빅오만 보지 않기).

### 실무/디버깅 포인트
- move 이후 객체는 “유효하지만 상태는 미정(지정된 범위만 보장)”이라는 원칙을 지킨다.
- 성능 튜닝은 측정 없이는 단정하지 않는다(필요하면 `확인 필요`로 남기고 측정 조건을 문서화).

### 프로젝트 연결
- 프레임 단위 데이터/버퍼 처리에서 이동/복사 정책이 성능과 직결될 수 있다.
  - 관련: [[프로젝트/RobotPal]]

## C++20 변화

### 개념
- C++20은 “문법 추가”보다 “표현력/안전성/비동기/제약 조건”을 강화하는 변화가 많다.
- 면접/실무에서 자주 묻는 축:
  - Concepts(템플릿 제약)
  - Ranges(알고리즘/뷰)
  - Coroutines(비동기/제너레이터)
  - (그 외) `std::span`, `constexpr` 강화 등

### 왜 필요한가
- 템플릿 에러는 길고 어렵기 때문에, Concepts로 “요구 조건을 타입 수준에서” 명확히 하면 유지보수가 좋아질 수 있다.
- Coroutines는 비동기 흐름을 “콜백 지옥 없이” 표현할 여지를 제공한다.

### 동작 원리
- Concepts:
  - 템플릿 인자에 요구 조건을 선언해 “컴파일 타임 제약”을 걸 수 있다.
  - 잘못된 타입 사용 시 에러가 더 의미 있게 나온다.
- Ranges:
  - 기존 알고리즘 + 이터레이터 조합보다 선언적으로 파이프라인을 구성할 수 있다.
  - 뷰(view)는 지연 평가(lazy) 형태가 많아, 수명/참조를 주의해야 한다.
- Coroutines:
  - 실행을 중단/재개할 수 있는 함수 형태로, 런타임/스케줄러/프레임 관리와 결합해 사용된다.
  - “코루틴 = 스레드”가 아니라, 비동기 상태 머신을 컴파일러가 생성하는 형태로 이해하는 게 안전하다.

### 장단점 / 한계
- 장점: 표현력, 타입 제약으로 인한 안정성, 비동기 코드 가독성
- 한계: 학습 비용, 컴파일 시간/디버깅 난이도, 팀 합의 없는 도입 시 코드 스타일 혼란

### 대안 / 선택 기준
- 도입 기준:
  - 팀의 C++20 지원 환경(컴파일러/표준 라이브러리) 확인
  - 적용할 가치가 큰 영역(템플릿 라이브러리/비동기 파이프라인)에 제한적으로 도입

### 실무/디버깅 포인트
- C++20 기능은 “문법이 편하다”보다 “오류를 줄이고 유지보수를 개선하는가”로 판단한다.
