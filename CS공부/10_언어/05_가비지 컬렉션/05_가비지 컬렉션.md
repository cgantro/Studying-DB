# 가비지 컬렉션

## 한 줄 정의
가비지 컬렉션(Garbage Collection, GC)은 **더 이상 프로그램에서 사용되지 않는 메모리를 자동으로 찾아 회수하는 기능**이다.

쉽게 말하면, 사람이 일일이 치우지 않아도 런타임이 **"안 쓰는 메모리"를 정리해 주는 자동 청소 시스템**이다.

---

## 왜 배우는가
- 메모리를 직접 해제하는 건 실수하기 쉽다.
- 해제를 깜빡하면 메모리 누수가 생긴다.
- 이미 해제한 메모리를 다시 쓰면 치명적인 버그가 난다.
- GC는 이런 위험을 줄여 준다.
- 하지만 GC가 있다고 메모리를 신경 안 써도 되는 건 아니다.
- 성능 문제, 지연 시간 문제, 메모리 사용량 문제를 이해하려면 GC를 알아야 한다.

---

## 초보자용 한 줄 기억법
- **GC = 안 쓰는 메모리를 자동으로 치우는 기능**

---

## 가비지 컬렉션을 한 문장으로 다시 말하면
가비지 컬렉션은 **도달 불가능한 객체를 찾아 메모리를 자동 회수하는 메모리 관리 기법**이다.

---

## 전체 흐름

```text
객체 생성
   ↓
참조가 유지됨
   ↓
더 이상 참조되지 않음
   ↓
GC가 탐색
   ↓
회수 대상 판단
   ↓
메모리 정리
```

---

## 핵심 개념
- **힙(Heap)**: 객체가 주로 저장되는 영역
- **참조(Reference)**: 객체를 가리키는 연결
- **도달 가능성(Reachability)**: 코드에서 아직 접근 가능한가
- **세대별 수집(Generational GC)**: 객체의 생존 패턴을 이용한 GC
- **정지 시간(Pause Time)**: GC 동안 프로그램이 멈추는 시간
- **메모리 회수**: 더 이상 필요 없는 공간을 다시 사용 가능하게 만드는 것

---

## 왜 GC가 필요한가
### 수동 메모리 관리의 문제
C/C++ 같은 환경에서는 직접 메모리를 해제해야 하는 경우가 많다.

#### 실수 1: 해제를 안 함
- 메모리 누수 발생
- 오래 실행되면 점점 메모리가 커짐

#### 실수 2: 이미 해제한 메모리를 다시 사용
- 댕글링 포인터
- 크래시나 예측 불가 동작

#### 실수 3: 중복 해제
- 같은 메모리를 두 번 해제
- 프로그램이 불안정해질 수 있음

### GC의 역할
- 이런 실수를 런타임이 대신 줄여 준다
- 개발자는 비즈니스 로직에 더 집중할 수 있다

---

## GC가 작동하는 기본 흐름
GC는 보통 **"이 객체가 아직 살아 있는가?"**를 본다.

### 핵심 질문
- 아직 참조되고 있는가?
- 루트(root)에서 도달 가능한가?
- 더 이상 어떤 코드도 이 객체를 가리키지 않는가?

### 루트(root) 예시
- 스택 변수
- 전역 변수
- 레지스터
- 일부 런타임 내부 참조

---

## 도달 가능성(reachability)
GC는 보통 객체를 단순히 "안 쓰는 것 같음"으로 판단하지 않는다.
**루트에서 출발해서 도달 가능한가**를 본다.

### 예시
```text
루트 → A → B → C
```

- A, B, C는 도달 가능
- 루트에서 끊기면 더 이상 접근 불가
- 그때 회수 대상이 될 수 있다

### 쉽게 말하면
- 누군가 계속 잡고 있으면 살아 있다
- 아무도 잡고 있지 않으면 청소 대상이 된다

---

## GC의 기본 흐름

```text
1. 살아 있는 객체 찾기
2. 죽은 객체 찾기
3. 죽은 객체 메모리 회수
4. 필요하면 메모리 재배치
```

---

## 대표적인 GC 방식

### 1) 참조 추적(Tracing GC)
- 루트에서 시작해서 살아 있는 객체를 따라간다
- 도달하지 못한 객체를 회수한다
- Java, Go, Python 등에서 자주 언급된다

### 2) 참조 카운팅(Reference Counting)
- 각 객체를 몇 개가 참조하는지 센다
- 참조 수가 0이 되면 회수한다
- 단순하지만 순환 참조 문제를 해결하기 어렵다

---

## 참조 카운팅의 한계
### 순환 참조 문제
```text
A → B
B → A
```

이 경우 외부에서 둘을 참조하지 않아도 서로를 참조하고 있어서 카운트가 0이 되지 않을 수 있다.

### 그래서
- 참조 추적 방식이 더 자주 언급된다
- 참조 카운팅은 단독으로는 한계가 있다

---

## 세대별 GC(generational GC)
가비지 컬렉션에서 자주 나오는 중요한 개념이다.

### 아이디어
- **새로 생성된 객체는 빨리 죽는 경우가 많다**
- **오래 살아남는 객체는 계속 살아남는다**

그래서 객체를 나눠서 관리하면 효율이 좋아진다.

### 일반적인 이해
- Young Generation: 새 객체가 많은 곳
- Old Generation: 오래 살아남은 객체가 있는 곳

### 왜 좋은가
- 새 객체를 자주, 빠르게 정리할 수 있다
- 전체 힙을 매번 다 뒤지지 않아도 된다

---

## 세대별 GC 흐름 그림

```text
객체 생성
   ↓
Young 영역
   ↓ 살아남음
Old 영역 이동
   ↓ 오래 생존
다음 GC에서도 관리
```

---

## Stop-The-World(STW)
GC 이야기에서 자주 나오는 매우 중요한 개념이다.

### 뜻
GC가 동작하는 동안 애플리케이션 스레드가 잠시 멈추는 현상이다.

### 왜 멈추나
- 메모리를 안전하게 조사하려면 실행 중인 프로그램을 잠시 멈춰야 하는 경우가 있다
- 그래야 객체 그래프를 정확히 볼 수 있다

### 문제
- 멈추는 시간이 길면 응답 지연이 생긴다
- 실시간 서비스에서는 특히 민감하다

### 기억법
- STW = 런타임이 잠깐 멈추고 청소하는 시간

---

## 정지 시간(Pause Time)
정지 시간은 **GC 때문에 프로그램이 멈추는 시간**이다.

### 왜 중요하나
- 평균 처리량보다 사용자 체감이 더 큰 문제가 될 수 있다
- 짧은 요청이 갑자기 오래 멈추면 서비스 품질이 떨어진다

### 예시
- 웹 요청이 평소에는 20ms인데
- GC가 걸리면 순간적으로 200ms 이상 늘어날 수 있다

---

## GC의 장점
- 메모리 해제를 사람이 직접 하지 않아도 된다
- 메모리 안전성이 좋아진다
- use-after-free 같은 문제를 줄일 수 있다
- 개발 생산성이 올라간다
- 복잡한 소유권 관리 부담이 줄어든다

---

## GC의 단점
- 언제 멈출지 예측하기 어렵다
- 정지 시간이 생길 수 있다
- 메모리를 더 많이 쓸 수 있다
- GC 튜닝이 필요할 수 있다
- "자동"이라고 해서 무조건 편한 건 아니다

---

## GC가 만드는 트레이드오프

| 항목 | 장점 | 단점 |
|---|---|---|
| 개발 편의성 | 메모리 해제 부담 감소 | GC 동작 원리를 알아야 함 |
| 안전성 | 누수/이중 해제 위험 감소 | GC 지연 가능성 |
| 성능 | 런타임 최적화 가능 | pause time, 메모리 오버헤드 |
| 관리 | 자동화 | 튜닝 필요할 수 있음 |

---

## GC가 필요한 언어 예시
- Java
- Go
- Python
- JavaScript
- C#
- Ruby

### 기억법
이 언어들은 보통 GC가 기본이거나 매우 밀접하게 연결되어 있다.

---

## GC가 없는/덜 직접적인 언어 이해
- C/C++는 보통 수동 메모리 관리 이해이 강하다
- Rust는 GC 대신 소유권/빌림 규칙으로 안전성을 추구한다

### 중요한 점
- GC가 없다고 메모리 관리 문제가 사라지는 것은 아니다
- 그냥 관리 방식이 다를 뿐이다

---

## 메모리 누수와 GC
GC가 있다고 해서 메모리 누수가 절대 안 생기는 것은 아니다.

### 왜냐하면
- 참조를 계속 잡고 있으면 GC가 회수하지 못한다
- 전역 캐시나 컬렉션이 객체를 계속 붙잡고 있을 수 있다
- 의도치 않은 참조 유지가 누수처럼 보일 수 있다

### 예시
- 캐시에 넣은 객체를 빼지 않음
- 이벤트 리스너가 객체 참조를 계속 유지함
- 컬렉션에 쌓아두기만 함

---

## GC와 힙
GC는 주로 힙 메모리를 관리한다.

### 힙 특징
- 동적 할당 객체가 저장된다
- 생명주기를 런타임이 관리한다
- 객체가 많고 복잡할수록 GC 부담이 커질 수 있다

---

## 스택과 GC
스택은 보통 함수 호출과 지역 변수에 사용되고, 힙은 객체에 사용된다.

### 차이 이해
- 스택: 함수가 끝나면 자동 정리되는 느낌
- 힙: 런타임이 생존 여부를 확인해야 하는 느낌

### 따라서
GC는 주로 힙 쪽과 더 깊게 연결된다.

---

## GC의 성능 이슈

### 1) 자주 멈춤
- 짧은 pause가 자주 생길 수 있다

### 2) 오래 멈춤
- 큰 힙이나 많은 객체 그래프를 스캔할 때 발생할 수 있다

### 3) 메모리 사용량 증가
- GC가 덜 자주 일어나면 메모리 사용이 커질 수 있다

### 4) 튜닝 필요
- 언어/런타임마다 GC 설정을 조정해야 할 때가 있다

---

## 실무에서의 판단 기준
### 웹 서버
- 많은 요청이 오고 객체가 자주 생성/폐기된다
- GC pause가 응답 지연으로 보일 수 있다

### 배치 작업
- 메모리를 많이 쓰더라도 지연 시간보다 처리량이 더 중요할 수 있다

### 실시간 시스템
- pause time이 매우 민감할 수 있다
- GC 특성이 중요한 설계 요소가 된다

---

## 실전 예시

### 예시 1: 자바에서 객체 생성
- 메서드 안에서 객체를 만들고
- 더 이상 참조하지 않으면
- GC 대상이 될 수 있다

### 예시 2: 캐시 객체
- 캐시에 넣은 객체는 계속 참조되므로 회수되지 않을 수 있다
- 캐시 정책이 없으면 메모리가 점점 커질 수 있다

### 예시 3: 대용량 서비스
- 객체를 너무 많이 만들면 GC 부담이 커질 수 있다
- 불필요한 객체 생성 줄이기가 중요하다

---

## 자주 하는 실수
- GC가 있으면 메모리 관리를 신경 안 써도 된다고 생각함
  - 바로잡기: GC가 있어도 객체 참조 관리, 리소스 close, 메모리 사용량 관찰은 여전히 필요하다.
- 메모리 누수는 수동 메모리 관리에서만 생긴다고 생각함
  - 바로잡기: 스택 오버플로는 스택 공간 초과이고, 메모리 누수는 더 이상 필요 없는 힙 객체가 해제/수거되지 않는 상태다.
- 참조가 남아 있으면 GC가 못 치운다는 점을 놓침
  - 바로잡기: GC는 도달 가능한 객체를 살아 있다고 보므로, 더 이상 필요 없는 참조는 끊어야 한다.
- STW와 pause time을 구분하지 못함
  - 바로잡기: STW는 애플리케이션 스레드를 멈추는 현상이고, pause time은 그 멈춤이 지속된 시간이다.
- 세대별 GC의 의미를 모름
  - 바로잡기: 세대별 GC는 대부분의 객체가 짧게 산다는 가정으로 young/old 영역을 나누어 수거 효율을 높인다.
- 참조 추적과 참조 카운팅을 섞어 말함
  - 바로잡기: 참조 추적은 루트에서 도달 가능한 객체를 찾고, 참조 카운팅은 객체별 참조 수가 0인지로 수거한다.
- GC가 성능에 주는 영향을 과소평가함
  - 바로잡기: GC는 메모리 회수를 자동화하지만 STW, 힙 크기, 객체 생성률에 따라 지연 시간과 처리량에 영향을 줄 수 있다.

---

## 디버깅할 때 보는 순서
1. 객체 참조가 계속 남아 있는지 본다
2. 캐시/컬렉션/전역 변수에 붙잡혀 있는지 본다
3. GC가 자주 도는지 본다
4. pause time이 길어지는지 본다
5. 객체 생성량이 과도한지 본다
6. 오래 살아남는 객체가 너무 많은지 본다
7. 메모리 사용량이 계속 증가하는지 본다

---

## 비교해서 이해하기

### GC vs 수동 메모리 관리
- GC: 자동 회수, 편하지만 pause 가능
- 수동: 직접 관리, 제어 가능하지만 실수 위험 큼

### 참조 추적 vs 참조 카운팅
- 참조 추적: 루트부터 살아 있는 객체를 찾음
- 참조 카운팅: 참조 수를 센다

### Young vs Old
- Young: 새 객체가 많은 영역
- Old: 오래 살아남은 객체가 있는 영역

## ASCII 그림

### 참조 추적 이해
```text
루트
 ↓
객체 A ─→ 객체 B ─→ 객체 C

루트에서 닿지 못한 객체는 회수 대상
```

### 세대별 GC
```text
새 객체 → Young
        ↘ 살아남음
          Old
```

### STW 이해
```text
애플리케이션 실행
   ↓
GC 시작
   ↓
잠깐 멈춤(STW)
   ↓
청소
   ↓
재개
```

## 면접 포인트
면접에서는 GC를 "자동 청소" 정도로만 말하면 약하다. 트레이드오프가 중요하다.

### 꼭 말할 수 있어야 하는 것
- GC는 더 이상 참조되지 않는 메모리를 자동으로 회수하는 기법이다
- 참조 추적 방식이 일반적으로 많이 언급된다
- 세대별 GC는 객체 생존 패턴을 이용한 최적화다
- STW 때문에 pause time이 생길 수 있다
- GC는 편리하지만 성능과 메모리 사용량의 트레이드오프가 있다

### 한 번에 말하는 답변 예시
> 가비지 컬렉션은 더 이상 참조되지 않는 힙 메모리를 자동으로 회수하는 메커니즘입니다. 보통 참조 추적 방식으로 살아 있는 객체를 찾고, 세대별 GC처럼 객체 생존 패턴을 이용해 효율을 높입니다. 장점은 메모리 해제를 자동화해 안전성과 개발 편의성을 높인다는 점이고, 단점은 Stop-The-World로 인한 정지 시간과 성능 오버헤드가 생길 수 있다는 점입니다.

### 꼬리 질문 예시
- 참조 추적과 참조 카운팅 차이는 뭔가요?
- 세대별 GC는 왜 효율적인가요?
- STW는 왜 발생하나요?
- GC가 있어도 메모리 누수가 생길 수 있나요?
- GC가 실시간 시스템에 왜 부담이 될 수 있나요?

## 한 번 더 요약
- GC는 안 쓰는 메모리를 자동 회수하는 기능이다
- 참조 추적, 세대별 수집, STW가 핵심이다
- 메모리 안전성과 개발 편의성을 높여 준다
- 대신 pause time과 성능 오버헤드가 있을 수 있다
- GC가 있다고 메모리 관리가 사라지는 것은 아니다

## 초보자용 한 줄 기억법
가비지 컬렉션은 **"안 쓰는 힙 메모리를 런타임이 대신 치워 주는 시스템"**이다.
