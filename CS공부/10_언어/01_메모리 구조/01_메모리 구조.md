# 메모리 구조

## 한 줄 정의
메모리 구조는 **프로그램이 실행될 때 메모리가 어떤 용도로 나뉘어 쓰이는지** 이해하는 주제다.

쉽게 말하면, 프로그램이 실행되면 데이터를 아무 데나 넣는 게 아니라 **스택, 힙, 텍스트, 데이터, BSS 같은 영역에 역할을 나눠 저장**한다.

---

## 왜 배우는가
- 메모리를 이해하면 버그 원인을 더 잘 찾을 수 있다.
- 스택 오버플로, 힙 메모리 누수, 잘못된 참조 같은 문제를 설명할 수 있다.
- 지역 변수와 동적 할당 객체가 왜 다르게 동작하는지 이해할 수 있다.
- GC, 재귀, 함수 호출, 객체 생명주기, 성능 문제와 모두 연결된다.
- 면접에서도 스택과 힙 차이는 거의 기본 질문이다.

---

## 초보자용 한 줄 기억법
- **스택 = 함수가 쓰는 임시 책상**
- **힙 = 오래 쓰는 물건 창고**
- **텍스트/데이터/BSS = 프로그램 자체의 고정된 구역**

---

## 메모리 구조를 한 문장으로 다시 말하면
메모리 구조는 **프로그램이 실행될 때 코드, 전역 데이터, 지역 변수, 동적 객체를 어디에 두고 어떻게 생명주기를 관리하는지 설명하는 구조**다.

---

## 전체 그림

```text
프로그램 실행 시 메모리

┌───────────────────────┐
│   텍스트(text)        │  ← 코드
├───────────────────────┤
│   데이터(data)        │  ← 초기화된 전역/정적 변수
├───────────────────────┤
│   BSS                 │  ← 초기화되지 않은 전역/정적 변수
├───────────────────────┤
│   힙(heap)            │  ← 동적 할당, 위로 자라는 느낌
│        ...            │
│                       │
├───────────────────────┤
│   (빈 공간/가상주소)   │
├───────────────────────┤
│   스택(stack)         │  ← 함수 호출, 지역 변수, 아래로 자라는 느낌
└───────────────────────┘
```

> 실제 주소 배치는 운영체제, 언어, 아키텍처에 따라 다를 수 있지만, **역할 중심**으로 이해하는 것이 중요하다.

---

## 메모리 구조의 큰 분류
- **코드 영역(텍스트)**: 실행할 명령어
- **전역/정적 영역**: 프로그램 시작부터 끝까지 유지되는 데이터
- **힙**: 동적으로 생성되는 객체
- **스택**: 함수 호출과 지역 변수
- **기타**: 공유 라이브러리, 메모리 맵 영역 등

---

## 텍스트(text) 영역
### 한 줄 정의
텍스트 영역은 **실행할 프로그램 코드가 저장되는 곳**이다.

### 특징
- 보통 읽기 전용으로 취급된다
- CPU가 실행할 명령어가 들어 있다
- 함수의 실제 기계어 코드가 들어간다고 생각하면 된다

### 예시
- `main()` 함수
- `add()` 함수의 기계어 코드
- 클래스 메서드 구현 코드

### 쉽게 말하면
- 프로그램의 본문이 들어 있는 곳이다.

---

## 데이터(data) 영역
### 한 줄 정의
데이터 영역은 **초기값이 있는 전역 변수와 정적 변수**가 저장되는 곳이다.

### 예시
```c
int count = 10;
static int total = 5;
```

### 특징
- 프로그램 시작 시 값이 준비된다
- 프로그램 종료 시까지 유지된다
- 전역 상태와 연결되기 쉽다

### 주의
- 전역 변수는 편하지만, 남용하면 설계가 꼬이기 쉽다

---

## BSS 영역
### 한 줄 정의
BSS 영역은 **초기화되지 않은 전역 변수와 정적 변수**가 저장되는 곳이다.

### 예시
```c
int x;
static int y;
```

### 특징
- 실행 파일에 실제 값이 크게 저장되지 않을 수 있다
- 실행 시 0으로 초기화되는 경우가 많다
- 데이터 영역과 함께 전역/정적 데이터를 담당한다

### 기억법
- 데이터 영역: 초기값 있음
- BSS: 초기값 없음(실행 시 기본값)

---

## 스택(stack) 영역
### 한 줄 정의
스택은 **함수 호출과 지역 변수가 쌓이는 메모리 영역**이다.

### 특징
- LIFO(후입선출) 구조와 비슷하다
- 함수가 호출되면 스택 프레임이 쌓인다
- 함수가 끝나면 자동으로 정리된다
- 크기가 제한적이다
- 너무 깊은 재귀나 큰 지역 배열은 위험할 수 있다

### 예시
- 지역 변수
- 함수 매개변수
- 반환 주소
- 임시 계산 값

### 비유
- 접시를 차곡차곡 쌓았다가 맨 위부터 치우는 느낌

---

## 스택 프레임(stack frame)
함수 하나가 호출될 때 스택에 쌓이는 정보 묶음을 스택 프레임이라고 한다.

### 보통 들어있는 것
- 지역 변수
- 매개변수
- 반환 주소
- 이전 프레임 정보

### 흐름
```text
함수 호출
  ↓
스택 프레임 push
  ↓
함수 실행
  ↓
함수 종료
  ↓
스택 프레임 pop
```

---

## 힙(heap) 영역
### 한 줄 정의
힙은 **동적으로 할당한 객체가 저장되는 메모리 영역**이다.

### 특징
- 프로그램 실행 중 필요할 때 할당한다
- 크기 확장이 스택보다 유연하다
- 할당과 해제가 상대적으로 비싸다
- GC 또는 개발자가 직접 관리한다
- 오래 살아남는 객체가 많을 수 있다

### 예시
- `new`로 만든 객체
- 동적 배열
- 리스트/맵/트리 노드
- 대용량 버퍼

### 비유
- 필요한 물건을 창고에서 꺼내 쓰는 느낌

---

## 스택과 힙 비교표

| 항목 | 스택 | 힙 |
|---|---|---|
| 저장 대상 | 함수 호출, 지역 변수 | 동적 객체, 런타임 생성 데이터 |
| 생명주기 | 함수 종료 시 자동 정리 | 직접 해제 또는 GC |
| 속도 | 빠름 | 상대적으로 느림 |
| 크기 | 제한적 | 비교적 큼 |
| 관리 방식 | 자동 | 수동 또는 GC |
| 위험 | 스택 오버플로 | 메모리 누수, 단편화 |

---

## 스택과 힙 이해
- 스택은 **빠르지만 작다**
- 힙은 **크지만 관리가 필요하다**

### 기억법
- 스택: 잠깐 쓰는 임시 작업대
- 힙: 오래 보관할 수 있는 창고

---

## 스택 오버플로
스택이 너무 많이 쌓이면 문제가 생길 수 있다.

### 원인
- 재귀가 너무 깊음
- 큰 지역 변수를 너무 많이 잡음
- 함수 호출이 지나치게 중첩됨

### 예시
- 종료 조건 없는 재귀
- 아주 깊은 DFS 재귀

### 증상
- 프로그램 크래시
- stack overflow 에러

---

## 힙 메모리 문제
힙은 유연하지만 문제가 생길 수 있다.

### 1) 메모리 누수
- 더 이상 필요 없는 객체가 계속 참조되어 회수되지 않음

### 2) 단편화
- 메모리가 여기저기 조각나서 큰 연속 공간을 잡기 어려움

### 3) 과도한 할당
- 객체를 너무 많이 만들면 성능 부담이 커질 수 있다

---

## 생명주기 관점
메모리 구조를 이해할 때 가장 중요한 이해 포인트 중 하나가 **생명주기**다.

### 스택
- 함수가 시작되면 생기고
- 끝나면 사라진다

### 힙
- 필요할 때 만들고
- 더 이상 참조되지 않으면 정리된다

### 전역/정적
- 프로그램이 시작할 때 준비되고
- 끝날 때까지 유지된다

---

## 지역 변수와 동적 객체

### 지역 변수
```text
void f() {
  int a = 10;
}
```
- 보통 스택에 놓인다
- 함수가 끝나면 사라진다

### 동적 객체
```text
obj = new Object()
```
- 힙에 놓인다
- 더 이상 참조되지 않을 때까지 살아 있을 수 있다

### 차이 핵심
- 지역 변수는 함수와 함께 짧게 산다
- 동적 객체는 더 오래 살 수 있다

---

## 함수 호출과 스택
함수 호출은 스택과 매우 밀접하다.

### 흐름
```text
main()
  ↓
foo()
  ↓
bar()
```

각 함수가 호출될 때마다 스택 프레임이 쌓인다.

### 왜 중요한가
- 재귀와 연결된다
- 지역 변수 생명주기를 설명할 수 있다
- 함수 호출 흐름을 추적할 수 있다

---

## 재귀와 스택
재귀는 스택 구조와 직결된다.

### 예시
```text
factorial(3)
factorial(2)
factorial(1)
```

호출이 깊어질수록 스택이 쌓인다.

### 문제
- 종료 조건이 없으면 무한히 쌓인다
- 스택 오버플로가 날 수 있다

---

## 메모리 구조와 GC
GC는 주로 힙과 연결된다.

### 이유
- 힙의 객체는 생명주기가 유동적이다
- 더 이상 참조되지 않는 객체를 자동으로 정리해야 한다
- 스택은 함수 종료와 함께 자동 정리되므로 GC와 느낌이 다르다

### 연결 관계 이해
- 스택: 관리가 비교적 단순
- 힙: 자동 관리(GC) 또는 직접 관리가 필요

---

## 실행 파일 관점
실행 파일은 메모리에 올라올 때 여러 영역으로 나뉜다.

### 다시 보기
- 텍스트: 코드
- 데이터: 초기화된 전역/정적
- BSS: 초기화되지 않은 전역/정적
- 힙: 동적 할당
- 스택: 함수 호출

---

## 메모리 구조 ASCII 그림

```text
높은 주소
┌────────────────────┐
│ 스택               │
│ 함수 호출/지역 변수 │
├────────────────────┤
│ (빈 공간)          │
├────────────────────┤
│ 힙                 │
│ 동적 할당 객체     │
├────────────────────┤
│ BSS                │
├────────────────────┤
│ 데이터             │
├────────────────────┤
│ 텍스트             │
│ 실행 코드          │
└────────────────────┘
낮은 주소
```

> 방향은 개념 이해용이다. 실제 메모리 배치는 시스템마다 다를 수 있다.

---

## 실전 예시

### 예시 1: 함수 안 지역 변수
```text
int x = 10;
```
- 보통 스택
- 함수가 끝나면 사라짐

### 예시 2: 객체 생성
```text
Person p = new Person();
```
- 객체는 힙
- 참조 변수 p는 스택에 있을 수 있다

### 예시 3: 전역 카운터
```text
int count = 0;
```
- 데이터 영역 또는 BSS
- 프로그램 전체 동안 유지

### 예시 4: 재귀
- 스택 프레임이 계속 쌓임
- 깊어지면 위험

---

## 자주 하는 실수
- 지역 변수와 객체를 같은 곳에 둔다고 생각함
  - 바로잡기: 지역 변수 자체는 스택에 있을 수 있지만, 참조가 가리키는 객체는 힙에 생성될 수 있다.
- 스택과 힙을 단순히 "빠름/느림"으로만 암기함
  - 바로잡기: 스택/힙은 단순 속도 차이가 아니라 생명주기, 할당 방식, 소유권, 관리 방식이 다르다.
- 전역 변수와 정적 변수를 구분 못함
  - 바로잡기: 전역 변수는 접근 범위 관점이고, 정적 변수는 저장 기간/링크 방식과 관련된다.
- BSS가 무엇인지 모름
  - 바로잡기: BSS는 명시적으로 0 초기화되거나 초기값이 없는 정적 저장 기간 변수를 위한 영역이다.
- 스택 오버플로와 메모리 누수를 헷갈림
  - 바로잡기: 스택 오버플로는 스택 공간 초과이고, 메모리 누수는 더 이상 필요 없는 힙 객체가 해제/수거되지 않는 상태다.
- GC가 스택까지 관리한다고 생각함
  - 바로잡기: GC는 주로 힙 객체를 관리하고, 스택은 참조 루트로 스캔될 수 있지만 스택 프레임 자체를 GC가 임의로 정리하지 않는다.
- 힙이 크니까 무조건 안전하다고 생각함
  - 바로잡기: 힙이 커도 참조가 계속 남거나 할당이 폭증하면 GC 비용과 메모리 압박이 커질 수 있어 안전하다고 볼 수 없다.

---

## 디버깅할 때 보는 순서
1. 문제가 스택 오버플로인지 확인한다
2. 힙 메모리 누수인지 확인한다
3. 참조가 계속 남아 있는지 본다
4. 재귀 깊이와 함수 호출 수를 본다
5. 전역/정적 변수 남용 여부를 본다
6. 객체 생명주기가 예상과 다른지 본다
7. GC가 동작하는 언어라면 pause와 메모리 사용량도 본다

---

## 비교해서 이해하기

### 스택 vs 힙
- 스택: 자동 정리, 빠름, 제한적
- 힙: 유연, 동적, 관리 필요

### 텍스트 vs 데이터 vs BSS
- 텍스트: 코드
- 데이터: 초기화된 전역/정적
- BSS: 초기화되지 않은 전역/정적

### 지역 변수 vs 전역 변수
- 지역 변수: 함수 안에서 짧게 살음
- 전역 변수: 프로그램 전체 동안 살음

---

## 스택 프레임 흐름 그림

```text
함수 호출
   ↓
프레임 생성
   ↓
지역 변수/매개변수 저장
   ↓
함수 종료
   ↓
프레임 제거
```

- [ ] 텍스트/데이터/BSS의 역할을 설명할 수 있다
- [ ] 재귀와 스택 오버플로 관계를 안다
- [ ] 메모리 누수와 스택 오버플로를 구분할 수 있다
- [ ] GC가 주로 힙과 연결된다는 점을 안다
- [ ] 생명주기 관점으로 메모리 구조를 설명할 수 있다

## 면접 포인트
면접에서는 스택/힙을 외우는 것보다 **생명주기와 관리 방식**을 같이 말해야 한다.

### 꼭 말할 수 있어야 하는 것
- 스택은 함수 호출과 지역 변수, 힙은 동적 할당 객체를 다룬다
- 스택은 빠르고 자동 정리되지만 크기가 제한적이다
- 힙은 유연하지만 할당/해제 또는 GC 관리가 필요하다
- 텍스트/데이터/BSS는 실행 파일의 다른 영역이다
- 재귀가 깊으면 스택 오버플로가 날 수 있다

### 한 번에 말하는 답변 예시
> 메모리 구조는 프로그램 실행 시 메모리가 역할별로 나뉘는 방식입니다. 스택은 함수 호출과 지역 변수가 쌓이는 영역이라 빠르고 자동으로 정리되지만 크기가 제한적이고, 힙은 동적 객체가 저장되는 곳이라 유연하지만 직접 해제하거나 GC가 필요합니다. 텍스트, 데이터, BSS는 실행 파일과 전역/정적 데이터의 영역을 설명할 때 사용합니다.

### 꼬리 질문 예시
- 스택 프레임에는 뭐가 들어가나요?
- 힙 메모리 누수는 왜 생기나요?
- BSS 영역은 뭐예요?
- 재귀와 스택 오버플로는 왜 연결되나요?
- GC는 스택도 관리하나요?

## 한 번 더 요약
- 메모리 구조는 프로그램이 메모리를 어떻게 나눠 쓰는지 설명한다
- 스택은 함수 호출/지역 변수, 힙은 동적 객체다
- 텍스트/데이터/BSS는 실행 파일과 전역/정적 데이터와 관련 있다
- 생명주기와 관리 방식이 핵심이다
- 재귀, GC, 메모리 누수, 스택 오버플로와 연결된다
