# 컴파일과 인터프리터

## 한 줄 정의
컴파일과 인터프리터는 **소스 코드를 어떻게 실행 가능한 형태로 바꾸고 실행할지에 대한 두 가지 대표적인 방식**이다.

- **컴파일**: 미리 번역해 두고 실행하는 방식
- **인터프리터**: 실행할 때 바로 해석하며 실행하는 방식

쉽게 말하면:
- 컴파일 = 먼저 번역해서 완성본을 만든 뒤 실행
- 인터프리터 = 읽으면서 바로 실행

---

## 왜 배우는가
- 언어의 속도 차이와 실행 방식 차이를 이해할 수 있다.
- 왜 어떤 언어는 빌드가 필요하고, 어떤 언어는 바로 실행되는지 이해할 수 있다.
- 디버깅, 배포, 최적화, 메모리 관리와도 연결된다.
- 면접에서 "컴파일 언어/인터프리터 언어" 차이를 자주 묻는다.
- 실제로는 둘이 섞인 언어도 많아서, 단순 분류보다 실행 구조를 이해하는 게 중요하다.

---

## 초보자용 한 줄 기억법
- **컴파일 = 미리 번역**
- **인터프리터 = 실행하면서 번역**

---

## 컴파일과 인터프리터를 한 문장으로 다시 말하면
컴파일과 인터프리터는 **사람이 쓴 코드를 기계가 이해하는 수준으로 바꾸고 실행하는 시점이 다르다**는 개념이다.

---

## 전체 흐름

```text
소스 코드
   ↓
컴파일 or 인터프리트
   ↓
실행 가능한 형태
   ↓
CPU가 실행
```

---

## 크게 보면

### 컴파일 방식
```text
소스 코드
   ↓
컴파일러
   ↓
기계어/바이트코드/중간 산출물
   ↓
실행
```

### 인터프리터 방식
```text
소스 코드
   ↓
인터프리터가 읽음
   ↓
해석하면서 실행
```

---

## 컴파일이란
컴파일은 **소스 코드를 실행 전에 다른 형태로 번역해 두는 것**이다.

### 특징
- 실행 전에 번역 단계가 있다
- 번역 결과를 저장한다
- 보통 최적화를 많이 한다
- 실행 속도가 빠른 편인 경우가 많다

### 예시 언어
- C
- C++
- Rust
- Go
- Java는 조금 다르게 보지만 보통 컴파일/JIT 계열로 함께 설명된다

---

## 인터프리터란
인터프리터는 **코드를 실행할 때 한 줄 또는 단위별로 해석하면서 실행하는 것**이다.

### 특징
- 실행 전 대규모 번역이 없을 수 있다
- 바로 실행해 보기 쉽다
- 개발/실험/디버깅이 편한 경우가 많다
- 실행 시 해석 비용이 들 수 있다

### 예시 언어
- Python
- Ruby
- JavaScript
- PHP

---

## 컴파일 vs 인터프리터 비교표

| 항목 | 컴파일 | 인터프리터 |
|---|---|---|
| 실행 시점 | 실행 전에 번역 | 실행하면서 해석 |
| 속도 | 보통 빠른 편 | 보통 느릴 수 있음 |
| 배포 | 실행 파일/산출물 배포 | 소스와 런타임 의존 가능 |
| 디버깅 | 번역 후 실행 | 바로 실행하며 확인 가능 |
| 최적화 | 컴파일 시 최적화 가능 | 런타임 최적화에 의존할 수 있음 |
| 개발 편의성 | 빌드 과정이 있을 수 있음 | 빠른 실험이 쉬운 편 |

---

## 번역의 시점이 핵심
이 주제에서 가장 중요한 것은 **"언제 번역하느냐"**다.

### 컴파일
- 실행하기 전에 한 번 크게 번역한다
- 실행 시에는 번역 비용이 적다

### 인터프리터
- 실행하면서 필요한 부분을 해석한다
- 번역과 실행이 동시에 느껴진다

---

## CPU와의 관계
최종적으로 CPU는 기계어를 실행한다.

### 그래서 필요한 것
- 사람이 쓴 코드를 CPU가 직접 이해할 수 없기 때문이다
- 중간 단계가 필요하다

### 흐름 이해
```text
사람 코드 → 번역/해석 → CPU 실행
```

---

## 컴파일 결과물
컴파일은 꼭 하나의 형태만 있는 건 아니다.

### 가능 형태
- 실행 파일
- 바이트코드
- 중간 언어
- 라이브러리

### 예시
- C/C++: 네이티브 실행 파일
- Java: 바이트코드(.class)
- Kotlin/Scala: JVM 바이트코드

---

## 인터프리터 결과물 이해
인터프리터는 보통 실행 중 해석하므로, 결과물이 눈에 덜 드러날 수 있다.

### 쉽게 이해하기
- 소스 코드를 바로 실행한다
- 중간에 해석 비용이 들어간다
- 런타임이 더 많은 역할을 맡는다

---

## 컴파일만 있는 언어, 인터프리터만 있는 언어?
현실은 그렇게 단순하지 않다.

### 중요한 사실
- 많은 언어가 **혼합형**이다
- 컴파일과 인터프리터를 완전히 양분하면 오해가 생긴다

### 예시
- Java: 소스 → 바이트코드 → JVM이 실행
- Python: 구현에 따라 바이트코드와 인터프리터 구조가 섞여 있음
- JavaScript: 현대 엔진은 JIT 등 복합 구조를 사용

즉, "이 언어는 100% 컴파일, 저 언어는 100% 인터프리터"라고만 말하면 부족하다.

---

## JIT(Just-In-Time)
JIT는 컴파일과 인터프리터의 중간 느낌으로 자주 설명된다.

### 한 줄 정의
실행 중에 자주 쓰이는 코드를 즉석에서 최적화/컴파일하는 방식이다.

### 왜 쓰나
- 인터프리터의 편의성
- 컴파일의 속도 이점
둘을 함께 얻고 싶어서다.

### 대표 예
- JVM
- JavaScript 엔진(V8 등)

---

## 컴파일/인터프리터/JIT 비교표

| 방식 | 번역 시점 | 장점 | 단점 |
|---|---|---|---|
| 컴파일 | 실행 전 | 빠른 실행, 최적화 | 빌드 시간이 있을 수 있음 |
| 인터프리터 | 실행 중 | 빠른 실험, 단순한 실행 | 실행 비용이 있을 수 있음 |
| JIT | 실행 중(동적) | 빠름과 유연성 균형 | 런타임 복잡도 증가 |

---

## 컴파일과 인터프리터의 장단점

### 컴파일 장점
- 실행 속도가 빠른 편
- 미리 최적화 가능
- 배포할 때 실행 파일만 주면 되는 경우가 있다

### 컴파일 단점
- 빌드가 필요하다
- 컴파일 에러를 먼저 해결해야 한다
- 수정 후 다시 빌드해야 한다

### 인터프리터 장점
- 빠르게 실행해 볼 수 있다
- 개발 초기 실험이 쉽다
- 소규모 수정과 테스트가 편하다

### 인터프리터 단점
- 실행 시 해석 비용이 들 수 있다
- 대규모 서비스에서는 성능 최적화가 중요해질 수 있다
- 런타임 의존성이 클 수 있다

---

## 실제 예시

### 예시 1: C 프로그램
- 소스 코드 작성
- 컴파일
- 실행 파일 생성
- 실행

### 예시 2: Python 스크립트
- 소스 코드 작성
- 바로 실행
- 런타임이 해석하며 동작

### 예시 3: Java 프로그램
- 소스 코드 작성
- 바이트코드 생성
- JVM 위에서 실행

---

## 빌드와 실행 흐름

```text
[코드 작성]
   ↓
[컴파일 or 해석]
   ↓
[실행 산출물 또는 즉시 실행]
   ↓
[프로그램 실행]
```

---

## 왜 컴파일 언어가 빠른 편인가
- 실행 전에 번역과 최적화를 할 수 있다
- 실행 시에는 이미 준비된 형태를 쓰므로 오버헤드가 적을 수 있다
- CPU가 바로 처리하기 좋은 형태로 만들어 둔다

### 주의
- "무조건 빠르다"는 아니다
- 알고리즘, 메모리 사용, I/O, 런타임 구조가 더 중요할 때도 많다

---

## 왜 인터프리터 언어는 편하다고 느끼나
- 코드를 바로 바꿔 실행해 볼 수 있다
- 실험이 빠르다
- 스크립트성 작업에 좋다
- 작은 수정과 반복 테스트가 쉽다

---

## 디버깅 기준
### 컴파일러는
- 문법/타입 오류를 미리 잡아준다
- 실행 전에 문제를 발견하기 좋다

### 인터프리터는
- 실행하면서 바로 확인할 수 있다
- 동작을 보면서 수정하기 좋다

---

## 컴파일 에러 vs 런타임 에러

| 구분 | 언제 발생 | 예시 |
|---|---|---|
| 컴파일 에러 | 실행 전 | 문법 오류, 타입 오류 |
| 런타임 에러 | 실행 중 | null 참조, divide by zero |

컴파일 방식이든 인터프리터 방식이든, 런타임 에러는 따로 생길 수 있다.

---

## 정적 타이핑/동적 타이핑과의 관계
컴파일/인터프리터와 정적/동적 타이핑은 다른 축이다.

### 구분
- 컴파일/인터프리터: 언제 번역/실행하느냐
- 정적/동적 타이핑: 타입을 언제 검사하느냐

### 중요한 점
- 이 둘을 섞어 말하면 헷갈린다
- Python은 인터프리터로 자주 설명되지만 타입 시스템은 별개 축이다
- Java도 컴파일과 런타임이 함께 얽혀 있다

---

## 자주 하는 오해
- 컴파일 언어는 무조건 빠르다 → 아니다
- 인터프리터 언어는 무조건 느리다 → 아니다
- Python은 컴파일이 전혀 없다 → 구현에 따라 바이트코드 등 과정이 있다
- Java는 인터프리터 언어다 → 보통 바이트코드/JIT까지 함께 이해해야 한다
- 실행 방식과 타입 시스템을 같은 것으로 생각함

---

## 실행 방식이 설계에 주는 영향
- 배포 방식이 달라진다
- 빌드 과정이 달라진다
- 에러를 발견하는 시점이 달라진다
- 성능 특성이 달라진다
- 런타임 요구 사항이 달라진다

---

## 실전 예시

### 예시 1: 빠르게 테스트하고 싶은 스크립트
- Python 같은 인터프리터 성향 언어가 편할 수 있다
- 빠르게 실행/수정/재실행 가능

### 예시 2: 고성능 네이티브 프로그램
- C/C++ 같은 컴파일 언어가 적합할 수 있다
- 실행 속도와 메모리 제어가 중요하다

### 예시 3: 서버 애플리케이션
- Java, Go, Node.js, Python 등 다양한 선택이 있다
- 언어 자체보다 런타임 특성과 생태계가 중요할 수 있다

---

## 자주 하는 실수
- 컴파일/인터프리터를 언어 종류로만 외움
  - 바로잡기: 컴파일과 인터프리터는 언어 자체보다 실행 구현 방식이며, 한 언어도 여러 실행 방식을 가질 수 있다.
- 실행 시점 차이를 모름
  - 바로잡기: 컴파일은 실행 전에 코드를 다른 형태로 변환하는 방식이고, 인터프리터는 실행 중 코드를 읽어가며 처리하는 방식이다.
- JIT를 컴파일과 완전히 분리해서 이해함
  - 바로잡기: JIT는 실행 중 자주 쓰는 코드를 기계어로 컴파일하는 방식으로, 컴파일과 인터프리트의 중간 성격을 가진다.
- 컴파일 에러와 런타임 에러를 섞음
  - 바로잡기: 컴파일 에러는 실행 전 변환/검사 단계에서 발생하고, 런타임 에러는 실행 중 입력이나 상태 때문에 발생한다.
- 타입 시스템과 실행 방식을 같은 축으로 생각함
  - 바로잡기: 타입 시스템은 값의 종류와 연산 가능성을 다루고, 실행 방식은 코드를 언제 어떻게 기계가 실행하는지의 문제다.
- 성능 차이를 단순 언어 이름으로만 판단함
  - 바로잡기: 성능은 언어 이름보다 런타임, JIT, 최적화 수준, 메모리 모델, 라이브러리, 작업 특성에 좌우된다.

---

## 디버깅할 때 보는 순서
1. 이 언어/환경이 어떤 실행 방식을 쓰는지 확인한다
2. 실행 전 빌드가 필요한지 본다
3. 컴파일 에러인지 런타임 에러인지 구분한다
4. JIT/VM 같은 런타임이 있는지 본다
5. 성능 문제면 번역 비용인지 실행 비용인지 나눠 본다
6. 배포 환경에서 런타임이 동일한지 확인한다

---

## 비교해서 이해하기

### 컴파일 vs 인터프리터
- 컴파일: 실행 전에 번역
- 인터프리터: 실행하면서 해석

### 컴파일 vs JIT
- 컴파일: 미리 번역
- JIT: 실행 중 필요한 부분을 즉석 최적화

### 컴파일 에러 vs 런타임 에러
- 컴파일 에러: 실행 전 발견
- 런타임 에러: 실행 중 발생

- [ ] 실행 방식과 타입 시스템이 다른 축임을 안다
- [ ] 언어 이름만 보고 성능을 단정하지 않는다
- [ ] 배포/디버깅에 미치는 영향을 설명할 수 있다

## 면접 포인트
면접에서는 "컴파일 언어와 인터프리터 언어 차이"를 단순 암기보다 **실행 구조와 장단점**으로 설명해야 한다.

### 꼭 말할 수 있어야 하는 것
- 컴파일은 실행 전에 번역하고, 인터프리터는 실행하면서 해석한다
- 컴파일은 성능과 최적화에 유리할 수 있고, 인터프리터는 빠른 실험과 편의성이 장점일 수 있다
- 현실에서는 JIT처럼 혼합형이 많다
- 컴파일 에러와 런타임 에러는 다르다
- 실행 방식은 배포와 디버깅에도 영향을 준다

### 한 번에 말하는 답변 예시
> 컴파일은 소스 코드를 실행 전에 번역해 두는 방식이고, 인터프리터는 실행하면서 해석하는 방식입니다. 컴파일은 보통 실행 속도와 최적화에 유리하고, 인터프리터는 바로 실행해 보며 개발하기 편합니다. 다만 실제 언어들은 JIT나 바이트코드처럼 혼합된 구조가 많아서, 단순히 언어 이름만 보고 분류하기보다는 실행 구조를 이해하는 것이 중요합니다.

### 꼬리 질문 예시
- JIT는 왜 필요한가요?
- 컴파일 에러와 런타임 에러 차이는 뭔가요?
- Python은 왜 인터프리터 언어로 많이 부르나요?
- Java는 컴파일 언어인가요 인터프리터 언어인가요?
- 실행 방식이 디버깅에 어떤 영향을 주나요?
