# 함수형 프로그래밍

## 한 줄 정의
함수형 프로그래밍(Functional Programming)은 **상태 변경과 부작용을 줄이고, 함수 조합을 중심으로 프로그램을 구성하는 방식**이다.

쉽게 말하면, 프로그램을 "값을 바꾸는 명령들의 모음"보다 **"데이터를 함수에 넣고 새로운 결과를 만드는 과정"**으로 보는 방식이다.

---

## 왜 배우는가
- 상태 변경이 적으면 코드가 예측 가능해진다.
- 테스트하기 쉬워진다.
- 병렬 처리나 동시성에서 유리한 경우가 있다.
- 데이터 변환 파이프라인을 만들기 좋다.
- 객체지향과는 다른 관점이라, 둘을 비교해서 보면 사고가 넓어진다.
- 면접에서도 함수형/명령형 차이를 자주 묻는다.

---

## 초보자용 한 줄 기억법
- **함수형 = 바꾸지 말고 새로 만들어서 조합하기**

---

## 함수형 프로그래밍을 한 문장으로 다시 말하면
함수형 프로그래밍은 **불변 데이터와 순수 함수를 바탕으로, 함수의 합성과 고차 함수를 통해 프로그램을 구성하는 방식**이다.

---

## 전체 흐름

```text
입력 데이터
   ↓
순수 함수 1
   ↓
순수 함수 2
   ↓
순수 함수 3
   ↓
새 결과 데이터
```

### 쉽게 이해하기
- 중간 상태를 여기저기 직접 바꾸기보다
- 각 단계가 새로운 결과를 만들어 넘긴다

---

## 핵심 개념
- **불변성(immutability)**: 값이 한 번 만들어지면 바꾸지 않는다
- **순수 함수(pure function)**: 같은 입력이면 항상 같은 출력을 내고, 외부 상태를 바꾸지 않는다
- **부작용(side effect)**: 함수 외부에 영향을 주는 행동
- **고차 함수(higher-order function)**: 함수를 인자로 받거나 함수를 반환하는 함수
- **함수 합성(composition)**: 여러 함수를 이어 붙여 파이프라인처럼 만드는 것

---

## 불변성
불변성은 **데이터를 직접 수정하지 않고, 새로운 값을 만들어 쓰는 것**이다.

### 예시
- 리스트를 직접 바꾸기보다 새 리스트를 만든다
- 문자열을 수정하는 대신 새 문자열을 만든다
- 상태를 덮어쓰기보다 새 상태를 반환한다

### 왜 좋나
- 상태 추적이 쉬워진다
- 버그가 줄어들 수 있다
- 여러 곳에서 동시에 읽어도 안전한 경우가 많다

### 쉽게 말하면
- 기존 것을 바꾸지 않고 복사본을 새로 만든다

---

## 순수 함수
순수 함수는 **같은 입력이면 항상 같은 결과를 내고, 외부 상태를 바꾸지 않는 함수**다.

### 예시
```text
add(2, 3) → 5
```

언제 호출해도 `2 + 3`은 `5`다.

### 순수 함수가 아닌 예
- 전역 변수를 바꿈
- 파일에 기록함
- DB를 수정함
- 시간, 랜덤 값, 네트워크 결과에 의존함

### 왜 중요한가
- 테스트가 쉬워진다
- 결과를 예측하기 쉽다
- 함수가 독립적으로 이해되기 쉽다

---

## 부작용(side effect)
부작용은 함수가 결과값 외에 외부 세계에 영향을 주는 것이다.

### 예시
- 화면에 출력하기
- 파일 저장하기
- DB 수정하기
- 로그 남기기
- 네트워크 요청 보내기

### 함수형에서의 관점
- 부작용을 없애는 것보다 **부작용을 경계로 분리하는 것**이 중요하다
- 핵심 계산은 순수 함수로 만들고, 입출력만 바깥에서 처리하는 식으로 설계한다

---

## 고차 함수
고차 함수는 **함수를 인자로 받거나 함수를 반환하는 함수**다.

### 예시로 이해하기
- `map`, `filter`, `reduce`
- 콜백 함수 받는 함수

### 왜 유용한가
- 행동을 값처럼 다룰 수 있다
- 재사용성이 높아진다
- 코드가 조합 가능한 형태가 된다

### 비유
- 함수를 부품처럼 넣고 꺼낸다

---

## 함수 합성
함수 합성은 **작은 함수들을 이어 붙여 큰 처리를 만드는 것**이다.

### 흐름
```text
입력 → f → g → h → 결과
```

### 왜 좋은가
- 기능이 잘게 나뉜다
- 각 단계가 단순해진다
- 읽기와 테스트가 쉬워질 수 있다

### 비유
- 생산 라인처럼 한 단계씩 처리한다

---

## map / filter / reduce
함수형 프로그래밍에서 자주 나오는 대표 패턴이다.

### map
- 각 요소를 변환한다
- 예: 숫자 목록을 제곱하기

### filter
- 조건에 맞는 것만 남긴다
- 예: 짝수만 남기기

### reduce
- 여러 요소를 하나로 합친다
- 예: 합계 구하기

---

## map / filter / reduce 비교표

| 함수 | 하는 일 | 예시 |
|---|---|---|
| map | 형태를 바꿈 | 이름 목록을 대문자로 변환 |
| filter | 거르는 작업 | 100점 이상만 남김 |
| reduce | 하나로 합침 | 합계, 개수, 최댓값 |

---

## 예시로 보는 흐름

### 숫자 목록
```text
[1, 2, 3, 4, 5]
```

### map
```text
각 숫자에 2를 곱함
→ [2, 4, 6, 8, 10]
```

### filter
```text
짝수만 남김
→ [2, 4]
```

### reduce
```text
합계를 구함
→ 15
```

---

## 함수형 프로그래밍 이해
함수형은 **데이터를 변형하는 파이프라인**처럼 생각하면 쉽다.

```text
입력 데이터
   ↓
정제
   ↓
필터링
   ↓
변환
   ↓
집계
   ↓
결과
```

---

## 함수형과 명령형의 차이

### 명령형
- "어떻게 바꿀지"를 직접 쓴다
- 변수 상태를 계속 바꾼다
- 반복문과 대입이 많다

### 함수형
- "무엇을 변환할지"를 중심으로 쓴다
- 상태 변경을 줄인다
- 함수 조합이 많다

### 예시로 이해하기
- 명령형: 요리 레시피를 하나씩 직접 조작
- 함수형: 재료를 단계별 필터/변환/결합

---

## 명령형 vs 함수형 비교표

| 항목 | 명령형 | 함수형 |
|---|---|---|
| 중심 | 상태 변경 | 함수 조합 |
| 스타일 | 어떻게 | 무엇을 |
| 부작용 | 많아질 수 있음 | 줄이는 편 |
| 테스트 | 상태 의존이 커질 수 있음 | 상대적으로 쉬운 경우 많음 |
| 읽기 | 절차를 따라감 | 데이터 흐름을 따라감 |

---

## 함수형의 장점
- 예측 가능성이 높아질 수 있다
- 테스트하기 쉬워질 수 있다
- 병렬 처리에 유리한 경우가 있다
- 코드가 더 간결해질 수 있다
- 작은 함수들을 조합하기 좋아진다

---

## 함수형의 단점
- 처음 배우기 어렵다
- 익숙하지 않으면 오히려 읽기 힘들 수 있다
- 상태가 필요한 문제는 설계가 복잡해질 수 있다
- 모든 언어/프로젝트에 무조건 맞는 것은 아니다

---

## 불변성과 동시성
불변 데이터는 동시성에서 유리한 경우가 많다.

### 이유
- 여러 스레드가 동시에 읽어도 값이 바뀌지 않는다
- 공유 상태 변경이 줄어든다
- 락 사용을 줄이는 데 도움이 될 수 있다

### 주의
- 무조건 문제가 사라지는 건 아니고, 설계 전체를 봐야 한다

---

## 함수형이 잘 맞는 곳
- 데이터 변환 파이프라인
- 리스트/컬렉션 처리
- 로그/이벤트 처리
- 병렬 계산의 일부
- 순수 계산 로직

### 예시
- CSV 정제
- 숫자 리스트 가공
- 검색 결과 필터링
- 통계 계산

---

## 함수형이 덜 맞을 수 있는 곳
- GUI 이벤트가 많은 프로그램
- 상태가 많은 게임 로직
- 외부 시스템과 강하게 붙는 코드
- DB/네트워크/파일 I/O가 많은 코드

### 하지만
현실에서는 완전 함수형만 쓰기보다 **핵심 로직은 함수형 스타일로, 부작용은 경계에서 처리**하는 식이 많다.

---

## 부작용 분리 전략
함수형 스타일에서는 부작용을 아예 없애기보다 **가장 바깥으로 몰아넣는 것**이 중요하다.

### 예시 구조
```text
입력 받기
  ↓
순수 함수로 계산
  ↓
출력/저장/전송
```

### 쉽게 이해하기
- 계산은 깨끗하게
- I/O는 경계에서만

---

## 실전 예시

### 예시 1: 점수 처리
- 학생 점수 목록에서 60점 이상만 남김
- 평균 계산
- 최고점 찾기

이런 작업은 map/filter/reduce와 잘 맞는다.

### 예시 2: 결제 금액 계산
- 할인 적용
- 세금 계산
- 최종 금액 산출

핵심 계산은 순수 함수로 만들면 테스트가 쉬워진다.

### 예시 3: 데이터 정제
- 비어 있는 값 제거
- 문자열 정리
- 타입 변환
- 결과 집계

파이프라인 형태가 잘 어울린다.

---

## 자주 하는 실수
- 함수형을 map/filter/reduce만으로 이해함
  - 바로잡기: 함수형 프로그래밍은 map/filter/reduce뿐 아니라 순수 함수, 불변성, 합성, 부작용 격리가 핵심이다.
- 불변성을 너무 엄격하게만 봄
  - 바로잡기: 불변성은 모든 변경을 금지하는 것이 아니라 상태 변경 지점을 줄여 추론과 동시성을 쉽게 만드는 도구다.
- 부작용을 모두 나쁘다고 생각함
  - 바로잡기: 부작용은 I/O와 상태 변경에 필요할 수 있으며, 핵심은 부작용을 예측 가능하게 격리하는 것이다.
- 명령형과 함수형을 절대적인 이분법으로 봄
  - 바로잡기: 함수형 프로그래밍은 map/filter/reduce뿐 아니라 순수 함수, 불변성, 합성, 부작용 격리가 핵심이다.
- 순수 함수와 고차 함수 개념을 혼동함
  - 바로잡기: 순수 함수는 같은 입력에 같은 출력과 부작용 없음이 핵심이고, 고차 함수는 함수를 인자로 받거나 반환하는 함수다.
- 실제 프로젝트에서는 혼합 스타일이 많다는 걸 놓침
  - 바로잡기: 실제 프로젝트에서는 순수 함수와 불변성을 핵심 로직에 적용하고, I/O나 상태 변경은 경계에서 명령형으로 처리하는 혼합 방식이 흔하다.

---

## 디버깅할 때 보는 순서
1. 상태 변경이 어디서 일어나는지 본다
2. 순수 함수와 부작용이 섞여 있는지 확인한다
3. 데이터 흐름이 잘 보이는지 본다
4. map/filter/reduce 중 어떤 단계가 문제인지 분리한다
5. 불변 데이터가 잘 유지되는지 본다
6. 테스트가 쉬운 구조인지 확인한다

---

## 비교해서 이해하기

### 함수형 vs 명령형
- 함수형: 상태 변경을 줄이고 함수 합성
- 명령형: 변수와 상태를 직접 바꾸며 절차를 기술

### 순수 함수 vs 일반 함수
- 순수 함수: 같은 입력 → 같은 출력, 부작용 없음
- 일반 함수: 상태 변경/외부 영향 가능

### 불변 vs 가변
- 불변: 바꾸지 않고 새로 만듦
- 가변: 기존 값을 수정함

---

## ASCII 그림

### 데이터 파이프라인
```text
입력 → map → filter → reduce → 결과
```

### 순수 함수 흐름
```text
입력 1 → f → 출력 1
입력 2 → f → 출력 2
```

### 부작용 분리
```text
[순수 계산] + [I/O 경계]
```

## 면접 포인트
면접에서는 "함수형 프로그래밍이 뭐예요?"에 대해 map/filter/reduce만 말하면 약하다.

### 꼭 말할 수 있어야 하는 것
- 함수형 프로그래밍은 불변성과 순수 함수를 강조하는 방식이다
- 부작용을 줄이고 함수 합성을 통해 프로그램을 구성한다
- map/filter/reduce는 함수형 스타일의 대표 패턴이다
- 명령형은 상태를 직접 바꾸는 반면, 함수형은 데이터 변환에 초점을 둔다
- 실제 개발에서는 함수형과 명령형을 함께 쓰는 경우가 많다

### 한 번에 말하는 답변 예시
> 함수형 프로그래밍은 상태 변경과 부작용을 줄이고, 순수 함수와 불변 데이터를 중심으로 프로그램을 구성하는 방식입니다. map, filter, reduce 같은 고차 함수를 활용해 데이터 변환 파이프라인을 만들 수 있고, 명령형보다 예측 가능성과 테스트 용이성이 좋아질 수 있습니다. 다만 실제 프로젝트에서는 함수형과 명령형을 섞어 쓰는 경우가 많습니다.

### 꼬리 질문 예시
- 순수 함수가 왜 중요한가요?
- 불변성이 왜 동시성에서 도움이 되나요?
- map/filter/reduce는 각각 언제 쓰나요?
- 부작용은 왜 분리해야 하나요?
- 함수형이 안 맞는 경우도 있나요?

## 한 번 더 요약
- 함수형 프로그래밍은 상태 변경을 줄이고 함수 조합으로 설계하는 방식이다
- 불변성과 순수 함수가 핵심이다
- map/filter/reduce는 대표적인 함수형 패턴이다
- 명령형과는 "상태 변경" vs "데이터 변환"의 차이로 보면 쉽다
- 실제 프로젝트에서는 혼합해서 쓰는 경우가 많다

## 초보자용 한 줄 기억법
함수형 프로그래밍은 **"데이터를 바꾸는 대신, 함수를 이어서 새로운 결과를 만드는 방식"**이다.
