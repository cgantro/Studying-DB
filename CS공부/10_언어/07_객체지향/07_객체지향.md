# 객체지향

## 한 줄 정의
객체지향(Object-Oriented Programming, OOP)은 **데이터와 그 데이터를 다루는 행동을 객체라는 단위로 묶어 설계하는 방식**이다.

쉽게 말하면, 프로그램을 만들 때 **"사람이 현실을 생각하는 방식처럼 역할과 책임을 가진 덩어리"**로 나누는 것이다.

---

## 왜 배우는가
- 프로그램이 커질수록 코드가 금방 복잡해진다.
- 객체지향은 복잡한 시스템을 역할과 책임으로 나누는 데 유용하다.
- 유지보수, 확장, 테스트, 재사용성을 높이는 데 도움이 된다.
- 면접에서 객체지향은 거의 항상 묻는 기본 주제다.
- 실무에서는 클래스 설계와 객체 간 협력이 매우 중요하다.

---

## 초보자용 한 줄 기억법
- **객체지향 = 데이터 + 행동을 하나의 객체로 묶는 설계 방식**

---

## 객체지향을 한 문장으로 다시 말하면
객체지향은 **프로그램을 객체들의 협력으로 보고, 각 객체가 자신의 상태와 행동을 책임지게 만드는 설계 철학**이다.

---

## 전체 흐름

```text
현실의 개념 파악
   ↓
역할 나누기
   ↓
객체 설계
   ↓
객체 간 책임 분배
   ↓
협력 구조 만들기
   ↓
유지보수 가능한 코드
```

---

## 객체(Object)란
객체는 **상태(state)** 와 **행동(behavior)** 을 함께 가지는 단위다.

### 예시: 주문 객체
- 상태: 주문번호, 주문상태, 결제금액
- 행동: 결제하기, 취소하기, 상태 변경하기

### 예시: 사용자 객체
- 상태: 이름, 이메일, 비밀번호, 권한
- 행동: 로그인하기, 권한 확인하기, 비밀번호 변경하기

---

## 객체지향의 핵심 개념
- **캡슐화(Encapsulation)**
- **상속(Inheritance)**
- **다형성(Polymorphism)**
- **추상화(Abstraction)**

이 4가지를 보통 객체지향의 4대 특성으로 말한다.

---

## 4대 특성 요약표

| 특성 | 의미 | 쉽게 말하면 |
|---|---|---|
| 캡슐화 | 내부 상태와 구현을 숨기고 필요한 기능만 공개 | 안쪽은 숨기고 버튼만 보여줌 |
| 상속 | 기존 클래스의 성질을 물려받음 | 비슷한 것끼리 확장 |
| 다형성 | 같은 인터페이스로 다른 행동을 함 | 같은 명령, 다른 반응 |
| 추상화 | 중요한 것만 뽑아 단순화 | 본질만 남기기 |

---

## 캡슐화
캡슐화는 **객체 내부 상태를 외부에서 마음대로 바꾸지 못하게 하고, 필요한 기능만 공개하는 것**이다.

### 왜 필요한가
- 내부 상태가 함부로 바뀌면 버그가 생기기 쉽다
- 객체가 자기 데이터를 스스로 관리하게 하면 안전하다
- 구현을 바꿔도 외부 영향이 적다

### 예시
```text
class BankAccount {
  private balance

  deposit(amount)
  withdraw(amount)
}
```

외부에서 `balance`를 직접 바꾸지 않고, `deposit`과 `withdraw` 같은 메서드로만 접근하게 한다.

### 쉽게 말하면
- 중요한 건 안에 숨기고
- 밖에서는 정해진 버튼만 누르게 한다

---

## 상속
상속은 **기존 클래스의 속성과 동작을 물려받아 새 클래스를 만드는 것**이다.

### 예시
- 동물 → 개, 고양이
- 차량 → 자동차, 오토바이

### 장점
- 공통 코드를 재사용할 수 있다
- 계층 구조를 표현하기 쉽다

### 주의할 점
- 상속이 많아지면 설계가 굳을 수 있다
- 부모 클래스가 바뀌면 자식 클래스에도 영향이 갈 수 있다
- 무조건 상속이 좋은 건 아니다

### 기억법
- 상속은 "비슷한 것끼리 공통 부분을 물려주는 것"

---

## 다형성
다형성은 **같은 이름의 인터페이스나 메서드가 상황에 따라 다른 동작을 하는 것**이다.

### 예시
```text
동물.makeSound()
개 → 멍멍
고양이 → 야옹
```

같은 `makeSound()`를 호출해도 객체 종류에 따라 다르게 반응한다.

### 왜 중요한가
- 코드가 유연해진다
- 새로운 타입을 추가해도 구조를 크게 안 바꿀 수 있다
- 조건문 덕지덕지 붙는 코드를 줄일 수 있다

### 쉽게 말하면
- 같은 버튼을 눌렀는데, 대상에 따라 다른 동작을 한다

---

## 추상화
추상화는 **복잡한 것에서 중요한 부분만 뽑아 단순하게 보는 것**이다.

### 예시
- 자동차를 운전할 때 엔진 내부를 전부 알 필요는 없다
- 핸들, 엑셀, 브레이크 같은 핵심 조작만 알면 된다

### 코드에서의 이해
- 복잡한 내부 로직을 숨기고
- 필요한 기능만 인터페이스로 보여준다

### 쉽게 말하면
- 본질만 남기고 나머지는 덜어내는 것

---

## 객체지향 4대 특성 비교표

| 특성 | 무엇을 다루나 | 목적 |
|---|---|---|
| 캡슐화 | 내부 상태, 접근 제어 | 안전성, 유지보수성 |
| 상속 | 클래스 계층 | 재사용, 확장 |
| 다형성 | 동일한 호출의 다른 동작 | 유연성, 확장성 |
| 추상화 | 핵심만 남기기 | 복잡도 감소 |

---

## 응집도와 결합도
객체지향 설계에서 매우 중요한 개념이다.

### 응집도(Cohesion)
- 한 객체나 모듈이 얼마나 자기 일에 집중하는가
- 높을수록 좋다

### 결합도(Coupling)
- 객체들끼리 얼마나 서로 의존하는가
- 낮을수록 좋다

### 쉽게 말하면
- 응집도 높음 = 한 덩어리가 자기 역할을 잘함
- 결합도 낮음 = 서로 덜 묶여 있어서 수정이 쉬움

### 좋은 설계 기준
- 응집도는 높이고
- 결합도는 낮추는 방향으로 간다

---

## 객체지향 설계의 목표
- 변경에 강한 코드
- 읽기 쉬운 코드
- 테스트하기 쉬운 코드
- 재사용 가능한 코드
- 역할이 분명한 코드

---

## 객체 간 협력
객체지향은 혼자 있는 클래스 모음이 아니라 **객체들끼리 협력하는 구조**다.

### 예시: 주문 시스템
- User: 사용자 정보 담당
- Order: 주문 상태 담당
- Payment: 결제 담당
- Inventory: 재고 담당

각 객체가 자기 역할만 잘하면 전체 시스템이 안정적이다.

---

## 객체지향 설계 흐름

```text
문제 파악
   ↓
명사와 동사 찾기
   ↓
객체 후보 나누기
   ↓
각 객체의 책임 정하기
   ↓
객체 간 메시지 설계
   ↓
코드로 구현
```

### 명사와 동사 찾기
- 명사 → 객체 후보
- 동사 → 객체의 행동 후보

예:
- 명사: 주문, 사용자, 상품, 결제
- 동사: 결제하다, 취소하다, 조회하다

---

## 객체지향의 장점
- 유지보수가 쉬워진다
- 역할 분리가 명확해진다
- 테스트하기 쉬워질 수 있다
- 코드 재사용이 쉬워질 수 있다
- 현실 문제를 모델링하기 편하다

---

## 객체지향의 단점
- 설계가 과하면 복잡해진다
- 클래스가 너무 많아질 수 있다
- 상속을 잘못 쓰면 더 어려워진다
- 초보자가 이해하기 어려울 수 있다
- 무조건 객체지향이 정답은 아니다

---

## 객체지향의 오해
### 1) 클래스만 많이 만들면 객체지향이다?
- 아니다
- 중요한 건 역할과 책임, 협력 구조다

### 2) 상속만 쓰면 객체지향이 잘 된 건가?
- 아니다
- 상속은 도구 중 하나일 뿐이다

### 3) 객체지향은 무조건 좋은가?
- 아니다
- 상황에 따라 절차지향/함수형이 더 나을 수 있다

### 4) 객체지향은 현실을 그대로 복사하는 건가?
- 아니다
- 현실을 단순화해서 적절히 모델링하는 것이다

---

## 실제 예시

### 예시 1: 은행 계좌
- balance는 외부에서 직접 바꾸지 않는다
- deposit, withdraw 메서드로만 상태를 바꾼다
- 캡슐화가 중요한 예시다

### 예시 2: 결제 시스템
- 카드 결제, 계좌 이체, 포인트 결제를 같은 인터페이스로 다룰 수 있다
- 다형성이 유리한 예시다

### 예시 3: 동물 소리
- Dog, Cat, Bird가 같은 makeSound()를 다르게 구현한다
- 다형성 설명에 자주 쓰는 예시다

## 코드로 이해하기 예시

```text
PaymentService
  ├─ CardPayment
  ├─ BankTransferPayment
  └─ PointPayment
```

같은 `pay()`를 호출해도 실제 구현은 다를 수 있다.

## 설계에서 자주 보는 질문
- 이 객체가 정말 하나의 책임만 가지는가?
- 이 기능은 어디에 있어야 하는가?
- 상속이 맞는가, 조합이 맞는가?
- 외부에서 내부 상태를 직접 건드려도 되는가?
- 새로운 기능 추가 시 기존 코드를 많이 수정해야 하는가?

---

## 상속 vs 조합
상속은 객체지향에서 자주 나오지만, 항상 정답은 아니다.

### 상속
- 공통된 성질을 물려준다
- 계층 구조가 분명할 때 적합

### 조합(Composition)
- 필요한 기능을 가진 객체를 붙여서 사용한다
- 더 유연한 경우가 많다

### 기억법
- 상속: "is-a"
- 조합: "has-a"

---

## is-a / has-a 이해
- `Dog is an Animal` → 상속이 자연스러울 수 있음
- `Car has an Engine` → 조합이 자연스러울 수 있음

---

## 객체지향과 SOLID 이해
SOLID는 객체지향 설계를 더 잘하기 위한 원칙들이다.

### 아주 짧게만 보면
- 단일 책임 원칙
- 개방-폐쇄 원칙
- 리스코프 치환 원칙
- 인터페이스 분리 원칙
- 의존 역전 원칙

이 노트에서 다 외울 필요는 없지만, 객체지향이 **좋은 설계 원칙과 함께 간다**는 정도는 알아두면 좋다.

---

## 자주 하는 실수
- 클래스만 많으면 객체지향이라고 생각함
  - 바로잡기: 객체지향은 클래스 개수보다 책임 분리, 캡슐화, 다형성, 협력 구조가 핵심이다.
- 상속 구조를 너무 깊게 만듦
  - 바로잡기: 상속이 깊어지면 결합도가 커지므로, 재사용 목적이면 조합을 우선 고려하는 것이 좋다.
- 객체가 자기 책임을 넘어서 너무 많은 일을 함
  - 바로잡기: 객체는 자신이 맡은 책임을 중심으로 작게 설계하고, 다른 객체의 책임까지 대신하지 않게 해야 한다.
- 전역 상태를 많이 씀
  - 바로잡기: 전역 상태는 테스트와 변경 추적을 어렵게 하므로 의존성을 명시적으로 전달하는 편이 낫다.
- 다형성을 안 쓰고 if/else를 남발함
  - 바로잡기: 타입별 분기가 반복된다면 다형성이나 전략 패턴으로 책임을 분산할 수 있다.
- 추상화 수준이 너무 높거나 너무 낮음
  - 바로잡기: 추상화는 공통 규칙을 드러내되 실제 변경 이유와 사용 흐름을 숨기지 않는 수준으로 맞춰야 한다.
- 캡슐화를 무시하고 내부 필드를 직접 만짐
  - 바로잡기: 캡슐화는 상위 계층 데이터에 각 계층의 헤더가 붙어 내려가는 과정이다.

---

## 디버깅할 때 보는 순서
1. 객체가 너무 많은 책임을 갖고 있지 않은지 본다
2. 외부에서 내부 상태를 직접 건드리는지 본다
3. 상속이 과도하지 않은지 본다
4. 다형성으로 대체할 수 있는 if/else가 많은지 본다
5. 결합도가 너무 높지 않은지 본다
6. 테스트가 어려운 구조인지 본다

---

## 비교해서 이해하기

### 객체지향 vs 절차지향
- 객체지향: 객체와 협력 중심
- 절차지향: 순서와 함수 호출 중심

### 객체지향 vs 함수형
- 객체지향: 상태와 행동을 묶음
- 함수형: 상태 변화를 줄이고 함수 조합을 중시

### 상속 vs 조합
- 상속: 공통 성질 공유
- 조합: 필요한 기능을 붙임

## 면접 포인트
면접에서는 4대 특성을 외우는 것보다 **실제 설계와 연결**해야 한다.

### 꼭 말할 수 있어야 하는 것
- 객체지향은 데이터와 행동을 객체로 묶는 설계 방식이다
- 캡슐화는 내부 상태 보호, 상속은 재사용, 다형성은 유연성, 추상화는 복잡도 감소다
- 응집도는 높이고 결합도는 낮추는 방향이 좋다
- 상속보다 조합이 더 나은 경우도 많다
- 객체지향은 현실을 그대로 복사하는 것이 아니라 적절히 모델링하는 것이다

### 한 번에 말하는 답변 예시
> 객체지향은 데이터와 행동을 하나의 객체로 묶어 설계하는 방식입니다. 캡슐화는 내부 상태를 숨겨 안전성을 높이고, 상속은 공통 기능 재사용, 다형성은 같은 호출에 다른 동작을 가능하게 하며, 추상화는 복잡한 시스템에서 중요한 부분만 드러내는 역할을 합니다. 실제 설계에서는 응집도는 높이고 결합도는 낮추는 것이 중요합니다.

### 꼬리 질문 예시
- 상속보다 조합이 좋은 이유는 뭔가요?
- 다형성은 왜 중요한가요?
- 캡슐화가 왜 유지보수에 좋나요?
- 객체지향과 함수형의 차이는 뭔가요?
- 현실을 그대로 객체로 옮기면 안 되는 이유는 뭔가요?

## 한 번 더 요약
- 객체지향은 데이터와 행동을 객체로 묶는 설계 방식이다
- 4대 특성은 캡슐화, 상속, 다형성, 추상화다
- 응집도는 높이고 결합도는 낮추는 게 좋다
- 상속만이 답은 아니고 조합이 더 적합할 때도 많다
- 객체지향은 현실을 적절히 모델링하는 도구다
