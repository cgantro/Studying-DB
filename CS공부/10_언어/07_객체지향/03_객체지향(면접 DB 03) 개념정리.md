## OOP 기본

### 개념
- 객체지향(OOP)은 “데이터(상태) + 행동(메서드)”을 객체로 묶고, 객체들의 협력으로 시스템을 구성하는 설계 방식이다.
- 핵심은 “문법(클래스)”이 아니라 “설계 관점”이다.
  - 역할(Role): 어떤 책임을 수행하는가
  - 책임(Responsibility): 무엇을 보장하는가(입력/출력/불변식)
  - 협력(Collaboration): 객체들이 어떤 메시지로 상호작용하는가
- 4대 특성(캡슐화/상속/다형성/추상화)은 “도구”이고, 목적은 유지보수/확장/테스트 용이성이다.

### 왜 필요한가
- 시스템이 커질수록 변경이 잦아지고, 변경을 “국소화”하지 못하면 비용이 폭발한다.
- 객체지향은 변경을 흡수할 수 있는 경계(인터페이스/추상화/캡슐화)를 제공한다.
- 면접/실무 모두에서 중요한 질문은 “이 설계가 왜 변경에 강한가?”다.

### 동작 원리
- 캡슐화:
  - 외부는 “무엇을 할 수 있는지(인터페이스)”만 알게 하고, 내부 구현/상태는 숨긴다.
  - 불변식을 깨지 않게 만드는 장치다(예: 상태 변경은 메서드를 통해서만).
- 다형성:
  - 호출자는 “인터페이스”에 의존하고, 실제 구현은 런타임에 교체될 수 있다.
  - 구현 교체가 가능해지면 테스트 대역/전략 교체가 쉬워진다.
- 추상화:
  - 변하는 것과 변하지 않는 것을 구분하고, 변하는 부분을 추상화 뒤로 숨긴다.
  - 좋은 추상화는 “사용자 관점에서 의미가 선명하고, 변경 이유가 하나”로 모인다.

### 장단점 / 한계
- 장점:
  - 변경 비용 감소(의존성 분리), 테스트 용이성(대역 주입), 재사용성(역할 중심 설계).
- 한계:
  - 과도한 추상화/레이어링은 복잡도를 증가시킨다(“유연성”이 오히려 독이 될 수 있음).
  - 객체지향이 모든 문제의 정답은 아니다(데이터 파이프라인/단순 CRUD 등에서는 단순 구조가 더 낫기도 함).

### 대안 / 선택 기준
- 단순한 문제는 단순하게:
  - “변경 가능성이 낮고 규칙이 단순”하면 과한 패턴/계층은 피하는 게 낫다.
- 함수형/데이터 중심 설계:
  - 불변 데이터 + 순수 함수 접근은 동시성/테스트 측면에서 강점이 있다.
  - 다만 실제 시스템은 I/O/상태를 다루므로, “어디까지 순수하게 갈지”가 설계 선택이다.

### 실무/디버깅 포인트
- “객체지향스럽게”가 아니라 아래를 체크한다.
  - 한 객체/클래스가 책임을 과도하게 가지지 않는가(변경 이유가 여러 개인가)
  - 상태 변경 경로가 통제되는가(불변식이 보호되는가)
  - 인터페이스가 사용자의 관점에서 의미 있는가(추상화 누수는 없는가)

### 프로젝트 연결
- 실시간 시스템(예: WebSocket/MQTT)은 “상태 전이/도메인 이벤트”가 많아 역할/책임 분리가 특히 중요하다.
  - 관련 흐름: [[프로젝트/Autowing-Car/01-구조-흐름]]

## 설계 품질

### 개념
- 설계 품질은 “코드가 예쁘다”가 아니라, 변경/확장/테스트/운영이 수월한지를 의미한다.
- 자주 쓰는 품질 축:
  - 응집도(cohesion): 한 모듈이 하나의 목적에 집중하는가
  - 결합도(coupling): 모듈 간 의존이 얼마나 강한가
  - 변경 용이성: 요구사항 변화에 국소 수정으로 대응 가능한가
  - 관측 가능성: 장애 시 원인 추적이 쉬운가(로그/경계/에러 처리)

### 왜 필요한가
- “한 번 만들고 끝”이 아니라, 대부분의 서비스는 계속 수정된다.
- 설계 품질이 낮으면 신규 기능 추가보다 버그 수정/리팩터링에 시간이 더 든다.
- 팀 협업에서는 모듈 경계가 흐리면 커밋 충돌/중복 구현/책임 떠넘기기가 늘어난다.

### 동작 원리
- 응집도 높이기:
  - 한 클래스/모듈의 변경 이유를 1개로 줄인다(SRP로 연결).
  - “데이터 중심”으로 기능을 붙이기보다 “행동 중심”으로 책임을 묶는다.
- 결합도 낮추기:
  - 구현 클래스 대신 인터페이스/추상 타입에 의존한다(DIP로 연결).
  - 외부 시스템/프레임워크 의존을 경계(어댑터)로 몰아둔다(헥사고날/클린 아키텍처 아이디어).
- 경계의 명확화:
  - Controller/Service/Repository 같은 계층 분리뿐 아니라, 도메인/인프라 경계를 분리해 테스트를 단순화한다.

### 장단점 / 한계
- 품질을 올리려면 초기 설계 비용이 늘 수 있다.
- 다만 “변경이 잦은 축”에만 투자하면 전체 비용이 내려가는 경우가 많다.
- 품질을 지표화하기 어렵다(결국 코드 리뷰/운영 경험이 중요).

### 대안 / 선택 기준
- 리팩터링 타이밍:
  - 변경이 반복되는 영역부터(“바뀌는 축”을 먼저 찾는다)
  - 장애/성능에 직접 영향이 있는 경계(동시성/트랜잭션/외부 연동)
- “완벽한 설계”를 목표로 하기보다, 현재 요구사항과 다음 변화 가능성을 보고 선택한다.

### 실무/디버깅 포인트
- 냄새(Smell) 체크:
  - God Object/Service, Feature Envy, Shotgun Surgery(한 변경에 수정 파일이 너무 많음)
  - 조건문 폭발(전략/상태 패턴으로 흡수할 여지)
  - 순환 의존(패키지/모듈 단위로 끊기)
- 장애 분석 관점:
  - 예외가 어느 경계에서 변환/처리되는지(전역 예외 처리, 에러 코드 설계 등)도 설계 품질이다.

### 프로젝트 연결
- 배포/운영 경험이 있으면 “관측 가능성(로그/추적)”이 설계 품질과 연결된다는 점을 설명할 수 있다.
  - CI/CD 흐름: [[프로젝트/Autowing-Car/02-CI-CD-배포-이슈]]

관련 질문: [[CS공부/12_면접/전체 질문 DB/03-객체지향-질문#설계 품질]]

## 상속 / 인터페이스 / 추상화

### 개념
- 상속은 “is-a” 관계를 표현하지만, 실제로는 결합도를 강하게 만들 수 있어 조심해야 한다.
- 인터페이스/추상 클래스는 “계약(무엇을 제공하는가)”를 분리해 다형성을 가능하게 한다.
- 좋은 추상화는 사용자에게 의미가 있고, 변경 이유가 하나로 모인다.

### 왜 필요한가
- 구현을 교체해야 하는 요구(결제 수단, 저장소, 전송 계층 등)가 많기 때문에, 계약과 구현을 분리해야 변경이 쉬워진다.
- 상속 남용은 “부모 변경이 자식 전체에 전파되는 구조”를 만들기 쉽다.

### 동작 원리
- 상속의 위험:
  - 부모의 protected/내부 상태에 자식이 의존하면, 부모 변경이 곧 API 변경처럼 된다.
  - LSP(리스코프 치환)를 깨면 다형성이 오히려 버그를 만든다(부모 타입 기대를 자식이 깨는 경우).
- 조합(Composition):
  - “기능을 가진 객체를 필드로 들고 위임”하면, 구현 교체가 쉬워지고 결합도가 낮아질 때가 많다.
  - 특히 정책/전략/알고리즘은 상속보다 조합이 자연스러운 경우가 많다.
- 인터페이스의 역할:
  - 호출자는 구현이 아니라 “계약”에 의존한다(DIP).
  - 테스트에서 mock/fake를 넣기 쉬워진다.

### 장단점 / 한계
- 인터페이스가 너무 많으면 구조가 복잡해지고 추적이 어려워질 수 있다.
- 상속은 프레임워크 확장 포인트(템플릿 메서드 등)에서 유용하지만, 도메인 모델에서는 조합이 더 안전한 경우가 많다.

### 대안 / 선택 기준
- 선택 기준(경험칙):
  - “진짜 is-a(대체 가능)”인가? 아니면 “has-a(구성)”인가?
  - 상속을 택하면 “부모 변경이 자식 전체에 영향”을 감수할 수 있는가?
  - 인터페이스는 “사용자 관점의 의미 있는 계약”이 될 때만 만든다.

### 실무/디버깅 포인트
- 추상화 누수:
  - 호출자가 구현 디테일을 알아야만 올바르게 쓰는 API는 추상화가 깨진 신호다.
- 상속 트리 디버깅:
  - 문제를 찾기 어려우면 “상속 깊이” 자체가 문제일 수 있다(조합으로 리팩터링 고려).

### 프로젝트 연결
- 전송 계층(TCP/WebSocket/MQTT)처럼 “구현 교체”가 필요한 축은 인터페이스+조합이 유리하다.
  - 관련: [[프로젝트/RobotPal]]

관련 질문: [[CS공부/12_면접/전체 질문 DB/03-객체지향-질문#상속 / 인터페이스 / 추상화]]

## SOLID

### 개념
- SOLID는 객체지향 설계 원칙 묶음이며, 목적은 “변경에 강한 구조”를 만드는 것이다.
  - SRP: 단일 책임
  - OCP: 확장에는 열려 있고 변경에는 닫혀 있음
  - LSP: 하위 타입은 상위 타입으로 대체 가능
  - ISP: 인터페이스 분리
  - DIP: 추상에 의존, 구현에 비의존

### 왜 필요한가
- 원칙은 “규칙 암기”가 아니라, 코드 냄새를 발견하고 개선 방향을 잡는 체크리스트로 쓴다.
- 특히 협업에서 SRP/OCP/DIP는 구조를 단순화하고 변경 충돌을 줄이는 데 도움이 된다.

### 동작 원리
- SRP:
  - 한 클래스/모듈의 변경 이유를 1개로 만든다.
  - 로그/DB/외부 호출 같은 횡단 관심사는 분리할 여지가 있다.
- OCP:
  - 기존 코드를 덜 건드리고 기능을 추가하려면, 변하는 축을 “전략/인터페이스”로 분리해야 한다.
- LSP:
  - “부모 타입의 계약(전제/결과)”을 자식이 깨지 않도록 설계한다.
- ISP:
  - 큰 인터페이스 하나보다, 클라이언트가 필요한 것만 보게 쪼갠다.
- DIP:
  - 상위 정책(비즈니스)은 하위 구현(인프라)에 의존하지 않게 하고, 둘 다 추상에 의존하게 만든다.

### 장단점 / 한계
- 원칙을 기계적으로 적용하면 과도한 추상화로 복잡해질 수 있다.
- 특히 OCP를 “무조건 인터페이스로 만들기”로 오해하면 구조가 커진다.
- 원칙 충돌 상황도 있다(예: 성능 최적화로 응집도가 깨질 수 있음).

### 대안 / 선택 기준
- 적용 우선순위(실무 기준):
  - SRP/DIP를 먼저(구조가 정리되면 나머지 문제도 보인다).
  - LSP/ISP는 “상속/인터페이스가 실제로 등장할 때” 구체적으로 점검한다.
- “변경이 잦은 영역”에만 원칙을 강하게 적용하는 게 비용 대비 효과가 좋다.

### 실무/디버깅 포인트
- 원칙을 위반하면 어떤 문제가 생겼는지 “사례”로 설명할 수 있어야 한다.
  - SRP 위반: 한 변경이 예상치 못한 곳을 깨뜨림(회귀 버그)
  - DIP 위반: 테스트가 어려워지고, 외부 시스템 장애가 비즈니스 로직 테스트에 전파됨

### 프로젝트 연결
- 외부 의존성이 많은 시스템에서 DIP/어댑터 분리는 테스트/장애 대응을 쉽게 만든다.
  - 관련(외부 연동/운영): [[프로젝트/Autowing-Car]]

## 패턴 / 실무 설계

### 개념
- 디자인 패턴은 “자주 반복되는 설계 문제의 이름 붙은 해법”이다.
- 패턴을 외우기보다 “문제 → 패턴이 주는 구조적 해결”로 이해해야 한다.
  - 전략(Strategy): 알고리즘/정책 교체
  - 상태(State): 상태에 따른 동작 분기 캡슐화
  - 데코레이터(Decorator): 기능을 조합으로 확장
  - 템플릿 메서드(Template Method): 고정 흐름 + 확장 포인트(상속)
  - 어댑터(Adapter): 외부 의존을 내부 인터페이스로 맞춤

### 왜 필요한가
- 패턴은 팀 커뮤니케이션을 빠르게 만든다(“전략으로 빼자” 같은 대화가 가능).
- 실무에서는 프레임워크가 이미 패턴을 많이 품고 있어, 동작 이해/확장에 도움이 된다.

### 동작 원리
- 패턴의 공통 목표:
  - 변경이 잦은 부분을 분리한다.
  - 호출자와 구현의 결합을 줄인다.
  - 테스트 대역/대체 구현이 가능해진다.
- 패턴 오해:
  - 패턴은 “추상화 추가”이므로, 단순한 문제에 쓰면 오히려 복잡해진다.

### 장단점 / 한계
- 장점: 확장성, 재사용, 커뮤니케이션
- 단점: 클래스/구조가 늘어 추적이 어려워질 수 있음, 초기 비용 증가

### 대안 / 선택 기준
- 패턴 도입 기준:
  - 동일한 if/switch 분기가 여러 곳에서 반복되는가?
  - 요구사항이 자주 바뀌는 정책/규칙인가?
  - 테스트가 너무 어렵고 외부 의존이 강한가?
- “아직 변화가 없다면” 단순 구현을 유지하고, 반복/변화가 확실해질 때 리팩터링한다.

### 실무/디버깅 포인트
- 서비스 로직에서 if/else 폭발은 “정책/상태 분리” 신호일 수 있다.
- 패턴으로 분리하면 관측(로그/메트릭) 지점을 경계에 심기 쉬워 운영에도 유리하다.

### 프로젝트 연결
- 이벤트 처리/실시간 통신은 상태/전략 분리가 중요해지는 대표 영역이다.

## 객체지향 면접 꼬리질문

### 개념
- “꼬리질문”은 특정 개념의 암기 여부가 아니라, 설계 판단 기준과 트레이드오프를 묻는 질문이 많다.
- CS공부 문서에서는 꼬리질문을 “오해 포인트/판단 기준/실무 체크리스트”로 재해석해 정리한다.

### 왜 필요한가
- 같은 개념이라도 상황에 따라 정답이 바뀌므로, 상황 변수를 먼저 확인하는 습관이 필요하다.
- 예: “상속 vs 조합”은 성능이 아니라 변경/결합/확장 포인트가 핵심 변수다.

### 동작 원리
- 판단 순서(권장):
  1) 요구사항에서 변하는 축이 무엇인지 찾는다
  2) 그 축을 격리할 추상화/경계를 만든다
  3) 단순성이 깨지지 않는 수준에서만 확장한다
- 흔한 오해:
  - “OOP = 무조건 클래스 많이 만들기”
  - “인터페이스 = 무조건 좋은 것”
  - “패턴을 쓰면 고급 개발자”

### 장단점 / 한계
- 꼬리질문 대비는 문서만으로 끝나지 않고, 실제 코드/리팩터링 경험이 있어야 강해진다.
- 이 문서는 판단 기준을 정리하는 것이고, 실제 적용은 프로젝트/코드 근거와 함께 보완해야 한다.

### 대안 / 선택 기준
- 답을 내기 전에 확인해야 할 변수 예시:
  - 변경 빈도, 팀 합의된 아키텍처, 테스트 요구 수준, 성능/지연 목표, 운영 관측 체계

### 실무/디버깅 포인트
- 면접/실무 공통으로 안전한 답변 구조:
  - “상황 변수”를 먼저 말하고, 그 다음 선택 기준과 trade-off를 말한다.
