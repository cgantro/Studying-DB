# 트랜잭션

## 한 줄 정의
트랜잭션(transaction)은 **여러 데이터 변경 작업을 하나의 논리적 단위로 묶어서, 전부 성공하거나 전부 실패하게 만드는 개념**이다.

쉽게 말하면, DB에서 어떤 작업을 할 때 **중간에 멈추면 안 되는 일**을 안전하게 처리하는 묶음이다.

---

## 왜 배우는가
- 계좌이체, 주문, 결제, 재고 차감처럼 틀리면 큰 문제가 나는 작업이 많다.
- 중간에 실패했는데 일부만 반영되면 데이터가 망가진다.
- 트랜잭션은 이런 반쪽짜리 실패를 막는다.
- 동시에 여러 사용자가 DB를 쓸 때 안전하게 처리하려면 트랜잭션과 격리 수준을 함께 알아야 한다.
- 면접에서도 ACID와 commit/rollback은 거의 기본 질문이다.

---

## 초보자용 한 줄 기억법
- **트랜잭션 = 여러 DB 작업을 한 번에 묶어서, 같이 성공하거나 같이 실패하게 하는 것**

---

## 트랜잭션을 한 문장으로 다시 말하면
트랜잭션은 **DB의 일관성을 지키기 위해 관련 작업들을 하나의 단위로 다루는 실행 방식**이다.

---

## 전체 흐름

```text
작업 시작
   ↓
여러 SQL 실행
   ↓
모두 성공? ── 아니오 → ROLLBACK
   │
   예
   ↓
COMMIT
   ↓
변경 확정
```

---

## 트랜잭션이 필요한 이유
### 1) 중간 실패가 생길 수 있다
네트워크 오류, 서버 장애, 코드 버그, DB 에러는 언제든 생길 수 있다.

### 2) 일부만 반영되면 안 된다
출금만 되고 입금이 안 되면 큰일이다.

### 3) 데이터 정합성이 중요하다
DB는 단순 저장소가 아니라 **정확한 상태**를 유지해야 한다.

---

## 트랜잭션의 핵심 개념
- **원자성(Atomicity)**: 전부 성공하거나 전부 실패한다
- **일관성(Consistency)**: 제약 조건을 지킨다
- **격리성(Isolation)**: 서로 간섭하지 않는다
- **지속성(Durability)**: 커밋된 결과는 남는다

이 4개를 합쳐 **ACID**라고 한다.

---

## ACID 비교표

| 성질 | 의미 | 쉽게 말하면 |
|---|---|---|
| Atomicity | 전부 또는 전무 | 중간에 깨지면 통째로 취소 |
| Consistency | 규칙 유지 | 데이터 규칙을 지킴 |
| Isolation | 트랜잭션 분리 | 서로 방해하지 않음 |
| Durability | 결과 보존 | 커밋되면 남음 |

---

## 원자성(Atomicity)
원자성은 **트랜잭션 안의 작업이 전부 실행되거나 전부 취소되어야 한다**는 성질이다.

### 쉬운 예
계좌이체:
- A 계좌에서 10,000원 출금
- B 계좌에 10,000원 입금

중간에 서버가 죽어서 출금만 반영되면 안 된다.

### 원자성이 없으면
- 출금만 됨
- 입금만 됨
- 둘 다 안 됨

이 중 출금만 되고 입금이 안 되는 상황이 가장 위험하다.

---

## 일관성(Consistency)
일관성은 **트랜잭션 전후로 DB 제약 조건이 깨지지 않아야 한다**는 뜻이다.

### 예시
- 잔액은 음수가 되면 안 됨
- 주문 수량은 재고보다 많으면 안 됨
- 외래 키는 참조 대상이 존재해야 함

### 쉽게 말하면
- DB가 정한 규칙을 지키는 상태를 유지해야 한다

---

## 격리성(Isolation)
격리성은 **동시에 실행되는 트랜잭션이 서로에게 너무 영향을 주지 않도록 하는 성질**이다.

### 왜 필요한가
- 여러 사용자가 동시에 수정하면 충돌할 수 있다
- 격리 수준이 여기와 연결된다

### 예시로 이해하기
- 내가 작업 중인 값을 다른 사람이 너무 일찍 보면 안 될 수 있다

---

## 지속성(Durability)
지속성은 **커밋된 결과는 장애가 나도 남아 있어야 한다**는 성질이다.

### 쉽게 말하면
- 저장 완료 버튼을 누른 뒤에는 데이터가 유지되어야 한다
- 서버가 갑자기 죽어도 커밋된 내용은 복구되어야 한다

---

## ACID를 한 번에 이해하기

```text
계좌이체 트랜잭션

1. 출금
2. 입금
3. commit

중간에 실패하면?
→ rollback

결과:
- 전부 성공
- 또는 전부 실패
```

---

## commit과 rollback

### commit
- 트랜잭션의 변경을 확정한다
- 이제 DB에 반영된 결과로 본다

### rollback
- 트랜잭션 중 변경을 취소한다
- 아직 확정되지 않은 작업을 되돌린다

### 비유
- commit = 최종 제출
- rollback = 제출 전 되돌리기

---

## commit/rollback 흐름 그림

```text
BEGIN
  ↓
SQL 1
  ↓
SQL 2
  ↓
SQL 3
  ↓
성공 → COMMIT → 확정
실패 → ROLLBACK → 취소
```

---

## 트랜잭션 예시

### 예시 1: 계좌이체
```text
출금 계좌에서 10,000원 감소
입금 계좌에 10,000원 증가
```

중간에 실패하면 둘 다 취소해야 한다.

### 예시 2: 주문 처리
```text
주문 생성
재고 차감
결제 기록 저장
포인트 적립
```

이 중 하나라도 실패하면 반쪽짜리 주문이 된다.

### 예시 3: 회원가입
```text
회원 정보 저장
권한 정보 저장
프로필 생성
```

일부만 저장되면 계정 상태가 꼬인다.

---

## 트랜잭션의 시작과 끝
DB마다 문법은 조금 다르지만, 개념적으로는 다음과 같다.

```text
BEGIN
  ... 작업 ...
COMMIT 또는 ROLLBACK
```

### 쉽게 이해하기
- BEGIN: 묶음 시작
- COMMIT: 확정
- ROLLBACK: 취소

---

## 트랜잭션이 깨지면 생기는 문제
- 돈이 사라짐
- 재고가 틀어짐
- 주문이 중복되거나 반쪽만 저장됨
- 권한이나 상태가 이상해짐
- 데이터 정합성이 깨짐

즉, 트랜잭션은 DB의 "안전벨트"다.

---

## 내부 메커니즘
트랜잭션은 단순히 "묶는다"로 끝나지 않는다. DB는 이를 위해 여러 장치를 사용한다.

### 1) 로그 기록
- 변경을 기록해 둔다
- 장애 복구에 필요하다

### 2) 롤백 정보
- 취소할 수 있도록 이전 상태를 기억한다

### 3) 락
- 동시에 충돌하지 않도록 막는다

### 4) MVCC
- 읽기와 쓰기를 분리하거나 버전을 관리한다

---

## 로그와 복구
트랜잭션이 커밋되면 DB는 결과를 보존해야 한다.

### 그래서 필요한 것
- 장애가 나도 복구 가능해야 한다
- 일부만 반영되면 안 된다
- 로그를 바탕으로 REDO/UNDO 같은 복구가 가능하다

### 쉽게 이해하기
- 작업 내역을 따로 적어 둬야
- 망가졌을 때 다시 복구할 수 있다

---

## 트랜잭션 격리와 동시성
트랜잭션은 서로 간섭하지 않도록 해야 하지만, 동시에 너무 꽉 막으면 성능이 떨어진다.

### 그래서 나오는 개념
- 락
- 격리 수준
- MVCC
- 데드락

### 연결 관계
- 트랜잭션: 작업 단위
- 격리 수준: 서로를 얼마나 볼지
- 락/MVCC: 이를 구현하는 기술

---

## 트랜잭션과 격리 수준의 연결
트랜잭션이 여러 개 동시에 돌 때 어떤 이상이 생길지 조절하는 것이 격리 수준이다.

### 함께 보면 좋은 이유
- 트랜잭션이 있어도 격리 수준이 낮으면 이상 읽기가 생길 수 있다
- 격리 수준이 높아지면 안전하지만 성능이 떨어질 수 있다

---

## 트랜잭션과 락
락은 트랜잭션의 충돌을 줄이는 대표적인 방법이다.

### 쉽게 이해하기
- 락 = 지금 이 자원은 내가 쓰는 중이니 잠깐 기다려

### 장점
- 데이터가 꼬이지 않게 한다

### 단점
- 대기가 생긴다
- 데드락 가능성이 있다

---

## 트랜잭션과 MVCC
MVCC는 여러 버전을 관리해서 읽기와 쓰기를 좀 더 부드럽게 다루는 방식으로 자주 설명된다.

### 쉽게 이해하기
- 읽는 사람은 예전 버전을 볼 수 있고
- 쓰는 사람은 새 버전을 만든다

### 장점
- 읽기 동시성에 유리할 수 있다

### 주의
- DB마다 세부 구현이 다르다

---

## 트랜잭션이 너무 크면?
트랜잭션은 너무 길거나 너무 큰 것도 문제다.

### 문제
- 락이 오래 유지됨
- 경합 증가
- 실패 시 롤백 비용 증가
- 다른 요청이 대기

### 쉽게 이해하기
- 꼭 필요한 작업만 하나의 트랜잭션으로 묶는다

---

## 작은 트랜잭션 vs 큰 트랜잭션

| 항목 | 작은 트랜잭션 | 큰 트랜잭션 |
|---|---|---|
| 범위 | 짧음 | 김 |
| 롤백 비용 | 적음 | 큼 |
| 락 유지 시간 | 짧음 | 김 |
| 동시성 | 좋음 | 나쁠 수 있음 |
| 관리 | 쉬움 | 어려움 |

---

## 트랜잭션 설계 기준
### 좋은 습관
- 꼭 필요한 작업만 묶기
- 너무 오래 잡고 있지 않기
- 외부 API 호출은 트랜잭션 안에서 조심하기
- 재시도/보상 로직 고려하기

### 주의할 점
- DB 작업과 외부 서비스 호출을 너무 크게 묶으면 복잡해진다
- 트랜잭션이 길면 장애 시 영향이 커진다

```text
COMMIT
```

중간 실패하면 계정 상태가 꼬이지 않게 해야 한다.

## 자주 하는 실수
- 트랜잭션을 너무 크게 잡음
  - 바로잡기: 트랜잭션은 원자적으로 묶어야 하는 최소 작업 범위로 잡아야 락 보유 시간과 충돌을 줄일 수 있다.
- commit/rollback 개념을 헷갈림
  - 바로잡기: Commit은 변경을 확정하고, rollback은 트랜잭션 안의 변경을 취소해 이전 일관 상태로 되돌린다.
- ACID를 외우기만 하고 의미를 모름
  - 바로잡기: ACID는 원자성, 일관성, 격리성, 지속성으로 트랜잭션이 안전하게 처리되기 위한 성질이다.
- 락과 트랜잭션을 같은 개념으로 생각함
  - 바로잡기: 트랜잭션은 작업 단위이고, 락은 격리성을 구현하기 위한 수단 중 하나다.
- 격리 수준과 트랜잭션을 섞어 말함
  - 바로잡기: 격리 수준은 동시 트랜잭션이 서로를 얼마나 보게 할지 정하는 정책이고, 락/MVCC는 이를 구현하는 방식이다.
- 외부 API 호출까지 DB 트랜잭션에 길게 묶음
  - 바로잡기: 외부 API 호출을 DB 트랜잭션 안에 오래 묶으면 락 시간이 길어지고 실패 복구가 복잡해진다.
- 롤백이 모든 시스템 문제를 해결한다고 생각함
  - 바로잡기: 배포 전에는 이전 버전 복구 방법, DB 변경 되돌림 가능성, 설정 복구 절차를 준비해야 한다.

7. 외부 서비스 호출이 트랜잭션 안에 길게 묶여 있지 않은지 본다

## 비교해서 이해하기

### 트랜잭션 vs 단일 SQL
- 단일 SQL: 한 문장 실행
- 트랜잭션: 여러 작업을 하나로 묶음

### commit vs rollback
- commit: 확정
- rollback: 취소

### 락 vs MVCC
- 락: 충돌 방지
- MVCC: 버전 분리로 동시성 개선

### 트랜잭션 vs 격리 수준
- 트랜잭션: 작업 묶음
- 격리 수준: 서로의 간섭 정도

## ASCII 그림

### 트랜잭션 흐름
```text
BEGIN
  ↓
작업 1
  ↓
작업 2
  ↓
작업 3
  ↓
성공? ─ 예 → COMMIT
        └ 아니오 → ROLLBACK
```

### 계좌이체
```text
[A 계좌 잔액 -10,000]
        ↓
[B 계좌 잔액 +10,000]
        ↓
둘 다 성공해야 완료
```

### ACID 이해
```text
A: 전부 또는 전무
C: 규칙 지킴
I: 서로 방해 적게
D: 커밋되면 남음
```

## 면접 포인트
면접에서는 ACID를 외우는 것보다 **왜 필요한지**와 **어떻게 깨지는지**를 말해야 한다.

### 꼭 말할 수 있어야 하는 것
- 트랜잭션은 여러 작업을 하나의 단위로 다루는 개념이다
- ACID는 원자성, 일관성, 격리성, 지속성이다
- commit은 확정, rollback은 취소다
- 락과 MVCC는 트랜잭션 동시성을 조절하는 데 쓰인다
- 트랜잭션이 너무 크면 성능과 경합에 문제가 생길 수 있다

### 한 번에 말하는 답변 예시
> 트랜잭션은 여러 DB 작업을 하나의 논리적 단위로 묶어서 전부 성공하거나 전부 실패하게 만드는 개념입니다. ACID로 설명하며, 원자성은 전부 또는 전무, 일관성은 제약 조건 유지, 격리성은 트랜잭션 간 간섭 최소화, 지속성은 커밋된 결과가 남는 것을 의미합니다. 실제로는 commit과 rollback, 락과 MVCC, 격리 수준과 함께 이해해야 합니다.

## 한 번 더 요약
- 트랜잭션은 여러 DB 작업을 하나로 묶는 안전장치다
- ACID는 트랜잭션의 성질을 설명한다
- commit은 확정, rollback은 취소다
- 락/MVCC/격리 수준과 연결해서 봐야 한다
- 너무 큰 트랜잭션은 성능과 경합에 불리할 수 있다
