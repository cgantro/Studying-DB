# 쿼리 최적화

## 한 줄 정의
쿼리 최적화(query optimization)는 **SQL을 더 빠르고 효율적으로 실행하도록, DB가 실행 계획을 고르고 사람이 쿼리를 개선하는 과정**이다.

쉽게 말하면, 같은 결과를 내는 SQL이라도 **어떻게 실행하느냐에 따라 속도가 크게 달라지므로**, DB가 덜 일하게 만드는 방법을 찾는 일이다.

---

## 왜 배우는가
- SQL은 맞게 썼는데도 너무 느릴 수 있다.
- 느린 이유는 문법이 아니라 실행 방식일 수 있다.
- 대용량 데이터에서는 작은 차이도 성능 차이를 크게 만든다.
- 실무에서 가장 흔한 문제 중 하나가 "느린 쿼리"다.
- 면접에서도 EXPLAIN, 인덱스, JOIN 순서 같은 질문이 자주 나온다.

---

## 초보자용 한 줄 기억법
- **쿼리 최적화 = DB가 덜 힘들게 일하도록 SQL을 다듬는 것**

---

## 쿼리 최적화를 한 문장으로 다시 말하면
쿼리 최적화는 **실행 계획을 보고, 인덱스와 조건, 조인 방식, 데이터 접근량을 줄여 쿼리 성능을 개선하는 과정**이다.

---

## 전체 흐름

```text
SQL 작성
   ↓
DB가 실행 계획 생성
   ↓
테이블 접근 방식 선택
   ↓
인덱스 사용 여부 결정
   ↓
JOIN 순서/방식 결정
   ↓
실행
   ↓
느리면 분석 후 개선
```

---

## 핵심 개념
- **실행 계획(Execution Plan)**: DB가 이 SQL을 어떻게 실행할지 적은 계획
- **인덱스**: 검색을 빠르게 해 주는 구조
- **풀 테이블 스캔**: 테이블 전체를 읽는 방식
- **선택도(selectivity)**: 조건이 얼마나 데이터를 잘 걸러내는지
- **JOIN 순서**: 어떤 테이블부터 합칠지
- **카디널리티(cardinality)**: 행 수나 분포 이해
- **EXPLAIN**: 실행 계획을 보는 도구

---

## 왜 느려지는가
쿼리가 느린 이유는 보통 SQL 한 줄이 아니라 여러 원인의 합이다.

### 흔한 원인
- 인덱스를 못 탐
- 테이블 전체를 읽음
- JOIN 순서가 비효율적임
- 조건절에서 함수/형변환을 써서 인덱스가 깨짐
- 너무 많은 데이터를 가져옴
- 정렬/그룹화 비용이 큼
- 서브쿼리가 비효율적임
- 통계 정보가 오래됨

---

## 실행 계획이란
실행 계획은 **DB가 SQL을 실제로 어떤 순서와 방식으로 실행할지 정한 계획서**다.

### 실행 계획에서 보는 것
- 어떤 테이블을 먼저 읽는지
- 인덱스를 쓰는지
- 풀 스캔인지
- JOIN 방식은 무엇인지
- 정렬이 필요한지
- 필터링이 얼마나 되는지

### 왜 중요한가
겉보기에는 같은 SQL이어도 실행 계획이 다르면 속도가 완전히 달라질 수 있다.

---

## EXPLAIN이란
EXPLAIN은 실행 계획을 확인하는 대표적인 방법이다.

### 보통 보는 포인트
- 테이블 접근 방식
- 인덱스 사용 여부
- 예상 행 수
- 조인 순서
- 비용(cost)

### 쉽게 이해하기
- EXPLAIN을 보면 DB가 "무슨 일을 얼마나 많이 할지" 대략 볼 수 있다.

---

## 인덱스와 쿼리 최적화
인덱스는 최적화에서 가장 자주 언급되는 요소다.

### 인덱스가 잘 먹는 조건
- WHERE 조건이 선택도가 높다
- 조인 키로 자주 쓴다
- 정렬과 범위 검색에 유리하다

### 인덱스를 못 타는 경우
- 컬럼에 함수가 걸림
- 컬럼에 형변환이 걸림
- 조건이 너무 애매함
- 너무 많은 행이 걸려 선택도가 낮음
- 옵티마이저가 풀 스캔이 더 낫다고 판단함

### 예시
```sql
-- 인덱스를 못 탈 수 있는 예
WHERE DATE(created_at) = '2026-04-08'

-- 더 나은 예
WHERE created_at >= '2026-04-08 00:00:00'
  AND created_at <  '2026-04-09 00:00:00'
```

---

## 풀 테이블 스캔 vs 인덱스 스캔

| 방식 | 의미 | 장점 | 단점 |
|---|---|---|---|
| 풀 테이블 스캔 | 테이블 전체를 읽음 | 단순함 | 데이터가 많으면 느림 |
| 인덱스 스캔 | 인덱스를 따라 필요한 행만 읽음 | 빠를 수 있음 | 인덱스가 필요하고, 항상 유리하진 않음 |

### 기억법
- 풀 스캔: 책 한 권을 처음부터 끝까지 읽기
- 인덱스 스캔: 색인 보고 필요한 페이지만 찾기

---

## 선택도(selectivity)
선택도는 **조건이 데이터를 얼마나 잘 걸러내는지**를 나타낸다.

### 예시
- `성별 = 남` → 많은 행이 걸릴 수 있음 → 선택도 낮을 수 있음
- `주문번호 = 12345` → 거의 한 행만 걸림 → 선택도 높음

### 왜 중요한가
- 선택도가 높을수록 인덱스가 유리할 가능성이 커진다
- 선택도가 낮으면 인덱스를 타도 큰 이득이 없을 수 있다

---

## JOIN 최적화
JOIN은 느린 쿼리의 핵심 원인이 되기도 한다.

### JOIN이 느려질 수 있는 이유
- 조인할 행 수가 많다
- 조인 키에 인덱스가 없다
- 조인 순서가 비효율적이다
- 잘못된 조건으로 카티전 폭발에 가까운 결과가 생긴다

### 좋은 판단 기준
- 먼저 작은 결과 집합을 만들고
- 그다음 큰 테이블과 조인하면 유리할 수 있다

### 예시 느낌
```text
작은 테이블 → 큰 테이블
보다
큰 테이블 → 다시 큰 테이블
```

---

## JOIN 순서가 중요한 이유
DB 옵티마이저는 자동으로 순서를 고르지만, 설계나 통계가 안 좋으면 비효율적인 경로를 택할 수 있다.

### 생각 기준
- 먼저 많이 걸러지는 테이블은 무엇인가?
- 조인 키는 인덱스가 있는가?
- 필터 조건은 어디에 있는가?
- 결과 행 수가 급격히 늘어나는 지점은 어디인가?

---

## WHERE 조건 최적화
### 1) 함수 사용 주의
```sql
WHERE DATE(created_at) = '2026-04-08'
```
이렇게 쓰면 인덱스를 못 타는 경우가 있다.

### 2) 범위 조건 활용
```sql
WHERE created_at >= '2026-04-08 00:00:00'
  AND created_at <  '2026-04-09 00:00:00'
```
이런 식이 더 유리할 수 있다.

### 3) 컬럼 가공을 피하기
- 가능하면 컬럼 자체를 직접 비교한다
- 함수는 조건 절에서 조심한다

---

## ORDER BY / GROUP BY 최적화
정렬과 집계도 비용이 크다.

### ORDER BY
- 정렬할 행이 많으면 느려진다
- 인덱스를 활용하면 정렬 비용을 줄일 수 있다

### GROUP BY
- 그룹이 많을수록 비용이 커진다
- 필요한 컬럼만 모으는 것이 좋다

### 쉽게 이해하기
- 정렬과 집계는 "읽기"만이 아니라 "계산"도 많이 하는 작업이다

---

## 서브쿼리 vs JOIN
서브쿼리가 항상 나쁜 것은 아니지만, 경우에 따라 비효율적일 수 있다.

### 생각 기준
- 결과를 먼저 좁힐 수 있는가?
- 조인으로 더 명확하게 표현할 수 있는가?
- DB 옵티마이저가 더 잘 처리하는 형태는 무엇인가?

### 중요한 점
- 무조건 JOIN이 답도 아니고, 무조건 서브쿼리가 답도 아니다
- 실제 실행 계획을 보고 판단한다

---

## 통계 정보와 옵티마이저
DB는 통계 정보를 보고 실행 계획을 세운다.

### 통계 정보가 중요한 이유
- 어떤 조건이 얼마나 행을 걸러내는지 추정해야 하기 때문
- 통계가 오래되면 잘못된 실행 계획을 선택할 수 있다

### 쉽게 이해하기
- 옵티마이저는 "추측"을 한다
- 그 추측의 재료가 통계 정보다

---

## 카디널리티(cardinality)
카디널리티는 보통 **행 수나 값의 분포 이해**으로 이해한다.

### 왜 중요하나
- 어떤 조합이 얼마나 많은 결과를 만들지 예측하는 데 필요하다
- 실행 계획 선택에 영향을 준다

---

## 쿼리 최적화 흐름도

```text
느린 쿼리 발견
   ↓
EXPLAIN 확인
   ↓
인덱스 사용 여부 확인
   ↓
WHERE/JOIN/ORDER BY 검토
   ↓
불필요한 컬럼 제거
   ↓
조건 수정
   ↓
재측정
```

---

## 실전 예시

### 예시 1: 사용자 검색
```sql
SELECT * FROM users WHERE name = 'junoeasy';
```

#### 최적화 포인트
- name 컬럼에 인덱스가 있는지 확인
- LIKE `%jun%`처럼 앞에 와일드카드가 붙으면 인덱스 사용이 어려울 수 있음

---

### 예시 2: 주문 목록 조회
```sql
SELECT *
FROM orders
WHERE user_id = 10
ORDER BY created_at DESC
LIMIT 20;
```

#### 최적화 포인트
- `user_id`, `created_at` 조합 인덱스 고려
- 필요한 컬럼만 조회
- 페이징 방식도 고려

---

### 예시 3: 날짜 조회
```sql
WHERE DATE(created_at) = '2026-04-08'
```

#### 문제
- 함수 때문에 인덱스가 깨질 수 있음

#### 개선
```sql
WHERE created_at >= '2026-04-08 00:00:00'
  AND created_at <  '2026-04-09 00:00:00'
```

---

### 예시 4: JOIN이 많은 보고서
- 고객 → 주문 → 주문상세 → 상품
- 중간 결과가 너무 커질 수 있다
- 먼저 조건으로 데이터를 줄이고 조인하는지 확인해야 한다

---

## 자주 하는 실수
- 인덱스가 있으면 무조건 빠르다고 생각함
  - 바로잡기: 인덱스는 조건과 정렬, 선택도에 맞을 때 빠르며, 쓰기 비용과 저장 공간 비용도 증가시킨다.
- 쿼리 문법만 보고 성능을 판단함
  - 바로잡기: 성능은 문법보다 실행 계획, 인덱스 사용, 조인 순서, 읽는 행 수에 의해 결정된다.
- WHERE절에 함수가 들어가도 괜찮다고 생각함
  - 바로잡기: 컬럼에 함수를 적용하면 인덱스를 타기 어려울 수 있으므로 조건식을 인덱스 친화적으로 바꾸는 것이 좋다.
- EXPLAIN을 안 보고 감으로 튜닝함
  - 바로잡기: EXPLAIN으로 실제 접근 방식, 예상 행 수, 조인 방식, 인덱스 사용 여부를 확인해야 한다.
- SELECT *를 아무 생각 없이 사용함
  - 바로잡기: 필요한 컬럼만 조회하면 I/O, 네트워크 전송량, 커버링 인덱스 활용 가능성을 개선할 수 있다.
- 조인 순서와 결과 행 수를 무시함
  - 바로잡기: 조인은 중간 결과 행 수가 커질수록 비용이 커지므로 선택도가 높은 조건과 적절한 인덱스를 고려해 실행 계획을 봐야 한다.
- 통계 정보가 오래돼도 신경 안 씀
  - 바로잡기: 오래된 통계는 옵티마이저가 잘못된 실행 계획을 고르게 만들 수 있다.
- 조회가 느리면 DB만 탓함
  - 바로잡기: 느린 조회는 DB뿐 아니라 네트워크, 애플리케이션 N+1, 직렬화, 캐시 부재, 락 대기, 커넥션 풀 부족도 함께 확인해야 한다.

---

## 초보자용 비교
### 빠른 쿼리 vs 좋은 쿼리
- 빠른 쿼리: 지금 당장 잘 도는 쿼리
- 좋은 쿼리: 유지보수, 인덱스 활용, 확장성까지 고려한 쿼리

### 인덱스가 좋은 경우 vs 아닌 경우
- 좋은 경우: 특정 행을 빨리 찾을 때
- 아닌 경우: 너무 많은 행이 걸릴 때

### 풀 스캔 vs 인덱스
- 풀 스캔: 다 읽음
- 인덱스: 필요한 것만 좁혀 읽음

## ASCII 그림

### 느린 쿼리 분석 흐름
```text
느림 발견
   ↓
EXPLAIN 보기
   ↓
풀 스캔? 인덱스 미사용?
   ↓
WHERE/JOIN/ORDER BY 수정
   ↓
재측정
```

### 인덱스 이해
```text
테이블 전체 읽기 = 책 한 권 처음부터 끝까지
인덱스 사용 = 색인 보고 필요한 페이지로 이동
```

### JOIN 이해
```text
작게 줄이기 → 조인
큰 것끼리 바로 조인 → 비싸질 수 있음
```

- [ ] EXPLAIN으로 무엇을 보는지 안다
- [ ] 인덱스가 왜 중요한지 설명할 수 있다
- [ ] 풀 테이블 스캔과 인덱스 스캔 차이를 안다
- [ ] WHERE 절의 함수가 왜 위험할 수 있는지 안다
- [ ] JOIN 순서가 성능에 영향을 줄 수 있다는 점을 안다
- [ ] SELECT *가 항상 좋은 선택이 아님을 안다

## 면접 포인트
면접에서는 "어떻게 빨라지게 할 건가요?"를 묻는 식으로 나온다.

### 꼭 말할 수 있어야 하는 것
- 쿼리 최적화는 실행 계획을 보고 병목을 찾는 과정이다
- 인덱스, JOIN 순서, WHERE 조건, 정렬, 집계가 중요하다
- EXPLAIN으로 실행 계획을 확인한다
- 인덱스가 항상 좋은 것은 아니며, 선택도와 데이터 분포를 봐야 한다
- 조회가 느리다고 무작정 DB만 탓하면 안 된다

### 한 번에 말하는 답변 예시
> 쿼리 최적화는 SQL이 실제로 어떤 실행 계획으로 동작하는지 확인하고, 인덱스 활용 여부, JOIN 순서, WHERE 조건, 정렬과 집계 비용을 줄여 성능을 개선하는 과정입니다. 보통 EXPLAIN으로 실행 계획을 확인하고, 불필요한 풀 스캔이나 함수로 인한 인덱스 미사용을 줄이며, 필요한 컬럼만 조회하도록 조정합니다.

### 꼬리 질문 예시
- EXPLAIN에서 무엇을 보나요?
- 인덱스는 왜 항상 좋은 게 아닌가요?
- WHERE 절에 함수가 있으면 왜 느려질 수 있나요?
- JOIN 순서가 왜 중요한가요?
- SELECT *가 왜 비효율적일 수 있나요?

## 한 번 더 요약
- 쿼리 최적화는 더 빠르게 실행되도록 SQL과 실행 계획을 다듬는 과정이다
- EXPLAIN으로 병목을 본다
- 인덱스, JOIN, WHERE, ORDER BY, GROUP BY가 핵심이다
- 인덱스는 무조건 좋은 게 아니라 선택도가 중요하다
- 빠른 쿼리보다 좋은 쿼리를 목표로 해야 한다

## 초보자용 한 줄 기억법
쿼리 최적화는 **"DB가 덜 힘들게 일하도록 SQL을 고치는 일"**이다.
