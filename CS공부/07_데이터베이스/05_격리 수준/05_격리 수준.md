# 격리 수준

## 한 줄 정의
격리 수준(isolation level)은 **동시에 실행되는 트랜잭션끼리 서로를 얼마나 보지 못하게 할지 정하는 기준**이다.

쉽게 말하면, 여러 사람이 같은 DB를 동시에 수정하고 읽을 때 **서로의 작업을 어느 정도까지 숨길지** 정하는 규칙이다.

---

## 왜 배우는가
- DB는 여러 사용자가 동시에 쓰기 때문에, 충돌을 완전히 피할 수 없다.
- 너무 느슨하면 데이터가 흔들리고, 너무 엄격하면 성능이 떨어진다.
- 격리 수준은 **정합성 vs 성능**의 균형을 정하는 핵심이다.
- 실무에서 장애나 이상 조회를 분석할 때 격리 수준을 모르면 원인을 놓치기 쉽다.
- 면접에서도 4단계와 각 이상 현상은 자주 묻는다.

---

## 초보자용 한 줄 기억법
- **낮은 격리 수준 = 빠르지만 서로 많이 보임**
- **높은 격리 수준 = 안전하지만 느려질 수 있음**

---

## 전체 흐름

```text
트랜잭션 A 시작 ──┐
                  ├─ 같은 데이터에 접근
트랜잭션 B 시작 ──┘
        ↓
격리 수준이 낮음 → 서로의 변경이 더 많이 보일 수 있음
격리 수준이 높음 → 서로의 변경을 덜 보지만 비용이 커짐
```

---

## 핵심 개념
- **동시성**: 여러 작업이 동시에 돌아가는 성질
- **격리**: 내 작업이 다른 작업에 얼마나 영향을 덜 받는지
- **이상 현상(anomaly)**: 동시에 돌 때 생길 수 있는 비정상적인 읽기/쓰기 결과
- **트레이드오프**: 성능과 일관성은 함께 최대화되기 어렵다

---

## 격리 수준을 왜 조절하나
DB는 항상 동시에 사용된다.

예를 들어:
- 결제 시스템은 정확성이 매우 중요하다
- 통계 조회는 약간의 오차를 허용할 수 있다
- 재고 시스템은 중복 주문이나 누락이 치명적일 수 있다
- 대시보드는 약간 늦은 값이어도 괜찮을 수 있다

즉, 서비스마다 필요한 안전 수준이 다르다.

---

## 4단계 한눈에 보기

| 격리 수준 | 안전성 | 동시성/성능 | 대표 이상 현상 |
|---|---|---|---|
| Read Uncommitted | 가장 낮음 | 가장 높을 수 있음 | Dirty Read, Non-repeatable Read, Phantom Read |
| Read Committed | 중간 | 좋음 | Dirty Read 방지, 나머지는 가능 |
| Repeatable Read | 높음 | 더 낮아질 수 있음 | Non-repeatable Read 방지, Phantom Read는 DB에 따라 다름 |
| Serializable | 가장 높음 | 가장 낮음 | 이상 현상 대부분 방지 |

---

## 4단계의 이해
- **Read Uncommitted**: 남의 작업을 너무 일찍 봄
- **Read Committed**: 일단 커밋된 것만 봄
- **Repeatable Read**: 같은 걸 두 번 읽어도 같은 값이어야 함
- **Serializable**: 아예 순서를 직렬로 실행한 것처럼 보이게 함

---

## 이상 현상이란
격리 수준을 이해하려면 먼저 **어떤 이상이 생기는지** 알아야 한다.

### 1) Dirty Read
아직 커밋되지 않은 다른 트랜잭션의 변경을 읽는 것

#### 쉽게 말하면
- 상대가 아직 확정하지 않은 값을 내가 먼저 읽음
- 나중에 상대가 롤백하면 내가 본 값은 사실 존재하지 않았던 값이 된다

#### 예시
- 트랜잭션 A가 계좌 잔액을 100 → 50으로 바꿈
- 아직 커밋 안 됨
- 트랜잭션 B가 50을 읽음
- A가 롤백함
- B는 존재하지 않았던 50을 본 셈

---

### 2) Non-repeatable Read
같은 데이터를 두 번 읽었는데 중간에 값이 바뀌는 것

#### 예시
- 트랜잭션 A가 주문 금액을 읽음: 10,000원
- 다른 트랜잭션 B가 그 값을 수정하고 커밋
- A가 다시 읽음: 12,000원
- 같은 트랜잭션 안인데 값이 달라짐

---

### 3) Phantom Read
같은 조건으로 다시 조회했는데 행이 새로 추가되거나 사라지는 것

#### 예시
- 트랜잭션 A가 `WHERE age >= 20`으로 조회
- 결과가 10건
- 다른 트랜잭션 B가 새로운 20세 이상 사용자를 추가하고 커밋
- A가 같은 조건으로 다시 조회
- 결과가 11건
- 기존 행 값이 아니라 **행의 수 자체가 바뀜**

---

## 이상 현상 비교표

| 이상 현상 | 무엇이 문제인가 | 한 줄 예시 |
|---|---|---|
| Dirty Read | 커밋 안 된 값을 읽음 | 롤백될 값을 먼저 봄 |
| Non-repeatable Read | 같은 행의 값이 바뀜 | 두 번 읽었는데 숫자가 달라짐 |
| Phantom Read | 조회 결과 행 수가 바뀜 | 조건 조회했더니 행이 새로 생김 |

---

## 격리 수준과 이상 현상 관계

| 격리 수준 | Dirty Read | Non-repeatable Read | Phantom Read |
|---|---|---|---|
| Read Uncommitted | 가능 | 가능 | 가능 |
| Read Committed | 방지 | 가능 | 가능 |
| Repeatable Read | 방지 | 방지 | DB에 따라 다름 |
| Serializable | 방지 | 방지 | 방지 |

> 주의: DB 제품마다 구현 방식이 달라서, 실무에서는 정확한 동작을 DB 문서와 함께 봐야 한다.

---

## 각 단계 자세히 보기

### 1. Read Uncommitted
- 가장 느슨한 수준
- 커밋되지 않은 변경도 볼 수 있다
- 성능은 좋을 수 있지만 위험하다
- 실무에서는 거의 조심해서 봐야 한다

#### 비유
- 상대가 아직 다 말하지 않았는데 내가 중간 내용을 먼저 들음

#### 적합성
- 매우 드물게, 품질보다 빠른 응답이 더 중요한 특정 상황에서 고려할 수 있지만 일반적으로 조심한다

---

### 2. Read Committed
- 커밋된 데이터만 읽는다
- Dirty Read를 막는다
- 하지만 같은 행을 두 번 읽으면 중간에 값이 바뀔 수 있다

#### 비유
- 발표가 끝난 내용만 듣는다

#### 장점
- 가장 많이 쓰이는 기본적 수준으로 자주 언급된다
- 성능과 안전성의 균형이 괜찮다

---

### 3. Repeatable Read
- 같은 트랜잭션에서 같은 데이터를 다시 읽어도 같은 값이 보이도록 한다
- Non-repeatable Read를 막는다
- Phantom Read는 DB 구현에 따라 다르게 보일 수 있다

#### 비유
- 한 번 읽은 문장은 다음에도 같은 문장으로 보이게 함

#### 장점
- 읽기 일관성이 더 좋다

#### 주의
- 실제 DB별 구현 차이를 꼭 알아야 한다

---

### 4. Serializable
- 가장 엄격한 수준
- 여러 트랜잭션을 순서대로 실행한 것처럼 보이게 한다
- 정합성은 최고 수준이지만 성능 비용이 가장 크다

#### 비유
- 동시에 처리하지 않고 줄 세워서 하나씩 하는 것처럼 보이게 함

#### 장점
- 이상 현상을 가장 잘 막는다

#### 단점
- 동시성이 떨어지고 충돌이 많아질 수 있다

---

## 격리 수준 흐름 그림

```text
낮은 격리 수준
Read Uncommitted → Read Committed → Repeatable Read → Serializable
   빠름 -------------------------------------------→ 안전함
   위험 ------------------------------------------→ 엄격함
```

---

## 트랜잭션과 격리 수준의 관계
트랜잭션은 여러 DB 작업을 하나의 논리 단위로 묶는 것이고, 격리 수준은 그 트랜잭션들이 서로를 얼마나 보게 할지 정한다.

### 같이 봐야 하는 이유
- 트랜잭션만 알면 동시성 문제를 다 설명하지 못함
- 격리 수준만 알면 왜 트랜잭션이 필요한지 흐려짐
- 둘은 세트로 이해해야 한다

---

## MVCC와 락
격리 수준은 보통 **락(lock)** 이나 **MVCC** 같은 기술과 연결된다.

### 락
- 한 번에 한쪽만 접근하게 막는 느낌
- 안전하지만 경합이 생길 수 있다

### MVCC
- 여러 버전을 두고 읽기와 쓰기를 분리하는 느낌
- 읽기 성능과 동시성에 도움이 될 수 있다

### 중요한 점
- 격리 수준은 정책이고, 락/MVCC는 구현 기법이다
- 정책과 구현을 섞어 말하면 헷갈린다

---

## 왜 높은 격리 수준이 항상 좋은 건 아닌가
- 성능이 떨어질 수 있다
- 트랜잭션 대기 시간이 길어질 수 있다
- 교착 상태나 경합이 증가할 수 있다
- 많은 사용자가 동시에 접근하는 서비스에서는 부담이 크다

즉, **무조건 높게**가 답은 아니다.

---

## 서비스별로 생각하기

### 결제/정산
- 정합성이 매우 중요
- 높은 격리 수준이나 엄격한 트랜잭션 관리가 중요

### 통계/로그 조회
- 약간 늦은 값이 괜찮을 수 있음
- 성능이 더 중요할 수 있음

### 재고 관리
- 중복 차감, 누락 방지가 중요
- 동시성 문제가 치명적일 수 있음

### 대시보드
- 초 단위의 약간 늦은 값은 허용 가능
- 성능/응답성이 중요할 수 있음

---

## 실전 예시

### 예시 1: 계좌 잔액 조회
- 트랜잭션 A가 계좌 잔액을 읽음
- 트랜잭션 B가 같은 계좌에 입금/출금 후 커밋
- A가 다시 읽었을 때 금액이 바뀌면 문제

### 예시 2: 재고 수량
- 주문이 동시에 들어옴
- 격리 수준이 낮으면 중복 차감 가능성
- 정합성이 중요하므로 더 엄격한 제어가 필요

### 예시 3: 게시글 목록
- 조회 중간에 새 글이 올라오면 결과 수가 바뀔 수 있음
- 이런 행 변화가 Phantom Read와 연결된다

---

## 자주 하는 실수
- Dirty Read, Non-repeatable Read, Phantom Read를 섞음
  - 바로잡기: Dirty Read는 미커밋 데이터 읽기, Non-repeatable Read는 같은 행 재조회 값 변경, Phantom Read는 조건 결과 집합 변경이다.
- Read Committed와 Repeatable Read의 차이를 흐림
  - 바로잡기: Read Committed는 커밋된 값만 읽고, Repeatable Read는 트랜잭션 동안 같은 행의 반복 조회 결과를 유지하려 한다.
- 격리 수준을 락과 같은 개념으로만 이해함
  - 바로잡기: 격리 수준은 동시 트랜잭션이 서로를 얼마나 보게 할지 정하는 정책이고, 락/MVCC는 이를 구현하는 방식이다.
- DB마다 구현 차이가 있다는 점을 놓침
  - 바로잡기: 같은 격리 수준 이름이라도 DBMS의 MVCC/락 구현에 따라 실제 동작이 다를 수 있다.
- 높은 격리 수준이 무조건 정답이라고 생각함
  - 바로잡기: 격리 수준은 동시 트랜잭션이 서로를 얼마나 보게 할지 정하는 정책이고, 락/MVCC는 이를 구현하는 방식이다.
- 성능 문제를 격리 수준만의 문제로 오해함
  - 바로잡기: 격리 수준은 동시 트랜잭션이 서로를 얼마나 보게 할지 정하는 정책이고, 락/MVCC는 이를 구현하는 방식이다.

## 디버깅할 때 보는 순서
1. 어떤 격리 수준인지 확인한다
2. 문제가 읽기 이상인지 쓰기 충돌인지 구분한다
3. 트랜잭션 범위를 확인한다
4. 같은 데이터에 동시에 접근하는지 본다
5. 락/MVCC 설정과 DB 문서를 확인한다
6. 커밋/롤백 타이밍을 본다
7. 읽기 일관성이 필요한지, 성능이 더 중요한지 판단한다

## 비교해서 이해하기

### 낮은 격리 vs 높은 격리
- 낮은 격리: 빠르지만 더 많이 보인다
- 높은 격리: 안전하지만 느려질 수 있다

### Dirty vs Non-repeatable vs Phantom
- Dirty: 커밋 안 된 값을 읽음
- Non-repeatable: 같은 행 값이 바뀜
- Phantom: 조회되는 행의 수가 바뀜

### 락 vs MVCC
- 락: 접근을 막는다
- MVCC: 버전을 나눠 읽고 쓴다


## 면접 포인트
면접에서는 4단계 암기보다 **이상 현상과 트레이드오프**를 같이 말해야 한다.

### 꼭 말할 수 있어야 하는 것
- 격리 수준은 트랜잭션 간 간섭 정도를 정하는 기준이다
- 낮을수록 성능은 좋고, 높을수록 일관성은 좋다
- Dirty Read, Non-repeatable Read, Phantom Read를 구분할 수 있어야 한다
- Serializable이 가장 엄격하지만 성능 비용이 크다
- DB마다 실제 구현 차이가 있을 수 있다

### 한 번에 말하는 답변 예시
> 격리 수준은 동시에 실행되는 트랜잭션끼리 서로를 얼마나 보지 못하게 할지 정하는 기준입니다. Read Uncommitted, Read Committed, Repeatable Read, Serializable 순으로 엄격해지며, 낮을수록 성능은 좋아질 수 있지만 Dirty Read나 Non-repeatable Read, Phantom Read 같은 이상 현상이 더 쉽게 발생합니다. 반대로 높을수록 정합성은 좋아지지만 동시성 비용이 커집니다.

- 서비스 특성에 맞춰 성능과 정합성의 균형을 잡아야 한다

격리 수준은 **"동시에 일하는 트랜잭션들이 서로를 얼마나 보게 할지 정하는 규칙"**이다.
