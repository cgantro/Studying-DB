## DB 기본

### 개념
- 데이터베이스(DB)는 데이터를 “저장”하는 것 자체보다, **동시성/정합성/조회 성능/운영 기능**을 함께 제공하는 시스템이다.
- 파일 시스템에 직접 저장해도 데이터는 보관되지만, 아래를 애플리케이션이 직접 구현해야 한다.
  - 동시 수정 충돌 처리(락/버전)
  - 무결성(중복/참조 관계) 강제
  - 검색/정렬 최적화(인덱스)
  - 장애 복구/백업/권한/감사 로그
- RDBMS는 관계형 모델을 기반으로 테이블/키/제약조건/조인을 제공한다.

### 왜 필요한가
- 실무에서 가장 자주 터지는 문제는 “기능이 안 됨”보다 **데이터가 틀어지는 문제**다.
  - 중복 데이터, 참조가 끊긴 데이터, 일부만 반영된 업데이트(부분 성공) 등이 대표적이다.
- 대용량에서 조회 성능은 인덱스/실행 계획/스키마 설계가 결정한다.
- 운영에서는 백업/복제/복구(RPO/RTO)와 권한/감사 같은 요구가 필수로 등장한다.

### 동작 원리
- 제약조건:
  - PK/UK/NOT NULL/CHECK/FK 등은 “애플리케이션 버그/누락”이 있어도 DB에서 마지막 방어선을 제공한다.
- 키 설계:
  - Natural key(의미 있는 키)는 규칙이 바뀌거나 포맷이 바뀌면 위험해질 수 있다.
  - Surrogate key(의미 없는 키)는 조인이 단순하지만, “비즈니스 유일성”은 별도 유니크 제약으로 강제하는 경우가 많다.
- ID 전략:
  - Auto increment는 단순/효율적이지만 외부 노출 식별자로 쓰면 추측/스크래핑 위험이 생길 수 있다.
  - UUID는 분산에 유리하지만 인덱스/스토리지 비용과 랜덤성으로 인한 B-Tree 비효율 문제가 생길 수 있다.

### 장단점 / 한계
- DB 제약은 안전하지만, 과도하면 쓰기 성능/개발 유연성을 떨어뜨릴 수 있다.
- “무결성은 DB에서만”도, “무결성은 앱에서만”도 위험하다. 핵심 규칙은 DB에 두고, 복잡한 규칙은 앱/도메인 로직과 분리해 설계한다.

### 대안 / 선택 기준
- RDB vs NoSQL은 “정합성 요구/쿼리 패턴/운영 복잡도”로 결정한다.
  - 연결: [[CS공부/07_데이터베이스/01_RDB와 NoSQL/01_RDB와 NoSQL]]
- ID 전략은 “분산 생성 필요 여부 + 외부 노출 여부 + 인덱스/조인 패턴”으로 선택한다.

### 실무/디버깅 포인트
- 데이터가 꼬였을 때는 “어떤 무결성 규칙이 깨졌는지”부터 정리한다.
  - 중복(UK), 참조 끊김(FK), 필수 값 누락(NOT NULL), 범위 오류(CHECK)
- 장애 대응 관점에서는 “백업/복제/복구 계획”이 없는 DB는 운영이 불가능에 가깝다.

### 프로젝트 연결
- 프로젝트에서 DB 스키마/트랜잭션/배포를 다룬 사례를 이 문서의 근거로 연결해두면, 면접 답변이 단단해진다.
  - 구체 연결 문서 정리는 `확인 필요`

## SQL 기본

### 개념
- SQL은 “데이터를 선언적으로 질의/변경”하는 언어다. 작성자는 “무엇을” 말하고, DB는 “어떻게” 실행할지(실행 계획)를 결정한다.
- 기본 문장 단위:
  - 조회: SELECT
  - 변경: INSERT/UPDATE/DELETE
  - 결합: JOIN
  - 집계: GROUP BY/HAVING
  - 정렬/페이징: ORDER BY/LIMIT/OFFSET
- SQL은 NULL 때문에 3값 논리(TRUE/FALSE/UNKNOWN)가 존재하므로 조건식 설계가 중요하다.

### 왜 필요한가
- ORM을 써도 결국 DB에는 SQL이 나가고, 성능/락/실행 계획 문제는 SQL로 해결해야 한다.
- WHERE/HAVING를 잘못 쓰면 쿼리가 느려지거나 결과가 달라질 수 있다.
- JOIN/집계는 실무에서 가장 자주 튜닝 포인트가 된다.

### 동작 원리
- 논리적 처리 순서(이해용):
  - FROM/JOIN → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT/OFFSET
- 물리적 실행(실제):
  - 옵티마이저가 통계/비용을 기반으로 인덱스 스캔, 조인 순서/방식 등을 선택한다.
- EXISTS vs IN:
  - 의미/NULL 처리/최적화 가능성이 달라질 수 있어 실행 계획으로 확인하는 습관이 중요하다.

### 장단점 / 한계
- 선언형 언어라 “왜 느린지”가 코드처럼 직관적이지 않다.
- 같은 SQL이라도 통계/데이터 분포/인덱스 상태에 따라 성능이 크게 바뀐다.

### 대안 / 선택 기준
- 복잡한 조회는 SQL로 풀되, 도메인 규칙이 깊으면 애플리케이션에서 후처리할지(단, 대량 데이터 후처리는 비효율) 선택한다.
- 페이징은 OFFSET 기반이 단순하지만, 데이터가 커지면 cursor/seek 기반을 고려한다.

### 실무/디버깅 포인트
- NULL은 “값이 없음”이지 “빈 문자열/0”이 아니다.
  - `= NULL`이 아니라 `IS NULL`을 쓰는 이유가 여기에 있다.
- ORDER BY는 큰 비용이 될 수 있어 “인덱스로 정렬을 만족할 수 있는지”를 먼저 본다.

### 프로젝트 연결
- N+1, 페이징, 조인 튜닝 같은 이슈는 보통 ORM+SQL 경계에서 발생한다.
  - 프로젝트 근거 연결은 `확인 필요`

## 정규화 / 모델링

### 개념
- 정규화는 중복을 줄이고 이상 현상을 막기 위해 테이블을 분해하는 설계 기법이다.
- 1NF: 원자값(반복/배열 컬럼을 피함)
- 2NF: 부분 함수 종속 제거(복합키의 일부에만 의존하는 컬럼 분리)
- 3NF: 이행적 함수 종속 제거(키가 아닌 컬럼끼리 의존하면 분리)
- 모델링은 “어떤 엔터티를 어떤 키로, 어떤 관계로” 저장할지 결정하는 작업이다.

### 왜 필요한가
- 중복은 결국 데이터 불일치로 이어진다(한 곳은 업데이트되고 다른 곳은 누락).
- FK/제약조건을 제대로 잡지 않으면 “없는 것을 참조하는 데이터”가 쌓인다.
- 관계(1:1, 1:N, N:M)를 정확히 모델링해야 쿼리/인덱스가 단순해진다.

### 동작 원리
- N:M은 중간 테이블로 풀어낸다.
  - 관계 자체가 의미/속성을 가지면(상태/정렬/생성일) 중간 테이블은 독립 엔터티가 된다.
- ERD 설계 순서(권장):
  1) 엔터티 후보와 경계
  2) 식별자(PK)와 유일성(UK)
  3) 관계(FK)와 삭제/수정 정책
  4) 조회/변경 패턴 기반 인덱스
- 타입 설계:
  - 문자열/숫자/시간대는 “표현”이 아니라 “의미/규칙”을 저장한다는 관점으로 선택한다.

### 장단점 / 한계
- 정규화를 과도하게 하면 조인이 늘어 조회가 복잡해질 수 있다.
- 반정규화는 조회 성능에 도움이 될 수 있지만, 동기화/무결성 비용이 늘어난다.

### 대안 / 선택 기준
- 기본은 정규화로 시작하고, 병목이 “측정으로 확인”되면 반정규화를 고려한다.
- 반정규화를 한다면 “갱신 정책(어디서 언제 동기화하는지)”을 명확히 문서화해야 한다.

### 실무/디버깅 포인트
- 모델링에서 가장 큰 실수는 “조회 편의로 중복을 늘리는 것”과 “키/제약을 약하게 두는 것”이다.
- 날짜/시간은 “UTC 저장 + 표시 변환” 같은 팀 규칙을 먼저 정하는 편이 안정적이다.

### 프로젝트 연결
- 실무에서 데이터 모델은 기능 변경보다 “조회/리포팅 요구”에서 흔히 흔들린다.
  - 프로젝트 근거 연결은 `확인 필요`

## 트랜잭션 / ACID

### 개념
- 트랜잭션은 여러 DB 작업을 하나의 작업 단위로 묶어 정합성을 지키는 메커니즘이다.
- ACID는 트랜잭션이 제공하는 성질을 요약한 모델이다.
- 중요한 점: “트랜잭션은 성능 기능이 아니라 정합성 기능”이며, 동시성/지연과 항상 trade-off가 있다.

### 왜 필요한가
- 잔액/재고/상태 전이 같은 문제는 “부분 성공”이 허용되지 않는다.
- 트랜잭션을 모르면 동시 수정, 데이터 유실, 중복 처리 같은 문제를 예방하기 어렵다.

### 동작 원리
- Commit/Rollback:
  - 커밋은 변경 확정, 롤백은 변경 취소다.
- 트랜잭션이 길어지면:
  - 락 점유가 길어져 대기/데드락 가능성이 커지고,
  - 커넥션 풀 점유가 길어져 애플리케이션까지 장애가 전파될 수 있다.

### 장단점 / 한계
- 트랜잭션을 강하게 쓰면 정합성은 좋아지지만 처리량/지연이 나빠질 수 있다.
- 격리 수준을 올리면 이상 현상은 줄지만 충돌/재시도가 늘 수 있다.

### 대안 / 선택 기준
- 모든 것을 한 트랜잭션으로 묶기보다 “진짜 원자성이 필요한 경계”를 좁힌다.
- 재시도 가능한 설계를 위해 멱등성 키/중복 방지(유니크 제약) 같은 방어선을 둔다.

### 실무/디버깅 포인트
- “느려졌다”면 트랜잭션 길이/락 대기/커넥션 풀 상태를 같이 본다.
- 애플리케이션에서는 “예외 발생 시 롤백 규칙”이 프레임워크 설정에 좌우될 수 있어 명확히 해두는 게 안전하다.

### 프로젝트 연결
- 트랜잭션 경계/재시도 설계는 실무에서 자주 면접 꼬리질문으로 이어진다.
  - 근거 문서 연결은 `확인 필요`

## Lock / 동시성 제어

### 개념
- 락은 동시 접근에서 정합성을 지키기 위한 제어 장치다.
- 크게 “읽기(shared)”와 “쓰기(exclusive)”로 설명할 수 있지만, 실제 DB는 더 다양한 락 모드를 갖는다.
- 동시성 제어 전략은 락뿐 아니라 MVCC(버전 관리)도 포함한다.

### 왜 필요한가
- 동시에 같은 데이터를 수정하면 lost update, 데드락, 대기 폭증 같은 문제가 생긴다.
- 락을 이해하면 “왜 타임아웃이 늘었는지”, “왜 일부 요청만 멈췄는지”를 설명할 수 있다.

### 동작 원리
- Row lock vs Table lock:
  - 행 락은 동시성을 높이고, 테이블 락은 단순하지만 영향 범위가 크다.
- 데드락:
  - 서로가 서로의 락을 기다리는 순환 대기이며, DB는 이를 감지해 일부 트랜잭션을 중단시킨다.
- 낙관적/비관적 락:
  - 낙관: 버전 비교로 충돌 감지 → 실패 시 재시도
  - 비관: 선점 락(예: SELECT FOR UPDATE)으로 충돌 자체를 방지

### 장단점 / 한계
- 락을 과하게 쓰면 처리량이 떨어지고, 데드락이 증가할 수 있다.
- 낙관적 락은 실패/재시도 UX가 필요하고, 비관적 락은 지연/대기 체인이 길어질 수 있다.

### 대안 / 선택 기준
- 충돌이 드물면 낙관적 락, 충돌이 잦고 반드시 성공시켜야 하면 비관적 락을 고려한다.
- 무엇보다 “트랜잭션을 짧게” 유지하는 것이 가장 강력한 동시성 전략 중 하나다.

### 실무/디버깅 포인트
- 데드락/락 대기는 “락 획득 순서 통일”과 “인덱스 조건으로 잠금 범위 축소”가 핵심이다.
- 운영에서는 블로킹 쿼리/대기 시간을 관측해야 한다(어느 쿼리가 누구를 막는지).

### 프로젝트 연결
- 동시성 제어는 DB만의 문제가 아니라, 애플리케이션 스레드/커넥션 풀과 결합되어 장애가 커진다.
  - 프로젝트 근거 연결은 `확인 필요`

## Index 기본

### 개념
- 인덱스는 조회를 빠르게 하기 위한 보조 자료구조다.
- B-Tree는 범위/정렬에 강하고, 복합 인덱스는 실제 쿼리 패턴에 맞춰 설계해야 한다.
- 인덱스는 “조회만 빠르게”가 아니라 “쓰기 비용을 증가”시키므로 균형이 필요하다.

### 왜 필요한가
- 대용량에서 Seq Scan은 비용이 커질 수 있어, 조건 조회/정렬/조인을 빠르게 하려면 인덱스가 필요하다.
- 인덱스를 이해하면 “왜 인덱스가 있는데도 안 타는지”, “왜 느려졌는지”를 실행 계획으로 설명할 수 있다.

### 동작 원리
- B-Tree:
  - 정렬된 구조라 범위 조건과 ORDER BY 최적화에 유리하다.
- 복합 인덱스:
  - 선행 컬럼부터 정렬되므로 leftmost prefix 규칙이 중요하다.
- LIKE:
  - 접두어 검색은 유리할 수 있지만, `%keyword%` 같은 포함 검색은 다른 인덱스/전문 검색이 필요할 수 있다.

### 장단점 / 한계
- 인덱스는 많을수록 좋지 않다(쓰기 비용/유지 비용/플랜 선택 혼란).
- 선택도가 낮은 컬럼은 인덱스 효과가 작을 수 있다.

### 대안 / 선택 기준
- 인덱스는 “자주 쓰는 WHERE/JOIN/ORDER BY” 패턴을 기준으로 최소로 설계한다.
- 조회가 특수하면(전문검색/JSONB/유사도) DB가 제공하는 특수 인덱스를 고려한다.

### 실무/디버깅 포인트
- 인덱스 설계는 항상 실행 계획(EXPLAIN)과 함께 검증한다.
- 인덱스 추가는 빠른 해결처럼 보이지만, 운영에서는 쓰기/디스크/락 영향까지 같이 본다.

### 프로젝트 연결
- 인덱스 튜닝 경험은 실제 쿼리/실행 계획을 근거로 정리하면 가장 강한 답변 재료가 된다.
  - 근거 문서 연결은 `확인 필요`

## Query 성능 / 실행 계획

### 개념
- 실행 계획은 DB가 쿼리를 처리하기 위해 선택한 접근 경로/연산 순서다.
- EXPLAIN은 계획을 보여주고, EXPLAIN ANALYZE는 실제 실행(실측)을 포함한다.
- 성능 튜닝은 “쿼리만”이 아니라 “통계/인덱스/데이터 분포/운영 상태”까지 포함한다.

### 왜 필요한가
- 느린 쿼리는 시스템 전체를 느리게 만든다(커넥션 풀 고갈, 락 대기 증가, 타임아웃 증가).
- N+1은 작은 쿼리 다수가 누적되어 DB와 네트워크에 큰 부담을 준다.
- 페이징은 데이터가 커질수록 전략 선택이 중요해진다(OFFSET vs cursor).

### 동작 원리
- Seq Scan vs Index Scan:
  - 조건 범위/선택도/캐시 상태에 따라 유리한 방식이 달라진다.
- 통계:
  - 옵티마이저는 통계로 비용을 추정하므로, 통계가 stale하면 플랜이 나빠질 수 있다.
- 페이징:
  - OFFSET은 뒤로 갈수록 버리는 작업이 누적될 수 있어 cursor/seek 방식이 도움이 된다.

### 장단점 / 한계
- 실행 계획은 “추정”이 포함될 수 있어, 실측과 차이가 나면 원인(통계/분포/캐시)을 찾아야 한다.
- 무작정 인덱스를 늘리면 쓰기/운영 비용이 커진다.

### 대안 / 선택 기준
- 튜닝 우선순위(권장):
  1) 느린 쿼리 식별(로그/모니터링)
  2) 실행 계획/실측 확인(EXPLAIN ANALYZE)
  3) 필터 조건/반환 행 수 줄이기
  4) 인덱스/조인 전략/쿼리 구조 개선
  5) 스키마/반정규화/캐시/아키텍처 확장

### 실무/디버깅 포인트
- “DB가 느린지”와 “애플리케이션이 느린지”를 분리해야 한다(커넥션 대기 vs 쿼리 실행 시간).
- 반환 행 수가 크면 “정말 그만큼 필요한가”부터 확인한다.

### 프로젝트 연결
- N+1/페이징/인덱스/실행 계획은 JPA 같은 ORM 사용 프로젝트에서 자주 문제로 나타난다.
  - 근거 문서 연결은 `확인 필요`

## PostgreSQL 특화

### 개념
- PostgreSQL은 MVCC 기반으로 동시성을 제공하고, JSONB/확장/다양한 인덱스 유형이 강점인 RDBMS다.
- PostgreSQL에서 “성능/디스크” 이슈는 MVCC와 Vacuum/Autovacuum 이해와 직접 연결된다.

### 왜 필요한가
- PostgreSQL은 기능이 풍부한 대신, 운영에서 Vacuum/통계/커넥션 관리가 부족하면 성능이 서서히 나빠질 수 있다.
- JSONB 같은 기능을 잘 쓰면 유연성을 얻지만, 핵심 쿼리/인덱스 설계를 놓치면 튜닝이 어려워질 수 있다.

### 동작 원리
- Schema:
  - DB 내 네임스페이스이며, 같은 DB에서 논리적 분리를 제공한다.
- SERIAL vs IDENTITY:
  - 둘 다 자동 증가 키를 구현하는 방법으로, IDENTITY는 표준에 가까운 표현이다(세부는 버전/옵션 확인 필요).
- TEXT vs VARCHAR:
  - 길이 제한 강제 필요 여부가 주된 선택 기준이다.
- TIMESTAMP vs TIMESTAMPTZ:
  - 시간대 정책(저장/표시 변환)을 먼저 정하고 타입을 선택한다.

### 장단점 / 한계
- MVCC는 읽기/쓰기 동시성에 유리하지만 dead tuple 관리가 필요하다.
- JSONB는 유연하지만, 핵심 조건/조인은 정규 컬럼으로 두는 것이 더 단순/효율적인 경우가 많다.

### 대안 / 선택 기준
- JSONB는 “유연한 속성/부가 필드”에 제한적으로 쓰고, 조회 핵심 축은 정규 컬럼 + B-Tree 인덱스로 두는 전략이 안전하다.
- 커넥션은 애플리케이션 풀 + 필요 시 PgBouncer 같은 풀러로 “적정 수”를 유지하는 것이 중요하다.

### 실무/디버깅 포인트
- 성능이 “서서히” 나빠지면:
  - dead tuple 증가(Autovacuum), 통계(stale), 인덱스 비대 등을 의심한다.
- 장기 트랜잭션/idle in transaction은 Vacuum을 막을 수 있어 특히 위험하다.

### 프로젝트 연결
- PostgreSQL을 실제 운영한 프로젝트 근거가 있으면 연결한다: `확인 필요`

## PostgreSQL 인덱스 / 성능

### 개념
- PostgreSQL은 B-Tree 외에도 GIN/GiST/부분 인덱스/표현식 인덱스 등 다양한 인덱스를 제공한다.
- 플랜 선택은 통계에 크게 의존하므로 ANALYZE/Autovacuum이 성능에 영향을 준다.

### 왜 필요한가
- JSONB/전문검색/유사도 같은 요구는 B-Tree만으로 커버가 어렵다.
- 잘못된 통계/플랜 선택은 “인덱스가 있는데도 Seq Scan” 같은 현상으로 나타날 수 있다.

### 동작 원리
- GIN:
  - 다중 키(토큰/요소)를 빠르게 찾는 데 유리(배열/JSONB/FTS 등).
- GiST:
  - 범위/공간/근접 같은 질의에 유용한 프레임워크형 인덱스.
- Partial index:
  - 조건을 만족하는 행만 인덱싱해 비용을 줄인다(soft delete 같은 패턴에 유용).
- Expression index:
  - `LOWER(email)` 같은 표현식 결과로 검색할 때 효과적이다.

### 장단점 / 한계
- 특수 인덱스는 강력하지만, 쓰기 비용/유지 비용과 “쿼리가 인덱스를 타는 형태로 작성”되어야 한다는 제약이 있다.

### 대안 / 선택 기준
- 먼저 쿼리 패턴과 연산자를 확정하고, 그에 맞는 인덱스 타입을 고른다.
- 가능하면 “정규 컬럼으로 풀기”가 운영/튜닝을 단순화하는 대안이 된다.

### 실무/디버깅 포인트
- 플랜이 이상하면:
  - 통계 갱신(ANALYZE), 데이터 분포 변화, 파라미터 스니핑/플랜 캐시 영향(환경 의존)을 의심한다.

### 프로젝트 연결
- 특수 인덱스 사용 사례를 실제 쿼리/EXPLAIN과 함께 남기면 근거가 강해진다.
  - 근거 문서 연결은 `확인 필요`

## PostgreSQL 트랜잭션 / MVCC

### 개념
- PostgreSQL의 MVCC는 “읽기 스냅샷”과 “행 버전”으로 읽기/쓰기 충돌을 줄인다.
- MVCC의 비용은 dead tuple이 쌓이는 것이고, 이를 Vacuum/Autovacuum으로 관리한다.

### 왜 필요한가
- 트랜잭션이 길어지면 락 문제뿐 아니라 Vacuum이 막혀 성능/디스크 문제로 번질 수 있다.
- 격리 수준은 이상 현상/성능/재시도 모델을 결정한다.

### 동작 원리
- dead tuple:
  - 업데이트/삭제가 즉시 덮어쓰지 않고 버전을 남기면서 생긴다.
- 장기 트랜잭션:
  - 오래된 스냅샷이 유지되면, Vacuum이 정리할 수 있는 범위가 줄어든다.
- idle in transaction:
  - 트랜잭션이 열린 채로 멈춰서 위험(락/커넥션 점유 + Vacuum 방해).

### 장단점 / 한계
- MVCC는 동시성에 유리하지만 운영 난이도가 올라간다.
- Serializable 같은 강한 격리는 재시도(충돌) 모델을 요구할 수 있다.

### 대안 / 선택 기준
- 트랜잭션은 짧게, 락 범위는 좁게, 커넥션은 빠르게 반환하는 구조가 기본이다.
- 충돌 가능성이 높으면 “비관적 락” 또는 “낙관적 락+재시도” 중 사용자 경험/업무 규칙에 맞는 전략을 택한다.

### 실무/디버깅 포인트
- 성능 저하가 누적되면 autovacuum 상태/테이블 비대/장기 트랜잭션을 우선 확인한다.

### 프로젝트 연결
- 실무에서는 DB만 보는 게 아니라 애플리케이션 풀/스레드와 같이 봐야 한다.
  - 근거 문서 연결은 `확인 필요`

## PostgreSQL 운영 / 백업

### 개념
- 운영에서 핵심은 RPO/RTO를 만족하는 백업/복구 전략과, 장애 시나리오(복제/페일오버)를 준비하는 것이다.
- WAL은 내구성/복제/복구(PITR)의 기반 로그다.
- 커넥션은 무작정 늘리면 성능이 좋아지지 않고 오히려 나빠질 수 있어 “적정 커넥션 수”가 중요하다.

### 왜 필요한가
- 백업이 없으면 DB 장애는 곧 데이터 유실로 이어질 수 있다.
- 복제(Read replica)는 읽기 확장과 고가용성에 도움이 되지만, replication lag은 일관성 문제를 만든다.
- 커넥션 폭증은 DB 리소스(메모리/스케줄링/락 관리)를 갉아먹어 전체 성능을 악화시킬 수 있다.

### 동작 원리
- 논리 백업(pg_dump):
  - 덤프/복원 형태로 이식/부분 복구에 유용하지만, 대용량에서는 시간/부하를 고려해야 한다.
- 물리 백업:
  - 파일 레벨/베이스 백업 + WAL로 복구(구체 방식은 운영 구성에 따라 다름).
- 복제:
  - Primary 변경 사항을 Replica로 전달하는 구조이며, 네트워크/부하에 따라 지연(lag)이 생길 수 있다.
- Connection pooler(PgBouncer):
  - 클라이언트 연결 수를 줄이고 DB 커넥션 재사용을 통해 오버헤드를 완화한다.

### 장단점 / 한계
- 운영 안정성은 기능이 아니라 “프로세스/정책/관측”이 핵심이라, 문서화/훈련이 없으면 실제 장애에서 무너질 수 있다.
- Read replica는 조회 확장을 돕지만, “쓰기 직후 읽기” 요구가 강하면 라우팅 정책이 필요하다.

### 대안 / 선택 기준
- 백업은 논리/물리를 목적에 맞게 조합한다(이식/부분 복구 vs 빠른 복구).
- 레플리카는 기능별로 “어떤 읽기를 레플리카로 보내도 되는지” 정책을 정한다.

### 실무/디버깅 포인트
- 커넥션이 많아져 느려지면:
  - 애플리케이션 풀 설정, DB max connections, PgBouncer 도입 여부를 같이 점검한다.
- lag 문제는:
  - 기능 요구(일관성)과 함께 트레이드오프를 합의하고, 모니터링/알람을 둔다.

### 프로젝트 연결
- 운영/배포 이슈 문서가 있으면 “백업/복구/장애 대응” 관점으로 연결해두면 강한 근거가 된다.
  - 근거 문서 연결은 `확인 필요`

## 면접 우선순위(추천)

### 개념
- 면접 대비는 “전체를 조금씩”보다 “핵심 축을 깊게”가 효과가 좋다.
- DB는 질문이 많지만, 대부분이 아래 6축으로 압축된다.

### 왜 필요한가
- 신입/주니어 면접에서는 “정의 암기”보다 **문제 상황에서 어떤 순서로 분석하는지**를 본다.
- 그래서 SQL/트랜잭션/인덱스/실행 계획을 먼저 잡아두면 나머지 질문도 대응이 쉬워진다.

### 동작 원리
```text
1) SQL 기본: JOIN, GROUP BY, WHERE/HAVING, NULL
2) 트랜잭션: ACID, Isolation Level, Lock
3) 인덱스: B-Tree, 복합 인덱스, 실행 계획
4) 성능 분석: EXPLAIN, Seq Scan, Index Scan, Pagination
5) PostgreSQL 특화: MVCC, Vacuum, JSONB, TIMESTAMPTZ
6) 운영: Connection Pool, Backup, Replication, WAL
```

### 장단점 / 한계
- 이 순서는 “면접 대비 우선순위”이고, 실무의 우선순위는 서비스 병목(쓰기 vs 읽기, OLTP vs OLAP)과 운영 요구에 따라 달라질 수 있다.

### 대안 / 선택 기준
- 특정 회사/직무가 DB를 더 깊게 본다면, PostgreSQL 운영(진단/복구/튜닝)을 더 앞에 두는 것도 가능하다.

### 실무/디버깅 포인트
- 성능 질문은 거의 항상 “실행 계획 → 인덱스/통계 → 쿼리 구조 → 스키마/아키텍처” 순서로 답하면 안전하다.
