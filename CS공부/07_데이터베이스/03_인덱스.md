# 인덱스

## 한 줄 정의
데이터를 빠르게 찾기 위해 별도로 유지하는 탐색용 구조다.

## 세부 정리
### 왜 필요한가
- 전체 테이블을 다 읽는 비용을 줄인다
- WHERE, JOIN, ORDER BY에서 효과가 크다

### 구조
- B-Tree 계열이 대표적이다
- 키 순서를 유지해서 범위 검색에 강하다
- 인덱스가 많을수록 쓰기 비용은 늘어난다

### 주의할 점
- 선택도가 낮으면 효과가 떨어진다
- 작은 테이블에는 오히려 손해일 수 있다
- 무작정 많이 만드는 것이 답이 아니다

## 예시
- 사용자 이메일 검색, 게시글 작성일 범위 검색

## 면접 포인트
- 인덱스가 왜 읽기는 빠르게 하고 쓰기는 느리게 하는지 설명한다

## 더 자세한 설명
- 인덱스는 책의 색인처럼 원하는 데이터를 더 빨리 찾게 해 주는 별도 구조다.
- 검색은 빨라지지만, 쓰기 작업은 인덱스 유지 비용 때문에 느려질 수 있다.
- 인덱스가 많다고 무조건 좋은 것이 아니라, 실제 질의 패턴에 맞춰야 한다.
- 인덱스는 "읽기 최적화"와 "쓰기 비용"의 교환이라고 이해하면 된다.

## 초보자용 한 줄 기억법
- 인덱스: 인덱스를 "일상 언어"로 다시 바꿔 말해 보는 노트다.
## 더 깊게 보기
- DB는 정확성, 속도, 동시성을 함께 맞추는 시스템이다.
- 인덱스는 조회를 빠르게 하지만, 삽입/수정/삭제 때 유지 비용을 만든다.
- 컬럼의 선택도가 높을수록 인덱스 효과가 좋아지는 경우가 많다.
- 실행 계획을 보지 않으면 인덱스가 실제로 타는지 확인할 수 없다.

## 초보자용 기억법
- 인덱스: "이게 왜 필요한지"까지 설명할 수 있으면 이해한 것이다.
