# RDB와 NoSQL

## 한 줄 정의
RDB와 NoSQL은 **데이터를 저장하고 조회하는 방식의 철학이 다른 두 큰 계열**이다.

- **RDB(관계형 데이터베이스)**: 표 형태의 테이블과 관계를 중심으로 관리한다
- **NoSQL**: 관계형 모델에만 묶이지 않고, 유연한 구조와 확장성을 중시한다

쉽게 말하면:
- RDB = 규칙이 잘 정해진 정리된 서랍장
- NoSQL = 목적에 따라 형태를 바꿀 수 있는 유연한 보관함

---

## 왜 배우는가
- DB는 무조건 하나만 쓰는 게 아니다.
- 서비스 성격에 따라 RDB가 맞을 수도, NoSQL이 맞을 수도 있다.
- 어떤 DB를 쓸지 모르면 성능, 확장성, 정합성에서 실수가 나온다.
- 면접에서도 RDB와 NoSQL의 차이는 기본 질문이다.
- 실무에서는 둘을 섞어 쓰는 경우도 많다.

---

## 초보자용 한 줄 기억법
- **RDB는 관계와 규칙이 강한 구조**
- **NoSQL은 유연성과 확장성이 강한 구조**

---

## RDB와 NoSQL을 한 문장으로 다시 말하면
RDB는 **관계형 모델과 SQL을 기반으로 정합성과 구조화된 데이터를 다루는 방식**이고, NoSQL은 **더 유연한 데이터 모델로 대규모 분산과 빠른 확장에 유리한 방식**이다.

---

## 전체 흐름

```text
요구사항 확인
   ↓
정합성이 더 중요한가?
   ├─ 예 → RDB 쪽이 유리할 수 있음
   └─ 아니오
        ↓
유연한 스키마/대규모 확장이 중요한가?
        ├─ 예 → NoSQL 검토
        └─ 아니오 → 둘 다 검토
```

---

## 핵심 개념
- **RDB**: 테이블, 행, 열, 관계, SQL, JOIN, ACID
- **NoSQL**: 문서형, 키-값형, 컬럼형, 그래프형, 수평 확장, 유연한 스키마
- **스키마**: 데이터 구조 규칙
- **정합성**: 데이터가 일관되고 틀어지지 않는 것
- **확장성**: 더 많은 데이터/트래픽을 처리할 수 있는 성질

---

## RDB란
RDB(Relational Database)는 **관계형 모델**을 따르는 데이터베이스다.

### 특징
- 데이터를 테이블로 저장한다
- 행(row)과 열(column)로 표현한다
- 테이블 간 관계를 FK로 연결한다
- SQL로 조회/수정한다
- 정규화와 ACID에 잘 어울린다

### 예시
- MySQL
- PostgreSQL
- Oracle
- SQL Server
- SQLite

---

## RDB 이해
RDB는 **정형화된 데이터와 명확한 관계**에 강하다.

### 예시
- 회원
- 주문
- 결제
- 재고
- 계좌

이런 데이터는 관계가 분명하고, 정확성이 중요해서 RDB가 잘 맞는다.

---

## NoSQL이란
NoSQL은 **Not Only SQL**의 의미로, 관계형 DB의 틀에만 묶이지 않는 데이터 저장 방식이다.

### 특징
- 스키마가 유연할 수 있다
- 문서/키-값/컬럼/그래프 등 다양한 형태가 있다
- 수평 확장에 유리한 경우가 많다
- 대규모 분산 저장에 강점을 보일 수 있다

### 예시
- MongoDB
- Redis
- Cassandra
- DynamoDB
- Neo4j

---

## NoSQL 이해
NoSQL은 **자주 바뀌는 구조, 아주 큰 규모, 빠른 확장**에 강한 경우가 많다.

### 예시
- 로그
- 캐시
- 세션 저장
- 이벤트 데이터
- 소셜 그래프
- 추천 시스템 일부

---

## RDB와 NoSQL 비교표

| 항목 | RDB | NoSQL |
|---|---|---|
| 데이터 모델 | 테이블 | 문서/키값/컬럼/그래프 등 |
| 스키마 | 엄격한 편 | 유연한 편 |
| 관계 | 강함 | 약하거나 모델마다 다름 |
| JOIN | 강점 | 보통 약하거나 제한적 |
| ACID | 강함 | 제품별로 다름 |
| 확장 방식 | 수직 확장 중심으로 시작하는 경우 많음 | 수평 확장에 유리한 경우 많음 |
| 쿼리 | SQL | 제품별 쿼리 언어/SDK |
| 적합한 데이터 | 정형 데이터 | 반정형/비정형 데이터 |

---

## RDB의 장점
- 정합성이 좋다
- 관계 표현이 쉽다
- JOIN과 쿼리가 강력하다
- SQL 표준을 통해 이해와 운영이 편하다
- 트랜잭션과 ACID에 강하다
- 데이터 구조가 안정적이다

---

## RDB의 단점
- 스키마 변경이 부담될 수 있다
- 아주 큰 규모의 수평 확장이 까다로울 수 있다
- JOIN이 많아지면 성능 이슈가 생길 수 있다
- 비정형 데이터에 유연하지 않을 수 있다

---

## NoSQL의 장점
- 스키마가 유연하다
- 구조가 자주 바뀌는 데이터에 맞을 수 있다
- 수평 확장에 유리한 경우가 많다
- 특정 사용 사례에서는 매우 빠를 수 있다
- 데이터 형태에 따라 적합한 모델을 고를 수 있다

---

## NoSQL의 단점
- 정합성 관리가 더 어려울 수 있다
- 제품마다 방식이 달라 배우는 부담이 있다
- JOIN 같은 관계형 기능이 약할 수 있다
- 설계가 잘못되면 중복이 많아질 수 있다
- 쿼리의 표준화가 약할 수 있다

---

## RDB와 NoSQL의 선택 기준

### RDB가 잘 맞는 경우
- 주문/결제/계좌처럼 정확성이 중요
- 관계가 명확함
- 트랜잭션이 중요함
- JOIN이 많이 필요함
- 데이터 구조가 비교적 안정적임

### NoSQL이 잘 맞는 경우
- 로그/캐시/세션처럼 빠른 접근이 중요
- 구조가 자주 바뀜
- 대규모 분산이 중요함
- 반정형/비정형 데이터가 많음
- 특정 사용 패턴에 최적화하고 싶음

---

## 선택 기준 비교표

| 상황 | 추천 방향 | 이유 |
|---|---|---|
| 결제/주문/정산 | RDB | 정합성과 트랜잭션 중요 |
| 게시글/댓글 | RDB 또는 NoSQL | 구조와 트래픽에 따라 다름 |
| 로그 저장 | NoSQL | 대량 저장/유연성 |
| 캐시/세션 | NoSQL(예: Redis) | 빠른 접근 |
| 소셜 관계 그래프 | 그래프형 NoSQL | 관계 탐색에 적합 |
| 자주 바뀌는 문서 구조 | 문서형 NoSQL | 유연한 스키마 |

---

## NoSQL의 종류

### 1) 문서형(Document)
- JSON 비슷한 문서로 저장
- 유연한 구조에 강하다
- 예: MongoDB

### 2) 키-값형(Key-Value)
- key로 value를 찾는 단순 구조
- 매우 빠른 조회에 적합
- 예: Redis, DynamoDB 일부 활용

### 3) 컬럼형(Column Family)
- 컬럼 중심으로 대량 데이터를 저장
- 분석/대규모 처리에 유리한 경우가 있음
- 예: Cassandra

### 4) 그래프형(Graph)
- 정점과 간선으로 관계를 저장
- 관계 탐색이 중요한 경우에 강함
- 예: Neo4j

---

## NoSQL 종류 비교표

| 종류 | 특징 | 잘 맞는 경우 |
|---|---|---|
| 문서형 | 문서 단위 저장, 유연한 구조 | 게시글, 사용자 프로필 |
| 키-값형 | key로 value 바로 찾음 | 캐시, 세션 |
| 컬럼형 | 대규모 분산/분석 | 로그, 이벤트 |
| 그래프형 | 관계 탐색 강함 | SNS, 추천, 네트워크 관계 |

## RDB 예시

### 회원과 주문
```text
회원 테이블
- 회원ID(PK)
- 이름
- 이메일

주문 테이블
- 주문ID(PK)
- 회원ID(FK)
- 주문일자
- 총액
```

### 왜 좋은가
- 회원과 주문의 관계가 명확하다
- 정합성이 중요하다
- 트랜잭션으로 안전하게 처리할 수 있다

---

## NoSQL 예시

### 사용자 프로필 문서형
```json
{
  "userId": 1,
  "name": "junoeasy",
  "profile": {
    "age": 20,
    "skills": ["CS", "Java", "DB"]
  }
}
```

### 왜 좋은가
- 구조가 유연하다
- 필요한 필드를 쉽게 확장할 수 있다
- 프로필처럼 가변적인 데이터에 적합할 수 있다

---

## 관계와 JOIN
RDB가 강한 이유 중 하나는 JOIN이다.

### JOIN이 좋은 이유
- 서로 다른 테이블의 관계를 자연스럽게 표현한다
- 중복을 줄일 수 있다
- 관계형 모델의 장점을 살릴 수 있다

### NoSQL에서는
- JOIN을 덜 쓰거나
- 데이터 중복을 허용해 조회를 빠르게 하거나
- 애플리케이션 레벨에서 조합하기도 한다

---

## 정규화와 NoSQL
RDB는 보통 정규화를 통해 중복을 줄인다.
NoSQL은 상황에 따라 중복을 허용해서 읽기 성능을 높이기도 한다.

### 쉽게 이해하기
- RDB: 정합성과 구조를 우선
- NoSQL: 조회 방식과 확장성을 우선

---

## CAP 이해
NoSQL 이야기를 할 때 CAP를 같이 언급하는 경우가 많다.

### CAP
- Consistency: 일관성
- Availability: 가용성
- Partition tolerance: 분산 환경에서 네트워크 분할을 견딤

### 쉽게 이해하기
- 분산 시스템에서는 세 가지를 모두 완벽히 만족하기 어렵다
- 어떤 쪽을 더 중시하는지에 따라 설계가 달라진다

---

## 일관성/확장성/유연성의 균형
이 세 가지는 동시에 최고가 되기 어렵다.

### RDB 쪽 강점
- 일관성
- 정합성
- 관계 표현

### NoSQL 쪽 강점
- 유연성
- 확장성
- 대규모 분산

---

## 실전 예시

### 예시 1: 쇼핑몰
- 회원, 주문, 결제, 재고는 RDB가 잘 맞는 경우가 많다
- 정합성이 매우 중요하다

### 예시 2: 캐시 서버
- Redis 같은 키-값형 NoSQL이 잘 맞는다
- 빠른 응답이 중요하다

### 예시 3: 로그 저장
- 스키마가 자주 바뀔 수 있다
- 대량 입력이 중요하다
- NoSQL이 잘 맞을 수 있다

### 예시 4: SNS 관계
- 친구/팔로우/추천 관계 탐색이 중요하다
- 그래프형 NoSQL이 유리할 수 있다

---

## 자주 하는 실수
- RDB가 무조건 좋은 줄 앎
  - 바로잡기: RDB는 정합성, 관계, 트랜잭션에 강하지만 스키마와 확장 요구에 따라 항상 최선은 아닐 수 있다.
- NoSQL이 무조건 빠른 줄 앎
  - 바로잡기: NoSQL은 모델과 구현에 따라 장점이 다르며, 빠르다는 말은 접근 패턴과 일관성 요구에 따라 달라진다.
- 정합성보다 유연성만 보고 선택함
  - 바로잡기: 데이터 선택은 유연성뿐 아니라 정합성, 조회 패턴, 트랜잭션 요구, 확장 방식을 함께 고려해야 한다.
- 스키마 차이만 보고 관계 차이를 무시함
  - 바로잡기: RDB와 NoSQL의 차이는 스키마 유무만이 아니라 관계 표현, 질의 방식, 트랜잭션 범위, 확장 전략까지 포함한다.
- 여러 NoSQL 종류를 한 덩어리로 생각함
  - 바로잡기: 문서형, 키-값, 컬럼 패밀리, 그래프 DB는 데이터 모델과 최적화 대상이 다르다.
- ACID와 CAP를 섞어 말함
  - 바로잡기: ACID는 트랜잭션 성질이고, CAP는 분산 시스템에서 일관성/가용성/분할 내성의 trade-off를 설명한다.
- 실무에서는 둘을 함께 쓰는 경우가 많다는 점을 놓침
  - 바로잡기: 실무에서는 핵심 거래 데이터는 RDB에 두고, 검색/캐시/로그/세션처럼 접근 패턴이 다른 데이터는 NoSQL을 함께 쓰기도 한다.

---

## 디버깅할 때 보는 순서
1. 데이터의 형태가 정형인지 반정형인지 본다
2. 정합성이 중요한지 본다
3. 관계(JOIN)가 많은지 본다
4. 스키마 변경이 잦은지 본다
5. 데이터량과 트래픽 규모를 본다
6. 수평 확장이 필요한지 본다
7. 읽기/쓰기 패턴을 본다
8. RDB와 NoSQL을 분리해서 써야 하는지 검토한다

---

## 비교해서 이해하기

### RDB vs NoSQL
- RDB: 관계형, 정합성, SQL, JOIN, ACID
- NoSQL: 유연성, 확장성, 다양한 모델

### SQL vs NoSQL
- SQL: 질의 언어
- NoSQL: 관계형 틀 밖의 저장 방식 전체를 부르는 말

### 문서형 vs 키-값형
- 문서형: 구조화된 문서 저장
- 키-값형: 빠른 조회와 단순 구조


- RDB는 JOIN과 ACID가 강점이다
- NoSQL은 문서형, 키-값형, 컬럼형, 그래프형 등으로 나뉜다
- 정답은 요구사항에 따라 달라진다

### 한 번에 말하는 답변 예시
> RDB는 테이블과 관계를 기반으로 정합성과 트랜잭션 처리에 강한 반면, NoSQL은 문서형, 키-값형, 컬럼형, 그래프형처럼 다양한 모델을 통해 유연성과 수평 확장에 강점이 있습니다. 그래서 결제나 주문처럼 정합성이 중요한 경우에는 RDB가 잘 맞고, 로그나 캐시, 유연한 문서 구조가 필요한 경우에는 NoSQL이 잘 맞을 수 있습니다.

### 꼬리 질문 예시
- 왜 RDB는 JOIN이 강점인가요?
- NoSQL은 왜 스키마가 유연하다고 하나요?
- ACID와 CAP는 어떤 차이가 있나요?
- MongoDB와 Redis는 같은 NoSQL인가요?
- 둘을 같이 쓰는 이유는 뭔가요?

---

## 한 번 더 요약
- RDB는 관계와 정합성 중심이다
- NoSQL은 유연성과 확장성 중심이다
- RDB는 테이블, JOIN, ACID가 강점이다
- NoSQL은 문서/키-값/컬럼/그래프 등 다양한 모델이 있다
- 선택은 데이터 성격과 서비스 요구사항에 따라 달라진다

## 초보자용 한 줄 기억법
RDB는 **"규칙이 강한 표 기반 DB"**, NoSQL은 **"유연하고 확장하기 쉬운 DB"**다.
