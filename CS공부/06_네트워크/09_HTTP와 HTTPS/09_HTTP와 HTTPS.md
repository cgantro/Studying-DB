# HTTP와 HTTPS

## 한 줄 정의
HTTP는 **웹에서 클라이언트와 서버가 요청과 응답을 주고받는 규칙**이고, HTTPS는 **그 HTTP 통신에 TLS 보안을 더한 버전**이다.

쉽게 말하면:
- HTTP = 웹에서 하는 기본 대화
- HTTPS = 그 대화에 암호화와 신원 확인을 붙인 것

---

## 왜 배우는가
- 브라우저가 페이지를 여는 과정의 핵심이 HTTP다.
- 로그인, 조회, 파일 업로드, API 호출은 대부분 HTTP 위에서 일어난다.
- HTTPS는 보안, 인증서, 암호화, 중간자 공격 방지와 연결된다.
- 실무에서 가장 자주 보는 프로토콜이므로 모르면 개발/운영/디버깅이 모두 불편해진다.
- 면접에서도 요청/응답 구조, 메서드, 상태 코드, 보안 차이는 거의 기본 질문이다.

---

## 초보자용 한 줄 기억법
- **HTTP는 웹의 대화 규칙**
- **HTTPS는 그 대화에 자물쇠를 건 것**

---

## 전체 흐름

```text
[브라우저]
   ↓ 1. DNS로 서버 IP 찾기
[서버 IP]
   ↓ 2. TCP 연결
[HTTP 요청 생성]
   ↓ 3. 요청 전송
[서버]
   ↓ 4. 요청 처리
[HTTP 응답 생성]
   ↓ 5. 응답 전송
[브라우저가 화면 렌더링]
```

HTTPS는 여기서 **TCP 위에 TLS 단계**가 추가된다고 생각하면 된다.

---

## HTTP가 하는 일
HTTP는 클라이언트와 서버가 웹 자원에 대해 소통하는 규칙이다.

### HTTP의 핵심 특징
- **요청/응답 구조**
  - 클라이언트가 요청하고 서버가 응답한다
- **무상태성(stateless)**
  - 각 요청은 기본적으로 독립적이다
- **메서드가 있다**
  - GET, POST, PUT, PATCH, DELETE 등
- **상태 코드가 있다**
  - 200, 301, 404, 500 등
- **헤더가 있다**
  - 캐시, 타입, 인증, 압축 같은 부가정보를 담는다

---

## 요청/응답 구조

### HTTP 요청(Request)
보통 다음 요소를 가진다.
- 요청 라인: 메서드, 경로, 버전
- 헤더: 부가 정보
- 바디: 전송할 데이터

### 예시
```http
GET /users/1 HTTP/1.1
Host: example.com
Accept: application/json
```

### HTTP 응답(Response)
- 상태 라인: 버전, 상태 코드, 메시지
- 헤더: 응답 부가 정보
- 바디: 실제 데이터

### 예시
```http
HTTP/1.1 200 OK
Content-Type: application/json

{"id":1,"name":"junoeasy"}
```

---

## 메서드란
HTTP 메서드는 **이 요청이 무슨 일을 하려는지**를 표현한다.

### 자주 쓰는 메서드
- **GET**: 조회
- **POST**: 생성/전송
- **PUT**: 전체 수정
- **PATCH**: 일부 수정
- **DELETE**: 삭제

### 메서드 이해
- GET = 꺼내보기
- POST = 새로 보내기
- PUT = 덮어쓰기
- PATCH = 부분만 고치기
- DELETE = 지우기

---

## 메서드 비교표

| 메서드 | 의미 | 보통 쓰는 상황 | 특징 |
|---|---|---|---|
| GET | 조회 | 페이지 열기, 목록 보기 | 바디 없이 쓰는 경우가 많음 |
| POST | 전송/생성 | 로그인, 회원가입, 업로드 | 서버에 상태 변화를 줄 수 있음 |
| PUT | 전체 수정 | 전체 객체 교체 | 같은 요청을 반복해도 결과가 같게 설계하는 경우가 많음 |
| PATCH | 부분 수정 | 이름만 변경 | 일부 필드만 바꿀 때 |
| DELETE | 삭제 | 자원 제거 | 위험하므로 신중히 사용 |

---

## 상태 코드란
상태 코드는 서버가 요청을 어떻게 처리했는지 알려 주는 숫자다.

### 대표 상태 코드
- **2xx**: 성공
- **3xx**: 리다이렉트
- **4xx**: 클라이언트 오류
- **5xx**: 서버 오류

### 자주 보는 코드
- **200 OK**: 성공
- **201 Created**: 생성 성공
- **301 Moved Permanently**: 영구 이동
- **302 Found**: 임시 이동
- **400 Bad Request**: 요청이 잘못됨
- **401 Unauthorized**: 인증 안 됨
- **403 Forbidden**: 권한 없음
- **404 Not Found**: 없음
- **500 Internal Server Error**: 서버 내부 오류
- **502 Bad Gateway**: 중간 서버가 이상함
- **503 Service Unavailable**: 서비스 이용 불가

---

## 상태 코드로 이해하기
- **200대**: 잘 됨
- **300대**: 다른 곳으로 가라고 함
- **400대**: 네가 보낸 요청이 문제
- **500대**: 서버가 문제

---

## 상태 코드 비교표

| 범위 | 의미 | 책임 주체 | 예시 |
|---|---|---|---|
| 2xx | 성공 | 서버 | 200, 201 |
| 3xx | 이동 | 서버/브라우저 | 301, 302 |
| 4xx | 요청 문제 | 클라이언트 | 400, 401, 403, 404 |
| 5xx | 서버 문제 | 서버 | 500, 502, 503 |

---

## HTTP의 무상태성(stateless)
HTTP는 기본적으로 **이전 요청을 기억하지 않는다**.

### 왜 그런가
- 매 요청을 독립적으로 처리하면 단순하다
- 서버 확장이 쉬워진다
- 요청 간 결합도가 낮아진다

### 하지만 불편한 점
- 로그인 상태를 기억하기 어렵다
- 장바구니, 세션, 사용자 인증처럼 상태가 필요할 수 있다

### 그래서 나오는 보완책
- 쿠키
- 세션
- 토큰(JWT 등)

---

## 쿠키와 세션
### 쿠키
- 브라우저에 저장되는 작은 데이터 조각
- 주로 상태 식별자나 설정값 저장에 쓴다

### 세션
- 서버 쪽에서 사용자 상태를 관리하는 방식
- 쿠키에 세션 ID를 넣고 서버가 그 ID로 상태를 찾기도 한다

### 아주 쉽게 말하면
- 쿠키 = 내 쪽 메모지
- 세션 = 서버 쪽 보관함

---

## 헤더란
헤더는 요청/응답에 붙는 **부가 정보**다.

### 요청 헤더 예시
- `Host`
- `Accept`
- `Authorization`
- `User-Agent`
- `Content-Type`
- `Cookie`

### 응답 헤더 예시
- `Content-Type`
- `Cache-Control`
- `Set-Cookie`
- `Location`
- `Content-Encoding`

### 헤더가 중요한 이유
- 캐시 정책
- 인증 정보
- 데이터 형식
- 압축
- 쿠키 전달

---

## 바디(body)란
바디는 요청이나 응답의 **실제 내용**이다.

### 예시
- 로그인 아이디/비밀번호
- JSON 데이터
- 파일 내용
- HTML 문서

### 주의
- GET은 보통 바디를 쓰지 않는 경우가 많지만, 프로토콜상 완전히 불가능하다는 뜻은 아니다
- 실무에서는 일반적인 패턴을 따라가는 것이 중요하다

---

## HTTP와 연결 재사용
### keep-alive
- 매 요청마다 연결을 새로 만들지 않고 재사용한다
- TCP 핸드셰이크 비용을 줄일 수 있다

### 왜 중요한가
- 웹은 요청이 많다
- 연결을 매번 새로 만들면 비효율적이다

### 간단한 그림
```text
연결 1개로
GET /index
GET /style.css
GET /script.js
```

---

## HTTPS란
HTTPS는 HTTP에 **TLS(Transport Layer Security)** 를 더한 것이다.

### HTTPS가 주는 것
- **기밀성**: 중간에서 내용을 못 봄
- **무결성**: 중간에서 내용을 바꾸기 어려움
- **인증**: 서버가 진짜인지 확인

### 쉽게 말하면
- HTTP: 편지 엽서
- HTTPS: 봉투 + 봉인 + 신원 확인

---

## TLS가 하는 일
TLS는 HTTPS의 핵심 보안 계층이다.

### TLS의 목적
- 데이터를 암호화한다
- 통신 중 변조를 막는다
- 서버가 진짜인지 인증한다

### 인증서와 공개키
- 서버는 인증서를 가진다
- 인증서는 공개키와 신원 정보를 담는다
- 브라우저는 인증서를 검증한 뒤 통신을 시작한다

---

## HTTPS 연결 흐름(개념)

```text
브라우저
   ↓
서버 인증서 검증
   ↓
암호화에 사용할 키 협상
   ↓
안전한 통신 채널 생성
   ↓
HTTP 요청/응답 전송
```

---

## HTTPS가 필요한 이유
### 1) 도청 방지
- 네트워크 중간에서 내용을 읽기 어렵게 한다

### 2) 위변조 방지
- 중간자가 내용을 바꾸기 어렵게 한다

### 3) 서버 신원 확인
- 접속한 서버가 진짜 그 서버인지 검증한다

### 4) 사용자 신뢰
- 브라우저 자물쇠 아이콘, 보안 경고 방지

---

## HTTP vs HTTPS 비교표

| 항목 | HTTP | HTTPS |
|---|---|---|
| 보안 | 암호화 없음 | TLS로 암호화 |
| 기밀성 | 낮음 | 높음 |
| 무결성 | 낮음 | 높음 |
| 서버 인증 | 없음 | 있음 |
| 기본 포트 | 80 | 443 |
| 브라우저 표시 | 자물쇠 없음 | 자물쇠 표시 |

---

## HTTPS의 오해
- HTTPS는 느리다 → 예전보다 훨씬 최적화되었고, 보안 비용보다 얻는 이점이 크다
- HTTPS는 비싸다 → 인증서와 운영 비용은 있지만, 대부분 필수에 가깝다
- HTTPS는 그냥 포트 443만 쓰면 된다 → 단순 포트가 아니라 TLS 검증이 핵심이다

---

## HTTP/HTTPS와 포트
- HTTP 기본 포트: **80**
- HTTPS 기본 포트: **443**

### 기억법
- 80 = HTTP
- 443 = HTTPS

---

## 요청 흐름 예시
### 예시 1: 로그인
```text
브라우저 → POST /login → 서버
서버 → 200 OK + 쿠키/토큰 → 브라우저
```

### 예시 2: 게시글 조회
```text
브라우저 → GET /posts/1 → 서버
서버 → 200 OK + 게시글 JSON → 브라우저
```

### 예시 3: 주소 이동
```text
브라우저 → GET /old-page → 서버
서버 → 301 Moved Permanently + Location: /new-page
브라우저 → 새 주소로 재요청
```

---

## 실전에서 자주 보는 상황
### 1) 로그인 후에도 로그인이 유지된다
- 쿠키/세션/토큰 덕분이다

### 2) 새로고침해도 페이지가 보인다
- HTTP가 HTML/CSS/JS를 받아 렌더링한다

### 3) 특정 API가 404를 돌려준다
- 경로가 없거나 잘못된 주소일 수 있다

### 4) 프록시 뒤에서 502가 뜬다
- 게이트웨이/백엔드 연결 문제가 있을 수 있다

---

## 자주 하는 실수
- HTTP와 HTTPS를 포트 번호만 다른 것이라고 생각함
  - 바로잡기: HTTPS는 HTTP에 TLS 암호화와 서버 인증을 더한 것으로 포트 번호 차이만이 핵심이 아니다.
- GET/POST 차이를 정확히 모름
  - 바로잡기: GET은 주로 조회와 URL 기반 파라미터 전달, POST는 서버 상태 변경이나 본문 전송에 주로 사용된다.
- 상태 코드 의미를 헷갈림
  - 바로잡기: 상태 코드는 응답 결과의 범주를 나타내며 2xx 성공, 3xx 리다이렉션, 4xx 클라이언트 오류, 5xx 서버 오류로 나뉜다.
- 쿠키와 세션을 같은 것처럼 생각함
  - 바로잡기: 쿠키는 클라이언트에 저장되는 작은 데이터이고, 세션은 서버가 사용자 상태를 관리하는 방식이다.
- 인증과 인가를 구분하지 못함
  - 바로잡기: 인증은 누구인지 확인하는 것이고, 인가는 확인된 사용자가 무엇을 할 수 있는지 판단하는 것이다.
- 4xx와 5xx 책임 주체를 혼동함
  - 바로잡기: 4xx는 클라이언트 요청 형식, 인증, 권한, 경로 문제에 가깝고, 5xx는 서버나 게이트웨이가 요청 처리에 실패한 경우다.
- 헤더와 바디를 뒤섞어 생각함
  - 바로잡기: 헤더는 메타데이터와 제어 정보, 바디는 실제 전송하려는 본문 데이터다.

---

## 인증과 인가
이 둘도 자주 같이 나온다.

### 인증(Authentication)
- 너가 누구인지 확인하는 것
- 로그인

### 인가(Authorization)
- 너가 무엇을 할 수 있는지 확인하는 것
- 관리자 페이지 접근 권한

### 기억법
- 인증 = 신원 확인
- 인가 = 권한 확인

---

## 캐시와 HTTP
HTTP는 캐시와 아주 많이 연결된다.

### 캐시가 필요한 이유
- 같은 리소스를 매번 다시 받아오면 낭비다
- 브라우저/프록시/CDN이 캐시를 활용한다

### 자주 보는 캐시 관련 헤더
- `Cache-Control`
- `Expires`
- `ETag`
- `If-None-Match`

### 의미
- 서버가 이 응답을 얼마나 오래 저장해도 되는지 알려 준다
- 바뀌지 않았으면 다시 다운로드하지 않게 한다

---

## 리다이렉션
HTTP는 한 주소에서 다른 주소로 보내는 기능도 제공한다.

### 대표 상태 코드
- 301: 영구 이동
- 302: 임시 이동
- 307/308: 메서드 보존 이동

### 사용 예
- `/old` → `/new`
- HTTP → HTTPS 이동
- 도메인 변경

---

## API 관점에서 HTTP
요즘 웹은 그냥 페이지뿐 아니라 **API 통신**이 많다.

### JSON 예시
```http
POST /api/users HTTP/1.1
Content-Type: application/json

{"name":"junoeasy","age":20}
```

### 응답 예시
```http
HTTP/1.1 201 Created
Content-Type: application/json

{"id":123,"name":"junoeasy","age":20}
```

---

## REST 이해과 함께 보기
HTTP는 REST 스타일 API와 자주 함께 쓰인다.

### REST 느낌
- 자원을 URL로 표현한다
- 메서드로 동작을 표현한다
- 상태 코드를 적절히 쓴다

---

## HTTP와 웹 페이지 로딩
브라우저가 페이지를 열 때 HTTP는 여러 번 쓰인다.

```text
HTML 요청
 ↓
CSS 요청
 ↓
JS 요청
 ↓
이미지 요청
 ↓
API 추가 요청
```

즉, 한 페이지를 여는 데 HTTP 요청이 여러 번 오간다.

---

## 디버깅할 때 보는 순서
1. 요청 URL이 맞는가
2. 메서드가 맞는가
3. 상태 코드가 무엇인가
4. 헤더가 올바른가
5. 바디가 기대한 형식인가
6. 인증 정보가 있는가
7. HTTPS 인증서/리다이렉션 문제가 없는가
8. 캐시가 문제를 만들고 있지 않은가

## 비교해서 이해하기

### HTTP vs HTTPS
- HTTP: 기본 통신 규칙
- HTTPS: 보안이 추가된 HTTP

### GET vs POST
- GET: 조회
- POST: 전송/생성

### 쿠키 vs 세션
- 쿠키: 브라우저 쪽
- 세션: 서버 쪽

### 인증 vs 인가
- 인증: 누구냐
- 인가: 무엇을 할 수 있냐

## ASCII 다이어그램

### 요청/응답
```text
클라이언트 ---- 요청 ----> 서버
클라이언트 <--- 응답 ---- 서버
```

### HTTPS 추가
```text
클라이언트 --TLS/암호화--> 서버
그 위에서 HTTP 요청/응답
```

### 웹 접속 순서
```text
DNS → TCP → TLS → HTTP → 렌더링
```

## 면접 포인트
면접에서는 "HTTP와 HTTPS 차이"를 단순 암기보다 흐름으로 말하면 좋다.

### 꼭 말할 수 있어야 하는 것
- HTTP는 웹의 요청/응답 규칙이다
- HTTPS는 HTTP에 TLS를 더한 것이다
- HTTPS는 암호화, 무결성, 인증을 제공한다
- 메서드와 상태 코드의 의미를 알아야 한다
- 쿠키/세션/토큰은 상태 관리를 위한 수단이다

### 한 번에 말하는 답변 예시
> HTTP는 클라이언트와 서버가 요청과 응답을 주고받는 웹 통신 규칙이고, HTTPS는 여기에 TLS를 적용해 암호화와 서버 인증을 추가한 것입니다. HTTP는 메서드와 상태 코드로 동작을 표현하고, 쿠키나 세션을 통해 무상태성을 보완합니다.

### 꼬리 질문 예시
- GET과 POST 차이는 뭔가요?
- 왜 HTTPS가 필요한가요?
- 401과 403 차이는 뭔가요?
- 쿠키와 세션 차이는 뭔가요?
- 리다이렉션 상태 코드 차이는 뭔가요?

## 한 번 더 요약
- HTTP는 웹의 기본 대화 규칙이다
- HTTPS는 그 규칙에 보안(TLS)을 더한 것이다
- 메서드, 상태 코드, 헤더, 바디가 HTTP의 핵심이다
- 쿠키와 세션은 HTTP의 무상태성을 보완한다
- HTTPS는 기밀성, 무결성, 인증을 제공한다

## 초보자용 한 줄 기억법
HTTP는 **"웹이 서로 말하는 방식"**, HTTPS는 **"그 말을 안전하게 보내는 방식"**이다.
