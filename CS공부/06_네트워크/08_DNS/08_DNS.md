# DNS

## 한 줄 정의
DNS(Domain Name System)는 **사람이 읽기 쉬운 도메인 이름을 컴퓨터가 통신할 수 있는 IP 주소로 바꿔 주는 이름 해석 시스템**이다.

예를 들면 `www.google.com`을 입력했을 때, 브라우저가 실제로 접속할 서버의 IP를 찾는 과정이 DNS다.

---

## 왜 배우는가
- 사람은 이름이 편하지만, 네트워크 장비는 숫자(IP)가 편하다.
- DNS가 없으면 웹사이트 주소를 전부 숫자로 외워야 한다.
- 웹이 느릴 때, 서버가 안 열릴 때, 접속이 엉킬 때 DNS를 의심해야 하는 경우가 많다.
- 면접에서도 DNS 조회 흐름, 캐시, TTL, 레코드 타입은 자주 나온다.

---

## 초보자용 한 줄 기억법
- **도메인 = 사람용 이름**
- **IP = 컴퓨터용 주소**
- **DNS = 이름을 주소로 바꿔 주는 전화번호부**

---

## DNS를 한 문장으로 다시 말하면
DNS는 **"도메인 이름을 보고, 그 이름에 연결된 서버의 IP를 찾는 시스템"**이다.

---

## 전체 흐름

```text
[브라우저에 주소 입력]
        ↓
[로컬 캐시 확인]
        ↓ 없으면
[OS/브라우저/라우터 캐시 확인]
        ↓ 없으면
[재귀 리졸버(Recursive Resolver)]
        ↓
[루트 DNS 서버]
        ↓
[최상위 도메인(TLD) 서버]
        ↓
[권한 서버(Authoritative DNS)]
        ↓
[IP 주소 응답]
        ↓
[브라우저가 해당 IP로 접속]
```

---

## DNS가 실제로 하는 일
DNS는 단순한 변환기가 아니다. 실제로는 아래 역할을 함께 한다.

- 도메인 이름을 IP로 해석한다
- 여러 서버 중 어느 쪽으로 갈지 안내한다
- 캐시를 이용해 빠르게 응답한다
- 장애나 서버 이동 시 새 주소를 반영한다
- 서비스 분산, CDN, 로드밸런싱에 도움을 준다

---

## DNS가 필요한 이유
### 1) 사람은 이름을 기억하기 쉽다
- `google.com`은 기억하기 쉽다
- `142.250.206.14` 같은 숫자는 외우기 어렵다

### 2) 서버 IP는 바뀔 수 있다
- 서버를 옮기거나 확장하면 IP가 달라질 수 있다
- 도메인을 쓰면 주소 변경을 숨길 수 있다

### 3) 여러 서버를 유연하게 연결할 수 있다
- 같은 도메인에 여러 IP를 연결할 수 있다
- 지역별, 부하별로 다른 서버를 안내할 수 있다

---

## DNS의 구조: 계층형 분산 시스템
DNS는 한 서버가 다 아는 구조가 아니다. **계층적으로 나뉜 분산 구조**다.

### 계층 구조
```text
루트 DNS 서버
  ↓
TLD DNS 서버 (.com, .kr, .org)
  ↓
권한 DNS 서버(예: example.com의 공식 정보 보유)
```

### 쉽게 말하면
- 루트 서버: "어느 최상위 도메인으로 가야 하지?"
- TLD 서버: "example.com 정보는 어디에 있지?"
- 권한 서버: "example.com의 실제 IP는 뭐지?"

---

## 재귀 질의 vs 반복 질의
DNS는 조회 방식이 두 가지로 자주 설명된다.

### 재귀 질의(Recursive Query)
- 클라이언트가 리졸버에게 "네가 끝까지 찾아와 줘"라고 맡기는 방식
- 보통 우리가 실제로 쓰는 조회 방식에 가깝다

### 반복 질의(Iterative Query)
- 질의받은 서버가 "나는 여기까지 알고 있고, 다음은 저기로 가봐"라고 안내만 하는 방식
- 루트 → TLD → 권한 서버를 따라가며 단계적으로 찾는다

### 한 줄 비교
- 재귀: 대신 찾아줘
- 반복: 다음 길 알려줘

---

## DNS 조회 흐름 예시
예를 들어 `www.example.com`을 입력했다고 하자.

1. 브라우저가 로컬 캐시를 먼저 본다
2. 없으면 운영체제 캐시를 본다
3. 그래도 없으면 재귀 리졸버에 문의한다
4. 리졸버가 루트 서버에 물어본다
5. 루트 서버가 `.com` TLD 서버를 알려준다
6. TLD 서버가 `example.com` 권한 서버를 알려준다
7. 권한 서버가 실제 IP를 알려준다
8. 리졸버가 그 결과를 캐시에 저장하고 브라우저에 준다
9. 브라우저는 그 IP로 TCP 연결을 시도한다

---

## DNS 조회 ASCII 그림

```text
브라우저
   ↓
로컬 캐시
   ↓ 없으면
재귀 리졸버
   ↓
루트 서버
   ↓
TLD 서버 (.com)
   ↓
권한 서버(example.com)
   ↓
IP 응답
```

## 권한 서버(authoritative DNS)란
권한 서버는 **그 도메인의 공식 답을 가지고 있는 서버**다.

### 쉽게 이해하기
- 루트/TLD는 길 안내에 가깝다
- 권한 서버는 최종 주소를 가진 원본 관리자다

### 예시
`example.com`의 실제 A 레코드가 어디 있는지, 권한 서버가 최종 답을 준다.

---

## DNS 레코드 타입
DNS는 단순히 IP만 저장하는 게 아니다. 여러 종류의 레코드가 있다.

### 자주 보는 레코드

| 레코드   | 의미                 | 예시                                |     |
| ----- | ------------------ | --------------------------------- | --- |
| A     | 도메인 → IPv4 주소      | `example.com -> 93.184.216.34`    |     |
| AAAA  | 도메인 → IPv6 주소      | `example.com -> 2606:2800:...`    |     |
| CNAME | 별칭을 다른 도메인으로 연결    | `www.example.com -> example.com`  |     |
| MX    | 메일 서버 주소           | `example.com -> mail.example.com` |     |
| NS    | 이 도메인을 관리하는 DNS 서버 | 권한 서버 지정                          |     |
| TXT   | 텍스트 정보             | SPF, DKIM, 소유권 확인                 |     |
| SRV   | 서비스 위치 정보          | 특정 서비스의 호스트/포트                    |     |
| PTR   | IP → 도메인 역방향 조회    | 역방향 DNS                           |     |

---

## 레코드 타입을 쉽게 이해하기
### A 레코드
- 제일 기본
- 도메인 이름을 IPv4 주소로 바꾼다

### AAAA 레코드
- IPv6 버전
- IPv6 네트워크에서 사용

### CNAME
- 별명(alias)이다
- `www.example.com`을 `example.com`으로 연결할 때 자주 쓴다

### MX
- 메일 받을 서버를 지정한다
- 이메일 시스템에서 중요하다

### TXT
- 텍스트 메모장 같은 느낌
- 인증 정보나 보안 정책을 저장한다

---

## 레코드 비교표

| 레코드 | 방향 | 주 용도 | 기억 포인트 |
|---|---|---|---|
| A | 이름 → IPv4 | 일반 웹 접속 | 가장 기본 |
| AAAA | 이름 → IPv6 | IPv6 접속 | IPv6 버전 |
| CNAME | 이름 → 이름 | 별칭 처리 | 다른 이름으로 연결 |
| MX | 도메인 → 메일 서버 | 이메일 | 메일 수신 |
| NS | 도메인 → DNS 서버 | 권한 위임 | 누구가 관리하나 |
| TXT | 텍스트 | 검증/정책 | 메모장 |

---

## TTL과 캐시
DNS가 빠른 이유 중 하나는 **캐시**다.

### TTL(Time To Live)
- 이 레코드를 얼마나 오래 믿고 저장할지 정한 시간
- TTL이 남아 있으면 다시 묻지 않고 캐시를 쓴다

### 왜 중요한가
- TTL이 길면 빠르다
- TTL이 짧으면 변경이 빨리 반영된다
- 둘 사이에서 균형을 잡아야 한다

### 아주 쉬운 예시
- TTL이 300초면 5분 동안은 같은 답을 재사용할 수 있다
- 서버 IP를 바꿨는데 TTL이 길면, 일부 사용자는 오래된 IP를 볼 수 있다

### 캐시가 있는 곳
- 브라우저
- 운영체제
- 로컬 DNS 캐시
- 재귀 리졸버
- 중간 네트워크 장비

---

## 캐시가 DNS를 빠르게 만드는 방식

```text
[브라우저 입력]
   ↓
[브라우저 캐시 있나?]
   ├─ 있으면 바로 사용
   └─ 없으면 OS 캐시 확인
                ↓
         [리졸버 캐시 확인]
                ↓
          [권한 서버 질의]
```

---

## DNS가 느려지면 생기는 현상
- 웹사이트 접속 시작이 늦다
- 첫 화면이 뜨기 전까지 오래 멈춘 느낌이 난다
- 같은 사이트인데 어떤 때는 빠르고 어떤 때는 느리다
- 회사/학교/공유기 환경에 따라 체감 차이가 난다

---

## DNS와 웹 접속 흐름
DNS는 웹 요청의 첫 단계에 가깝다.

```text
도메인 입력
   ↓
DNS로 IP 조회
   ↓
TCP 연결
   ↓
HTTP 요청
   ↓
HTTP 응답
```

DNS가 느리면 뒤 단계가 시작되기도 전에 기다리게 된다.

---

## DNS와 로드밸런싱
DNS는 한 도메인에 여러 IP를 연결할 수 있다.

### 예시
- `example.com -> 서버 A`
- `example.com -> 서버 B`
- `example.com -> 서버 C`

이렇게 하면 사용자를 여러 서버로 분산시킬 수 있다.

### 주의점
- DNS 기반 분산은 즉시 반영되지 않을 수 있다
- 캐시 때문에 실제로는 균등 분배가 안 보일 수 있다

---

## DNS와 CDN
CDN은 사용자와 가까운 위치의 서버로 연결하기 위해 DNS를 활용하는 경우가 많다.

### 직관적으로
- 사용자에게 가장 가까운 엣지 서버를 안내한다
- 지연 시간을 줄인다
- 대역폭을 분산한다

---

## DNS와 장애 상황
DNS 문제는 생각보다 흔하다.

### 예시 장애
- 도메인을 입력했는데 사이트가 안 열림
- 특정 네트워크에서만 접속이 안 됨
- 변경한 IP가 아직 반영되지 않음
- CNAME 체인이 잘못됨
- MX 레코드가 잘못돼 메일이 안 감

### 장애일 때 먼저 볼 것
1. 도메인 자체가 맞는가
2. DNS 레코드가 맞는가
3. TTL 때문에 이전 값이 남아 있지 않은가
4. 로컬 캐시가 오래된 값을 들고 있지 않은가
5. 권한 서버가 정상인지

---

## DNS 문제 디버깅 순서

### 1) 도메인 확인
- 오타가 없는지 본다
- 하위 도메인을 잘 썼는지 본다

### 2) 캐시 확인
- 브라우저 캐시
- OS 캐시
- 리졸버 캐시

### 3) 레코드 확인
- A/AAAA/CNAME/MX/NS가 맞는지 본다

### 4) TTL 확인
- 변경 직후라면 옛 값이 남아 있을 수 있다

### 5) 네트워크 확인
- 방화벽이나 DNS 서버 차단이 없는지 본다

### 6) 권한 서버 확인
- 공식 정보가 올바른지 확인한다

---

## 실전 예시 1: 브라우저에서 사이트가 안 열림
### 가능한 원인
- DNS 조회 실패
- 잘못된 CNAME
- 캐시가 오래된 IP를 가리킴
- 인터넷은 되지만 DNS만 막힘

### 확인 방법
- 다른 도메인은 열리는지 확인
- `nslookup`, `dig`로 조회
- 캐시 비우기
- 다른 DNS 서버로 바꿔 테스트

---

## 실전 예시 2: 서버 IP를 바꿨는데 일부 사용자만 접속 실패
### 가능한 원인
- TTL이 길어서 기존 IP가 캐시에 남음
- 일부 리졸버가 옛 정보를 유지 중

### 해결 기준
- TTL을 낮춰두고 변경
- 변경 전후를 고려해 롤아웃
- 캐시 만료 시간을 감안

---

## 실전 예시 3: 메일이 안 감
### 가능한 원인
- MX 레코드 오류
- SPF/DKIM/TXT 설정 문제
- 메일 서버 주소 오타

### 포인트
DNS는 웹뿐 아니라 이메일에도 핵심이다.

---

## 자주 쓰는 도구
- `nslookup`
- `dig`
- `host`
- 브라우저 개발자 도구
- 시스템 DNS 설정 확인

### `dig`가 주는 느낌
```text
dig example.com
```
- 어떤 DNS 서버가 어떤 답을 주는지 자세히 본다

## 자주 하는 실수
- DNS와 IP를 같은 개념으로 생각함
  - 바로잡기: DNS는 이름을 IP 등 레코드로 해석하는 시스템이고, IP는 네트워크에서 호스트를 식별하는 주소다.
- CNAME과 A 레코드를 헷갈림
  - 바로잡기: A 레코드는 이름을 IPv4 주소로 매핑하고, CNAME은 다른 도메인 이름의 별칭을 만든다.
- TTL이 왜 필요한지 모름
  - 바로잡기: TTL은 DNS 응답을 캐시에 얼마나 오래 보관할지 정해 부하와 변경 반영 속도의 균형을 잡는다.
- 캐시를 무시하고 서버만 의심함
  - 바로잡기: 테스트에서는 이 요소를 그대로 의존하면 결과가 흔들리므로 fixture, mock, 고정 seed, 테스트 전용 설정으로 통제해야 한다.
- DNS 문제인데 네트워크 문제로 오해함
  - 바로잡기: 연결 실패는 서버 코드뿐 아니라 포트 개방, 방화벽, 라우팅, DNS, 바인딩 주소 문제도 함께 확인해야 한다.
- DNS는 웹에서만 쓰인다고 생각함
  - 바로잡기: MX는 메일 서버, TXT는 검증/정책 문자열, NS는 권한 있는 네임서버 정보를 담는다.
- MX, TXT, NS 같은 레코드도 중요하다는 걸 놓침
  - 바로잡기: MX는 메일 서버, TXT는 검증/정책 문자열, NS는 권한 있는 네임서버 정보를 담는다.

---

## 비교해서 이해하기

### DNS vs IP
- DNS: 이름
- IP: 숫자 주소

### 재귀 vs 반복 질의
- 재귀: 끝까지 찾아줘
- 반복: 다음 줄 알려줘

### A vs CNAME
- A: 이름을 IP로 바로 연결
- CNAME: 이름을 다른 이름으로 연결

### TTL 길다 vs 짧다
- 길다: 캐시 효율 좋음, 변경 반영 느림
- 짧다: 변경 반영 빠름, 조회 빈도 증가

## ASCII 그림 모음

### 이름 해석 흐름
```text
도메인 이름
   ↓
DNS 조회
   ↓
IP 주소
   ↓
TCP 연결
   ↓
서비스 접속
```

### 계층 구조
```text
Root
 └─ TLD(.com)
     └─ Authoritative(example.com)
         └─ A/AAAA/CNAME...
```

### 캐시 우선 순서
```text
브라우저 캐시
   ↓
OS 캐시
   ↓
로컬 리졸버 캐시
   ↓
외부 DNS 조회
```

---

## 초보자 체크리스트
- [ ] DNS가 IP를 찾는 시스템이라는 걸 설명할 수 있다
- [ ] 루트/TLD/권한 서버의 역할을 안다
- [ ] 재귀 질의와 반복 질의를 구분할 수 있다
- [ ] A, AAAA, CNAME, MX, NS, TXT를 들어본 적 있다
- [ ] TTL이 왜 필요한지 설명할 수 있다
- [ ] DNS 캐시가 어디에 있을 수 있는지 안다
- [ ] DNS가 느리면 웹 접속도 느려질 수 있다는 점을 안다

## 면접 포인트
면접에서는 "DNS가 뭐예요?"보다 조회 흐름을 말할 수 있어야 한다.

### 꼭 말할 수 있어야 하는 것
- DNS는 도메인을 IP로 바꾸는 시스템이다
- 계층형 분산 구조를 가진다
- 캐시와 TTL이 성능에 중요하다
- 재귀 리졸버가 루트/TLD/권한 서버를 거쳐 찾는다
- A/AAAA/CNAME/MX 정도는 최소 알아야 한다

### 한 번에 말하는 답변 예시
> DNS는 사람이 읽는 도메인 이름을 IP 주소로 바꿔 주는 이름 해석 시스템입니다. 사용자가 도메인을 입력하면 브라우저나 OS 캐시를 먼저 보고, 없으면 재귀 리졸버가 루트 DNS, TLD DNS, 권한 DNS를 순서대로 거쳐 최종 IP를 찾습니다. 이때 캐시와 TTL 때문에 빠르게 동작하지만, 변경 반영이 늦어질 수도 있습니다.

### 꼬리 질문 예시
- 왜 캐시가 필요한가요?
- TTL이 길면 어떤 문제가 있나요?
- CNAME과 A 레코드 차이는 뭔가요?
- 메일이 안 갈 때 DNS에서 무엇을 보나요?
- 재귀 질의와 반복 질의는 어떻게 다른가요?

---

## 한 번 더 요약
- DNS는 이름을 IP로 바꿔 주는 시스템이다
- 계층형이고 분산되어 있다
- 캐시와 TTL 덕분에 빠르다
- A/AAAA/CNAME/MX/NS/TXT 같은 레코드가 있다
- 웹뿐 아니라 메일, CDN, 로드밸런싱에도 중요하다

## 초보자용 한 줄 기억법
DNS는 **"도메인 이름을 IP 주소로 바꿔 주는 전화번호부"**다.
