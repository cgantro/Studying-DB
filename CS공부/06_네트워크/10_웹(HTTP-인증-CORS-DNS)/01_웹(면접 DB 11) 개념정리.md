## HTTP 기본

### 개념
- HTTP는 웹에서 클라이언트와 서버가 요청/응답 메시지를 교환하는 애플리케이션 계층 프로토콜이다.
- 메시지 구조:
  - Request: 시작줄(Method/Path/Version) + Headers + Body(선택)
  - Response: 시작줄(Version/Status) + Headers + Body(선택)
- HTTP는 기본적으로 무상태(stateless)라서, 상태는 쿠키/세션/토큰 같은 별도 메커니즘으로 보완한다.

### 왜 필요한가
- 웹/백엔드/운영/보안 모든 영역에서 HTTP는 기본 전제다.
- 상태 코드/메서드 의미를 틀리면 클라이언트가 재시도/캐싱을 잘못해 장애로 이어질 수 있다.
- “서버가 느리다/안 된다”를 설명할 때도 결국 HTTP의 동작(요청 흐름/헤더/캐시/인증)을 알아야 한다.

### 동작 원리
- 메서드:
  - GET: 조회(캐싱/프리페치 영향을 받을 수 있음)
  - POST: 생성/처리(비멱등이 될 수 있음)
  - PUT: 전체 치환(멱등으로 설계하는 경우가 많음)
  - PATCH: 부분 수정(멱등 보장은 설계에 달림)
  - DELETE: 삭제(멱등으로 설계하는 경우가 많음)
- 상태 코드:
  - 2xx: 성공, 3xx: 리다이렉트, 4xx: 클라이언트 오류, 5xx: 서버 오류
  - 401(인증 필요) vs 403(권한 없음) 구분은 운영/디버깅에서 중요하다.

### 장단점 / 한계
- HTTP는 범용적이지만, 실시간/양방향 통신에는 한계가 있어 WebSocket/SSE 같은 다른 모델이 등장한다.
- 상태 코드/헤더가 표준이긴 해도, 실제 시스템에서는 프록시/브라우저/서버 구현 차이로 예외가 생길 수 있다.

### 대안 / 선택 기준
- 단순 API는 HTTP가 기본이고, 실시간이면 WebSocket/SSE를 선택한다.
- 내부 서비스 통신에서는 gRPC 같은 대안도 있지만, 운영/디버깅/브라우저 호환성 trade-off가 있다.

### 실무/디버깅 포인트
- 장애 분석에서 최소한 분리해야 할 것:
  - 서버가 요청을 받았는가(access log)
  - 애플리케이션이 처리했는가(app log/trace)
  - 업스트림/DB가 느렸는가(upstream timing)
- 브라우저/프록시/서버 어디에서 캐싱/리다이렉트가 발생했는지 구분한다.

### 프로젝트 연결
- WebSocket/STOMP, JWT 인증, Nginx 프록시 같은 요소는 HTTP 기본 이해 위에 쌓인다.
  - 관련: [[프로젝트/Autowing-Car]]

## HTTP 버전 / 연결

### 개념
- HTTP/1.1은 연결 재사용(keep-alive) 기반이고, HTTP/2는 멀티플렉싱으로 병렬성을 개선한다.
- HTTP/3는 QUIC(UDP 기반) 위에서 동작해 연결/지연 특성이 달라진다(세부는 구현/환경에 따라 달라 `확인 필요`).

### 왜 필요한가
- “왜 느린가”를 분석할 때 연결 수립 비용(TCP/TLS)과 전송 모델(1.1 vs 2)이 영향을 준다.
- 모바일/고지연 환경에서는 연결 재사용/핸드셰이크 비용이 체감 성능에 크게 영향을 줄 수 있다.

### 동작 원리
- Keep-Alive:
  - 연결을 매 요청마다 새로 만들지 않고 재사용해 TCP/TLS 비용을 줄인다.
  - 단, 서버/프록시 타임아웃으로 연결이 끊길 수 있어 클라이언트도 재연결을 대비한다.
- HOL(Head-of-Line) blocking:
  - HTTP/1.1에서는 한 연결에서 순차 처리/대기가 생기면 뒤 요청이 밀릴 수 있다.
  - HTTP/2는 스트림 단위 멀티플렉싱으로 응답이 섞여 전송될 수 있어 HOL 문제를 줄인다(단, 하위 계층(TCP)에서의 HOL은 다른 차원).
- TLS handshake:
  - HTTPS는 TCP 연결 후 TLS 핸드셰이크가 추가되고, 인증서 검증/키 합의가 수행된다.

### 장단점 / 한계
- HTTP/2는 성능에 유리할 수 있지만, 헤더 압축/다중화로 인해 장애 분석이 더 어려워질 수 있다.
- HTTP/3/QUIC은 이점이 있지만, 인프라/클라이언트 지원과 운영 복잡도가 변수가 된다.

### 대안 / 선택 기준
- 일반 서비스는 서버/프록시가 지원하면 HTTP/2를 고려할 수 있다.
- 다만 “성능은 측정으로 판단”해야 하고, 네트워크/클라이언트 다양성을 고려한다.

### 실무/디버깅 포인트
- 연결 관련 문제는:
  - keep-alive 타임아웃 미스매치, 프록시/로드밸런서 설정, TLS 설정 문제로 자주 나타난다.
- 클라이언트 개발자 도구에서 connection/SSL 타이밍을 보고 어느 구간이 긴지부터 본다.

### 프로젝트 연결
- 실시간/웹소켓은 “연결 유지”가 기본이라 keep-alive/타임아웃 이해가 중요하다.
  - 관련: [[프로젝트/Autowing-Car/00-상태-요약]]

## Header / Cookie / Caching

### 개념
- 헤더는 메시지의 메타데이터(타입, 인증, 캐시, 압축, 협상)를 담는다.
- 쿠키는 브라우저가 도메인/경로 규칙에 따라 자동으로 전송하는 상태 전달 메커니즘이다.
- 캐싱은 성능을 올리지만, “일관성/무효화” 문제가 항상 따라온다.

### 왜 필요한가
- 인증(JWT/세션)도 결국 헤더/쿠키로 전달된다.
- 캐시 설정 실수는 “배포했는데 안 바뀜”, “특정 사용자만 이상” 같은 장애로 보일 수 있다.
- 쿠키 옵션(SameSite/Secure/HttpOnly)은 보안(CSRF/XSS)과 직결된다.

### 동작 원리
- `Content-Type` vs `Accept`:
  - Content-Type: 내가 보내는 바디의 타입
  - Accept: 내가 받고 싶은 응답 타입
- 쿠키 전달:
  - 브라우저는 도메인/Path/만료/보안 옵션에 맞으면 자동으로 요청에 쿠키를 붙인다.
- 쿠키 옵션:
  - HttpOnly: JS에서 접근 제한(XSS로 인한 탈취 위험 완화)
  - Secure: HTTPS에서만 전송
  - SameSite: 크로스 사이트 요청에서 쿠키 전송 제한(CSRF 완화)
- 캐싱:
  - `Cache-Control`로 캐시 정책을 지정한다.
  - ETag/If-None-Match, Last-Modified/If-Modified-Since로 조건부 요청(304)을 만들 수 있다.

### 장단점 / 한계
- 쿠키는 편하지만 자동 전송 특성 때문에 CSRF 표면이 생긴다.
- 캐시는 성능을 올리지만, 무효화 정책이 없으면 오래된 데이터/리소스가 남는다.

### 대안 / 선택 기준
- 토큰 저장:
  - 쿠키(HttpOnly) vs 브라우저 저장소(LocalStorage) 선택은 위협 모델(XSS/CSRF)에 따라 결정한다.
- 캐시 전략:
  - 정적 리소스는 강한 캐시 + 파일명 해시(무효화 단순)
  - API 응답은 캐싱 여부를 신중히(개인화/권한/일관성 고려)

### 실무/디버깅 포인트
- 쿠키가 안 붙으면:
  - Domain/Path/Secure/SameSite 설정과 “요청이 HTTPS인지/출처가 무엇인지”를 먼저 본다.
- 캐시 문제는:
  - 브라우저 캐시/프록시 캐시/CDN 캐시를 분리해 본다.

### 프로젝트 연결
- JWT/Refresh Token을 쿠키/헤더로 전달하는 설계는 보안/운영과 결합된다.
  - 관련: [[프로젝트/Autowing-Car/03-인증-세션-토큰]]

## CORS / SOP / 브라우저 보안

### 개념
- SOP(Same-Origin Policy)는 브라우저가 “출처(origin)가 다른 리소스 접근”을 기본적으로 제한하는 보안 정책이다.
- origin은 보통 `scheme + host + port`로 정의된다.
- CORS는 SOP를 완화하기 위한 표준으로, 서버가 허용 헤더를 내려주면 브라우저가 교차 출처 요청을 허용한다.

### 왜 필요한가
- 프론트/백 분리(서브도메인/다른 도메인) 구조에서는 CORS 설정이 필수로 등장한다.
- CORS는 “서버가 막는 것”이 아니라 “브라우저가 막는 것”이라, 서버-서버 통신에는 같은 형태로 적용되지 않는다.
- 인증(쿠키/헤더)과 결합되면 `credentials` 옵션, wildcard origin 제한 같은 추가 규칙이 생긴다.

### 동작 원리
- Simple request vs Preflight:
  - 특정 조건을 만족하면 preflight 없이 요청이 가고,
  - 조건을 벗어나면 브라우저가 OPTIONS preflight로 허용 여부를 먼저 확인한다.
- 주요 헤더:
  - `Access-Control-Allow-Origin`
  - `Access-Control-Allow-Methods`
  - `Access-Control-Allow-Headers`
  - `Access-Control-Allow-Credentials`
- credentials:
  - 쿠키/인증 정보를 포함하는 요청에서 `Allow-Credentials: true`가 필요하고,
  - 이 경우 `Allow-Origin: *`는 사용할 수 없다는 제약이 있다.

### 장단점 / 한계
- CORS는 브라우저 보안 정책이므로, 공격자가 브라우저가 아닌 클라이언트를 쓰면(Curl 등) CORS로는 막을 수 없다.
- 즉, CORS는 “인증/인가 대체재”가 아니라, 브라우저 환경에서의 보호막이다.

### 대안 / 선택 기준
- 프론트/백이 같은 origin으로 붙으면 CORS 복잡도가 줄어든다(리버스 프록시로 같은 도메인에서 서빙).
- 도메인을 분리해야 하면:
  - 허용 origin을 최소화하고,
  - 쿠키 기반이면 CSRF까지 같이 고려한다.

### 실무/디버깅 포인트
- 브라우저에서만 실패하면 CORS를 의심한다.
- 서버 로그에 OPTIONS가 찍히는지, preflight 응답 헤더가 올바른지 확인한다.
- Security가 있는 서버는 “MVC에서만 CORS 설정”하면 실패할 수 있다(필터 체인에서 처리되는지 확인).

### 프로젝트 연결
- 브라우저 클라이언트가 있는 실시간 시스템은 CORS/인증/쿠키 정책이 합쳐져 복잡해진다.
  - 관련: [[프로젝트/Autowing-Car]]

## 인증 / 인가

### 개념
- 인증(Authentication): “누구인가”를 확인
- 인가(Authorization): “무엇을 할 수 있는가”를 확인
- 웹에서는 보통:
  - 세션 기반(stateful)
  - 토큰 기반(JWT, stateless)
중 하나(또는 혼합)로 구현한다.

### 왜 필요한가
- 인증 방식 선택은 운영/확장/보안 trade-off를 결정한다.
- 토큰 탈취, 세션 고정, CSRF/XSS 같은 위협 모델이 실제 사고로 이어질 수 있어 기본 설계를 탄탄히 해야 한다.

### 동작 원리
- 세션:
  - 로그인 시 서버가 세션을 만들고, 클라이언트는 세션 식별자(쿠키)를 보낸다.
  - 서버는 세션 저장소(메모리/Redis 등)에 상태를 유지한다.
- JWT:
  - 서버가 서명된 토큰을 발급하고, 클라이언트가 헤더(또는 쿠키)로 전달한다.
  - 서버는 토큰의 서명/만료/클레임을 검증한다.
- Refresh Token:
  - Access Token을 짧게 가져가고, Refresh Token으로 재발급하는 패턴이 흔하다.
  - Refresh Token을 서버에 저장하면 “강제 로그아웃/회수” 같은 운영 정책을 구현하기 쉬워진다.
- CSRF/XSS:
  - CSRF는 “브라우저가 쿠키를 자동 전송”하는 특성과 결합해 발생한다.
  - XSS는 스크립트 주입으로 토큰/세션 정보를 훔치거나 악성 요청을 수행하게 만든다.

### 장단점 / 한계
- 세션은 서버 통제가 쉬우나, 상태 저장/스케일링 문제가 생길 수 있다.
- JWT는 확장에 유리하지만, 탈취/회수/즉시 차단이 어려울 수 있다(정책 설계 필요).

### 대안 / 선택 기준
- 브라우저 중심이면:
  - 쿠키(HttpOnly) 기반 + CSRF 전략(SameSite/토큰 등)을 같이 설계
- API/모바일 중심이면:
  - Authorization 헤더 기반 토큰을 고려(저장/탈취 대응은 여전히 중요)

### 실무/디버깅 포인트
- “로그인은 됐는데 API가 401/403”은:
  - 토큰 만료/서명 키, 쿠키 전송 여부(SameSite/Secure), CORS/프리플라이트 차단을 분리한다.
- 만료/회전 정책은 문서로 고정해야 운영에서 혼선이 줄어든다.

### 프로젝트 연결
- JWT + Refresh Token을 Redis에 저장하는 형태의 경험이 기록돼 있다.

## REST / API 설계

### 개념
- REST는 리소스 기반 설계이며, URL은 리소스를 표현하고 동작은 HTTP 메서드로 표현하는 것이 기본 철학이다.
- API 설계에서 중요한 비기능 요구:
  - 버저닝, 호환성, 에러 모델, 멱등성, 페이징, 관측(로그/추적)

### 왜 필요한가
- API는 외부/내부 클라이언트와의 계약이라 변경 비용이 크다.
- 에러 모델/멱등성/재시도 정책이 불분명하면 운영 장애가 커진다.

### 동작 원리
- 리소스 중심 URL:
  - 명사 기반(`/users/1`), 행위는 메서드로 표현
- Pagination:
  - offset은 단순하지만 데이터가 커지면 비용이 커지고, 중간 삽입/삭제에 약하다.
  - cursor는 일관성/성능에 유리할 수 있지만 구현/디버깅이 복잡해질 수 있다.
- 멱등성:
  - “같은 요청을 여러 번 보내도 결과가 같다”는 성질로, 재시도 가능한 API 설계에 핵심이다.

### 장단점 / 한계
- 모든 기능이 CRUD로 떨어지지는 않는다(비동기 작업/워크플로우).
- 공통 응답 포맷은 일관성을 주지만, 지나치면 상태 코드 의미가 약해질 수 있다.

### 대안 / 선택 기준
- 조회가 복잡하면 GraphQL 같은 대안이 있을 수 있지만, 운영/캐시/권한 모델 trade-off가 있다.
- 내부 통신은 gRPC 같은 대안도 가능하지만, 브라우저/프록시 호환성까지 고려한다.

### 실무/디버깅 포인트
- 에러는 사용자 메시지와 디버깅 정보를 분리한다(코드/trace id 등).
- 재시도/서킷브레이커는 멱등성/부하를 근거로 적용한다.

### 프로젝트 연결
- API 설계는 인증/실시간/배포와 같이 움직이므로, 프로젝트 문서에서 구체 사례로 확장한다.
  - [[프로젝트/Autowing-Car]]

## 실시간 통신 / 웹소켓

### 개념
- Polling/Long Polling: 클라이언트가 주기적으로(또는 오래 열어) 서버에 요청해 이벤트를 받는 방식
- SSE: 서버→클라이언트 단방향 스트리밍(HTTP 기반)
- WebSocket: 양방향(full-duplex) 장기 연결(HTTP 업그레이드로 시작)
- STOMP 같은 프로토콜은 WebSocket 위에서 메시징 규약(구독/주제)을 제공한다.

### 왜 필요한가
- 실시간 요구(채팅/관제/알림/게임/스트리밍)에서 요청-응답만으로는 지연/비용이 커진다.
- 실시간 시스템은 연결 유지/재연결/하트비트/백프레셔 같은 운영 고려가 필요하다.

### 동작 원리
- WebSocket handshake:
  - HTTP 요청에 Upgrade 헤더를 포함하고, 서버가 101 응답으로 프로토콜 전환한다.
- 재연결 전략:
  - 모바일/불안정 네트워크에서는 끊김이 정상이라, 지수 백오프/재구독/중복 처리 전략이 필요하다.
- 메시지 브로커:
  - 다중 인스턴스/팬아웃에서는 메시지를 중계/브로드캐스트하기 위한 브로커(또는 중앙 저장소)가 필요할 수 있다.

### 장단점 / 한계
- WebSocket은 양방향이 강력하지만, 프록시/타임아웃/연결 수 관리가 어렵다.
- SSE는 단방향이라 단순하지만, 양방향 상호작용에는 부족하다.

### 대안 / 선택 기준
- 단순 알림/피드면 SSE가 더 단순할 수 있다.
- 양방향 상호작용/저지연이면 WebSocket이 적합한 경우가 많다.
- “최신성 vs 순서/재전송” 요구에 따라 TCP 기반(WebSocket)과 UDP 기반 설계의 trade-off가 생길 수 있다.

### 실무/디버깅 포인트
- 끊김 문제는:
  - 프록시 타임아웃, 로드밸런서 idle timeout, 하트비트 주기 불일치가 흔한 원인이다.
- 백프레셔는:
  - 소비자가 못 따라올 때 버퍼가 무한히 쌓이지 않게 제어하는 설계다.

### 프로젝트 연결
- MQTT → 서버 도메인 처리 → WebSocket(STOMP) 브로드캐스트 흐름이 기록돼 있다.

## 웹 성능 / 운영

### 개념
- 웹 성능은 “네트워크 + 렌더링 + 서버 처리”의 합이다.
- CDN은 사용자 가까운 엣지에서 정적/캐시 가능한 리소스를 제공해 지연과 오리진 부하를 줄인다.
- 운영에서는 “어디가 병목인지”를 분리하는 관측 지점이 핵심이다.

### 왜 필요한가
- 느린 웹은 사용성/비즈니스 지표에 직접 영향을 준다.
- 성능 문제는 대개 한 지점이 아니라 여러 지점이 누적되어 나타난다.

### 동작 원리
- 브라우저 렌더링(개념):
  - HTML 파싱 → DOM/CSSOM → 렌더 트리 → 레이아웃 → 페인트/컴포지팅
- Critical Rendering Path:
  - 초기 렌더에 필요한 리소스(HTML/CSS/JS)의 로딩/실행 경로를 최적화하는 개념이다.
- Lazy loading:
  - 당장 보이지 않는 리소스를 늦게 로드해 초기 로딩을 가볍게 한다.

### 장단점 / 한계
- 최적화는 비용이 있으므로, “측정(타이밍/지표) 기반”으로 투자해야 한다.
- CDN/캐시는 무효화/일관성 문제가 있어 운영 규칙이 필요하다.

### 대안 / 선택 기준
- 서버 성능:
  - 캐시, 쿼리 최적화, 비동기 처리, 커넥션 풀/스레드 풀 조정
- 클라이언트 성능:
  - 번들 분리, 압축, 캐시, 이미지 최적화

### 실무/디버깅 포인트
- 장애 분석 순서(예시):
  - 클라이언트(콘솔/네트워크 탭) → Nginx/게이트웨이 로그 → WAS 로그/메트릭 → DB/외부 API
- TTFB가 길면 서버 처리/업스트림 지연을 우선 의심한다.

### 프로젝트 연결
- 실시간 시스템은 “성능/운영”이 기능 요구와 같이 움직인다.

## DNS / 요청 흐름 / 네트워크 디버깅

### 개념
- URL 입력 시 대표 흐름(요약):
  - DNS 조회 → TCP 연결(3-way handshake) → TLS handshake(HTTPS) → HTTP 요청/응답
- DNS는 이름→주소 해석 시스템이고, TTL/캐시/전파가 실무 문제를 만든다.
- 네트워크 디버깅은 “어느 구간이 느린지/깨지는지”를 계층별로 분리하는 작업이다.

### 왜 필요한가
- “도메인은 맞는데 접속이 안 됨”은 DNS/방화벽/Nginx/업스트림 중 어디가 원인인지 분리가 필요하다.
- DNS 변경이 바로 반영되지 않는(일부 사용자만 안 됨) 상황은 TTL/캐시로 설명해야 한다.
- 컨테이너/서버 환경에서만 DNS가 깨지는 경우가 흔하다.

### 동작 원리
- DNS 캐시 계층:
  - 브라우저/OS/공유기/리졸버/중간 캐시 등 여러 계층에서 캐시될 수 있다.
- 3-way handshake:
  - 새 TCP 연결이 필요할 때 HTTP 요청 전 수행된다(keep-alive로 재사용하면 반복되지 않음).
- TLS handshake:
  - 서버 인증서 검증과 암호화 키 합의를 수행하고, SNI/ALPN 등 확장으로 도메인/프로토콜 협상을 한다.
- 개발자 도구 타이밍:
  - DNS Lookup / Initial Connection / SSL / TTFB를 보면 어느 구간이 긴지 1차 분리가 가능하다.

### 장단점 / 한계
- DNS 기반 라우팅/CDN은 유연하지만, 캐시/전파로 인해 일시적 불일치가 생길 수 있다.
- “내 컴퓨터는 되는데”는 캐시/리졸버 차이로 쉽게 발생한다.

### 대안 / 선택 기준
- 문제를 줄이려면:
  - TTL을 전략적으로 설계하고(너무 길면 전파 느림, 너무 짧으면 부하/불안정),
  - 배포/이전 시에는 전환 절차(이중화/점진 전환)를 둔다.
- 컨테이너 환경에서는:
  - 서비스 이름 기반 통신과 네트워크 스코프를 명확히 한다.

### 실무/디버깅 포인트
- 증상별 체크:
  - DNS 실패: NXDOMAIN/Unknown host, resolver 설정, 사내 DNS/프록시
  - TCP 실패: 방화벽/보안그룹/포트, 서버 다운
  - TLS 실패: 인증서/시간 동기화/신뢰 체인
  - TTFB 지연: 서버 처리/업스트림/DB 지연
- 도구:
  - `nslookup`/`dig`로 특정 리졸버의 응답/TTL/레코드를 확인한다.
- “클라이언트는 되는데 서버(컨테이너)만 안 됨”은:
  - 컨테이너 DNS 설정(`/etc/resolv.conf`), 네트워크 격리, 런타임 DNS 캐시를 순서대로 본다.
