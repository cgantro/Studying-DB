# TCP와 UDP

## 한 줄 정의
TCP와 UDP는 **전송 계층에서 프로그램끼리 데이터를 주고받는 대표 프로토콜**이다.

- **TCP**: 연결을 먼저 만들고, 순서와 신뢰성을 중요하게 여긴다
- **UDP**: 연결 절차를 줄이고, 빠르고 가볍게 보내는 데 초점을 둔다

쉽게 말하면:
- TCP = 안전하게 보내기
- UDP = 빨리 보내기

---

## 왜 배우는가
- 웹, 로그인, 파일 전송, 게임, 스트리밍까지 전송 계층은 거의 모든 네트워크 서비스의 바닥에 있다.
- 같은 데이터를 보내더라도 어떤 상황에서는 안정성이 중요하고, 어떤 상황에서는 지연 시간이 더 중요하다.
- TCP와 UDP를 알면 왜 서비스가 느린지, 왜 끊기는지, 왜 순서가 바뀌지 않는지를 설명할 수 있다.
- 면접에서도 "왜 이 서비스는 TCP를 쓰고, 저 서비스는 UDP를 쓰나요?"가 자주 나온다.

---

## 초보자용 한 줄 기억법
- **TCP는 편지를 등기처럼 보내는 느낌**
- **UDP는 빨리 툭 던지는 느낌**

---

## 전체 흐름

```text
애플리케이션
   ↓
전송 계층(TCP 또는 UDP)
   ↓
IP 계층
   ↓
데이터 링크/물리 계층
```

TCP와 UDP는 모두 **전송 계층**에 있지만, 동작 방식이 많이 다르다.

---

## 먼저 큰 그림
전송 계층은 단순히 "보내는" 게 아니라,
- 어떤 프로그램으로 보낼지(포트)
- 신뢰성 있게 보낼지
- 순서를 보장할지
- 재전송이 필요한지
- 흐름과 혼잡을 제어할지
를 결정한다.

즉, TCP와 UDP는 같은 계층이지만 **철학이 다르다**.

---

## TCP란
TCP(Transmission Control Protocol)는 **연결 지향적이고 신뢰성 있는 전송**을 제공하는 프로토콜이다.

### 핵심 특징
- 연결을 먼저 만든다
- 데이터 순서를 보장한다
- 손실되면 재전송한다
- 흐름 제어를 한다
- 혼잡 제어를 한다

### TCP를 쓰는 이유
- 빠뜨리면 안 되는 데이터
- 순서가 중요한 데이터
- 한 번 잘못 보내면 큰 문제가 나는 데이터

---

## TCP의 동작 이해
TCP는 그냥 데이터를 던지지 않는다. 먼저 상대와 "우리 통신 시작하자"를 합의한다.

### 3-way handshake
```text
클라이언트 → 서버 : SYN
클라이언트 ← 서버 : SYN+ACK
클라이언트 → 서버 : ACK
```

### 이 과정이 의미하는 것
- 서로 연결할 준비가 됐는지 확인한다
- 양쪽이 통신 상태를 맞춘다
- 이후 실제 데이터 전송을 시작한다

### 비유
- 문을 두드림(SYN)
- 상대가 응답함(SYN+ACK)
- 나도 확인함(ACK)
- 그다음 본격적으로 대화 시작

---

## TCP가 신뢰성을 만드는 방법
TCP는 여러 장치를 이용해 데이터를 안정적으로 전달한다.

### 1) 순서 번호(sequence number)
- 데이터 조각마다 순서를 붙인다
- 도착 순서가 뒤죽박죽이어도 다시 정렬할 수 있다

### 2) ACK(응답 확인)
- 받았다고 알려준다
- 못 받았으면 다시 보내게 할 수 있다

### 3) 재전송
- 패킷이 사라지면 다시 보낸다
- 타임아웃이나 중복 ACK 등을 기반으로 동작한다

### 4) 흐름 제어(flow control)
- 상대가 처리할 수 있는 속도에 맞춰 보낸다
- 수신 버퍼가 넘치지 않게 한다

### 5) 혼잡 제어(congestion control)
- 네트워크 자체가 너무 막히지 않게 조절한다
- 전송 속도를 상황에 맞게 조정한다

---

## TCP가 다루는 핵심 개념

| 개념 | 의미 | 한 줄 기억 |
|---|---|---|
| 연결 지향 | 먼저 연결을 맺음 | 시작 전에 인사함 |
| 신뢰성 | 빠진 데이터 복구 | 안 오면 다시 보냄 |
| 순서 보장 | 도착 순서 정렬 | 번호표 붙이기 |
| 흐름 제어 | 수신자가 감당 가능한 만큼 | 상대 속도 맞추기 |
| 혼잡 제어 | 네트워크가 막히지 않게 | 길이 막히면 속도 줄임 |

---

## UDP란
UDP(User Datagram Protocol)는 **연결 절차 없이 빠르게 데이터를 보내는 프로토콜**이다.

### 핵심 특징
- 연결을 따로 만들지 않는다
- 순서 보장이 없다
- 재전송을 기본으로 하지 않는다
- 오버헤드가 적다
- 지연 시간이 짧은 편이다

### UDP를 쓰는 이유
- 조금 잃어도 괜찮다
- 늦게 오는 것보다 빨리 오는 게 중요하다
- 실시간성이 중요하다
- 애플리케이션이 직접 보정할 수 있다

---

## UDP 이해
UDP는 상대에게 "일단 이걸 보내둘게" 하고 빠르게 던지는 느낌이다.

### 비유
- TCP: 택배 + 송장 + 수령 확인
- UDP: 말 그대로 바로 던짐

---

## TCP와 UDP 비교표

| 항목 | TCP | UDP |
|---|---|---|
| 연결 | 연결 지향 | 비연결 지향 |
| 순서 보장 | 있음 | 없음 |
| 신뢰성 | 높음 | 낮음 |
| 재전송 | 있음 | 기본 없음 |
| 흐름 제어 | 있음 | 없음 |
| 혼잡 제어 | 있음 | 없음 |
| 속도/오버헤드 | 상대적으로 무거움 | 가벼움 |
| 지연 시간 | 상대적으로 큼 | 작음 |
| 사용 예 | 웹, 파일 전송, 로그인 | 게임, 음성, 스트리밍 |

---

## TCP와 UDP가 다르게 보이는 이유
둘은 단순히 "느리다/빠르다"의 차이가 아니라, **목표가 다르다**.

### TCP의 목표
- 정확하게
- 순서대로
- 빠진 것 없이

### UDP의 목표
- 빠르게
- 단순하게
- 가볍게

### 선택 기준
- 누락이 치명적이면 TCP
- 지연이 더 치명적이면 UDP

---

## 어떤 서비스가 어떤 걸 쓰나

### TCP가 잘 맞는 예
- 웹 페이지 요청/응답
- 로그인
- 파일 다운로드
- 결제
- 이메일
- DB 통신

### UDP가 잘 맞는 예
- 온라인 게임
- 실시간 음성/영상 통화
- 스트리밍
- 센서 데이터
- 빠른 상태 동기화

### 중요한 포인트
- UDP가 무조건 빠르고 TCP가 무조건 느린 것은 아니다
- 실제 성능은 네트워크 상황, 구현 방식, 애플리케이션 설계에 따라 달라진다

---

## 흐름 차이를 그림으로 보기

### TCP
```text
연결 수립
   ↓
데이터 전송
   ↓
ACK 확인
   ↓
재전송/순서 정리
   ↓
연결 종료
```

### UDP
```text
데이터 전송
   ↓
끝
```

---

## TCP와 UDP의 포트
둘 다 포트를 사용해서 어떤 프로그램으로 보낼지 구분한다.

### 예시
- HTTP: 80/TCP
- HTTPS: 443/TCP
- DNS: 53/UDP 또는 53/TCP
- 게임 서버: 서비스에 따라 UDP/TCP 모두 가능

### 포트의 역할
- IP는 어느 장비인지
- 포트는 그 장비 안의 어느 프로그램인지

---

## TCP의 상태 이해
TCP는 연결을 관리하므로 상태가 있다.

### 예시 상태 이해
- 연결 전
- 연결 중
- 데이터 주고받는 중
- 종료 중

### 왜 중요한가
- 연결이 언제 시작되고 끝나는지 추적할 수 있다
- 서버 입장에서는 여러 연결 상태를 관리해야 한다

---

## UDP의 상태 이해
UDP는 연결을 따로 맺지 않으므로 TCP보다 단순하다.

### 장점
- 빠르게 시작 가능
- 연결 유지 비용이 적다
- 상태 관리 부담이 적다

### 단점
- 잃어버린 데이터는 애플리케이션이 처리해야 할 수 있다
- 순서와 신뢰성을 직접 보강해야 할 수 있다

---

## TCP에서 자주 나오는 추가 개념

### 재전송
- 데이터가 안 오면 다시 보낸다

### 슬라이딩 윈도우
- 한 번에 너무 많이 보내지 않도록 조절하는 방식

### ACK
- 받았다는 응답

### 윈도우 크기
- 지금 얼마나 더 받을 수 있는지 나타내는 이해

### 혼잡 제어
- 네트워크가 막히면 전송 속도를 줄인다

---

## UDP에서 자주 나오는 추가 개념

### 체크섬
- 기본적인 오류 검출에 도움을 준다
- 하지만 TCP처럼 강한 보장까지는 아니다

### 실시간성
- 지연 시간이 더 중요할 때 선택한다

### 애플리케이션 책임
- 순서 보정, 누락 보정, 재조립 등을 앱이 맡을 수 있다

---

## 신뢰성 vs 지연 시간
이 두 가지가 가장 중요하다.

### 신뢰성이 중요할 때
- 파일이 하나라도 빠지면 안 됨
- 주문이나 결제 데이터가 틀리면 안 됨
- 로그인 세션이 안정적으로 유지되어야 함

### 지연 시간이 중요할 때
- 게임 캐릭터 이동
- 음성 통화
- 실시간 스트리밍
- 빠른 상태 동기화

---

## 실전 예시

### 예시 1: 웹 사이트 열기
- 브라우저는 보통 TCP로 서버에 연결한다
- 순서와 신뢰성이 중요하다
- 응답이 하나라도 빠지면 페이지가 깨질 수 있다

### 예시 2: 파일 다운로드
- 일부 조각이 빠지면 파일이 손상된다
- TCP가 잘 맞는다

### 예시 3: 온라인 게임
- 위치 정보가 조금 늦게 오면 게임이 어색해진다
- 빠름이 중요하므로 UDP가 유리할 수 있다

### 예시 4: 화상 통화
- 약간의 프레임 손실보다 지연이 더 문제다
- UDP가 자주 거론된다

### 예시 5: DNS 조회
- 보통 빠르게 질의하고 응답을 받는 방식이므로 UDP가 자주 쓰인다
- 경우에 따라 TCP도 사용한다

---

## 자주 하는 실수
- TCP = 느림, UDP = 빠름만 외우고 이유를 모름
  - 바로잡기: TCP는 연결, 순서 보장, 재전송, 흐름/혼잡 제어를 제공하기 때문에 오버헤드가 있고 UDP는 그 기능을 애플리케이션이 선택적으로 구현한다.
- UDP는 아무 보장도 없다고 단순하게 생각함
  - 바로잡기: UDP 자체는 전달/순서/재전송을 보장하지 않지만, 애플리케이션 프로토콜이 필요한 보장을 직접 추가할 수 있다.
- TCP의 순서 보장과 재전송 원리를 모름
  - 바로잡기: TCP는 sequence number, ACK, 재전송, 수신 버퍼를 통해 순서와 신뢰성을 제공한다.
- 흐름 제어와 혼잡 제어를 헷갈림
  - 바로잡기: 흐름 제어는 수신자가 감당 가능한 속도를 맞추는 것이고, 혼잡 제어는 네트워크 전체 혼잡을 줄이는 것이다.
- 포트와 프로토콜을 섞어 생각함
  - 바로잡기: 프로토콜은 통신 규칙이고, 포트는 한 호스트 안에서 어떤 프로세스/서비스로 전달할지 구분하는 번호다.
- UDP를 쓰면 무조건 게임이 좋다고 단정함
  - 바로잡기: 게임도 위치 동기화, 재전송 필요 여부, 지연 허용 범위에 따라 UDP 위에 별도 신뢰성 로직을 얹거나 TCP/QUIC을 선택할 수 있다.
- TCP와 HTTP를 같은 개념으로 착각함
  - 바로잡기: TCP는 전송 계층의 연결/전송 프로토콜이고, HTTP는 그 위에서 요청/응답 의미를 정의하는 애플리케이션 프로토콜이다.

## 디버깅할 때 보는 순서
1. TCP인지 UDP인지 확인
2. 포트가 맞는지 확인
3. 연결이 실제로 맺어졌는지 확인(TCP)
4. 패킷 손실이 있는지 확인
5. 지연이 문제인지 손실이 문제인지 구분
6. 서버의 처리 속도와 버퍼 상태 확인
7. 방화벽/네트워크 장비에서 막히지 않는지 확인

## 비교해서 이해하기

### TCP vs UDP
- TCP: 연결 지향, 신뢰성, 순서 보장
- UDP: 비연결, 가벼움, 지연 시간 중심

### TCP vs 편지
- TCP: 등기 + 수령 확인 + 재전송
- UDP: 메모를 빨리 전달

### UDP vs 브로드캐스트
- UDP도 브로드캐스트/멀티캐스트와 함께 언급될 수 있다
- 하지만 둘은 같은 개념은 아니다

## ASCII 그림

### TCP 연결 수립
```text
클라이언트 -- SYN --> 서버
클라이언트 <-- SYN+ACK -- 서버
클라이언트 -- ACK --> 서버
```

### UDP 전송
```text
클라이언트 -------------> 서버
(연결 절차 없음)
```

### 선택 기준
```text
빠짐이 치명적? → TCP
지연이 치명적? → UDP
```

## 면접 포인트
면접에서는 "TCP와 UDP 차이"를 암기식으로만 말하면 약하다. 이유와 선택 기준이 중요하다.

### 꼭 말할 수 있어야 하는 것
- TCP는 연결 지향이고 신뢰성을 제공한다
- UDP는 비연결 지향이고 지연이 짧고 가볍다
- TCP는 3-way handshake, 재전송, 흐름 제어, 혼잡 제어가 있다
- UDP는 이런 추가 보장을 기본으로 하지 않는다
- 어떤 프로토콜을 쓸지는 서비스 요구사항에 따라 정한다

### 한 번에 말하는 답변 예시
> TCP는 연결을 먼저 맺고 순서 보장, 재전송, 흐름 제어, 혼잡 제어를 통해 신뢰성 있게 데이터를 전달합니다. UDP는 연결 절차가 없어서 가볍고 지연이 짧지만, 순서 보장이나 재전송을 기본으로 하지 않습니다. 그래서 웹이나 파일 전송처럼 정확성이 중요하면 TCP를, 게임이나 실시간 음성처럼 지연 시간이 더 중요하면 UDP를 고려합니다.

### 꼬리 질문 예시
- 3-way handshake는 왜 필요한가요?
- 흐름 제어와 혼잡 제어 차이는 뭔가요?
- UDP는 왜 재전송을 안 하나요?
- DNS는 왜 UDP를 많이 쓰나요?
- 게임은 왜 UDP를 쓰는 경우가 많나요?

---

## 한 번 더 요약
- TCP는 연결 지향, 신뢰성, 순서 보장
- UDP는 비연결, 가벼움, 지연 시간 중심
- TCP는 보장을 많이 하는 대신 오버헤드가 있다
- UDP는 빠르지만 누락과 순서를 애플리케이션이 다뤄야 할 수 있다
- 선택은 "안전"과 "속도"의 균형이다

## 초보자용 한 줄 기억법
TCP는 **"안전하게 보내는 프로토콜"**, UDP는 **"빨리 보내는 프로토콜"**이다.
