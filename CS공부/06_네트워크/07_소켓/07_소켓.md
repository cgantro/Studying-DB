# 소켓

## 한 줄 정의
소켓(socket)은 **프로그램이 네트워크와 통신하기 위해 사용하는 출입구이자 인터페이스**다.

쉽게 말하면, 소켓은 **"이 프로그램이 인터넷/네트워크와 대화하는 창문"**이다.

---

## 왜 배우는가
- 네트워크 통신은 결국 프로그램이 실제로 어떻게 보내고 받는지 이해해야 끝난다.
- 서버/클라이언트 구조를 이해하려면 소켓을 알아야 한다.
- 장애가 났을 때 "연결이 안 됨", "응답이 늦음", "포트가 안 열림" 같은 문제를 설명하려면 소켓 개념이 필요하다.
- 웹 서버, 채팅 서버, 게임 서버, 파일 전송, 마이크로서비스 통신까지 전부 소켓과 연결된다.

---

## 초보자용 한 줄 기억법
- **IP는 집 주소**
- **포트는 그 집의 호수/문 번호**
- **소켓은 그 문을 실제로 여닫는 프로그램용 출입구**

---

## 소켓을 한 문장으로 다시 말하면
소켓은 **IP 주소와 포트를 이용해, 프로그램이 네트워크 데이터를 보내고 받도록 해 주는 연결점**이다.

---

## 전체 흐름

```text
[클라이언트 프로그램]
   ↓ connect()
[소켓 생성]
   ↓
[IP + 포트 대상 연결]
   ↓
[TCP/UDP 프로토콜]
   ↓
[네트워크]
   ↓
[서버 소켓]
   ↓ accept()/recv()/send()
[서버 프로그램]
```

---

## 소켓의 핵심 역할
소켓은 단순한 "통신용 객체"가 아니다. 아래 역할을 함께 가진다.

- 네트워크 주소를 프로그램과 연결한다
- 연결 상태를 관리한다
- 데이터를 읽고 쓰는 통로 역할을 한다
- TCP/UDP 같은 전송 방식 차이를 감싼다
- 서버가 여러 클라이언트와 통신하게 만든다

---

## 소켓이 다루는 기본 요소

### 1) IP 주소
- 어디로 갈지 나타내는 논리적 주소
- 예: `192.168.0.10`, `8.8.8.8`

### 2) 포트 번호
- 같은 장비 안에서 어느 프로그램으로 갈지 구분하는 번호
- 예: `80`, `443`, `3306`, `8080`

### 3) 프로토콜
- TCP인지 UDP인지
- 어떤 방식으로 데이터를 전달할지 정한다

### 4) 소켓 주소
보통 IP + 포트 + 프로토콜이 함께 의미를 가진다.

```text
192.168.0.10:8080/TCP
```

---

## 서버와 클라이언트

### 서버
- 먼저 대기한다
- 연결 요청을 받는다
- 클라이언트 여러 명을 상대할 수 있다

### 클라이언트
- 먼저 연결을 시도한다
- 요청을 보내고 응답을 기다린다
- 보통 서버보다 먼저 움직인다

### 한 줄 비교
- 서버 = 문 열고 손님 기다리는 사람
- 클라이언트 = 문 열고 들어가는 사람

---

## 소켓의 기본 흐름

### 서버 흐름
```text
socket()
  ↓
bind()
  ↓
listen()
  ↓
accept()
  ↓
recv()/send()
  ↓
close()
```

### 클라이언트 흐름
```text
socket()
  ↓
connect()
  ↓
send()/recv()
  ↓
close()
```

---

## 각 단계가 무슨 뜻인가

### socket()
- 네트워크 통신을 위한 소켓을 만든다.
- 아직 연결은 안 됐고, 그릇만 준비한 상태다.

### bind()
- 서버가 특정 IP와 포트에 자신을 묶는다.
- "이 포트로 들어오는 요청을 내가 받겠다"는 의미다.

### listen()
- 서버가 연결 요청을 기다린다.
- 대기열(backlog)을 둘 수 있다.

### accept()
- 클라이언트 연결을 실제로 받아들인다.
- 새 연결용 소켓이 만들어지는 것처럼 이해하면 쉽다.

### connect()
- 클라이언트가 서버에 연결을 요청한다.

### send()/recv()
- 데이터를 보내고 받는다.
- 실제 통신은 이 단계에서 일어난다.

### close()
- 연결을 종료한다.

---

## 소켓의 흐름 그림

```text
[클라이언트]                        [서버]
    |                                 |
    | ---- connect 요청 ------------> |
    |                                 |  socket()
    |                                 |  bind()
    |                                 |  listen()
    |                                 |  accept()
    | <--- 연결 수락 ---------------- |
    |                                 |
    | ---- send 요청 ------------->   |
    |                                 |  recv()
    |                                 |  처리
    | <--- 응답 -------------------   |
    |                                 |
    | -------- close ---------------> |
```

## TCP 소켓 vs UDP 소켓
소켓은 전송 계층 프로토콜에 따라 성격이 달라진다.

### TCP 소켓
- 연결 지향
- 순서 보장
- 신뢰성 높음
- 흐름 제어와 혼잡 제어가 있음
- 서버/클라이언트 흐름이 뚜렷하다

### UDP 소켓
- 비연결 지향
- 빠르고 가벼움
- 순서/재전송 보장이 없음
- 실시간성 중요할 때 자주 사용

### 비교표

| 항목 | TCP 소켓 | UDP 소켓 |
|---|---|---|
| 연결 | 필요 | 불필요 |
| 신뢰성 | 높음 | 낮음 |
| 순서 보장 | 있음 | 없음 |
| 속도/오버헤드 | 상대적으로 무거움 | 가벼움 |
| 적합한 예 | 웹, 파일 전송, 로그인 | 게임, 음성, 스트리밍 |

---

## 블로킹과 논블로킹

### 블로킹 소켓
- 데이터가 올 때까지 함수가 멈춰 기다린다.
- 코드가 단순해 보이지만, 대기 때문에 성능 설계가 필요하다.

### 논블로킹 소켓
- 바로 돌아온다.
- 데이터가 없으면 "없다"고 알려준다.
- 이벤트 루프, 셀렉트/폴/이폴 같은 방식과 함께 쓴다.

### 비유
- 블로킹: 식당에서 음식 올 때까지 자리에서 기다림
- 논블로킹: 주문하고 다른 일 하다가 나중에 다시 확인함

---

## 동기 I/O vs 비동기 I/O

### 동기 I/O
- 호출한 쪽이 결과를 직접 기다린다.
- 흐름이 직관적이다.

### 비동기 I/O
- 요청만 던지고 나중에 완료를 받는다.
- 대규모 동시 처리에 유리할 수 있다.

### 헷갈리지 말 것
- 블로킹/논블로킹은 "기다리느냐"의 성격
- 동기/비동기는 "결과를 언제 받느냐"의 성격
- 서로 비슷하지만 같은 개념은 아니다

---

## 소켓이 실제로 쓰이는 곳
- 웹 서버와 브라우저
- 채팅 앱
- 게임 서버
- DB 클라이언트와 DB 서버
- 파일 업로드/다운로드
- 메시지 큐
- 마이크로서비스 간 API 호출

---

## 실전 예시 1: 웹 브라우저가 서버에 요청
1. 브라우저가 서버 주소를 알아낸다
2. 소켓을 열고 서버의 IP:포트로 연결한다
3. HTTP 요청을 보낸다
4. 서버가 응답을 돌려준다
5. 연결을 닫거나 keep-alive로 유지한다

### 한 번에 보기
```text
브라우저 ──socket──> 서버
브라우저 <─응답────── 서버
```

---

## 실전 예시 2: 채팅 서버
- 클라이언트는 서버에 연결을 유지한다
- 서버는 여러 소켓을 동시에 관리한다
- 한 사용자가 보낸 메시지를 다른 사용자에게 전달한다

### 특징
- 연결 수가 많다
- 실시간성이 중요하다
- 동시 접속 처리가 중요하다

---

## 실전 예시 3: 파일 전송
- 클라이언트가 파일을 조각내서 보낸다
- 서버는 조각을 받아 순서대로 재조립한다
- TCP를 쓰면 순서와 신뢰성이 유리하다

---

## 포트 번호를 왜 쓰는가
한 장비에는 여러 프로그램이 동시에 돌아간다. IP만으로는 어느 프로그램으로 보낼지 모른다.
그래서 포트가 필요하다.

### 예시
- `:80` → HTTP
- `:443` → HTTPS
- `:3306` → MySQL
- `:6379` → Redis

---

## 소켓과 포트의 관계
- IP는 장비를 찾는다
- 포트는 그 장비 안의 프로그램을 찾는다
- 소켓은 그 둘을 묶어 통신을 만든다

```text
IP = 건물 주소
포트 = 호수
소켓 = 그 호수로 통신하는 창구
```

---

## 소켓 통신이 어려운 이유
- 연결이 항상 성공하는 게 아니다
- 상대 프로그램이 꺼져 있을 수 있다
- 네트워크가 끊길 수 있다
- 패킷이 늦거나 순서가 꼬일 수 있다
- 서버는 여러 연결을 동시에 관리해야 한다

---

## 자주 하는 실수
- IP와 포트를 혼동함
  - 바로잡기: IP는 호스트를 찾는 주소이고, 포트는 그 호스트 안의 애플리케이션 엔드포인트를 구분한다.
- TCP와 UDP 소켓의 차이를 무시함
  - 바로잡기: TCP 소켓은 연결 기반 스트림이고, UDP 소켓은 연결 없이 datagram을 주고받는 방식이다.
- 서버는 무조건 accept만 하고 끝이라고 생각함
  - 바로잡기: 서버는 accept 이후에도 클라이언트별 연결 처리, read/write, 종료, 에러 처리를 계속 수행해야 한다.
- 블로킹과 논블로킹을 같은 말로 씀
  - 바로잡기: 블로킹은 호출한 스레드가 기다리는 방식이고, 논블로킹은 준비되지 않으면 즉시 반환해 나중에 다시 시도하거나 이벤트로 처리하는 방식이다.
- 연결이 안 되면 무조건 서버 코드만 의심함
  - 바로잡기: 연결 실패는 서버 코드뿐 아니라 서버 실행 여부, 바인딩 주소, 포트, 방화벽, 라우팅, DNS, 클라이언트 설정을 함께 확인해야 한다.
- 방화벽, 포트 개방, DNS 문제를 놓침
  - 바로잡기: 연결 실패는 서버 코드뿐 아니라 포트 개방, 방화벽, 라우팅, DNS, 바인딩 주소 문제도 함께 확인해야 한다.

---

## 디버깅할 때 보는 순서
1. 서버 프로세스가 살아 있는가
2. 포트가 실제로 열려 있는가
3. IP/포트가 맞는가
4. 같은 네트워크에서 접근 가능한가
5. 방화벽이 막고 있지 않은가
6. DNS가 올바르게 해석되는가
7. TCP인지 UDP인지 맞는가
8. 블로킹/타임아웃 설정이 이상하지 않은가

## 비교해서 이해하기

### 소켓 vs TCP/UDP
- 소켓: 프로그램이 네트워크와 통신하는 인터페이스
- TCP/UDP: 어떤 방식으로 통신할지 정하는 프로토콜

### 서버 소켓 vs 연결 소켓
- 서버 소켓: 연결을 기다리는 문
- 연결 소켓: 실제로 클라이언트와 데이터 주고받는 문

### 블로킹 vs 논블로킹
- 블로킹: 기다리면서 멈춤
- 논블로킹: 바로 돌아옴

## 면접 포인트
면접에서는 "소켓이 뭔가요?"보다 흐름을 설명할 수 있어야 한다.

### 꼭 말할 수 있어야 하는 것
- 소켓은 네트워크 통신을 위한 프로그램의 인터페이스다
- IP와 포트를 함께 사용해 통신 상대를 찾는다
- 서버는 bind/listen/accept 흐름을, 클라이언트는 connect 흐름을 따른다
- TCP와 UDP 소켓은 성격이 다르다
- 블로킹/논블로킹, 동기/비동기를 구분할 수 있어야 한다

### 한 번에 말하는 답변 예시
> 소켓은 프로그램이 네트워크와 통신할 때 사용하는 인터페이스입니다. IP와 포트를 통해 상대를 식별하고, 서버는 보통 bind, listen, accept 순서로 연결을 받으며, 클라이언트는 connect로 연결을 시작합니다. TCP 소켓은 연결과 신뢰성이 중요할 때, UDP 소켓은 지연이 더 중요할 때 자주 사용합니다.
