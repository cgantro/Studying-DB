## 1. 전체 구조 파악

### 1-1. 서버 전체 아키텍처 요약

**[코드 확인]**

이 프로젝트는 자율주행 견인차(Towing Car)와 항공기(Aircraft)를 관제하는 **실시간 모니터링 및 제어 관제 서버**입니다.

- **디바이스 ↔ 서버 (MQTT):** 견인차 하드웨어(Orin/STM32)와는 MQTT 프로토콜을 통해 상태 데이터(Telemetry)를 10Hz 이상의 고주파수로 수신하고, 제어 명령(Command)을 송신합니다.
- **클라이언트 ↔ 서버 (HTTP / WebSocket):** 프론트엔드(웹) 클라이언트와는 REST API로 CRUD를 처리하고, STOMP 기반 WebSocket으로 차량의 실시간 위치 및 미션 상태를 브로드캐스팅합니다.
- **데이터 저장 (Redis / DB):** 실시간 차량 최신 상태는 Redis에 캐싱하여 조회 성능을 높이고, 주행 로그 및 히스토리는 PostgreSQL(TimescaleDB 확장)에 저장하여 시계열 데이터 최적화를 도모합니다.

### 1-2. 패키지 구조와 계층 책임

**[코드 확인]** 패키지 구조는 도메인형(Domain-Driven) 구조와 인프라 분리를 혼합한 헥사고날 아키텍처(Ports and Adapters)의 과도기적 형태를 띄고 있습니다.

- **`domain` (비즈니스 로직)**
    - **역할:** 각 도메인(차량, 미션, 맵 등)별 Controller, Service, Repository, Entity, DTO를 응집도 있게 관리.
    - **대표 클래스:** `TowingCarService`, `MissionStateMachine`
    - **구조상 장점:** 도메인별 응집도가 높아 특정 기능 수정 시 영향 범위가 제한적임.
    - **구조상 아쉬운 점:** 엔티티(`Entity`)와 영속성(`Repository`)이 도메인 내부에 강하게 결합되어 있어 완전한 인프라 독립(헥사고날)을 이루지는 못함.
- **`infra` (외부 통신 어댑터)**
    - **역할:** 외부 프로토콜(MQTT, WebSocket)과의 통신 채널 설정 및 메시지 파싱.
    - **대표 클래스:** `MqttGateway`, `MqttMessageRoutingService`
    - **구조상 장점:** 도메인 로직이 특정 통신 기술(MQTT 등)에 종속되지 않도록 분리함.
    - **구조상 아쉬운 점:** 인프라에서 수신한 데이터를 다시 도메인 서비스로 넘기는 의존성 방향 정리가 완벽히 추상화되어 있진 않음.
- **`global` (공통 관심사)**
    - **역할:** Security(JWT), AOP(로깅, 성능 측정), Error(Global Exception), Config(Redis, DB, OpenAPI).
    - **대표 클래스:** `SecurityConfig`, `GlobalExceptionHandler`, `PerformanceAspect`

### 1-3. 핵심 기능 흐름 한눈에 보기

1. **차량 모니터링 (Telemetry):**
    
    차량 MQTT Publish → `MqttInboundHandler` 수신 → `MonitoringHandler` 파싱 → `TowingCarCacheService` (Redis 최신값 업데이트) → `DrivingLogRepository` (DB 저장) → `TowingCarWSController` (웹소켓 브로드캐스트)
    
2. **미션 상태 제어:**
    
    클라이언트 HTTP 요청 → `MissionController` → `MissionService` 검증 → `MissionStateMachine` 상태 전이 → `TowingCarMqttService` (차량으로 명령 MQTT Publish) → DB 반영 및 WS 전파.

## 2. 기능별 상세 분석

### [차량 실시간 상태 수신 및 모니터링 (Telemetry)]

1. **기능 목적:** 차량에서 초당 여러 번 들어오는 위치 및 상태 데이터를 처리하고 사용자에게 실시간 전파.
2. **관련 핵심 클래스:** `MonitoringMqttMessageHandler`, `TowingCarCacheService`, `TimescaleDBInitializer`
3. **요청 → 처리 → 저장/응답 흐름:** MQTT 수신 → JSON 파싱(`CarMonitoringPayloadParser`) → Redis 갱신 (차량 최신 상태) → DB 비동기/동기 저장 (시계열 데이터) → WebSocket 구독자에게 전파.
4. **사용 기술:** Spring Integration MQTT, Redis, TimescaleDB, STOMP.
5. **설계 의도:** 트래픽이 몰리는 상태 업데이트를 Redis로 캐싱하여 RDBMS 부하를 줄이고, 이력 데이터는 TimescaleDB의 Hypertable을 활용해 대용량 시계열 쓰기/조회 성능을 확보.
6. **장점:** 캐시와 영속성 DB의 역할을 완벽히 분리하여 성능 한계를 극복함.
7. **한계 / 개선점:** **[추정]** 차량 대수가 수백/수천 대로 늘어날 경우 건건이 DB에 Insert 하는 구조라면 병목이 생길 수 있음. (Batch Insert 고려 필요).
8. **면접에서 말할 핵심 한 줄:** "고빈도 IoT 센서 데이터를 처리하기 위해 최신 상태는 Redis로 서빙하고, 이력은 TimescaleDB로 분리하여 쓰기 성능과 조회 성능을 동시에 확보했습니다."
9. **과장 주의 포인트:** "초당 수만 건을 처리할 수 있다" (실제 부하테스트 결과값이 없다면 절대 금물).

### [MQTT 명령어 전송 및 ACK 보장 시스템]

1. **기능 목적:** 차량에 제어 명령(이동, 정지 등)을 내리고, 차량이 명령을 수신했는지 확인(ACK).
2. **관련 핵심 클래스:** `CommandAckService`, `AckHandler`, `MqttGateway`
3. **요청 → 처리 → 저장/응답 흐름:** 백엔드 제어 명령 발행 → `CommandAckService`에 상태 등록(대기) → 차량이 해당 명령 수행 후 ACK 토픽 발행 → 백엔드 `AckHandler` 수신 → `CommandAckService`에서 완료 처리.
4. **사용 기술:** CompletableFuture 또는 In-memory Map 상태 관리, MQTT.
5. **설계 의도:** 단순 QoS 1, 2 에 의존하지 않고, 애플리케이션 레벨에서 실제 차량이 명령을 받아 로직을 시작했음을 보장하기 위해.
6. **장점:** 네트워크 단절이나 차량 에러 시 타임아웃 처리가 가능해 시스템 신뢰성이 높음.
7. **한계 / 개선점:** 서버 다중화(Scale-out) 시 로컬 메모리에 대기중인 ACK 상태가 유실되거나 분산될 수 있음. (Redis 분산 락/캐시 기반 ACK 상태 관리로 고도화 필요).
8. **면접에서 말할 핵심 한 줄:** "불안정한 무선 환경을 고려해 MQTT QoS에만 의존하지 않고, 애플리케이션 레벨의 ACK-Wait 로직을 구현해 제어 신뢰성을 높였습니다."
9. **과장 주의 포인트:** "서버 다운에도 모든 메시지 유실을 100% 방지했다" (현재 코드는 서버 재시작 시 인메모리 대기열 증발 위험 존재).

### [미션 및 차량 상태 머신 (State Machine)]

1. **기능 목적:** 미션의 진행 상태(대기-할당-이동-견인-완료 등)와 차량의 상태(대기-주행-오류 등)를 안전하게 관리.
2. **관련 핵심 클래스:** `MissionStateMachine`, `TowingCarStateMachine`
3. **요청 → 처리 → 저장/응답 흐름:** 비즈니스 로직(할당, 취소 등) → StateMachine의 `canTransition()` 체크 → 유효할 경우 상태 변경 및 이벤트 발행.
4. **사용 기술:** Java 순수 객체 지향 설계 (if-else 최소화, 패턴 활용).
5. **설계 의도:** 복잡한 상태 변경 흐름에서 개발자 실수로 인한 비정상 상태 전이(예: 대기 상태가 아닌데 작업 할당)를 원천 차단하기 위함.
6. **장점:** 도메인 규칙을 한 곳에 캡슐화하여 응집도가 높고 테스트가 쉬움.
7. **한계 / 개선점:** 상태 전이 이력 자체가 엔티티로 완벽히 분리 관리되는지 확인 필요(Audit 로직).
8. **면접에서 말할 핵심 한 줄:** "복잡한 비즈니스 로직의 사이드 이펙트를 막기 위해, 도메인 레이어에 상태 머신(State Machine) 패턴을 적용하여 안전한 상태 전이만 허용하도록 설계했습니다."

## 3. Spring Boot 관점 분석

### [Spring Integration 기반 MQTT 처리]

1. **쓰인 방식:** `@ServiceActivator`, `MqttPahoMessageDrivenChannelAdapter`를 통해 Inbound/Outbound 채널 구성 (`config/MqttInboundConfig.java` 등).
2. **이유:** Spring Boot 환경에서 외부 MQTT 브로커(Mosquitto 등)와의 결합도를 낮추고 비동기 메시지 채널 구조를 우아하게 구성하기 위함.
3. **예상 질문:** "왜 Paho MQTT 클라이언트를 직접 쓰지 않고 Spring Integration을 사용했나요?"
4. **답변 포인트:** "메시지 라우팅, 채널링, 에러 핸들링을 프레임워크가 제공하는 추상화 레이어를 통해 처리함으로써, 도메인 로직과 통신 인프라 로직을 깔끔하게 분리할 수 있기 때문입니다."
5. **주의할 점:** 프레임워크 내부 스레드 풀 구조나 Channel 방식에 대해 깊이 묻는다면 헷갈릴 수 있으므로 `DirectChannel` 등 기본 개념 숙지 필요.

### [AOP 기반 로깅 및 성능 측정]

1. **쓰인 방식:** `global/aop/PerformanceAspect.java`, `LogExecutionTime` 커스텀 어노테이션 기반 타겟팅. 메서드 실행 전후 시간을 측정하여 로그 출력.
2. **이유:** 비즈니스 로직을 더럽히지 않고 특정 API나 메서드의 병목을 찾기 위한 관심사 분리.
3. **예상 질문:** "AOP를 활용해 성능 측정을 하셨는데, Proxy 내부 호출 시 동작하지 않는 문제는 어떻게 해결/이해하고 있나요?"
4. **답변 포인트:** "Spring AOP는 CGLIB/Dynamic Proxy 기반이므로 객체 내부 호출(Self-invocation) 시 프록시를 타지 않아 AOP가 적용되지 않음을 인지하고 있습니다. 컨트롤러 진입점이나 파사드(Facade) 레이어에 어노테이션을 부착해 이를 회피했습니다."

5. **답변 예시:** "기존 코드는 전혀 수정할 필요가 없습니다. `MqttTopicHandler` 인터페이스를 구현하는 새로운 Handler 클래스를 만들고 `@Component`로 등록만 하면, 라우팅 서비스가 다형성을 이용해 자동으로 매핑해줍니다."

## 5. DB / JPA / Redis 분석

### 5-1. Entity 구조 및 트랜잭션 경계

- **코드 확인:** 엔티티 설계 시 식별자와 주요 필드를 정의. 차량의 이력 관리를 위해 `DrivingLog`와 같은 히스토리 엔티티 별도 존재.
- **트랜잭션:** 서비스 계층에 `@Transactional`을 적용하여 작업 단위 분리. 읽기 전용 로직에는 `@Transactional(readOnly = true)`를 적절히 섞었는지 소스 상 디테일한 확인은 필요 **[추정 포함]**.
- **면접 포인트:** 단순 DB 저장 외에 차량의 상태를 바꾸면서 동시에 MQTT로 메시지를 발송해야 할 때 트랜잭션 처리가 중요함. (DB는 롤백되었는데 MQTT는 이미 발송된 경우 등).

### 5-5. 캐시와 DB 역할 분리 (핵심)

- **설계 해석:** 실시간 대시보드나 클라이언트에서 '현재 차량 위치'를 요청할 때마다 DB를 찌르면 심각한 부하 발생. 따라서 최신 상태는 Redis(`TowingCarCacheService`)에 덮어쓰기로 유지하고, RDBMS에는 로그를 적재하는 역할로 철저히 나눔.
- **운영상 이슈:** Redis 인메모리 특성상 서버 다운 시 실시간 위치 데이터 유실 가능성. (이때 DB의 가장 마지막 로그를 끌어와 복구하는 로직이 있는지 질문받을 수 있음).

### 5-7. TimescaleDB (PostgreSQL 기반) 활용

- **코드 확인:** `global/config/TimescaleDBInitializer.java` 존재. 앱 기동 시 혹은 스키마 초기화 시 `SELECT create_hypertable('driving_log', 'created_at');` 형태의 쿼리를 수행하는 것으로 **[코드 확인]**.
- **장점:** 일반적인 B-Tree 인덱스 기반 RDBMS는 시계열 데이터(차량 주행 로그)가 누적될수록 Insert/Select 속도가 급감함. TimescaleDB는 청크(Chunk) 단위 파티셔닝을 통해 최신 데이터 쓰기 및 조회에 극적인 성능 향상을 보장함.

## 10. 면접 대비 질문 리스트

### [백엔드 전체 구조 & 아키텍처]

**[질문] 이 프로젝트에서 가장 병목이 될 만한 구간은 어디였고 어떻게 해결했나요?**

1. **의도:** 시스템 아키텍처 관점의 트러블슈팅 능력 평가.
2. **핵심:** 빈번한 DB I/O를 Redis 캐싱과 TimescaleDB로 분산/해소한 경험.
3. **예시 답변:** "초당 수십 번 올라오는 차량 센서 데이터가 가장 큰 병목이었습니다. 이를 모두 RDBMS에 넣으면 락(Lock) 경합과 디스크 I/O로 서버가 터질 것이라 판단했습니다. 최신 상태는 메모리 기반 Redis에 저장해 프론트 서빙을 해결하고, 로그 보관은 시계열에 특화된 TimescaleDB로 이원화하여 해결했습니다."
4. **주의할 점:** "모든 데이터를 다 Redis에 넣었다"고 하면 OOM(Out of Memory) 공격 질문을 받음. (어떻게 TTL이나 영속화를 나눴는지 정확히 알아야 함).

### [Spring Boot / 인프라 (MQTT/WebSocket)]

**[질문] MQTT 통신에서 ACK 검증 로직을 별도로 구현한 이유가 뭔가요? QoS 1이나 2를 쓰면 되지 않나요?**

1. **의도:** 프로토콜에 대한 이해와 실무(도메인) 적합성 판단.
2. **핵심:** 네트워크 딜리버리 보장과 도메인 로직(실행) 보장의 차이 인지.
3. **예시 답변:** "MQTT QoS는 브로커와 클라이언트 간의 메시지 '도달'을 보장할 뿐입니다. 저희 관제 시스템에서는 차량이 메시지를 받았다는 것뿐만 아니라, **실제로 그 제어 명령(이동, 정지 등)을 하드웨어가 정상적으로 실행 시작했는지** 판단해야 했습니다. 그래서 애플리케이션 레벨의 로직 ACK를 별도로 구현했습니다."
