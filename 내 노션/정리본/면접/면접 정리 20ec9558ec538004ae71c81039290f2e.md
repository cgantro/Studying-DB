[버퍼 캐시 관리]
운영체제 수업 중 원형 연결리스트와 LRU 알고리즘을 적용해 순환 구조에서 오래된 데이터를 신속하게 찾아 교체하여, 버퍼 캐시를 관리하는 프로그램을 구현했습니다. 이를 통해 제한된 메모리 내에서 데이터를 순환 관리하는 방법과, 메모리 활용을 효율적으로 하는 방법을 익혔습니다.

        1. 구현하면서 어려웠던 점
            - 답변

        2. Queue Stack의 차이와 활용 예시
            
            스택은 후입선출(LIFO, Last In First Out) 구조로, 가장 나중에 들어온 데이터가 가장 먼저 나가는 특징이 있습니다. 이런 특성 때문에 함수 호출 시 함수의 실행 순서를 관리하거나 계산기에서 연산자 우선순위를 처리할 때 주로 사용됩니다.
            
            반면 큐는 선입선출(FIFO, First In First Out) 구조로, 먼저 들어온 데이터가 먼저 처리됩니다. 이 때문에 프로세스 스케줄링, 네트워크 메시지 큐 등에서 순서대로 작업을 처리하는 데 적합합니다.
            
    - LRU 쪽 질문
        1. 운영체제 수업에서**원형 연결리스트**과 lRU**알고리즘**을 활용해**버퍼 캐시 관리 프로그램**을 구현했다고 말씀하셨는데,이때**LRU 알고리즘**의 동작 원리를 아주 깊고 상세하게 설명해주세요.
            - 답변
                
                LRU 알고리즘은 **Least Recently Used**의 약자로, **가장 오랫동안 사용되지 않은 데이터를 캐시에서 우선적으로 교체**합니다.
                
                예를 들어, **캐시에서 접근(캐시 히트를) 한 페이지나 데이터**가 있을 경우 해당 데이터를 **가장 앞이나 최신로 옮깁니다**.
                
                그렇게 함으로써 **나중에 교체해야 할 대상**은 **가장 오랫동안 접근되지 않은 데이터**가 되며, 이는 캐시의 활용 효율을 더욱 상승시킬 수 있습니다.
                
                이런 방식을 통해 **캐시의 miss 빈도를 줄일 수고**, 성능을 개선할 수 있습니다.
                
        2. LRU를 구현할 때 **원형 연결리스트**를 선택하게 된 이유는 뭘까요?
            - 답변
                
                캐시에서 **데이터의 삽입이나 삭제가 빈번하게 발생하기 때문에 연결 리스트를 선택하게 되었고**,
                
                이 중에서도 **원형 연결 리스트**를 선택하게 된 이유는 **원형 연결 리스트가 순환 구조를 가지고 있어 캐시의 교체나 업데이트가 더욱 자연스러운 방식으로 이뤄지기 때문이다**고 생각했습니다.
                
                이러한 **순환 구조**는 **캐시에서 페이지를 한 바퀴 돌며 교체하거나 확인하기 쉽다는 장점**을 제공합니다.
                
                이로 인해 **원형 연결 리스트**는 **캐시의 LRU 구현에도 유리하게 활용되었습니다**.
                
                만일 원형 연결 리스트가 아니더라도 **원형 연결 큐가 캐시 히트** 
                
        3. 원형쓴 이유
            - 답변
                
                원형 연결 리스트는 **첫 번째 노드와 마지막 노드가 연결되어 있다는 점에서** 일반 연결 리스트보다 유리합니다.
                
                연결 리스트는**리스트의 맨 앞에서만 탐색을 시작해야 하지만**, 원형 연결 리스트는
                
                **끝과 앞이 연결되어 있어 어느 지점에서든 바로 탐색을 계속할 수 있습니다**.이로 인해 **탐색이나 접근이 더욱 유연하게 가능합니다.**
                
        4. 구현/관리가 복잡할 것 같은데?
            - 답변
                
                테스트 코드를 작성/주석을 꼼꼼하게 작성
                
                → **리스트의 상태가 변경되는 단계마다 검증로직이나 assert를 포함시켜 오류를 빠르게 확인.**
                
    - 메모리 관련 질문
        
        메모리 누수란 프로그램이 동적으로 할당한 메모리를 사용한 후에 적절히 해제하지 않아, 해제되지 않은 메모리가 계속 남아 시스템 자원이 점점 부족해지는 현상을 말합니다. 메모리 누수가 심해지면 프로그램이 점점 메모리를 많이 차지해 시스템 전체 성능 저하나 비정상 종료가 발생할 수 있습니다.
        
        메모리 누수를 방지하기 위해서는 할당한 메모리를 반드시 사용 후에 해제하는 습관을 들이는 것이 중요하며, 코드 리뷰와 함께 메모리 누수를 자동으로 탐지할 수 있는 도구(예: Valgrind, Visual Studio CRT Debug Library 등)를 활용하는 것이 효과적입니다.

### ✅ 1. 지원 동기 및 목표

1. SSAFY에 지원한 이유는 무엇인가요?
    - **답변**
        
        음악 장비를 사용하면서 임베디드 시스템에 관심을 가지게 되었습니다. SSAFY는 기초적인 소프트웨어 역량부터 시작해 임베디드 특화 트랙까지 이어지는 구조를 갖추고 있어, 경험이 부족한 저에게 가장 적합하다고 판단했습니다. 실전 프로젝트와 협업을 통해 구조적인 사고와 자원 효율 설계를 배워나가고 싶습니다.
        
2. SSAFY를 통해 어떤 개발자가 되고 싶나요?
    - **답변**
        
        하드웨어와 소프트웨어 양쪽을 이해하고 설계할 수 있는 개발자가 되고 싶습니다. 싸피는 이를 위한 기술적 기반을 다질 수 있는 최적의 환경이라고 생각합니다. 다양한 사람과 협업하며 여러 분야의 지식을 얻고 탐구해나가며 제 목표를 달성해나가겠습니다.
        
        키워드 : 하드웨어/소프트웨어 이해
        
3. SSAFY를 수료한 후의 진로 계획은 어떻게 되나요?
    - **답변**
        - 음악 산업과 임베디드를 융합한 스마트 음악 장비 개발자로 성장하는 것입니다.
        - SSAFY 수료 후에는 스마트 스피커, 이펙터 등 **음악장비을 개발하는 임베디드 개발자**로 성장하고 싶습니다..
        
        키워드 : → 내 목표인 음악 관련 산업에 기여(**공연 연출** 등)
        
4. 다른 교육기관이 아닌 SSAFY를 선택한 이유는 무엇인가요?
    - 답변
        
        다양한 분야의 협업 + 지원금 → 학습에만 몰두할 수 있는 환경
        
5. SSAFY 교육과정을 1년 동안 성실히 이수할 자신이 있나요?
    - 답변
        
        당연 → 만약 임베디드 분반에 배정되지 않고/ 다른 분야에 배정되더라도 제 목표를 위한 능력을 기르는 데에 필요하다고 생각함 → 따라서 어느 상황에서도 성실히 이수할 것.
        
        또한 1년은 짧은 기간이 아닙니다. 장기적인 체력을 기르기 위해 SSAFY 동료들과 운동 소모임을 만들어 병행하겠습니다.

### ✅ 3. 협업 및 커뮤니케이션

1. 팀 프로젝트에서 갈등이 생긴다면 어떻게 해결하시겠습니까?
    - 답변
        
        ---
        
        객관적 근거 + 현재 상황에 맞춘 선택 에 기반하여 갈등 경험이 있습니다
        
        팀 프로젝트에서 **쿠키 기반 인증과 헤더 기반 인증 방식** 중 어떤 걸 사용할지 의견 차이가 있었습니다.
        
        **쿠키 기반 인증**   
        
        **JWT 토큰을 헤더**
        
        우선 각 방식의 장단점을 정리해 팀원들과 공유했습니다.
        
        - 쿠키 기반 인증은 자동으로 브라우저가 쿠키를 관리해주고, HttpOnly, Secure 옵션으로 보안 강화가 가능하지만, CSRF 공격에 취약할 수 있습니다.
        - 헤더 기반 인증(JWT)은 CSRF 공격에 비교적 자유롭고, 모바일 앱이나 다양한 클라이언트에서도 일관되게 사용할 수 있지만, 토큰 관리와 만료 처리 등 구현이 더 복잡할 수 있습니다.
        
        그 다음에는 서비스 환경을 고려해 어느 쪽이 더 적합한지 논의했습니다. 
        
        성능이 좋지 못한 서버를 사용했기에, 최대한 서버에 부담이 덜 가는 방향으로 타협을 했다. → 헤더 기반 인증 
        
        이 과정에서 **객관적인 근거를 바탕으로 열린 소통을 하고, 타협점을 찾아가는 경험**을 쌓을 수 있었습니다.
        
        - 쿠키 기반 인증
            - **개념**
                
                서버가 클라이언트에 `Set-Cookie` 헤더를 통해 세션 ID 또는 인증 토큰을 쿠키로 전달
                
                이후 클라이언트는 자동으로 HTTP 요청 시 쿠키를 포함해 서버에 인증 정보 전달
                
            - **장점**
                - 브라우저가 쿠키를 자동으로 관리(저장/전송)해 개발 편리
                - `HttpOnly`, `Secure` 플래그로 쿠키 보호 가능
                - 서버에서 세션 상태 관리 가능 (서버 세션 스토리지 필요)
                - SameSite 옵션으로 CSRF 공격 완화 가능
            - **단점**
                - CSRF(Cross-Site Request Forgery) 공격에 취약 (특히 SameSite 설정이 없거나 느슨할 때)
                - 쿠키는 도메인에 종속적이고, 모바일 앱이나 비브라우저 클라이언트에 적용하기 어려움
                - 서버에 세션 정보를 저장해야 하므로 서버 부하 증가 가능
            - **중요 설정**
                - `HttpOnly`: JS 접근 불가 (XSS 방어)
                - `Secure`: HTTPS 연결에서만 전송
                - `SameSite`: `Strict`, `Lax`, `None` 설정 가능
                    - `SameSite=None`은 반드시 `Secure`(HTTPS) 필요
                - 예:
                    
                    ```java
                    
                    Set-Cookie: sessionId=abc123; HttpOnly; Secure; SameSite=Lax
                    
                    ```
                    
        - 헤더 기반 인증
            - **개념**
                
                JSON Web Token (JWT)을 서버가 생성해 클라이언트에 전달
                
                클라이언트는 이후 요청 시 `Authorization: Bearer <token>` 헤더에 JWT를 포함해 전송
                
            - **장점**
                - CSRF 공격에 상대적으로 안전 (쿠키 자동 전송과 다르게 명시적으로 헤더에 넣어 전송)
                - 무상태(stateless) 인증 가능: 서버에 별도의 세션 저장 불필요
                - 모바일, SPA, 다양한 클라이언트에서 동일한 인증 방식 사용 가능
                - 토큰 페이로드에 사용자 정보, 권한 등 담아 추가 인증 정보 활용 가능
            - **단점**
                - 토큰 탈취 시 장기간 악용 위험 (토큰 만료, 재발급 관리 필요)
                - 토큰 저장 위치 문제 (LocalStorage는 XSS 공격에 취약, 쿠키는 CSRF 위험)
                - 구현 복잡도 증가 (토큰 갱신, 만료 처리, 블랙리스트 등 관리 필요)
            - **JWT 구조**
                - Header: 토큰 타입, 서명 알고리즘 정보
                - Payload: 클레임(사용자 ID, 권한, 만료시간 등)
                - Signature: 비밀키로 서명해 변조 방지
            - **인증 흐름 예시**
                1. 로그인 성공 시 서버가 JWT 생성 후 클라이언트 전달
                2. 클라이언트는 JWT를 저장(LocalStorage, 쿠키, 메모리 등)
                3. 요청 시 `Authorization` 헤더에 JWT 포함
                4. 서버는 JWT 유효성 검사 후 요청 처리
            
        - JWT 란?
            
            **JWT + 액세스 토큰 + 리프레쉬 토큰**의 역할과 처리 방안을 한데 모아 깔끔하게 요약할게요.
            
            ---
            
            ## 🔑 JWT + Access Token + Refresh Token 요약
            
            |  | **Access Token** | **Refresh Token** |
            | --- | --- | --- |
            | 용도 | API 접근시 권한 확인 | 만료된 Access Token을 재발급 |
            | 만료 | 일반적으로 아주 짧고 (15분~30분) | 길고 (일~주, 심지어 한달) |
            | 내용 | 회원 정보(id, roles...) 포함 | 회원 식별 정보(id)가 포함되어 있음 |
            | 저장 | 클라이언트의 메모리, 상태관리 | HttpOnly, SameSite, Secure Cookie |
            | 전송 | API마다 `Authorization:Bearer …`로 전달 | 자동 전달(쿠키) 또는 별도의 API로 전달 |
            | 보안 | 만료가 짧고 정보가 한정되어 있어 탈취시 피해가 적은편 | 만료가 길고 계정 탈취로 연결 가능 → 엄격하게 보호해야함 |
            | 만료시 | 만료 -> Refresh로 신규 발급 | 만료/유효 -> 신규 Access Token 발급 |
            
            ---
            
            ## 🔑 처리 흐름 (일반적으로)
            
            1️⃣ **회원 가입/로그인시**
            
            - Access Token + Refresh Token 발급
            - Refresh Token은 HttpOnly-Secure-SameSite-Strict 쿠키로 전달
            - Access Token은 API 요청시 전달 (Authorization)
            
            2️⃣ **API 접근시**
            
            - Access Token 확인 → 만료 아니고, 서명과 권한이 유효하다면 접근 허용
            - 만료시 401 Unauthorized 응답
            
            ---
            
            ## 🔑 Access Token 만료시 (리프레쉬)
            
            1️⃣ 클라이언트는 refresh API로 refresh_token 전달 (자동로 cookies 포함)
            
            2️⃣ 서버는 refresh_token의 유효성과 만료, 상태 등을 확인
            
            (DB 또는 Blacklist 확인)
            
            3️⃣ 새로운 Access Token 발급
            
            4️⃣ (선택) Refresh Token도 재생성 (Sliding Sessions)
            
            ---
            
            ## 🔑 로그아웃시, 계정 탈취시
            
            ✅ 해당 계정과 연결되어 있는 refresh_token을 무효화 (DB is_revoked 변경)
            
            ✅ 클라이언트를 통해 refresh_token을 삭제 (만료일 0로 세팅)
            
            ✅ Access Token은 만료가 짧으니 그냥 만료를 기다리면 된다
            
            ✅ 계정 탈취시 refresh_token을 모두 무효화함으로써 새로운 Access Token 발급을 막을 수 있음
            
            ---
            
            ## 🔑 보안 권장사항
            
            ✅ Refresh Token : HttpOnly, SameSite, Secure
            
            ✅ Access Token : 만료를 아주 짧게
            
            ✅ 계정 탈취시 refresh_token 무효화
            
            ✅ jti(id)가 포함되어 있어 개별 무효화 가능
            
            ✅ Blacklist, Database로 refresh_token 상태 확인
            
            ---
            
            ✨ 만일:
            
            ✅ C++로 어떻게 구현해야 할지
            
            ✅ Spring Security에서 Filter, Exception Handling, UserDetails, SecurityContext과 연결하기
            
            ✅ Redis Blacklist, Database Schema
            
            ✅ JWT 생성, 파서, 검증, 만료 처리
            
            ✅ Exception Handling, Security Filter Chain, 테스트 방법
            
        - 보안 측면
            - 토큰 탈취 방지 방안
                
                **HTTPS 사용**, **토큰 만료 시간 설정**, **토큰 저장 위치의 신중한 선택(예: HttpOnly 쿠키 사용)**, **리프레시 토큰과 액세스 토큰 분리**
                
            - HTTPS와 HTTP의 차이점은 무엇이고, 왜 HTTPS를 사용하는 것이 중요한가요?
                - **HTTP는 암호화하지 않은 평문 통신이다**
                - **HTTPS는 암호화된 통신을 지원한다** — TLS(Transport Layer Security)를 통해 데이터를 암호화해 안전하게 전송합니다. ‘SSL’ 보다는 TLS가 정확한 명칭이에요.
                - **인증서 역할** — 인증서는 서버의 신원을 검증하고, 클라이언트와 서버 간에 안전한 암호화 통신 채널을 만드는 데 사용됩니다.
                - **왜 중요한가?** — HTTPS를 사용하면 중간자 공격, 데이터 탈취, 변조 등을 방지할 수 있어 사용자 정보 보호에 필수적입니다.

### ✅ 4. 자기 이해 및 태도

    - 답변

        > 
        > 
        > 이 과정에서 저는 **창의성과 협업, 기술적 완성도를 동시에 끌어올리는 경험**을 했고,
        > 
        > 특히 음악을 구조적으로 다루고 쌓아나가는 과정 속에서 자연스럽게 **음향과 시스템 간 연결**, 나아가 **임베디드 시스템과의 접점**에 흥미를 느끼게 되었습니다.
        > 
        > SSAFY에서 이런 도전 정신을 활용해 **실제 시스템 구현 능력으로 발전시키며 실시간 음향 제어 시스템과 같은 실제 상황에서도 유용한 제품을 만들고 싶습니다.**
        > 
        > 키워드 : 자작곡 → 창의성, 협업, 완성도 → 이런 도전 정신 → 싸피에서도 해보지 못한 경험을 하며 실제 상황에서도 유용한 제품을 만들고 싶음
        > 
        > 저의 단점은 때때로 직설적인 피드백을 하는 것입니다. 
        > 
        > 밴드 합주를 할 때 피드백이 너무 직설적이어서 분위기를 해칠 수 있다는 지적을 받았습니다. 이를 개선하기 위해 최근 1년간 공부방에서 초등학생과 중학생을 가르치면서, 학생 개개인의 성향에 맞는 피드백의 중요성을 배웠습니다. 예를 들어, 학원에서 소심하고 말이 없는 아이에게는 직접적인 질문을 피하고, 아이가 문제를 풀 때 그 사고 과정을 질문하는 방식으로 접근했습니다. 그 아이는 점차 자신감을 얻고 문제를 풀 때 더 적극적으로 의견을 표현하게 되었습니다. 이 경험을 통해, 사람마다 다른 성향을 이해하고, 그에 맞춰 적절한 피드백을 주는 것이 얼마나 중요한지를 깨달았습니다.”
        > 
        > 키워드 → 직설적 → 학원 근무 → 개개인 성향에 맞는 피드백/칭찬 →
        > 
2. 주변 사람들은 본인을 어떤 사람이라고 하나요?
    - 답변
        
        2년 전만 하더라도 주변 동아리원들은 저를 무섭다고 생각했다. 왜냐하면 앞서 말한 직설적인 표현 때문이다.
        
        그러나 방금 말했듯이, 학원일도 하고 동아리 활동 연차가 쌓이면서 개개인의 성향에 맞는 접근법을 알아가려노 노력했다.
        
        따라서 최근에 축제 대회를 준비할 때, 따뜻한 사람이란 칭찬을 받은적이 있다. 정말 기분 좋았다.
        
3. 최근 스스로 성장했다고 느낀 순간은 언제인가요?
    - 답변
        
        최근 축제 준비를 하며 개개인 성향 및 실력에 따른 **연습 가이드/피드백**을 제시 후 **팀원들의 실력이 향상된 것**을 발견했다.
        
        → 인간적/기술적으로 성장했다고 느꼈다.
        
4. 실패를 경험한 적이 있다면, 그때 어떻게 대처했나요?
    - **답변**
        
        개발 경험? 내 개발 경험과 연관지어야겠다.
        
5. 스트레스를 받을 때 어떻게 해소하나요?
    - 답변
        
        음악과 야구를 좋아한다. 베이스나 피아노를 연주하거나 합주를하고 캐치볼하면서 공을 던지다 보면 긍정적인 에너지가 생긴다.
