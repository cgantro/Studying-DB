# Spring boot

## Filter

Java Servlet 요청과 응답을 가로채는 역할

체이닝, MVC와의 독립성

```java
@WebFilter(urlPatterns = "/*") // 모든 URL에 적용
public class MySimpleFilter implements Filter {

    @Override
    public void init(FilterConfig filterConfig) throws ServletException {
        System.out.println("MySimpleFilter 초기화!");
    }

    @Override
    public void doFilter(ServletRequest request, ServletResponse response, FilterChain chain)
            throws IOException, ServletException {
        HttpServletRequest httpRequest = (HttpServletRequest) request;
        System.out.println("들어오는 요청 URL: " + httpRequest.getRequestURL());

        // 전처리
        // ...

        // 체인의 다음 필터 또는 서블릿으로 요청 전달
        chain.doFilter(request, response);

        // 후처리 (서블릿이 요청을 처리한 후)
        // ...
        System.out.println("나가는 응답 URL: " + httpRequest.getRequestURL());
    }

    @Override
    public void destroy() {
        System.out.println("MySimpleFilter 소멸!");
    }
}
```

## 인터셉터

### Interceptor (인터셉터)

인터셉터는 Spring MVC 프레임워크의 일부입니다. 요청이 서블릿 필터를 통과한 **후**에, 하지만 Spring MVC 컨트롤러 메서드에 도달하기 **전**에 요청을 가로챕니다. 이는 Spring MVC 요청 라이프사이클에 대한 더 세밀한 제어를 제공합니다.

컨텍스트, 빈, 핸들러 정보에 접근 가능

세가지 지점:

- `preHandle()`: 핸들러 메서드 실행 전에 실행됩니다. `true` (계속 처리) 또는 `false` (처리 중단)를 반환할 수 있습니다.
- `postHandle()`: 핸들러 메서드 실행 *후*에 실행되지만, 뷰가 렌더링되기 *전*입니다. `ModelAndView`를 수정할 수 있습니다.
- `afterCompletion()`: 뷰 렌더링을 포함하여 요청이 완료된 *후*에 실행됩니다. 리소스 정리 등에 유용합니다.

| 특징 | 필터 (Filter) | 인터셉터 (Interceptor) | 인증 필터 (Authentication Filter) |
| --- | --- | --- | --- |
| **레벨** | 서블릿 컨테이너 | Spring MVC (`DispatcherServlet`) | 서블릿 컨테이너 (종종 Spring Security의 일부) |
| **API** | `javax.servlet.Filter` | `org.springframework.web.servlet.HandlerInterceptor` | `javax.servlet.Filter` (특정 목적) |
| **Spring 인식** | 기본적으로 Spring을 인식하지 못함 | Spring을 인식함 (Spring 빈에 접근 가능) | Spring을 인식함 (Spring Security의 일부인 경우) |
| **실행 시점** | `DispatcherServlet` 이전 (전처리 및 후처리) | `DispatcherServlet` 이후, 컨트롤러 이전 (전처리/후처리/완료 후) | `DispatcherServlet` 이전 (인증에 특화됨) |
| **사용 사례** | 광범위한 관심사 (인코딩, 모든 요청 로깅, 기본 보안) | 세밀한 Spring MVC 특정 관심사 (컨트롤러 로깅, 성능, 특정 권한 부여) | 인증 처리 전용 (자격 증명 추출, 유효성 검사, 보안 컨텍스트 설정) |
- 면접 예상 질문 들

    ### Spring Boot란?
    
    - Spring Framework 기반의 어플리케이션을 더욱 쉽고 빠르게 만들 수 있도록 만든 **자동설정**, **내장서버(Tomcat)**, **의견주도**, **starter** 등을 포함한 프레임워크

    ### 4️⃣ Spring Beans의 생성과 관리
    
    - Spring Container가 Singleton 빈을 만든 후, DI로 연결
    - `@Component, @Service, @Repository, @Controller, @RestController, @Bean` 등을 통해 Spring Container에 등록
    - IOC(Inversion of Control)로 객체의 생성과 연결을 Spring Container에게 위임함
        
        > 꼬리질문: IOC의 역할과 DI의 장점?
        > 
        > - 답변
        >     
        >     ## IOC (Inversion of Control) 역할
        >     
        >     - **제어의 역전**이라고 번역됩니다.
        >     - 전통적인 프로그래밍에서는 객체가 자신이 사용할 의존 객체를 직접 생성하거나 관리합니다.
        >     - IOC는 이 제어 권한을 객체 자신이 아닌 외부(컨테이너, 프레임워크 등)가 가져가도록 합니다.
        >     - 즉, 객체가 필요한 의존성을 외부에서 주입해 줌으로써, 객체는 자신의 핵심 역할에 집중할 수 있습니다.
        >     - 주로 프레임워크(예: Spring)에서 객체 생성과 생명주기 관리, 의존성 주입을 담당합니다.
        >     
        >     **IOC 역할 정리:**
        >     
        >     - 객체 생성과 관리 책임을 외부에 위임
        >     - 객체 간 결합도(의존성) 낮추기
        >     - 모듈 간 유연한 교체 및 테스트 용이성 증가
        >     
        >     ---
        >     
        >     ## DI (Dependency Injection)의 장점
        >     
        >     - DI는 IOC의 구체적인 구현 방법 중 하나입니다.
        >     - 필요한 객체(의존성)를 직접 생성하지 않고 외부에서 주입(Injection)받는 방식입니다.
        >     
        >     **DI의 대표적인 장점은 다음과 같습니다:**
        >     
        >     1. **결합도 감소**
        >         - 객체가 직접 의존성을 생성하지 않아, 객체 간 결합도가 낮아집니다.
        >     2. **유지보수 및 확장성 향상**
        >         - 의존성 주입을 통해 객체 교체가 쉽고, 코드 수정 없이 기능 확장이 가능합니다.
        >     3. **테스트 편의성 증가**
        >         - Mock 객체나 테스트용 객체를 주입하여 단위 테스트가 쉬워집니다.
        >     4. **코드의 명확성과 재사용성**
        >         - 의존성이 명시적으로 드러나므로 코드 이해가 쉬워지고, 재사용성이 높아집니다.

    ### 6️⃣ @RestController vs @Controller
    
    - **@RestController**: API로 직접 데이터를 반환 (@Controller + @ResponseBody)
    - **@Controller**: View 이름을 반환 -> ViewResolver로 연결
    
    > 꼬리질문: ViewResolver의 역할과 Thymeleaf의 역할?
    > 
    > - 답변
    >     
    >     ## 🔹 ViewResolver의 역할 🔹
    >     
    >     ViewResolver는 **컨트를러에서 반환되는 View 이름을 어떻게 View로 연결할 것인지 결정합니다**.
    >     
    >     - *컨트를러에서 `return "home"`*이라는 View 이름을 넘겼다면,
    >     - ViewResolver는 이 이름과 미리 정의된 규칙(ex: prefix, suffix)가 연결되어서
    >     - 실제 View로 forward합니다.
    >     
    >     ```java
    >     
    >     // 예시 (Spring)
    >     return "home";
    >     // ViewResolver가 "/templates/home.html"로 연결
    >     
    >     ```
    >     
    >     ---
    >     
    >     ## 🔹 Thymeleaf의 역할 🔹
    >     
    >     Thymeleaf는 **View 계층에서 HTML을 동적으로 생성하기 위해 사용하는 템플릿 엔진**입니다.
    >     
    >     - HTML 내부에서 Thymeleaf의 문법(ex: `th:text`)을 사용할 수 있습니다.
    >     - 모델에서 전달되는 데이터를 이 문법을 통해 View에 풀어줍니다.
    >     
    >     ```html
    >     
    >     <p th:text="${msg}">Hello</p>
    >     
    >     ```
    >     
    >     → 모델에서 `msg`이라는 이름의 데이터가 전달되어 있고
    >     
    >     → 이 값을 HTML에 동적으로 넣어줍니다.
    >     
    >     ---
    >     
    >     ## 🔹 Spring MVC에서의 연결 🔹
    >     
    >     일반적으로 Spring Boot에서:
    >     
    >     ```java
    >     
    >     @EnableSpringBootApplication
    >     @Controller
    >     public class HelloController {
    >     
    >         @GetMapping("/hello")
    >         public String hello(Model model) {
    >             model.addAttribute("msg", "안녕하세요.");
    >             return "hello";
    >         }
    >     }
    >     
    >     ```
    >     
    >     → ViewResolver가 `return "hello"`를 받고
    >     
    >     → Thymeleaf가 `src/main/resources/templates/hello.html`이라는 View를 찾아서
    >     
    >     → 모델과 합쳐서 결과 HTML을 만든 후
    >     
    >     → 클라이언트를 통해 전달합니다.
    >

    ### 7️⃣ Spring MVC의 DispatcherServlet 역할
    
    - Spring MVC의 **Front Controller**
    - 클라이언트의 요청을 받고 해당 **Handler Mapping**이나 **Controller**로 전달
    - 결과를 View로 전달하거나 바로 API로 응답
    
    > 꼬리질문: DispatcherServlet의 서블릿 라이프사이클?
    > 
    > - 답변
    >     
    >     `DispatcherServlet`은 Spring MVC의 핵심 서블릿으로, 클라이언트 요청을 처리하는 중앙 컨트롤러 역할을 합니다. 서블릿 라이프사이클은 일반 서블릿과 비슷하지만, Spring MVC 동작에 맞게 특화되어 있습니다.
    >     
    >     1. **서블릿 초기화 (init)**
    >         - WAS가 시작되거나 최초 요청 시 `DispatcherServlet`이 생성되고 초기화됩니다.
    >         - `web.xml` 또는 Spring Boot에서는 자동으로 서블릿이 등록됨
    >         - `init()` 메서드 호출 시 Spring `WebApplicationContext`를 생성하고, 컨트롤러, 핸들러 매핑, 뷰 리졸버 등 관련 빈을 로딩함
    >     2. **요청 처리 (service / doGet, doPost)**
    >         - 클라이언트 요청이 오면 `DispatcherServlet`의 `service()`가 호출됨
    >         - 요청을 핸들러 매핑에서 알맞은 컨트롤러로 전달 (핸들러 탐색)
    >         - 컨트롤러 실행 후 모델과 뷰 정보를 반환받음
    >         - 뷰 리졸버를 통해 실제 뷰(JSP, Thymeleaf 등)를 렌더링함
    >     3. **서블릿 종료 (destroy)**
    >         - WAS가 종료되거나 서블릿이 언로드될 때 호출됨
    >         - 관련 리소스 정리 및 종료 작업 수행

    ### 9️⃣ Spring Data JPA
    
    - JpaRepository 인터페이스 상속만으로 CRUD 가능
    - 쿼리메소드로 간단하게 데이터 접근 가능(ex: findByUsername)
    - JPQL, NativeQuery, QueryDSL과의 연결 가능
    - 영속성 컨텍스트, 변경 감지, 지연로딩, N+1문제 등을 고려해야 함
    
    > 꼬리질문: 영속성 컨텍스트의 역할과 변경 감지의 원리?
    > 
    > - 답변
    >     - **영속성 컨텍스트(Persistence Context)**
    >         - 엔티티를 1차 캐시에 저장하는 공간으로, 같은 트랜잭션 내에서 엔티티 객체의 일관성을 유지하고 관리합니다.
    >         - DB와 엔티티 객체 간의 상태를 동기화하는 중간 관리자로 작동하며, 같은 엔티티 조회 시 매번 DB에 가지 않고 캐시된 객체를 반환해 성능을 향상시킵니다.
    >     - **변경 감지(Dirty Checking)**
    >         - 영속성 컨텍스트가 관리하는 엔티티 객체의 상태 변화를 감지하는 메커니즘입니다.
    >         - 트랜잭션이 커밋되거나 `flush()` 호출 시점에 영속성 컨텍스트가 엔티티의 현재 상태와 처음 로딩된 상태를 비교합니다.
    >         - 변경된 필드가 있으면 자동으로 해당 변경 내용을 반영하는 SQL `UPDATE` 쿼리를 생성해 DB에 반영합니다.
    >         - 개발자가 명시적으로 수정 쿼리를 작성하지 않아도 되는 장점이 있습니다.

    ### 11️⃣ Spring Security의 역할
    
    - Spring 어플리케이션의 인증(authentication)과 **권한**(authorization)을 처리합니다.
    - Filter Chain과 Security Filter로 요청마다 검사합니다.
    - 커스터마이즈로 UserDetails, AuthenticationProvider 등을 사용할 수 있습니다.
    
    > 꼬리질문: UserDetailsService의 역할과 AuthenticationProvider의 역할?
    > 
    > - 답변
    >     - **UserDetailsService**
    >         - Spring Security에서 사용자 정보를 조회하는 인터페이스입니다.
    >         - 주로 데이터베이스 등에서 사용자 정보를 로드(load)하여 `UserDetails` 객체로 반환합니다.
    >         - 인증 과정에서 사용자의 아이디(또는 사용자명)를 받아서 사용자 정보를 찾아내는 역할을 담당합니다.
    >     - **AuthenticationProvider**
    >         - 실제 인증(authentication) 로직을 처리하는 인터페이스입니다.
    >         - `Authentication` 객체를 받아서 인증 절차를 수행하고, 인증이 성공하면 인증된 `Authentication` 객체를 반환합니다.
    >         - 인증 정보 검증(비밀번호 확인, 토큰 검사 등)을 담당하며, 여러 인증 방식을 지원하기 위해 커스터마이징할 수 있습니다.

    ### 12️⃣ Filter Chain의 역할 (Spring Security)
    
    - DispatcherServlet보다 **우선적으로** 요청을 인터셉트를 수행합니다.
    - Security Filter Chain에서 CSRF, CORS, Authentication 등을 확인합니다.
    
    > 꼬리질문: Filter Chain과 DispatcherServlet의 실행순서?
    > 
    > - 답변
    >     - 클라이언트 요청 →
    >     - Filter Chain 내의 필터들 순차 실행 →
    >     - DispatcherServlet 실행 →
    >     - 컨트롤러 및 뷰 처리 →
    >     - 응답 반환 (DispatcherServlet → Filter Chain → 클라이언트)

    ### 13️⃣ CSRF 보호
    
    - Spring Security가 **자동 활성화**합니다.
    - CSRF Token이라는 임시 토큰을 만들어서 **POST, PUT, PATCH, DELETE** 요청마다 포함되어 있어야합니다.
    
    > 꼬리질문: CSRF Token을 비우거나 무시해야 할 수 있는 API는?
    > 
    > - 답변
    >     - **비회원용 공개 API**
    >         - 인증 없이 공개적으로 접근 가능한 API (예: 공공 데이터 조회, 이미지 다운로드 등)
    >         - CSRF 보호가 필요 없고, 인증 정보도 없기 때문에 토큰 검사 생략 가능
    >     - **RESTful API / SPA 백엔드에서 토큰 기반 인증 사용 시**
    >         - JWT 등 헤더 기반 인증을 사용하는 API는 CSRF 공격 위험이 적어 토큰 검사를 비활성화하는 경우가 많음
    >     - **외부 시스템과 통신하는 API**
    >         - 서버 간 통신용 API로, 브라우저를 거치지 않는 요청
    >         - CSRF 공격 대상이 아니므로 무시 가능

    ### 14️⃣ CORS
    
    - CORS는 **서로 다른 도메인끼리 리소스를 사용할 수 있도록 허용하거나 제한**합니다.
    - Spring에서 `@CrossOrigin` 또는 `CorsConfigurationSource`로 CORS를 세팅합니다.
    
    > 꼬리질문: CORS의 Preflight 요청(OPTIONS)은 왜 발생할까요?
    > 
    > - 답변
    >     - **브라우저가 실제 요청 전에 서버에 허용 여부를 확인하기 위해 보냄**
    >     - 특히, **“비단순 요청”(non-simple request)** 에 대해 발생
    >         - 예: `Content-Type`이 `application/json` 같은 특정 타입일 때,
    >         - 커스텀 헤더를 사용할 때,
    >         - HTTP 메서드가 `GET`, `POST`, `HEAD`가 아닌 경우 (`PUT`, `DELETE` 등)
    >     - **서버가 CORS 요청을 허용하는지 판단하고, 실제 요청 전에 보안을 강화하기 위함**
    >     - Preflight 요청은 `OPTIONS` 메서드로 보내지고,
    >         - 서버는 `Access-Control-Allow-Origin`, `Access-Control-Allow-Methods`, `Access-Control-Allow-Headers` 등을 응답 헤더로 보내서 권한을 알려줌
    >     - 만약 서버가 허용하지 않으면 실제 요청은 차단됨

    ### 28️⃣ Spring Boot와 Tomcat
    
    - 내장 톰캣 서버를 내장해 별도 WAS 설치 없이 실행 가능
    - `spring-boot-starter-web` 의존 시 자동 포함
    - `application.properties`로 포트, 스레드 수 등 설정 가능
    
    > 꼬리질문: 톰캣을 내장하는 것과 독립 서버로 운영하는 것 차이?
    > 
    > - 답변
    >     
    >     ## 🔹 내장 Tomcat 🔹 (Spring Boot)
    >     
    >     ✅ Spring Boot 어플리케이션 안에 Tomcat이라는 웹서버가 포함되어 있음
    >     
    >     ✅ `java -jar 어플리케이션.jar`만 실행하면 바로 웹서버가 올라온다는 의미
    >     
    >     ✅ 별도의 Tomcat 설치, 관리가 필요없고, 아주 간편하게 사용할 수 있음
    >     
    >     ✅ 빠릇하게 테스트하거나, microservice, container로 사용할때 유리
    >     
    >     ✅ Spring Boot의 `starter-web` 의존성을 포함하면서 Tomcat을 같이 가져온 것
    >     
    >     ✅ `port` 변경이나 `thread pool` 변경 같은 세부 세팅 역시 Spring Application properties에서 가능
    >     
    >     (ex: `application.yaml`)
    >     
    >     ```yaml
    >     yaml
    >     복사편집
    >     server:
    >       port: 8080
    >       tomcat:
    >         max-threads: 200
    >     
    >     ```
    >     
    >     ---
    >     
    >     ## 🔹 독립 Tomcat 🔹 (External Tomcat)
    >     
    >     ✅ Tomcat이라는 웹서버를 별도로 설치/관리함
    >     
    >     ✅ Spring 어플리케이션을 **WAR**로 만든 후 Tomcat의 `webapps` 폴더에 넣고 Tomcat이라는 별도의 프로세스에서 실행
    >     
    >     ✅ 운영이나 리소스, 성능, 보안 등을 더욱 세부적으로 Tomcat 수준에서 커스터마이즈 가능
    >     
    >     ✅ 기존의 Tomcat 관리 인력이나 운영 체계가 존재하거나, 어플리케이션마다 Tomcat을 별도로 풀려서 운영해야 할 경우 유리
    >     
    >     ✅ 이는 Spring Framework의 고전 방식이었다고 생각할 수 있음
    >     
    >     ```bash
    >     bash
    >     복사편집
    >     $ CATALINA_HOME/bin/catalina.sh start
    >     # 그리고 war deploy
    >     
    >     ```
    >     
    >     ---
    >     
    >     ## 🔹 요약 🔹
    >     
    >     |  | **내장 Tomcat** | **독립 Tomcat** |
    >     | --- | --- | --- |
    >     | 배포 | Spring Boot jar 하나로 가능 | WAR로 만든 후 Tomcat에 배포 |
    >     | 서버관리 | Spring Application과 한데 포함 | 별도의 Tomcat 인스턴스 |
    >     | 초기구동 | java -jar로 바로 가능 | Tomcat 별도의 start |
    >     | 환경 | Docker, microservice 친화 | 전통적 어플리케이션, 운영 환경과 밀접 |
    >     | 유지보수 | 일괄/자동 | 별도로 Tomcat과 어플리케이션 관리 |
    >     | 유연성 | Spring properties에서 세부 세팅 가능 | Tomcat server.xml에서 세부 세팅 가능 |
    >     
    >     ---
    >     
    >     ✨ 요약:
    >     
    >     내장 Tomcat은 간단하게 어플리케이션과 한데 포함되어 있어 **배포, 테스트, 운영이 쉽고**,
    >     
    >     독립 Tomcat은 **보다 세부적으로 커스터마이즈하거나 기존 운영 환경과 연결하기 쉽다는 장점**을 가지고 있습니다.
