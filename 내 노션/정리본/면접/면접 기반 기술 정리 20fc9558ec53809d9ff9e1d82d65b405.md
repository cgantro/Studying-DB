    1. **LRU 알고리즘**
        - **LRU**
            - 간단하면서 효율적
            - 시간 지역성
            - 메모리 & 디스크 블록이 제한적일 때 강함
            
            Clock(Seceond chance) 알고리즘 → Referece bit 0이면 없애버림, 1이면 살림
            
            | 알고리즘 | 기준 | 장점 | 단점 | 예시 |
            | --- | --- | --- | --- | --- |
            | LRU | 최근 사용 시간 | 단순, 효율 | 단기 사용 무시 | OS,DB,브라우저 |
            | LFU | 사용 횟수 | 장기 보존 | 최신성 고려 x | DB 캐시 |
            | FIFO | 삽입 순서 | 구현 쉬움 | 성능 불안 정 |  |
            | MRU | 최근 사용 | 반복 처리에 유리 | 일반적 사용에 부적절 | Prefetch |
            | ARC | 자동 적응 | 강력한 캐시 효율 | 구현 복잡 |  |

            - 네트워크
            - 모바일 + 임베디드

                - 데이터와 인덱스 캐시(자주 사용 데이터를 메모리에서 직접 처리)

            데이터 접근이 지역성을 가짐

        **7. 균형 BST**
        
        이진 탐색 트리에서 균형이 무너지면, 트리가 **한쪽으로 치우친 형태(편향 트리)**가 되어 **탐색 시간이 O(n)**까지 떨어질 수 있습니다.
        
        > 이를 방지하기 위해 AVL 트리나 Red-Black 트리 같은 자가 균형 이진 탐색 트리가 사용됩니다.
        > 
        
        | 종류 | 특징 |
        | --- | --- |
        | **AVL Tree** | 삽입/삭제마다 노드 간 높이 차를 ±1로 유지, **더 엄격한 균형** → 검색 빠름 |
        | **Red-Black Tree** | 노드에 색상(R/B) 속성을 부여해 **느슨한 균형** 유지 → 삽입/삭제 성능이 좋음 |

- 알고리즘
    1. 트리 순회
        
        > DFS(Depth-First Search) 기반 순회는 트리를 깊이 방향으로 재귀적으로 탐색하는 방식입니다.
        > 
        
        | 순회 방식 | 순서 (방문 순서) | 예시 설명 |
        | --- | --- | --- |
        | **전위 순회 (Pre-order)** | **Root → Left → Right** | 루트 먼저 방문 |
        | **중위 순회 (In-order)** | **Left → Root → Right** | 왼쪽 먼저, 루트는 가운데 |
        | **후위 순회 (Post-order)** | **Left → Right → Root** | 자식 다 방문 후 루트 |
        
        ### BFS 기반 순회 – 너비 우선 탐색
        
        > **BFS(Breadth-First Search)**는 트리를 레벨 단위(같은 깊이)로 탐색합니다.
        > 
        > 
        > 구현 시 **Queue(큐)**를 사용합니다.
        > 
        - 루트 → 루트의 자식들 → 그들의 자식들 … 순서로 탐색
        - 흔히 **"레벨 순회(Level-order Traversal)"** 라고 부릅니다.
    
    ### BFS vs DFS 비교표
    
    | 항목 | BFS | DFS |
    | --- | --- | --- |
    | 탐색 방식 | 너비 우선 | 깊이 우선 |
    | 자료구조 | 큐 (Queue) | 스택 (Stack) 또는 재귀 |
    | 대표 사용처 | 최단 거리, 레벨별 순회 | 백트래킹, 경로 존재 확인, 위상 정렬 |
    | 구현 난이도 | 비교적 쉬움 | 재귀일 경우 스택 오버플로우 주의 |
    | 경로 탐색 | 가장 짧은 경로 보장 | 경우에 따라 비효율 가능 |
- 운영체제
    - 과제 쪽
        1. 버퍼캐시란
            - 운영체제에서 디스크 I/O 성능 향상을 위해 사용하는 메모리 공간
            - 디스크 블록 데이터를 메모리에 임시 저장하여 디스크 접근 횟수를 줄임
            - 데이터 일관성을 위해 읽기/쓰기 작업에서 중간 저장소 역할 수행
        2. 버퍼 vs 캐시
            - **버퍼**: 데이터 전송 과정에서 임시 저장 공간, 입출력 속도 차이 완화 목적
            - **캐시**: 자주 사용하는 데이터를 미리 저장해 빠른 접근을 목표
            - 차이점: 버퍼는 주로 데이터 흐름 제어용, 캐시는 데이터 재사용성에 초점

        Race Condition
        
        - DeadLock이란
            - 둘 이상의 프로세스가 **서로가 가진 자원**을 기다리며 무한정 대기 상태에 빠지는 현상
            - 상호배제/점유와 대기/비선점/순환 대기
        - 여러 스레드나 프로세스가 동시에 같은 자원에 접근
            - 예측 불가능한 결과
            - 상호 배제, 데드락, 기아 상태
            
            해결
            
            ### 데드락 해결 방법
            
            - **예방(Prevention)**: 위 조건 중 하나라도 없애서 데드락이 발생하지 않도록 설계
            - **회피(Avoidance)**: 자원 할당 시 안전 상태인 경우에만 허용 (예: 뱅커 알고리즘)
            - **검출(Detection) 및 회복(Recovery)**: 데드락 발생 여부를 주기적으로 검사하고, 발견 시 프로세스 종료나 자원 강제 회수
            - **발생 방지(기본적 동기화 기법)**: 뮤텍스(Mutex), 세마포어(Semaphore) 같은 동기화 기법을 사용해 자원 접근 조절
            
            mutex → 하나의 스레드만 공유 자원에 접근
            
            Semaphore → 한 번에 접근할 수 있는 스레드 수를 제한한다.
            
            **데이터 보안 + 자원 격리 + 충돌 시 영향 가는 경우에는 프로세스 사용.**

        **페이지 단위 관리**
        
        - **메모리 보호**: 각 프로세스가 독립된 주소 공간을 가져 다른 프로세스의 메모리를 침범하지 않음
        - **효율적 메모리 사용**: 자주 사용하지 않는 데이터를 디스크로 옮겨 실제 RAM 사용량을 줄임
        - **프로그램 크기 제약 완화**: 실제 메모리보다 큰 프로그램도 실행 가능
        
        **디스크 공간 이용**
        
        - 독립적 공간 제공 + 충돌 방지
        
        **한꺼번에 메모리에 올리지 않음, 필요한 부분만**
        
        나머지는 디스크에 저장

        - **Page Fault**는 현재 실행 중인 프로세스가 접근하려는 
        **가상 페이지가 물리 메모리**에 존재하지 않을 때 발생하는 예외(Exception)**입니다.
        
        **Demand Paging (지연 로딩)**
        
        필요할 때만 로딩 → 메모리 절약 가능/페이지 폴트 많이 발생
        
        **Pre-Paging(선 로딩)**
        
        미리 예측적으로 로딩 → 연속 접근이 많으면 성능 향상/메모리 낭비 위험 있음

    13. 프로세스 상태
        - **Running (실행 중)**
            - 현재 CPU에서 실행되고 있거나, 실행 준비가 된 상태
        - **Ready (준비 상태)**
            - CPU를 할당받기를 기다리고 있는 상태
        - **Waiting (대기 상태) / Blocked (차단 상태)**
            - I/O 작업 등 이벤트가 완료되기를 기다리며 CPU 할당을 기다리지 않는 상태
        - **Stopped (중지 상태)**
            - 프로세스가 일시 중지된 상태 (예: 디버깅, 신호로 중단됨)
        - **Zombie (좀비 상태)**
            - 프로세스가 종료되었지만 부모 프로세스가 아직 종료 상태 정보를 수집하지 않은 상태
    14. **인터럽트 (Interrupt)**
        - 외부 장치나 이벤트가 발생했을 때 CPU에게 즉시 알림을 보내는 신호
        - CPU는 현재 작업을 잠시 멈추고, 인터럽트 처리 루틴으로 전환하여 신속히 대응
        - 효율적이며, CPU가 불필요하게 기다리지 않도록 함
    15. **폴링 (Polling)**
        - CPU가 주기적으로 장치 상태를 확인해서 이벤트 발생 여부를 검사하는 방법
        - CPU가 계속해서 장치 상태를 체크하기 때문에 비효율적일 수 있음
        - 간단한 시스템에서 주로 사용

        ## 3-Way Handshake: TCP 연결 설정
        
        > TCP는 양쪽 통신 주체가 연결을 맺기 전에 서로의 상태를 확인하고 준비하는 과정을 거칩니다.
        > 
        
        ### 📶 동작 단계
        
        - **SYN (Synchronize)**:
            
            클라이언트가 서버에 연결 요청을 보내며, 초기 시퀀스 번호(ISN)를 포함한 SYN 패킷을 전송합니다.
            
        - **SYN-ACK (Synchronize-Acknowledgement)**:
            
            서버는 클라이언트의 SYN을 받고, 클라이언트의 시퀀스 번호에 대한 ACK를 보내면서 자신의 SYN과 초기 시퀀스 번호를 포함해 응답합니다.
            
        - **ACK (Acknowledgement)**:
            
            클라이언트가 서버의 SYN-ACK에 대해 ACK를 보내면, 연결이 성립되고 양측은 데이터를 주고받을 준비가 됩니다.
            
        
        ✅ 이 과정을 통해 **양쪽이 송수신 준비가 되었음을 확인** → 안정적인 연결 성립
        
        ---
        
        ## 4-Way Handshake: TCP 연결 해제
        
        > 연결을 끊을 때는 양방향 종료가 필요하기 때문에 4단계로 처리됩니다.
        > 
        
        ### 🔻 동작 단계
        
        - **FIN (Finish) - 클라이언트**:
            
            연결을 종료하려는 쪽(예: 클라이언트)이 더 이상 데이터를 보내지 않겠다는 신호인 FIN 패킷을 서버에 보냅니다.
            
        - **ACK - 서버**:
            
            서버는 클라이언트의 FIN 패킷을 받고, 확인 응답(ACK)을 보냅니다. 이때 서버는 아직 데이터 전송이 가능할 수 있습니다.
            
        - **FIN - 서버**:
            
            서버도 연결을 종료할 준비가 되면, 클라이언트에게 FIN 패킷을 보냅니다.
            
        - **ACK - 클라이언트**:
            
            클라이언트는 서버의 FIN 패킷을 받고, 확인 응답(ACK)을 보냅니다. 이후 양쪽 모두 연결이 종료됩니다.

    - 정규화 & 반정규화
        
        데이터의 중복을 줄이고 이상 현상을 방지하기 위한 작업
        
        제1 정규형 → 속성 원자값
        
        제2 정규형 → 부분 종속 제거(완전 종속 관계를 가지도록) → 어떤 속성이  기본키 일부에만 종속되면 안됨
        
        제3 정규형 → 이행적 함수 종속 제거 → (A→B→C) → B를 기준으로 다른 테이블로 분리
        
        과다 정규화
        
        → JOIN 연산 → 성능저하, 복잡성
        
        반정규화
        
        성능 향상을 위한 작업
        
    - 트랜잭션 과 ACID
        
        트랜잭션 → DB에서의 작업 단위 (All or Nothing)
        
        ACID → 원자성, 일관성, 격리성, 지속성
        
        - **원자성**: 계좌 이체 도중 입금만 성공 → ❌ 전체 취소
        - **일관성**: A 계좌 마이너스가 되면 안 됨 → 제약 조건 위반 ❌
        - **격리성**: 다른 트랜잭션이 중간값을 읽어가면 안 됨
        - **지속성**: 커밋된 후 전원 꺼져도 다시 복구되어야 함 (WAL 로그 등)
    - 인덱스(Index)
        
        빠른 데이터 조회를 위해 사용하는 자료구조
        
        | **B+ Tree (범용 인덱스)** | 범위 조회, 정렬된 탐색에 강함 → 대부분의 RDBMS 기본 인덱스 |
        | --- | --- |
        
        | **Hash (해시 인덱스)** | **정확히 일치하는 값 조회에만** 빠름 (범위 조회는 부적합) |
        | --- | --- |
        
        장점 
        
        - **조회 속도 향상**: 수십만 건 중 필요한 데이터도 **O(log n)** 수준으로 탐색
        - **WHERE 절, JOIN, ORDER BY** 등의 연산에서 성능 개선
        - **PRIMARY KEY, UNIQUE 제약**을 구현하는 데 필수
        
        단점
        
        - **쓰기 성능 저하 (INSERT / UPDATE / DELETE 시 오버헤드 발생)**
            - 인덱스도 **함께 갱신**되어야 하므로 성능 하락 가능
            - 예: 값이 바뀌면 **인덱스도 재정렬 또는 재구성** 필요
        - **디스크 공간 추가 소모**
            - 인덱스는 테이블과 **별도의 구조**로 저장됨 → 대용량이면 부담
        - **잘못된 컬럼에 인덱스를 만들면 오히려 느려짐**
            - *선택도(Selectivity)**가 낮은 컬럼 (ex: 성별, TRUE/FALSE 등)에는 인덱스 효과 적음
            - 선택도(Selectivity): 결과가 얼마나 잘 분포되어 있는지
                - 성별,삭제여부와 같은 True/False 컬럼
                    - 결과가 대부분 True일 경우 인덱스를 써도 다 읽어야함
            - **WHERE 조건이 함수나 연산을 포함할 때**
                
                `SELECT * FROM users WHERE UPPER(name) = 'JUNHO';`
                
                함수 적용으로 인한 인덱스 사용 불가
                
        - **과도한 인덱싱은 오히려 전체 시스템 부하 증가**
        
    - JOIN
        1. INNER JOIN
            1. 조건이 일치하는 행만 반환
        2. OUTER JOIN
            1. 한쪽에는 데이터가 있지만 다른 쪽엔 없는 행까지 포함
            
            `SELECT * FROM A LEFT JOIN B ON [A.id](http://a.id/) = B.a_id;`
            
            - **A는 모두**, B는 일치하는 것만 → 없으면 NULL로 채움
            
            RIGHT OUTER JOIN
            
            - B는 모두, A는 일치하는 것만
            
            FULL OUTER JOIN (DBMS에 따라 지원 여부 다름)
            
            - A와 B **모두를 기준**으로 → 어느 쪽이 NULL이어도 출력
        3. CROSS JOIN

        | 항목 | JOIN | 서브쿼리 |
        | --- | --- | --- |
        | **구조** | 여러 테이블을 **한 번에 연결** | **하위 쿼리 결과를 상위 쿼리에 제공** |

        | **용도** | 여러 테이블의 **행 단위 결합** | 필터링, 조건 비교 등 **열 단위 연산**에 유리 |

- 서버 쪽 부하 처리
    1. 서버 자원 관리효율
        - 답변
            1. **부하 분산 (Load Balancing)**
                - 여러 서버에 요청을 고르게 분산해 각 서버의 과부하를 방지합니다.
                - 라운드 로빈, IP 해시, 가중치 기반 분산 등 다양한 방식이 있습니다.
            2. **비동기 처리 및 이벤트 기반 아키텍처**
                - 동기 방식보다 비동기 처리로 I/O 대기 시간을 줄이고, 서버 자원을 더 효율적으로 사용합니다.
                - Node.js, Nginx 등이 이런 방식을 활용합니다.
            3. **캐싱 (Caching)**
                - 자주 조회되는 데이터를 메모리 캐시(Redis, Memcached 등)에 저장해 디스크나 DB 접근을 줄여 서버 부하를 감소시킵니다.
            4. **자원 제한 및 관리 (Rate Limiting, Throttling)**
                - 한 사용자가 과도하게 요청하는 것을 제한해 전체 서비스 안정성을 유지합니다.
            5. **스케일링 (Scaling)**
                - 수평적 스케일링(서버 수 증가)과 수직적 스케일링(서버 성능 향상)을 적절히 조합해 처리 능력을 확보합니다.
- git
    - *브랜치(branch)**는 독립적인 작업 공간을 만들어서 여러 기능이나 버그 수정을 동시에 진행할 수 있게 해줍니다. 이렇게 하면 메인 코드에 영향을 주지 않고 안전하게 개발할 수 있죠.
    - *병합(merge)**은 두 개 이상의 브랜치를 합쳐서 작업 내용을 하나로 통합하는 작업입니다.
    - *리베이스(rebase)**는 한 브랜치의 변경 사항을 다른 브랜치 기반으로 재적용하는 작업으로, 커밋 기록을 깔끔하게 유지할 때 유용합니다.
