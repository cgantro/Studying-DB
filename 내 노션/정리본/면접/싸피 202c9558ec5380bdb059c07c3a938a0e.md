    - **질문:** `project_analysis_report.txt`에 Spring Security를 적용했다고 나와 있는데, "토큰 무효화 전략 미비"가 보안 문제로 지적되어 있습니다. Spring Security를 사용했음에도 불구하고 이러한 문제가 발생한 원인은 무엇이라고 생각하며, Spring Security 내에서 토큰 무효화 기능을 구현한다면 어떤 방법을 사용할 수 있을까요?
        - **답변 가이드:**
            - **원인:** "Spring Security는 인증/인가의 전반적인 프레임워크를 제공하지만, JWT와 같은 토큰 기반 인증의 경우 **토큰 무효화 로직은 개발자가 직접 구현**해야 하는 부분이 많습니다. Spring Security는 필터 체인이나 `AuthenticationProvider` 등을 통해 토큰 유효성 검증 및 사용자 권한 부여를 처리하지만, 토큰이 탈취되거나 로그아웃 시 강제로 무효화하는 로직은 프레임워크가 기본으로 제공하지 않습니다. 저희 프로젝트에서는 이 부분에 대한 깊은 고민과 구현이 미비했습니다."
            - **구현 방법 (Spring Security 연동):** "Spring Security 내에서 토큰 무효화를 구현한다면, `LogoutFilter`나 커스텀 필터를 추가하여 로그아웃 시 전달받은 토큰을 Redis 기반의 블랙리스트에 저장하는 방식으로 구현할 수 있습니다. 이후 모든 요청의 `JwtAuthenticationFilter`에서 토큰 유효성 검사 이전에 이 블랙리스트를 확인하여, 블랙리스트에 있는 토큰은 유효하지 않은 것으로 처리하도록 할 것입니다."

    - 답변
        
        큰 갈등은 아니었지만, 프론트엔드 팀과 서버 간 쿠키 동작 문제로 협업과 조율이 필요한 상황이 있었습니다.
        
        서버에서는 정상적으로 `Set-Cookie` 헤더를 내려주고 있었지만, 프론트에서는 브라우저에 쿠키가 저장되지 않아 로그인 세션이 유지되지 않았습니다.
        
        처음에는 프론트 코드 문제로만 생각했지만, 원인을 함께 분석해보니 **서버와 클라이언트의 도메인이 달라서 발생한 로스도메인 쿠키 문제**였습니다.
        
        특히, **SameSite=None 속성은 HTTPS와 함께 사용해야만 쿠키가 저장**되는데, 개발 환경에서는 HTTP였기 때문에 쿠키가 거부되고 있었습니다. 
        
        그래서 개발 환경에서는 HTTPS를 적용해 테스트했고 프론트 요청에 withCredentials: true 설정을 추가해 문제를 해결했습니다. 또한 이 설정은 CSRF문제가 발생할 수 있기에 개발 환경에서만 적용시키고, 배포 시에는 **프론트와 백엔드의 도메인을 일치시켰습니다(프로토콜 호스트 포트 일치)** 
        
        이 경험을 통해 웹 보안 정책, 브라우저 동작 방식, 그리고 CORS와 같은 프론트-백엔드 연계 지식의 중요성을 실감했고, **문제를 함께 분석하고 조율해 나가는 팀 커뮤니케이션의 중요성**도 배울 수 있었습니다.
        
        - **관련 지식**
            
            ### 1. **쿠키와 세션의 차이점은?**
            
            | 항목 | 쿠키 | 세션 |
            | --- | --- | --- |
            | 저장 위치 | 클라이언트 브라우저 | 서버 |
            | 보안성 | 상대적으로 낮음 (클라이언트에 노출) | 상대적으로 높음 |
            | 용량 제한 | 약 4KB | 제한 없음 (서버 자원에 따라) |
            | 속도 | 빠름 (매 요청마다 자동 전송) | 상대적으로 느릴 수 있음 |
            | 유지 방식 | 브라우저에 저장, 만료 기간 설정 가능 | 세션 ID를 쿠키에 저장해 관리 |
            
            📌 **면접 팁**: “서버 부하나 상태 저장 이슈 때문에 토큰 기반 인증(JWT)으로 전환되는 경우가 많습니다.”
            
            ---
            
            ### 2. **쿠키 속성들 설명**
            
            - `Domain` : 쿠키를 사용할 수 있는 도메인
            - `Path` : 쿠키를 사용할 수 있는 URL 경로
            - `Expires`, `Max-Age` : 쿠키 만료 시간
            - `HttpOnly` : JS로 접근 불가 (XSS 방어용)
            - `Secure` : HTTPS 연결에서만 전송
            - `SameSite` : 크로스사이트 요청 제어
            
            📌 예시:
            
            ```
            http
            복사편집
            Set-Cookie: token=abc123; HttpOnly; Secure; SameSite=None; Path=/; Domain=example.com
            
            ```
            
            ---
            
            ### 3. **SameSite 속성이란? (면접에서 자주 출제)**
            
            - `Strict` : **모든 Cross-Origin 요청에 쿠키 미포함**
            - `Lax` : **GET 요청 허용 (ex. 링크 클릭)**, POST 등은 쿠키 미포함
            - `None` : **쿠키 포함 허용**, 단 **Secure 필수**
            
            📌 **실전 팁**: OAuth 인증처럼 외부 로그인 후 쿠키로 세션 관리하려면 `SameSite=None; Secure`가 필수
            
            ---
            
            ### 4. **JWT vs 쿠키 기반 인증의 차이**
            
            | 항목 | JWT | 쿠키 |
            | --- | --- | --- |
            | 저장 위치 | 로컬 스토리지/세션 스토리지/쿠키 | 쿠키 |
            | 서버 상태 유지 | Stateless | Stateful (또는 쿠키에 JWT 담아서 stateless하게 사용 가능) |
            | 보안 이슈 | XSS 위험 | CSRF 위험 (Secure + SameSite로 완화) |
            
            📌 쿠키 + `HttpOnly` + `Secure` + `SameSite` 설정 조합으로 CSRF 대응 가능
            
            ---
            
            ### 5. **쿠키가 안 담길 때 확인할 점**
            
            1. **`SameSite` 설정** 확인
            2. **`Secure` 속성** 있는데 HTTP 환경일 경우 → 쿠키 무시됨
            3. **프론트에서 withCredentials 설정 필요**
                
                ```
                js
                복사편집
                axios.get('/api', { withCredentials: true })
                
                ```
                
            4. **서버에서 Access-Control-Allow-Credentials: true** 설정 필요
            5. **CORS 허용 도메인과 실제 Origin 일치 여부**
            
            ---
            
            ### 6. **프론트-백엔드 도메인이 다를 때 (Cross-Origin)**
            
            - 반드시:
                - `SameSite=None`
                - `Secure` (HTTPS)
                - `Access-Control-Allow-Origin: 도메인 명시`
                - `Access-Control-Allow-Credentials: true`
            
            📌 기본적으로 **쿠키 포함 요청은 단순 요청(CORS preflight 없이)**이 아니라서 **OPTIONS 요청 허용 설정 필요**
            
            ---
            
            ### 7. **XSS, CSRF와 쿠키의 연관성**
            
            - **XSS 방지**: `HttpOnly` → JS 접근 차단
            - **CSRF 방지**:
                - 쿠키는 자동 전송되기 때문에 **추가 인증 수단 필요**
                - 토큰을 헤더에 직접 담거나, **SameSite=Lax/Strict** 설정으로 방지
                - **CSRF 토큰 (anti-CSRF token)**
                    - 서버가 클라이언트에 **랜덤 토큰 발급** → 이후 요청 시 함께 전송
                    - 서버는 요청의 토큰이 유효한지 검증
                    - 이 토큰은 JS에서 추출해서 `header`나 `body`에 수동 전송

1. 개발자로서 가장 중요하다고 생각하는 윤리 의식은 무엇인가요?
    - **답변**
        
        윤리 의식은 바로 **사용자 데이터 보호 및 프라이버시 존중**입니다. 이는 단순히 법적 준수를 넘어선 **적극적인 책임감**을 의미합니다.
        
        최근 SK에서 발생한 사태를 보며, 법적인 최소한의 기준을 지키는 것을 넘어, **사용자의 개인 정보를 다루는 모든 과정에서 발생할 수 있는 잠재적 위험을 예측하고, 이를 사전에 방지하기 위한 강력한 보호 장치를 마련하는 것이 개발자의 필수적인 윤리 의식**이라는 점을 다시 한번 절감했습니다. 단순히 정보 유출을 막는 것을 넘어, 수집된 정보가 어떻게 활용되고 관리되는지에 대한 투명성을 확보하고, 사용자가 자신의 데이터에 대한 통제권을 가질 수 있도록 지원해야 합니다.
        
2. 본인이 생각하는 '책임감'이란 무엇이며, 어떤 경험에서 책임감을 발휘했나요?
    - **답변**
    제가 생각하는 **책임감**은 단순히 맡은 업무를 완수하는 것을 넘어, **함께 일하는 팀 전체의 역량을 파악하고, 모든 팀원이 각자의 자리에서 최대한 기여할 수 있도록 돕고 이끄는 것**입니다. 궁극적으로는 **팀의 목표를 달성하기 위해 개인이 할 수 있는 모든 노력을 다하는 태도**라고 생각합니다.
        
        **밴드 동아리에서 밴드 마스터 역할을 자주 맡았을 때**입니다. 여러 악기를 다룰 줄 알고 악기를 오래 다뤄왔기에 자연스럽게 팀원들을 리드하는 위치에 서곤 했습니다.
        
        밴드 활동을 하다 보면 팀원 간의 실력 차이에서 오는 불만이 생기기 마련입니다. 예를 들어, 특정 멤버가 곡의 특정 파트를 연주하기 어려워하거나, 전체적인 합주에서 소외감을 느끼는 경우가 있었습니다. 이때 저는 단순히 "네 파트니까 네가 알아서 해"라고 말하는 대신, 밴드 마스터로서 **팀원 개개인의 강점과 약점, 그리고 악기 연주 수준을 면밀히 파악**했습니다.
        
        **개인 연습 방향 제시:** 특정 파트에서 어려움을 겪는 팀원에게는 그 파트를 세분화하여 연습할 수 있는 구체적인 방법을 알려주거나, 난이도를 낮춘 연습곡을 추천하는 등 **맞춤형 연습 방향을 제시**했습니다.
        
        **유연한 편곡:** 모든 팀원이 소외되지 않고 곡에 기여할 수 있도록, 원곡의 난이도를 팀원의 실력에 맞춰 **새롭게 편곡**했습니다. 예를 들어, 어려운 솔로 파트는 다른 악기가 함께 연주하여 부담을 덜거나, 각자의 장기를 살릴 수 있는 새로운 세션을 추가하기도 했습니다.
        
        하지만 이렇더라도 전반적으로 불가능 할 때는, 단호하게 곡 변경을 시키기도 하였습니다.
        
3. 업무의 우선순위를 정할 때 어떤 기준을 가지고 있나요?

        - **긴급성과 중요성 (Impact & Urgency):** 가장 기본적인 기준이죠. 급하게 처리해야 하면서도 전체 프로젝트나 팀에 미치는 영향이 큰 업무를 최우선으로 둡니다. 예를 들어, 서비스의 핵심 기능에 영향을 주거나 고객에게 직접적인 불편을 초래하는 버그 해결 같은 일이죠. 아이젠하워 매트릭스를 활용해 긴급하지만 중요하지 않은 일, 중요하지만 긴급하지 않은 일 등을 구분하여 효율적으로 관리하려고 노력합니다.

        - **예상 소요 시간 대비 효과 (Effort vs. Value):** 짧은 시간 안에 큰 가치를 창출할 수 있는 '퀵 윈(Quick Win)' 과제가 있다면 이를 먼저 처리하여 팀의 사기를 높이거나 즉각적인 성과를 보여주는 것을 선호합니다. 반대로, 시간이 오래 걸리더라도 장기적으로 중요한 영향을 미 미치는 작업은 미리 계획을 세워 꾸준히 진행합니다. 특히 **제한된 자원을 효율적으로 활용해야 하는 상황**에서는 이 기준이 더욱 중요해지죠.

    - XSS : 공격자가 웹 페이지에 악성 스크립트를 삽입하고, 그 스크립트가 다른 사용자의 브라우저에서 실행되도록 하는 공격
        - 사용자의 입력을 검증없이 HTML에 출력할 때 발생
        - 예를 들어 자바스크립트 코드를 입력했을 때, 웹 페이지가 그대로 출력하면 문제가 생김
        
        해결방안
        
        - HttpOnly 쿠키 설정
        - 인풋 정제
    - CSRF : 사용자가 인증된 상태를 악용해 공격자가 원하지 않는 요청을 전송
        - 주로 로그인된 상태에서 악성 사이트에 전송 → 위조된 요청
        - 사용자가 은행 사이트에 로그인하고 세션 유지 중
        - 공격자가 만든 사이트에 접속
        - `<img src="https://bank.com/transfer?to=attacker&amount=1000">`
        - 사용자의 세션 쿠키가 자동 전송되어 **송금이 실행됨**
        
        해결 방안
        
        - SameSite

        - 헤더 기반 인증(자동 전송 차단)
