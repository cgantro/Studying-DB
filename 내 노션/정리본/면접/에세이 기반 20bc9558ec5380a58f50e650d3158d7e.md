        저희 프로젝트에서는 **브라우저에 저장된 쿠키가 서버로 잘 전송되지 않는 문제**가 발생했습니다. 이 문제의 원인은 단순히 기술적인 설정 문제뿐만 아니라, **클라이언트와 서버 개발 간의 이해 부족**에서 비롯된 것이었습니다.
        
        초기에는 각각의 영역에 집중한 나머지, 클라이언트 측은 백엔드의 쿠키 처리 방식을 잘 이해하지 못했고, 백엔드 역시 클라이언트의 동작 방식이나 브라우저의 보안 정책에 대한 이해가 부족했습니다.
        
        이에 저희 팀은 **첫날 회의를 통해 서로의 영역에 대해 공부해오자고 결정**했고, 다음날 다시 모여 함께 분석을 진행했습니다. 그 과정에서 다음과 같은 기술적인 원인을 파악할 수 있었습니다:
        
        1. **도메인이 달랐던 문제**
            
            클라이언트와 서버가 서로 다른 도메인을 사용하고 있었기 때문에, 기본 SameSite 정책(`Lax`)에서는 쿠키가 전송되지 않았습니다. 이 경우에는 `SameSite=None`과 `Secure` 설정이 함께 필요하다는 것을 알게 되었습니다.
            
        2. **Secure 설정의 영향**
            
            초기 백엔드에서는 `Secure` 옵션을 활성화해 쿠키를 설정하고 있었는데, 클라이언트는 HTTPS 환경이 아니거나, 프록시 환경에서 HTTPS가 완전히 적용되지 않아 쿠키가 전송되지 않았습니다.
            
        
        이러한 경험을 통해, 단순히 백엔드나 프론트엔드 개별 기술만 잘 아는 것이 아니라, **웹 보안 정책(CORS, SameSite, Secure 등)**과 **서로의 역할과 동작 방식에 대한 상호 이해**가 중요하다는 교훈을 얻었습니다.
        
        이후에는 다음과 같은 조치를 통해 문제를 해결했습니다:
        
        - 클라이언트 요청에 `withCredentials: true`를 명시.
        - 서버에서 `Access-Control-Allow-Credentials: true` 및 정확한 `origin`을 허용.
        - 쿠키에 `SameSite=None`과 `Secure`를 명확히 설정.
        - 프록시/도메인 구성을 함께 점검하여 HTTPS 환경을 완전하게 적용.

        ## 쿠키 기반 + HttpOnly + SameSite + CSRF 방지 조합이 적합한 이유
        
        | 요소 | 설명 | 장점 |
        | --- | --- | --- |
        | **쿠키 기반 인증** | 로그인 후 서버가 세션 ID를 쿠키로 발급 (`Set-Cookie`) | 브라우저가 자동으로 전송하므로 편리 |
        | **`HttpOnly`** | JS에서 쿠키에 접근 못 하도록 막음 | XSS 공격에 대한 방어 |
        | **`SameSite=Lax` 또는 `Strict`** | 다른 사이트에서 요청할 때 쿠키 전송 제한 | CSRF 위험 최소화 |
        | **CSRF 토큰 사용** | form 요청이나 ajax 요청에 **CSRF 토큰**을 추가하여 이중 방어 | SameSite 정책 우회한 CSRF에도 대응 가능 |
        
        ### 1. **`SameSite=None`과 `Secure`는 왜 같이 써야 하나요?**
        
        - **답변:** `SameSite=None`은 쿠키를 cross-site 요청에서도 보내겠다는 뜻인데, 이 설정은 보안상 위험할 수 있어 브라우저들은 `Secure=true`가 같이 설정되지 않으면 해당 쿠키를 **차단**합니다.
        
        ---
        
        ### ✅ 2. **`withCredentials: true`를 사용하면 어떤 일이 발생하나요?**
        
        - **답변:** 브라우저가 cross-origin 요청 시에도 쿠키, 인증 헤더 등 **자격 증명(credentials)**을 포함해서 전송하게 됩니다. 서버도 `Access-Control-Allow-Credentials: true`로 이를 명시적으로 허용해줘야 합니다.
        
        ---
        
        ### ✅ 3. **CORS와 쿠키 문제는 어떻게 연관되어 있나요?**
        
        - **답변:** CORS는 브라우저가 다른 origin 간의 리소스 공유를 제어하는 보안 정책입니다. 쿠키를 전송하려면 CORS 정책에 따라:
            - `credentials: true`
            - `Access-Control-Allow-Origin`에 와일드카드()가 아닌 정확한 도메인 명시
            - `Access-Control-Allow-Credentials: true` 설정이 필요합니다.
        
        ---
        
        ### ✅ 4. **도메인이 다르면 왜 쿠키가 안 가나요?**
        
        - **답변:** 쿠키는 기본적으로 **same-origin 정책**을 따르기 때문에, 도메인이 다르면 브라우저가 자동으로 쿠키를 전송하지 않습니다. 이를 해결하려면 쿠키에 `SameSite=None`, `domain=공통 도메인`, CORS 설정 등이 모두 조화롭게 구성되어야 합니다.
        
        ---
        
        ### ✅ 5. **쿠키 대신 토큰을 localStorage에 저장하면 이런 문제가 해결되지 않나요?**
        
        - **답변:** 기술적으로는 localStorage + Authorization 헤더 방식으로도 인증을 구현할 수 있습니다. 하지만 이 경우 XSS에 더 취약하고, 브라우저가 자동으로 전송하지 않으므로 **직접 헤더에 넣는 구현**이 필요합니다. 쿠키 방식은 자동 전송되는 장점이 있고, `HttpOnly` 속성으로 XSS를 막을 수 있습니다.
        
        ---
        
        ### ✅ 6. **프록시(Nginx, Vercel 등) 환경에서 쿠키 관련해서 주의할 점은?**
        
        - **답변:**
            - `Set-Cookie` 헤더가 프록시에서 누락되거나 제거되지 않도록 설정해야 함
            - HTTPS termination 지점이 어디인지 명확히 파악하고, 백엔드에서 `req.secure`를 정확히 인식할 수 있도록 `X-Forwarded-Proto`를 사용하는 설정 필요
        
        ---
        
        ### ✅ 7. **브라우저는 어떤 조건일 때 쿠키를 전송하나요?**
        
        - **답변:**
            - 요청 도메인과 쿠키 도메인이 일치하거나 포함될 때
            - Secure 쿠키는 HTTPS에서만 전송
            - SameSite 정책에 따라:
                - `Strict`: 동일 출처 요청에서만
                - `Lax`: 일부 GET 요청에서는 가능
                - `None`: cross-site 전송 허용 (Secure 필수)
