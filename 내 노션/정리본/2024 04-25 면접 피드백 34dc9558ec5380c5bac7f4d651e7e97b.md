# SDS 면접 피드백 정리

## 1. 기술 PT 준비 방향

작년 기술 PT 기출을 참고해서 준비해야 한다.

단순 기술 설명이 아니라, **AI 내용 + SDS 관련 뉴스 + 비즈니스 문제 해결 관점**을 섞어야 한다.

준비 방식은 다음과 같다.

- Claude, GPT 등을 활용해 기술 PT 주제 20개 정도를 뽑아본다.
- 주제는 단순 CS 문제가 아니라, 보통 **비즈니스 문제를 기술로 해결하는 형태**로 준비한다.
- 답변할 때는 설계도를 함께 그리는 연습을 한다.
- 내 설계에 대해 CS적으로 질문받는 상황까지 대비한다.

예상 질문 방향은 다음과 같다.

- 왜 이런 구조로 설계했는가?
- 병목은 어디서 생길 수 있는가?
- 장애가 나면 어떻게 대응할 것인가?
- 확장성은 어떻게 확보할 것인가?
- AI를 어디에 붙이면 실제 비즈니스 효과가 나는가?

## 2. 답변 태도와 말하는 방식

현재 답변이 다소 외워서 말하는 느낌이 있다.

따라서 앞으로는 **흥미, 경험, 문제 해결 경험 중심으로 자연스럽게 꺼내는 연습**이 필요하다.

핵심은 다음과 같다.

- 답변을 암기하지 말고 내재화해야 한다.
- 질문을 받으면 먼저 내 경험에서 꺼내는 습관을 들여야 한다.
- 기술 키워드를 말할 때도 “제가 이걸 왜 중요하게 봤는지”를 함께 말해야 한다.
- 면접관이 궁금한 것은 완벽한 문장이 아니라, 실제로 고민해본 사람인지 여부다.

예를 들면 다음 방향이 좋다.

> 저는 이 부분을 프로젝트에서 직접 겪어봤습니다. 단순히 기능을 붙이는 것보다, 데이터 흐름이 꼬이지 않게 만드는 게 더 중요하다고 느꼈습니다.
>

## 3. Python 관련 준비

SDS 내부에서 Python을 많이 사용한다는 피드백이 있었다.

따라서 Python 경험을 어느 정도 만들어두면 좋다.

다만 핵심 메시지는 이것이다.

> 언어 자체는 큰 문제가 아니다. 새로운 언어를 배우는 데 큰 저항이 없고, 필요한 상황에서는 빠르게 익혀 적용할 수 있다.
> 

답변 방향은 다음과 같다.

- 주력은 Java/Spring이지만 Python도 활용할 수 있다.
- Python은 자동화, 로그 분석, 데이터 처리, AI API 활용 등에 적합하다고 이해하고 있다.
- 언어보다 중요한 것은 문제를 구조화하고 해결하는 능력이다.
- 새로운 언어를 배우는 데 부담이 없다는 태도를 보여준다.

예시 답변:

> 주력 경험은 Java/Spring 기반 백엔드이지만, Python도 로그 분석이나 자동화, AI API 활용 같은 영역에서 충분히 활용할 수 있다고 생각합니다. 언어 자체보다 중요한 것은 문제를 나누고 적절한 도구를 선택하는 능력이라고 보고, 필요한 언어는 빠르게 익혀 적용할 수 있습니다.
>

## 4. Java 17과 CS 기본기 준비

Java 17을 단순 문법 위주로 외우기보다, **왜 쓰는지**와 CS 개념을 연결해서 공부해야 한다.

공부 방향은 다음과 같다.

- Java 17의 주요 특징
    - record
    - sealed class
    - switch expression
    - var
    - Stream API
    - Optional
    - GC 개선
- Spring Boot와 Java 17 조합의 장점
- JVM 메모리 구조
- GC 동작 원리
- Thread, 동시성, synchronized, Lock
- Collection 내부 구조
- HashMap, ArrayList, LinkedList, PriorityQueue 동작 방식
- 예외 처리 방식
- 객체지향 원칙

중요한 것은 “이 기능을 안다”가 아니라 다음처럼 말하는 것이다.

> Java 17의 기능을 단순히 문법으로 보기보다, 코드 안정성, 유지보수성, 타입 안정성을 높이는 도구로 이해하고 있습니다.

## 5. UX, 안전성, 기본기

피드백에서 중요한 키워드는 **UX, 안전성, 기본기**다.

여기서 말하는 기본기는 단순 CS 지식만 의미하지 않는다.

내가 정의해야 할 기본기는 다음과 같이 잡을 수 있다.

## 내가 생각하는 기본기

개발자의 기본기는 단순히 문법을 많이 아는 것이 아니라, **사용자가 안정적으로 쓸 수 있는 구조를 만들기 위해 문제를 끝까지 확인하는 태도와 능력**이다.

구체적으로는 다음이다.

- 데이터 흐름을 이해하는 능력
- 장애 가능성을 미리 생각하는 능력
- 로그와 지표로 원인을 확인하는 습관
- 기능 구현 후 검증하는 태도
- 협업자가 이해할 수 있게 문서화하는 능력
- 사용자가 불편하지 않게 안정성과 UX를 함께 고려하는 태도

필살기 방향은 다음과 같다.

> 저는 기본기를 단순 문법 지식이 아니라, 문제가 생겼을 때 원인을 추적하고, 재현하고, 검증 가능한 구조로 개선하는 능력이라고 생각합니다. 프로젝트에서도 Redis 분리, ACK 기반 상태 전이, 비동기 처리 구조 개선처럼 안정성을 높이는 방향으로 기본기를 적용해왔습니다.

## 6. AI 활용 도구 구체화

AI 활용은 반드시 구체화해야 한다.

단순히 “ChatGPT를 사용했습니다” 수준이면 약하다.

면접에서는 **AI를 어떻게 업무 도구로 통제하고 활용했는지**를 말해야 한다.

핵심 방향은 다음과 같다.

- AI에게 단순 명령만 내린 것이 아니다.
- 역할, 범위, 금지사항, 출력 형식을 명확히 줬다.
- AGENTS.md, SKILLS, 가이드라인 등을 활용해 일관된 결과를 유도했다.
- Codex를 코드 작성뿐 아니라 계획, 테스트, 리뷰, 리팩토링, 생산성 검증에 활용했다.
- Git worktree 기반으로 여러 에이전트를 병렬로 돌려 작업을 분리해봤다.(이건 안해봄 ㅋ)
- 토큰 사용량, 작업 시간, 리뷰 품질 등을 비교하며 활용 방식을 개선했다.

정리하면 키워드는 다음이다.

- AI 프롬프팅
- Codex 활용
- Claude 활용 의사
- AGENTS.md
- SKILLS
- 프로젝트 가이드라인
- Git worktree 기반 병렬 작업
- 플래닝
- 테스트 생성
- 코드 리뷰
- 리팩토링
- 생산성 검증
- 평가 기준 수립

답변 방향:

> 저는 AI를 단순히 코드를 대신 짜는 도구로 쓰기보다, 작업을 계획하고 검증하는 보조 개발자로 활용하려고 했습니다. 예를 들어 Codex에 바로 구현을 맡기기보다 AGENTS.md에 백엔드 레이어 규칙, 금지사항, 테스트 기준을 먼저 정의하고, Git worktree를 나눠 여러 접근을 병렬로 비교했습니다. 이후 테스트 결과와 코드 리뷰 기준으로 어떤 결과가 더 적합한지 판단했습니다.

AI 활용 답변은 수치나 사례가 있으면 훨씬 좋다.

준비할 만한 예시는 다음과 같다.

- 스토리포인트 3점짜리 작업을 AI 없이 했을 때와 AI 활용했을 때 비교

- 테스트 케이스 생성 속도 개선

- 문서 초안 작성

예시 문장:

> 예를 들어 스토리포인트 3점 정도의 백엔드 API 작업을 할 때, 먼저 AI에게 요구사항을 태스크 단위로 분해하게 하고, 이후 제가 레이어 규칙과 예외 처리 기준을 검토했습니다. 구현 후에는 테스트 케이스와 코드 리뷰 체크리스트를 AI로 생성해 누락 가능성을 줄였습니다. 최종 판단은 제가 했고, AI는 생산성과 검토 범위를 넓히는 도구로 활용했습니다.

## 9. 장단점 답변 방향

장단점은 너무 정형적으로 말하지 말고, 조금 더 진짜 내 인생에서 나온 답변으로 준비해야 한다.

재미있고 사람 냄새가 나는 답변이 좋다.

> 장점 : 소통/협업 → 10년동안 밴드하면서 다양한 기조를 가진 사람들을 만날 수 있었다.
실제로 사설 밴드 꾸려서 대회 나갈 때, 멤버 전부가 추구하는 음악이 달랐다.
이러한 환경에서 곡을 만들 때, 각각의 데모를 만들고, 그 데모에서 좋은 부분을 따와서 곡에 합치면서 곡 초안을 만들었고, 그 후 노래의 장르 및 방향성 주제를 정한 뒤, 각자의 장점을 살려서 곡을 만들었다.

나의 경우엔 펑크/블루스 피아노를 주로 쳤었기에 베이스 라인을 만들 때 건반 라인처럼 묻어나오는 게 있었는데, 이러한 장점을 살렸다.
> 

## 장점 방향

- 여러 가지를 직접 해보는 사람
- 새로운 도구나 기술에 거부감이 적은 사람
- 문제를 끝까지 파고드는 사람
- 구조를 정리하는 사람
- 팀에서 기준을 잡고 흐름을 맞추는 사람
- 밴드 리딩 경험처럼 사람들과 함께 결과를 만드는 경험이 있는 사람

예시:

> 제 장점은 새로운 걸 직접 해보는 데 거리낌이 적다는 점입니다. 밴드 활동도 직접 팀을 만들고 공연을 준비했고, 개발에서도 C++, Unreal, Spring, AI 도구까지 필요하면 직접 부딪혀봤습니다. 덕분에 처음 보는 문제도 피하기보다 구조를 나눠서 접근하는 편입니다.
> 

## 단점 방향

- 처음에는 너무 많이 벌리는 경향
- 관심이 생기면 깊게 파다가 우선순위가 흐려질 때가 있음
- 그래서 최근에는 Jira, 문서화, 작업 기준을 통해 정리하려고 함

예시:

> 단점
무던한 편이라 여자친구한테 공감능력 없다는 소리를 자주 들었다~ㅋ
알빠노?

## 11. 녹음/녹화 연습

면접 답변은 반드시 녹음 또는 녹화해서 확인해야 한다.

확인할 포인트는 다음이다.

- 외운 티가 나는지
- 말이 너무 길어지는지
- 결론부터 말하는지
- 경험 기반으로 말하는지
- 표정과 시선이 자연스러운지
- 기술 설명이 추상적인지
- 수치와 사례가 들어가는지
- “제가 한 일”이 명확한지

특히 지금 필요한 것은 **암기한 답변을 말하는 연습이 아니라, 질문을 받고 경험에서 꺼내는 연습**이다.

## 12. 마음가짐

면접은 심문이 아니라 동료를 구하는 자리라고 생각해야 한다.

중요한 마음가짐은 다음이다.

- 너무 쫄 필요 없다.
- 내가 부족한 점도 있지만, 더 잘할 수 있다는 태도를 보여주면 된다.
- 완벽한 답보다, 같이 일하고 싶은 사람이라는 인상을 주는 게 중요하다.
- 기술을 아는 사람보다, 문제를 책임지고 해결하려는 사람이 되어야 한다.
- 면접관은 내 적이 아니라, 같이 일할 동료를 찾는 사람이다.

정리하면 다음 문장으로 압축 가능하다.

> 나는 모든 걸 완벽히 아는 사람은 아니지만, 문제를 구조화하고 끝까지 개선해본 경험이 있다. 새로운 도구와 언어도 빠르게 익히고, 팀 안에서 더 안정적인 결과를 만드는 사람이라는 점을 보여주자.
