## 1. 자신을 소개해 주세요

안녕하세요. 삼성SDS 지원자 홍윤표입니다.

저는 **꾸준히 해내고, 팀의 기준을 맞추는 개발자**

10년간 밴드 활동을 하며 서로 다른 성향의 사람들과 하나의 결과물을 만들어왔고, 협업은 기준을 맞추고 끝까지 완성해내는 과정이라고 느꼈습니다.

개발 프로젝트에서도 Jira, 브랜치 규칙, API 문서처럼 팀이 같은 방식으로 일하기 위한 기준을 정리했습니다. 최근에는 AI 도구를 활용할 때도 AGENTS.md 같은 작업 규칙을 문서화해, AI가 팀 기준에 맞게 동작하도록 관리하려고 했습니다.

삼성SDS에서도 안정적으로 기능을 구현하면서, 팀이 함께 일하기 편한 흐름을 만드는 개발자로 성장하고 싶습니다.

- 2안
    
    저는 **기준을 분명히 세우되, 사람들과 호흡을 맞추는 과정을 중요하게 생각하는 사람**입니다.
    
    대학 시절 밴드 활동을 오래 했고, 밴드 마스터 역할도 맡았습니다. 합주를 하다 보면 각자 연주 스타일도 다르고 실력 차이도 있어서, 그냥 제 생각만 밀어붙이면 분위기도 무너지고 결과도 잘 나오지 않았습니다. 그래서 저는 어떤 부분이 맞지 않는지 먼저 기준을 정리하고, 그걸 상대가 받아들일 수 있는 방식으로 전달하려고 많이 노력했습니다.
    
    그 과정을 겪으면서 좋은 결과는 혼자 잘해서 나오는 게 아니라, 서로 다른 사람들을 같은 방향으로 움직이게 할 때 나온다는 걸 배웠습니다. 지금도 개발할 때 문제를 구조적으로 보는 편이지만, 결국 협업 안에서 기준을 맞추고 흐름을 정리하는 태도는 이 경험에서 많이 만들어졌다고 생각합니다.
    

## 2. 왜 삼성 SDS에 지원했나요?

저는 협업 과정의 혼선과 정보 불일치를 줄이는 제품을 만들고 싶어서 삼성SDS에 지원했습니다. 
밴드 활동과 프로젝트를 하면서 협업은 단순히 소통을 많이 하는 게 아니라, 
서로 다른 기준을 가진 사람들이 같은 기준 안에서 움직이게 만드는 일이라고 느꼈습니다. 
브리티웍스는 메일, 메신저, 미팅, 드라이브를 연결하고 코파일럿으로 회의록 요약까지 지원해서, 
협업 과정에서 생기는 맥락 단절과 정보 불일치를 줄여주는 제품이라고 봤습니다. 
이런 협업 구조를 실제 업무 환경에서 더 잘 작동하게 만드는 데 기여하고 싶어 지원했습니다.

## 4. 본인의 강점은 무엇인가요?

제 강점은 **무던하게 버티고 끝까지 해내는 태도**입니다.

군 복무 중 허리를 다쳐 전역하게 된 경험이 있습니다. 당시에는 걷는 것도 쉽지 않았고, 꾸준히 해오던 운동도 제대로 할 수 없어 많이 힘들었습니다.

하지만 상황을 바꿀 수 없다면, 지금 할 수 있는 것부터 하자는 생각으로 재활과 근력 보강을 꾸준히 이어갔습니다. 그 결과 현재는 많이 회복된 상태이고, 이 경험을 통해 어려운 상황에서도 감정에만 머무르지 않고 할 일을 찾아 지속하는 태도를 갖게 됐습니다.

삼성SDS에서도 처음 마주하는 업무나 어려운 상황이 있더라도 조급해하기보다, 필요한 것을 차근차근 배우고 끝까지 책임지는 구성원이 되겠습니다.

## 5. 본인의 약점은 무엇인가요?

제 약점은 문제 상황에서 상대의 감정보다 해결 방법을 먼저 생각하는 편이라는 점입니다.

밴드마스터로 활동할 때 합주나 곡 방향이 맞지 않으면 빠르게 고치고 싶어서 “이건 이렇게 해야 한다”는 식으로 바로 피드백을 준 적이 있습니다. 제 의도는 팀 결과물을 좋게 만들기 위한 것이었지만, 나이 차이가 있는 후배들에게는 그 방식이 다소 직설적이고 어렵게 느껴질 수 있다는 말을 들었습니다.

이후에는 바로 해결책을 말하기보다 먼저 상대가 어떤 의도로 연주했는지, 어떤 부분이 어려웠는지 듣고 나서 방향을 제안하려고 노력했습니다. 이 경험을 통해 협업에서는 맞는 답을 말하는 것만큼, 상대가 받아들일 수 있는 방식으로 전달하는 것도 중요하다고 느꼈습니다.

## 7. 팀에서의 역할은 어떤 편인가요?

저는 팀에서 **기준을 정리하고 흐름을 맞추는 역할** 을 자주 맡았습니다.

개발 프로젝트에서는 Jira, 브랜치 규칙, pre-commit, AI AGENT.md 오같은 기준을 정리해 팀원들이 같은 방식으로 작업할 수 있도록 했습니다.

밴드 활동에서도 팀 합주를 할 때 곡의 완성도 기준이나 합주 방향을 맞추는 역할을 했습니다. 예를 들어 각자 연주 스타일이나 해석이 다를 때, 어느 부분을 더 강조할지, 어떤 구간을 맞춰야 할지 정리하면서 팀 전체의 합이 맞도록 조율했습니다.

그래서 저는 팀 안에서 각자 맡은 일이 따로 움직이지 않고, 하나의 결과물로 이어지도록 기준을 맞추는 데 강점이 있다고 생각합니다.

## 8. 어떠한 갈등 상황이 있었나요?

팀 프로젝트에서 **쿠키 기반 인증과 헤더 기반 인증 방식** 중 어떤 걸 사용할지 의견 차이가 있었습니다.

세션 vs 쿠키

우선 각 방식의 장단점을 정리해 팀원들과 공유했습니다.

| 기준 | 쿠키 기반 JWT | 서버 세션 기반 |
| --- | --- | --- |
| 클라이언트 저장 | JWT를 쿠키에 저장 | sessionId만 쿠키에 저장 |
| 인증 상태 위치 | 토큰 자체에 claim 포함 | 서버 세션 저장소 |
| 서버 확장성 | 세션 저장소 의존 적음 | 공용 세션 저장소 필요 |
| 강제 로그아웃 | 추가 설계 필요 | 세션 삭제로 쉬움 |
| CSRF | 쿠키 자동 전송이라 방어 필요 | 쿠키 자동 전송이라 방어 필요 |
| XSS 토큰 탈취 | HttpOnly면 JS 접근 어려움 | HttpOnly면 sessionId 접근 어려움 |
| 구현 복잡도 | Refresh/만료/무효화 설계 필요 | 세션 저장소/클러스터링 필요 |

그 다음에는 서비스 환경을 고려해 어느 쪽이 더 적합한지 논의했습니다. 

성능이 좋지 못한 서버를 사용했기에, 최대한 서버에 부담이 덜 가는 방향으로 타협을 했다. → 헤더 기반 인증 

이 과정에서 **객관적인 근거를 바탕으로 열린 소통을 하고, 타협점을 찾아가는 경험**을 쌓을 수 있었습니다.
