## 1. RobotPal 프로젝트 정체 파악

### 1-1. 이 프로젝트는 무엇인가

RobotPal은 C++ 기반으로 개발된 크로스 플랫폼 로봇 시뮬레이터 및 제어 엔진입니다.

### 1-2. 어떤 문제를 해결하려는가

가상의 로봇 시뮬레이션 환경(SandboxScene)을 제공하고, 로봇의 카메라 뷰를 외부로 실시간 스트리밍(TCP/WebSocket)하며, 이를 Python 기반의 외부 에이전트나 웹 클라이언트와 연동하여 알고리즘을 테스트하거나 제어할 수 있는 환경을 제공하려는 것으로 추정됩니다. 특히 카메라 스트리밍 시 발생하는 부하를 분산시켜 메인 렌더링 틱에 영향을 주지 않으려 노력하고 있습니다.

### 1-3. 어떤 실행 환경을 지원하는가

- **Windows (데스크톱)**: CMake 기반으로 빌드되며 MSVC 컴파일러를 사용(`out/build/x64-debug/RobotPal/RobotPal.exe`).
- **Web (브라우저)**: Emscripten을 통한 WebAssembly 빌드 및 배포를 지원 (`emscripten-build.yml`, `RobotPal-web/` 폴더 존재).
- **Python 클라이언트**: 시뮬레이터와 통신하기 위한 Python 패키지(`RobotPal-python`)를 별도로 제공.

### 1-4. 한 문장 요약

> **"멀티 스레드 기반의 영상 인코딩 최적화와 ECS 아키텍처를 적용하여, C++ 네이티브부터 웹 WebAssembly 환경까지 지원하는 확장성 높은 로봇 시뮬레이터 엔진"**

## 2. 저장소 전체 구조 분석

### 2-1. 루트 디렉터리 구조 [코드 확인: 메타데이터 기반]

- `RobotPal/`: C++ 코어 엔진 소스코드 (`include/`, `src/`)
- `RobotPal-python/`: Python 제어/연동 라이브러리 소스 (`src/robotpal/`)
- `RobotPal-web/`: WebAssembly 구동을 위한 웹 템플릿 및 서비스 워커
- `Assets/`: 3D 모델(`.glb`), 셰이더(`.glsl`), 폰트(`.ttf`), 텍스처 등 리소스
- `External/`: 외부 의존성 (glad 등)
- `tests/`: 벤치마크 및 테스트 스크립트 (`.cpp`, `.py`)
- `.github/workflows/`: CI/CD 파이프라인 (Emscripten 빌드/배포)

### 2-2. 주요 모듈 분류 [코드 확인]

1. **코어 모듈**: `EngineApp`, `Window`, `SceneManager`. (flecs 기반 ECS 구조 활용).
2. **렌더링 모듈**: `RenderSystemModule`, `Shader`, `Texture`, `Framebuffer`, `PBR/IBL` 처리.
3. **네트워크 모듈**: `NetworkEngine`, `TcpNetworkTransport`, `WebSocketTransport`.
4. **스트리밍 모듈**: `StreamingSystemModule`, `StreamingWorker`, `JpegEncoder`. 카메라 화면을 JPEG로 압축하여 외부로 전송.
5. **시뮬레이션 모듈**: `SimController`, `RealController`, `HybridController`.

### 2-3. 실행 진입점과 초기화 흐름 [코드 확인]

프로그램의 진입점은 `main.cpp`입니다.

1. `EngineApp app;` 객체 생성.
2. `app.Run();` 호출.
3. `EngineApp` 내부에서 `Init()` -> `MainLoop()` -> `Shutdown()`의 생명주기를 가짐.
4. `MainLoop` 내에서 `flecs::world`를 기반으로 등록된 시스템(렌더링, 네트워크, 스트리밍 등)들이 매 프레임마다 업데이트될 것으로 추정됨.

### 2-4. 전체 데이터/제어 흐름 한눈에 보기 [추정 및 코드 확인]

- **[렌더링 루프]**: Scene의 카메라에서 프레임버퍼 렌더링 -> 화면 출력.
- **[스트리밍 흐름]**: 렌더링된 프레임버퍼 픽셀 읽기 -> 작업 큐(Queue)에 삽입 -> `StreamingWorker` 스레드 풀에서 JPEG 인코딩 -> `NetworkEngine`을 통해 패킷 전송 (TCP/WebSocket).

## 3. 기능/모듈별 상세 분석

### [앱 초기화 및 코어 관리 - EngineApp]

1. **기능 목적**: 엔진의 생명주기 관리 및 하위 매니저(Scene, Editor, Window) 초기화. ECS(flecs) 월드 소유.
2. **관련 핵심 파일**: `EngineApp.h`, `EngineApp.cpp`.
3. **실행 흐름**: `Init()`으로 Window와 flecs 초기화 -> `MainLoop()`에서 델타타임 계산 및 flecs 시스템 진행 -> 종료 시 `Shutdown()`.
4. **프로젝트 전체에서의 역할**: 모든 서브 시스템을 묶어주는 척추 역할.
5. **이 부분을 꼭 이해해야 하는 이유**: 어떤 시스템이 먼저 업데이트되는지(예: 입력 -> 물리 -> 렌더링 -> 스트리밍) 순서를 제어하는 주체이기 때문입니다.
6. **장점**: `flecs::world`를 멤버로 소유하여 Entity 기반의 데이터 지향 설계(DOD)를 강제함.
7. **한계 / 개선점**: `MainLoop`가 하드코딩되어 있다면 플랫폼(Web/Emscripten의 `emscripten_set_main_loop`) 분기가 복잡해질 수 있음. (단, `emscripten_mainloop.h`가 있는 것으로 보아 매크로나 별도 파일로 분리했을 것으로 추정됨).
8. **면접 핵심 한 줄**: "EngineApp은 flecs 기반 ECS 구조의 최상위 관리자로서, 메인 스레드의 렌더링 틱과 생명주기를 통제합니다."
9. **안전한 방어 포인트**: "제가 EngineApp을 직접 설계하지 않았더라도, 제 모듈이 flecs의 어떤 Phase에서 실행되도록 EngineApp에 등록되는지는 정확히 인지하고 개발했습니다."

### [네트워크 처리 모듈 - NetworkEngine]

1. **기능 목적**: 외부(Python 스크립트 등)와의 양방향 통신 지원.
2. **관련 핵심 파일**: `NetworkEngine.h`, `NetworkTransport.h`, `NetworkQueue.h`.
3. **실행 흐름**: `TryConnect()`로 연결 -> 백그라운드 스레드에서 수신된 데이터를 `m_RecvQueue`에 적재 -> 메인 틱에서 `GetPacket()`으로 폴링.
4. **프로젝트 전체에서의 역할**: 로봇 제어 명령을 수신하고 센서/카메라 데이터를 외부로 송출하는 인터페이스.
5. **이 부분을 꼭 이해해야 하는 이유**: 네트워크 지연이나 병목이 전체 렌더링 프레임(APP FPS)에 영향을 주지 않도록 스레드가 어떻게 분리되어 있는지 알아야 합니다.
6. **장점**: `NetworkTransport`라는 추상화 레이어를 두어 TCP와 WebSocket(Emscripten 용) 구현을 다형성으로 분리함.
7. **면접 핵심 한 줄**: "네트워크 I/O를 Transport 인터페이스로 추상화하여, 데스크톱(TCP)과 웹(WebSocket) 통신을 비동기 큐 기반으로 안전하게 처리합니다."

### [스트리밍 최적화 벤치마크 모듈]

1. **기능 목적**: 렌더링 루프와 분리된 다중 스레드 인코딩 파이프라인의 성능 검증 및 프레임 드랍률 측정.
2. **관련 핵심 파일**: `streaming_frame_drop_benchmark.cpp`, `카메라스트리밍 병목분석.md`.
3. **실행 흐름**: Producer(렌더링 루프 모사)가 원본 RGBA를 큐에 밀어 넣음 -> `workerCount` 개수의 Worker 스레드가 큐에서 작업을 꺼내 JPEG로 인코딩 -> Drop Rate 및 FPS 계산.
4. **프로젝트 전체에서의 역할**: 스트리밍 병목의 원인(인코딩 부하)을 증명하고, 멀티스레딩 도입의 타당성을 수치화하는 검증 도구.
5. **장점**: `std::condition_variable`과 `std::mutex`를 사용한 전형적이고 안정적인 생산자-소비자(Producer-Consumer) 패턴 구현. 큐 사이즈 제한(`maxQueueSize`)을 두어 메모리 폭발(OOM)을 방지함.
6. **면접 핵심 한 줄**: "스트리밍 시 발생하는 JPEG 인코딩 병목을 해결하기 위해, 메인 루프를 블로킹하지 않는 멀티 워커 구조를 설계하고 벤치마크를 통해 증명했습니다."

## 4. 팀 코드까지 이해하기 위한 관점 정리

### [Flecs 기반 ECS 모듈]

1. **책임**: 로봇의 위치, 속도, 카메라 컴포넌트 등의 데이터를 분리하고 캐시 효율성(Data Locality) 극대화.
2. **연결 방식**: 각 System(Render, Streaming, Transform)이 Entity를 순회하며 작업 수행.
3. **설명 시 알아야 할 핵심**: "로봇의 위치 데이터(Transform)와 렌더링 로직(System)이 분리되어 있어, 시스템 확장이 용이하다."
4. **면접 안전 표현**: "제가 ECS 프레임워크 자체를 연동한 것은 아니지만, 제 기능을 구현할 때 flecs의 쿼리를 통해 필요한 컴포넌트만 추출하여 캐시 미스를 줄이는 구조를 적극 활용했습니다."

## 5. 코드 구조와 설계 평가

- **모듈 분리 [코드 확인]**: `EngineApp`, 네트워크, 렌더링 모듈이 `.h` / `.cpp` 로 깔끔하게 분리되어 있으며, Transport 인터페이스를 통한 추상화가 돋보임.
- **성능 대응 구조 [코드 확인]**: 네트워크 송수신(`NetworkQueue`)과 스트리밍 인코딩에 락(mutex) 기반 큐를 적용하여 메인 스레드 블로킹을 방지함.
- **확장성 [추정]**: `RealController.h`와 `SimController.h`가 있는 것으로 보아, 가상 시뮬레이션 환경뿐만 아니라 실제 물리 로봇 연동까지 고려된 구조임.
- **아쉬운 점 / 개선점 [코드 확인]**: 벤치마크 코드에서 `std::deque`와 단일 `std::mutex`를 사용하는데, 워커 스레드가 19개까지 늘어나면 큐 접근 시 락 경합(Lock Contention) 오버헤드가 발생할 수 있음. *Lock-free queue나 Ring Buffer 구조로 개선을 제안해 볼 수 있음.*

## 6. 성능 / 최적화 / 병목 분석 (가장 중요한 셀링 포인트)

해당 프로젝트의 가장 핵심적인 엔지니어링 성과는 **카메라 스트리밍 병목 해결**입니다.

- **문제 상황 [문서 확인]**: 카메라 렌더 타겟(224x224)을 JPEG로 압축하는 과정이 메인 렌더 루프와 경쟁하여 전체 애플리케이션 FPS가 하락하는 병목 발생.
- **해결 방식 [문서 및 코드 확인]**: 인코딩 워커 스레드 수를 1개에서 하드웨어 지원 최대 스레드(19개)로 확장(`hardware_concurrency`).
- **측정 지표 [문서 확인]**: `RobotPal.exe`의 메인 루프 FPS (APP FPS), 수신 측의 프레임레이트 (SINK FPS).
- **결과 수치 [문서 확인]**:
    - **외장 GPU (RTX 4050) 환경**: 싱글 스레드 55.94 FPS -> 멀티 스레드 59.41 FPS (**+6.2% 향상**).
    - **내장 GPU (Intel Arc) 강제 환경**: 싱글 스레드 평균 58.10 FPS -> 멀티 스레드 평균 66.95 FPS (**+15.24% 향상**).
- **성능상 의미**: 비동기 처리를 통해 CPU 바운드 작업(JPEG 인코딩)을 분산시킴으로써 메인 렌더링 파이프라인을 보호함. 특히 상대적으로 CPU 부하에 민감한 내장 그래픽 환경에서 개선 효과가 매우 두드러짐.
- **면접 포인트**: "최적화를 '감'으로 하지 않고, `tests/tcp_frame_sink.py` 스크립트를 작성하여 수신 측 부하를 통제한 상태에서 정확히 `[APP FPS]`와 `[SINK FPS]`를 분리하여 측정했다는 점이 핵심입니다."

## 7. 테스트 / 벤치마크 / 검증 결과 분석

### 7-1. 테스트 코드 존재 여부 [코드 확인]

- **벤치마크 코드**: `tests/streaming_frame_drop_benchmark.cpp`를 통해 멀티 스레드 환경에서 처리량(Processed)과 버려짐(Dropped) 비율을 검증함.
- **수신 검증기**: `tests/tcp_frame_sink.py` 를 통해 네트워크 부하 실험을 수행함.

### 7-2. 실제 결과 요약 및 해석 [문서 확인]

- **측정 내용**: 스레드 개수에 따른 프레임 처리 효율.
- **해석 시 주의점**: SINK FPS가 16~17 수준에 머무는 것은 인코딩 문제가 아니라 시스템 주기가 `.rate(2)`로 설정되어 있어 원본 생성 자체가 절반으로 줄어들었기 때문임을 문서에서 정확히 인지하고 있음. (이는 문제 원인을 오판하지 않고 파이프라인 전체를 이해하고 있다는 강력한 증거임).

### 7-3. 테스트 개선 제안

- **메모리 누수 테스트 (Valgrind/AddressSanitizer)**: 스트리밍 큐에 패킷이 쌓일 때 메모리 반환이 지연되는지 확인.
- **Lock Contention 측정**: 워커가 많아질수록 큐 락 대기 시간이 얼마나 길어지는지 프로파일링 추가.

## 8. 내가 면접에서 프로젝트 전체를 설명하는 법

### 8-1. 프로젝트 전체 설명 1분 버전

"RobotPal은 C++와 ECS 기반으로 구축된 크로스 플랫폼 로봇 시뮬레이션 엔진입니다. 단순 렌더링에 그치지 않고, 로봇의 카메라 뷰를 외부 AI 에이전트나 웹으로 실시간 스트리밍하기 위해 네트워크와 인코딩 파이프라인을 비동기로 설계했습니다. 저는 이 프로젝트에서 각 모듈이 메인 렌더 스레드를 방해하지 않도록 구조를 파악하고, 병목 구간인 영상 압축 과정을 벤치마크하여 멀티 스레드로 최적화하는 과정을 거쳤습니다."

### 8-4. 다른 사람 코드까지 이해하고 있다고 보여주는 답변 방식

"예를 들어 `EngineApp` 클래스의 생명주기 관리는 다른 팀원이 주도했지만, 저는 제 스트리밍 모듈이 이 생명주기 안에서 `flecs::world`의 어느 시점에 데이터를 읽어가야 화면 찢김(Tearing)이나 락 오버헤드가 없는지 코드 수준에서 분석하고 통합했습니다."

### 8-6. 테스트/검증까지 했다고 설명하는 안전한 방식

"기능 구현에서 멈추지 않고 C++ 마이크로 벤치마크 코드와 Python 더미 싱크 서버(`tcp_frame_sink.py`)를 만들어 정량적 평가를 진행했습니다. 실제로 내장 GPU 환경에서는 멀티스레드 도입으로 앱 FPS를 15% 이상 개선할 수 있었습니다."

## 9. 예상 질문 리스트

### [성능 최적화]

- **질문**: 카메라 스트리밍 인코딩 과정에서 왜 프레임 드랍이 발생했나요?
- **의도**: 병목 지점의 원리 이해 및 CPU vs GPU 바운드 파악.
- **답변 핵심**: "단일 스레드 구조에서는 JPEG 압축 연산(CPU 바운드)이 완료될 때까지 다음 프레임버퍼 리드백이나 렌더링 루프가 대기해야 했기 때문입니다. 이를 해결하기 위해 Producer-Consumer 패턴 기반의 큐와 워커 스레드 풀을 도입했습니다." [과장 금지: 완전히 드랍이 0이 된 것은 아니며, 네트워크 지연에 의해 SINK FPS는 한계가 있음을 명시]

### [네트워크 / 스트리밍]

- **질문**: TCP와 WebSocket을 동시에 지원하는 이유는 무엇이고, 어떻게 구현했나요?
- **의도**: 크로스 플랫폼 아키텍처 및 인터페이스 추상화 이해도.
- **답변 핵심**: "데스크톱/Python 클라이언트 통신에는 빠르고 범용적인 TCP를, Emscripten으로 빌드된 웹 환경에서는 브라우저 제약 상 WebSocket을 사용해야 했습니다. 이를 위해 `NetworkTransport` 인터페이스를 추상화하고 다형성을 활용해 런타임/빌드타임에 구현체를 교체하도록 설계되어 있습니다."

## 10. 최종 정리

- **10-1. 한 문장 요약**: C++ ECS 구조를 기반으로 다중 스레드 인코딩 최적화를 적용한 크로스 플랫폼 로봇 제어/스트리밍 시뮬레이터.
- **10-2. 핵심 모듈 5개**: `EngineApp` (생명주기), `NetworkEngine` (통신), `StreamingWorker` (최적화), flecs 기반 ECS (데이터 구조), `SceneManager` (상태 관리).
- **10-3. 자신 있게 말할 수 있는 포인트**:
    1. 데이터 지향 설계(ECS) 환경에서의 협업.
    2. 뮤텍스와 조건변수를 활용한 스레드 풀 구현.
    3. 명확한 지표(FPS, Drop Rate) 기반의 성능 검증 방법론.
- **10-4. 조심해서 말해야 하는 포인트 (확인 불가)**: 물리 엔진(Collision) 내부 처리, OpenGL 셰이더 세부 수학 모델 (직접 짰다고 단정하면 안 됨).
- **10-6. 구조적으로 개선할 부분**: 현재 락 기반(std::mutex) 큐를 Lock-Free Queue로 마이그레이션하여 스레드 경합 최소화.

## 11. 협업 프로젝트 면접용 안전한 표현 모음

- **테스트/성능 검증 결과를 설명할 때**: "단순히 '스레드를 늘려서 빨라졌다'고 추정하지 않았습니다. 환경변수(`ROBOTPAL_ENCODE_WORKERS`)를 통해 워커 수를 통제하고, 외장/내장 GPU별로 앱 렌더링 프레임과 네트워크 수신 프레임을 분리하여 측정하는 벤치마크 환경을 구축해 가설을 검증했습니다."
- **깊게 물어보면 선을 긋는 법**: "네트워크 패킷 큐잉 및 전송 로직의 흐름 제어는 제가 구조를 정확히 파악하고 수정할 수 있지만, 웹 소켓 전송 시 Emscripten의 세부 C-API 통신 내부 구현(JS 매핑 로직)까지 처음부터 짠 것은 아니며 프로젝트 팀 내에서 추상화된 인터페이스를 활용했습니다."
