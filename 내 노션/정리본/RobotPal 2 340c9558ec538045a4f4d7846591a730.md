## 1. 프로젝트 정체와 목적

### 1-1. RobotPal은 무엇인가

C++와 flecs(ECS) 기반으로 개발되어 데스크톱(Windows)과 웹(WebAssembly) 환경을 모두 지원하는 경량 로봇 시뮬레이터 엔진.

### 1-2. 어떤 문제를 해결하려는가

가상 환경 내 로봇의 카메라 시점(프레임버퍼)을 외부 Python 에이전트나 웹 클라이언트로 실시간 스트리밍하여, 외부 알고리즘이 로봇을 제어할 수 있는 폐쇄 루프(Closed-loop) 테스트 환경 제공.

### 1-3. 어떤 실행 환경을 지원하는가

Windows (MSVC), Web 브라우저 (Emscripten/WebAssembly).

### 1-4. 한 문장 요약

"ECS 아키텍처와 비동기 인코딩 파이프라인을 결합하여, 메인 렌더링 루프의 성능 저하 없이 로봇 시점 영상을 외부로 스트리밍할 수 있는 크로스 플랫폼 시뮬레이터."

## 3. 기능/모듈별 상세 분석

### [스트리밍 시스템 - StreamingSystemModule]

1. **기능 목적**: 렌더링된 화면을 캡처하여 압축 후 외부로 전송.
2. **관련 핵심 파일**: `StreamingSystemModule.cpp`, `StreamingWorker.cpp`, `JpegEncoder.h`.
3. **실행 흐름**: 매 프레임 ECS에서 카메라 컴포넌트 데이터 추출 -> 프레임버퍼에서 이미지 추출 -> Worker Queue에 전달 -> 인코딩 완료 후 Network Queue로 전달.
4. **장점**: 메인 스레드와 인코딩 스레드가 철저히 분리된 생산자-소비자(Producer-Consumer) 패턴.
5. **한계 / 개선점**: `glReadPixels` 자체의 동기적 대기가 남아있을 가능성(PBO 미적용 시).
6. **면접 핵심 한 줄**: "스트리밍 모듈은 렌더링된 픽셀을 비동기 워커에 위임하여 메인 렌더 루프의 블로킹을 방지하는 방파제 역할을 합니다."

## 4. 내 역할 심화 분석 (영상 처리 병렬화 및 성능 개선)

### 4-1. 내가 맡은 핵심 영역 요약

화면 렌더링 완료 후 영상 데이터를 JPEG로 압축하여 네트워크로 넘기는 파이프라인의 **병목 해소 및 멀티 워커 아키텍처 재설계**.

### 4-2. 관련 핵심 파일 / 클래스 / 함수

- `tests/streaming_frame_drop_benchmark.cpp` (성능 측정 로직)
- `RobotPal/src/Util/StreamingWorker.cpp` (스레드 풀 관리)
- `카메라스트리밍 병목분석.md` (원인 분석 문서)

### 4-3. 원래 어떤 문제/병목이 있었는가

- **코드 근거**: 이전 구조(Single-worker) 및 분석 문서(`카메라스트리밍 병목분석.md`).
- **문제 지점**: 메인 렌더 루프 혹은 단일 백그라운드 스레드에서 시뮬레이션 물리 연산, 렌더링, 그리고 무거운 **JPEG 압축(CPU Bound)**이 순차적으로 처리됨.
- **병목 이유**: 고해상도(혹은 높은 FPS) 상황에서 1장의 프레임을 압축하는 시간이 다음 프레임 생성 시간보다 길어지면서 큐에 데이터가 쌓이거나 메인 루프가 대기하는 현상(Stall) 발생.

### 4-4. 문제점 도출 과정

- 체감이 아닌 **측정 기반**으로 접근.
- `streaming_frame_drop_benchmark.cpp`를 작성하여 재현 환경을 고정함. 큐의 압력을 모니터링하고 큐 사이즈를 초과할 때 버려지는 프레임을 카운팅함.

### 4-5. 내가 맡은 해결 방식

- **구조 변경**: Single-worker → Multi-worker (스레드 풀) 도입.
- `std::mutex`와 `std::condition_variable`을 활용한 **Producer-Consumer 큐** 구축.
- 렌더링 틱(Producer)은 픽셀 데이터를 큐에 밀어 넣기만 하고 즉시 리턴하여 메인 FPS 방어.
- 여러 개의 Worker 스레드가 큐에서 픽셀을 꺼내 병렬로 JPEG 인코딩 수행.

### 4-6. 성능상 의미

- **장점**: CPU Bound 작업인 인코딩 부하를 멀티 코어로 분산시켜 **Frame Drop을 획기적으로 감소**시키고, 앱 자체의 메인 렌더링 FPS 저하를 방지함.
- **한계**: 스레드가 무한정 늘어나면 Context Switching 오버헤드와 Mutex 경합(Contention) 발생.
- **면접 핵심 한 줄**: "인코딩 병목을 해결하기 위해 단일 처리 흐름을 멀티 워커 기반의 비동기 큐 구조로 분리하여 렌더링 파이프라인을 보호했습니다."
- **과장 주의 포인트**: "네트워크 지연까지 완벽히 해결했다"고 말하면 안 됨. 네트워크 대역폭 한계는 별개의 문제이며, 본 작업은 **CPU 인코딩 병목**을 해결한 것임.

## 6. 내가 맡지 않은 팀 코드까지 이해하기

### [RenderSystemModule]

1. **책임**: flecs 시스템을 순회하며 메쉬를 그리고 프레임버퍼에 색상 값을 씀.
2. **내 코드와의 연결점**: 렌더링이 끝난 프레임버퍼에서 `glReadPixels` 등으로 데이터를 가져와 내 스트리밍 큐에 넣어야 함. 동기화 시점이 매우 중요.
3. **이해해야 할 핵심**: 더블 버퍼링이나 PBO(Pixel Buffer Object)를 사용하지 않으면 GPU 파이프라인이 CPU 리드백을 기다리느라 렌더링 성능이 하락할 수 있음을 인지.

### [NetworkEngine]

1. **책임**: 스트리밍 워커가 압축한 JPEG 바이트 배열을 TCP/WebSocket으로 실제 송출.
2. **이해해야 할 핵심**: 비동기로 전송되므로 내 워커 코드가 네트워크 전송 완료를 블로킹하며 기다리지 않음을 구조적으로 파악.

## 7. 성능 / 최적화 / 병목 분석 심화

- **GPU → CPU Readback 병목 (면접관의 핵심 타겟)**: 영상 스트리밍 시 가장 큰 비용 중 하나는 GPU에 있는 화면 픽셀을 CPU 메모리로 복사(`glReadPixels`)하는 시간입니다.
- **저장소 기준 확인**: 현재 코드베이스가 PBO(Pixel Buffer Object)를 활용한 비동기 Readback을 구현했는지는 명시적으로 확인되지 않음 [확인 불가/추정].
- **면접 답변 포인트 (약점 방어)**: *"현재 워커 스레드 분리를 통해 CPU 인코딩 병목은 크게 개선했습니다. 하지만 구조상 `glReadPixels` 호출 시 GPU와 CPU의 동기화 대기(Stall)가 미세하게 남아있을 수 있습니다. 추후 PBO 기반의 비동기 텍스처 리드백(Async Readback)을 도입하여 렌더 파이프라인과 스트리밍 파이프라인을 완전히 디커플링하는 것이 다음 목표입니다."*

## 8. 테스트 / 벤치마크 / 검증 결과 전체 분석

- **장점**: 최적화를 감이나 추측이 아닌 명확한 코드(Test script)와 지표(Drop Rate, FPS)로 증명함. 이는 시니어 개발자가 가장 선호하는 접근법임.

## 9. 문제 발생 → 문제점 도출 → 개선 → 결과 정리집 (STAR)

- **상황(Situation)**: 로봇의 카메라 뷰를 외부로 스트리밍하는 과정에서 화면이 자주 끊기고 렌더링 프레임이 저하됨.
- **과제(Task)**: 병목 지점을 찾고 스트리밍 처리량을 높여 프레임 드랍 완화.
- **행동(Action)**:
    1. `streaming_frame_drop_benchmark.cpp` 작성하여 병목 구간(JPEG 압축 구간) 특정 (단일 워커 시 17.56% 드랍).
    2. 연산과 영상 처리를 병렬화하기 위해 `StreamingWorker` 클래스를 구현하고 Thread Pool 기반의 Multi-worker 구조로 파이프라인 재설계.
    3. 락 경합 방지를 위해 조건 변수와 큐 사이즈 제한 도입.
- **결과(Result)**: 동일 통제 조건 하에서 프레임 드랍률을 17.56% → 0.78% 로 감소시킴.

### 10-3. 내 역할 설명 (집중형)

"저는 시뮬레이터의 실시간성을 보장하기 위해 스트리밍 파이프라인 최적화를 담당했습니다. 초기엔 영상 인코딩과 시뮬레이션 틱이 순차적으로 맞물려 17%에 달하는 프레임 드랍이 발생했습니다. 저는 이를 감으로 고치지 않고 벤치마크 코드를 작성해 CPU 병목을 특정했습니다. 이후 Thread Pool과 큐를 활용한 멀티 워커 구조로 아키텍처를 변경하여, 메인 렌더 루프를 보호함과 동시에 프레임 드랍률을 0.78% 수준으로 안정화시켰습니다."

### 10-5. "이 수치(17% -> 0.78%) 정말 직접 측정한 건가요? 코드 근거 있나요?" 대응

"네, 정확히 코드 기반으로 측정한 수치입니다. 저장소의 `tests/` 디렉터리에 `streaming_frame_drop_benchmark.cpp`라는 벤치마크 모듈을 작성했습니다. 생성기(Producer)가 1800개의 더미 프레임을 일정한 FPS로 큐에 푸시하도록 통제하고, 워커가 이를 소화하지 못해 큐 사이즈 제한에 걸려 버려진(Dropped) 횟수를 정확히 로깅하여 계산한 신뢰할 수 있는 데이터입니다."

### 10-7. "네트워크나 렌더링 시스템은 직접 짜지 않은 것 같은데, 어떻게 연동했나요?" 대응

"네트워크 코어와 ECS 엔진은 다른 팀원이 구축했습니다. 하지만 제 파이프라인을 얹기 위해 전체 흐름을 이해하는 것은 필수였습니다. 예를 들어 렌더링이 끝난 프레임버퍼에서 어느 타이밍에 픽셀을 읽어와야 찢어짐(Tearing)이 없는지, 그리고 제 인코딩 워커가 완료된 데이터를 `NetworkQueue`에 밀어 넣었을 때 네트워크 스레드가 이를 어떻게 비동기로 가져가는지 인터페이스 단위로 구조를 분석하고 협업했습니다."

## 11. 최종 정리

### 11-1. RobotPal을 한 문장으로 설명

크로스 플랫폼 대응이 가능한 C++ ECS 기반 로봇 시뮬레이터로, 특히 다중 스레드 인코딩 최적화를 통해 실시간 제어와 스트리밍에 강점을 가진 엔진.

### 11-2. 내가 가장 자신 있게 말할 수 있는 기술 5개

1. C++ 멀티스레딩 (Mutex, Condition Variable) 활용 스레드 풀 설계
2. Producer-Consumer 패턴을 통한 메인 스레드 병목 해소
3. 마이크로 벤치마크 환경 구축 및 지표(Throughput/Drop rate) 기반 성능 개선
4. ECS(Entity Component System) 아키텍처 상에서의 모듈 설계
5. 객체 지향 및 인터페이스 기반의 모듈 디커플링 (네트워크/스트리밍 분리)

### 11-3. 내가 조심해서 말해야 할 부분 5개 (과장 주의)

1. PBO/GPU 비동기 리드백의 완벽한 달성 (코드 미확인 시 언급 주의)
2. 네트워크 환경 (Bandwidth/Latency)에 의한 최종 끊김 완벽 해소 (인코딩 병목만 해결한 것임)
3. OpenGL/WebGL 렌더링 파이프라인 밑바닥 구현 (다른 팀원 역할일 가능성)
4. WebAssembly(Emscripten)의 메모리 관리 및 JS 브릿지 깊은 수준 (포팅 구조만 이해)
5. 무한정 스레드를 늘리는 것이 정답이라는 뉘앙스 (락 경합 문제 인지 필요)

### 11-8. 면접관이 좋아할 설명 포인트

- **측정 기반의 접근**: "문제를 체감으로 느끼지 않고, 벤치마크 코드를 작성해 1800프레임 중 316프레임 드랍이라는 정확한 정량적 수치를 뽑아냈다."
- **전체 파이프라인 이해**: "내 코드가 빨라지는 것에서 멈추지 않고, 내 코드가 메인 렌더 루프(FPS)를 방해하지 않도록 아키텍처를 분리(Decoupling)했다."

### 11-9. 면접관이 물고 늘어질 수 있는 약점

- **Q. "Mutex 기반 큐면 스레드가 많아질 때 락 경합(Lock Contention) 오버헤드는 없었나?"**
    - *방어:* "맞습니다. 현재 `std::mutex` 기반이라 스레드가 무거워지면 대기가 발생합니다. 이를 개선하기 위해 추후 Lock-Free Queue나 Ring Buffer 구조로 발전시킬 계획이 있습니다."
