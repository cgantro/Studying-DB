### Spring 관련

### 기초 지식

- **Spring Framework**란
    
    자바 개발을 편리하게 해주는 오픈소스 프레임워크
    
    - 경량 컨테이너로서 자바 객체를 직접 관리
        - 객체 생성,소멸과 같은 라이프 사이클 관리
    - 제어의 역전(**IoC**)라는 기술을 통해 느슨한 결합 도모
        - 컨트롤의 제어권이 사용자가 아닌 프레임 워크에 있다
        - 프레임워크가 사용자의 코드를 호출
        - POJO의 생성, 소멸, 서비스에 대한 권한을 가짐
            - 특정 자바 모델이나 프레임워크를 따르지 않는 순수한 자바객체
            - 도메인과 비즈니스 로직을 수행하는 대상
    - **DI**
        - 각각의 계층이나 서비스들 간에 의존성이 존재할 경우 프레임워크가 서로 연결시켜준다
        - 생성자 주입, 수정자 주입, 필드 주입
            - **생성자 주입** : **생성자로 객체를생성하는 시점에 필요한 빈을 주입 → 순환 참조 해결 가능**
                - **final(불변성 보장) 가능**
                - DI 컨테이너를 사용하지 않고도 **테스트 가능**
                - 코드
                    
                    ```java
                    @Service
                    public class UserService {
                    
                        private UserRepository userRepository;
                        private MemberService memberService;
                    
                        public UserService(UserRepository userRepository, MemberService memberService) {
                            this.userRepository = userRepository;
                            this.memberService = memberService;
                        }
                    
                    }
                    ```
                    
            - **필드 주입**
                - 코드 간결
                - 외부 수정 불가능
                - DI를 지원하는 컨테이너가 있어야 함
                - 코드
                    
                    ```java
                    @Service
                    public class UserService {
                    
                        @Autowired
                        private UserRepository userRepository;
                        @Autowired
                        private MemberService memberService;
                    
                    }
                    ```
                    
            - **수정자 주입: Setter 주입**
                - public으로 구현 → 관계를 주입받는 객체의 변경 가능성을 열어둠
                - 객체가 변경될 필요성이 있을 때만 사용
                - 코드
                    
                    ```java
                    @Service
                    public class UserService {
                    
                        private UserRepository userRepository;
                        private MemberService memberService;
                    
                        @Autowired
                        public void setUserRepository(UserRepository userRepository) {
                            this.userRepository = userRepository;
                        }
                    
                        @Autowired
                        public void setMemberService(MemberService memberService) {
                            this.memberService = memberService;
                        }
                    }
                    ```
                    
            
            "**생성자 -> 필드 -> 메서드**" 순으로 호출
            
    - **AOP**
        - 트랜잭션, 로깅, 보안과 같이 여러 모듈에서 공통적으로 사용하는 기능의 경우 **해당 기능을 분리하여 관리 가능**
    - 독립 컨테이너에서 동작 가능 → 별도의 WAS 설치 없이 내장된 톰캣으로 실행
- **WAS와 WS의 차이**
    
    **WAS(Web Application Server)**
    
    - 비즈니스 로직 넣기 가능
    - Tomcat, PHP, ASP, .NET 등
    
    **WS(Web Server)**
    
    - 비즈니스 로직을 넣을 수 없음
    - Nginx, Apache 등
- **Spring과 Spring boot의 차이**
    
    가장 큰 차이점은 **Auto Configuration**의 차이인 것 같습니다. Spring은 프로젝트 초기에 다양한 환경설정을 해야 하지만,
    **Spring Boot**는 설정의 많은 부분을 자동화하여 사용자가 편하게 스프링을 활용할 수 있도록 돕습니다.
    spring boot starter dependency만 추가해주면 설정은 끝나고, **내장된 톰캣을 제공해 서버를 바로 실행**할 수 있습니다.
    
- RequestBody, RequestParam, ModelAttribute의 차이
    
    **RequestBody**
    
    - 클라이언트가 전송하는 JSON 형태의 HTTP Body 내용을 Java 객체로 변환 시켜주는 역할
    - 값을 주입하지 않고 변환 → 생성자,Getter,Setter가 없어도 정상적으로 할당
    
    **RequestParam**
    
    - HTTP 요청 파라미터를 받기위해 사용
    - 반드시 파라미터가 전송되어야함
    - 반드시 필요하지 않다면 required의 값을 false로 설정해줘야함
    
    **ModelAttribute**
    
    - HTTP Body 내용과 HTTP 파라미터의 값들을 생성자, Getter, Setter를 통해 주입하기 위해 사용
    - 값 변환이 아닌 값을 주입 → 변수들의 생성자, Getter, Setter가 없으면 변수들이 저장되지 않음
- VO, BO, DAO, DTO
    - DAO(Data Access Object)
        - DB의 데이터에 접근을 위한 객체
        - Mapper 또는 Repository
    - BO(Business Object)
        - 여러 DAO를 활용해 비즈니스 로직을 처리하는 객체
        - Service
    - DTO(Data transfer Object)
        - 각 계층간의 데이터 교환을 위한 계층
            - Controller, View, BusinessLayer, Persistent Layer와 같은 게층
    - VO (Value Object)
        - 실제 데이터만을 저장하는 객체
        - 변하지 않는 데이터 객체
            - read, getter만 가능
    - Entity
        - 실제 Database의 테이블과 1:1 매핑되는 클래스
        - 상속을 받거나 구현체여서는 안됨
- 아파치와 톰캣
    
    아파치
    
    - 정적인 데이터를 처리하는 웹 서버
    - 기본적으로 멀티 프로세스
        - 설정에 따라 멀티 쓰레드를 같이 운용 간으
    
    ---
    
    톰캣
    
    - WAS로서 요청을 처리하기 위한 Thread Pool내 쓰레드를 제공
    - 멀티 쓰레드

### Security

- **base64**
    
    8비트 이진 데이터(실행 파일, ZIP 파일 등)를 문자 코드에 영향 받지 않는 공통 ASCII 영역의 문자들로만 이루어진 일련의 문자열로 바꾸는 인코딩 방식
    
    - 64진법, 2의 제곱수 ASCII 문자를 써서 표현할 수 있는 가장 큰 진법
    - 128진법은 쓰이지 않음 (ASCII 문자가 128개보다 적음)
- **MAC**
    
    Message Authentication code
    
    **메시지를 인증하기 위해 사용하는 정보**
    
    - 메시지의 무결성과 인증
        
        무결성 : 변조되지 않았다
        
        인증 : 올바른 송신자로부터 온 메시지
        
- **HMAC**
    
    **REST API 요청을 받았을 때, 이 요청이 신뢰할 수 있는 호출인지 확인하는 방법 중 하나**
    
    사용자의 ID와 암호 같은 민감한 정보를 받는 대신, 사전에 공유한 secretKey와 전송할 message를  입력 받아 Hash 기반의 MAC을 생성해서 전송하며 서버는 Secret Key와 message를 기반으로 MAC을 검증
    
    - HTTPS를 같이 사용하는게 좋음
- **HTTPS**
    
    하이퍼텍스트 전송 프로토콜 Security
    
    모든 웹사이트(특히 로그인 자격 증명)는 HTTPS를 사용해야 한다
    
    암호화 프로토콜을 사용하여 통신을 암호화
    
    - **SSL**로 알려졌지만 **TLS**라고 불린다
    - **SSL** : 암호화 기반 인터넷 보안 프로토콜
        - 웹에서 전송되는 데이터를 암호화
        - 핸드셰이크라는 인증 프로세스
    - **TLS**
        - SSL의 발전 버전
        - 암호화, 인증, 무결성 달성
            - **암호화** : 제 3자로부터 전송되는 데이터를 숨김
            - **인증** : 정보를 교환하는 당사자가 요청된 당사자임을 보장
            - **무결성 : 데이터가 위조, 변조 되지 않음**
    
    ---
    
    작동 방식 : 두 개의 서로 다른 키 사용
    
    - 개인 키 : 웹 사이트 소유자가 관리
        - **비공개**, 웹 서버에 저장

    - 공개 키
        - 안전한 방식으로 서버와 사용하고자 하는 모든 사람이 사용 가능
        - **공개키로 암호화된 정보는 개인 키로만 해독 가능**

    - HTTPS : 443

### Batch Size

- N+1 문제를 안 일어나게 하는 방법은 아님
    - N+1 문제가 발생하더라도 `select * from user where team_id = ?` 가 아닌
    - `select * from user where team_id in (?,?,?)` 방식으로 N+1문제가 발생하게 하는 방법

`application.yml`

```yaml
spring:
  jpa:
    properties:
      hibernate:
        default_batch_fetch_size: 1000
```

`application.properties`

```yaml
spring.jpa.properties.hibernate.default_batch_fetch_size=1000
```

간단하게 설정 하나 세팅해주면 아래와 같이 in 절로 쿼리가 나간다.

```sql
Hibernate: select team0_.id as id1_0_, team0_.name as name2_0_ from team team0_
Hibernate: select users0_.team_id as team_id1_1_1_, users0_.users_id as users_id2_1_1_, user1_.id as id1_2_0_, user1_.first_name as first_na2_2_0_, user1_.last_name as last_nam3_2_0_, user1_.team_id as team_id4_2_0_ from team_users users0_ inner join user user1_ on users0_.users_id=user1_.id where users0_.team_id in (?, ?, ?, ?)
```

---

**실무에서 N+1문제로 DB가 죽어버리는 문제 방지법**

- 연관 관계에 대한 설정이 필요하다면
    - FetchType을 성능 최적화를 하기 어려운 EAGER가 아닌 LAZY모드로 사용
    - 성능 최적화가 필요한 부분 → Fetch Join 사용
    - Batch Size의 값을 1000 이하로 설정
- 연관관계를 끊어버리는 것도 방법

### 기초 지식

- **HTTP METHOD**
    
    종류
    
    - get, post, put, delete
    
    get, post의 차이
    
    - get : 클라에서 서버로 어떠한 리소스로부터 정보를 요청하기 위해 사용되는 method
        - 값을 가져와서 보여줄 때, 내용, 상태 등을 바꾸지 않을 때
        - 브라우저에 캐싱됨
    - post : 리소스를 생성/업데이트 하기 위해 사용되는 method
        - 서버 상의 데이터 값이나 상태를 바꾸기 위해 사용
- **HTTP 상태 코드**
    
    **주요 코드**
    
    - 200 : OK
    - 404 : Not Found
        - 요청받은 리소스를 찾을 수 없음
            - 브라우저에서는 알려지지 않은 URL
        - 서버는 인증받지 않은 클라이언트로 부터 리소스를 숨기기 위해 이 응답을 403대신에 전송할 수 있음
    - 503 : 서버가 요청을 처리할 준비가 되지 않음
        - 유지 보수를 위해 작동이 중단되거나 과부하가 걸린 서버일 경우 발생
    
    ---
    
    100번대(정보) : 요청 받음, 프로세스 계속 진행
    
    200번대(성공) : 요청을 성공적으로 받고, 인식하고, 수행
    
    300번대(리다이렉션) : 요청 완료를 위해 추가 작업 조치가 필요합니다.
    
    400번대(클라이언트 오류) : 요청의 문법이 잘못되었거나 요청을 처리 불가
    
    500번대(서버 오류) : 서버가 명백히 유효한 요청에 대해 충족을 실패

- **RDB와 NoSQL**
    
    ![img1.daumcdn.png](%EB%A9%B4%EC%A0%91%20%EB%8C%80%EB%B9%84/img1.daumcdn.png)
    
    RDB
    
    - 엄격한 Schema 요구, table 구조
        - 엄격한 Schema는 데이터 중복을 방지 → update가 많을 때 유리
    - 수평 확장이 쉽지 않음
    
    NoSQL
    
    - 비정형 데이터
        - 데이터 중복이 있음 → update시 모든 컬렉션에서 수정 필요
        - update가 적고 조회가 많을 때 유리
        - 데이터가 많을 때 사용
    
- **DB에서 인덱스의 장점, 사용 방법**
    
    개념
    
    - 검색 속도를 높이기 위한 색인 기술
    - 컬럼에 대해 인덱스 설정 가능
    
    **사용법**
    
    ```sql
    -- single column index for id
    CREATE INDEX si_id ON t_student (id);
    -- single column index for name
    CREATE INDEX si_name ON t_student (name);
    -- multi column index for id, name
    CREATE INDEX mi_id_name ON t_student (id, name);
    -- multi column index for id, name, age
    CREATE INDEX mi_id_name_age ON t_student (id, name, age);
    ...
    ```
    
    ---
    
    ### 인덱스 설정 기준
    
    **카디널리티 (Cardinality)**
    
    - 높을 수록 인덱스 설정에 좋음
        - 한 컬럼이 갖고 있는 값의 중복도가 낮다
    
    **선택도 (Selectivity)**
    
    - 낮을 수록 인덱스 설정에 좋다(5~10%가 적당)
        - 한 컬럼이 갖고 있는 값 하나로 적은 row가 찾아진다
        
        **선택도 계산법**
        
        $(= 컬럼의 특정 값의 row 수 / 테이블의 총 row 수 * 100)$
        
        ex) 10개의 데이터에서 고유한 학번(grade) 컬럼, 2명씩 같은 이름(name) 컬럼, 5명씩 같은 나이(age) 컬럼인 경우
        
        ① 학번(grade) 컬럼 선택도: 1 / 10 = 10%
        
        ② 이름(name) 컬럼 선택도: 2 / 10 = 20%
        
        ③ 나이(age) 컬럼 선택도: 5 / 10 = 50%
        
    
    **조회 활용도**
    
    - 높을 수록 좋음
        - Where의 대상 컬럼으로 많이 활용되는지
    
    **수정 빈도**
    
    - 낮을 수록 좋음

    ### DML에 미치는 영향
    
    - UPDATE, DELETE
        - 인덱스로 설정된 컬럼에 대해 Where 사용할 수도 있음
            - 즉 수정/삭제할 데이터를 찾는게 빠름
    - INSERT
        - 효율 낮음
        - 새로운 데이터 추가 → 인덱스가 설정되어 있던 컬럼의 테이블이 수정되야함
    
- **수직 확장, 수평 확장**
    
    수직 확장
    
    - 단일 서버의 스펙을 단순히 더 좋은 것으로 업그레이드
    - 서버 장애에 매우 치명적
    
    ---
    
    수평 확장
    
    - 여러대의 서버를 추가로 설치
    - 여러 서버를 한 서버처럼 사용하기 위한 클러스터링 작업 필요
        - ex) 로드 밸런서
