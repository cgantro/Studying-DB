### 1. 자기소개: 나는 어떤 사람입니까?

[문제 해결의 확장성]

주어진 제약을 피해 가기보다, 기술로 정면에서 풀어내는 엔지니어입니다.

SSAFY 프로젝트에서 AGV 알고리즘을 검증하던 당시, 실제 장비를 사용할 수 있는 시간이 매우 제한적이어서 충분한 테스트가 어려웠습니다. 처음에는 ROS나 Gazebo와 같은 상용 시뮬레이터 도입도 고려했습니다. 그러나 이 문제는 일시적인 환경 문제가 아니라, 같은 커리큘럼을 거치는 프로젝트라면 누구나 반복해서 겪게 될 구조적인 한계라고 판단했습니다.

그래서 눈앞의 과제를 넘기는 방식 대신, 바로 활용 가능한 자체 시뮬레이터를 직접 구현하기로 결정했습니다. 팀의 검증 시간을 확보하는 동시에 이후 프로젝트에서도 재사용할 수 있는 기반을 만들고자 했기 때문입니다. 시스템 구조를 처음부터 설계하며 메모리 관리 방식과 성능 병목을 직접 점검했고, 그 과정에서 단순히 동작하는 코드를 넘어 왜 이러한 구조가 필요한지까지 이해할 수 있었습니다. 그 결과 장비가 없는 환경에서도 테스트를 할 수 있었고, 후배들도 활용할 수 있는 가능성을 남겼습니다.

- 2. 본인의 과거 경험과 핵심역량을 바탕으로 지원하신 직무를 성공적으로 수행해낼 수 있는 본인만의 차별화된 강점을 구체적 사례와 함께 기술해주십시오
    
    [프로토콜 설계]
    
    RobotPal 프로젝트에서는 시뮬레이터와 서버 간 통신을 담당하는 데이터 송수신 모듈을 설계하고 구현했습니다.
    
    개발 초기, 대용량 이미지 데이터를 TCP 기반으로 전송하면서 패킷이 분리되거나 여러 개가 한 번에 수신되는 현상으로 인해 파싱 오류가 빈번하게 발생했습니다. 상용 네트워크 라이브러리를 그대로 적용하기 어려운 자체 개발 환경이었기 때문에, 임시적인 예외 처리로 문제를 넘기는 대신 근본적인 해결이 필요하다고 판단했습니다.
    
    이에 클라이언트(C++)와 서버(Python) 간 통신 방식을 Length-Prefix 구조로 통일했습니다. 모든 데이터 앞에 4바이트 길이 정보를 명시하고, 수신 측에서는 asyncio기반 비동기 스트림 버퍼링을 통해 헤더에 정의된 크기만큼 데이터가 완전히 수신된 경우에만 디코딩하도록 구현했습니다. 이를 통해 TCP 특성으로 인한 데이터 경계 문제를 안정적으로 해결할 수 있었습니다.
    
    ### [병렬 처리]
    
    실시간 영상 스트리밍 기능 구현 중 발생한 전송 지연 문제를 해결하여 성능을 최적화했습니다. 초기 개발 단계에서는 고해상도 영상 전송 시 프레임 저하가 심해 평균 15FPS 수준에 머물렀습니다.
    
    1차적으로 네트워크 대역폭과 메모리 복사 비용을 줄이기 위해 JPEG 압축 전송 방식을 도입했습니다. 그러나 단일 스레드 환경에서 이미지를 인코딩하는 연산 비용 자체가 커지면서, 데이터 크기는 줄었지만 전체 처리 시간은 단축되지 않는 한계를 확인했습니다.
    
    이를 개선하기 위해 2차적으로 SIMD 최적화가 적용된 libjpeg-turbo 라이브러리를 도입하여 인코딩 속도 자체를 높였습니다. 하지만 인코딩과 네트워크 전송이 하나의 실행 흐름에서 순차적으로 처리되는 구조 탓에, 부하가 누적되어 실시간성을 확보하기 어려웠습니다.
    
    따라서 이 문제를 단순한 라이브러리 성능 이슈가 아닌 파이프라인 구조의 병목으로 재정의하고, 아키텍처를 멀티스레딩 기반으로 전면 재설계했습니다. CPU 코어 수에 맞춰 스레드 풀을 구성한 뒤, 인코딩 작업과 네트워크 전송 작업을 분리하는 Producer-Consumer 패턴을 적용했습니다. 두 작업 사이에 큐를 두어 속도 차이를 완충하고 병렬 처리가 가능하도록 구현한 결과, 병목 현상을 해소하며 60FPS 이상의 안정적인 스트리밍 성능을 달성했습니다.
    
    [통신 규격 표준화]
    
    마지막으로 제어팀(C++)과 비전팀(Python) 간 협업 과정에서 반복적으로 발생하던 통신 오류를 해결하기 위해 통신 규격을 정리하고 표준화했습니다.
    
    서로 다른 언어 환경에서 데이터 형식이 통일되지 않아 제어 명령이 누락되거나 의도와 다르게 해석되는 문제가 지속적으로 발생했습니다. 이를 해결하기 위해 로봇 제어 입력을 주행 제어와 관절 제어로 구분하고, 각 명령에 반드시 포함되어야 할 필수 필드인 type, id, angle, speed를 명확히 정의한 JSON 기반 통신 규격을 직접 수립했습니다.
    
    이후 Python 서버에서는 해당 규격을 따르는 JSON 데이터만 생성하도록 제한하고, C++ 엔진에서는 이를 기준으로 파싱 로직과 제어 로직을 연결하도록 구조를 고정했습니다. 그 결과 통신 오류를 사전에 차단할 수 있었고, 팀 간 협업 과정에서 발생하던 불필요한 혼선 또한 크게 줄일 수 있었습니다.
    
    마주한 문제를 임시로 넘기기보다 근본적인 원인을 해결해 온 경험을 바탕으로, 안정적이고 확장 가능한 서버 개발을 통해 서비스 품질 향상에 기여하고자 합니다.
