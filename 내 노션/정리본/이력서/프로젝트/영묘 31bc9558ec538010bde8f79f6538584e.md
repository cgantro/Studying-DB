# **🎙️ 음성 채팅 시스템 전면 개편 보고서 (WebSocket → UDP 및 품질 개선)**

기존 시스템이 안고 있던 구조적인 한계점(WebSocket의 지연, Raw PCM의 대역폭 한계)을 극복하기 위해 **네트워크 통신 규격부터 코덱, 3D 사운드 렌더링까지 전 영역에 걸쳐 진행된 대대적인 리팩토링 및 픽스 내용**을 총정리했습니다.

---

## **1. 네트워크 통신 구조 대공사 (WebSocket → UDP 전환)**

- **문제 상황 (TCP의 한계):** 기존 음성 채팅은 캐릭터 이동 정보 등과 함께 WebSocket(TCP 기반)으로 전송되었습니다. TCP는 데이터가 순서대로 오지 않거나 단 하나라도 유실되면, **해당 패킷을 다시 받을 때까지 뒤에 쌓인 모든 데이터를 멈추는 "Head-of-Line Blocking"** 현상이 있습니다. 이로 인해 오디오가 수 초 단위로 밀리거나 로봇처럼 심하게 끊기는 현상(버퍼링 랙)이 발생했습니다.
- **해결 및 변경 사항:**
    - **[서버 측 변경]**: 기존 WebSocket 서버와 병렬로 동작하는 **순수 UDP 릴레이 서버** 포트를 신설했습니다. 음성 데이터는 서버에서 무거운 파싱이나 검증 과정 없이, 단순 헤더(방 번호, 화자 ID)만 확인하고 즉각 브로드캐스트(Relay)하도록 가볍게 설계했습니다.
    - **[클라이언트 측 변경]**: 멀티플레이어 통신 모듈을 분리하여 `GameNetworkSubsystem` 및 비동기 `UDPHandler` 클래스를 신설했습니다.
- **기대 효과:** 중간에 패킷 하나가 유실되더라도 집착하지 않고 버린 뒤 무조건 **"가장 최신 목소리"**를 재생시켜, 통신 환경이 나빠도 절대 음성이 밀리지(Delay) 않는 **실시간 VoIP** 환경이 구축되었습니다.

## **2. 업계 표준 코덱 도입 (Raw PCM → libOpus 압축)**

- **문제 상황 (대역폭 병목):** 기존 방식은 압축되지 않은 날것의 PCM 데이터를 초당 `256,000` 비트씩 때려 박았습니다. 사람이 많아지면 네트워크 대역폭 한계로 게임 전체 랙을 유발할 수준이었습니다. (언리얼 내장 `IVoiceEncoder`를 시도했으나, 스트림 기반이라 UDP 환경에서 순서가 섞이면 `Header Corrupted` 에러를 뿜으며 소리가 박살났습니다.)
- **해결 방식:**
    - 코덱 모듈(`VoiceCodec`)을 갈아엎고 **Raw `libopus` API (`opus_encode`, `opus_decode`)**를 직접 호출하도록 C++ 레벨에서 전면 재구축했습니다.
    - 패킷 하나하나가 다른 패킷에 의존하지 않는 독립적(Stateless) 구조를 채택해 UDP 패킷 로스에 강력해졌습니다.
    - 자체 프레이밍(`[2바이트 길이][Opus Payload]`)을 도입하고 코덱의 **FEC(오류 정정) 및 패킷 로스 대처 옵션**을 활성화했습니다.
- **결과:** 음성 전송량이 초당 `24,000` 비트로 줄어 **대역폭 사용량이 10분의 1로 압축**되었습니다.

## **3. 오디오 파편화 및 묵음 지글거림 픽스**

- **문제 상황 (Buffer Starvation):** 마이크 캡처 함수가 일정하지 않은 크기(예: 1000바이트, 256바이트)의 오디오 쪼가리를 내주는 탓에, 규격에 모자란 용량을 **무음(Zero)으로 패딩**해서 압축하는 바람에 목소리 사이사이에 묵음과 틱틱거리는 노이즈가 섞였습니다.
- **해결 방식:**
    - 인코더 내부에 **PCM 누적 링버퍼(`PCMBuffer`)** 아키텍처를 도입했습니다.
    - 조각난 데이터가 들어오면 무조건 버퍼에 누적시키고, 정확히 **1프레임 규격 크기(20ms)**가 모였을 때만 두부 썰듯 잘라서 인코딩을 수행해 무음 이빨 빠짐 현상을 원천 차단했습니다.

## **4. 성대모사 랙(Buffer Bloat) 및 무한 루프 폭주 방지**

- **문제 상황:** 무전기처럼 마이크를 열고 길게 대화하면, 캡처 스레드와 게임 로직 간의 틱 속도 차이로 어느 순간 캡처 버퍼 공간이 초과(`Resetting UncompressedAudioBuffer`)되면서 수 초간 마이크가 완전히 먹통이 되거나 극심한 랙이 발생했습니다.
- **해결 방식:**
    - `VoiceCodec` 내부에 **안전장치(Safeguard)**를 추가했습니다.
    - 통신 지연으로 인해 버퍼에 **1초 분량(32,000 Bytes)** 이상의 데이터가 적체되면, 영구적인 밀림을 막기 위해 과거 데이터를 즉시 쳐내서(Drop) 랙 유발과 인코딩 지연 폭주를 막았습니다.

## **5. 하이톤 변조(헬륨가스) 및 재생 속도 오류 해결**

- **문제 상황:** 마이크로 말한 목소리가 3배 빠르게 감기며 하이톤으로 들리고 뚝뚝 끊겼습니다.
- **해결 방식:**
    - 언리얼 기본 마이크 캡처는 **16,000Hz**로 녹음되는데 반해, Opus 코덱과 재생 프로시저(`USoundWave`) 규격은 **48,000Hz**로 엇갈려 있었습니다.
    - 네트워크 파이프라인 전체(캡처, 인코딩, 디코딩, 스피커)의 기준 주파수를 시스템 캡처 스펙인 **16,000Hz**로 일괄 통일하여 본래의 자연스러운 목소리 톤과 속도로 복구했습니다.

## **6. 오디오 공간화 (3D Spatialization) 및 감쇠 세팅**

- **문제 상황:** 기존 로직은 플레이어 간 거리를 수학으로 계산해서 특정 거리가 넘으면 하드 코딩으로 사운드를 Mute 처리했습니다. 가까이 있어도 100% 볼륨으로 때려박혀 거리감과 방향성이 체감되지 않았습니다.
- **해결 방식:**
    - `VoiceAudioReceiver`에 언리얼 사운드 엔진 고유 클래스 **`FSoundAttenuationSettings`*를 적용했습니다.
    - 가까운 거리(`InnerRadius: 200`)에서는 100%로 들리고, 대상과 멀어질수록 점진적으로 자연스럽게 볼륨이 감소하다(`Linear Algorithm`) 최대 거리(`FalloffDistance: 1800`)에서 완전히 소멸하도록 변경하여 현실적인 3D 보이스를 완성시켰습니다.

## **7. 로컬 지연(Latency) 한계 최소화 및 시스템 버그 픽스**

- **문제 상황 및 해결:**
    - **응답성 극대화:** 백그라운드 스레드에서 UDP 수신을 대기하는 `FUdpSocketReceiver` 주기를 `10ms`에서 **`2ms`*로 대폭 단축해 게임 스레드로 넘어가는 응답 지연을 최대한 단축했습니다.
    - **루프 글리치 픽스:** `USoundWaveProcedural` 큐에 데이터가 미세하게 늦게 들어올 때 이전 소리가 반복되어 끊기는 것처럼 들리는 증상이 있어, 해당 컴포넌트의 `bLooping = false`로 강제 오프(Off)하여 사운드 버그를 방지했습니다.
    - **초기화 에러 픽스:** 시스템 마이크 장치 조회 시 0x88780078 발생을 방지하기 위해 검색 문자열을 강제 초기화(`""`) 하고 WASAPI 사운드 믹서를 타도록 **Build.cs** 의존성을 정리했습니다.

## **2. 🏗️ 적용된 CS 개념 및 객체지향/아키텍처 설계 원칙 (OOP)**

- **단일 책임 원칙 (SRP, Single Responsibility Principle)** 가장 집중적으로 개선한 원칙입니다. **LobbyServer**에 몰려있던 책임을 쪼개어, **세션 관리**(**LobbyClientManager**), **메시지 분배**(**LobbyPacketRouter**), **기능별 로직 처리**(**RoomHandler**, **GameHandler**, **ChatHandler**)로 역할을 명확하게 구분했습니다.
- **의존성 역전 원칙 (DIP, Dependency Inversion Principle) 및 결합도 완화** 데이터 모델인 **Player**나 **RoomManager**가 구체적인 특정 라이브러리의 클래스(`uWS::WebSocket`)를 직접 참조하지 않도록 했습니다. 대신 추상화된 함수 포인터(`std::function` 형태의 **SendFunc**)만 들고 있게 변경하여 도메인 모델과 네트워크 계층 간의 결합을 끊어냈습니다.
- **다형성 (Polymorphism)에 기반한 확장성** **IServer**와 같은 공통 추상 클래스를 상속받게 하여 메인 루프(**main.cpp**)에서는 내부 로직을 신경쓰지 않고도 서버 인터페이스를 다룰 수 있게 구성했습니다.

## **3. 🧩 활용된 디자인 패턴 (Design Patterns)**

- **라우터 & 핸들러 패턴 (Router-Handler Architecture / MVC Controller 패턴)** 들어오는 모든 요청(Packet)을 중앙의 라우터(**LobbyPacketRouter**)가 파싱하여 타입(type)을 분석한 뒤, 해당 임무를 전담하는 외부 전문가(Handler: **RoomHandler** 등)에게 위임하는 방식입니다. Spring MVC나 Express.js 등에서 흔히 볼 수 있는 웹 서버 표준 아키텍처를 C++ 소켓 서버에 접목했습니다.
- **Pimpl 이디엄 (Pointer to Implementation / Opaque Pointer)** **LobbyPacketRouter** 헤더에서 다른 핸들러 클래스의 존재를 철저히 숨기기 위해, 헤더에는 껍데기만 두고 실제 구현부(**Impl**) 포인터만을 선언했습니다. 이를 통해 코드가 수정되어도 이를 참조하는 다른 수많은 소스들이 **다시 컴파일되는 것을 막는 'C++의 방화벽(Compile Firewall)'** 역할을 하도록 고도화 시켰습니다.

## **4. 💻 C++ 문법과 언어적 최적화 포인트**

- **`std::function` & `Lambda Expression` (모던 C++ 활용)** Map(`std::unordered_map`) 구조체 안에 동적인 람다 함수들을 보관해두고, Key값 콜 호출 기법으로 If/Switch 문의 늪에서 벗어났습니다. 콜백 지향적 언어 특징을 백분 활용한 설계입니다.
- **스마트 포인터 (`std::unique_ptr`) & RAII 원칙** 객체의 생명 주기를 엄격히 관리하고 메모리 누수(Memory Leak)를 원천 차단하기 위해, Pimpl 패턴에서 원시 포인터() 대신 스마트 포인터를 통해 자동 소멸이 이뤄지도록 구현했습니다.
- **헤더와 파일 구조의 엄격한 분리** 프로토타이핑 단계에서는 클래스 블록 안에 함수를 같이 구현하는 Inline 방식이 편할 수 있으나, 프로젝트 덩치가 커지면 빌드 속도 저하와 순환 참조 에러를 유발합니다. 이번에 모든 클래스를 **.h**와 **.cpp**로 분리하여 코드의 구조적 성숙도를 올렸습니다.

2. **글로벌 에러 코드 시스템 확립** **sendError(ws, "에러 메시지")** 같은 하드코딩된 스트링 대신, Enum 기반의 통합 ErrorCode 시스템을 만들어 클라이언트 개발팀과 명세화(Documentation) 해야 합니다.
3. **Thread-Safety 및 동시성 문제 파악** 현 단계는 uWebSockets의 싱글 스레드 이벤트 루프 덕에 Data Race에서 자유로운 편입니다. 하지만 향후 병렬 DB 요청 스레드가 결합되거나, 멀티스레드 기반 물리 처리 로직이 들어올 경우 전역 메모리(`std::unordered_map` 등)의 동시 쓰기 문제를 방지하기 위해 `std::shared_mutex`나 Lock 시스템 도입 논의가 필요합니다.
4. **Heartbeat 및 Zombie Connection 관리** 갑자기 랜선이 뽑히거나 비정상 접속 종료 시 클라이언트 연결을 완전히 Clean-Up 하고, 핑퐁(Ping-Pong)을 활용해 "유령 방(Zombie Room)" 및 권한 위임 체계를 더 깐깐하게 정립해야 합니다.

### 03-12

프로젝트 구조는 Unreal Dedicated Server 단독형이 아니라, 외부 사설 C++ 서버 + Unreal 게임 서버/클라이언트의 하이브리드 구조로 보는 것이 맞다.

외부 사설 C++ 서버가 담당하는 역할:

- Room 생성/입장
- 텍스트 채팅
- 보이스 채팅 서버
- 게임 시작 가능 여부 판정

Unreal 쪽이 담당하는 역할:

- 실제 월드 시뮬레이션
- 캐릭터 스폰/빙의
- 아이템 획득/사용
- 전투/데미지
- 복제(Replication) 기반 인게임 상태 동기화

즉, 로비/보이스/채팅/게임 시작은 외부 서버가 맡고, 실제 인게임 authoritative gameplay는 Unreal 쪽이 맡는 구조다.

**현재 구조에서 괜찮은 점**

- A302.Target.cs, A302Server.Target.cs, A302Editor.Target.cs가 존재한다.
- A302Server.Target.cs에서 TargetType.Server를 사용하고 있다.
- GameInstance + WebSocket/UDP 기반으로 외부 서버와 연동하는 구조는 하이브리드 아키텍처 관점에서 타당하다.
- RPC, HasAuthority(), GetNetMode(), GameState, PlayerState 복제 구조가 일부 존재한다.
- 외부 서버를 통해 Room/Chat/Voice/GameStart를 관리하려는 방향 자체는 맞다.

**현재 구조의 핵심 문제**

- 외부 서버를 쓰는 것은 문제 없지만, 인게임 authoritative gameplay 경계가 아직 약하다.
- 단일 A302 모듈 안에 UI, Input, Voice Capture, Gameplay, Authority 로직이 섞여 있다.
- GameMode, Character, Component에서 클라이언트 전용 코드와 서버 전용 코드가 명확히 분리되지 않았다.
- Dedicated Server 대응 여부와 별개로, “인게임 로직은 Unreal 서버가 최종 판정해야 한다”는 원칙이 아직 완전히 지켜지지 않는다.

**특히 주의할 부분**

- 아이템 지급이 일부 클라이언트 RPC 기반으로 처리된다.
- 인벤토리/QuickSlot/Shield 상태가 복제 중심이라기보다 로컬 상태 중심으로 보인다.
- 공격 사용과 데미지 처리 흐름이 로컬 입력에서 바로 이어지는 부분이 있다.
- InteractComponent, VoiceCaptureProcessor, 일부 UI 생성 코드가 공용 gameplay 코드에 들어가 있다.
- GameMode가 UI 참조를 직접 갖고 있는 부분은 역할 분리상 좋지 않다.

**정리된 판단**

- 외부 사설 서버 연동 구조 자체는 설계상 맞다.
- 따라서 GameInstance에서 WebSocket 연결, Voice 서버 직접 연결, 채팅 JSON 송신 같은 부분은 단점으로 단정하면 안 된다.
- 하지만 Unreal 내부에서는 여전히 다음 원칙이 필요하다:
- 외부 서버는 세션/보이스/로비 담당
- Unreal 서버는 인게임 authoritative gameplay 담당
- 클라이언트는 UI/Input/로컬 연출 담당

**나중에 꼭 정리해야 할 우선순위**

1. 인게임 판정 로직을 Unreal 서버 authoritative로 명확히 고정
2. 인벤토리/QuickSlot/Shield 같은 핵심 상태를 복제 기준으로 재정리
3. UI/Input/Voice Capture를 gameplay 공용 코드에서 분리
4. GameMode에서 UI 참조 제거
5. 가능하면 장기적으로 Shared / Client / Server 성격으로 모듈 구조 분리

    ## **2. 프로젝트 개요**
    
    이 프로젝트는 Unreal Engine 기반 멀티플레이 환경에서, 기존 구조의 한계를 해결하기 위해 서버 아키텍처와 프로젝트 구조를 함께 개선한 작업이다. 목표는 단순히 “기능이 동작하는 상태”가 아니라, 실제 운영 단계에서 문제가 되는 세 가지 축을 동시에 개선하는 것이었다.
    
    - 축 1: 단일 물리 서버에서 여러 게임 세션을 안정적으로 운영할 수 있는가
    - 축 2: 서버/클라이언트 경계를 분리해 빌드 및 배포 비용을 줄일 수 있는가
    - 축 3: 실시간 보이스를 세션 격리와 결합해 비용과 품질을 함께 관리할 수 있는가
    
    이 문제는 기술적으로 중요한 이유가 명확하다. 멀티플레이 프로젝트에서 세션 경계가 약하면 게임 로직과 통신 데이터가 서로 오염될 수 있고, 서버 타깃 의존성이 비대하면 빌드/배포 속도와 안정성이 함께 떨어진다. 또한 보이스는 전송량과 실시간성이 직접 충돌하는 영역이기 때문에, 기능 구현만으로는 운영 가능한 품질을 확보하기 어렵다.
    
    따라서 이 프로젝트는 기능 추가가 아니라, **운영 가능한 시스템 경계를 재설계한 프로젝트**로 정의할 수 있다.

    ## **3. 배경과 기존 한계**
    
    기존 구조의 핵심 문제는 “기능은 존재하지만 경계가 없다”는 점이었다.
    
    - 서버와 클라이언트 책임이 코드 레벨에서 혼재
    - 모듈 경계가 약해 변경 영향 범위가 넓음
    - 권한 판정, UI, 입력, 네트워크 처리가 명확히 분리되지 않음
    - 서버 전용 배포물 최소화가 어려움
    - 분리되지 않은 의존성 때문에 서버 타깃이 불필요한 모듈까지 포함
    - 세션 분리 기준이 약해 보이스/게임 로직 교차 오염 위험 존재
    - 글로벌 보이스 구조의 비효율: 전체 전달로 인한 네트워크 비용 증가 및 문맥 외 음성 재생
    
    이 한계는 개발 단계에서는 “당장 동작”으로 가려지지만, 운영 단계에서 비용과 리스크로 드러난다.
    
    특히 서버 관점에서 보면, 코드가 한 덩어리로 묶여 있으면 특정 변경이 서버 빌드 파이프라인 전체에 파급되고, 이는 개발 생산성 저하뿐 아니라 배포 안정성 저하로 이어진다.

    ## **4. 내가 맡은 역할**
    
    내 역할은 기능 단위 구현자가 아니라 **경계 설계자**에 가까웠다.
    
    실제로 담당한 핵심 책임은 다음과 같다.
    
    - 사설 서버 구조 개선 방향 수립: 단일 물리 서버 기반 다중 세션 운영 구조 설계
    - 룸 코드 기반 논리 분리 설계: 게임 상태와 보이스 라우팅의 공통 키 정의
    - Unreal 구조 분리 실행: A302 / A302Client / A302Server / A302Shared 재편
    - Shared 계층 책임 재정의: 공통 요소를 유지하되 서버/클라이언트 실행 책임 분리
    - 보이스 경로 표준화: Capture부터 Playback까지 단계별 필터 구조 정의
    - Opus 도입 및 전송 단위 설계: 16kHz/mono/20ms 프레임 기반 전송 모델 적용
    - 거리 기반 보이스 정책 반영: 몰입감과 네트워크 비용 동시 관리
    - 보이스 서버 동시성 개선: 룸 샤딩 + 워커 풀(2) 구성 적용
    
    즉, 단순 구현 목록이 아니라 “어떤 경계를 어떻게 나누고, 그 경계가 운영 결과를 어떻게 바꾸는지”를 책임지는 역할을 수행했다.

    ### **5-2. 빌드 경계 변화**
    
    구조 분리는 빌드 파이프라인 수치로도 확인된다.
    
    - 서버 타깃 빌드 대상 파일 수: 156 -> 132 (감소율 -15.38%)
    - 서버 타깃 의존 모듈 집합: 19 -> 11 (감소율 -42.11%)
    
    분리 후 서버 타깃 빌드 대상 파일 구성은 다음과 같다.
    
    - A302 = 3
    - A302Server = 30
    - A302Shared = 99
    
    제거된 주요 의존성은 아래와 같다.
    
    - AudioCapture
    - AudioMixer
    - Voice
    - libOpus
    - NetCore
    - Networking
    - Sockets
    - Niagara
    
    이 수치는 “서버가 필요 없는 기능 의존성에서 벗어났다”는 구조적 증거다.
    
    ### **5-3. 서버 측 세션 구조 개선**
    
    사설 서버는 물리 자원 증설보다 먼저 논리 분리를 선택했다.
    
    핵심 방향은 단일 물리 서버 위에서 다중 게임 세션을 독립 운영하는 것이다.
    
    - 룸 코드 기반 논리 분리
    - 플레이어-룸 매핑을 멤버십 레지스트리로 관리
    - 룸 런타임 lifecycle 관리
    - 유휴 세션 정리 고려
    
    즉, 서버를 많이 띄우는 방식이 아니라, **같은 서버 안에서 경계를 안전하게 나누는 방식**으로 운영 효율을 확보했다.
    
    ### **5-4. 룸 코드 기반 공통 키 설계**
    
    룸 코드를 게임/보이스 공통 키로 사용한 이유는 단일 기준 강제에 있다.
    
    - 게임 상태 라우팅 기준과 보이스 라우팅 기준 일치
    - 세션 분리 기준의 불일치로 인한 오염 가능성 축소
    - 디버깅 시 “세션 단위” 추적 가능성 향상
    
    룸 코드가 공통 키가 되면, 게임 로직과 통신이 동일 경계에서 움직이기 때문에 운영 관점의 예측 가능성이 크게 높아진다.

    ## **6. 보이스 시스템 상세**
    
    ### **6-1. 보이스 데이터 흐름 표준화**
    
    보이스 경로를 단계별로 명시했다.
    
    - Capture
    - Opus Encode
    - Session Filter
    - Distance Filter
    - Decode / Playback
    
    구체 흐름은 아래와 같다.
    
    - PCM 수집(20ms 프레임)
    - Opus 인코딩
    - 룸 코드 필터
    - 거리 기반 대상 선정
    - 수신측 Opus 디코드
    - 재생
    
    핵심은 “보이스를 전송한다”가 아니라, **어떤 필터를 어떤 순서로 통과시키는지**를 고정한 것이다.
    
    ### **6-2. 코덱 선택과 수치**
    
    보이스 코덱 기준:
    
    - 16kHz
    - mono
    - 20ms frame
    - 320 samples
    
    비트레이트 기준:
    
    - PCM 기준: 256kbps
    - Opus 목표: 24kbps
    - 프레임 길이 헤더 오버헤드: 2B * 50pps = 0.8kbps
    
    절감 효과:
    
    - 애플리케이션 페이로드 기준 절감: 약 90.31%
    - UDP/IP + custom header 포함 wire 기준 절감: 약 80.17%
    
    이 수치는 보이스를 “고품질 기능”이 아니라 **운영 가능한 네트워크 비용 모델**로 다뤘다는 근거다.
    
    ### **6-3. 거리 기반 보이스 정책**
    
    거리 기반 보이스는 UX 기능이면서 동시에 비용 정책이다.
    
    - 글로벌 전달 대신 근접 전달
    - 전달 대상 수 축소로 네트워크 비용 제어
    - 플레이 문맥과 맞는 음성만 재생되어 몰입감 증가
    
    즉, 거리 기반 정책은 “재미”와 “비용”을 분리하지 않고 동시에 해결하는 설계다.
    
    ### **6-4. 상태 기반 청취 정책**
    
    보이스 정책에는 상태(생존/사망/탈출)도 반영했다.
    
    - 단순 거리 조건만으로는 게임 룰과 충돌 가능
    - 상태 조건을 포함해 실제 게임 규칙에 맞는 청취 정책 구성
    
    이로써 보이스가 독립 기능이 아니라 게임 규칙의 일부로 동작하게 된다.
    
    ### **6-5. 세션 격리 방식**
    
    세션 격리 없는 보이스는 운영 리스크가 크다.
    
    해결 방식은 다음과 같다.
    
    - 보이스 패킷 헤더에 룸 코드 포함
    - 룸 코드 기준으로 송수신 필터 적용
    - 세션 간 음성 교차를 구조적으로 차단

    ## **7. 동시성/운영 구조**
    
    보이스 서버는 동시 처리 경합 완화를 위해 룸 샤딩을 적용했다.
    
    - 샤딩 방식: hash(roomCode) % 2
    - 워커 수: 2
    
    의도는 단순 병렬화가 아니라, **같은 룸 트래픽을 같은 처리 축으로 모아 경합을 줄이는 것**이다.
    
    추가로 운영 구조에서는 lifecycle을 포함했다.
    
    - 멤버십 레지스트리 기반 플레이어-룸 추적
    - 룸 런타임 관리
    - 유휴 세션 정리 고려
    
    즉, 동시성 처리와 세션 수명 관리를 분리하지 않고 하나의 운영 모델로 다뤘다.

    ## **9. 설계 의사결정 근거 (WHY 중심)**
    
    ### **9-1. 왜 Client / Server / Shared로 분리했는가**
    
    문제의 본질이 “기능 부족”이 아니라 “경계 부재”였기 때문이다.
    
    클라이언트/서버 책임이 섞인 상태에서는 기능이 늘어날수록 변경 영향이 넓어지고, 서버 전용 배포 최적화가 불가능하다.
    
    Shared는 공통 요소를 담되, 실행 책임은 Client/Server가 갖도록 분리해야 런타임 책임이 선명해진다.
    
    대안은 기존 모놀리식 유지 + 규칙 기반 개발이었지만, 이는 시간이 지나며 규칙 위반과 경계 침범이 누적될 가능성이 높다. 구조 분리는 강제력이 있는 해결책이다.
    
    ### **9-2. 왜 이것이 단순 리팩토링이 아니라 빌드 파이프라인 최적화인가**
    
    폴더만 나눈 것이 아니라 서버 타깃 입력과 의존 집합이 실제로 줄었다.
    
    156 -> 132, 19 -> 11이라는 수치는 빌드 파이프라인의 복잡도가 낮아졌음을 의미한다.
    
    즉, 구조 변경이 곧 빌드 비용 구조 변경으로 이어졌다.
    
    ### **9-3. 왜 룸 코드가 게임 로직과 보이스 라우팅의 공통 키여야 하는가**
    
    게임 로직과 보이스가 서로 다른 기준으로 라우팅되면 세션 오염 위험이 생긴다.
    
    공통 키를 강제하면 “같은 세션 경계”를 공유하므로 교차 오염 가능성을 구조적으로 낮출 수 있다.
    
    운영 디버깅도 키 하나로 추적 가능해진다.
    
    ### **9-4. 왜 단일 물리 서버에서 논리 세션 분리가 먼저 필요한가**
    
    초기 운영 단계에서는 물리 서버 증설보다 구조 효율이 우선이다.
    
    논리 분리를 먼저 확보하면 같은 자원에서 더 안정적으로 세션을 운영할 수 있고, 이후 물리 확장은 이 구조 위에서 단계적으로 수행할 수 있다.
    
    ### **9-5. 왜 Opus가 적절한 선택이었는가**
    
    실시간 보이스는 대역폭과 지연의 균형이 핵심이다.
    
    PCM(256kbps)은 비용이 크고 세션 수 증가 시 부담이 급격히 커진다.
    
    Opus 목표 24kbps는 실시간성과 비용 사이에서 운영 가능한 지점을 제공한다.
    
    ### **9-6. 왜 거리 기반 보이스가 네트워크 비용과 몰입감을 동시에 해결하는가**
    
    글로벌 보이스는 비용이 크고 문맥 노이즈가 많다.
    
    거리 기반은 전달 대상 자체를 줄여 트래픽을 낮추고, 플레이어 입장에서는 주변 상황 중심 정보만 듣게 되어 몰입이 증가한다.
    
    즉, 전달 범위 정책 자체가 비용 최적화 수단이다.
    
    ### **9-7. 왜 세션 격리 없는 보이스가 운영상 위험한가**
    
    보이스는 실시간·빈번 전송 데이터라 오염 시 영향이 즉각적이고 광범위하다.
    
    게임 이벤트보다 체감 장애가 빠르게 나타나며, 재현/추적도 어렵다.
    
    패킷 단위 룸 코드 포함 및 라우팅 필터가 필수인 이유다.
    
    ### **9-8. 왜 Shared에는 최소 공통 요소만 두어야 하는가**
    
    Shared가 비대해지면 다시 모놀리식 중심이 되고, 분리 효과가 사라진다.
    
    Shared는 계약/공통 데이터/브릿지 성격에 집중하고, 실행 책임이 있는 코드(UI, 입력, 서버 권한 처리)는 각 계층에 둬야 경계가 유지된다.
    
    ### **9-9. 왜 서버 타깃 의존성 축소가 유지보수성과 운영 효율성으로 이어지는가**
    
    의존성이 많을수록 변경 전파 경로와 빌드 경로가 늘어난다.
    
    의존성 축소는 컴파일 대상 축소 이상의 의미가 있다.
    
    - 원인 추적 범위 축소
    - 릴리즈 리스크 축소
    - 서버 전용 검증 경로 단순화
    
    ### **9-10. 왜 멀티플레이에서는 논리 분리와 자원 분리를 함께 봐야 하는가**
    
    논리 분리(세션 격리)만 있고 자원 관점이 없으면 운영 병목이 생긴다.
    
    자원 분리만 있고 논리 경계가 없으면 데이터 오염이 생긴다.
    
    멀티플레이 운영성은 두 축을 동시에 맞출 때 확보된다.

    결과: 처리 경합 완화 방향 확보

    - Unreal 프로젝트를 런타임/빌드 관점에서 동시에 보는 역량
    - 실시간 통신을 품질이 아닌 비용/운영 모델까지 포함해 설계하는 역량
    - 멀티세션 운영 구조와 라우팅 기준을 데이터 키(룸 코드)로 통일하는 아키텍처 사고
    - 정량 지표(파일 수/의존성/전송량)로 개선을 설명하는 엔지니어링 커뮤니케이션 역량
    
    포트폴리오에서 이 프로젝트를 제시하면, “복잡한 시스템을 구조적으로 단순화하고 운영 가능 상태로 만드는 사람”이라는 인상을 줄 수 있다.

    ## **14. 면접 대비 포인트**
    
    ### **14-1. 예상 질문**
    
    - 왜 Client / Server / Shared 분리가 필요했나
    - Shared에 무엇을 두고 무엇을 두지 않았나
    - 서버 타깃 파일 수 감소가 실제로 어떤 의미가 있나
    - 룸 코드를 공통 키로 쓴 이유는 무엇인가
    - Opus 목표 비트레이트를 왜 24kbps로 잡았나
    - 거리 기반 보이스의 장단점은 무엇인가
    - 워커 수를 2로 둔 이유는 무엇인가
    - 세션 격리가 깨지면 어떤 운영 문제가 발생하나
    
    ### **14-2. 답변 시 강조 포인트**
    
    - “기능 추가”가 아니라 “경계 설계” 문제로 접근했다는 점
    - 런타임 분리와 빌드 분리를 함께 해결했다는 점
    - 수치와 구조를 함께 제시한다는 점
    - 룸 코드 통일로 게임/보이스 라우팅 경계를 일관화했다는 점
    - 보이스에서 품질, 비용, 몰입감을 동시에 본 설계라는 점
    
    ### **14-3. 방어가 필요한 포인트**
    
    - 빌드 시간 자체의 실측 대신 빌드 대상 파일/의존성 수치를 사용한 이유
        - 답변 포인트: 시간은 환경 편차가 크므로 구조 지표를 우선 사용했고, 구조 지표는 재현 가능성이 높음
    - Shared 비대화 위험
        - 답변 포인트: Shared를 계약 계층으로 제한하는 규칙이 필요하며, 회귀 점검이 필수
    - 워커 수 2의 근거
        - 답변 포인트: 룸 샤딩 구조를 먼저 정착시키는 단계였고, 워커 수 자체는 운영 조건에 맞춰 조정 가능한 파라미터

    ## **15. 기억 보존용 사실 목록 (재사용용 원본 데이터)**
    
    ### **15-1. 구조/커밋/규모**
    
    - 구조 분리 커밋 날짜: 2026-03-17
    - 구조 분리 커밋 변경 규모: 소스 코드 207개 파일, +6672 / -3166 lines
    - 모듈 재편: A302, A302Client, A302Server, A302Shared
    
    ### **15-2. 서버 빌드 관련 수치**
    
    - 서버 타깃 빌드 대상 파일: 156 -> 132 (-15.38%)
    - 분리 후 서버 타깃 파일 구성: A302=3, A302Server=30, A302Shared=99
    - 서버 타깃 의존 모듈 집합: 19 -> 11 (-42.11%)
    
    ### **15-3. 서버 경로에서 제거된 주요 의존성**
    
    - AudioCapture
    - AudioMixer
    - Voice
    - libOpus
    - NetCore
    - Networking
    - Sockets
    - Niagara
    
    ### **15-4. 세션/라우팅 구조**
    
    - 단일 물리 서버 기반 다중 세션 운영 구조 고려
    - 룸 코드를 게임 상태와 보이스 라우팅의 공통 키로 사용
    - 플레이어-룸 매핑을 멤버십 레지스트리로 관리
    - 룸 런타임 lifecycle 포함
    - 유휴 세션 정리 고려
    - 보이스 패킷 헤더에 룸 코드 반영
    - 세션 간 이벤트/음성 교차 오염 방지 설계
    
    ### **15-5. 보이스 데이터 흐름**
    
    - Capture -> Opus Encode -> Session Filter -> Distance Filter -> Decode / Playback
    - 상세: PCM 수집(20ms) -> Opus 인코딩 -> 룸 코드 필터 -> 거리 기반 대상 선정 -> 수신측 디코드 -> 재생
    
    ### **15-6. 보이스 코덱/대역폭 수치**
    
    - 코덱 기준: 16kHz, mono, 20ms frame, 320 samples
    - PCM 기준 전송량: 256kbps
    - Opus 목표 비트레이트: 24kbps
    - 프레임 길이 헤더 오버헤드: 2B * 50pps = 0.8kbps
    - 애플리케이션 페이로드 기준 절감: 약 90.31%
    - UDP/IP + custom header 포함 wire 기준 절감: 약 80.17%
    
    ### **15-7. 보이스 정책**
    
    - 거리 기반 보이스 적용
    - 상태(생존/사망/탈출) 기반 청취 정책 반영
    - 세션 격리 + 네트워크 비용 + 몰입감 동시 고려
    
    ### **15-8. 동시성/운영**
    
    - 보이스 서버 룸 샤딩 방식: hash(roomCode) % 2
    - 워커 풀 수: 2
    - 목적: 동시 처리 경합 완화
    
    ### **15-9. 기존 문제 정의 키워드**
    
    - 모놀리식 구조
    - 구조 분리 부재
    - 의존성 과다
    - 런타임 경계 불명확
    - 빌드 경계 불명확
    - 서버 배포물 최소화 어려움
    - 세션 격리 약함
    - 글로벌 보이스 비효율
