## 📂 프로젝트 개요: RobotPal (JETANK 로봇 시뮬레이터)

### 1. 프로젝트 한 줄 요약

**"물리적 제약 없이 로봇 제어 알고리즘을 검증할 수 있는 크로스 플랫폼(Web/Desktop) 기반의 JETANK 로봇 시뮬레이션 환경 개발"**

### 2. 핵심 기술 스택 (Tech Stack)

- **Core Engine:** C++17, CMake
- **Architecture:** ECS (Entity Component System) - `flecs` 라이브러리 활용
- **Web Porting:** Emscripten (C++ → WebAssembly)
- **Networking:** WebSocket (Web), TCP (Desktop), Python `asyncio`
- **Client SDK:** Python (JetBot API 호환), Traitlets

### 3. 주요 기술적 성과 (자소서/면접 대비용)

그래픽 구현을 제외하고, 시스템 엔지니어링 관점에서 강조할 수 있는 4가지 핵심 포인트입니다.

### A. 고성능 비동기 네트워크 아키텍처 설계

> "실시간 영상 스트리밍과 제어 명령의 지연(Latency)을 최소화하기 위해 수신과 처리를 분리한 비동기 네트워크 구조를 설계했습니다."
> 
- **문제:** 시뮬레이터에서 보내는 대용량 이미지 데이터와 클라이언트의 제어 명령이 섞일 경우 병목 현상 발생.
- **해결:**
    - **Python 서버 (`server.py`):** `asyncio`를 활용하여 네트워크 수신부(`Receiver`)와 데이터 처리부(`Processor`)를 분리하고, 내부 `Queue`를 통해 데이터를 전달하는 **Producer-Consumer 패턴**을 적용했습니다.
    - **Ring Buffer & 패킷 처리:** TCP 통신 시 패킷이 쪼개지거나 합쳐지는 현상(TCP Stickiness)을 해결하기 위해, 헤더(4바이트 길이 정보)를 포함한 커스텀 프로토콜을 정의하고 링 버퍼(Ring Buffer) 방식으로 데이터를 완벽하게 파싱했습니다.
    - **C++ 네트워크 엔진 (`NetworkEngine.cpp`):** 전송 계층을 추상화(`NetworkTransport`)하여 상황에 따라 TCP 또는 WebSocket을 유연하게 교체할 수 있도록 설계했습니다.

### B. C++ 엔진의 웹(WebAssembly) 포팅 및 크로스 플랫폼 지원

> "설치가 복잡한 시뮬레이터의 접근성을 높이기 위해 Emscripten을 활용하여 C++ 엔진을 웹 브라우저에서 실행 가능한 WebAssembly로 빌드했습니다."
> 
- **구현:** `CMakeLists.txt`와 Emscripten 툴체인을 연동하여, 동일한 C++ 소스 코드가 윈도우/맥/리눅스 네이티브 앱뿐만 아니라 웹 환경에서도 빌드되도록 구성했습니다.
- **성과:** 별도의 설치 과정 없이 URL 접속만으로 시뮬레이션을 실행할 수 있는 환경을 구축하여 사용자 접근성을 극대화했습니다.

### C. ECS (Entity Component System) 기반의 모듈형 설계

> "객체 지향 상속의 복잡성을 피하고 성능과 유지보수성을 높이기 위해 ECS 아키텍처를 도입했습니다."
> 
- **구현:** 로봇의 `Position`, `Rotation`, `Controller` 등을 컴포넌트로 분리하고, 이를 처리하는 로직을 `System`(`ControllerSystemModule.cpp` 등)으로 구현했습니다.
- **장점:** 네트워크 모듈, 물리 계산 모듈, 로봇 제어 모듈이 서로 강하게 결합되지 않고 독립적으로 동작하여, 새로운 기능을 추가할 때 기존 코드에 영향을 주지 않는 유연한 구조를 완성했습니다.

### D. 하드웨어 추상화 및 실제 로봇 API 호환성 확보

> "사용자가 시뮬레이션 코드를 실제 로봇에 그대로 적용할 수 있도록, 실제 하드웨어 제어 라이브러리와 동일한 인터페이스를 제공했습니다."
> 
- **구현:** Python 측에서 `traitlets` 라이브러리를 사용하여 값의 변경을 감지하고 자동으로 네트워크 패킷을 전송하는 반응형(Reactive) 구조를 설계했습니다.
- **로직:** `robot.py` 클래스를 통해 사용자는 `robot.left_motor.value = 0.5`와 같이 직관적인 코드로 로봇을 제어할 수 있으며, 이는 내부적으로 JSON 패킷으로 변환되어 C++ 엔진으로 전송됩니다.

## 📝 코드 구조 요약 (Remind용)

자소서 작성 시 구체적인 파일명이나 로직을 언급해야 할 때 참고하세요.

1. **Python 클라이언트 (`RobotPal-python/`)**
    - `src/robotpal/_core/server.py`: **네트워크의 핵심**. `asyncio`로 비동기 루프를 돌며 이미지 디코딩(cv2)과 명령 송신을 담당. 큐(Queue)를 써서 버퍼링 구현.
    - `src/robotpal/robot.py`: 사용자 인터페이스. 모터 값을 설정하면 `SimulatorServer`를 통해 패킷을 전송.
2. **C++ 엔진 (`RobotPal/src/`)**
    - `Network/NetworkEngine.cpp`: 시뮬레이터 측 네트워크 관리자. 연결 상태 관리 및 수신된 패킷을 큐에 적재.
    - `Systems/ControllerSystemModule.cpp`: **제어 로직의 핵심**. 네트워크로 받은 JSON 패킷을 파싱하여 로봇의 속도(`Move`)나 관절 각도(`Servo`)를 실제로 반영. 관절 회전축(Axis) 계산 로직 포함.
    - `SimController.cpp`: **물리/이동 로직**. 목표 속도까지 부드럽게 가속/감속하는 보간(Interpolation) 로직과, 속도에 따라 다음 위치를 계산하는 추측 항법(Dead Reckoning) 구현.

### 💡 자소서 활용 팁

그래픽이 본인의 영역이 아니라면, **"그래픽 라이브러리(OpenGL 등)는 도구로써 활용하고, 그 위에서 돌아가는 '시스템의 구조', '데이터의 흐름', '사용자 인터페이스'를 설계하고 최적화하는 데 주력했다"**는 점을 강조하시면 됩니다. 특히 **WebSocket/TCP 네트워크 최적화**와 **C++의 웹 포팅 경험**은 백엔드나 시스템 엔지니어링 직무에서 매우 매력적인 포인트입니다.

- jpeg
    
    ## 1. 문제 정의 (Problem Definition)
    
    C++ 기반 영상 전송 시스템에서 **JPEG 인코딩 단계가 전체 파이프라인의 주요 병목**으로 작용하였다.
    
    초기 구현에서는 `stb_image_write`를 사용하여 JPEG 인코딩을 수행했으나, 고해상도 프레임을 연속적으로 처리하는 과정에서 **CPU 사용률 급증 및 프레임 처리 지연**이 발생하였다.
    
    특히 단일 스레드 환경에서 인코딩 작업이 직렬로 수행되면서, **실시간 처리 요구사항을 만족하지 못하는 문제**가 확인되었다.
    
    ---
    
    ## 2. 원인 분석 (Root Cause Analysis)
    
    성능 저하의 원인을 분석한 결과 다음과 같은 문제가 확인되었다.
    
    - `stb_image_write`는 JPEG 인코딩에 대한 SIMD 최적화가 제한적이며, 고해상도 프레임 처리 시 성능 한계가 명확함
    - JPEG 인코딩 과정(RGB → YCbCr 변환, DCT, 양자화, 엔트로피 코딩)이 **CPU 집약적 작업**임에도 단일 스레드로 처리됨
    - 인코딩 라이브러리 교체만으로는 **전체 파이프라인 성능 개선에 한계**가 존재
    
    이를 통해 단순한 라이브러리 교체가 아닌 **구조적 성능 개선이 필요**하다는 결론에 도달하였다.
    
    ---
    
    ## 3. 해결 과정 (Solution Approach)
    
    ### 3.1 JPEG 인코딩 라이브러리 교체
    
    - 기존 `stb_image_write`를 **SIMD 최적화가 적용된 `libjpeg-turbo`** 기반 인코더로 전환
    - `turbojpeg API`를 사용하여 인코딩 성능을 최대한 활용
    
    이를 통해 **단일 프레임 인코딩 시간은 감소**했으나, 전체 시스템 관점에서의 성능 향상은 제한적임을 확인하였다.
    
    ---
    
    ### 3.2 단일 스레드 병목 식별
    
    라이브러리 교체 이후에도 인코딩 단계가 **단일 스레드로 직렬 처리**되고 있음을 확인하였으며,
    
    JPEG 인코딩이 프레임 단위로 **완전히 독립적인 작업**이라는 점에 착안하여 병렬화 가능성을 검토하였다.
    
    ---
    
    ### 3.3 멀티 스레드 인코딩 구조 설계
    
    - 프레임 단위 작업을 분리하여 **멀티 스레드 인코딩 파이프라인**을 설계
    - 스레드 풀 기반 구조를 적용하여 여러 프레임을 병렬로 인코딩
    - CPU 코어 수를 효율적으로 활용할 수 있도록 작업 큐 기반 처리 방식 도입
    
    이를 통해 인코딩 단계의 **직렬 병목을 제거**하였다.
    
    ---
    
    ## 4. 결과 및 효과 (Results & Impact)
    
    - JPEG 인코딩 처리량이 **단일 스레드 대비 2배 이상 향상**
    - CPU 사용률이 단일 코어 집중 구조에서 **멀티 코어 분산 구조로 개선**
    - 고해상도 영상 프레임에 대해 **실시간 처리 안정성 확보**
    - 라이브러리 교체 + 병렬화 적용을 통해 **구조적 성능 개선 경험 확보**
    
    본 개선을 통해 단순한 최적화 기법 적용이 아닌,
    
    **병목 분석 → 원인 분리 → 단계적 해결**이라는 성능 개선 프로세스를 실무적으로 경험하였다.
    
    ---
    
    ## 5. 핵심 정리 (Summary)
    
    - JPEG 인코딩 성능 저하의 원인을 라이브러리 수준과 구조적 수준으로 분리하여 분석
    - SIMD 최적화 라이브러리 도입 이후에도 남아 있는 단일 스레드 병목을 멀티 스레딩으로 해결
    - 실시간 영상 처리 요구사항을 만족하는 인코딩 파이프라인을 구현
