## 1. Git

- 질문: Git의 핵심 개념을 면접 답변처럼 1분 안에 설명해 보세요.
- 답변: Git는 소스 코드 변경 이력을 관리하는 도구입니다. 핵심은 여러 사람이 동시에 작업하면 변경 추적, 병합, 복구가 필요하기 때문입니다. 그래서 작업 내용을 commit으로 저장하고 branch로 흐름을 나눈 뒤 merge 또는 rebase로 통합합니다. 면접에서는 정의만 말하지 말고, 이 개념이 어떤 위험을 줄이고 어떤 기준으로 성능이나 안정성에 영향을 주는지까지 연결하면 좋습니다.
- 답변 키워드: Git, 스냅샷, 히스토리, 브랜치 전략, 충돌 해결, 복구 명령, 정의, 핵심 목적, 면접식 요약
- 관련 문서: [[CS공부/11_개발 기본기/01_Git|Git]]

## 2. commit

- 질문: commit이/가 필요한 이유와 해결하려는 문제는 무엇인가요?
- 답변: commit가 필요한 이유는 여러 사람이 동시에 작업하면 변경 추적, 병합, 복구가 필요하기 때문입니다. 이를 해결하기 위해 시스템은 작업 내용을 commit으로 저장하고 branch로 흐름을 나눈 뒤 merge 또는 rebase로 통합합니다. 결과적으로 애플리케이션이나 사용자는 하드웨어, 저장소, 네트워크, 런타임의 세부 구현을 모두 직접 다루지 않고도 일관된 규칙 위에서 동작할 수 있습니다.
- 답변 키워드: commit, 스냅샷, 히스토리, 브랜치 전략, 충돌 해결, 복구 명령, 문제 배경, 도입 이유, 대안 비교
- 관련 문서: [[CS공부/11_개발 기본기/01_Git|Git]]

## 3. branch

- 질문: branch의 동작 과정을 단계별로 설명해 보세요.
- 답변: branch의 동작 과정은 작업 내용을 commit으로 저장하고 branch로 흐름을 나눈 뒤 merge 또는 rebase로 통합합니다 이 흐름을 설명할 때는 입력이 무엇이고, 중간 상태가 어디에 저장되며, 결과가 어느 시점에 반영되는지를 순서대로 말해야 합니다. 주요 확인 기준은 merge/rebase, reset/revert, branch 전략, conflict 해결입니다.
- 답변 키워드: branch, 스냅샷, 히스토리, 브랜치 전략, 충돌 해결, 복구 명령, 단계별 흐름, 입력/처리/출력, 상태 변화
- 관련 문서: [[CS공부/11_개발 기본기/01_Git|Git]]

## 4. merge

- 질문: merge의 장점과 한계를 실제 사례와 함께 설명해 보세요.
- 답변: merge의 장점은 여러 사람이 동시에 작업하면 변경 추적, 병합, 복구가 필요하는 문제를 줄인다는 점입니다 한계는 merge/rebase, reset/revert, branch 전략, conflict 해결 사이에서 트레이드오프가 생긴다는 것입니다. 따라서 실제 사례에서는 입력 규모, 지연 시간, 일관성, 구현 난이도 중 무엇을 우선하는지에 따라 평가가 달라집니다.
- 답변 키워드: merge, 스냅샷, 히스토리, 브랜치 전략, 충돌 해결, 복구 명령, 장점, 한계, 실무 사례
- 관련 문서: [[CS공부/11_개발 기본기/01_Git|Git]]

## 5. rebase

- 질문: rebase에서 성능 병목이나 비용이 커지는 지점은 어디인가요?
- 답변: rebase에서 성능 병목은 주로 conflict 크기, binary file, rebase 충돌, CI 대기에서 커집니다. 병목을 확인할 때는 먼저 처리량, 지연 시간, 메모리 사용량처럼 관측 가능한 지표를 잡고, force push 사고, conflict 오해, reset 남용, 큰 PR 가능성을 하나씩 좁혀야 합니다. 개선은 가장 큰 비용을 만든 구간을 측정으로 확인한 뒤 적용해야 합니다.
- 답변 키워드: rebase, 스냅샷, 히스토리, 브랜치 전략, 충돌 해결, 복구 명령, 병목 지점, 비용, 측정 지표
- 관련 문서: [[CS공부/11_개발 기본기/01_Git|Git]]

- 답변 키워드: cherry-pick, 스냅샷, 히스토리, 브랜치 전략, 충돌 해결, 복구 명령, 비교 기준, 차이점, 선택 상황

## 7. stash

- 질문: stash 관련 장애나 버그가 발생했을 때 어떤 순서로 원인을 추적하겠습니까?
- 답변: stash 관련 장애는 먼저 재현 조건과 영향 범위를 고정한 뒤 force push 사고, conflict 오해, reset 남용, 큰 PR를 의심해 볼 수 있습니다. 그 다음 로그, 지표, 상태 덤프, 설정값을 확인하면서 작업 내용을 commit으로 저장하고 branch로 흐름을 나눈 뒤 merge 또는 rebase로 통합합니다 이 흐름 중 어느 단계에서 기대와 달라졌는지 찾습니다. 수정 후에는 같은 조건을 검증하는 테스트나 운영 체크를 남겨야 합니다.
- 답변 키워드: stash, 스냅샷, 히스토리, 브랜치 전략, 충돌 해결, 복구 명령, 장애 증상, 원인 추적, 검증 도구

- 답변: 운영 가능한 코드에 대한 흔한 오해는 이름만 보고 항상 같은 성능이나 보장을 제공한다고 생각하는 것입니다. 실제로는 리뷰 깊이, 문서 비용, 기술 부채, 릴리스 품질에 따라 동작과 비용이 달라집니다. 따라서 답변에서는 개별 판단만으로 개발하면 스타일, 설계, 운영 기준이 흔들리기 때문이라는 범위와 대형 PR, 리뷰 형식화, 문서 부재, 책임 경계 불명확 같은 예외 상황을 함께 말해야 정확합니다.
- 답변 키워드: 운영 가능한 코드, 코드 리뷰, 문서화, 컨벤션, 기술 부채, 릴리스 품질, 흔한 오해, 반례, 정확한 표현
