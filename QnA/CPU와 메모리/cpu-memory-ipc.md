# CPU, Memory, IPC Q&A

## 1. CPU는 빠르고 RAM은 느리다는 말의 의미

CPU가 빠르고 RAM이 느리다는 말은 보통 **한 번의 연산 또는 데이터 접근에 걸리는 시간**이 다르다는 뜻이다.

CPU는 덧셈, 비교, 분기 같은 명령을 매우 짧은 시간에 처리할 수 있다. 반면 RAM에서 데이터를 가져오려면 CPU 입장에서는 오래 기다려야 한다. CPU 내부의 레지스터나 캐시는 CPU와 매우 가까워 접근이 빠르지만, RAM은 CPU 밖의 별도 칩에 있고 메모리 컨트롤러와 버스를 거쳐야 하므로 더 느리다.

속도 계층은 대략 다음과 같다.

```text
빠름
CPU 레지스터
L1 캐시
L2 캐시
L3 캐시
RAM
SSD/HDD
느림
```

핵심은 **CPU의 계산 속도에 비해 RAM에서 데이터를 가져오는 속도가 느려서 CPU가 기다리는 일이 생길 수 있다**는 것이다. 그래서 캐시는 자주 쓰는 데이터를 CPU 가까이에 두어 RAM 접근을 줄이는 역할을 한다.

## 2. IPC 성능은 누가 정하는가

IPC는 **Instructions Per Cycle**의 약자로, CPU가 클럭 1사이클 동안 평균적으로 처리한 명령어 수를 뜻한다.

IPC는 CPU에 고정된 단일 숫자가 아니라 여러 요소의 결과로 결정된다.

- CPU 설계: 명령어를 몇 개 가져오고 디코딩할 수 있는지, 실행 유닛이 몇 개인지, 분기 예측이 얼마나 정확한지, 캐시 구조가 어떤지 등이 영향을 준다.
- 프로그램 코드: 서로 독립적인 명령이 많으면 CPU가 동시에 처리하기 쉬워 IPC가 높아질 수 있다. 앞 명령의 결과를 다음 명령이 기다려야 하는 코드라면 IPC가 낮아질 수 있다.
- 메모리 접근: 캐시 미스가 자주 나고 RAM 접근을 많이 기다리면 사이클은 지나가지만 명령어 완료가 늦어져 IPC가 낮아진다.
- 컴파일러 최적화: 불필요한 명령 제거, 루프 최적화, 벡터 명령 사용, 명령 순서 재배치 등이 IPC에 영향을 줄 수 있다.

정리하면 다음과 같다.

```text
IPC = CPU 구조 + 프로그램 특성 + 메모리 접근 + 컴파일러 최적화의 결과
```

같은 CPU라도 실행하는 프로그램에 따라 IPC는 달라진다.

## 3. CPU가 접근하는 순서는 레지스터 -> 캐시 -> 메모리인가

대체로 비슷하지만, 정확히는 CPU가 항상 `레지스터 -> 캐시 -> 메모리` 순서로 차례대로 찾아간다는 뜻은 아니다.

CPU의 실제 연산은 보통 **레지스터에 있는 값**으로 수행된다. 필요한 데이터가 레지스터에 없으면 `load` 명령을 통해 캐시 또는 메모리 계층에서 값을 가져와 레지스터에 넣는다.

예를 들어 `a = b + c` 같은 연산은 보통 다음 흐름으로 처리된다.

```text
1. b 값을 레지스터로 load
2. c 값을 레지스터로 load
3. 레지스터에 있는 b와 c를 더함
4. 결과를 레지스터에 저장
5. 필요하면 결과를 메모리에 store
```

이때 `load`가 발생하면 CPU는 보통 캐시 계층을 확인한다.

```text
L1 캐시에 있나?
없으면 L2 캐시에 있나?
없으면 L3 캐시에 있나?
없으면 RAM에서 가져온다
```

더 정확한 그림은 다음과 같다.

```text
CPU 연산
  ↓
레지스터
  ↑
Load/Store 명령
  ↓
L1 캐시 -> L2 캐시 -> L3 캐시 -> RAM
```

핵심은 **CPU는 계산할 때 레지스터를 직접 사용하고, 레지스터에 값을 채우거나 결과를 저장할 때 캐시와 메모리 계층을 거친다**는 것이다.

## 4. 필요한 데이터가 미리 레지스터에 있을 수 있는가

필요한 데이터가 이미 레지스터에 있을 수 있다. 방금 계산한 값이거나 곧 계속 사용할 값이면 컴파일러가 그 값을 레지스터에 둔 채 다음 연산에서 바로 사용하게 만들 수 있다.

예를 들어 다음 코드가 있다고 하자.

```c
int x = a + b;
int y = x * 3;
```

`x`는 굳이 RAM에 저장했다가 다시 가져올 필요가 없다. CPU는 보통 `a + b`의 결과를 레지스터에 넣고, 그 레지스터 값을 바로 `* 3` 연산에 사용할 수 있다.

다만 레지스터는 캐시처럼 모든 데이터를 자동으로 보관하는 공간이라기보다는, CPU 명령어가 직접 지정해서 쓰는 **초소형 작업 공간**에 가깝다. 어떤 값을 레지스터에 둘지는 주로 컴파일러가 정하고, 실행 중 CPU는 명령어를 따라 레지스터를 사용한다.

레지스터는 CPU 안에서도 **각 코어 내부**에 있다.

```text
CPU 칩
├─ 코어 0
│  ├─ 레지스터
│  ├─ L1 캐시
│  └─ 실행 유닛
├─ 코어 1
│  ├─ 레지스터
│  ├─ L1 캐시
│  └─ 실행 유닛
└─ L2/L3 캐시
```

일반적으로 다음처럼 볼 수 있다.

- 레지스터: 각 CPU 코어 안에 있음
- L1 캐시: 각 코어 가까이에 있음
- L2 캐시: 보통 코어별 또는 일부 공유
- L3 캐시: 여러 코어가 공유하는 경우가 많음
- RAM: CPU 밖의 메인 메모리 칩

레지스터는 RAM처럼 큰 저장공간이 아니라 몇십 개에서 몇백 개 수준의 매우 작은 저장공간이다. 값이 이미 레지스터에 있으면 CPU는 캐시나 메모리에 가지 않고 바로 연산할 수 있다.

## 5. CPU에게 명령은 커널이 주는가

CPU가 실행하는 명령은 **커널만 주는 것이 아니다**. CPU는 현재 실행 중인 프로그램의 **기계어 명령어**를 메모리에서 가져와 실행한다.

예를 들어 사용자가 만든 프로그램이 실행 중이면, CPU는 그 프로그램의 명령어를 실행한다. 커널 코드가 실행 중이면 CPU는 커널의 명령어를 실행한다. 즉, CPU 입장에서는 사용자 프로그램이든 커널이든 모두 메모리에 올라와 있는 기계어 명령어일 뿐이다.

커널의 역할은 CPU에게 모든 명령을 하나씩 직접 내려주는 것이 아니라, 다음과 같은 실행 환경을 관리하는 것이다.

- 어떤 프로그램을 실행할지 정함
- 각 프로그램에 CPU 시간을 나눠 줌
- 메모리, 파일, 네트워크, 장치 접근을 관리함
- 사용자 프로그램이 직접 하면 위험한 작업을 대신 수행함
- 인터럽트나 시스템 콜이 발생했을 때 CPU 실행 흐름을 커널 코드로 전환함

흐름을 단순화하면 다음과 같다.

```text
프로그램 코드가 메모리에 올라감
        ↓
CPU가 명령어를 하나씩 가져와 실행
        ↓
필요하면 시스템 콜이나 인터럽트로 커널 코드 실행
        ↓
다시 사용자 프로그램으로 돌아감
```

예를 들어 일반 계산은 사용자 프로그램의 명령어로 실행된다.

```c
int x = a + b;
```

이런 계산은 커널이 매번 명령하지 않는다. CPU가 프로그램의 기계어 명령을 직접 가져와 실행한다.

반면 파일 읽기 같은 작업은 사용자 프로그램이 직접 디스크를 제어하지 않고 커널에 요청한다.

```c
read(fd, buffer, size);
```

이 경우 사용자 프로그램이 시스템 콜을 호출하고, CPU 실행 흐름이 커널 코드로 넘어간다. 커널은 파일 시스템과 장치를 확인해서 데이터를 읽고, 작업이 끝나면 다시 사용자 프로그램으로 돌아가게 한다.

정리하면 다음과 같다.

```text
CPU가 실행하는 명령 = 현재 실행 흐름에 있는 기계어 명령
커널 = 어떤 프로그램이 실행될지, 자원과 권한을 어떻게 쓸지 관리하는 운영체제 핵심
```

따라서 **CPU에게 명령을 주는 주체가 항상 커널인 것은 아니고, CPU는 사용자 프로그램과 커널 코드를 번갈아 실행한다**고 이해하면 된다.

## 6. 제어장치는 실제 하드웨어인가

제어장치는 단순히 개념적인 이름만 있는 것이 아니라, **CPU 안에 실제 회로로 구현된 하드웨어적 구성 요소**다. 다만 메모리 칩이나 SSD처럼 따로 떼어낼 수 있는 독립 부품이라기보다는, CPU 내부에 포함된 제어 로직을 부르는 말이다.

CPU를 크게 나누면 보통 다음처럼 설명한다.

```text
CPU
├─ 제어장치
├─ 산술논리장치(ALU)
├─ 레지스터
└─ 캐시
```

제어장치는 CPU가 명령어를 실행할 때 전체 흐름을 조율한다.

- 명령어를 가져옴
- 명령어를 해석함
- 어떤 레지스터를 읽을지 정함
- ALU가 덧셈을 할지 비교를 할지 정함
- 메모리에서 값을 읽을지 저장할지 정함
- 다음에 실행할 명령어 주소를 정함

즉, 제어장치는 데이터를 직접 계산하는 장치라기보다는 **CPU 내부의 다른 부품들에게 제어 신호를 보내는 회로**라고 보면 된다.

예를 들어 다음 명령이 있다고 하자.

```text
ADD R1, R2, R3
```

의미는 `R2 + R3`을 계산해서 `R1`에 넣으라는 것이다. 이때 제어장치는 대략 다음 일을 한다.

```text
1. R2와 R3 레지스터 값을 읽게 함
2. ALU에게 덧셈을 하라고 신호를 보냄
3. 결과를 R1 레지스터에 저장하게 함
4. 다음 명령어로 넘어가게 함
```

현대 CPU에서는 제어장치가 교과서 그림처럼 하나의 단순한 블록으로만 존재하지는 않는다. 명령어 디코더, 스케줄러, 분기 예측기, 파이프라인 제어 로직, 마이크로코드 ROM 같은 여러 회로와 로직이 함께 명령 실행을 제어한다.

정리하면 다음과 같다.

```text
제어장치 = CPU 내부 동작을 조율하는 실제 하드웨어 회로
하지만 독립 부품이라기보다는 CPU 안에 구현된 제어 로직 전체를 설명하는 이름
```

따라서 제어장치는 **하드웨어이기도 하고, CPU 동작을 설명하기 위한 논리적 구성 요소 이름이기도 하다**고 이해하면 된다.

## 7. compare r1, r2를 하면 ALU는 무엇을 반환하는가

`compare r1, r2` 같은 비교 명령은 보통 결과값을 일반 레지스터에 저장하지 않는다. 대신 ALU가 내부적으로 `r1 - r2` 같은 연산을 해 보고, 그 결과에 따라 **상태 레지스터의 플래그**를 바꾼다.

예를 들어 다음 비교가 있다고 하자.

```text
compare r1, r2
```

CPU는 대략 다음처럼 처리한다.

```text
1. r1 값을 읽음
2. r2 값을 읽음
3. ALU가 r1 - r2를 계산해 봄
4. 계산 결과를 일반 레지스터에 저장하지는 않음
5. 대신 상태 플래그를 갱신함
```

대표적인 플래그는 다음과 같다.

- Zero Flag: 결과가 0인지
- Sign Flag: 결과가 음수인지
- Carry Flag: 자리올림/빌림이 발생했는지
- Overflow Flag: 부호 있는 연산에서 오버플로가 났는지

예를 들어 `r1 = 5`, `r2 = 5`라면 내부적으로 `5 - 5 = 0`이므로 Zero Flag가 켜진다. 그 다음 조건 분기 명령은 이 플래그를 보고 이동할지 말지 결정한다.

```text
compare r1, r2
jump_if_equal label
```

위 흐름은 다음 뜻이다.

```text
r1과 r2를 비교한다
같으면 Zero Flag가 켜진다
jump_if_equal은 Zero Flag를 보고 label로 이동한다
```

따라서 `compare`의 결과는 함수의 `return`처럼 어떤 값이 나오는 것이 아니라, **CPU 상태 레지스터의 플래그가 바뀌는 것**이라고 이해하면 된다.

정리하면 다음과 같다.

```text
add r1, r2 -> 계산 결과가 레지스터에 저장될 수 있음
compare r1, r2 -> 계산 결과는 버리고 상태 플래그만 갱신함
조건 분기 -> 그 플래그를 보고 실행 흐름을 바꿈
```

## 8. 플래그들은 어디에 있고 compare는 어떤 플래그를 바꾸는가

플래그들은 CPU 내부의 **상태 레지스터**에 들어 있는 비트들이다. CPU마다 이름은 조금씩 다르지만, x86 계열에서는 `RFLAGS` 또는 `EFLAGS` 같은 레지스터가 이 역할을 한다.

상태 레지스터는 일반적인 데이터 계산에 쓰는 범용 레지스터와 목적이 다르다.

```text
범용 레지스터: r1, r2 같은 실제 데이터 값을 담음
상태 레지스터: 직전 연산 결과의 상태를 비트 단위로 기록함
```

플래그가 정확히 4개만 있는 것은 아니다. 교재에서 자주 설명하는 대표 플래그가 Zero, Sign, Carry, Overflow일 뿐이고, 실제 CPU에는 더 많은 상태 비트가 있을 수 있다.

대표 플래그는 다음과 같다.

- Zero Flag: 연산 결과가 0이면 켜짐
- Sign Flag: 연산 결과의 부호 비트가 1이면 켜짐
- Carry Flag: 부호 없는 연산에서 자리올림이나 빌림이 발생하면 켜짐
- Overflow Flag: 부호 있는 연산에서 표현 범위를 넘으면 켜짐

`compare r1, r2`는 보통 내부적으로 `r1 - r2`를 계산해 보고, 그 결과에 따라 **하나의 플래그만 바꾸는 것이 아니라 관련 플래그들을 여러 개 갱신**한다.

예를 들어 다음과 같다고 하자.

```text
r1 = 5
r2 = 5

compare r1, r2
```

내부적으로 `5 - 5 = 0`이므로 대표적으로 다음 상태가 된다.

```text
Zero Flag = 1
Sign Flag = 0
```

이후 `jump_if_equal` 같은 조건 분기 명령은 Zero Flag를 확인한다.

```text
compare r1, r2
jump_if_equal label
```

의미는 다음과 같다.

```text
비교 결과가 0인가?
즉, Zero Flag가 1인가?
맞으면 label로 이동
```

하지만 모든 조건 분기가 Zero Flag 하나만 보는 것은 아니다. 예를 들어 크다/작다 비교는 부호 있는 비교인지, 부호 없는 비교인지에 따라 Sign Flag, Carry Flag, Overflow Flag 등을 조합해서 판단할 수 있다.

단순화하면 다음처럼 이해하면 된다.

```text
compare 실행
  ↓
ALU가 r1 - r2를 계산해 봄
  ↓
결과값은 버림
  ↓
상태 레지스터의 여러 플래그 비트 갱신
  ↓
조건 분기 명령이 필요한 플래그를 읽음
```

정리하면 다음과 같다.

```text
플래그 = CPU 내부 상태 레지스터에 있는 비트
플래그 개수 = 대표적으로 4개를 많이 설명하지만 실제로는 더 많을 수 있음
compare = 하나의 결과값을 반환하지 않고 여러 상태 플래그를 갱신함
조건 분기 = 그 플래그 하나 또는 조합을 보고 판단함
```

## 9. IR은 Instruction Register인데 명령어도 레지스터에 저장하는가

IR은 **Instruction Register**의 약자다. 이름 그대로 현재 CPU가 가져와서 해석 중인 **명령어를 잠깐 담는 레지스터**다.

레지스터라고 해서 모두 `int x = 10` 같은 데이터 값만 저장하는 것은 아니다. 레지스터는 CPU 내부의 아주 빠른 저장 공간이고, 목적에 따라 여러 종류가 있다.

```text
범용 레지스터: 계산에 쓰는 데이터 저장
PC: 다음 명령어 주소 저장
SP: 현재 스택 위치 저장
상태 레지스터: 연산 결과 플래그 저장
IR: 현재 실행할 명령어 저장
```

CPU의 기본 명령어 처리 흐름은 보통 다음처럼 설명한다.

```text
1. PC가 다음 명령어 주소를 가리킴
2. CPU가 그 주소에서 명령어를 메모리/캐시에서 가져옴
3. 가져온 명령어를 IR에 저장
4. 제어장치가 IR 안의 명령어를 해석
5. 필요한 레지스터, ALU, 메모리 동작을 제어
6. PC를 다음 명령어 주소로 갱신
```

예를 들어 메모리에 이런 기계어 명령어가 있다고 하자.

```text
주소 1000: ADD R1, R2, R3
```

PC가 `1000`을 가리키면 CPU는 그 위치의 명령어를 가져와 IR에 넣는다.

```text
PC = 1000
IR = ADD R1, R2, R3
```

그 다음 제어장치는 IR에 들어 있는 명령어를 보고 다음 제어 신호를 만든다.

```text
R2 읽기
R3 읽기
ALU에게 덧셈 지시
결과를 R1에 저장
```

따라서 IR은 프로그램의 모든 명령어를 저장하는 공간이 아니다. 프로그램의 명령어 전체는 메모리의 코드 영역에 있고, IR은 그중 **현재 처리 중인 명령어 하나를 잠깐 담는 레지스터**라고 보면 된다.

정리하면 다음과 같다.

```text
명령어 전체 = 메모리의 코드 영역에 저장
PC = 다음에 가져올 명령어 주소
IR = 지금 가져와서 해석 중인 명령어
제어장치 = IR을 보고 CPU 내부 동작을 제어
```

현대 CPU에서는 내부 구조가 더 복잡해서 명령어 큐, 디코더, 마이크로연산 캐시 같은 단계가 추가될 수 있다. 하지만 기본 개념을 이해할 때는 `PC가 주소를 가리키고, 그 주소의 명령어를 IR에 담아 제어장치가 해석한다`고 보면 된다.

## 10. 마이크로프로그램 제어는 어떻게 동작하는가

마이크로프로그램 제어는 CPU 명령어 하나를 더 작은 내부 제어 단계들로 나누고, 그 단계들을 **마이크로명령어** 형태로 저장해 두었다가 순서대로 실행하는 제어 방식이다.

일반 프로그램 명령어와 마이크로명령어는 층위가 다르다.

```text
프로그램 명령어: ADD, LOAD, JUMP 같은 ISA 명령어
마이크로명령어: CPU 내부 회로에 보낼 세부 제어 신호 묶음
```

예를 들어 프로그램에 다음 명령어가 있다고 하자.

```text
LOAD R1, [1000]
```

의미는 메모리 주소 `1000`에 있는 값을 읽어 `R1`에 넣으라는 것이다. 이 명령어 하나를 실행하려면 CPU 내부에서는 여러 작은 동작이 필요하다.

```text
1. 주소 1000을 메모리 주소 버퍼에 넣음
2. 메모리 읽기 신호를 보냄
3. 메모리에서 온 데이터를 내부 버퍼에 받음
4. 그 값을 R1에 저장함
5. 다음 명령어로 이동함
```

마이크로프로그램 제어 방식에서는 이런 내부 단계들이 제어 저장소에 마이크로명령어 형태로 저장되어 있다.

```text
명령어 LOAD 해석
        ↓
제어 저장소에서 LOAD용 마이크로프로그램 시작 주소 찾기
        ↓
마이크로명령어 1 실행
        ↓
마이크로명령어 2 실행
        ↓
마이크로명령어 3 실행
        ↓
LOAD 명령 완료
```

구조를 단순화하면 다음과 같다.

```text
IR
↓
명령어 디코더
↓
제어 저장소(Control Store)
↓
마이크로명령어들
↓
CPU 내부 제어 신호 생성
```

제어 저장소는 CPU 내부에 있는 ROM 또는 고속 메모리 같은 공간으로 설명할 수 있다. 여기에 각 명령어를 어떻게 세부 단계로 실행할지에 대한 마이크로프로그램이 들어 있다.

예를 들어 `ADD R1, R2, R3`는 단순한 명령이라 몇 단계면 끝날 수 있다.

```text
R2 읽기
R3 읽기
ALU 덧셈
R1에 결과 저장
```

반면 복잡한 명령어는 더 많은 마이크로명령어 단계가 필요할 수 있다.

하드와이어드 제어와 비교하면 다음과 같다.

```text
하드와이어드 제어
- 제어 신호 생성 로직을 회로로 직접 구성
- 빠를 수 있음
- 복잡한 명령어를 구현하기 어렵거나 수정이 어려움

마이크로프로그램 제어
- 제어 신호 순서를 마이크로명령어로 저장
- 복잡한 명령어 구현이 상대적으로 쉬움
- 제어 저장소를 읽는 단계 때문에 느릴 수 있음
```

정리하면 다음과 같다.

```text
프로그램 명령어 = CPU가 외부에 공개한 명령어
마이크로명령어 = 그 명령어를 실제 회로 동작으로 바꾸기 위한 내부 제어 단계
마이크로프로그램 제어 = 명령어별 내부 실행 절차를 저장해 두고 실행하는 방식
```

즉, 마이크로프로그램 제어는 **명령어 하나를 실행하기 위한 CPU 내부용 작은 프로그램을 제어장치가 따라가는 방식**이라고 이해하면 된다.

## 11. 명령어 사이클에서 Fetch, Decode, Execute는 누가 담당하는가

큰 방향은 맞지만, `Fetch는 PC`, `Decode는 제어장치`, `Execute는 ALU`처럼 한 부품이 단독으로 담당한다고 보면 조금 단순화가 심하다. 실제로는 여러 구성요소가 함께 동작한다.

기본 명령어 사이클은 보통 다음처럼 설명한다.

```text
Fetch -> Decode -> Execute -> Memory -> Write Back
```

### Fetch

Fetch는 다음에 실행할 명령어를 메모리나 캐시에서 가져오는 단계다.

여기서 PC는 **명령어의 주소를 제공**한다. PC 자체가 명령어를 가져오는 일을 직접 하는 것은 아니고, PC에 들어 있는 주소를 바탕으로 CPU의 명령어 가져오기 회로, 캐시, 메모리 인터페이스가 동작한다.

```text
PC가 다음 명령어 주소를 가리킴
        ↓
CPU가 그 주소로 명령어를 가져옴
        ↓
IR 또는 명령어 큐에 저장
        ↓
PC를 다음 명령어 주소로 갱신
```

따라서 정확히는 다음과 같다.

```text
Fetch = PC + 명령어 캐시 + 메모리 접근 회로 + 제어 로직이 함께 수행
```

### Decode

Decode는 가져온 명령어가 어떤 동작을 의미하는지 해석하는 단계다.

이 단계에서는 제어장치, 명령어 디코더, 경우에 따라 마이크로프로그램 제어 로직이 관여한다.

예를 들어 IR에 다음 명령어가 들어 있다고 하자.

```text
ADD R1, R2, R3
```

Decode 단계에서는 이것을 다음처럼 해석한다.

```text
연산 종류: ADD
읽을 레지스터: R2, R3
쓸 레지스터: R1
필요한 실행 장치: ALU
```

따라서 Decode는 주로 제어장치와 디코더가 담당한다고 볼 수 있다.

### Execute

Execute는 명령어가 요구하는 실제 동작을 수행하는 단계다.

산술/논리 연산이면 ALU가 핵심 역할을 한다.

```text
ADD R1, R2, R3
-> ALU가 R2 + R3 수행
```

하지만 모든 Execute가 ALU만으로 끝나는 것은 아니다.

```text
LOAD -> 주소 계산은 ALU가 할 수 있지만, 실제 데이터 읽기는 캐시/메모리 계층이 필요
STORE -> 주소 계산과 메모리 쓰기가 필요
JUMP -> PC 변경이 핵심
COMPARE -> ALU가 비교하고 상태 플래그 갱신
```

따라서 Execute는 명령어 종류에 따라 ALU, 분기 장치, 로드/스토어 유닛, 캐시, 레지스터 파일 등이 함께 관여한다.

정리하면 다음과 같다.

```text
Fetch
- PC가 주소를 제공
- 명령어 캐시/메모리 접근 회로가 명령어를 가져옴

Decode
- 제어장치/디코더가 명령어 의미를 해석
- 필요한 제어 신호를 준비

Execute
- ALU, 분기 장치, 로드/스토어 유닛 등이 실제 동작 수행
```

가장 짧게 표현하면 다음과 같다.

```text
PC는 Fetch의 기준 주소를 제공한다
제어장치는 Decode에서 명령어를 해석하고 제어 신호를 만든다
ALU는 Execute 중 산술/논리 연산을 담당한다
```

즉, 질문의 방향은 맞지만 **각 단계는 특정 부품 하나가 아니라 CPU 내부의 여러 장치가 협력해서 수행한다**고 이해하는 것이 더 정확하다.

## 12. add r1, r2, #5는 명령어이고 ALU가 실행하는가

`add r1, r2, #5`는 CPU가 실행할 **기계어 명령어를 사람이 읽기 쉽게 표현한 어셈블리 명령어**다.

의미는 보통 다음과 같다.

```text
r1 = r2 + 5
```

여기서 `#5`는 즉시값(immediate value)이다. 즉, 다른 레지스터나 메모리에서 값을 가져오는 것이 아니라 명령어 안에 숫자 `5`가 직접 들어 있다는 뜻이다.

이 명령어 하나가 실행될 때는 ALU만 단독으로 움직이는 것이 아니다. 여러 CPU 구성요소가 함께 동작한다.

```text
1. Fetch
   PC가 가리키는 주소에서 add r1, r2, #5 명령어를 가져옴

2. Decode
   제어장치/디코더가 이 명령어를 해석함
   - 연산: add
   - 읽을 값: r2, 즉시값 5
   - 저장할 곳: r1
   - 사용할 장치: ALU

3. Execute
   ALU가 r2 + 5를 계산함

4. Write Back
   계산 결과를 r1 레지스터에 저장함
```

따라서 역할을 나누면 다음과 같다.

```text
add r1, r2, #5
= CPU에게 시키는 명령어

제어장치/디코더
= 이 명령어가 덧셈이고 r2와 5를 써서 r1에 저장해야 한다고 해석

ALU
= 실제 덧셈 r2 + 5를 수행

레지스터 파일
= r2 값을 제공하고, 결과를 r1에 저장
```

즉, `add r1, r2, #5`라는 줄 자체가 하드웨어는 아니다. 이 줄은 **CPU가 해석할 명령어**이고, 그 명령어에 포함된 실제 산술 연산을 수행하는 하드웨어가 **ALU**다.

정리하면 다음과 같다.

```text
명령어 = 무엇을 할지 적어 둔 코드
ALU = 그중 산술/논리 계산을 실제로 수행하는 회로
제어장치 = 명령어를 해석해서 ALU와 레지스터에 제어 신호를 보내는 회로
```

## 13. ALU는 하드웨어인가, 칩인가

ALU는 **Arithmetic Logic Unit**, 즉 산술논리장치다. ALU는 실제 하드웨어 회로가 맞다. 덧셈, 뺄셈, AND, OR, XOR, 비교 같은 산술/논리 연산을 수행하는 회로다.

하지만 보통 현대 컴퓨터에서 ALU를 독립된 칩 하나로 보지는 않는다. ALU는 **CPU 칩 안의 CPU 코어 내부에 들어 있는 회로 블록**이다.

```text
CPU 칩
└─ CPU 코어
   ├─ 제어장치
   ├─ 레지스터
   ├─ ALU
   ├─ 캐시
   └─ 기타 실행 유닛
```

즉, 다음처럼 구분하면 된다.

```text
ALU = 실제 하드웨어 회로
ALU = CPU 내부 구성 요소
ALU != 보통 별도로 꽂는 독립 칩
```

예전에는 74181 같은 ALU 기능을 하는 독립 IC 칩도 있었다. 하지만 현재 우리가 쓰는 일반적인 CPU에서는 ALU가 CPU 내부에 통합되어 있다.

CPU 안에는 ALU가 하나만 있는 것도 아니다. 현대 CPU 코어에는 여러 실행 유닛이 있고, 정수 ALU가 여러 개 들어 있을 수 있다. 또 정수 연산용 ALU, 부동소수점 연산 장치, 벡터 연산 장치처럼 역할이 나뉜 회로들도 있다.

정리하면 다음과 같다.

```text
ALU는 하드웨어다.
하지만 보통 독립된 칩이 아니라 CPU 코어 내부의 연산 회로 블록이다.
CPU는 명령어를 해석한 뒤 ALU에 연산을 수행하도록 제어 신호를 보낸다.
```

## 14. add r1, r2, #5에서 r1과 r2는 immediate인가

`add r1, r2, #5`에서 `r1`과 `r2`는 immediate가 아니라 **레지스터**다. immediate 방식인 것은 `#5`다.

보통 이 명령어는 다음 뜻이다.

```text
r1 = r2 + 5
```

각 부분을 나누면 다음과 같다.

```text
add  r1, r2, #5
     │   │   └─ 즉시값(immediate) 5
     │   └──── 원본 레지스터(source register)
     └──────── 목적지 레지스터(destination register)
```

즉, `r2`라는 레지스터 안에 들어 있는 값에 숫자 `5`를 더하고, 결과를 `r1` 레지스터에 저장한다.

```text
r2 안의 값 = 10

add r1, r2, #5

r1 = 15
```

여기서 중요한 점은 `r1`, `r2`가 메모리 주소를 뜻하는 것이 아니라는 것이다. 이들은 CPU 내부 레지스터 이름이다.

반면 `#5`는 레지스터 이름도 아니고 메모리 주소도 아니다. 명령어 안에 직접 들어 있는 상수 값이다.

```text
r1  -> 레지스터
r2  -> 레지스터
#5  -> 즉시값
```

주소를 사용하는 명령어는 보통 문법이 다르게 표현된다. 예를 들어 가상의 어셈블리로 표현하면 다음처럼 볼 수 있다.

```text
load r1, [1000]
```

이 경우 `[1000]`은 메모리 주소 `1000`에 있는 값을 읽으라는 의미로 볼 수 있다.

또는 레지스터가 주소를 담고 있고, 그 주소에 접근하는 방식도 있다.

```text
load r1, [r2]
```

이 경우 `r2` 자체의 값이 데이터가 아니라 메모리 주소로 해석된다. 그래서 `r2`가 가리키는 메모리 위치의 값을 읽어 `r1`에 넣는다.

정리하면 다음과 같다.

```text
add r1, r2, #5
- r1: 결과를 저장할 레지스터
- r2: 입력값이 들어 있는 레지스터
- #5: 명령어 안에 직접 들어 있는 즉시값

load r1, [r2]
- r1: 결과를 저장할 레지스터
- r2: 메모리 주소가 들어 있는 레지스터
- [r2]: r2가 가리키는 메모리 위치
```

따라서 `add r1, r2, #5`는 **레지스터 직접 사용 + 즉시값 사용**이 섞인 명령어라고 보면 된다.

## 15. 실제 명령어에서도 r1, r2라고 쓰는가

레지스터 이름은 ISA마다 다르다. `r1`, `r2`는 교재나 설명에서 자주 쓰는 일반적인 표기이고, 실제 ISA에서도 비슷하게 쓰는 경우가 있지만 항상 그런 것은 아니다.

예를 들어 ARM 계열에서는 예전 문법에서 `r0`, `r1`, `r2` 같은 이름을 많이 쓴다.

```asm
add r1, r2, #5
```

의미는 보통 다음과 같다.

```text
r1 = r2 + 5
```

RISC-V는 `x0`, `x1`, `x2` 같은 레지스터 이름을 쓰거나, 관례적 별칭인 `a0`, `sp`, `ra` 등을 쓴다.

```asm
addi x1, x2, 5
```

MIPS는 `$t0`, `$t1`, `$s0` 같은 이름을 쓴다.

```asm
addi $t0, $t1, 5
```

x86-64는 `rax`, `rbx`, `rcx`, `rdx`, `r8` 같은 이름을 쓴다.

```asm
add rax, 5
```

따라서 `r1`, `r2`는 "레지스터 1번, 레지스터 2번" 같은 설명용 표기로 이해하면 된다. 실제로 어떤 이름을 쓰는지는 CPU가 제공하는 ISA와 어셈블리 문법에 따라 달라진다.

```text
ISA = CPU가 이해하는 명령어 규칙
어셈블리 = 그 명령어를 사람이 읽기 쉽게 쓴 표기
레지스터 이름 = ISA/어셈블러마다 다름
```

## 16. direct와 indirect 주소 지정은 어떻게 쓰는가

주소 지정 방식은 명령어가 피연산자를 어디서 가져오는지 표현하는 방식이다.

먼저 `immediate`는 값이 명령어 안에 직접 들어 있는 방식이다.

```asm
add r1, r2, #5
```

```text
r2 값에 숫자 5를 더함
#5 자체가 데이터
```

`register addressing`은 피연산자가 레지스터에 있는 방식이다.

```asm
add r1, r2, r3
```

```text
r1 = r2 + r3
r2와 r3 안의 값을 사용
```

`direct addressing`은 명령어 안에 메모리 주소가 직접 들어 있는 방식이다.

```asm
load r1, [1000]
```

```text
메모리 주소 1000에 있는 값을 읽어서 r1에 저장
```

여기서 `1000`은 데이터 자체가 아니라 메모리 주소다.

```text
메모리[1000] = 42

load r1, [1000]

r1 = 42
```

`indirect addressing`은 명령어 안에 실제 메모리 주소가 직접 들어 있는 것이 아니라, 어떤 레지스터나 메모리 위치에 들어 있는 값을 주소로 해석하는 방식이다.

대표적으로 레지스터 간접 주소 지정은 다음처럼 쓴다.

```asm
load r1, [r2]
```

의미는 다음과 같다.

```text
r2 안에 들어 있는 값을 주소로 사용
그 주소의 메모리 값을 읽어서 r1에 저장
```

예를 들어 다음과 같다고 하자.

```text
r2 = 1000
메모리[1000] = 42

load r1, [r2]

r1 = 42
```

즉, `r2` 자체의 값 `1000`을 데이터로 쓰는 것이 아니라, `1000`번지를 가리키는 주소로 사용한다.

차이를 정리하면 다음과 같다.

```text
immediate
- 명령어 안의 값 자체를 사용
- add r1, r2, #5

register
- 레지스터 안의 값을 사용
- add r1, r2, r3

direct
- 명령어 안의 주소가 가리키는 메모리 값을 사용
- load r1, [1000]

indirect
- 레지스터 안의 값을 주소로 해석하고, 그 주소의 메모리 값을 사용
- load r1, [r2]
```

주의할 점은 ISA마다 실제 문법이 다르다는 것이다. 어떤 ISA는 `[r2]`처럼 쓰고, 어떤 ISA는 `0(r2)`처럼 쓰고, x86은 `[rbx]`처럼 쓴다. 하지만 핵심 개념은 같다.

```text
direct = 주소가 명령어에 직접 있음
indirect = 주소가 레지스터나 다른 메모리 위치를 한 번 거쳐서 나옴
```

## 17. 캐시에서 말하는 지역성이란 무엇인가

지역성(locality)은 프로그램이 메모리에 접근할 때 보이는 반복적인 패턴이다. 캐시는 이 지역성을 이용해서 CPU가 곧 쓸 가능성이 높은 데이터를 미리 가까운 곳에 보관한다.

핵심은 다음과 같다.

```text
방금 쓴 데이터는 곧 다시 쓸 가능성이 높다.
방금 쓴 데이터 근처의 데이터도 곧 쓸 가능성이 높다.
```

지역성은 보통 두 가지로 나눈다.

### 시간 지역성

시간 지역성은 **최근에 접근한 데이터가 가까운 미래에 다시 접근될 가능성이 높다**는 성질이다.

예를 들어 반복문에서 같은 변수를 계속 사용하면 시간 지역성이 생긴다.

```c
for (int i = 0; i < 100; i++) {
    sum += i;
}
```

여기서 `sum`은 반복문이 돌 때마다 계속 사용된다. 그래서 한 번 캐시에 올라오면 다시 쓰일 가능성이 높다.

```text
sum 접근
sum 다시 접근
sum 또 접근
```

캐시는 이런 데이터를 RAM보다 가까운 곳에 두어 빠르게 다시 사용할 수 있게 한다.

### 공간 지역성

공간 지역성은 **어떤 주소에 접근하면 그 근처 주소도 곧 접근될 가능성이 높다**는 성질이다.

예를 들어 배열을 순서대로 읽으면 공간 지역성이 좋다.

```c
for (int i = 0; i < 100; i++) {
    sum += arr[i];
}
```

`arr[0]` 다음에는 `arr[1]`, 그 다음에는 `arr[2]`를 읽는다. 메모리상 배열 원소들은 보통 연속적으로 배치되어 있으므로, 근처 데이터가 계속 사용된다.

```text
arr[0] -> arr[1] -> arr[2] -> arr[3]
```

캐시는 보통 데이터를 한 바이트나 한 정수만 가져오지 않고, **캐시 라인(cache line)** 단위로 근처 데이터를 함께 가져온다. 예를 들어 `arr[0]`을 가져올 때 그 주변의 `arr[1]`, `arr[2]`도 같이 캐시에 들어올 수 있다.

그래서 다음 접근이 `arr[1]`이면 RAM까지 가지 않고 캐시에서 바로 찾을 가능성이 높아진다.

### 지역성이 나쁜 경우

반대로 메모리를 불규칙하게 여기저기 접근하면 지역성이 낮다.

```text
주소 1000 접근
주소 800000 접근
주소 42 접근
주소 9000000 접근
```

이런 패턴은 캐시에 올려둔 데이터가 다시 쓰이기 어렵기 때문에 캐시 효율이 떨어진다.

정리하면 다음과 같다.

```text
시간 지역성
- 최근 사용한 데이터를 다시 사용할 가능성
- 예: 반복문 안에서 같은 변수 재사용

공간 지역성
- 접근한 데이터 근처의 데이터를 사용할 가능성
- 예: 배열을 순서대로 읽기

캐시
- 지역성을 이용해서 자주 쓰거나 곧 쓸 가능성이 높은 데이터를 CPU 가까이에 보관
```

따라서 캐시가 지역성을 이용한다는 말은 **프로그램의 메모리 접근 패턴을 이용해, CPU가 다음에 필요로 할 가능성이 높은 데이터를 미리 빠른 저장공간에 두는 것**을 의미한다.

## 18. 캐시라인은 메모리에서 캐시로 가져오는 단위인가, 크기인가

캐시라인(cache line)은 **캐시가 메모리 데이터를 가져오고 저장하고 교체하는 기본 단위**다. 그래서 "단위"라고 해도 맞고, 그 단위가 보통 몇 바이트인지 정해져 있으므로 "크기"라고 해도 맞다.

예를 들어 어떤 CPU의 캐시라인 크기가 64바이트라고 하자. CPU가 메모리 주소 `1000`의 4바이트 정수 하나만 필요하더라도, 캐시는 그 정수 하나만 가져오는 것이 아니라 그 주변을 포함한 64바이트 덩어리를 가져온다.

```text
CPU가 필요한 값: 메모리[1000]의 4바이트
실제로 캐시에 올라오는 것: 1000 근처의 64바이트 캐시라인
```

단순화하면 다음과 같다.

```text
메모리
[주소 1000 ~ 1063]  64바이트
        ↓
캐시
캐시라인 1개
```

이렇게 하는 이유는 공간 지역성 때문이다. `arr[0]`을 읽은 뒤 곧 `arr[1]`, `arr[2]`를 읽을 가능성이 높기 때문에 근처 데이터를 함께 가져오는 것이 유리하다.

```c
sum += arr[0];
sum += arr[1];
sum += arr[2];
```

만약 `arr[0]`, `arr[1]`, `arr[2]`가 같은 캐시라인 안에 있으면, 첫 접근 때만 RAM에서 캐시라인을 가져오고 이후 접근은 캐시에서 처리할 수 있다.

캐시라인은 가져올 때만의 단위가 아니라 캐시 내부에서 관리되는 단위이기도 하다.

```text
캐시에 저장되는 단위 = 캐시라인
캐시에서 교체되는 단위 = 캐시라인
캐시 일관성 관리 단위 = 보통 캐시라인
```

정리하면 다음과 같다.

```text
캐시라인 = 캐시와 메모리 사이에서 오가는 데이터 블록
캐시라인 크기 = 그 블록의 바이트 수
일반적인 크기 = 64바이트인 경우가 많음
```

따라서 캐시라인은 **데이터를 가져오는 단위**이면서, 동시에 **그 단위의 크기**를 말할 때도 쓰는 표현이다.

## 19. 서로 다른 메모리 블록이 같은 캐시 위치에서 충돌한다는 뜻

캐시는 RAM보다 훨씬 작다. 그래서 메모리의 모든 블록을 캐시에 동시에 올릴 수 없다. 캐시는 메모리 블록을 아무 위치에나 두는 것이 아니라, **매핑 방식**이라는 규칙에 따라 캐시 안의 어느 위치에 둘지 정한다.

여기서 메모리 블록은 캐시라인 크기만큼 나눈 메모리 덩어리라고 보면 된다.

```text
메모리 블록 = 캐시에 올라올 수 있는 메모리 덩어리
캐시라인 = 캐시에 저장되는 한 칸짜리 데이터 덩어리
```

예를 들어 캐시에 칸이 4개밖에 없다고 해보자.

```text
캐시
인덱스 0: [    ]
인덱스 1: [    ]
인덱스 2: [    ]
인덱스 3: [    ]
```

메모리에는 블록이 훨씬 많다.

```text
메모리 블록 0
메모리 블록 1
메모리 블록 2
메모리 블록 3
메모리 블록 4
메모리 블록 5
메모리 블록 6
메모리 블록 7
...
```

직접 매핑(direct mapped cache)에서는 각 메모리 블록이 들어갈 수 있는 캐시 위치가 딱 하나로 정해진다. 보통 단순화하면 다음 규칙처럼 생각할 수 있다.

```text
캐시 인덱스 = 메모리 블록 번호 % 캐시 칸 개수
```

캐시 칸이 4개라면 다음처럼 배정된다.

```text
메모리 블록 0 -> 캐시 인덱스 0
메모리 블록 1 -> 캐시 인덱스 1
메모리 블록 2 -> 캐시 인덱스 2
메모리 블록 3 -> 캐시 인덱스 3
메모리 블록 4 -> 캐시 인덱스 0
메모리 블록 5 -> 캐시 인덱스 1
메모리 블록 6 -> 캐시 인덱스 2
메모리 블록 7 -> 캐시 인덱스 3
```

여기서 메모리 블록 0과 메모리 블록 4는 둘 다 캐시 인덱스 0에만 들어갈 수 있다.

```text
블록 0 -> 캐시 0번 칸
블록 4 -> 캐시 0번 칸
```

만약 프로그램이 블록 0을 사용해서 캐시 0번 칸에 올려 둔 상태에서, 블록 4에 접근하면 블록 4도 캐시 0번 칸에 들어가야 한다. 그러면 기존의 블록 0은 밀려난다.

```text
처음:
캐시 0번 칸 = 블록 0

블록 4 접근:
캐시 0번 칸 = 블록 4
블록 0은 제거됨
```

이것이 **서로 다른 메모리 블록이 같은 캐시 위치에서 충돌한다**는 뜻이다. 둘 다 같은 캐시 칸에만 들어갈 수 있어서 동시에 캐시에 있을 수 없는 상황이다.

문제가 되는 패턴은 다음과 같다.

```text
블록 0 접근 -> 캐시 0번 칸에 블록 0 적재
블록 4 접근 -> 캐시 0번 칸에 블록 4 적재, 블록 0 제거
블록 0 접근 -> 블록 0이 없으므로 다시 적재, 블록 4 제거
블록 4 접근 -> 블록 4가 없으므로 다시 적재, 블록 0 제거
```

이렇게 반복되면 캐시가 있는데도 계속 미스가 난다. 이것을 충돌 미스(conflict miss)라고 한다.

## 20. 캐시 매핑 방식이란 무엇인가

캐시 매핑 방식은 **메모리 블록을 캐시의 어느 위치에 둘 수 있는지 정하는 규칙**이다.

대표적으로 세 가지가 있다.

```text
직접 매핑
완전 연관 매핑
집합 연관 매핑
```

### 직접 매핑

직접 매핑은 메모리 블록 하나가 들어갈 수 있는 캐시 위치가 딱 하나인 방식이다.

```text
메모리 블록 N -> 정해진 캐시 인덱스 하나
```

예를 들어 캐시 칸이 4개면 다음처럼 된다.

```text
블록 0, 4, 8, 12 -> 캐시 0번 칸
블록 1, 5, 9, 13 -> 캐시 1번 칸
블록 2, 6, 10, 14 -> 캐시 2번 칸
블록 3, 7, 11, 15 -> 캐시 3번 칸
```

장점은 구조가 단순하고 빠르다는 것이다. 단점은 같은 위치로 배정되는 블록들이 자주 번갈아 사용되면 충돌이 많이 난다는 것이다.

### 완전 연관 매핑

완전 연관 매핑은 메모리 블록이 캐시의 어느 칸에나 들어갈 수 있는 방식이다.

```text
메모리 블록 N -> 캐시의 아무 빈 칸 가능
```

예를 들어 블록 0도 캐시 0번, 1번, 2번, 3번 어디든 들어갈 수 있다. 그래서 직접 매핑보다 충돌이 적다.

하지만 단점이 있다. 어떤 메모리 블록이 캐시에 있는지 확인하려면 캐시 전체를 비교해야 한다. 그래서 하드웨어가 복잡하고 비용이 커진다.

### 집합 연관 매핑

집합 연관 매핑은 직접 매핑과 완전 연관 매핑의 중간 방식이다.

캐시를 여러 집합(set)으로 나누고, 각 메모리 블록은 특정 집합 하나에 배정된다. 하지만 그 집합 안에서는 여러 칸 중 하나에 들어갈 수 있다.

예를 들어 2-way set associative cache는 각 집합에 칸이 2개 있는 방식이다.

```text
캐시
Set 0: [ way 0 ] [ way 1 ]
Set 1: [ way 0 ] [ way 1 ]
```

메모리 블록이 Set 0으로 배정되면, Set 0 안의 두 칸 중 하나에 들어갈 수 있다.

```text
블록 0 -> Set 0의 way 0 또는 way 1
블록 2 -> Set 0의 way 0 또는 way 1
```

직접 매핑처럼 딱 한 칸만 가능한 것보다는 유연하고, 완전 연관처럼 전체 캐시 어디든 가능한 것보다는 하드웨어 부담이 작다.

정리하면 다음과 같다.

```text
직접 매핑
- 갈 수 있는 캐시 위치가 딱 1개
- 빠르고 단순함
- 충돌이 많을 수 있음

완전 연관 매핑
- 캐시 어디든 갈 수 있음
- 충돌이 적음
- 비교 회로가 복잡함

집합 연관 매핑
- 정해진 set 안의 여러 way 중 하나로 감
- 직접 매핑과 완전 연관 매핑의 절충안
- 실제 CPU 캐시에서 많이 사용
```

핵심은 **캐시가 작기 때문에 메모리 블록을 캐시 위치에 배정하는 규칙이 필요하고, 그 규칙 때문에 서로 다른 메모리 블록이 같은 캐시 위치를 두고 충돌할 수 있다**는 것이다.

## 21. 메모리 블록은 언제 누가 캐시로 가져오는가

메모리 블록은 CPU가 어떤 주소의 데이터나 명령어를 필요로 했는데, 그 주소에 해당하는 데이터가 캐시에 없을 때 가져온다. 이 상황을 캐시 미스(cache miss)라고 한다.

흐름은 다음과 같다.

```text
CPU가 주소 X의 데이터를 요청
        ↓
L1 캐시 확인
        ↓
없으면 L2/L3 또는 RAM까지 확인
        ↓
해당 주소 X가 포함된 메모리 블록을 캐시라인 단위로 가져옴
        ↓
CPU에 필요한 값을 전달
        ↓
가져온 블록은 캐시에 남겨 둠
```

여기서 실제로 가져오는 일은 CPU 안의 캐시 컨트롤러, 메모리 컨트롤러, 캐시 계층 하드웨어가 처리한다. 프로그램이나 커널이 일반적인 메모리 접근마다 직접 "이 블록을 캐시에 넣어라"라고 명령하는 것은 아니다.

```text
누가 가져오나?
- CPU의 캐시/메모리 하드웨어가 자동으로 가져옴

언제 가져오나?
- CPU가 접근한 주소가 캐시에 없을 때
```

처음 접근해서 캐시에 없으면 cold miss가 난다.

```text
처음 arr[0] 접근
        ↓
캐시에 없음
        ↓
cold miss
        ↓
arr[0]이 포함된 캐시라인을 RAM에서 가져옴
```

## 22. 캐시에서 밀려난 블록은 없어지는가

캐시에서 어떤 블록이 밀려난다고 해서 원본 데이터가 RAM에서 사라지는 것은 아니다. 캐시는 RAM의 일부 데이터를 복사해서 가까이에 둔 공간이다.

```text
RAM = 원본이 있는 큰 저장 공간
캐시 = RAM 데이터 일부의 빠른 복사본
```

예를 들어 메모리 블록 A가 캐시에 올라와 있다가, 다른 블록 B 때문에 교체된다고 하자.

```text
캐시:
[블록 A]

블록 B가 필요함

캐시:
[블록 B]
```

이때 블록 A는 캐시에서는 사라진다. 하지만 RAM에는 여전히 블록 A에 해당하는 메모리 내용이 있다.

```text
캐시에서 제거됨 = 캐시 복사본이 사라짐
RAM에서 제거됨 = 아님
```

다만 쓰기(write)가 있었는지에 따라 처리가 달라진다.

### 읽기만 한 블록

캐시에 올라온 뒤 읽기만 했다면 캐시의 복사본과 RAM의 원본이 같다. 이 경우 캐시에서 밀려날 때 그냥 버려도 된다.

```text
RAM의 블록 A = 10
캐시의 블록 A = 10

캐시에서 블록 A 제거
-> 그냥 버려도 됨
```

### 수정된 블록

캐시에서 값을 수정했다면 캐시의 블록이 RAM보다 최신일 수 있다. 이런 블록을 dirty block 또는 dirty cache line이라고 한다.

```text
RAM의 블록 A = 10
캐시의 블록 A = 20
```

이 경우 캐시에서 밀려나기 전에 변경된 내용을 RAM에 써야 한다. 그렇지 않으면 수정 내용이 사라진다.

```text
dirty 블록 교체
        ↓
캐시의 최신 값을 RAM에 반영
        ↓
캐시에서 제거
```

정리하면 다음과 같다.

```text
깨끗한 블록(clean) = 캐시에서 그냥 버려도 됨
더러운 블록(dirty) = RAM에 먼저 써야 함
```

## 23. RAM 16GB가 미리 블록으로 다 나뉘어 있고 캐시 주소가 정해져 있는가

RAM이 실제로 "블록 객체"들로 미리 잘라져 저장되는 것은 아니다. RAM은 기본적으로 연속된 주소 공간이다.

```text
주소 0
주소 1
주소 2
주소 3
...
```

하지만 캐시는 데이터를 캐시라인 크기 단위로 가져오기 때문에, CPU와 캐시 하드웨어는 메모리 주소를 캐시라인 크기 기준으로 해석한다.

예를 들어 캐시라인 크기가 64바이트라면, 주소를 개념적으로 다음처럼 묶어 볼 수 있다.

```text
블록 0: 주소 0 ~ 63
블록 1: 주소 64 ~ 127
블록 2: 주소 128 ~ 191
블록 3: 주소 192 ~ 255
...
```

즉, 미리 물리적으로 블록을 만들어 놓는 것이 아니라, **주소를 계산할 때 캐시라인 크기 단위로 나누어 보는 것**이다.

메모리 블록 번호는 대략 다음처럼 계산할 수 있다.

```text
메모리 블록 번호 = 메모리 주소 / 캐시라인 크기
```

캐시 위치도 미리 표로 전부 저장해두는 것이 아니라, 주소의 일부 비트를 이용해 그때그때 계산한다.

직접 매핑을 단순화하면 다음과 같다.

```text
캐시 인덱스 = 메모리 블록 번호 % 캐시 라인 개수
```

실제 하드웨어에서는 주소 비트를 보통 다음처럼 나누어 사용한다.

```text
메모리 주소
[ tag ][ index ][ offset ]
```

- offset: 캐시라인 안에서 몇 번째 바이트인지
- index: 캐시의 어느 위치 또는 set으로 갈지
- tag: 그 캐시 위치에 들어 있는 것이 정확히 어떤 메모리 블록인지 확인하는 값

예를 들어 어떤 주소에 접근하면 CPU 캐시는 다음처럼 판단한다.

```text
1. 주소의 index 비트로 캐시 위치를 찾음
2. 그 위치에 저장된 tag와 현재 주소의 tag를 비교
3. tag가 같으면 cache hit
4. tag가 다르면 cache miss
5. miss면 해당 주소가 포함된 메모리 블록을 가져옴
```

따라서 "RAM 16GB를 미리 전부 블록으로 나누고 각 블록의 캐시 위치를 미리 저장해둔다"기보다는, **메모리 주소가 들어올 때 캐시 하드웨어가 주소 비트를 이용해 어느 블록이고 어느 캐시 위치로 갈지 계산한다**고 보는 것이 정확하다.

정리하면 다음과 같다.

```text
메모리 블록
- 캐시라인 크기 기준으로 본 연속 주소 범위
- RAM에 별도 객체로 미리 만들어지는 것은 아님

캐시로 가져오는 시점
- CPU가 접근했는데 캐시에 없을 때

가져오는 주체
- CPU의 캐시/메모리 하드웨어

캐시에서 밀려난 블록
- 캐시 복사본은 사라짐
- RAM 원본은 남아 있음
- dirty면 RAM에 먼저 반영해야 함

매핑
- 미리 거대한 표를 만들어두는 것이 아니라 주소 비트로 계산
```

## 24. write-through와 write-back은 dirty 블록 이야기인가

write-through와 write-back은 **CPU가 캐시에 값을 쓸 때 RAM에는 언제 반영할지**를 정하는 캐시 쓰기 정책이다. dirty 블록 개념은 특히 write-back 정책에서 중요하다.

### write-through

write-through는 캐시에 쓸 때 메모리에도 바로 같이 쓰는 방식이다.

```text
CPU가 값 20을 씀
        ↓
캐시도 20으로 갱신
        ↓
RAM도 20으로 갱신
```

이 방식에서는 캐시와 RAM이 계속 같은 값을 유지하기 쉽다.

```text
캐시의 A = 20
RAM의 A = 20
```

그래서 캐시라인이 교체될 때 그냥 버려도 된다. 이미 RAM에 최신 값이 반영되어 있기 때문이다.

```text
write-through
- 캐시와 RAM을 동시에 갱신
- dirty 상태 관리가 단순함
- 대신 매번 RAM까지 써야 해서 느릴 수 있음
```

### write-back

write-back은 먼저 캐시에만 쓰고, RAM에는 나중에 반영하는 방식이다.

```text
CPU가 값 20을 씀
        ↓
캐시만 20으로 갱신
        ↓
RAM은 아직 예전 값 10
```

이때 캐시의 값이 RAM보다 최신이다.

```text
캐시의 A = 20
RAM의 A = 10
```

이런 캐시라인을 dirty cache line 또는 dirty block이라고 한다.

dirty 블록은 캐시에서 밀려날 때 그냥 버리면 안 된다. RAM에는 아직 최신 값이 없기 때문이다.

```text
dirty 블록 교체
        ↓
캐시의 최신 값을 RAM에 먼저 씀
        ↓
캐시에서 제거
```

따라서 dirty는 write-back에서 필요한 표시라고 볼 수 있다.

```text
write-back
- 먼저 캐시에만 씀
- RAM 반영은 나중에 함
- 캐시라인에 dirty 표시가 필요함
- 교체될 때 dirty면 RAM에 write-back 해야 함
```

정리하면 다음과 같다.

```text
write-through
- 쓸 때마다 캐시와 RAM을 같이 갱신
- 캐시에서 밀려나도 보통 그냥 버릴 수 있음
- dirty 관리 부담이 작음

write-back
- 쓸 때는 캐시만 갱신
- RAM은 나중에 갱신
- 캐시와 RAM 값이 달라질 수 있음
- 이때 dirty 표시가 필요함
```

즉, 사용자가 적은 write-back 설명의 "나중에 메모리에 반영한다"가 바로 dirty 블록과 연결된다. **dirty 블록은 write-back 방식에서 아직 RAM에 반영되지 않은 최신 캐시 데이터**라고 이해하면 된다.

## 25. 캐시 히트를 늘리고 미스를 줄이려면 어떻게 해야 하는가

캐시 히트를 늘리려면 프로그램이 **시간 지역성**과 **공간 지역성**을 잘 활용하도록 만들어야 한다. 즉, 최근에 쓴 데이터를 다시 쓰고, 메모리상 가까운 데이터를 연속해서 접근하는 방식이 유리하다.

### 1. 배열을 순서대로 접근하기

배열은 보통 메모리에 연속적으로 배치된다. 그래서 앞에서부터 순서대로 접근하면 공간 지역성이 좋아진다.

```c
for (int i = 0; i < n; i++) {
    sum += arr[i];
}
```

이 코드는 `arr[0]`, `arr[1]`, `arr[2]`처럼 가까운 주소를 순서대로 읽는다. 캐시라인에 주변 데이터가 같이 올라오기 때문에 캐시 히트가 늘어날 수 있다.

반대로 무작위 접근은 캐시 효율이 나쁘다.

```c
for (int i = 0; i < n; i++) {
    sum += arr[random_index[i]];
}
```

주소가 여기저기 튀면 캐시에 올려둔 데이터가 다시 쓰일 가능성이 낮아진다.

### 2. 2차원 배열은 메모리 배치 순서에 맞게 순회하기

C 계열 언어의 2차원 배열은 보통 row-major 방식으로 저장된다. 즉, 같은 행의 원소들이 메모리에 연속적으로 배치된다.

좋은 접근:

```c
for (int i = 0; i < rows; i++) {
    for (int j = 0; j < cols; j++) {
        sum += arr[i][j];
    }
}
```

나쁜 접근:

```c
for (int j = 0; j < cols; j++) {
    for (int i = 0; i < rows; i++) {
        sum += arr[i][j];
    }
}
```

첫 번째 코드는 메모리에 연속된 방향으로 접근한다. 두 번째 코드는 행을 건너뛰며 접근하므로 캐시라인을 효율적으로 쓰지 못할 수 있다.

### 3. 자주 쓰는 데이터는 가까이 모아두기

함께 자주 쓰이는 데이터는 메모리상 가까이 두는 것이 좋다. 그래야 한 캐시라인에 같이 들어올 가능성이 커진다.

예를 들어 어떤 연산에서 `x`, `y`, `z`만 자주 쓰는데 구조체 안에 잘 모여 있으면 캐시 효율이 좋아질 수 있다.

```c
typedef struct {
    float x;
    float y;
    float z;
} Position;
```

반대로 실제로 거의 쓰지 않는 큰 데이터가 중간에 끼어 있으면 캐시라인 낭비가 생길 수 있다.

### 4. 한 번 가져온 데이터를 최대한 재사용하기

캐시에 올라온 데이터를 밀려나기 전에 여러 번 쓰면 시간 지역성이 좋아진다.

예를 들어 큰 행렬 연산에서는 전체 데이터를 한 번에 훑기보다 작은 블록 단위로 나누어 처리하는 기법을 사용한다. 이것을 blocking 또는 tiling이라고 한다.

```text
큰 데이터를 한 번에 처리
-> 캐시에 다 못 들어가서 자주 밀려남

작은 블록 단위로 처리
-> 캐시에 들어온 블록을 여러 번 재사용
```

### 5. 불필요하게 큰 데이터 구조를 피하기

캐시는 작다. 데이터 구조가 너무 크거나, 실제로 필요한 값보다 많은 값을 같이 끌고 다니면 캐시 공간을 낭비한다.

```text
필요한 데이터가 작고 가까이 있음
-> 캐시 효율 좋음

필요 없는 데이터까지 크고 넓게 퍼져 있음
-> 캐시 효율 나쁨
```

### 6. 연결 리스트보다 배열이 캐시에 유리한 경우가 많다

연결 리스트는 각 노드가 메모리 여기저기에 흩어질 수 있다.

```text
노드 A -> 멀리 있는 노드 B -> 또 멀리 있는 노드 C
```

이런 구조는 공간 지역성이 낮다. 반면 배열은 연속된 메모리에 배치되므로 순차 접근에서 캐시에 유리하다.

```text
배열: arr[0] arr[1] arr[2] arr[3]
```

그래서 단순 순회가 많다면 연결 리스트보다 배열, 벡터 같은 연속 자료구조가 더 빠를 수 있다.

### 7. 충돌 미스를 줄이도록 접근 패턴을 조정하기

직접 매핑이나 낮은 associativity의 캐시에서는 서로 다른 메모리 블록이 같은 캐시 위치로 몰려 충돌할 수 있다. 특정 간격으로 큰 배열 여러 개를 번갈아 접근하면 이런 문제가 생길 수 있다.

```text
A의 어떤 블록 -> 캐시 0번 위치
B의 어떤 블록 -> 캐시 0번 위치
A 접근할 때 B가 밀려남
B 접근할 때 A가 밀려남
```

이런 경우 데이터 배치, 패딩, 접근 순서 변경으로 충돌을 줄일 수 있다.

### 정리

캐시 히트를 늘리는 핵심 전략은 다음과 같다.

```text
순차 접근하기
자주 쓰는 데이터 가까이 배치하기
한 번 가져온 데이터 여러 번 재사용하기
큰 데이터를 작은 블록 단위로 처리하기
불필요하게 큰 자료구조 피하기
무작위 접근 줄이기
연결 리스트보다 연속 배열을 고려하기
```

한 줄로 정리하면, **CPU가 다음에 쓸 데이터를 캐시가 예측하기 쉽게 메모리를 접근하도록 만드는 것**이 캐시 히트를 늘리는 방법이다.

## 26. C의 2차원 배열은 왜 행 기준으로 저장되는가

C 계열 언어의 2차원 배열은 보통 **행 우선(row-major)** 방식으로 저장된다. 예를 들어 다음 배열이 있다고 하자.

```c
int arr[3][4];
```

이 배열은 개념적으로는 3행 4열이다.

```text
arr[0][0] arr[0][1] arr[0][2] arr[0][3]
arr[1][0] arr[1][1] arr[1][2] arr[1][3]
arr[2][0] arr[2][1] arr[2][2] arr[2][3]
```

하지만 실제 메모리는 2차원 격자가 아니라 1차원 주소 공간이다. 그래서 2차원 배열도 결국 한 줄로 펼쳐서 저장해야 한다.

C에서는 다음 순서로 저장한다.

```text
arr[0][0]
arr[0][1]
arr[0][2]
arr[0][3]
arr[1][0]
arr[1][1]
arr[1][2]
arr[1][3]
arr[2][0]
arr[2][1]
arr[2][2]
arr[2][3]
```

즉, 같은 행의 원소들이 먼저 연속해서 저장된다.

이렇게 저장하는 이유는 C에서 `arr[3][4]`가 "4개짜리 int 배열 3개"로 해석되기 때문이다.

```text
arr = 행 3개짜리 배열
arr[0] = int 4개짜리 배열
arr[1] = int 4개짜리 배열
arr[2] = int 4개짜리 배열
```

그래서 `arr[0]`의 원소 4개가 연속으로 저장되고, 그 다음에 `arr[1]`의 원소 4개가 이어서 저장된다.

주소 계산도 이 구조를 따른다. `arr[i][j]`의 위치는 단순화하면 다음과 같다.

```text
arr[i][j] 주소 = 시작 주소 + ((i * 열 개수) + j) * 원소 크기
```

예를 들어 `int arr[3][4]`에서 `arr[1][2]`는 다음 위치다.

```text
((1 * 4) + 2) = 6번째 원소
```

## 27. 2차원 배열에서 행 기준 접근과 열 기준 접근 중 캐시 적중률이 높은 것은?

C처럼 행 우선 저장(row-major)인 언어에서는 보통 **행 기준 접근이 캐시 적중률이 높다**.

행 기준 접근은 다음과 같다.

```c
for (int i = 0; i < rows; i++) {
    for (int j = 0; j < cols; j++) {
        sum += arr[i][j];
    }
}
```

접근 순서는 다음과 같다.

```text
arr[0][0] -> arr[0][1] -> arr[0][2] -> arr[0][3]
arr[1][0] -> arr[1][1] -> arr[1][2] -> arr[1][3]
```

이 순서는 실제 메모리 저장 순서와 같다. 그래서 캐시라인 하나를 가져오면 그 안에 다음에 접근할 원소들이 같이 들어 있을 가능성이 높다.

반면 열 기준 접근은 다음과 같다.

```c
for (int j = 0; j < cols; j++) {
    for (int i = 0; i < rows; i++) {
        sum += arr[i][j];
    }
}
```

접근 순서는 다음과 같다.

```text
arr[0][0] -> arr[1][0] -> arr[2][0]
arr[0][1] -> arr[1][1] -> arr[2][1]
```

C의 메모리 배치에서는 `arr[0][0]` 다음에 바로 `arr[1][0]`이 있는 것이 아니다. 중간에 `arr[0][1]`, `arr[0][2]`, `arr[0][3]` 같은 같은 행의 원소들이 있다.

```text
메모리 저장 순서:
arr[0][0] arr[0][1] arr[0][2] arr[0][3] arr[1][0] arr[1][1] ...
```

따라서 열 기준 접근은 메모리상 멀리 떨어진 위치로 점프하면서 접근하는 패턴이 된다. 배열이 작아서 캐시에 다 들어가면 차이가 작을 수 있지만, 배열이 커지면 행 기준 접근이 보통 훨씬 유리하다.

정리하면 다음과 같다.

```text
C/C++/Java의 일반 2차원 배열 접근
- 행 기준 접근이 캐시 친화적
- 이유: 같은 행의 원소들이 메모리에 연속 배치됨

열 기준 접근
- 메모리상 큰 간격으로 건너뛰며 접근
- 캐시라인에 같이 올라온 근처 데이터를 잘 활용하지 못할 수 있음
```

다만 모든 언어가 같은 것은 아니다. Fortran, MATLAB, R 같은 환경은 열 우선(column-major) 저장을 사용한다. 그런 경우에는 열 기준 접근이 더 캐시 친화적일 수 있다.

```text
row-major 언어: 행 기준 접근 유리
column-major 언어: 열 기준 접근 유리
```
