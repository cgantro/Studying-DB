# DMA와 메모리 매핑 입출력 Q&A

## 1. DMA란 무엇인가

DMA는 **Direct Memory Access**의 약자다. 뜻은 장치가 CPU를 매번 거치지 않고 **메모리와 직접 데이터를 주고받는 방식**이다.

예를 들어 디스크에서 1GB 파일을 RAM으로 읽는다고 해보자.

CPU가 직접 옮긴다면 흐름은 단순히 다음처럼 볼 수 있다.

```text
디스크 -> CPU -> RAM
```

이 방식에서는 CPU가 계속 데이터 복사 작업에 묶인다. CPU는 원래 계산과 명령 실행을 해야 하는데, 큰 데이터를 옮기는 일에 시간을 많이 쓰게 된다.

DMA를 사용하면 흐름이 달라진다.

```text
CPU: 디스크야, 이 데이터를 RAM의 이 주소로 옮겨줘
        ↓
디스크 컨트롤러/DMA 컨트롤러가 직접 전송
        ↓
디스크 -> RAM
        ↓
전송 완료 후 인터럽트로 CPU에게 알림
```

즉, DMA는 **대량 데이터 전송을 CPU 대신 장치와 메모리 사이에서 처리하는 방식**이다.

정리하면 다음과 같다.

```text
DMA의 목적
- CPU가 큰 데이터 복사에 계속 묶이지 않게 함
- 장치와 메모리 사이의 전송을 효율적으로 처리함
```

## 2. 메모리 매핑 입출력이란 무엇인가

메모리 매핑 입출력(Memory-Mapped I/O, MMIO)은 **입출력 장치의 제어 레지스터를 메모리 주소처럼 다루는 방식**이다.

예를 들어 네트워크 카드에게 "전송 시작" 명령을 내리고 싶다고 하자. 특정 주소가 네트워크 카드의 제어 레지스터에 연결되어 있을 수 있다.

```text
0xFFFF0000 주소 = 네트워크 카드 제어 레지스터
```

CPU 입장에서는 특정 메모리 주소에 값을 쓰는 것처럼 보인다.

```c
*(int*)0xFFFF0000 = 1;
```

하지만 실제로는 RAM에 쓰는 것이 아니라, 네트워크 카드에게 다음 명령을 내리는 것이다.

```text
"전송 시작해"
```

즉, 메모리 매핑 입출력은 **장치 제어용 레지스터를 메모리 주소 공간에 배치해서 CPU가 load/store 명령으로 장치를 제어하게 하는 방식**이다.

## 3. DMA와 메모리 매핑 입출력의 차이

둘은 경쟁 개념이 아니다. 역할이 다르며 같이 쓰일 수 있다.

```text
메모리 매핑 입출력
- CPU가 장치를 제어하기 위해 특정 주소를 읽고 씀
- 장치에게 명령하거나 상태를 확인하는 창구

DMA
- 장치가 메모리와 직접 데이터를 주고받음
- 큰 데이터 전송을 CPU 대신 처리
```

예를 들어 디스크에서 데이터를 읽는 흐름은 다음처럼 볼 수 있다.

```text
1. CPU가 MMIO 주소에 설정값을 씀
   "디스크야, RAM 0x1000 위치로 4KB 읽어와"

2. 디스크 컨트롤러가 DMA로 데이터를 RAM에 직접 복사
   디스크 -> RAM

3. 전송이 끝나면 인터럽트 발생
   "CPU야, 다 읽었어"
```

한 줄로 정리하면 다음과 같다.

```text
메모리 매핑 입출력 = CPU가 장치에게 명령하는 창구
DMA = 장치가 CPU 대신 큰 데이터를 옮기는 방법
```

## 4. 디스크가 읽었다는 말의 의미

"디스크가 읽었다"는 말은 디스크에 데이터가 없던 것을 새로 만든다는 뜻이 아니다. 디스크에는 이미 파일 데이터가 저장되어 있다. 여기서 읽는다는 말은 **디스크에 저장된 데이터를 RAM으로 가져오는 작업**을 뜻한다.

저장 장치와 메모리의 역할은 다르다.

```text
디스크/SSD
- 파일이 오래 저장되는 장치
- 전원이 꺼져도 데이터가 유지됨
- CPU가 직접 명령어를 실행하거나 일반 변수처럼 바로 쓰기에는 느림

RAM
- 실행 중인 프로그램과 데이터가 올라오는 메모리
- CPU가 빠르게 접근할 수 있음
- 전원이 꺼지면 내용이 사라짐
```

예를 들어 `hello.txt`라는 파일이 SSD에 저장되어 있다고 하자.

```text
SSD
└─ hello.txt의 내용: "abc"
```

프로그램이 이 파일을 읽으려면, SSD에 있는 내용을 RAM의 버퍼로 가져와야 한다.

```c
read(fd, buffer, size);
```

흐름은 단순화하면 다음과 같다.

```text
1. 프로그램이 read 시스템 콜 호출
2. 커널이 디스크/SSD 컨트롤러에 읽기 요청
3. 디스크/SSD가 저장된 데이터를 읽어 RAM 버퍼로 전송
4. 전송이 끝나면 인터럽트로 CPU에게 알림
5. 프로그램은 RAM의 buffer에서 데이터를 사용
```

이때 "디스크 읽기 완료"는 다음 뜻이다.

```text
디스크/SSD에 저장되어 있던 데이터가
RAM의 지정된 위치로 복사 완료됨
```

즉, 디스크는 창고에 가깝고 RAM은 작업대에 가깝다.

```text
디스크/SSD = 장기 보관 창고
RAM = CPU가 작업하기 위해 올려두는 공간
```

파일이 디스크에 이미 있어도, CPU가 프로그램에서 바로 쓰려면 보통 그 내용이 RAM으로 올라와야 한다. 그래서 "파일을 읽는다"는 표현은 실제로는 **저장 장치에서 RAM으로 데이터를 가져온다**는 의미다.

정리하면 다음과 같다.

```text
디스크에 저장되어 있다
- 파일 데이터가 보관되어 있다는 뜻

디스크에서 읽는다
- 그 데이터를 RAM으로 가져온다는 뜻

디스크 읽기 완료 인터럽트
- 장치가 RAM으로 데이터 전송을 끝냈다고 CPU/커널에 알리는 것
```
