# 프로세스 주소 공간 Q&A

## 1. 자신만의 주소 공간이란 무엇인가

프로세스가 자신만의 주소 공간을 가진다는 말은, 각 프로세스가 **자기만의 메모리 지도**를 가진다는 뜻이다.

예를 들어 프로세스 A와 프로세스 B가 둘 다 `0x1000`이라는 주소를 사용한다고 해도, 실제 RAM에서는 서로 다른 위치를 가리킬 수 있다.

```text
프로세스 A의 0x1000 -> 물리 메모리의 어떤 위치 A
프로세스 B의 0x1000 -> 물리 메모리의 다른 위치 B
```

프로그램이 보는 주소는 보통 물리 주소가 아니라 **가상 주소**다. 운영체제와 CPU는 페이지 테이블을 이용해서 가상 주소를 실제 물리 주소로 변환한다.

```text
프로세스가 보는 주소 = 가상 주소
RAM의 실제 위치 = 물리 주소
페이지 테이블 = 가상 주소를 물리 주소로 바꾸는 표
```

## 2. 왜 프로세스마다 주소 공간을 분리하는가

주소 공간을 분리하면 프로세스끼리 서로의 메모리를 마음대로 건드릴 수 없다. 그래서 안정성과 보안이 좋아진다.

예를 들어 프로세스 A가 버그 때문에 잘못된 주소에 값을 써도, 운영체제가 프로세스 B의 메모리까지 망가지지 않도록 막을 수 있다.

```text
프로세스 A
├─ 코드 영역
├─ 데이터 영역
├─ 힙
└─ 스택

프로세스 B
├─ 코드 영역
├─ 데이터 영역
├─ 힙
└─ 스택
```

겉으로는 비슷한 구조를 가지지만, 서로 독립된 주소 공간이다.

## 3. 스레드는 주소 공간을 공유한다

같은 프로세스 안의 스레드들은 주소 공간을 공유한다.

```text
프로세스 A
├─ 코드 영역        공유
├─ 힙              공유
├─ 전역 변수        공유
├─ 스레드 1의 스택  따로
└─ 스레드 2의 스택  따로
```

그래서 같은 프로세스 안의 스레드들은 힙이나 전역 변수에 있는 데이터를 쉽게 공유할 수 있다. 대신 동시에 같은 데이터를 수정하면 동기화 문제가 생길 수 있다.

## 4. 프로세스와 스레드 비교

```text
프로세스
- 자신만의 주소 공간을 가짐
- 다른 프로세스 메모리에 마음대로 접근할 수 없음
- 격리와 안정성이 좋음

스레드
- 같은 프로세스의 주소 공간을 공유함
- 힙, 전역 변수 등을 같이 사용하기 쉬움
- 공유 데이터 동기화가 필요할 수 있음
```

한 줄로 정리하면, **자신만의 주소 공간**이란 프로세스마다 독립된 가상 메모리 공간을 가지고, 같은 주소 숫자를 쓰더라도 다른 실제 메모리를 가리킬 수 있다는 뜻이다.

## 5. 프로세스 메모리 구조: code, data, heap, stack

프로세스가 실행되면 운영체제는 그 프로세스에게 독립된 가상 주소 공간을 제공한다. 이 주소 공간 안에는 보통 다음 영역들이 있다.

```text
프로세스 주소 공간
├─ code 영역
├─ data 영역
├─ heap 영역
└─ stack 영역
```

단순화해서 그림으로 보면 다음과 같다.

```text
낮은 주소
┌────────────────────┐
│ code 영역           │
├────────────────────┤
│ data 영역           │
├────────────────────┤
│ heap 영역           │  동적 할당 시 커짐
│        ↓            │
│                    │
│        ↑            │
│ stack 영역          │  함수 호출 시 커짐
└────────────────────┘
높은 주소
```

실제 운영체제와 CPU 구조에서는 주소 배치가 더 복잡할 수 있지만, 기본 개념은 이 그림으로 이해하면 된다.

## 6. code 영역

code 영역은 실행할 **기계어 명령어**가 들어 있는 영역이다.

예를 들어 다음 C 코드가 있다고 하자.

```c
int add(int a, int b) {
    return a + b;
}

int main() {
    int result = add(1, 2);
    return 0;
}
```

컴파일 후에는 `add`, `main` 같은 함수 코드가 CPU가 실행할 기계어 명령어로 바뀐다. 그 명령어들이 code 영역에 들어간다.

```text
code 영역
├─ main 함수의 기계어 명령어
├─ add 함수의 기계어 명령어
└─ 기타 실행 코드
```

CPU의 PC는 이 code 영역 안의 명령어 주소를 가리키며 실행을 진행한다.

```text
PC -> code 영역의 다음 명령어
```

code 영역은 보통 읽기 전용으로 보호된다. 실행 중인 프로그램이 자기 명령어를 실수로 덮어쓰면 위험하기 때문이다.

## 7. data 영역

data 영역은 전역 변수와 static 변수처럼 프로그램 실행 동안 오래 살아 있는 데이터를 저장한다.

예를 들어 다음 코드가 있다고 하자.

```c
int global_count = 10;
static int static_value = 20;

int main() {
    return 0;
}
```

`global_count`, `static_value` 같은 값은 함수가 끝나도 사라지지 않는다. 프로그램이 실행되는 동안 계속 존재한다.

```text
data 영역
├─ 전역 변수
├─ static 변수
└─ 초기화된 전역 데이터
```

data 영역은 보통 더 세부적으로 나눠 설명하기도 한다.

```text
data 영역
- 초기화된 전역/static 변수

bss 영역
- 0으로 초기화되거나 명시적으로 초기화하지 않은 전역/static 변수
```

예를 들어 다음 변수는 bss 영역에 들어갈 수 있다.

```c
int global_zero;
static int static_zero;
```

## 8. heap 영역

heap 영역은 프로그램 실행 중에 필요한 만큼 동적으로 할당하는 메모리 영역이다.

C에서는 `malloc`, C++에서는 `new`, Java/Python/JavaScript 같은 언어에서는 객체 생성이 heap 사용과 관련된다.

```c
int* p = malloc(sizeof(int) * 100);
```

이 코드는 정수 100개를 저장할 공간을 실행 중에 heap에서 할당받는다.

```text
heap 영역
└─ malloc/new 등으로 동적 할당된 데이터
```

heap은 함수 호출이 끝난다고 자동으로 사라지지 않는다. C에서는 `free`, C++에서는 `delete`를 해야 하고, Java/Python 같은 언어에서는 가비지 컬렉터가 더 이상 쓰지 않는 객체를 회수한다.

```text
stack 변수
- 함수가 끝나면 자동 정리

heap 객체
- 직접 해제하거나 GC가 회수할 때까지 유지
```

heap은 크기가 유동적으로 변한다. 큰 배열, 객체, 리스트, 트리, 그래프 같은 자료구조는 heap에 만들어지는 경우가 많다.

## 9. stack 영역

stack 영역은 함수 호출과 관련된 정보를 저장하는 영역이다.

함수가 호출되면 그 함수 실행에 필요한 정보가 stack에 쌓인다. 이 한 덩어리를 보통 스택 프레임(stack frame)이라고 한다.

스택 프레임에는 다음 정보가 들어갈 수 있다.

```text
함수의 지역 변수
함수 인자
반환 주소
이전 스택 프레임 정보
임시 저장 값
```

예를 들어 다음 코드가 있다고 하자.

```c
void foo() {
    int x = 10;
}

int main() {
    foo();
    return 0;
}
```

`main`이 실행되면 main의 스택 프레임이 생기고, `foo()`를 호출하면 foo의 스택 프레임이 그 위에 쌓인다.

```text
stack 영역
┌──────────────┐
│ foo 프레임    │
├──────────────┤
│ main 프레임   │
└──────────────┘
```

`foo`가 끝나면 foo의 스택 프레임은 제거된다.

```text
stack 영역
┌──────────────┐
│ main 프레임   │
└──────────────┘
```

SP는 현재 스택의 위치를 가리키는 레지스터다. 함수 호출과 반환이 일어날 때 SP가 움직인다.

```text
SP = 현재 스택 위치
```

## 10. heap과 stack의 차이

heap과 stack은 둘 다 데이터를 저장하지만 성격이 다르다.

```text
stack
- 함수 호출 단위로 자동 관리
- 지역 변수, 반환 주소, 함수 인자 저장
- 빠름
- 크기가 제한적
- 함수가 끝나면 해당 프레임이 사라짐

heap
- 실행 중 필요한 만큼 동적 할당
- 객체, 큰 배열, 오래 살아야 하는 데이터 저장
- stack보다 관리 비용이 큼
- 직접 해제하거나 GC가 회수해야 함
- 비교적 큰 데이터에 사용
```

예를 들어 다음 코드에서 `p` 자체는 지역 변수라 stack에 있을 수 있고, `malloc`으로 할당된 100개짜리 배열은 heap에 있다.

```c
void func() {
    int local = 10;
    int* p = malloc(sizeof(int) * 100);
}
```

```text
stack
├─ local
└─ p 변수

heap
└─ int 100개짜리 배열
```

주의할 점은 `p`는 heap 데이터의 주소를 들고 있는 포인터일 뿐이라는 것이다.

## 11. 스레드와 메모리 영역의 관계

같은 프로세스 안의 스레드들은 주소 공간을 공유한다. 그래서 code, data, heap은 공유한다.

하지만 각 스레드는 자기 stack을 가진다.

```text
프로세스 A
├─ code 영역       모든 스레드가 공유
├─ data 영역       모든 스레드가 공유
├─ heap 영역       모든 스레드가 공유
├─ 스레드 1 stack  따로
├─ 스레드 2 stack  따로
└─ 스레드 3 stack  따로
```

각 스레드가 stack을 따로 가지는 이유는, 스레드마다 실행 중인 함수 호출 흐름이 다르기 때문이다.

```text
스레드 1은 A 함수 실행 중
스레드 2는 B 함수 실행 중
```

각자 다른 함수 호출 상태와 지역 변수를 가져야 하므로 stack이 분리된다.

반대로 heap과 전역 변수는 공유하므로 여러 스레드가 동시에 수정하면 동기화가 필요할 수 있다.

## 12. 전체 정리

```text
code
- 실행할 기계어 명령어
- PC가 이 영역의 명령어를 가리킴
- 보통 읽기 전용

data
- 전역 변수, static 변수
- 프로그램 실행 동안 유지

heap
- 동적 할당 메모리
- malloc/new/객체 생성
- 직접 해제하거나 GC가 회수

stack
- 함수 호출 정보
- 지역 변수, 인자, 반환 주소
- 함수 호출 시 쌓이고 반환 시 제거
```

프로세스와 스레드 기준으로 보면 다음이 중요하다.

```text
프로세스마다 주소 공간은 분리됨
같은 프로세스 안의 스레드들은 code/data/heap을 공유함
각 스레드는 자기 stack을 따로 가짐
```

## 13. code, data, heap, stack과 주소 공간의 관계

code, data, heap, stack은 **프로세스 주소 공간 안에 들어 있는 영역들**이다.

즉, 주소 공간은 더 큰 개념이고, 그 안을 용도별로 나누어 보면 code, data, heap, stack 같은 영역이 있는 것이다.

```text
프로세스 주소 공간
├─ code 영역
├─ data 영역
├─ heap 영역
└─ stack 영역
```

프로세스마다 주소 공간이 다르다는 말은, 프로세스마다 자기만의 code/data/heap/stack 영역을 가진다는 뜻이다.

예를 들어 프로세스 A와 프로세스 B가 있다고 하자.

```text
프로세스 A의 주소 공간
├─ A의 code
├─ A의 data
├─ A의 heap
└─ A의 stack

프로세스 B의 주소 공간
├─ B의 code
├─ B의 data
├─ B의 heap
└─ B의 stack
```

두 프로세스가 같은 주소 숫자를 사용해도 실제로는 다른 메모리를 가리킬 수 있다.

```text
프로세스 A의 0x1000 -> A의 code 또는 data 일부
프로세스 B의 0x1000 -> B의 code 또는 data 일부
```

주소 숫자가 같다고 같은 RAM 위치를 뜻하는 것이 아니다. 각 프로세스는 자기 페이지 테이블을 통해 가상 주소를 물리 주소로 변환한다.

```text
프로세스 A
가상 주소 0x1000 -> 물리 주소 0xAAAA

프로세스 B
가상 주소 0x1000 -> 물리 주소 0xBBBB
```

그래서 프로세스 A가 자기 주소 공간의 heap을 수정해도 프로세스 B의 heap이 바뀌지 않는다.

```text
프로세스 A의 heap에 값 저장
-> A의 주소 공간만 변경
-> B의 주소 공간은 영향 없음
```

이것이 프로세스 격리다.

반면 같은 프로세스 안의 스레드들은 같은 주소 공간을 공유한다.

```text
프로세스 A의 주소 공간
├─ code 영역       스레드 1, 2가 공유
├─ data 영역       스레드 1, 2가 공유
├─ heap 영역       스레드 1, 2가 공유
├─ 스레드 1 stack
└─ 스레드 2 stack
```

따라서 스레드 1이 heap에 만든 객체를 스레드 2도 같은 주소로 접근할 수 있다. 그래서 스레드는 데이터 공유가 쉽지만, 동시에 수정하면 동기화가 필요하다.

정리하면 다음과 같다.

```text
주소 공간 = 프로세스가 사용할 수 있는 가상 메모리 전체 지도
code/data/heap/stack = 그 주소 공간 안의 영역들
프로세스마다 주소 공간이 다름 = 프로세스마다 자기 code/data/heap/stack을 가짐
스레드는 주소 공간을 공유 = 같은 code/data/heap을 같이 봄
```

## 14. 커널 메모리란 무엇인가

커널 메모리는 운영체제의 핵심 부분인 **커널이 사용하는 메모리 영역**이다.

사용자 프로그램도 메모리를 쓰고, 커널도 메모리를 쓴다. 다만 둘은 목적과 접근 권한이 다르다.

```text
사용자 메모리
- 일반 프로그램이 사용하는 영역
- 프로세스의 code/data/heap/stack 등이 있음

커널 메모리
- 운영체제 커널이 사용하는 영역
- 커널 코드와 커널 자료구조가 있음
```

커널 메모리에는 다음과 같은 것들이 들어갈 수 있다.

```text
커널 코드
프로세스 관리 정보
PCB 또는 task_struct 같은 자료구조
페이지 테이블 관련 정보
파일 시스템 캐시
장치 드라이버 상태 정보
네트워크 버퍼
커널 스택
```

예를 들어 PCB는 커널이 프로세스를 관리하기 위해 저장하는 정보이므로 커널 메모리에 있다.

```text
커널 메모리
├─ 프로세스 A의 PCB
├─ 프로세스 B의 PCB
├─ 스케줄러 자료구조
└─ 파일/장치 관리 정보
```

사용자 프로그램은 커널 메모리에 마음대로 접근할 수 없다. 만약 일반 프로그램이 커널 메모리를 직접 읽고 쓸 수 있다면 다른 프로세스의 정보나 운영체제 자체를 망가뜨릴 수 있기 때문이다.

그래서 CPU는 보통 권한 모드를 나눈다.

```text
사용자 모드
- 일반 프로그램 실행
- 커널 메모리에 직접 접근 불가

커널 모드
- 운영체제 커널 실행
- 하드웨어와 커널 메모리에 접근 가능
```

사용자 프로그램이 파일 읽기, 네트워크 전송, 프로세스 생성 같은 커널 기능을 쓰고 싶으면 직접 커널 메모리를 만지는 것이 아니라 시스템 콜을 사용한다.

```text
사용자 프로그램
        ↓ 시스템 콜
커널 모드로 전환
        ↓
커널이 커널 메모리와 장치를 사용해 작업 처리
        ↓
사용자 프로그램으로 복귀
```

프로세스 주소 공간과 연결해서 보면, 프로세스마다 사용자 영역은 분리되어 있지만 커널 영역은 운영체제가 관리하는 특권 영역이다.

단순화하면 다음처럼 볼 수 있다.

```text
프로세스 A의 가상 주소 공간
├─ 사용자 영역: A의 code/data/heap/stack
└─ 커널 영역: 커널 코드/자료구조에 대한 매핑

프로세스 B의 가상 주소 공간
├─ 사용자 영역: B의 code/data/heap/stack
└─ 커널 영역: 커널 코드/자료구조에 대한 매핑
```

사용자 영역은 프로세스마다 다르지만, 커널 영역은 운영체제가 보호하면서 사용한다. 실제 세부 구조는 운영체제마다 다르지만 핵심은 다음과 같다.

```text
커널 메모리 = 커널이 쓰는 보호된 메모리 영역
사용자 프로그램은 직접 접근 불가
커널 모드에서만 접근 가능
PCB 같은 운영체제 자료구조가 여기에 있음
```

## 15. 커널 메모리와 사용자 메모리는 둘 다 RAM 안에 있는가

맞다. 커널 메모리와 사용자 메모리는 결국 모두 **RAM을 사용한다**. 차이는 물리적으로 완전히 다른 종류의 메모리라서가 아니라, 운영체제와 CPU가 **권한과 용도에 따라 나누어 관리한다**는 점이다.

단순화하면 다음과 같다.

```text
RAM
├─ 커널이 사용하는 영역
│  ├─ 커널 코드
│  ├─ PCB/TCB
│  ├─ 페이지 테이블
│  └─ 장치/파일 관리 정보
│
└─ 사용자 프로세스들이 사용하는 영역
   ├─ 프로세스 A의 code/data/heap/stack
   ├─ 프로세스 B의 code/data/heap/stack
   └─ 프로세스 C의 code/data/heap/stack
```

하지만 프로그램이 보는 주소는 보통 물리 RAM 주소가 아니라 가상 주소다. 그래서 더 정확히는 각 프로세스의 가상 주소 공간 안에 사용자 영역과 커널 영역이 구분되어 있고, 이들이 실제 RAM의 물리 페이지들로 매핑된다.

```text
가상 주소 공간
        ↓ 페이지 테이블
물리 RAM
```

사용자 프로그램은 자기 사용자 영역에는 접근할 수 있지만, 커널 영역에는 직접 접근할 수 없다.

```text
사용자 모드
- 사용자 메모리 접근 가능
- 커널 메모리 직접 접근 불가

커널 모드
- 커널 메모리 접근 가능
- 필요한 경우 사용자 메모리도 검증 후 접근
```

예를 들어 프로세스 A가 실행 중일 때, A의 변수와 heap 데이터는 사용자 메모리에 있고, A를 관리하기 위한 PCB/TCB는 커널 메모리에 있다. 둘 다 최종적으로는 RAM 어딘가에 있지만 접근 권한이 다르다.

```text
프로세스 A의 int x
-> 사용자 메모리
-> A의 코드가 직접 접근 가능

프로세스 A의 PCB/TCB
-> 커널 메모리
-> 커널만 직접 접근 가능
```

정리하면 다음과 같다.

```text
커널 메모리와 사용자 메모리
- 둘 다 RAM을 사용함
- 물리적으로 완전히 다른 장치가 아님
- 운영체제와 CPU 권한 보호로 구분됨

사용자 메모리
- 일반 프로그램의 code/data/heap/stack

커널 메모리
- 커널 코드, PCB/TCB, 페이지 테이블, 장치 관리 정보
```
