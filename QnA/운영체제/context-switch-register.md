# 문맥 교환과 레지스터 Q&A

## 1. 프로세스마다 레지스터가 따로 있는가

프로세스 A용 레지스터, 프로세스 B용 레지스터가 CPU 안에 따로 있는 것은 아니다. **CPU 코어 안의 물리적 레지스터는 하나의 실행 흐름이 사용한다.** 문맥 교환이 일어나면 운영체제가 그 레지스터 값들을 메모리에 저장했다가, 다른 프로세스의 값을 다시 CPU 레지스터에 복원한다.

흐름은 다음과 같다.

```text
CPU 물리 레지스터에 프로세스 A 상태가 들어 있음
        ↓
운영체제가 A의 레지스터 값들을 PCB 또는 커널 스택에 저장
        ↓
운영체제가 B의 저장된 레지스터 값들을 CPU 물리 레지스터에 복원
        ↓
CPU는 이제 프로세스 B의 다음 명령어부터 실행
```

즉, 프로세스마다 레지스터 주소가 따로 할당되는 것이 아니라, **프로세스마다 "저장된 레지스터 값 묶음"이 메모리에 존재**한다고 보는 편이 정확하다.

## 2. 제어장치가 ALU 명령만 바꿔주는 것인가

문맥 교환은 제어장치가 ALU 명령만 바꿔주는 작업이 아니다. 운영체제가 CPU 상태 전체를 바꾸는 작업에 가깝다.

대표적으로 바뀌는 정보는 다음과 같다.

- PC: 다음에 실행할 명령어 주소
- SP: 현재 스택 위치
- 범용 레지스터: 계산 중이던 값, 함수 인자, 임시 값 등
- 상태 레지스터: 조건 플래그, CPU 상태 정보 등
- 페이지 테이블 관련 레지스터: 현재 프로세스의 가상 메모리 공간 정보

CPU는 PC가 가리키는 주소에서 명령어를 가져오고, 그 명령어를 디코딩한 뒤 필요한 ALU 동작을 수행한다. 따라서 문맥 교환 후에는 **PC와 주소 공간이 바뀌기 때문에 CPU가 가져오는 명령어 자체가 달라진다.**

## 3. 프로세스 A와 B의 명령어들은 어디에 저장되어 있는가

각 프로세스의 명령어들은 그 프로세스의 메모리 공간에 저장된다. 보통 실행 파일의 코드 영역이 메모리에 올라와 있고, CPU는 PC가 가리키는 주소에서 명령어를 가져온다.

프로세스 메모리 구조를 단순화하면 다음과 같다.

```text
프로세스 메모리 공간
├─ 코드 영역: 실행할 기계어 명령어
├─ 데이터 영역: 전역 변수, 정적 변수
├─ 힙: 동적 할당 메모리
└─ 스택: 함수 호출, 지역 변수, 반환 주소
```

프로세스 A와 B는 서로 다른 메모리 공간을 가진다.

```text
프로세스 A
└─ 코드 영역: A의 명령어들

프로세스 B
└─ 코드 영역: B의 명령어들
```

CPU가 프로세스 A를 실행 중이면 PC는 A의 코드 영역 안 주소를 가리킨다. 문맥 교환 후 프로세스 B가 실행되면 PC는 B의 코드 영역 안 주소를 가리킨다.

## 4. 가상 주소와 페이지 테이블도 함께 중요하다

현대 운영체제에서는 프로세스마다 독립적인 가상 주소 공간을 가진다. 그래서 프로세스 A와 B가 같은 숫자의 주소를 사용하더라도 실제 물리 메모리에서는 서로 다른 곳을 가리킬 수 있다.

문맥 교환 때는 레지스터 값뿐 아니라, 현재 프로세스의 주소 공간을 가리키는 페이지 테이블 관련 정보도 바뀔 수 있다.

```text
프로세스 A 실행
PC = A의 가상 주소
페이지 테이블 = A의 주소 변환 규칙

문맥 교환

프로세스 B 실행
PC = B의 가상 주소
페이지 테이블 = B의 주소 변환 규칙
```

그래서 CPU가 같은 방식으로 "PC가 가리키는 명령어를 가져와 실행"하더라도, 현재 선택된 프로세스와 페이지 테이블에 따라 실제로 가져오는 명령어가 달라진다.

## 5. 정리

문맥 교환을 정확히 표현하면 다음과 같다.

```text
프로세스 A의 CPU 상태를 메모리에 저장
프로세스 B의 CPU 상태를 CPU 레지스터에 복원
프로세스 B의 주소 공간으로 전환
PC가 가리키는 B의 명령어를 가져와 실행
```

따라서 다음처럼 이해하면 된다.

```text
레지스터 = CPU 코어 안의 실제 작업 공간
PCB/커널 스택 = 프로세스별로 저장된 CPU 상태가 놓이는 메모리 공간
명령어 = 각 프로세스의 코드 영역에 저장된 기계어
PC = 다음에 가져올 명령어의 주소
제어장치 = PC가 가리키는 명령어를 해석해 CPU 내부 동작을 조율하는 회로
```

핵심은 **프로세스마다 CPU 레지스터가 따로 있는 것이 아니라, 운영체제가 같은 물리 레지스터의 내용을 저장하고 복원하면서 다른 프로세스가 이어서 실행되는 것처럼 만들어 준다**는 점이다.

## 6. PCB는 하드웨어인가, 커널 자료구조인가

운영체제에서 말하는 PCB는 **Process Control Block**의 약자다. 이름에 `Block`이 들어가서 하드웨어 부품처럼 느껴질 수 있지만, PCB는 하드웨어가 아니라 **커널이 메모리 안에 만들어 두는 자료구조**다.

PCB는 프로세스를 관리하기 위한 정보 묶음이다. 운영체제는 실행 중인 프로세스마다 PCB를 하나씩 가지고 있고, 그 안에 프로세스 상태와 CPU 상태를 저장한다.

대표적으로 PCB에는 다음 정보가 들어갈 수 있다.

- 프로세스 ID
- 프로세스 상태: ready, running, blocked 등
- PC 값: 다음에 실행할 명령어 주소
- SP 값: 스택 위치
- 범용 레지스터 값들
- 메모리 관리 정보: 페이지 테이블 관련 정보
- 열린 파일 정보
- 스케줄링 정보: 우선순위, CPU 사용 시간 등

즉, PCB는 CPU 안에 있는 장치가 아니라 커널 메모리 안에 있는 기록표에 가깝다.

```text
커널 메모리
├─ 프로세스 A의 PCB
│  ├─ PID
│  ├─ 상태
│  ├─ 저장된 PC
│  ├─ 저장된 레지스터 값들
│  └─ 메모리/파일/스케줄링 정보
│
└─ 프로세스 B의 PCB
   ├─ PID
   ├─ 상태
   ├─ 저장된 PC
   ├─ 저장된 레지스터 값들
   └─ 메모리/파일/스케줄링 정보
```

문맥 교환 때는 CPU 레지스터의 현재 값들을 실행 중이던 프로세스의 PCB 또는 커널 스택에 저장하고, 다음에 실행할 프로세스의 PCB 또는 커널 스택에서 값을 복원한다.

```text
CPU 레지스터
   ↓ 저장
프로세스 A의 PCB/커널 스택

프로세스 B의 PCB/커널 스택
   ↓ 복원
CPU 레지스터
```

정리하면 다음과 같다.

```text
CPU 레지스터 = 실제 하드웨어
PCB = 커널이 메모리에 만든 프로세스 관리용 자료구조
PCB 안의 레지스터 값 = 나중에 다시 실행하기 위해 저장해 둔 CPU 상태
```

따라서 PCB는 하드웨어가 아니라 **운영체제 커널이 프로세스를 멈췄다가 다시 실행할 수 있도록 보관하는 관리 데이터**라고 이해하면 된다.

### PCB에 CPU 레지스터 값이 들어간다는 뜻

PCB에 CPU 레지스터 값이 들어간다는 말은 **PCB가 어느 CPU 레지스터 안에 들어간다**는 뜻이 아니다.

정확한 의미는 다음과 같다.

```text
CPU 레지스터에 있던 값들을
커널 메모리에 있는 PCB 자료구조 안에 저장한다
```

즉, 방향은 이렇게 이해해야 한다.

```text
CPU 레지스터 -> PCB에 저장
PCB -> CPU 레지스터로 복원
```

PCB 자체는 CPU 레지스터 안에 들어가는 것이 아니라, 커널 메모리 어딘가에 있는 자료구조다.

```text
커널 메모리
└─ 프로세스 A의 PCB
   ├─ 저장된 PC 값
   ├─ 저장된 SP 값
   ├─ 저장된 범용 레지스터 값들
   ├─ 저장된 상태 레지스터 값
   └─ 기타 프로세스 관리 정보
```

예를 들어 프로세스 A가 실행 중일 때 CPU 레지스터 상태가 다음과 같다고 하자.

```text
PC = 0x400120
SP = 0x7FFF0000
R1 = 10
R2 = 20
상태 레지스터 = Zero Flag 0, Carry Flag 0 ...
```

운영체제가 프로세스 A를 잠시 멈추고 프로세스 B를 실행하려면, A가 나중에 이어서 실행될 수 있도록 이 값들을 저장해야 한다.

```text
프로세스 A의 PCB
├─ saved_PC = 0x400120
├─ saved_SP = 0x7FFF0000
├─ saved_R1 = 10
├─ saved_R2 = 20
└─ saved_status = ...
```

그 다음 프로세스 B의 PCB에 저장되어 있던 값들을 CPU 레지스터에 다시 넣는다.

```text
프로세스 B의 PCB
├─ saved_PC = 0x500080
├─ saved_SP = 0x7FFE0000
├─ saved_R1 = 33
└─ saved_R2 = 44

        ↓ 복원

CPU 레지스터
├─ PC = 0x500080
├─ SP = 0x7FFE0000
├─ R1 = 33
└─ R2 = 44
```

이후 CPU는 B의 PC가 가리키는 명령어부터 실행한다.

정리하면 다음과 같다.

```text
CPU 레지스터
- 실제 CPU 안에 있는 하드웨어 저장 공간
- 현재 실행 중인 스레드/프로세스의 상태가 들어 있음

PCB
- 커널 메모리에 있는 자료구조
- 실행을 멈춘 프로세스의 CPU 상태를 저장해 둠

문맥 교환
- 현재 CPU 레지스터 값을 이전 프로세스 PCB에 저장
- 다음 프로세스 PCB의 저장 값을 CPU 레지스터에 복원
```

따라서 "PCB에 레지스터 값이 들어간다"는 말은 **CPU의 현재 상태를 나중에 복원하기 위해 PCB라는 메모리 자료구조에 기록해 둔다**는 뜻이다.

## 7. PC와 SP의 차이

PC와 SP는 둘 다 주소를 담는 레지스터지만, 가리키는 대상이 다르다.

```text
PC = Program Counter
SP = Stack Pointer
```

PC는 **다음에 실행할 명령어의 주소**를 가리킨다. CPU는 PC가 가리키는 메모리 위치에서 기계어 명령어를 가져와 실행한다. 명령어를 하나 실행하면 보통 PC는 다음 명령어 주소로 이동한다.

SP는 **현재 스택의 위치**를 가리킨다. 스택은 함수 호출, 지역 변수, 함수 인자, 반환 주소, 임시 저장 값 등을 관리하는 메모리 영역이다. 함수가 호출되거나 끝날 때, 지역 변수가 잡히거나 해제될 때 SP가 움직인다.

예를 들어 다음 코드가 있다고 하자.

```c
void foo() {
    int x = 10;
}

int main() {
    foo();
}
```

`main`에서 `foo()`를 호출하면 CPU는 `foo` 함수의 명령어로 이동해야 한다. 이때 실행 흐름을 바꾸는 데 중요한 것이 PC다. 동시에 `foo` 함수가 끝난 뒤 다시 `main`으로 돌아와야 하므로 반환 주소나 함수 실행에 필요한 정보가 스택에 저장되고, 이 스택 위치를 관리하는 것이 SP다.

단순화하면 다음과 같다.

```text
PC: 지금 어느 명령어를 실행할 차례인가?
SP: 지금 스택을 어디까지 사용하고 있는가?
```

메모리 안에서 보면 다음처럼 구분할 수 있다.

```text
프로세스 메모리
├─ 코드 영역
│  └─ PC가 이 안의 다음 명령어 주소를 가리킴
│
└─ 스택 영역
   └─ SP가 현재 스택의 꼭대기 위치를 가리킴
```

문맥 교환 때 PC와 SP를 모두 저장하는 이유도 이 때문이다. 프로세스를 다시 실행하려면 "다음에 어떤 명령어부터 실행해야 하는지"와 "그때 스택 상태가 어디였는지"를 모두 복원해야 한다.

```text
PC를 복원해야 실행 위치를 이어갈 수 있음
SP를 복원해야 함수 호출/지역 변수/반환 흐름을 이어갈 수 있음
```

정리하면 다음과 같다.

```text
PC = 코드 영역을 가리키는 실행 위치 레지스터
SP = 스택 영역을 가리키는 스택 위치 레지스터
```

둘 다 주소를 담지만, **PC는 명령어 실행 흐름**, **SP는 함수 호출과 스택 상태**를 관리한다.

## 8. 멀티코어 병렬처리는 공유를 안 해서 동기화가 필요 없는가

멀티코어 병렬처리에서 동기화가 필요한지는 **여러 코어가 같은 데이터를 공유해서 접근하는지**에 따라 달라진다.

각 코어가 완전히 독립된 데이터만 처리한다면 동기화가 거의 필요 없을 수 있다.

```text
코어 0 -> 배열 A의 0~999 처리
코어 1 -> 배열 A의 1000~1999 처리
코어 2 -> 배열 A의 2000~2999 처리
```

각 코어가 서로 다른 영역만 읽고 쓰고, 중간에 공유 변수도 수정하지 않는다면 충돌할 일이 적다. 이런 경우에는 병렬화가 비교적 쉽다.

하지만 여러 코어가 같은 변수를 동시에 읽고 쓰면 문제가 생긴다.

```c
counter++;
```

이 코드는 한 줄이지만 CPU 내부에서는 보통 여러 단계로 나뉜다.

```text
1. counter 값을 읽음
2. 1을 더함
3. counter에 다시 저장
```

코어 0과 코어 1이 동시에 `counter++`를 하면 다음 문제가 생길 수 있다.

```text
counter = 10

코어 0: counter 읽음 -> 10
코어 1: counter 읽음 -> 10
코어 0: 10 + 1 = 11 저장
코어 1: 10 + 1 = 11 저장

결과: 12가 되어야 하는데 11이 됨
```

이런 상황을 race condition이라고 한다. 공유 데이터를 동시에 수정할 때 실행 순서에 따라 결과가 달라지는 문제다.

그래서 공유 데이터가 있으면 락, 뮤텍스, 세마포어, 원자적 연산, 메시지 큐 같은 동기화 방법이 필요하다.

```text
공유하지 않는 데이터
-> 동기화 필요 적음

공유해서 읽기만 하는 데이터
-> 보통 동기화 부담 적음

공유해서 쓰는 데이터
-> 동기화 필요
```

멀티코어에서는 캐시도 고려해야 한다. 각 코어는 자기 캐시를 가질 수 있고, 같은 메모리 값을 여러 코어가 캐시에 복사해서 가지고 있을 수 있다.

```text
코어 0의 캐시: X = 10
코어 1의 캐시: X = 10
RAM: X = 10
```

코어 0이 `X = 20`으로 바꾸면, 코어 1의 캐시에 있던 `X = 10`은 더 이상 최신 값이 아니다. 이런 문제를 해결하기 위해 CPU는 캐시 일관성(cache coherence) 프로토콜을 사용한다.

하지만 캐시 일관성이 있다고 해서 프로그램 수준의 동기화가 필요 없어지는 것은 아니다. 캐시 일관성은 "각 코어가 같은 메모리 위치에 대해 완전히 엉뚱한 값을 계속 보지 않도록" 도와주는 하드웨어 규칙이고, race condition을 자동으로 해결해 주지는 않는다.

정리하면 다음과 같다.

```text
멀티코어 병렬처리 = 여러 코어가 동시에 실행
공유 데이터 없음 = 동기화 부담 작음
공유 데이터 읽기만 함 = 비교적 안전
공유 데이터 쓰기 있음 = 동기화 필요
캐시 일관성 = 하드웨어가 캐시 값의 일관성을 관리
동기화 = 프로그램이 공유 데이터 접근 순서를 안전하게 관리
```

따라서 멀티코어 병렬처리는 **공유를 안 하면 동기화가 필요 없을 수 있지만, 공유 데이터가 있으면 동기화가 필요하다**고 이해하면 된다.

## 9. 멀티코어, 스레드, 프로세스 병렬 처리의 차이

멀티코어, 스레드, 프로세스는 같은 기준으로 나란히 비교할 개념이 아니다. 멀티코어는 **하드웨어 구조**이고, 스레드와 프로세스는 운영체제가 관리하는 **실행 단위**다.

```text
멀티코어 = 실제로 동시에 실행할 수 있게 해주는 CPU 하드웨어
스레드 = 한 프로세스 안에서 나뉘는 실행 흐름
프로세스 = 실행 중인 프로그램의 독립된 실행 단위
```

### 멀티코어

멀티코어 CPU는 CPU 안에 코어가 여러 개 있는 구조다. 코어가 여러 개면 여러 명령 흐름을 실제로 동시에 실행할 수 있다.

```text
코어 0 -> 작업 A 실행
코어 1 -> 작업 B 실행
코어 2 -> 작업 C 실행
```

단, 코어가 4개인데 작업을 100개로 쪼갠다고 해서 100개가 모두 동시에 실행되는 것은 아니다. 운영체제가 스케줄링해서 여러 작업을 코어들에 나눠 번갈아 실행한다.

```text
코어 수 = 실제 동시 실행 능력의 물리적 기반
작업 수 = OS가 스케줄링할 실행 단위 수
```

### 스레드 병렬 처리

스레드는 한 프로세스 안의 실행 흐름이다. 같은 프로세스 안의 스레드들은 코드, 힙, 전역 변수 같은 메모리 공간을 공유한다.

```text
프로세스 A
├─ 스레드 1
├─ 스레드 2
└─ 스레드 3
```

스레드마다 따로 가지는 것도 있다.

```text
스레드별로 따로 가짐
- PC
- 레지스터 상태
- 스택

같은 프로세스 안에서 공유
- 코드 영역
- 힙
- 전역 변수
- 열린 파일 정보 등
```

스레드는 같은 메모리를 쉽게 공유하므로 작업을 나누기 좋다. 예를 들어 큰 배열을 여러 구간으로 나눠 여러 스레드가 동시에 계산할 수 있다.

하지만 공유 메모리를 쉽게 사용할 수 있다는 장점은 동시에 위험이기도 하다. 여러 스레드가 같은 변수를 동시에 수정하면 race condition이 생길 수 있으므로 동기화가 필요하다.

```text
스레드 장점
- 생성/전환 비용이 프로세스보다 비교적 작음
- 같은 메모리 공유가 쉬움
- 작업 분할이 쉬움

스레드 단점
- 공유 데이터 동기화 필요
- 한 스레드의 버그가 같은 프로세스 전체에 영향을 줄 수 있음
```

### 프로세스 병렬 처리

프로세스는 실행 중인 프로그램의 독립된 단위다. 각 프로세스는 기본적으로 별도의 주소 공간을 가진다.

```text
프로세스 A
├─ 코드
├─ 힙
└─ 스택

프로세스 B
├─ 코드
├─ 힙
└─ 스택
```

프로세스끼리는 기본적으로 메모리가 분리되어 있다. 그래서 한 프로세스가 잘못된 메모리 접근을 해도 다른 프로세스에 직접 영향을 덜 준다. 이 점 때문에 격리와 안정성이 좋다.

하지만 데이터를 공유하려면 스레드보다 번거롭다. 파이프, 소켓, 공유 메모리, 메시지 큐 같은 IPC 기법이 필요할 수 있다.

```text
프로세스 장점
- 격리 좋음
- 안정성 좋음
- 한 프로세스 장애가 다른 프로세스에 덜 전파됨

프로세스 단점
- 생성 비용이 더 큼
- 문맥 교환 비용이 더 클 수 있음
- 메모리 공유가 상대적으로 어려움
```

### 셋의 관계

실제 병렬 처리는 보통 다음 구조로 이해하면 된다.

```text
멀티코어 CPU
├─ 코어 0이 스레드/프로세스 실행
├─ 코어 1이 스레드/프로세스 실행
└─ 코어 2가 스레드/프로세스 실행
```

즉, 스레드든 프로세스든 실제로 동시에 실행되려면 코어가 필요하다. 코어보다 실행 단위가 많으면 운영체제가 시간을 쪼개서 번갈아 실행한다.

```text
코어 4개, 스레드 4개
-> 조건이 맞으면 4개가 실제 병렬 실행 가능

코어 4개, 스레드 100개
-> 100개가 동시에 실행되는 것이 아니라 OS가 번갈아 스케줄링
```

정리하면 다음과 같다.

```text
멀티코어
- 하드웨어
- 실제 동시 실행 능력 제공

스레드
- 프로세스 안의 실행 흐름
- 메모리 공유 쉬움
- 동기화 주의
- 가볍지만 격리는 약함

프로세스
- 독립된 실행 단위
- 메모리 격리 좋음
- 공유는 어렵고 비용이 큼
- 안정성은 좋음
```

한 줄로 정리하면, **멀티코어는 동시에 실행할 수 있는 물리적 기반이고, 스레드와 프로세스는 그 위에서 작업을 나눠 실행하기 위한 운영체제의 실행 단위**다.

## 10. 코어는 프로세스인가, 한 코어가 여러 프로세스를 동시에 실행할 수 있는가

코어와 프로세스는 다른 개념이다.

```text
코어 = CPU 안의 실제 실행 하드웨어
프로세스 = 실행 중인 프로그램을 OS가 관리하는 단위
```

코어는 명령어를 실제로 실행하는 하드웨어이고, 프로세스는 운영체제가 관리하는 실행 대상이다.

```text
CPU
├─ 코어 0
├─ 코어 1
└─ 코어 2

OS
├─ 프로세스 A
├─ 프로세스 B
└─ 프로세스 C
```

한 코어는 한 순간에 보통 하나의 실행 흐름을 실행한다. 여기서 실행 흐름은 정확히는 프로세스라기보다 스레드다. 프로세스는 하나 이상의 스레드를 가질 수 있고, 실제로 CPU에 올라가 실행되는 것은 스레드다.

```text
한 순간:
코어 0 -> 스레드 A 실행
코어 1 -> 스레드 B 실행
```

그런데 한 코어에서도 여러 프로세스가 동시에 도는 것처럼 보일 수 있다. 이유는 운영체제가 아주 빠르게 문맥 교환을 하기 때문이다.

```text
시간 0~5ms: 코어 0이 프로세스 A 실행
시간 5~10ms: 코어 0이 프로세스 B 실행
시간 10~15ms: 코어 0이 프로세스 C 실행
시간 15~20ms: 다시 프로세스 A 실행
```

사람 입장에서는 너무 빠르게 바뀌므로 여러 프로세스가 동시에 실행되는 것처럼 보인다. 이것은 동시성(concurrency)이다.

반면 실제 병렬성(parallelism)은 여러 코어에서 여러 실행 흐름이 정말 같은 시간에 실행되는 것이다.

```text
실제 병렬 실행:
코어 0 -> 프로세스 A의 스레드 실행
코어 1 -> 프로세스 B의 스레드 실행
코어 2 -> 프로세스 C의 스레드 실행
```

정리하면 다음과 같다.

```text
코어 = 하드웨어
프로세스 = OS가 관리하는 실행 중인 프로그램
스레드 = 실제로 CPU에 스케줄링되는 실행 흐름

한 코어
- 한 순간에는 보통 하나의 스레드 실행
- 여러 프로세스를 아주 빠르게 번갈아 실행 가능

여러 코어
- 여러 스레드를 실제로 동시에 실행 가능
```

추가로, 하이퍼스레딩이나 SMT 같은 기술이 있으면 하나의 물리 코어가 여러 하드웨어 스레드를 동시에 처리하는 것처럼 동작할 수 있다. 하지만 이 경우도 코어와 프로세스가 같은 뜻이 되는 것은 아니다. 물리 코어 하나 안에 여러 논리 코어가 보이는 구조라고 이해하면 된다.

## 11. 프로세스가 자신만의 주소 공간을 가진다는 뜻

프로세스가 자신만의 주소 공간을 가진다는 말은, 각 프로세스가 **자기만의 메모리 지도**를 가진다는 뜻이다.

예를 들어 프로세스 A와 프로세스 B가 둘 다 `0x1000`이라는 주소를 사용한다고 해보자. 주소 숫자는 같아도, 실제 물리 메모리에서는 서로 다른 위치를 가리킬 수 있다.

```text
프로세스 A의 0x1000 -> 물리 메모리의 어떤 위치 A
프로세스 B의 0x1000 -> 물리 메모리의 다른 위치 B
```

이것이 가능한 이유는 현대 운영체제가 가상 메모리를 사용하기 때문이다. 프로그램이 보는 주소는 보통 물리 주소가 아니라 가상 주소다. CPU와 운영체제는 페이지 테이블을 이용해서 가상 주소를 실제 물리 주소로 변환한다.

```text
프로세스가 보는 주소 = 가상 주소
RAM의 실제 위치 = 물리 주소
페이지 테이블 = 가상 주소를 물리 주소로 바꾸는 표
```

프로세스마다 주소 공간이 분리되어 있으므로, 프로세스 A가 자기 메모리의 `0x1000`을 수정해도 프로세스 B의 `0x1000`이 같이 바뀌지 않는다.

```text
프로세스 A
├─ 코드 영역
├─ 데이터 영역
├─ 힙
└─ 스택

프로세스 B
├─ 코드 영역
├─ 데이터 영역
├─ 힙
└─ 스택
```

겉으로는 둘 다 비슷한 구조를 가지지만, 서로 독립된 주소 공간이다.

반면 같은 프로세스 안의 스레드들은 주소 공간을 공유한다.

```text
프로세스 A
├─ 코드 영역        공유
├─ 힙              공유
├─ 전역 변수        공유
├─ 스레드 1의 스택  따로
└─ 스레드 2의 스택  따로
```

그래서 같은 프로세스 안의 스레드들은 힙이나 전역 변수에 있는 데이터를 쉽게 공유할 수 있다. 하지만 이 때문에 동시에 같은 데이터를 수정하면 동기화 문제가 생길 수 있다.

정리하면 다음과 같다.

```text
프로세스
- 자신만의 주소 공간을 가짐
- 다른 프로세스 메모리에 마음대로 접근할 수 없음
- 격리와 안정성이 좋음

스레드
- 같은 프로세스의 주소 공간을 공유함
- 힙, 전역 변수 등을 같이 사용하기 쉬움
- 공유 데이터 동기화가 필요할 수 있음
```

한 줄로 정리하면, **자신만의 주소 공간**이란 프로세스마다 독립된 가상 메모리 공간을 가지고, 같은 주소 숫자를 쓰더라도 다른 실제 메모리를 가리킬 수 있다는 뜻이다.

## 12. PCB와 TCB는 문맥 교환 때 CPU 상태를 저장하는 곳인가

맞다. PCB와 TCB는 문맥 교환 때 **CPU에서 실행 중이던 상태를 저장하고 복원하기 위해 쓰이는 커널 자료구조**다.

다만 운영체제마다 구조 이름과 나누는 방식은 다를 수 있다. 개념적으로는 다음처럼 보면 된다.

```text
PCB = 프로세스 전체를 관리하는 정보
TCB = 스레드 하나의 실행 상태를 관리하는 정보
```

프로세스는 하나 이상의 스레드를 가질 수 있다. 실제로 CPU에 스케줄링되어 실행되는 단위는 보통 스레드다. 그래서 문맥 교환에서 특히 중요한 것은 스레드의 실행 상태다.

TCB에는 다음과 같은 정보가 저장될 수 있다.

```text
PC
SP
범용 레지스터 값들
상태 레지스터 값
스레드 상태
스레드 우선순위
커널 스택 정보
```

PCB에는 프로세스 전체에 대한 정보가 저장될 수 있다.

```text
프로세스 ID
주소 공간 정보
페이지 테이블 정보
열린 파일 정보
프로세스 상태
프로세스 안의 스레드 목록
```

문맥 교환 흐름은 단순화하면 다음과 같다.

```text
현재 실행 중인 스레드 A의 CPU 레지스터 값 저장
        ↓
A의 TCB/PCB 관련 구조에 기록
        ↓
다음에 실행할 스레드 B의 저장된 값 로드
        ↓
CPU 레지스터에 복원
        ↓
B가 멈췄던 지점부터 실행
```

중요한 점은 PCB나 TCB가 CPU 안에 들어 있는 것이 아니라는 것이다. 이들은 **커널 메모리에 있는 자료구조**다.

```text
CPU 레지스터
-> 현재 실행 중인 상태가 들어 있는 하드웨어

PCB/TCB
-> 멈춰 둔 실행 상태를 저장하는 커널 메모리의 자료구조
```

한 줄로 정리하면, **PCB/TCB는 CPU가 작업하던 내용 중 나중에 이어서 실행하는 데 필요한 상태를 커널 메모리에 저장해 둔 기록**이라고 보면 된다.

## 13. 실행 중인 작업이 I/O를 기다린다는 것을 어떻게 아는가

실행 중인 작업이 I/O를 기다린다는 것은 보통 **그 작업이 커널에 I/O 요청을 했기 때문에 커널이 알게 된다**.

예를 들어 프로세스가 파일을 읽는다고 하자.

```c
read(fd, buffer, size);
```

이 함수 호출은 내부적으로 시스템 콜을 통해 커널에 들어간다.

```text
사용자 프로그램
        ↓ read 시스템 콜
커널 진입
        ↓
커널이 파일/디스크 상태 확인
```

커널이 확인했을 때 데이터가 아직 준비되지 않았다면, 현재 실행 중인 스레드는 계속 CPU를 붙잡고 있어도 할 일이 없다. 그러면 커널은 이 스레드의 상태를 `blocked` 또는 `waiting`으로 바꾸고, I/O 완료를 기다리는 큐에 넣는다.

```text
스레드 A가 read 요청
        ↓
데이터가 아직 없음
        ↓
커널이 A 상태를 waiting으로 변경
        ↓
A를 I/O 대기 큐에 넣음
        ↓
다른 실행 가능한 스레드 B를 선택
        ↓
문맥 교환
```

나중에 디스크나 네트워크 장치가 작업을 끝내면 인터럽트를 발생시킨다.

```text
디스크 작업 완료
        ↓
인터럽트 발생
        ↓
커널의 인터럽트 핸들러 실행
        ↓
대기 중이던 스레드 A를 ready 상태로 변경
```

즉, 커널은 다음 두 순간에 I/O 대기 상태를 안다.

```text
1. 스레드가 시스템 콜로 I/O를 요청했을 때
2. 장치가 인터럽트로 I/O 완료를 알렸을 때
```

정리하면, 커널이 실행 중인 작업의 마음을 읽는 것이 아니라 **I/O 요청과 장치 완료 알림을 커널이 관리하기 때문에** 어떤 스레드가 기다려야 하는지 알 수 있다.

## 14. 커널 모드 전환 비용과 문맥 교환 비용은 왜 드는가

커널 모드 전환과 문맥 교환은 관련이 있지만 같은 말은 아니다.

```text
커널 모드 전환
- 사용자 모드에서 커널 모드로 CPU 권한 수준을 바꾸는 것
- 예: 시스템 콜, 인터럽트, 예외

문맥 교환
- 실행 중인 스레드/프로세스를 다른 스레드/프로세스로 바꾸는 것
- CPU 레지스터 상태 저장/복원, 주소 공간 전환 등이 필요
```

시스템 콜이 발생하면 먼저 커널 모드로 전환된다.

```text
사용자 프로그램 실행
        ↓ 시스템 콜
커널 모드로 전환
        ↓
커널 코드 실행
```

이때 비용이 드는 이유는 CPU가 단순히 함수 호출처럼 이동하는 것이 아니라, 안전하게 권한을 바꾸고 커널 진입 지점으로 이동해야 하기 때문이다.

커널 모드 전환 시에는 대략 다음 일이 필요하다.

```text
현재 사용자 실행 위치 저장
CPU 권한 모드 변경
커널 스택으로 전환
커널의 정해진 진입 코드 실행
필요한 레지스터 일부 저장
보안/권한 검사
```

그래서 커널 모드 전환 자체에도 비용이 있다.

하지만 커널 모드로 들어갔다고 항상 문맥 교환이 일어나는 것은 아니다.

예를 들어 아주 짧은 시스템 콜은 다음처럼 끝날 수 있다.

```text
프로세스 A 실행
        ↓ 시스템 콜
커널 모드 진입
        ↓
커널이 요청 처리
        ↓
다시 프로세스 A로 복귀
```

이 경우 커널 모드 전환은 있었지만, 다른 프로세스로 바뀌지 않았으므로 큰 의미의 문맥 교환은 없다.

반면 I/O 대기가 필요하면 문맥 교환이 일어날 수 있다.

```text
프로세스 A 실행
        ↓ read 시스템 콜
커널 모드 진입
        ↓
데이터가 아직 없음
        ↓
A를 waiting 상태로 변경
        ↓
B를 running 상태로 선택
        ↓
A의 CPU 상태 저장, B의 CPU 상태 복원
        ↓
프로세스 B 실행
```

문맥 교환 비용은 커널 모드 전환보다 더 클 수 있다. 이유는 다음 작업들이 추가되기 때문이다.

```text
현재 스레드의 레지스터 값 저장
다음 스레드의 레지스터 값 복원
PC, SP 복원
주소 공간 전환
페이지 테이블 관련 레지스터 변경
TLB 영향
캐시 지역성 손실
스케줄러 실행 비용
```

중요한 점은 문맥 교환이 "커널보다 밑 단계에서 자동으로 나는 것"이 아니라는 것이다. CPU 하드웨어가 시스템 콜, 인터럽트, 예외를 통해 커널 코드로 진입할 수 있게 도와주고, 실제로 어떤 스레드를 멈추고 어떤 스레드를 실행할지는 **커널 스케줄러가 결정**한다.

```text
CPU 하드웨어
- 현재 명령 실행
- 시스템 콜/인터럽트/예외 발생 시 커널 진입 지원
- 레지스터 저장/복원에 필요한 명령 제공

커널
- 현재 작업을 계속 실행할지, 대기시킬지 판단
- 다음에 실행할 작업 선택
- PCB/TCB에 상태 저장
- 다음 작업 상태를 CPU에 복원
```

따라서 문맥 교환은 CPU가 하드웨어적으로 지원하지만, **운영체제 커널이 수행하는 작업**이라고 이해하는 것이 정확하다.

짧게 정리하면 다음과 같다.

```text
I/O 대기 판단
- 프로세스가 시스템 콜로 I/O를 요청했기 때문에 커널이 알 수 있음
- 완료는 장치 인터럽트로 알림

커널 모드 전환 비용
- 권한 모드 변경, 커널 스택 전환, 진입 처리 등이 필요해서 발생

문맥 교환 비용
- 레지스터 저장/복원, 주소 공간 전환, 스케줄링, 캐시/TLB 영향 때문에 발생

누가 문맥 교환을 수행하는가
- CPU 하드웨어가 지원함
- 실제 결정과 절차는 커널 스케줄러가 수행함
```

## 15. Convoy effect란 무엇인가

Convoy effect는 운영체제 스케줄링에서 **오래 걸리는 작업 하나 때문에 짧은 작업들이 줄줄이 기다리게 되는 현상**이다.

`convoy`는 차량 행렬이라는 뜻이다. 느린 차량 하나가 앞에 있으면 뒤의 차들이 줄줄이 느려지는 것처럼, CPU를 오래 쓰는 프로세스 하나 때문에 다른 짧은 프로세스들이 대기하는 상황을 말한다.

대표적으로 FCFS(First-Come, First-Served) 스케줄링에서 잘 나타난다. FCFS는 먼저 온 작업을 먼저 실행한다.

예를 들어 다음 작업들이 순서대로 도착했다고 하자.

```text
프로세스 A: CPU 100ms 필요
프로세스 B: CPU 3ms 필요
프로세스 C: CPU 2ms 필요
프로세스 D: CPU 4ms 필요
```

FCFS에서는 A가 먼저 왔으므로 A가 끝날 때까지 B, C, D는 기다린다.

```text
시간 흐름:
A 실행 100ms
        ↓
B 실행 3ms
        ↓
C 실행 2ms
        ↓
D 실행 4ms
```

문제는 B, C, D는 각각 아주 짧은 작업인데도 A 때문에 오래 기다린다는 점이다.

```text
B는 3ms만 필요하지만 100ms 가까이 기다림
C는 2ms만 필요하지만 더 오래 기다림
D도 마찬가지
```

이렇게 긴 작업 뒤에 짧은 작업들이 줄줄이 묶이는 현상이 convoy effect다.

I/O 작업과 연결해서도 문제가 생길 수 있다. CPU를 오래 쓰는 작업이 CPU를 점유하는 동안, I/O 중심 작업들이 CPU를 조금만 쓰고 다시 I/O를 보내야 하는데 대기하게 된다. 그러면 CPU와 I/O 장치가 번갈아 놀 수 있다.

단순화하면 다음과 같다.

```text
긴 CPU 작업이 CPU 점유
        ↓
짧은 I/O 작업들이 CPU를 못 얻음
        ↓
I/O 요청도 늦게 발생
        ↓
I/O 장치 활용도 떨어짐
        ↓
전체 시스템 처리량 저하
```

Convoy effect를 줄이기 위해 운영체제는 보통 다음과 같은 스케줄링 방식을 사용한다.

```text
Round Robin
- 일정 시간마다 강제로 CPU를 넘김
- 긴 작업 하나가 계속 CPU를 독점하지 못하게 함

SJF / SRTF
- 짧은 작업을 우선 실행
- 평균 대기 시간을 줄일 수 있음

우선순위 기반 스케줄링
- I/O 중심 작업이나 대화형 작업에 더 빨리 CPU를 줄 수 있음
```

정리하면 다음과 같다.

```text
Convoy effect
- 긴 작업 하나 때문에 짧은 작업들이 줄줄이 기다리는 현상
- FCFS 같은 단순 비선점 스케줄링에서 잘 발생
- 평균 대기 시간과 시스템 응답성을 나쁘게 만들 수 있음
```

## 16. 스케줄링에서 aging이란 무엇인가

Aging은 스케줄링에서 오래 기다린 프로세스나 스레드의 **우선순위를 점점 높여 주는 기법**이다. 목적은 starvation, 즉 어떤 작업이 계속 선택받지 못하고 굶는 상황을 막는 것이다.

우선순위 스케줄링에서는 우선순위가 높은 작업을 먼저 실행한다.

```text
우선순위 높음 -> 먼저 실행
우선순위 낮음 -> 나중에 실행
```

그런데 높은 우선순위 작업이 계속 들어오면 낮은 우선순위 작업은 계속 밀릴 수 있다.

```text
낮은 우선순위 작업 A 대기 중
        ↓
높은 우선순위 작업 계속 도착
        ↓
A가 계속 실행되지 못함
```

이 문제를 막기 위해 aging을 적용한다. 오래 기다릴수록 우선순위를 조금씩 올려 준다.

```text
처음 우선순위: 낮음
10초 대기 후: 조금 상승
20초 대기 후: 더 상승
30초 대기 후: 높은 우선순위가 되어 실행 가능
```

즉, aging은 "가치"를 부여한다기보다는 **대기 시간에 따라 우선순위 점수를 보정하는 방식**이라고 보면 된다.

예를 들어 우선순위 숫자가 작을수록 높은 우선순위라고 하자.

```text
프로세스 A 초기 priority = 10
5초 기다릴 때마다 priority를 1씩 감소
```

그러면 다음처럼 변한다.

```text
대기 0초  -> priority 10
대기 5초  -> priority 9
대기 10초 -> priority 8
대기 15초 -> priority 7
```

반대로 어떤 시스템은 숫자가 클수록 우선순위가 높을 수도 있다. 이 경우에는 기다릴수록 점수를 더한다.

```text
대기 0초  -> priority 10
대기 5초  -> priority 11
대기 10초 -> priority 12
```

핵심은 숫자의 방향이 아니라, **오래 기다릴수록 스케줄러가 더 잘 선택하게 만든다**는 것이다.

정리하면 다음과 같다.

```text
Aging
- 오래 기다린 작업의 우선순위를 점점 올림
- starvation 방지 목적
- 우선순위 스케줄링에서 자주 설명됨
- 실제 구현은 OS마다 다를 수 있음
```

한 줄로 정리하면, aging은 **기다린 시간만큼 우선순위에 가산점을 주는 방식**이다.

## 17. 멀티레벨 큐 스케줄링이란 무엇인가

멀티레벨 큐(Multilevel Queue) 스케줄링은 ready queue를 하나만 두지 않고, **작업의 성격에 따라 여러 큐로 나누어 관리하는 방식**이다.

예를 들어 시스템에는 여러 종류의 작업이 있을 수 있다.

```text
대화형 작업
- 사용자 입력에 빠르게 반응해야 함
- 예: 터미널, 에디터, GUI 프로그램

배치 작업
- 오래 걸려도 됨
- 예: 대용량 계산, 백그라운드 처리

시스템 작업
- 운영체제 내부 작업
- 높은 우선순위가 필요할 수 있음
```

이런 작업들을 하나의 큐에 모두 넣으면 성격이 다른 작업을 같은 기준으로 처리해야 한다. 멀티레벨 큐는 이를 나누어 처리한다.

```text
Ready Queues
├─ Queue 0: 시스템 작업
├─ Queue 1: 대화형 작업
├─ Queue 2: 배치 작업
└─ Queue 3: 백그라운드 작업
```

각 큐는 서로 다른 스케줄링 정책을 가질 수 있다.

```text
대화형 작업 큐
- Round Robin
- 응답성 중요

배치 작업 큐
- FCFS
- 처리량 중요
```

큐 사이에도 우선순위가 있을 수 있다.

```text
Queue 0이 비어 있지 않으면 Queue 0 실행
Queue 0이 비면 Queue 1 실행
Queue 1도 비면 Queue 2 실행
```

단순화하면 다음과 같다.

```text
높은 우선순위 큐
├─ 시스템 작업
↓
중간 우선순위 큐
├─ 대화형 작업
↓
낮은 우선순위 큐
└─ 배치 작업
```

멀티레벨 큐는 **작업 종류가 명확히 다르고, 각 종류에 맞는 스케줄링 정책을 따로 적용하고 싶을 때** 사용한다.

예를 들어 대화형 작업은 사용자가 느리다고 체감하지 않도록 빨리 CPU를 받아야 한다. 반면 배치 작업은 응답성보다 전체 처리량이 중요할 수 있다.

```text
대화형 작업
-> 짧게 자주 CPU를 줌

배치 작업
-> 오래 기다려도 되지만 꾸준히 처리
```

주의할 점도 있다. 큐 사이 우선순위가 고정되어 있으면 낮은 큐의 작업이 계속 실행되지 못하는 starvation이 생길 수 있다.

```text
높은 우선순위 큐에 작업이 계속 들어옴
        ↓
낮은 우선순위 큐는 계속 대기
```

이를 줄이기 위해 큐마다 CPU 시간을 일정 비율로 배분할 수 있다.

```text
시스템/대화형 큐: CPU 80%
배치 큐: CPU 20%
```

또는 aging이나 멀티레벨 피드백 큐 같은 방식으로 보완할 수 있다.

정리하면 다음과 같다.

```text
멀티레벨 큐
- ready queue를 작업 종류별로 여러 개 둠
- 큐마다 다른 스케줄링 정책 가능
- 대화형/배치/시스템 작업처럼 성격이 다른 작업을 분리하기 좋음
- 큐 사이 우선순위 때문에 starvation이 생길 수 있음
```

비슷한 이름의 멀티레벨 피드백 큐와 차이도 중요하다.

```text
멀티레벨 큐
- 프로세스가 어느 큐에 속하는지 보통 고정

멀티레벨 피드백 큐
- 실행 패턴에 따라 프로세스가 큐 사이를 이동할 수 있음
```

## 18. 실시간 시스템은 왜 스케줄링에서 언급되는가

실시간 시스템(real-time system)은 작업을 단순히 빠르게 처리하는 것보다 **정해진 시간 안에 반드시 처리하는 것**이 중요한 시스템이다.

일반적인 스케줄링에서는 평균 대기 시간, 응답 시간, 처리량, 공정성 같은 지표가 중요하다.

```text
일반 시스템
- 평균 대기 시간 줄이기
- 응답성 높이기
- 처리량 높이기
- 공정하게 CPU 나누기
```

하지만 실시간 시스템에서는 deadline이 중요하다.

```text
실시간 시스템
- 이 작업은 반드시 10ms 안에 끝나야 함
- 이 센서 데이터는 5ms 안에 처리해야 함
- 이 제어 신호는 늦으면 안 됨
```

그래서 실시간 시스템은 스케줄링에서 따로 언급된다. 어떤 작업을 먼저 실행할지 결정할 때, 단순히 평균 성능이 아니라 **마감 시간을 지킬 수 있는지**를 고려해야 하기 때문이다.

예를 들어 자동차 에어백 제어 시스템을 생각해보자.

```text
충돌 감지
        ↓
정해진 아주 짧은 시간 안에 에어백 전개 판단
        ↓
늦으면 실패
```

이런 시스템에서는 "조금 늦었지만 평균적으로는 빨랐다"가 의미가 없다. 정해진 시간 안에 처리하지 못하면 시스템 실패다.

실시간 시스템은 보통 두 종류로 나눈다.

```text
Hard real-time
- deadline을 놓치면 치명적 실패
- 예: 항공 제어, 의료 장비, 자동차 안전 시스템

Soft real-time
- deadline을 놓치면 품질 저하
- 예: 동영상 재생, 음성 통화, 게임 프레임 처리
```

동영상 재생을 예로 들면, 프레임을 제때 처리하지 못하면 화면이 끊긴다. 치명적인 사고는 아니지만 품질이 나빠진다. 이런 경우는 soft real-time에 가깝다.

스케줄링 관점에서는 다음 질문이 중요해진다.

```text
어떤 작업의 deadline이 더 가까운가?
어떤 작업이 주기적으로 실행되어야 하는가?
최악의 경우에도 시간 안에 끝낼 수 있는가?
우선순위를 어떻게 줘야 deadline을 지킬 수 있는가?
```

그래서 실시간 스케줄링에서는 EDF(Earliest Deadline First), Rate Monotonic Scheduling 같은 정책이 언급된다.

```text
EDF
- deadline이 가장 가까운 작업을 먼저 실행

Rate Monotonic
- 주기가 짧은 작업에 더 높은 우선순위 부여
```

정리하면 다음과 같다.

```text
실시간 시스템이 스케줄링에서 중요한 이유
- CPU를 누구에게 먼저 줄지가 deadline 충족 여부를 결정하기 때문
- 단순히 빠른 평균 성능보다 시간 보장이 중요하기 때문
- 작업 우선순위를 deadline과 주기에 따라 정해야 하기 때문
```

한 줄로 정리하면, **실시간 시스템은 작업의 마감 시간을 지키는 것이 핵심이라 스케줄링 정책이 시스템의 성공 여부를 직접 좌우한다**.

## 19. 임계 구역은 race condition을 예방하기 위한 방법인가

임계 구역(critical section)은 race condition을 막기 위해 **동기화로 보호해야 하는 코드 구간**을 말한다. 즉, 임계 구역 자체가 방법이라기보다는 "여기를 동시에 실행하면 위험하니 보호해야 한다"고 표시하는 개념에 가깝다.

race condition은 여러 스레드나 프로세스가 공유 데이터를 동시에 접근하고, 실행 순서에 따라 결과가 달라지는 문제다.

예를 들어 다음 코드가 있다고 하자.

```c
counter++;
```

이 코드는 한 줄이지만 실제로는 다음 단계로 나뉠 수 있다.

```text
1. counter 읽기
2. 1 더하기
3. counter에 다시 쓰기
```

두 스레드가 동시에 실행하면 문제가 생길 수 있다.

```text
counter = 10

스레드 A: counter 읽음 -> 10
스레드 B: counter 읽음 -> 10
스레드 A: 11 저장
스레드 B: 11 저장

결과: 12가 되어야 하는데 11이 됨
```

이때 공유 변수 `counter`를 읽고 수정하고 다시 쓰는 코드 구간이 임계 구역이다.

```c
// 임계 구역 시작
counter++;
// 임계 구역 끝
```

임계 구역을 안전하게 만들려면 한 번에 하나의 실행 흐름만 들어가게 해야 한다. 이를 위해 mutex, lock, semaphore, atomic operation 같은 동기화 기법을 사용한다.

```c
lock(mutex);

counter++;

unlock(mutex);
```

이렇게 하면 스레드 A가 임계 구역에 들어간 동안 스레드 B는 기다린다.

```text
스레드 A가 lock 획득
        ↓
A가 counter 수정
        ↓
A가 unlock
        ↓
스레드 B가 lock 획득
        ↓
B가 counter 수정
```

정리하면 다음과 같다.

```text
race condition
- 공유 데이터 접근 순서에 따라 결과가 달라지는 문제

임계 구역
- 공유 데이터를 건드리는 위험한 코드 구간
- 동시에 여러 실행 흐름이 들어가면 안 됨

동기화 기법
- mutex, lock, semaphore, atomic 등
- 임계 구역을 한 번에 하나만 실행하게 만드는 방법
```

따라서 **임계 구역은 race condition을 예방하기 위해 보호해야 하는 구간이고, 실제 예방 방법은 그 구간에 lock 같은 동기화를 적용하는 것**이다.

## 20. 세마포어는 임계 구역 보호에 못 쓰는가

세마포어도 임계 구역 보호에 쓸 수 있다. 다만 **세마포어의 초기값을 몇으로 두느냐**에 따라 의미가 달라진다.

뮤텍스는 보통 한 번에 하나의 실행 흐름만 임계 구역에 들어가게 한다.

```text
mutex
- 허용 개수 1
- 한 스레드만 임계 구역 진입
```

세마포어는 내부 카운트를 가지고 있고, 그 카운트만큼 동시에 통과를 허용한다.

```text
semaphore(3)
- 최대 3개 실행 흐름이 동시에 통과 가능
```

따라서 공유 변수 `counter++`처럼 한 번에 하나만 실행해야 안전한 임계 구역에 `semaphore(3)`을 쓰면 race condition이 생길 수 있다.

예를 들어 다음 코드가 있다고 하자.

```c
wait(sem);   // sem 초기값 3

counter++;

signal(sem);
```

세마포어 값이 3이면 스레드 3개가 동시에 `counter++` 구간에 들어올 수 있다.

```text
스레드 A 진입
스레드 B 진입
스레드 C 진입
```

이 경우 `counter++`는 안전하지 않다.

```text
counter = 10

A: 10 읽음
B: 10 읽음
A: 11 저장
B: 11 저장
```

결과가 꼬일 수 있으므로 race condition이 발생할 수 있다.

하지만 세마포어 초기값을 1로 두면 한 번에 하나만 들어가게 할 수 있다. 이것을 binary semaphore라고 한다.

```c
semaphore sem = 1;

wait(sem);

counter++;

signal(sem);
```

이 경우 동작은 뮤텍스와 비슷하다.

```text
sem = 1
스레드 A 진입 -> sem = 0
스레드 B 대기
A가 signal -> sem = 1
B 진입
```

정리하면 다음과 같다.

```text
mutex
- 임계 구역에 한 번에 하나만 들어가게 하는 용도
- 소유권 개념이 있는 경우가 많음

binary semaphore
- 카운트가 1인 세마포어
- 한 번에 하나만 통과
- 임계 구역 보호에 사용 가능

counting semaphore
- 카운트가 N인 세마포어
- 최대 N개까지 통과
- 제한된 자원 N개를 관리할 때 적합
```

counting semaphore는 보통 "동시에 N개까지 써도 되는 자원"에 사용한다.

예를 들어 DB 연결이 3개뿐이라면 다음처럼 생각할 수 있다.

```text
DB connection pool 크기 = 3
semaphore(3)

최대 3개 스레드만 DB 연결 사용 가능
나머지는 대기
```

이 경우에는 3개가 동시에 들어와도 괜찮다. 왜냐하면 자원이 3개 있고, 각 스레드가 서로 다른 연결을 사용할 수 있기 때문이다.

하지만 `counter++`처럼 하나의 공유 값을 동시에 수정하면 안 되는 코드에는 `semaphore(1)`이나 mutex를 써야 한다.

한 줄로 정리하면 다음과 같다.

```text
세마포어는 임계 구역 보호에 쓸 수 있지만,
하나만 들어와야 하는 임계 구역에는 semaphore(1) 또는 mutex를 써야 한다.
semaphore(N)은 N개가 동시에 들어와도 안전한 자원 제한에 쓰는 것이 맞다.
```

## 21. 뮤텍스에서 락 관리를 잘못하면 왜 데드락이 되는가

뮤텍스에서 데드락(deadlock)은 여러 스레드가 서로 상대가 가진 락을 기다리면서 **아무도 더 이상 진행하지 못하는 상태**다.

가장 흔한 예시는 락을 여러 개 사용할 때 순서가 꼬이는 경우다.

예를 들어 락 A와 락 B가 있다고 하자.

```text
락 A
락 B
```

스레드 1은 A를 잡고 B를 기다린다.

```c
lock(A);
lock(B);

// 작업

unlock(B);
unlock(A);
```

스레드 2는 반대로 B를 잡고 A를 기다린다.

```c
lock(B);
lock(A);

// 작업

unlock(A);
unlock(B);
```

동시에 실행되면 다음 상황이 생길 수 있다.

```text
스레드 1: lock(A) 성공
스레드 2: lock(B) 성공

스레드 1: lock(B) 시도 -> B는 스레드 2가 가지고 있음 -> 대기
스레드 2: lock(A) 시도 -> A는 스레드 1이 가지고 있음 -> 대기
```

이제 둘 다 상대가 가진 락을 기다린다.

```text
스레드 1은 B를 기다림
스레드 2는 A를 기다림

하지만 A는 스레드 1이 놓아야 하고,
B는 스레드 2가 놓아야 함
```

문제는 둘 다 기다리는 상태라 unlock까지 도달하지 못한다는 것이다. 그래서 영원히 멈춘다.

```text
스레드 1: A를 들고 B를 기다림
스레드 2: B를 들고 A를 기다림
        ↓
아무도 진행 못 함
        ↓
데드락
```

또 다른 경우는 락 해제를 빼먹는 것이다.

```c
lock(mutex);

if (error) {
    return; // unlock 없이 빠져나감
}

unlock(mutex);
```

이렇게 되면 mutex가 계속 잠긴 상태로 남고, 이후 다른 스레드들은 영원히 기다릴 수 있다.

데드락이 생기려면 보통 다음 조건들이 함께 만족된다. 이를 데드락의 4가지 필요 조건이라고 한다.

```text
상호 배제
- 자원을 한 번에 하나만 사용할 수 있음

점유와 대기
- 어떤 자원을 가진 채 다른 자원을 기다림

비선점
- 다른 스레드가 가진 자원을 강제로 빼앗을 수 없음

순환 대기
- A는 B를 기다리고, B는 A를 기다리는 식의 고리가 생김
```

뮤텍스는 기본적으로 상호 배제 자원이므로, 여러 락을 잘못된 순서로 잡으면 나머지 조건들이 쉽게 만들어질 수 있다.

데드락을 줄이는 방법은 다음과 같다.

```text
락 획득 순서를 항상 통일하기
- 모든 스레드가 A -> B 순서로만 lock

락을 잡은 시간을 짧게 유지하기
- 임계 구역 최소화

락을 잡은 상태에서 오래 걸리는 I/O 하지 않기
- 다른 스레드를 불필요하게 막음

try-lock이나 timeout 사용하기
- 영원히 기다리지 않게 함

예외나 return 경로에서도 반드시 unlock하기
- finally, defer, RAII 같은 패턴 사용
```

가장 중요한 원칙은 락 순서를 통일하는 것이다.

나쁜 예:

```text
스레드 1: A -> B
스레드 2: B -> A
```

좋은 예:

```text
스레드 1: A -> B
스레드 2: A -> B
```

정리하면 다음과 같다.

```text
뮤텍스 데드락
- 스레드들이 서로 가진 락을 기다리며 멈추는 상태

주요 원인
- 락 획득 순서 불일치
- unlock 누락
- 락을 잡은 채 다른 자원을 기다림

예방
- 락 순서 통일
- 임계 구역 최소화
- 반드시 unlock 보장
- timeout/try-lock 고려
```

## 22. unlock 누락만으로 항상 데드락이 되는가

unlock을 빼먹었다고 해서 항상 데드락이 되는 것은 아니다. 다만 다른 실행 흐름이 그 락을 기다리게 되면 시스템 일부가 멈출 수 있고, 데드락이나 무한 대기 상태로 이어질 수 있다.

데드락을 설명할 때는 보통 4가지 필요 조건을 말한다.

```text
1. 상호 배제
   자원을 한 번에 하나만 사용할 수 있음

2. 점유와 대기
   어떤 자원을 가진 상태에서 다른 자원을 기다림

3. 비선점
   다른 실행 흐름이 가진 자원을 강제로 빼앗을 수 없음

4. 순환 대기
   A는 B를 기다리고, B는 A를 기다리는 식의 대기 고리가 있음
```

이 네 조건이 모두 성립하면 데드락이 발생할 수 있다. 그래서 데드락 예방은 보통 이 조건 중 하나가 성립하지 않게 만드는 방식으로 한다.

예를 들어 락 순서를 통일하면 순환 대기를 막을 수 있다.

```text
모든 스레드가 A -> B 순서로만 락 획득
        ↓
B를 잡고 A를 기다리는 상황이 없음
        ↓
순환 대기 방지
```

unlock 누락은 조금 다르게 봐야 한다. 한 스레드가 락을 잡고 반환해 버리면, 다른 스레드들은 그 락을 영원히 기다릴 수 있다.

```c
lock(A);

if (error) {
    return; // unlock(A) 누락
}

unlock(A);
```

이 경우는 엄밀한 의미의 순환 대기가 없을 수도 있다. 예를 들어 다른 스레드들이 모두 A 하나만 기다린다면 다음 상태가 된다.

```text
스레드 1: A를 잡은 채 종료 또는 반환
스레드 2: A 기다림
스레드 3: A 기다림
```

이건 실제 현상으로는 프로그램이 멈추지만, 교과서적인 "서로가 서로를 기다리는 순환 대기" 형태의 데드락과는 조금 다르게 설명될 수 있다. lock이 영원히 해제되지 않아 생긴 무한 대기 또는 lock leak에 가깝다.

하지만 실무에서는 이런 상황도 넓게 데드락처럼 표현하는 경우가 있다. 중요한 것은 용어보다 **락이 해제되지 않아 진행이 막힌다**는 사실이다.

정리하면 다음과 같다.

```text
데드락
- 보통 4가지 필요 조건이 모두 성립할 때 발생 가능
- 특히 순환 대기가 핵심

unlock 누락
- 항상 교과서적 데드락은 아닐 수 있음
- 하지만 다른 스레드가 영원히 기다리게 만들 수 있음
- 실무적으로는 데드락성 버그로 취급될 수 있음
```

따라서 "락이 없어져서 데드락"이라기보다는, **락이 해제되지 않거나 락 획득 순서가 꼬여서 실행 흐름들이 더 이상 진행할 수 없을 때 문제가 생긴다**고 이해하면 된다.
