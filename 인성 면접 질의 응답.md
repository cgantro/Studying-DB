### 1. 1분 자기소개

저는 프로젝트를 진행하며 두 가지 역량을 길러왔습니다.  

첫 번째는 **병목을 분석하고 구조적으로 개선하는 역량**입니다. RobotPal 프로젝트에서는 카메라 스트리밍 과정에서 렌더링과 인코딩 처리가 맞물리며 프레임 드랍이 발생하는 문제를 확인했고, 이를 생산자-소비자 구조로 분리해 처리 흐름을 개선한 경험이 있습니다.

두 번째는 **팀 기준을 문서+자동화로 고정하는 경험**입니다. Sticker에서 브랜치/커밋/MR에 Jira 키를 남기는 컨벤션을 문서로 정리했고(`CONVENTIONS.md`), `commit-msg` 훅으로 형식 위반을 커밋 단계에서 막았습니다. 백엔드는 `ApiResponse(code/message/data)` 표준 응답과 예외 매핑을 맞춘 뒤 테스트 기반으로 API 문서를 생성했습니다. 최근에는 `AGENTS.md`에 모노레포 경계와 금지사항을 적어 AI 도구도 같은 기준으로 작업하게 했습니다.

이러한 경험을 바탕으로 삼성SDS에서도 안정적이고 운영 가능한 소프트웨어를 만드는 데 기여하겠습니다..

### 2. 지원동기
키워드 : 실시간 데이터 처리, 상태 불일치 방지, 협업 기준 정리(특히 중요)
경험 : 팀 문서 협업 (좀 더 구체적으로)
제품군들을 보ㅇ

저는 프로젝트를 진행하며 실시간 데이터 처리, 상태 불일치 방지, 협업 기준 정리의 중요성을 경험했습니다. 특히 API 명세, Jira 기준, 커밋·브랜치 규칙을 정리하면서 협업의 핵심은 단순한 소통이 아니라, 팀이 같은 기준과 맥락으로 일할 수 있는 구조를 만드는 것이라고 느꼈습니다.

이후 삼성SDS의 Brity 제품군을 보며, 기업의 업무 흐름을 연결하고 반복 업무와 정보 불일치를 줄이는 시스템에 관심을 갖게 되었습니다. 저도 삼성SDS에서 기업 고객의 업무가 더 안정적이고 효율적으로 이어질 수 있도록, 협업과 업무 흐름을 시스템적으로 개선하는 개발자로 성장하고 싶습니다.

### 3. 장/단점
### 직무적 장점

**병목을 감으로 판단하지 않고, 구조와 흐름을 나눠서 분석하는 점**

> 제 직무적 장점은 문제를 감으로 해결하기보다, 데이터 흐름과 처리 구조를 나눠 원인을 찾는 점입니다.  
> RobotPal 프로젝트에서 카메라 스트리밍 중 프레임 드랍이 발생했을 때, 단순히 성능이 부족하다고 보기보다 렌더링, 인코딩, 큐 적재 흐름을 나눠 확인했습니다. 그 결과 렌더링과 인코딩이 같은 흐름에서 맞물리는 것이 병목이라는 점을 확인했고, 생산자-소비자 구조로 분리해 개선했습니다.  
> 저는 이런 식으로 문제를 구조적으로 쪼개고, 병목 지점을 찾아 개선하는 데 강점이 있습니다.

핵심 키워드:

- 병목 분석
- 데이터 흐름 분리
- 구조적 개선
- 측정 기반 개선

### 직무적 단점

제 직무적 단점은 문제를 처음 마주했을 때 구조와 기준을 먼저 명확히 잡으려는 성향이 있어, 초반 실행 속도가 늦어질 때가 있다는 점입니다.

예전에는 팀 프로젝트 초반에 Git 브랜치 규칙, Jira 상태 기준, API 명세 작성 방식, 커밋 규칙 등을 한 번에 정리하려다 보니, 실제 기능 구현보다 기준 정리에 시간이 먼저 쓰이는 경우가 있었습니다. 기준을 잡는 것 자체는 필요했지만, 모든 규칙을 처음부터 완성하려 하면 개발 속도가 늦어질 수 있다는 점을 느꼈습니다.

그래서 최근에는 방식을 바꿔, 처음부터 모든 기준을 만들기보다 **바로 충돌이 생길 수 있는 최소 기준만 먼저 정하고 구현을 진행**하고 있습니다. 예를 들어 API 연동이 필요한 기능이라면 응답 필드, 에러 형식, 인증 여부처럼 프론트와 백엔드가 바로 맞춰야 하는 부분만 먼저 고정하고, 나머지 규칙은 개발하면서 보완하는 식입니다.

AI Agent를 활용할 때도 마찬가지로, 처음부터 모든 자동화 규칙을 만들기보다 파일 위치, 코드 스타일, 수정 금지 범위, 테스트 실행 기준처럼 결과물에 바로 영향을 주는 항목부터 AGENTS.md에 정리하고 있습니다.

이 단점이 좋은 이유:

- 실제 단점처럼 들림
- 개발자 성향과 자연스럽게 연결됨
- 보완 방식이 구체적임
- SDS에서 “느린 사람”으로 보이지 않게 방어 가능

### 인성적 장점

고통에 내성이 강하다.

제 강점은 **무던하게 버티고 끝까지 해내는 태도**입니다.

군 복무 중 허리를 다쳐 전역하게 된 경험이 있습니다. 당시에는 걷는 것도 쉽지 않았고, 꾸준히 해오던 운동도 제대로 할 수 없어 많이 힘들었습니다.

하지만 상황을 바꿀 수 없다면, 지금 할 수 있는 것부터 하자는 생각으로 재활과 근력 보강을 꾸준히 이어갔습니다. 그 결과 현재는 많이 회복된 상태이고, 이 경험을 통해 어려운 상황에서도 감정에만 머무르지 않고 할 일을 찾아 지속하는 태도를 갖게 됐습니다.

삼성SDS에서도 처음 마주하는 업무나 어려운 상황이 있더라도 조급해하기보다, 필요한 것을 차근차근 배우고 끝까지 책임지는 구성원이 되겠습니다.

### 인성적 단점

제 약점은 문제 상황에서 상대의 감정보다 해결 방법을 먼저 생각하는 편이라는 점입니다.

해결도 중요하지만 팀으로서 상대의 감정 또한 고려하는 것도 팀의 효율에 큰 영향을 끼친다는 것을 알게되었습니다.

밴드마스터로 활동할 때 합주나 곡 방향이 맞지 않으면 빠르게 고치고 싶어서 “이건 이렇게 해야 한다”는 식으로 바로 피드백을 준 적이 있습니다. 제 의도는 팀 결과물을 좋게 만들기 위한 것이었지만, 나이 차이가 있는 후배들에게는 그 방식이 다소 직설적이고 어렵게 느껴질 수 있다는 말을 들었습니다.

이후에는 바로 해결책을 말하기보다 먼저 상대가 어떤 의도로 연주했는지, 어떤 부분이 어려웠는지 듣고 나서 방향을 제안하려고 노력했습니다. 이 경험을 통해 협업에서는 맞는 답을 말하는 것만큼, 상대가 받아들일 수 있는 방식으로 전달하는 것도 중요하다고 느꼈습니다.
