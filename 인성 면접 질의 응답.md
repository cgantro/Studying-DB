### 4. 어려움을 극복한 경험

C++에서는 멀티스레딩 기반으로 처리하던 구조가 웹 빌드 후에는 기대한 대로 동작하지 않았고, 카메라 프레임 처리 과정에서 프레임 드랍이 발생했습니다. 원인을 확인해보니 웹에서는 Emscripten pthread를 사용하더라도 SharedArrayBuffer를 쓰기 위한 브라우저 보안 조건, 즉 COOP와 COEP 설정이 필요했습니다.

그래서 빌드 옵션에서 pthread를 활성화하고, 정적 배포 환경에서도 해당 조건이 맞도록 설정을 보완했습니다. 이후 웹에서도 Web Worker 기반 멀티스레딩이 정상 동작하는 것을 확인했습니다.

이 경험을 통해 성능 문제를 해결할 때는 코드 구조뿐 아니라 빌드 옵션, 브라우저 보안 정책, 배포 환경까지 함께 봐야 한다는 점을 배웠습니다.
#### Q1. 웹은 싱글스레드 아닌가요?

> JavaScript에서 DOM을 직접 다루는 흐름은 메인 스레드 중심입니다. 하지만 Web Worker를 사용하면 DOM 접근이 필요 없는 계산, 인코딩, 데이터 처리 같은 작업은 별도 실행 흐름으로 분리할 수 있습니다. Emscripten pthread는 이 Web Worker를 활용해 C++의 pthread 모델을 웹 환경에서 동작하도록 매핑합니다.

#### Q2. Emscripten pthread를 쓰려면 뭘 설정해야 하나요?

> 컴파일과 링크 단계에 `-pthread` 옵션을 적용해야 합니다. 필요하면 `-sPTHREAD_POOL_SIZE`로 워커 풀 크기도 지정할 수 있습니다. 단, 브라우저에서 실제로 동작하려면 SharedArrayBuffer 사용 조건도 맞아야 합니다.

#### Q3. COOP, COEP가 뭔가요?

> SharedArrayBuffer는 보안상 아무 페이지에서나 사용할 수 없고, 페이지가 cross-origin isolated 상태여야 사용할 수 있습니다. 이를 위해 COOP와 COEP 응답 헤더가 필요합니다. 쉽게 말하면 브라우저가 “이 페이지는 안전하게 메모리를 공유해도 된다”고 판단하게 만드는 조건입니다.

#### Q4. 왜 GitHub Actions에서 문제가 됐나요?

> 정확히는 GitHub Actions의 문제가 아니라, 정적 배포나 테스트 서버 환경에서 COOP/COEP 응답 헤더를 직접 제어하기 어려웠던 문제였습니다. 빌드는 성공해도 브라우저 실행 시 보안 조건이 맞지 않으면 pthread가 정상 동작하지 않았습니다.

#### Q5. JS 코드를 넣었다는 게 무슨 뜻인가요?

> 서버에서 헤더를 직접 넣는 게 정석이지만, 당시 정적 배포 환경에서는 그게 어려웠습니다. 그래서 서비스워커 기반 스크립트를 빌드 산출물에 포함해 필요한 보안 조건을 보완했고, 이후 브라우저에서 멀티스레딩이 정상 동작하는 것을 확인했습니다.

#### Q6. 운영 환경이라면 어떻게 할 건가요?

> 운영 환경이라면 서비스워커 방식보다는 Nginx, Spring 서버, CDN 설정 등에서 COOP/COEP 응답 헤더를 명시적으로 내려주는 방식이 더 적절하다고 생각합니다. 당시에는 제한된 정적 배포 환경에서 문제를 해결하기 위한 방식이었습니다.

#### Q7. 이게 프레임 드랍 개선과 어떻게 연결되나요?

> 카메라 프레임 처리에서 인코딩 같은 무거운 작업이 메인 흐름에 영향을 주면 프레임 드랍이 발생할 수 있습니다. 그래서 Web Worker 기반으로 무거운 작업을 분리해 메인 처리 흐름의 부담을 줄이는 방향으로 개선했습니다.
### 5. 팀에서의 역할

팀에서는 주로 작업 기준과 개발 흐름을 맞추는 역할을 했습니다.
Jira에서는 Epic을 큰 기능 단위로 두고, 실제 구현은 Task로 나눴습니다. 각 Task에는 피보나치 점수를 부여하고, Epic에는 하위 Task 점수가 자동 집계되도록 해서 스프린트 작업량을 팀이 같이 볼 수 있게 했습니다.

또한 Git 브랜치와 커밋 규칙, 변경사항 공유 방식을 맞춰 어떤 작업이 어떤 이슈와 연결되는지 추적 가능하게 했습니다. 최근에는 AI Agent 활용 기준도 문서화해서, 생성된 코드가 팀의 구조와 컨벤션을 벗어나지 않도록 관리했습니다.

저는 팀에서 앞에서 지시하기보다는, 기준을 명확히 해서 협업 비용을 줄이는 역할을 주로 맡았습니다.

#### Q1. 왜 Epic에는 점수를 안줬는가?
> Epic은 기능 묶음 -> 추정단위가 너무 큼
> 실제 구현 및 리뷰 단위는 Task -> Task에만 점수

#### Q2. 피보나치 점수를 쓴 이유는?
> 작업량을 시간 단위로 정확히 예측하기 어려움
> 상대적 복잡도를 표현하려 함 -> 점점 간격이 커지기에, 복잡도가 큰 작업은 다시 쪼갤 수 있음
#### Q3. 점수는 누가 정하는가?
> 혼자 정하기 보다, 구현 담당자와 논의

#### Q5. 실제로 도움이 되었나?
> 점수로 보니 작업자의 작업량을 보기가 편했고, 기능이 얼마나 큰지 볼 수 있었음.

#### Q4. 구체적으로 어떤 Git 사용법을 알려줬나요?
1. 기본적으로 Develop브랜치에 올라온 최신 코드들을 Pull 후,  작업 단위에 맞게 브랜치를 생성한 후, 원격 저장소에 Push하는 흐름을 알려줌
2. 충돌 발생 시 어떤 부분이 다른지, 충돌난 부분에 대하여, 팀원과 소통 후 합쳐야 함.
#### Q5. 왜 진작 Git 사용법을 맞추지 않았는가?
1. SSAFY에서는 기본적으로 Git 사용법을 알려줌 -> 그래서 사용법 아는 줄 알았음
2. 실제로는 Git을 아는 것과 팀 프로젝트에서 같은 기준으로 사용하는 것은 다름
3. 브랜치 전략과 커밋 메시지 등의 최소 공통 규약으 맞추려고 함.
